## 核心考题：智慧体检全链路架构设计

### 业务背景

```text
某区域医疗集团（旗下 5 家三甲医院健康管理中心 + 20 家社区服务中心 + 30 家乡镇卫生院），
承接三类体检业务：

1. 公卫体检（65+ 老年人/慢病/孕产妇/0-6 岁儿童）- 财政全额拨款，免费
   - 单日单点峰值 200-500 人，体检车上门服务
   - 必须按国标上报省级公卫系统
   - 数据格式：国标 XML/JSON（非 HL7/FHIR）

2. 商业体检（个人/企业团检）
   - 套餐制（基础套餐 + 加项）
   - 个人支付 + 部分医保个人账户支付
   - 需对接集团内 5 家三甲医院 HIS/EMR

3. 入职体检 / 从业人员健康证体检
   - 企业付费 + 个人付费混合
   - 健康证需上传监管系统（疾控中心）

业务规模：
- 日均体检 5000 人，峰值 1w 人
- 5 家三甲 + 20 社区 + 30 乡镇 = 55 个机构
- 设备：B 超 100 台、CT 30 台、DR 50 台、心电 200 台、眼底 30 台、
        生化分析仪 50 台、血细胞分析仪 50 台、尿分析仪 50 台
- 检验：5 家三甲自有 LIS，社区/乡镇送检到三甲
- 影像：5 家三甲自有 PACS，社区/乡镇云 PACS

历史痛点：
  - 2025 年：公卫上报数据格式不规范，被省级退回 3 次
  - 2025 年：商业体检医保对接失败，居民需现场补交现金
  - 2025 年：体检车网络中断，30+ 人数据丢失
  - 2025 年：B 超厂家私有协议，影像不能进 PACS
  - 2025 年：5 家三甲 HIS 厂家不一（卫宁/东软/创业慧康），对接成本高
  - 2026 年：AI 肺结节筛查集成失败，主检医生看不到 AI 结果
  - 2026 年：体检报告跨机构调阅难，居民换机构后历史数据丢失
  - 2026 年：健康证体检数据未及时上传疾控，被监管处罚

目标：
  1. 全链路可用性 99.9%（年度宕机 < 8.7 小时）
  2. 体检数据完整率 100%（不丢失）
  3. 公卫上报成功率 99.5%+
  4. 医保结算成功率 99%+
  5. 跨机构调阅 < 3 秒
  6. 主检报告生成 < 30 分钟
  7. AI 辅诊集成率 80%+
  8. 多机构多标准兼容（HL7/FHIR/DICOM/国标）
```

### 设计目标

```text
1. 业务全覆盖：公卫/商业/健康证三类体检全流程数字化
2. 多机构协同：55 个机构数据互通，跨机构调阅
3. 多标准兼容：HL7 v2/v3、FHIR、DICOM、国标 WS445 共存
4. 多设备集成：B 超/CT/DR/心电/眼底/检验全设备协议适配
5. 多付费模式：公卫免费 + 商业自费 + 医保 + 混合支付
6. 资金安全：医保/商保/商业支付三方对账，差错自动处置
7. 高可用：体检车断网缓存，云端多 AZ 容灾
8. 可观测：全链路追踪，资金/数据双闭环
9. SLA：P99 < 500ms、可用性 99.9%、数据完整率 100%
```

### 二、全链路时序：一次完整体检的数据流

```text
T0  居民预约（小程序）
    - 选择套餐（公务员套餐 500 元）
    - 系统判定付费模式（INSURANCE_PERSONAL）
    - 调医保"待遇查询"接口核验参保状态
    - 生成预约号

T1  体检当日登记（社区中心）
    - 扫医保电子凭证
    - 调医保"身份核验"接口
    - 打印条码（多个标本条码）
    - 生成就诊号

T2  检中执行 - 各科室检查
    T2.1 身高体重/血压
        - 设备网关接收数据（蓝牙/串口）
        - 上传业务层
    T2.2 血常规/尿常规（采血）
        - 系统生成标本条码
        - 采血后送检验科
        - LIS 适配器接收结果（ASTM/HL7 ORU^R01）
        - 通过 Mirth Connect 转换为 FHIR Observation
        - 推送到业务层
    T2.3 B 超检查
        - 设备从 MWL 拉取当日体检清单（DICOM C-FIND）
        - 检查后 C-STORE 推送到 PACS
        - PACS 生成影像 ID
        - 通知业务层（影像就绪）
    T2.4 CT 检查（仅三甲有 CT）
        - 同 B 超，但走 CT 协议
        - AI 肺结节筛查自动触发
        - AI 推理结果生成 DICOM SR
        - 推送到 PACS + 通知业务层
    T2.5 心电检查
        - 设备网关接收数据（DICOM 或私有协议）
        - 转换为 FHIR Observation
        - 推送到业务层

T3  检中：医保事前审核
    - 每开一个检查项，实时调医保"事前审核"
    - 不合规医嘱提示医生

T4  检中：加项处理
    - 居民临时加项
    - 实时医保事前审核
    - 标记"加项未结算"（合并结算）

T5  全部检查完成
    - 业务层汇总所有结果
    - 触发主检报告生成

T6  主检报告生成
    - 主检医生审核异常项
    - 健康评估（基于检查结果）
    - 生成 PDF 报告
    - 生成 CDA 文档（ HL7 v3 标准）
    - 生成 FHIR DiagnosticReport 资源
    - 推送到居民端（小程序）

T7  医保结算
    - 拆分套餐（医保项目 450 + 自费项目 50）
    - 调医保"费用明细上传"
    - 调医保"结算" -> 医保支付 450
    - 商业支付 50（微信扫码）
    - 调医保"结算确认"
    - 生成发票 + 电子票据

T8  公卫上报（异步）
    - 国标格式打包（XML/JSON）
    - XMLDSig 签名
    - 定时批量上报省级公卫系统
    - 接收回执
    - 失败重试

T9  HIS/EMR 对接（异步）
    - 体检报告生成 FHIR DiagnosticReport
    - 推送到集团内 5 家三甲 HIS/EMR
    - 通过 Mirth Connect 路由
    - 各 HIS 接收后归档到 EMR

T10 跨机构调阅（FHIR）
    - 居民换机构体检
    - 新机构通过 FHIR Patient/$match 查询历史
    - 调阅历史 DiagnosticReport
    - 展示历史趋势

T11 随访工单（异步）
    - 异常项触发随访工单
    - 推送到家庭医生签约系统
    - 7 天/30 天后随访

T12 健康证体检特殊流程
    - 完成体检后
    - 数据上传疾控中心系统
    - 生成健康证（电子证照）
    - 推送给居民和用人单位
```

│  └─────────────────────────────────────────────────────────┘     │

**多标准协同设计原则**：

1. **业务领域模型统一**：内部用统一的领域模型，对外按需适配
2. **适配器模式**：每种标准一套适配器，互不干扰
3. **Mirth Connect 作为集成引擎**：负责 HL7 v2、FHIR、国标之间的消息转换
4. **HAPI FHIR Server 作为对外接口**：跨机构调阅走 FHIR 标准
5. **PACS 独立部署**：影像走 DICOM，不走 FHIR（FHIR 只存 ImagingStudy 引用）

**分片键选择**：

| 表 | 分片键 | 分片策略 |

| 居民档案 | id_card_hash | 按身份证 hash 分片 |

| 公卫上报记录 | (org_id, batch_no) | 按机构+批次 |

**断网缓存策略**：

| 数据类型 | 缓存策略 | 重传策略 |
|---|---|---|
| 体检订单 | 本地 DB 写入，标记"待上传" | 网络恢复后 FIFO 上传 |
| 检查结果 | 本地 DB + 本地 MQ 队列 | 异步上传，幂等去重 |
| 影像文件 | 本地 Orthanc + 本地文件系统 | STOW-RS 上传，失败重试 |
| 设备数据 | 本地 DB 缓存 | 上传后删除本地 |
| 医保结算 | **不可断网**（必须在线） | 网络恢复后到就近机构结算 |

**幂等设计**：

```java
// 体检车上传幂等
@Data
public class UploadTask {
    private String taskId;            // 任务 ID（UUID）
    private String bizType;           // 业务类型
    private String bizId;             // 业务 ID（订单号/结果 ID）
    private String payload;           // 上传内容（JSON）
    private Integer retryCount;       // 重试次数
    private String status;            // PENDING/SUCCESS/FAIL
    private LocalDateTime nextRetryTime;
}

// 上传逻辑
public void upload(UploadTask task) {
    try {
        UploadResponse resp = cloudApi.upload(task);
        if (resp.isSuccess()) {
            task.setStatus("SUCCESS");
            // 删除本地缓存
            localCache.delete(task.getBizId());
        } else {
            enqueueRetry(task, 60);  // 1 分钟后重试
        }
    } catch (NetworkException e) {
        enqueueRetry(task, 60);
    }
}

// 重试策略
void enqueueRetry(UploadTask task, int delaySeconds) {
    task.setRetryCount(task.getRetryCount() + 1);
    if (task.getRetryCount() > 10) {
        // 超过 10 次失败，告警
        alertService.alert("上传失败 10 次: " + task.getBizId());
        return;
    }
    // 指数退避
    int nextDelay = delaySeconds * (int) Math.pow(2, task.getRetryCount());
    task.setNextRetryTime(LocalDateTime.now().plusSeconds(nextDelay));
    task.setStatus("PENDING");
    save(task);
}
```

## 全链路架构师决策清单

### 决策 1：为什么用多标准协同，而不是统一用 FHIR？

| 方案 | 优点 | 缺点 | 选择 |
|---|---|---|---|
| 统一 FHIR | 标准化、简单 | 国内医保/HIS 不支持 FHIR | ❌ |
| 多标准协同 | 兼容现有系统、可落地 | 复杂度高 | ✓ |

> 架构师要点：理想与现实的平衡。FHIR 是趋势，但国内 HIS/医保/公卫都还不支持 FHIR。多标准协同是当前最优解。

### 决策 2：为什么用 Mirth Connect 而不是自研集成引擎？

| 方案 | 优点 | 缺点 | 选择 |
|---|---|---|---|
| 自研 | 完全可控 | 成本高、踩坑多 | ❌ |
| Mirth Connect | 开源、成熟、社区活跃 | 学习曲线、性能调优 | ✓ |
| 商业（Rhapsody） | 专业、稳定 | 贵 | 仅大型三甲 |

### 决策 3：为什么 PACS 用 Orthanc 而不是商业 PACS？

| 方案 | 优点 | 缺点 | 选择 |
|---|---|---|---|
| 商业 PACS | 稳定、专业 | 贵、定制难 | 仅大型三甲 |
| Orthanc | 开源、插件丰富、易扩展 | 性能需调优 | ✓ |
| dcm4chee | 开源、企业级 | 部署复杂 | 备选 |

### 决策 4：为什么用 HAPI FHIR Server 而不是自研？

| 方案 | 优点 | 缺点 | 选择 |
|---|---|---|---|
| 自研 | 完全可控 | 成本极高 | ❌ |
| HAPI FHIR | 开源、Java 生态、社区活跃 | 性能需调优 | ✓ |
| 商业 FHIR | 稳定、有支持 | 贵 | 备选 |

### 决策 5：体检车断网缓存用什么？

| 方案 | 优点 | 缺点 | 选择 |
|---|---|---|---|
| 本地文件 | 简单 | 不可靠 | ❌ |
| SQLite | 轻量 | 并发弱 | 仅单机 |
| PostgreSQL + RocketMQ | 可靠、并发好 | 资源占用 | ✓ |

### 决策 6：混合支付的回滚用 Saga 还是 XA？

| 方案 | 优点 | 缺点 | 选择 |
|---|---|---|---|
| XA | 强一致 | 医保/商保/支付不支持 XA | ❌ |
| Saga + 补偿 | 可落地 | 最终一致 | ✓ |
| TCC | 强一致 | 改造医保/商保接口 | ❌（不可行） |

### 决策 7：多机构 SaaS 隔离用哪种方式？

| 方案 | 适用场景 | 选择 |
|---|---|---|
| 数据库级 | 三甲（数据量大 + 合规） | ✓ 5 家三甲 |
| Schema 级 | 社区（数据量中） | ✓ 20 家社区 |
| 行级 | 乡镇（数据量小） | ✓ 30 家乡镇 |

> 架构师要点：没有"一刀切"的方案，按数据量、合规要求、成本综合选型。

1. **全链路视角**：智慧体检不只是"业务系统"，而是"业务+医疗+影像+资金"的多链路协同

6. **断网缓存**：体检车边缘节点 + 云端协同，断网不丢数据
7. **AI 辅诊集成**：DICOM SR + 异步推理 + 医生审核的工作流
