## 背景

Day01 聚焦智慧体检整体架构，题目三触及了"集成模式选型"（点对点 / 集成引擎 / HL7 v2 / FHIR），但未深入 HIS/EMR 本身。Day02 把镜头拉近到医疗信息化的"中枢系统"——HIS 与 EMR，搞清楚它们的边界、数据模型、与体检系统的对接方式，为 Day03（HL7/FHIR）、Day04（DICOM/PACS）、Day05（医保对接）打底。

为什么 HIS/EMR 对你重要？

1. **跳槽到更好的医疗公司**：医疗信息化公司（卫宁、东软、创业慧康、嘉和美康、医惠）的产品矩阵都以 HIS/EMR 为核心，体检系统是其中子产品。不懂 HIS/EMR 的架构师，在医疗信息化公司是"边缘岗位"。
2. **公卫体检的下游是医院**：公卫体检发现的异常项需要转诊到医院，居民在医院又有自己的 EMR。跨机构就诊数据归集是"居民健康档案"的核心难题，本质是 EMPI + EMR 互通。
3. **三甲医院体检中心是高价值客户**：体检系统作为 HIS 子系统对接 EMR/LIS/PACS，是三甲体检中心的标配。乡镇场景（无 HIS）和三甲场景（完整 HIS）的差异化设计，是医疗 SaaS 架构师的核心能力。
4. **医疗数据合规底线**：电子病历归档有法律要求（门诊 15 年、住院 30 年），病案归档不可篡改是法律红线。体检报告是否适用、归档格式如何选，是医疗架构师必须搞清的合规边界。

### 作答区

#### 1. HIS 与 EMR 的区别与联系

**概念辨析**：

| 系统 | 全称 | 范畴 | 重点 | 起源 |
|---|---|---|---|---|
| HIS | Hospital Information System | 医院全业务信息化 | 业务流转与管理（挂号/收费/药品/物资/医保） | 80 年代财务/管理起步 |
| EMR | Electronic Medical Record | 患者医疗记录电子化 | 临床数据与医疗文档（病程/医嘱/检查检验/手术/护理） | 90 年代临床起步 |
| EHR | Electronic Health Record | 跨机构健康记录 | 全生命周期健康档案 | 2000 年后区域卫生 |

**联系与边界**：
- EMR 是 HIS 的临床核心子系统，但越来越多被独立采购
- 国内现状：HIS 厂家通常打包 EMR（卫宁、东软、创业慧康），但大型三甲会独立采购 EMR（嘉和美康、医惠）
- 数据归属：HIS 持有就诊/医嘱/收费主数据，EMR 持有病程/文档/临床观察数据
- EHR 在国内由"区域卫生信息平台"承载，对应居民的"健康档案"

**三甲医院信息系统生态与数据流**：

```
                    ┌──────────────────────────┐
                    │   患者主索引 EMPI         │
                    │   （跨系统患者身份匹配）  │
                    └────────────┬─────────────┘
                                 │
        ┌────────────────────────┼────────────────────────┐
        │                        │                        │
        ▼                        ▼                        ▼
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│  HIS         │         │  EMR         │         │  体检系统    │
│              │         │              │         │              │
│ 挂号/收费    │◄────────│ 病程/医嘱    │         │ 预约/报到    │
│ 医保结算     │  就诊号  │ 手术/护理    │         │ 检中/主检    │
│ 药品/物资    │  回流    │ 文档归档     │         │ 报告/上报    │
│ 财务/统计    │         │ 临床决策支持 │         │ 随访工单     │
└──────┬───────┘         └──────┬───────┘         └──────┬───────┘
       │                        │                        │
       │  Order（医嘱/申请单）   │ Observation（结果回流） │ Order
       ▼                        ▼                        ▼
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│  LIS 检验    │         │  PACS 影像   │         │  RIS 放射    │
│ 标本/结果    │         │ 影像存储     │         │ 检查预约     │
│ 危急值推送   │         │ WADO/QIDO   │         │ 报告录入     │
└──────────────┘         └──────────────┘         └──────────────┘

       ┌───────────────────────────────────────────┐
       │  手麻系统 ｜ 合理用药 ｜ 院感 ｜ 病案统计 ｜ 抗菌药物  │
       └───────────────────────────────────────────┘
```

**数据流要点**：
- HIS 挂号 → 创建 Visit → EMR 接管临床记录
- EMR 开检查检验医嘱（Order）→ LIS/PACS/RIS 接收 → 结果回写 EMR（Observation）
- 体检系统独立运行，但需获取患者基本信息（从 HIS/EMPI）+ 回写体检报告（到 EMR）
- 手麻系统从 EMR 接手术通知 → 术后回写麻醉记录
- 合理用药从 EMR 接处方 → 校验药物相互作用 → 回写警示

**体检系统：HIS 子模块 vs 独立系统对比**：

| 维度 | HIS 子模块 | 独立系统 |
|---|---|---|
| 数据归属 | 体检数据在 HIS 库 | 体检数据独立库 |
| 升级节奏 | 跟 HIS 升级 | 独立升级 |
| 定制灵活性 | 受 HIS 框架约束 | 自由定制 |
| 集成成本 | 低（同库） | 高（需对接） |
| 厂家绑定 | 强（必须同厂家） | 弱（可异厂家） |
| 多机构支持 | 难（HIS 单机构部署） | 易（SaaS 天然支持） |
| 适用场景 | 三甲医院体检中心 | 体检中心/乡镇/连锁 |

**结论**：
- 三甲医院体检中心：可作为 HIS 子模块（同厂家），也可独立部署（异厂家，更常见）
- 乡镇卫生院：独立系统（HIS 简化或不存在）
- 体检连锁/公卫体检：必须独立（多机构 SaaS）
- 啄木鸟云健康场景：必须独立，因为服务多家乡镇卫生院，HIS 不可控

#### 2. HIS/EMR 核心数据模型

**五大核心实体**：

| 实体 | 含义 | 关键字段 | 关联 |
|---|---|---|---|
| Patient | 患者（人口学信息） | patient_id, id_card, name, gender, birth_date | 1:N Visit |
| Visit/Encounter | 就诊（一次就医过程） | visit_id, patient_id, visit_type, start_time, end_time, dept | 1:N Order, 1:N Document |
| Order | 医嘱/申请单 | order_id, visit_id, order_type, code, status, exec_time | 1:N Observation |
| Observation | 观察项（检验/生命体征） | observation_id, order_id, code, value, unit, time | N:1 Order |
| Document | 文档（病程/报告） | doc_id, visit_id, type, content, author, time | N:1 Visit |

**三种 Visit 模型对比**：

| 维度 | 门诊 Visit | 住院 Visit | 体检 Visit |
|---|---|---|---|
| visit_type | OUTPATIENT | INPATIENT | EXAM |
| 时长 | 几小时 | 几天 | 几小时 |
| 病床 | 无 | 必有 | 无 |
| 主治医生 | 门诊医生 | 主治医生/主任 | 主检医生 |
| 医嘱 | 门诊处方 | 长期/临时医嘱 | 体检项目（必检/选检） |
| 病程记录 | 简单门诊病历 | 完整病程（首次/日常/出院） | 体检报告 |
| 结算 | 当场结算 | 出院结算 | 免费（公卫）/当场（商业） |
| 与 EMR 关系 | 门诊病历归档 | 出院病案归档 | 体检报告归档 |

**体检流程到 Visit/Order 的映射**：

| 体检阶段 | 数据模型操作 |
|---|---|
| 预约 | 创建 Visit（status=BOOKED） |
| 报到 | Visit 状态转 CHECKED_IN，关联多个 Order（每检查项一个 Order） |
| 检中 | Order 状态变化（STARTED → COMPLETED），Observation 落库 |
| 主检 | 生成 Document（type=EXAM_REPORT，关联 Visit） |
| 报告 | Document 状态 PUBLISHED，触发归档 |

**关键设计点**：
- 体检 Visit 与门诊/住院 Visit 共享 Patient 主键（通过 EMPI），但 visit_type 隔离
- 体检 Order 与门诊医嘱共用 Order 表，但 order_type=EXAM_ITEM
- 体检报告是 Document 的一种，与门诊病历、出院小结并列
- 体检的"漏项"在 Order 层处理（status=PARTIAL_COMPLETED），不污染 Visit 状态机

**适配层放哪？**

| 部署位置 | 优势 | 劣势 | 适用场景 |
|---|---|---|---|
| 体检系统侧 | 开发简单，无需额外组件 | 每家体检系统都要重写适配 | 乡镇/小机构（无集成引擎） |
| 集成引擎侧 | 通用，多系统共享 | 需部署 Mirth Connect 等 | 三甲（多系统集成） |
| **混合方案** | 兼顾两者 | 运维复杂 | 大型 SaaS 平台 |

**结论**：
- 三甲场景：适配层放集成引擎侧（Mirth Connect），通过 Channel 配置屏蔽厂家差异
- 乡镇场景：适配层放体检系统侧（无需集成引擎），SPI 化的适配器插件加载
- 啄木鸟云健康场景：乡镇为主，适配层放体检系统侧；若承接三甲项目，私有化部署时改用集成引擎

4. 数据安全法/个保法/人口健康信息管理办法下，跨机构患者数据共享的合规边界？患者授权机制如何设计？数据脱敏与去标识化在 EMPI 中的应用？

### 作答区

#### 1. EMPI 概念与匹配算法

**EMPI（Enterprise Master Patient Index，企业级/区域级患者主索引）**：跨系统统一患者身份，将不同系统中的同一患者关联到一个全局 ID（Global Patient ID）。

**为什么必须有 EMPI**：

| 场景 | 没 EMPI 的痛点 | EMPI 解决 |
|---|---|---|
| 同一患者多次就诊 | 每次 VISIT 不同 ID，历史数据散落 | 全局 ID 串联所有 Visit |
| 跨科室转诊 | 门诊号 vs 住院号 vs 体检号 | 全局 ID 跨科室关联 |
| 跨机构就诊 | A 院就诊号 vs B 院就诊号 | 全局 ID 跨机构关联 |
| 居民健康档案 | 各机构数据无法归集 | 全局 ID 聚合健康档案 |
| 医保结算 | 医保号 vs 就诊号 | 全局 ID 关联医保 |

**匹配算法**：

**① 确定性匹配（Deterministic Matching）**：
- 用唯一标识精确匹配：身份证、社保卡、护照、军官证
- 简单可靠，O(1) 复杂度
- 国内场景：身份证是主键，覆盖率 95%+
- 实现：`SELECT * FROM patient WHERE id_card_hash = ?`

**② 概率性匹配（Probabilistic Matching，Fellegi-Sunter 算法）**：
- 多字段加权匹配，计算匹配概率
- 字段权重：姓名（0.3）+ 生日（0.2）+ 性别（0.1）+ 地址（0.1）+ 电话（0.3）
- 计算匹配概率：`P(同人|字段匹配) = Σ(w_i × match_i)`
- 三档阈值：
  - P > 0.85：自动匹配
  - 0.5 < P < 0.85：人工审核
  - P < 0.5：不匹配
- 适用：无唯一标识场景（新生儿、外籍、隐私脱敏）

**国内身份证主键策略**：

```java
public class EmpiMatcher {
    // 第一优先级：身份证号哈希匹配
    public MatchResult matchByIdCard(String idCard, String name, String birthDate) {
        String hash = sha256(idCard + SALT);
        Patient patient = patientRepo.findByHash(hash);
        if (patient != null && patient.name.equals(name)) {
            return MatchResult.autoMatch(patient.globalId);
        }
        return MatchResult.uncertain();
    }
    
    // 第二优先级：无身份证走概率匹配
    public MatchResult matchProbabilistic(PatientInfo info) {
        List<Patient> candidates = patientRepo.findByBirthDateAndName(info.birthDate, info.name);
        double maxScore = 0;
        Patient best = null;
        for (Patient p : candidates) {
            double score = calculateScore(info, p);
            if (score > maxScore) { maxScore = score; best = p; }
        }
        if (maxScore > 0.85) return MatchResult.autoMatch(best.globalId);
        if (maxScore > 0.5)  return MatchResult.manualReview(best, maxScore);
        return MatchResult.createNew();
    }
    
    private double calculateScore(PatientInfo a, Patient b) {
        double score = 0;
        if (a.name.equals(b.name)) score += 0.3;
        else if (pinyin(a.name).equals(pinyin(b.name))) score += 0.2; // 同音字
        if (a.birthDate.equals(b.birthDate)) score += 0.2;
        if (a.gender.equals(b.gender)) score += 0.1;
        if (a.phone != null && a.phone.equals(b.phone)) score += 0.3;
        if (a.address != null && a.address.equals(b.address)) score += 0.1;
        return score;
    }
}
```

**新生儿、外籍、隐私脱敏场景**：

| 场景 | 主键 | 处理 |
|---|---|---|
| 新生儿 | 母亲身份证 + 出生序号 | 生成临时 ID（如 MOM_ID + BIRTH_SEQ），办身份证后关联 |
| 外籍人士 | 护照号 + 国籍 | 护照号做主键，国籍做校验 |
| 港澳台 | 港澳通行证 / 台胞证 | 证件号 + 证件类型做主键 |
| 隐私脱敏 | 身份证 SHA-256 + 盐 | 跨机构用同一哈希匹配，原值不存储 |
| 无证件急诊 | 急诊号 + 临时特征 | 概率匹配，事后补身份证关联 |

#### 2. 跨系统患者匹配的难点

**难点与对策**：

| 难点 | 原因 | 对策 |
|---|---|---|
| 同人不同就诊号 | 各机构独立发号 | EMPI 全局 ID 关联多个本地就诊号 |
| 身份证升位（15→18 位） | 老档案 15 位，新档案 18 位 | 升位算法统一处理（出生年补全、末位校验码重算） |
| 身份证更正（重号/错号） | 公安重号清理、录入错误 | 走人工审核，关联历史记录，不删除 |
| 姓名同音不同字（张三/张叁） | 录入习惯、简繁转换 | 拼音匹配 + 概率加权 |
| 姓名变更（更名/婚姻） | 户政变更、结婚随夫姓 | 保留曾用名，曾用名也参与匹配 |
| 地址变更 | 搬家、拆迁 | 地址权重低，不作为关键匹配字段 |
| 电话变更 | 换号频繁 | 电话权重中，仅作辅助 |
| 匹配冲突 | 同身份证不同人 / 同人不同身份证 | 人工审核队列 |

**匹配冲突解决流程**：

```
匹配引擎发现冲突
   │
   ├─ 同身份证号 + 不同姓名 + 不同生日
   │    → 极少见，可能是录入错误或冒用身份证
   │    → 进入人工审核队列
   │
   ├─ 同身份证号 + 不同姓名 + 同生日
   │    → 可能是更名
   │    → 人工审核，核对户口本/公安证明
   │
   ├─ 同身份证号 + 同姓名 + 不同生日
   │    → 肯定是错误
   │    → 标记异常，人工介入
   │
   └─ 概率匹配模糊（0.5 < P < 0.85）
        → 人工审核队列
        → 审核员核对身份证照片、户口本等
```

**人工审核机制**：
- 审核队列：所有模糊匹配进队列，由病案室/EMPI 管理员审核
- 审核依据：身份证原件扫描、户口本、医保卡
- 审核留痕：审核人、审核时间、决策依据、关联结果
- 撤销机制：错误关联可撤销，但保留撤销日志

#### 3. 体检系统获取就诊历史的方式

**三种数据交换方式对比**：

| 方式 | 时延 | 适用场景 | 实现复杂度 | 国内现状 |
|---|---|---|---|---|
| 实时查询 | <1s | 体检前调阅历史 | 低 | 单机构内常用 |
| 批量同步 | 分钟级 | 跨机构数据归集 | 中 | 区域平台常用 |
| 文档共享 | 小时级 | 区域卫生平台 | 高 | IHE XDS 标准 |

**实时查询（同机构内）**：
```
体检系统 → FHIR/HL7 v2 查询接口 → HIS/EMR → 返回就诊记录
```
- 实现：体检报到时调 EMR 接口，拉取近 1 年门诊诊断、住院记录
- 限制：仅本机构数据，跨机构需区域平台

**批量同步（跨机构）**：
```
区域卫生信息平台
   ├─ CDC 同步 A 院就诊数据
   ├─ CDC 同步 B 院就诊数据
   └─ CDC 同步 C 院就诊数据
       │
       ▼
    居民健康档案（按 EMPI 全局 ID 聚合）
       │
       ▼
体检系统 → 查询居民健康档案 → 获取跨机构历史
```
- 实现：区域平台 CDC 同步各机构数据，按 EMPI 聚合
- 限制：依赖区域平台建设水平（江浙沪较好，西部差）

**文档共享（IHE XDS）**：
```
A 院 → 提交文档（CDA）→ 文档库（XDS.b）
B 院 → 查询文档（按 EMPI）→ 调阅 → 跨机构共享
```
- 实现：标准化文档共享，跨机构调阅
- 限制：国内 XDS 落地少，多用于区域平台试点

**区域卫生信息平台/居民健康档案平台的角色**：

| 角色 | 职责 |
|---|---|
| EMPI 中枢 | 跨机构患者身份匹配 |
| 数据聚合 | CDC 同步各机构就诊数据 |
| 健康档案 | 按居民全局 ID 聚合健康档案 |
| 数据共享 | 提供跨机构数据查询接口 |
| 监管上报 | 公卫数据上报的汇聚点 |

**啄木鸟云健康场景的现实**：
- 乡镇卫生院的居民往往在县医院看过病，县医院有 HIS
- 体检系统无法直接调县医院 EMR（无对接）
- 现实方案：体检时由村医/医生人工询问既往史，录入体检系统
- 长期方案：接入省级居民健康档案平台（如福建、浙江），跨机构调阅

#### 4. 跨机构数据共享的合规

**法律框架**：

| 法律 | 重点 | 对 EMPI 的影响 |
|---|---|---|
| 《数据安全法》 | 数据分级分类，医疗数据为敏感数据 | 跨机构共享需数据分级评估 |
| 《个人信息保护法》 | 敏感个人信息单独同意 | 跨机构共享需患者单独授权 |
| 《人口健康信息管理办法》 | 人口健康信息不得违规出境 | 跨境共享禁止 |
| 《健康医疗数据安全指南》 | 数据分类分级、脱敏规则 | 脱敏后共享 |
| 等保 2.0 | 三级等保是医疗系统标配 | EMPI 系统必须过等保三级 |

**合规边界**：

| 共享场景 | 合规要求 |
|---|---|
| 同机构内科室间共享 | 患者挂号即默认授权，无需单独授权 |
| 跨机构数据共享 | 必须患者单独授权，签署《数据共享授权书》 |
| 跨机构调阅影像 | 患者授权 + 影像脱敏（去除敏感标注） |
| 公卫上报 | 法定义务，无需患者授权（公卫规范要求） |
| 医保结算 | 医保协议覆盖，无需单独授权 |
| 科研使用 | 必须去标识化 + 伦理审查 |
| 商业保险理赔 | 患者授权 + 最小必要原则 |

**患者授权机制**：

```
体检报到时：
   ├─ 居民签署《数据共享授权书》（纸质/电子）
   ├─ 授权范围：明确授权哪些机构、哪些数据、用于什么目的
   ├─ 授权期限：1 年 / 长期 / 单次
   ├─ 授权撤销：患者可随时撤销
   └─ 授权记录：留痕（授权人、时间、范围、撤销记录），可审计

授权流转：
   体检系统 → 调区域平台查就诊历史
              ↓
   区域平台校验授权记录
              ↓
   授权有效 → 返回数据（脱敏）
   授权无效/未授权 → 拒绝 + 引导授权
```

**数据脱敏与去标识化**：

| 字段 | 脱敏规则 | EMPI 中应用 |
|---|---|---|
| 身份证号 | SHA-256 + 盐哈希 | 跨机构用哈希匹配，原值不存 |
| 姓名 | 显示脱敏（张*三） | EMPI 内部存原名，对外脱敏 |
| 电话 | 中间 4 位掩码 | EMPI 内部存原号，对外脱敏 |
| 地址 | 模糊到街道级 | EMPI 内部存详细地址，对外模糊 |
| 生日 | 模糊到月 | 科研用途时模糊到月 |
| 病历内容 | 去标识化（替换姓名/身份证/地址） | 科研用途时去标识 |

**关键设计**：
- EMPI 系统存储原值（用于匹配），对外接口返回脱敏值
- 跨机构共享的数据必须脱敏（除医疗必需场景）
- 脱敏规则按"用途"动态选择：临床用全量、科研用去标识、统计用匿名
- 审计日志：所有调阅记录留痕（谁调阅、调阅谁的、什么数据、什么时间）

### 作答区

#### 1. 体检结果回写 EMR 的方式

**三种回写方式对比**：

| 方式 | 实现 | 优势 | 劣势 | 适用场景 |
|---|---|---|---|---|
| 结构化回写 | CDA Entry/Observation（HL7 v3 模型） | 可查询、可统计、可二次分析 | 实现复杂，EMR 必须支持 CDA | 检验结果、生命体征、诊断 |
| 文档回写 | CDA Document / PDF | 简单、完整、人读友好 | 不可查询、不可统计 | 体检报告整体、影像报告 |
| 引用回写 | 影像 URL / PACS 链接 | 节省存储、避免数据冗余 | 依赖原系统可用，链接失效则失访 | 影像（DICOM 影像太大） |

**结构化回写示例（CDA Observation）**：

```xml
<Observation classCode="OBS" moodCode="EVN">
  <code code="39156-5" codeSystem="LOINC" displayName="Body mass index"/>
  <value xsi:type="PQ" value="24.5" unit="kg/m2"/>
  <effectiveTime value="20260707"/>
  <performer><assignedEntity>...</assignedEntity></performer>
</Observation>
```

**体检报告回写到 EMR 的章节**：

CDA Body 结构：
```
<ClinicalDocument>
  <header>...</header>
  <component>
    <structuredBody>
      <section> 基本信息 </section>
      <section> 检查项（血压/身高/体重/B超/心电） </section>
      <section> 检验项（血常规/生化/尿常规） </section>
      <section> 影像（CT/MRI/X光） </section>
      <section> 主检结论与建议 </section>
      <section> 异常项与随访建议 </section>
    </structuredBody>
  </component>
</ClinicalDocument>
```

**回写到 EMR 章节的映射**：

| 体检内容 | EMR 章节 | 回写方式 |
|---|---|---|
| 血压、BMI、生命体征 | 生命体征章节 | 结构化（Observation） |
| 检验结果（血常规、生化） | 检验结果章节 | 结构化（Observation） |
| 影像报告 | 检查报告章节 | 文档（PDF）+ 引用（PACS URL） |
| 主检结论 | 诊断章节 | 结构化（Diagnosis） |
| 体检报告整体 | 检查报告章节 | 文档（CDA + PDF） |

**关键设计**：
- 体检报告整体走文档回写（PDF + CDA 双格式）
- 关键指标（血压、血糖、血脂、肝肾功能）走结构化回写，便于 EMR 临床决策支持（如血压高自动提醒）
- 影像走引用回写，避免影像数据冗余
- 三种方式并存，覆盖不同场景

#### 2. 电子病历归档机制

**操作库 vs 归档库**：

| 维度 | 操作库（EMR 运行库） | 归档库（EMR 病案库） |
|---|---|---|
| 用途 | 日常诊疗读写 | 病案归档（不可篡改） |
| 时效 | 实时 | 离线 |
| 修改 | 允许 | 不允许（仅追加修改记录） |
| 保留 | 在院期间 + 1 个月 | 门诊 15 年、住院 30 年 |
| 存储 | 关系库（Oracle/MySQL） | WORM 存储 + 关系库索引 |
| 访问 | 高频读写 | 低频只读 |
| 备份 | 日备 + 增量 | 长期保留 + 异地容灾 |

**归档时机**：

| 文档类型 | 归档时机 |
|---|---|
| 门诊病历 | 就诊结束 + 报告全部出齐后归档（一般 7 天内） |
| 住院病历 | 出院后 7 天内归档（病案室审核） |
| 体检报告 | 报告发布后 30 天内归档（公卫上报成功后） |
| 急诊病历 | 急诊结束后归档 |

**不可篡改与法律效力**：

| 措施 | 实现方式 | 法律依据 |
|---|---|---|
| 电子签名 | CA 证书签名（医生签名 + 医院签章） | 《电子签名法》第 13 条可靠电子签名 |
| 时间戳 | 国家授时中心时间戳 | 《电子签名法》第 15 条 |
| 哈希校验 | SHA-256 哈希，定期校验完整性 | 司法证据规范 |
| 修改留痕 | 归档后修改必须留痕（修改人/时间/原因/原值/新值） | 《电子病历应用管理规范》第 14 条 |
| 防篡改存储 | WORM（Write Once Read Many）存储 | 病案管理规范 |

**归档库技术选型对比**：

| 方案 | 优势 | 劣势 | 适用场景 |
|---|---|---|---|
| 关系库 + WORM 存储 | 成熟、成本低 | 实施复杂 | 大型医院主流 |
| 区块链存证 | 不可篡改性最强 | 成本高、性能低 | 高价值病历 |
| 电子签名 + 时间戳 | 法律效力明确 | 仅证明签名时刻，不防事后篡改 | 中小医院 |
| 对象存储 + 版本控制 | 简单、可扩展 | 不防删除 | 体检报告等非核心病历 |

**推荐方案**：
- 三甲医院：关系库 + WORM 存储 + 电子签名 + 时间戳（全套）
- 区域平台/体检连锁：对象存储 + 版本控制 + 区块链存证（哈希上链）
- 乡镇卫生院：电子签名 + 时间戳（轻量）

#### 3. 体检报告归档要求

**保留期限**：

| 文档类型 | 保留期限 | 法律依据 |
|---|---|---|
| 门诊病历 | 15 年 | 《医疗机构病历管理规定》 |
| 住院病历 | 30 年 | 《医疗机构病历管理规定》 |
| 体检报告（医院内） | 15 年（按门诊标准） | 比照门诊病历 |
| 公卫体检健康档案 | 永久 | 《国家基本公共卫生服务规范》 |
| 急诊病历 | 15 年 | 比照门诊病历 |

**体检报告归档格式**：

| 格式 | 用途 | 优势 |
|---|---|---|
| CDA | 标准化交换 | 可结构化查询、跨机构交换 |
| PDF | 人读、法律证据 | 法律效力强、人读友好 |
| 原始结构化数据（JSON） | 二次分析、统计 | 可查询、可统计 |
| 影像（DICOM） | 影像调阅 | 标准影像格式 |

**建议四者都归档**：
- CDA：跨机构交换（区域平台）
- PDF：法律证据、患者调阅
- 原始 JSON：统计、二次分析
- DICOM：影像（仅影像类体检项目）

**归档后修改留痕**：

```
归档后修改流程：
   ├─ 不允许直接修改原报告
   ├─ 走"补充报告"流程
   ├─ 生成补充报告 V2（标注"补充"，关联原报告 V1）
   ├─ 修改审计日志：
   │    - 修改人（主检医生 + 审批人）
   │    - 修改时间
   │    - 修改原因
   │    - 原值
   │    - 新值
   └─ 法律效力：补充报告与原报告并存，不替代
```

**修改触发场景**：
- 主检结论错误（如漏诊、误诊）
- 检查数据错误（如设备故障导致血压值错）
- 患者信息错误（如身份证号错）
- 公卫上报后发现错误

#### 4. 公卫健康档案 vs 医院 EMR

**两者对比**：

| 维度 | 公卫健康档案 | 医院 EMR |
|---|---|---|
| 性质 | 居民健康档案（含基本信息 + 体检 + 随访） | 医疗病历（含就诊全流程记录） |
| 归属 | 区域卫生信息平台 | 医院 |
| 标准 | 《国家基本公共卫生服务规范》 | 《电子病历应用管理规范》 |
| 数据来源 | 公卫体检、随访、村医录入 | 医院门诊、住院、检查检验 |
| 修改 | 可修改（随访记录会更新） | 归档后不可修改（仅追加） |
| 保留 | 永久 | 门诊 15 年、住院 30 年 |
| 是否属于电子病历 | 严格意义上属于"健康档案"，不属于"电子病历" | 是电子病历 |

**公卫健康档案不是严格意义上的电子病历**：
- 电子病历：单次就诊的完整记录（门诊/住院）
- 健康档案：跨机构、跨时间的健康数据聚合
- 法律上：电子病历受《病历管理规定》约束，健康档案受《人口健康信息管理办法》约束

**归档差异**：

| 维度 | 公卫健康档案 | 医院 EMR |
|---|---|---|
| 归档时机 | 实时归档（每次随访/体检即更新） | 出院后批量归档 |
| 归档格式 | 国标 XML/JSON | CDA + PDF |
| 修改机制 | 可修改（追加新版本） | 归档后不可改（仅补充报告） |
| 归档主体 | 区域平台 | 医院病案室 |

**跨机构居民健康档案聚合**：

```
区域卫生信息平台
   ├─ A 院 EMR 数据（CDC 同步）
   ├─ B 院 EMR 数据（CDC 同步）
   ├─ 公卫体检数据（公卫系统同步）
   ├─ 随访数据（社区中心同步）
   │
   ▼
按 EMPI 全局 ID 聚合
   │
   ▼
居民健康档案（按时间线串联）
   ├─ 2026-01 A 院门诊（高血压诊断）
   ├─ 2026-03 公卫体检（血压 145/90）
   ├─ 2026-05 B 院住院（脑梗）
   ├─ 2026-06 社区随访（血压 130/80）
   └─ ...
```

**跨机构数据冲突解决**：

| 冲突类型 | 解决策略 |
|---|---|
| 同一时间不同机构诊断不同 | 保留所有诊断，标注来源机构 |
| 同一指标不同值（如血压） | 保留所有记录，按时间排序，不合并 |
| 基本信息冲突（如地址） | 以最近一次为准，旧值归档 |
| 诊断冲突（A 院高血压 vs B 院无高血压） | 保留所有诊断，由家庭医生人工调和 |

**区域卫生信息平台的角色**：
- 数据汇聚：CDC 同步各机构数据
- 身份匹配：EMPI 跨机构关联
- 数据共享：提供跨机构查询接口
- 数据治理：去重、清洗、标准化
- 监管上报：公卫数据上报

**啄木鸟云健康的现实约束**：
- 公卫体检数据上传省级公卫系统（按国标 XML）
- 公卫系统是居民健康档案的"主索引"，但调阅能力弱
- 跨机构调阅：依赖省级平台建设（江浙沪较好，西部差）
- 实战方案：体检时由医生人工询问既往史，不依赖跨机构调阅

## 与架构师水平的差距自评

> **知识点掌握**：HIS/EMR 边界、核心数据模型（Patient/Visit/Order/Observation/Document）、三种 Visit 类型差异能讲清，但 CDA 文档模型（Header/Body/Section/Entry）、EMPI 概率匹配算法（Fellegi-Sunter 的具体权重计算）、IHE PIX/XDS 标准的工程实现细节不足。
>
> **架构思维**：多 HIS 厂家适配层、SaaS 化配置屏蔽差异的思路清晰，但跨机构 EMPI 聚合、区域卫生信息平台的数据汇聚与治理的工程实战经验不足，对"区域平台 vs 体检系统 vs 医院 EMR"三方数据流转的全景图缺乏实战案例。
>
> **工程细节**：体检结果回写 EMR 的三种方式（结构化/文档/引用）能讲清，但 CDA 模板设计（体检报告 CDA 模板的字段映射）、归档库与操作库的同步机制、电子签名 + 时间戳 + WORM 存储的工程落地需要补足。
>
> **业务理解**：乡镇场景（公卫体检、省级公卫系统对接）熟悉，但三甲医院 HIS/EMR 的实际部署（厂家差异、医院信息科对接流程、病案室归档流程）、医院信息化的销售/交付/运维模式需要补充。
>
> **合规意识**：数据安全法/个保法/人口健康信息管理办法知道框架，但具体到 EMPI 跨机构数据共享的患者授权机制、数据脱敏规则的工程实现、医疗数据出境的合规边界需要补足。
>
> **补足方向**：
> 1. 深读《电子病历应用管理规范（2017）》《电子病历基本规范（试行）》《病历书写基本规范》
> 2. 学习 EMPI 概率匹配算法（Fellegi-Sunter），实践一个简化版 EMPI（Java + ElasticSearch）
> 3. 学习 CDA 文档模型（Header/Body/Section/Entry），设计体检报告 CDA 模板
> 4. 学习 IHE PIX（患者交叉引用）、IHE XDS（文档共享）标准的国内实践
> 5. 复盘啄木鸟云健康与乡镇卫生院公卫系统的 EMPI 实践，识别身份证匹配的失败案例
> 6. 调研三甲医院 HIS/EMR 部署案例（卫宁、东软、创业慧康、嘉和美康），准备面试讲清楚"多机构 SaaS + 三甲对接"的双线能力
> 7. 学习病案归档的法律要求与归档库设计（WORM 存储、电子签名、时间戳、区块链存证）
> 8. 学习区域卫生信息平台的建设案例（上海、福建、浙江），理解跨机构数据聚合的工程实现
