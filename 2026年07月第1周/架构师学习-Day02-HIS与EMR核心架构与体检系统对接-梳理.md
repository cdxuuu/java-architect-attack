## 一、HIS / EMR / EHR：三个最常被混淆的概念

### 1.1 核心一句话
> **HIS 是医院业务的"操作系统"，EMR 是临床数据的"源头"，EHR 是跨机构的"健康档案"**
>
> 国内现状：HIS 厂家通常打包 EMR（卫宁、东软、创业慧康），EHR 由区域卫生信息平台承载。架构师必须能讲清三者的边界，否则在医疗信息化公司面试会"露怯"。

### 1.2 三者对比

| 系统 | 全称 | 范畴 | 重点 | 国内现状 |
| --- | --- | --- | --- | --- |
| HIS | Hospital Information System | 医院全业务 | 业务流转与管理（挂号/收费/药品/物资/医保） | 卫宁/东软/创业慧康为主 |
| EMR | Electronic Medical Record | 患者医疗记录 | 临床数据与文档（病程/医嘱/检查检验/手术） | 嘉和美康/医惠/东软集团 |
| EHR | Electronic Health Record | 跨机构健康记录 | 全生命周期健康档案 | 区域卫生信息平台承载 |

### 1.3 数据归属的"金科玉律"
- **HIS 持有**：就诊主数据（Visit）、医嘱/申请单（Order）、收费
- **EMR 持有**：病程记录、临床文档（Document）、临床观察（Observation）
- **EHR 持有**：跨机构聚合的健康档案（按 EMPI 全局 ID 关联）

## 二、HIS/EMR 核心数据模型：架构师必须烂熟的五大实体

### 2.1 五大核心实体

| 实体 | 含义 | 关键字段 | 关联 |
| --- | --- | --- | --- |
| Patient | 患者（人口学信息） | patient_id, id_card, name, gender, birth_date | 1:N Visit |
| Visit/Encounter | 就诊（一次就医过程） | visit_id, patient_id, visit_type, start_time, end_time, dept | 1:N Order, 1:N Document |
| Order | 医嘱/申请单 | order_id, visit_id, order_type, code, status, exec_time | 1:N Observation |
| Observation | 观察项（检验/生命体征） | observation_id, order_id, code, value, unit, time | N:1 Order |
| Document | 文档（病程/报告） | doc_id, visit_id, type, content, author, time | N:1 Visit |

### 2.2 三种 Visit 类型对比

| 维度 | 门诊 Visit | 住院 Visit | 体检 Visit |
| --- | --- | --- | --- |
| visit_type | OUTPATIENT | INPATIENT | EXAM |
| 时长 | 几小时 | 几天 | 几小时 |
| 病床 | 无 | 必有 | 无 |
| 医嘱 | 门诊处方 | 长期/临时医嘱 | 体检项目（必检/选检） |
| 病程记录 | 简单门诊病历 | 完整病程（首次/日常/出院） | 体检报告 |
| 结算 | 当场结算 | 出院结算 | 免费（公卫）/当场（商业） |
| 归档 | 门诊病历归档 | 出院病案归档 | 体检报告归档 |

### 2.3 体检流程到数据模型的映射

| 体检阶段 | 数据模型操作 |
| --- | --- |
| 预约 | 创建 Visit（status=BOOKED） |
| 报到 | Visit 状态转 CHECKED_IN，关联多个 Order（每检查项一个） |
| 检中 | Order 状态变化（STARTED → COMPLETED），Observation 落库 |
| 主检 | 生成 Document（type=EXAM_REPORT） |
| 报告 | Document 状态 PUBLISHED，触发归档 |

**关键设计点**：
- 体检 Visit 与门诊/住院 Visit 共享 Patient 主键（通过 EMPI），但 visit_type 隔离
- 体检 Order 与门诊医嘱共用 Order 表，但 order_type=EXAM_ITEM
- 体检报告是 Document 的一种，与门诊病历、出院小结并列
- 体检的"漏项"在 Order 层处理（status=PARTIAL_COMPLETED），不污染 Visit 状态机

## 三、EMPI 患者主索引：医疗信息化的"地基"

### 3.1 为什么必须有 EMPI

| 场景 | 没 EMPI 的痛点 | EMPI 解决 |
| --- | --- | --- |
| 同一患者多次就诊 | 每次 VISIT 不同 ID，历史数据散落 | 全局 ID 串联所有 Visit |
| 跨科室转诊 | 门诊号 vs 住院号 vs 体检号 | 全局 ID 跨科室关联 |
| 跨机构就诊 | A 院就诊号 vs B 院就诊号 | 全局 ID 跨机构关联 |
| 居民健康档案 | 各机构数据无法归集 | 全局 ID 聚合健康档案 |

### 3.2 匹配算法：确定性 vs 概率性

**① 确定性匹配**：唯一标识精确匹配（身份证、社保卡、护照）—— 简单可靠，国内覆盖率 95%+

**② 概率性匹配（Fellegi-Sunter 算法）**：多字段加权
- 字段权重：姓名（0.3）+ 生日（0.2）+ 性别（0.1）+ 地址（0.1）+ 电话（0.3）
- 三档阈值：
  - P > 0.85：自动匹配
  - 0.5 < P < 0.85：人工审核
  - P < 0.5：不匹配

### 3.3 国内场景的主键策略

| 场景 | 主键 | 处理 |
| --- | --- | --- |
| 普通居民 | 身份证 SHA-256 + 盐哈希 | 跨机构用哈希匹配，原值不存 |
| 新生儿 | 母亲身份证 + 出生序号 | 临时 ID，办身份证后关联 |
| 外籍人士 | 护照号 + 国籍 | 护照号做主键 |
| 港澳台 | 通行证号 + 证件类型 | 证件号做主键 |
| 无证件急诊 | 急诊号 + 临时特征 | 概率匹配，事后补关联 |

### 3.4 匹配冲突的人工审核

```
匹配引擎发现冲突
   │
   ├─ 同身份证号 + 不同姓名 + 不同生日 → 录入错误或冒用 → 人工审核
   ├─ 同身份证号 + 不同姓名 + 同生日 → 可能更名 → 人工核对户口本
   ├─ 同身份证号 + 同姓名 + 不同生日 → 肯定错误 → 标记异常
   └─ 概率匹配模糊（0.5 < P < 0.85）→ 人工审核队列
```

## 四、跨机构就诊数据归集：三种方式

### 4.1 三种数据交换方式

| 方式 | 时延 | 适用场景 | 国内现状 |
| --- | --- | --- | --- |
| 实时查询 | <1s | 体检前调阅历史（同机构内） | 单机构内常用 |
| 批量同步 | 分钟级 | 跨机构数据归集（CDC） | 区域平台常用 |
| 文档共享 | 小时级 | 区域卫生平台（IHE XDS） | 试点阶段 |

### 4.2 区域卫生信息平台的角色

| 角色 | 职责 |
| --- | --- |
| EMPI 中枢 | 跨机构患者身份匹配 |
| 数据聚合 | CDC 同步各机构就诊数据 |
| 健康档案 | 按居民全局 ID 聚合健康档案 |
| 数据共享 | 提供跨机构数据查询接口 |
| 监管上报 | 公卫数据上报的汇聚点 |

### 4.3 跨机构调阅的现实约束
- 江浙沪区域平台建设较好，跨机构调阅可用
- 西部地区区域平台弱，跨机构调阅不可用
- 啄木鸟云健康场景的现实：体检时由村医/医生人工询问既往史，不依赖跨机构调阅
- 长期方案：接入省级居民健康档案平台

## 五、体检结果回写 EMR：三种方式

### 5.1 三种回写方式对比

| 方式 | 实现 | 优势 | 适用场景 |
| --- | --- | --- | --- |
| 结构化回写 | CDA Entry/Observation | 可查询、可统计 | 检验结果、生命体征、诊断 |
| 文档回写 | CDA Document/PDF | 简单、完整 | 体检报告整体、影像报告 |
| 引用回写 | 影像 URL/PACS 链接 | 节省存储 | 影像（DICOM 太大） |

### 5.2 体检报告回写到 EMR 章节的映射

| 体检内容 | EMR 章节 | 回写方式 |
| --- | --- | --- |
| 血压、BMI、生命体征 | 生命体征章节 | 结构化（Observation） |
| 检验结果（血常规、生化） | 检验结果章节 | 结构化（Observation） |
| 影像报告 | 检查报告章节 | 文档（PDF）+ 引用（PACS URL） |
| 主检结论 | 诊断章节 | 结构化（Diagnosis） |
| 体检报告整体 | 检查报告章节 | 文档（CDA + PDF） |

### 5.3 关键设计
- 体检报告整体走文档回写（PDF + CDA 双格式）
- 关键指标（血压、血糖、血脂）走结构化回写，便于 EMR 临床决策支持
- 影像走引用回写，避免数据冗余
- 三种方式并存，覆盖不同场景

## 六、电子病历归档：法律红线

### 6.1 操作库 vs 归档库

| 维度 | 操作库 | 归档库 |
| --- | --- | --- |
| 用途 | 日常诊疗读写 | 病案归档（不可篡改） |
| 修改 | 允许 | 不允许（仅追加修改记录） |
| 保留 | 在院期间 + 1 个月 | 门诊 15 年、住院 30 年 |
| 存储 | 关系库 | WORM 存储 + 关系库索引 |

### 6.2 不可篡改的四道防线

| 措施 | 实现方式 | 法律依据 |
| --- | --- | --- |
| 电子签名 | CA 证书签名 | 《电子签名法》第 13 条 |
| 时间戳 | 国家授时中心时间戳 | 《电子签名法》第 15 条 |
| 哈希校验 | SHA-256 哈希，定期校验 | 司法证据规范 |
| 修改留痕 | 归档后修改必须留痕 | 《电子病历应用管理规范》第 14 条 |

### 6.3 归档库技术选型

| 方案 | 优势 | 适用场景 |
| --- | --- | --- |
| 关系库 + WORM 存储 | 成熟、成本低 | 大型医院主流 |
| 区块链存证 | 不可篡改性最强 | 高价值病历 |
| 电子签名 + 时间戳 | 法律效力明确 | 中小医院 |
| 对象存储 + 版本控制 | 简单、可扩展 | 体检报告等非核心病历 |

### 6.4 体检报告归档要求

| 维度 | 要求 | 法律依据 |
| --- | --- | --- |
| 保留期限 | 医院内 15 年（门诊标准），公卫体检永久 | 《医疗机构病历管理规定》 |
| 归档格式 | CDA + PDF + 原始 JSON + DICOM（影像） | 行业最佳实践 |
| 修改机制 | 走"补充报告"流程，原报告不覆盖 | 《电子病历应用管理规范》 |
| 修改审计 | 修改人、时间、原因、原值、新值留痕 | 同上 |

## 七、公卫健康档案 vs 医院 EMR

### 7.1 两者对比

| 维度 | 公卫健康档案 | 医院 EMR |
| --- | --- | --- |
| 性质 | 居民健康档案（基本信息 + 体检 + 随访） | 医疗病历（就诊全流程记录） |
| 归属 | 区域卫生信息平台 | 医院 |
| 标准 | 《国家基本公共卫生服务规范》 | 《电子病历应用管理规范》 |
| 修改 | 可修改（随访会更新） | 归档后不可改（仅追加） |
| 保留 | 永久 | 门诊 15 年、住院 30 年 |
| 法律性质 | 健康档案，非电子病历 | 电子病历 |

### 7.2 跨机构居民健康档案聚合

```
区域卫生信息平台
   ├─ A 院 EMR 数据（CDC 同步）
   ├─ B 院 EMR 数据（CDC 同步）
   ├─ 公卫体检数据（公卫系统同步）
   ├─ 随访数据（社区中心同步）
   │
   ▼
按 EMPI 全局 ID 聚合 → 居民健康档案
```

### 7.3 跨机构数据冲突解决

| 冲突类型 | 解决策略 |
| --- | --- |
| 同一时间不同机构诊断不同 | 保留所有诊断，标注来源机构 |
| 同一指标不同值（如血压） | 保留所有记录，按时间排序，不合并 |
| 基本信息冲突（如地址） | 以最近一次为准，旧值归档 |
| 诊断冲突 | 保留所有诊断，由家庭医生人工调和 |

## 八、多 HIS 厂家适配层：医疗 SaaS 架构师的核心能力

### 8.1 国内主流 HIS 厂家对比

| 厂家 | 主营市场 | 特点 | 体检产品 |

```
        │  ├─ 接口标准化    │
        │  ├─ 字段映射      │
        │  └─ 协议转换      │
        └────────┬─────────┘
                 │
        ┌────────┼────────┬────────┬────────┐
        ▼        ▼        ▼        ▼        ▼
   东软适配   卫宁适配   慧康适配  用友适配  中联适配
```

### 8.4 适配层部署位置

| 部署位置 | 适用场景 |

## 九、三种场景的差异化定位

### 9.1 乡镇 vs 三甲 vs 体检中心

| 维度 | 乡镇卫生院 | 三甲医院 | 独立体检中心 |
| --- | --- | --- | --- |
| HIS | 简化版/无 | 完整 HIS/EMR | 无 HIS |
| LIS | 无/简化 | 完整 LIS | 简化 LIS |
| PACS | 无 | 完整 PACS | 无 |
| 体检系统定位 | 业务核心 | 子系统 | 独立核心 |
| 对接策略 | 直连公卫系统 | 通过集成引擎对接 | 独立运行 |
| 部署模式 | SaaS 共享 | 私有化为主 | SaaS 共享 |
| 网络条件 | 4G/卫星 | 院内万兆 | 院内局域网 |
| 体检量 | 批量（200-500/天） | 分散（个人预约） | 分散（个人预约） |
| 收费 | 免费（公卫） | 付费（医保/自费） | 付费（自费为主） |
| 上报要求 | 必报公卫 | 不强制 | 不强制（除健康证） |

### 9.2 啄木鸟云健康的定位
- 主战场：乡镇卫生院 + 社区中心（公卫体检）
- 体检系统独立部署（SaaS 共享）
- 直连省级公卫系统（无 HIS 对接）
- 设备网关本地部署（断网缓存）
- 跳槽目标：拓展三甲体检中心场景，需要补齐 HIS/EMR 对接能力

## 十、医疗数据合规框架

### 10.1 法律框架

| 法律 | 重点 | 对 EMPI 的影响 |
| --- | --- | --- |
| 《数据安全法》 | 数据分级分类 | 跨机构共享需数据分级评估 |
| 《个人信息保护法》 | 敏感个人信息单独同意 | 跨机构共享需患者单独授权 |
| 《人口健康信息管理办法》 | 不得违规出境 | 跨境共享禁止 |
| 《健康医疗数据安全指南》 | 分类分级、脱敏规则 | 脱敏后共享 |
| 等保 2.0 | 三级等保是医疗系统标配 | EMPI 系统必须过等保三级 |

### 10.2 跨机构共享的合规边界

| 共享场景 | 合规要求 |
| --- | --- |
| 同机构内科室间共享 | 挂号即默认授权 |
| 跨机构数据共享 | 必须患者单独授权 |
| 公卫上报 | 法定义务，无需患者授权 |
| 医保结算 | 医保协议覆盖 |
| 科研使用 | 必须去标识化 + 伦理审查 |
| 商业保险理赔 | 患者授权 + 最小必要原则 |

### 10.3 数据脱敏规则

| 字段 | 脱敏规则 |
| --- | --- |
| 身份证号 | SHA-256 + 盐哈希 |
| 姓名 | 显示脱敏（张*三） |
| 电话 | 中间 4 位掩码 |
| 地址 | 模糊到街道级 |

## 十二、Day02 关键考点速记

| 考点 | 一句话记忆 |
| --- | --- |
| HIS vs EMR | HIS 管业务，EMR 管临床，EHR 跨机构 |
| 五大数据模型 | Patient → Visit → Order → Observation → Document |
| 三种 Visit | OUTPATIENT / INPATIENT / EXAM |
| EMPI 算法 | 确定性（身份证）+ 概率性（Fellegi-Sunter） |
| 跨机构数据交换 | 实时查询 / 批量同步 / 文档共享 |
| 体检结果回写 | 结构化 + 文档 + 引用 三种方式 |
| CDA 模型 | Header + Body（Section + Entry） |
| 病历归档 | 操作库 vs 归档库，门诊 15 年、住院 30 年 |
| 不可篡改四道防线 | 电子签名 + 时间戳 + 哈希校验 + 修改留痕 |
| 公卫健康档案 | 不是电子病历，是动态档案，永久保留 |
| 多 HIS 厂家适配 | 三甲放集成引擎侧，乡镇放体检系统侧 |
| 乡镇 vs 三甲 | 乡镇直连公卫，三甲集成引擎对接 HIS/LIS/PACS |
| 数据合规 | 数据安全法 + 个保法 + 人口健康信息管理办法 + 等保三级 |
| 跨机构共享 | 必须患者单独授权，公卫上报是法定义务 |
