## 背景

你所在的啄木鸟云健康主要承接国家基本公共卫生服务项目（以下简称"公卫体检"），面向 65 岁以上老年人、高血压/糖尿病慢病患者、孕产妇、0-6 岁儿童等重点人群，提供**免费体检**服务。与商业体检（美年大健康、爱康国宾）不同，公卫体检有几个鲜明的工程约束：

1. **批量上门体检**：体检车开进乡镇卫生院/社区中心，单日单点峰值 200~500 人，网络条件差（4G/卫星），设备离线采数后回院再回传。
2. **强监管上报**：体检结果必须按《国家基本公共卫生服务规范》第三版格式打包，上传到省级公卫系统（慢病随访、居民健康档案），数据格式是国标 XML/JSON，不是医疗行业的 HL7/FHIR。
3. **多机构 SaaS**：一套系统服务几十家乡镇卫生院/社区中心，需要数据隔离、配置可定制、报告模板可独立维护。
4. **设备异构**：血压计、心电图、B 超、生化分析仪、尿分析仪、身高体重仪、骨密度仪——品牌五花八门（欧姆龙、迈瑞、理邦、迪瑞），接口协议有串口、网口、蓝牙、私有 SDK，没有统一标准。
5. **结果回传给居民**：检后报告要通过微信公众号/小程序推送给居民，体检异常项要触发随访工单。

本周 Day01 聚焦**整体架构与核心模块**，把智慧体检的"骨架"画清楚，为后续 HIS/EMR（Day02）、HL7/FHIR（Day03）、DICOM/PACS（Day04）、医保对接（Day05）、全链路串联（Day06）、架构深挖（Day07）打基础。

各模块职责边界：

| 模块 | 职责 | 不做什么 |
|---|---|---|
| **预约登记 Reserve** | 居民预约（小程序/村医代约）、人群资格校验（年龄/慢病标签）、套餐匹配、预约号生成、改约/取消 | 不做检中操作，不做报告生成 |
| **报到分流 CheckIn** | 现场签到、身份证/社保卡识别、条码打印（多个标本条码）、检查项目分发、排队叫号 | 不持有体检结果，只生成"标本-项目"关联 |
| **检中执行 ExamExec** | 各科室检查执行、结果录入、设备结果接收、异常项标记、医生电子签名 | 不调设备（交给设备网关），不做报告汇总 |
| **设备网关 DeviceGW** | 设备协议适配（串口/网口/蓝牙/SDK）、原始报文解析、结果标准化、断网缓存重传 | 不持有业务逻辑，只做"协议-数据"翻译 |
| **LIS 适配 LISAdapt** | 检验仪器结果接收（ASTM/HL7 ORU^R01）、条码双向通讯、危急值推送 | 不做检验项目临床解释 |
| **PACS 适配 PACSAdapt** | 影像接收（DICOM Store SCU/SCP）、Worklist 推送（Modality Work List）、影像调阅（WADO/QIDO-RS） | 不存影像（交给 PACS 存储） |
| **主检报告 Report** | 主检医生审核、异常项汇总、健康评估、报告生成（PDF/HTML）、居民端推送 | 不修改原始检查数据，只生成"主检结论" |
| **公卫上报 PublicHealthReporter** | 国标格式打包（XML/JSON）、签名（XMLDSig）、定时批量上报、上报回执管理、补报重试 | 不修改业务数据，只读 + 打包 |
| **随访工单 FollowUp** | 异常项触发随访规则、随访工单生成、随访结果录入、随访结果回写公卫档案 | 不替代医生诊断，只做规则触发 + 工单流转 |
| **套餐排班 Schedule** | 体检套餐管理（老年/慢病/孕产/儿童）、人群-套餐映射、体检车排班、机构排班 | 不持有居民档案 |
| **居民端 WechatApp** | 预约、报告查询、随访填写、健康教育推送 | 不直接调业务层，全部走网关 |
| **机构管理后台 TenantAdmin** | 机构信息、报告模板、设备绑定、人员权限、套餐定制 | 不查看其他机构数据 |

#### 2. 同步链路与异步链路

**同步链路（前台操作，居民/医生在等）**：

```
居民小程序 → 网关 → 预约登记（资格校验、号源锁定）
              → 返回预约号

现场报到 → 网关 → 报到分流（身份证识别、人群校验）
        → 套餐匹配 → 生成多个标本条码（条码打印机同步出票）
        → 检查项目分发到各科室工作站
        → 居民持条码去各科室检查

各科室检查 → 网关 → 检中执行（医生录入或设备结果弹窗确认）
         → 异常项即时标记 → 危急值弹窗告警 + 短信通知主检医生
```

**异步链路（后台处理，居民不在等）**：

```
设备出结果 → 设备网关（断网缓存） → 网络恢复后批量上传
         → MQ → LIS 适配器 → 标准化结果 → 检中执行落库
         → MQ → 危急值评估 → 触发主检医生告警

所有项目检查完 → 检中执行发"体检完成"事件 → MQ
              → 主检报告生成（自动汇总 + 主检医生审核）
              → 报告 PDF 生成
              → MQ → 居民端推送（小程序订阅消息）
              → MQ → 公卫上报任务入队列

定时任务（凌晨）：
  公卫上报批处理 → 国标 XML 打包 → 省级公卫系统 WebService
              → 回执落库 → 失败补报（指数退避）
  → 异常项 → 随访工单生成 → 推送村医/社区医生
```

**为什么必须异步？**

1. **设备数据回流天然异步**：体检设备出结果时间不可控（生化仪 5~20 分钟，B 超医生手动出报告），同步阻塞会让前台工作站卡死。设备网关必须先缓存（断网场景必须），再异步推送。
2. **报告生成依赖多源数据汇聚**：主检报告需要等所有检查项目（血压、检验、影像、心电）全部到齐，必须靠"事件计数 + 状态机"判断完成，不能用同步 RPC 串起来。
3. **公卫上报天然批量**：省级公卫系统对接接口性能差（通常单次 5~10 条，QPS 限制 1~2），必须本地攒批后凌晨批量上报，单条同步上报必死。
4. **居民端推送不可阻塞业务**：微信公众号订阅消息接口可能慢/失败，MQ 隔离避免拖垮报告生成。
5. **危急值告警必须可靠**：危急值（如血钾 6.5mmol/L）的告警必须保证送达，用 MQ + 多通道（弹窗 + 短信 + 电话）兜底，不能"同步推一次失败就丢"。
6. **同步路径仅保留必要的**：预约、报到、检中录入是同步的（用户在等），其余尽量异步。

#### 3. 业务订单 vs 体检流程实例

**关系**：体检任务（业务订单）与体检流程实例（ProcessInstance）是一对一关系，但一次"批量上门体检任务"对应多个体检流程实例。

```
批量上门体检任务 TaskBatch20260706-XX镇-老年体检
   ├─ 体检流程实例1 老人张三（预约→报到→检中→报告→上报）
   ├─ 体检流程实例2 老人李四
   ├─ 体检流程实例3 老人王五
   └─ ... 300 人
```

**为什么要拆分？**

1. **职责分离**：体检任务关心"今天去哪个乡镇、什么人群、共多少人、车队调度"，体检流程实例关心"张三这次体检走到了哪一步、各项结果如何"。混在一起会导致任务表频繁被流程实例更新污染。
2. **批量场景的统计聚合**：批量任务需要按乡镇、人群、时间维度统计完成率、异常率，这些聚合查询与单人体检流程查询是不同的访问模式。
3. **生命周期不同**：批量任务当周完成（一周内），体检流程实例长期保留（公卫档案要求永久），且要回传给居民健康档案。
4. **多业务线复用**：体检流程实例是核心能力，既能服务公卫体检（免费、批量），也能服务商业体检（付费、单人），还能服务入职体检、从业人员健康证体检（监管要求不同）。

**批量上门 vs 日常门诊的订单模型差异**：

| 维度 | 批量上门体检 | 日常门诊体检 |
|---|---|---|
| 触发方式 | 村医代约/集体组织 | 居民自主预约/医生开单 |
| 预约粒度 | 按批次（300人/天） | 按个人（按号源） |
| 报到方式 | 集中报到（身份证批量识别） | 一人一报到 |
| 套餐选择 | 人群固定套餐（老年/慢病） | 居民可选/医生开单 |
| 收费 | 免费（公卫项目） | 付费（医保/自费） |
| 上报要求 | 必须上报公卫系统 | 不强制上报（除健康证） |
| 网络条件 | 4G/卫星，可能断网 | 院内局域网稳定 |
| 模型差异 | TaskBatch + ProcessInstance 两层 | ProcessInstance 一层即可 |

**模型设计建议**：
- 公卫体检：`task_batch`（批次表）+ `process_instance`（流程实例表），process_instance.task_batch_id 外键关联
- 商业体检：`process_instance` 单表即可，task_batch_id 为空
- 通用模型：两表都建，task_batch 可空，公卫与商业体检统一处理

#### 4. 多机构 SaaS 租户隔离

**隔离强度分级**：

| 数据类型 | 隔离方式 | 理由 |
|---|---|---|
| 业务数据（体检记录、报告、随访） | **强隔离**：分库分表，按 org_id 分片 | 监管要求各机构数据独立，且数据量差异大（大机构日均 500 人，小机构日均 30 人），共享表会有热点与查询污染 |
| 居民健康档案 | **强隔离**：按 org_id 分片，但身份证全局唯一索引跨片查询 | 同一居民可能在多个机构体检（流动就医），需跨片查档案；但写入只允许在主机构 |
| 设备配置、套餐模板 | **弱隔离**：共享表 + org_id 字段 | 配置数据量小，共享表便于平台运营统一维护"标准套餐" |
| 用户权限、机构人员 | **强隔离**：分库 | 安全合规要求，人员数据不出机构 |
| 报告模板 | **中隔离**：共享表 + 模板继承（平台标准模板 + 机构自定义覆盖） | 减少重复维护，但允许机构定制 |
| 字典数据（ICD10、检查项目、检验项目） | **共享**：全平台一份字典 | 字典是行业标准，不需要隔离 |
| 系统配置（限流、网关路由） | **共享**：配置中心统一管理 | 平台运维统一管控 |

**分片键选择**：

- **首选 `org_id`**：99% 的查询都是机构维度（"今天本机构体检了多少人"），按 org_id 分片让同机构数据落在同一分片，避免跨片 JOIN。
- **次选 `id_card_hash % N`**：仅用于"跨机构查居民档案"的特定场景，作为二级索引。
- **绝对不用 `cuid`/`resident_id`**：居民 ID 在不同机构不同，会导致同一居民多次体检记录散落在不同分片。

**分片策略**：

```java
// ShardingSphere 配置示例
shardingRule:
  tables:
    process_instance:
      actualDataNodes: ds_${0..3}.process_instance_${0..15}
      databaseStrategy:
        standard:
          shardingColumn: org_id
          shardingAlgorithm: org_mod  // org_id % 4 路由到 ds_0~3
      tableStrategy:
        standard:
          shardingColumn: org_id
          shardingAlgorithm: org_mod_16  // 同库内再分 16 表
```

**关键约束**：
- 跨片查询限制：只有"居民档案跨片查询"这一种场景允许跨片，且必须走 ES 索引（CDC 同步到 ES），禁止在 DB 上做跨片 JOIN
- 全局唯一 ID：用雪花算法，dataCenterId = 机房号，workerId = 服务实例号，保证跨片不冲突
- 备份恢复粒度：按机构粒度备份，某机构数据出问题不影响其他机构

## 题目二（状态机设计题）：一次公卫体检的全生命周期与状态机设计

一次公卫体检从预约到报告上传公卫系统，会经历多个状态。请回答：

1. 画出公卫体检流程实例的完整状态机，至少包含：预约成功（BOOKED）、已报到（CHECKED_IN）、检中（IN_EXAM）、待主检（PENDING_REVIEW）、主检中（REVIEWING）、已出报告（REPORTED）、已上报公卫（REPORTED_TO_PH）、已关闭（CLOSED）、已取消（CANCELLED）。标注每个状态的进入条件、退出条件、允许的状态流转路径。
2. 哪些状态流转是"危险"的（一旦流转错误就可能导致漏检、错检、错误上报）？例如：IN_EXAM → REPORTED 与 IN_EXAM → PENDING_REVIEW 之间的并发竞争如何防止？报告回退（REPORTED → REVIEWING）如何处理？状态机如何与幂等结合？
3. 居民在检中"漏项"（如血压量了但心电图没做）是公卫体检最常见的场景，状态机如何处理"部分完成"？主检医生发现漏项后如何回退到检中？已出报告后发现漏项如何补检补报？
4. 公卫上报失败是高频问题（接口超时、签名错、数据格式不符），状态机如何处理"上报中-失败-补报"？已上报成功的报告发现错误，如何撤销重报？

### 作答区

#### 1. 公卫体检完整状态机

```
                  ┌─────────┐  预约成功（小程序/村医代约）
                  │ BOOKED  │  待检
                  └────┬────┘
                       │ 现场报到（身份证识别）
                       ▼
                  ┌─────────────┐
                  │ CHECKED_IN  │ 已报到
                  └────┬────────┘
                       │ 第一个科室开始检查
                       ▼
                  ┌─────────────┐
        ┌────────│  IN_EXAM    │────────┐
        │        │  检中        │        │
        │        └────┬────────┘        │
        │             │                 │
   居民离场未做完  所有项目完成       主动取消
   （漏项挂起）     （计数达标）        / 改约
        │             │                 │
        ▼             ▼                 ▼
   ┌─────────┐  ┌──────────────┐   ┌──────────┐
   │ PARTIAL │  │PENDING_REVIEW│   │CANCELLED │
   │ 漏项挂起 │  │ 待主检       │   │ 已取消   │
   └────┬────┘  └──────┬───────┘   └──────────┘
        │              │ 主检医生开始审核
        │              ▼
        │         ┌──────────┐
        │         │REVIEWING │ 主检中
        │         └────┬─────┘
        │              │
        │     ┌────────┼────────┐
        │     │        │        │
        │  发现漏项  审核通过   审核驳回（数据错）
        │     │        │        │
        │     ▼        ▼        ▼
        └─────┘   ┌──────────┐  回到 IN_EXAM
                  │ REPORTED │  已出报告
                  └────┬─────┘
                       │ 定时任务批量上报公卫
                       ▼
                  ┌──────────────────┐
                  │REPORTED_TO_PH    │ 已上报公卫
                  │ 上报中/已上报    │
                  └────┬─────────────┘
                       │ 上报回执成功
                       ▼
                  ┌──────────┐
                  │ CLOSED   │ 已关闭（终态）
                  └──────────┘
```

**状态流转条件与允许路径**：

| 当前状态 | 目标状态 | 进入条件 |
|---|---|---|
| BOOKED | CHECKED_IN | 现场报到，身份证验证通过 |
| BOOKED | CANCELLED | 居民主动取消 / 改约 |
| CHECKED_IN | IN_EXAM | 第一个科室开始检查 |
| CHECKED_IN | CANCELLED | 报到后未开始检查即离开 |
| IN_EXAM | PENDING_REVIEW | 所有必检项目完成（计数达标） |
| IN_EXAM | PARTIAL | 居民离场且未全部完成（漏项挂起） |
| IN_EXAM | CANCELLED | 主动取消 |
| PARTIAL | IN_EXAM | 居民回来补检 |
| PARTIAL | CANCELLED | 漏项超期未补（如 7 天未回） |
| PENDING_REVIEW | REVIEWING | 主检医生开始审核 |
| REVIEWING | REPORTED | 审核通过，生成报告 |
| REVIEWING | IN_EXAM | 审核驳回（发现数据错误，回检） |
| REVIEWING | PARTIAL | 审核发现漏项，挂起等补检 |
| REPORTED | REVIEWING | 报告回退（已出报告发现问题，未上报前） |
| REPORTED | REPORTED_TO_PH | 定时任务触发上报 |
| REPORTED_TO_PH | REPORTED | 上报失败需修改（数据格式错误） |
| REPORTED_TO_PH | CLOSED | 上报回执成功 |
| CLOSED | REVIEWING | 已上报后发现错误，撤销重报（特殊审批） |

**禁止的逆向流转**：
- CLOSED → REPORTED_TO_PH：必须走"撤销重报"特殊审批流程
- CANCELLED → 任何状态：取消是终态，需重新预约
- REPORTED → IN_EXAM：必须经过 REVIEWING，不能直接回检中

#### 2. 危险的状态流转与并发竞争

**最危险的三处**：

**① IN_EXAM → PENDING_REVIEW vs IN_EXAM → PARTIAL 并发竞争**

场景：居民做完 8 项中的 7 项就离开了，最后一个心电图结果在 1 分钟后才从设备回流。此时状态机：
- 线程A（设备结果回流）：所有项目计数达标 → 置 PENDING_REVIEW
- 线程B（漏项检测任务）：发现居民离场且未完成 → 置 PARTIAL

如果加锁/版本号不到位，可能出现"状态被 PARTIAL 覆盖，主检医生看不到此人待审"或"状态被 PENDING_REVIEW 覆盖，漏项检测误判已完成"。

防护手段：
- **乐观锁 + 状态前置校验**：`UPDATE process_instance SET status=PENDING_REVIEW, version=version+1 WHERE id=? AND status=IN_EXAM AND version=?`，影响行数=0 则放弃
- **状态机白名单**：每个状态有"允许的下一状态集合"，PARTIAL 不能从 PENDING_REVIEW 流转来（必须先回退到 IN_EXAM）
- **业务前置校验**：置 PENDING_REVIEW 前必须校验"必检项目计数达标"，置 PARTIAL 前必须校验"居民确实离场"（一段时间无操作 + 系统检测到签到状态变更）
- **设备结果回流幂等**：同一标本条码 + 同一检查项目，重复回流只生效一次（唯一索引）

**② REPORTED → REVIEWING 报告回退**

场景：报告已生成但未上报公卫前，主检医生发现异常项遗漏或结论错误，需要回退重新主检。
- 必须校验"未上报公卫"才能回退，已上报的报告不允许直接回退
- 回退时必须保留"原报告快照"用于审计，不能直接覆盖
- 回退后状态置 REVIEWING，主检医生修改后重新生成报告，新报告版本号 +1

```sql
-- 报告版本快照表
report_snapshot {
    process_instance_id, version, report_content, 
    reviewer_id, review_time, reason  -- 回退原因
}
```

**③ CLOSED → REVIEWING 已上报后撤销重报**

场景：报告已上报公卫系统后发现错误（如身份证号错、主检结论错），需要撤销原上报、重新主检、重新上报。
- 必须走特殊审批流程（人工审核）
- 必须先调公卫系统的"撤销上报"接口（如果支持），不支持则需要走线下流程
- 状态机记录"撤销原因 + 审批人 + 撤销时间"
- 重新上报时上报新版本，公卫系统侧用"上报流水号 + 版本号"区分

**状态机与幂等结合**：
- 状态流转的"动作"必须幂等：同一项目结果重复回流 → 落到同一流程实例 → 状态机判断当前已是目标状态则直接返回成功
- 状态机表设计：`from_status, to_status, action, biz_check` 四元组，所有流转走配置化表，禁止代码里裸写 setStatus
- 上报重试幂等：同一 process_instance 上报公卫，使用"上报流水号 + 流程实例ID + 版本号"做幂等键，公卫系统侧幂等

#### 3. 漏项处理：部分完成状态设计

**漏项场景的复杂性**：
- 公卫体检项目多（10+ 项），漏项是常态而非异常
- 漏项可能是居民主动放弃（如不愿做妇科检查）、临时无法做（设备故障）、漏做（流程混乱）
- 漏项必须能补检（公卫规范要求"必检项"不能漏，漏了要追补）
- 已出报告后补检，需要补出补充报告

**状态机设计**：

```
IN_EXAM
   │
   ├─ 居民离场 + 必检项未完成 → PARTIAL（漏项挂起）
   │                              │
   │                              │ 居民回来补检
   │                              ▼
   │                           IN_EXAM（继续检）
   │
   └─ 所有必检项完成 → PENDING_REVIEW

主检审核阶段发现漏项：
REVIEWING
   │
   ├─ 必检项漏（数据缺失）→ PARTIAL（挂起等补检）
   │                          │
   │                          │ 补检完成
   │                          ▼
   │                       IN_EXAM → PENDING_REVIEW → REVIEWING
   │
   └─ 可选项漏（居民放弃）→ 标记"放弃原因"，继续审核 → REPORTED

已出报告后补检：
REPORTED（已出报告，未上报）
   │
   │ 居民回来补做漏项
   ▼
REVIEWING（主检医生重新审核）
   │
   │ 生成补充报告（原报告 + 补充内容）
   ▼
REPORTED（新版本）

已上报公卫后补检：
REPORTED_TO_PH
   │
   │ 补检完成 → 走"撤销重报"流程
   ▼
CLOSED → REVIEWING（特殊审批）→ REPORTED → REPORTED_TO_PH（新版本上报）
```

**关键设计点**：

1. **PARTIAL 不是终态**：漏项挂起后必须能恢复，超期才转 CANCELLED
2. **必检项 vs 可选项区分**：必检项漏必须补，可选项漏可标记"放弃"直接通过
3. **补检走原流程**：补检的检查项目走原 IN_EXAM 流程，结果落到原流程实例，不新建实例
4. **补充报告机制**：已出报告后补检，生成"补充报告"（V2），原报告（V1）保留，公卫上报时上报最新版本
5. **漏项检测定时任务**：每 10 分钟扫描 IN_EXAM 状态超过 4 小时未操作的流程实例，自动转 PARTIAL

#### 4. 公卫上报失败与补报

**上报状态子状态机**：

```
REPORTED（已出报告）
   │
   │ 凌晨定时任务触发上报
   ▼
┌─────────────────────┐
│ REPORTED_TO_PH      │
│  ├─ UPLOADING（上报中）│
│  ├─ UPLOAD_FAILED    │ ← 上报失败（网络/接口超时）
│  └─ UPLOADED         │ ← 上报成功，等回执
└─────────────────────┘
   │
   │ 回执查询任务（每小时）
   ▼
┌─────────────────────┐
│  回执 SUCCESS        │ → CLOSED
│  回执 FAIL（数据错） │ → 回到 REPORTED，标记错误原因，人工介入
│  回执 PENDING        │ → 继续等
└─────────────────────┘
```

**上报失败处理**：

| 失败类型 | 处理方式 |
|---|---|
| 网络超时 / 接口 5xx | 自动重试，指数退避（5min、15min、1h、4h、24h），最多 5 次 |
| 签名错 / 证书过期 | 立即告警，不重试，人工介入更新证书 |
| 数据格式错（字段缺失/格式不符） | 标记错误原因，回到 REPORTED，人工修改后重新触发上报 |
| 重复上报（同流程实例已上报） | 幂等返回成功，不报错 |
| 公卫系统返回"该居民已上报" | 标记"重复上报"，记录公卫侧原始流程实例号，停止重试 |

**幂等键设计**：
- 上报幂等键：`process_instance_id + report_version + org_id`
- 公卫系统侧也用此键做幂等，重复上报返回原上报流水号
- 本地"上报任务表"建唯一索引：`(process_instance_id, report_version)`

**已上报成功后发现错误，撤销重报**：

```
REPORTED_TO_PH (UPLOADED)
   │
   │ 发现错误（如身份证号错、主检结论错）
   ▼
特殊审批流程（人工）
   │
   │ 调公卫系统"撤销上报"接口
   ▼
   ├─ 撤销成功 → REVIEWING（重新主检）
   │              │
   │              │ 修改后重新上报
   │              ▼
   │           REPORTED_TO_PH（新版本）
   │
   └─ 撤销失败（公卫系统不允许撤销，如已纳入统计）
                  │
                  │ 走"补充报告"流程
                  ▼
              REPORTED_TO_PH（补充报告 V2，标注"补充"）
```

**关键设计**：
- 撤销上报必须走人工审批，避免误撤销
- 撤销日志全留痕（谁撤的、为什么撤、什么时间）
- 撤销失败时的兜底是"补充报告"，不能强行覆盖公卫侧已确认的数据
- 公卫系统对接接口的"撤销"能力各省市不同，需要适配层屏蔽差异

## 题目三（选型权衡题，进阶）：体检系统集成 LIS/PACS/RIS 的模式选型

智慧体检系统对接 LIS（检验）、PACS（影像）、RIS（放射报告）、HIS（医院信息）有多种集成模式。请回答：

1. 体检系统对接 LIS/PACS/RIS 有哪些主流集成模式？至少对比四种：点对点直连、集成引擎（如 Mirth Connect）、HL7 v2/v3、FHIR。各自的工作原理、优缺点、适用场景是什么？
2. 公卫体检场景下，你倾向选哪种集成模式？为什么？特别是设备数据采集（血压计、心电、B 超、生化）与 LIS/PACS 的差异是什么？
3. 乡镇卫生院/社区中心的小机构（无 LIS/PACS） vs 三甲医院体检中心（已有完整 LIS/PACS）两种客户，集成模式应该如何差异化设计？SaaS 化部署对集成模式有什么约束？
4. 集成模式的演进路径是怎样的？从点对点起步，如何演进到集成平台 + FHIR 标准化？过渡期的混合架构如何设计？

### 作答区

#### 1. 四种主流集成模式对比

**① 点对点直连**

工作原理：体检系统直接调 LIS/PACS 的私有 API 或数据库视图，结果通过定时轮询/接口推送获取。

```
体检系统 ──HTTP/DB View──> LIS
体检系统 ──HTTP/DICOM────> PACS
体检系统 ──HTTP──────────> RIS
```

| 优点 | 缺点 |
|---|---|
| 实现简单，开发快 | M×N 矩阵：每接一个系统就要写一份适配代码 |
| 性能高（直连无中间层） | 强耦合：LIS 接口变更，体检系统必须改 |
| 调试直接，链路短 | 没有标准，每个 LIS/PACS 厂家接口都不同 |
| 无需额外中间件 | 难扩展，新接系统成本线性增长 |

适用场景：单体机构、LIS/PACS 厂家固定不变、系统数量少（< 3 个）。

**② 集成引擎（Mirth Connect / Rhapsody / Ensemble）**

工作原理：部署独立集成引擎，所有系统对接引擎，引擎负责路由、转换、协议适配。Mirth Connect 是开源医疗集成引擎，用 Channel（通道）+ Transformer（转换器）+ Connector（连接器）模型。

```
体检系统 ──┐                    ┌──> LIS
           │                    │
           ▼                    ▼
       ┌────────────────────────┐
       │  集成引擎 Mirth Connect │
       │  - Channel：消息通道    │
       │  - Transformer：转换    │
       │  - Router：路由         │
       └────────────────────────┘
           ▲                    ▲
           │                    │
PACS ──────┘                    └──> RIS
```

| 优点 | 缺点 |
|---|---|
| M+N 矩阵：每系统只接一次引擎 | 部署复杂，需运维集成引擎 |
| 协议转换、消息路由集中管理 | 引擎本身是单点/瓶颈，需高可用部署 |
| 可视化配置通道，少写代码 | 学习成本高（Mirth JavaScript 转换器） |
| 支持多种协议（HL7、DICOM、HTTP、DB、FTP） | 商业产品贵（Rhapsody 数十万） |
| 内置消息队列、重试、审计 | 调试链路长，问题定位难 |

适用场景：多系统集成（5+ 个）、有 IT 运维能力的中大型机构。

**③ HL7 v2.x（管道分隔文本协议）**

工作原理：医疗行业标准协议（HL7 v2.x 是事实标准），消息为管道分隔文本（`MSH|...|PID|...|PV1|...|OBX|...`），通过 MLLP（TCP）传输。

```
体检系统 ──HL7 v2 ORU^R01（检验结果）──> LIS
         ──HL7 v2 ORM^O01（订单）────> LIS
         ──HL7 v2 ADT^A04（就诊）────> HIS
```

| 优点 | 缺点 |
|---|---|
| 医疗行业标准，LIS/HIS 都支持 | 标准模糊，本地化变体多（"中国的 HL7"） |
| 消息结构化，字段语义明确 | 管道分隔格式可读性差 |
| 适用于就诊、订单、结果多种场景 | HL7 v2 是"事实标准"非"严格标准"，兼容性差 |
| 国内三甲医院 LIS 都能接 | 不适合新系统，新系统倾向 FHIR |

适用场景：与已有 LIS/HIS（厂家已固化 HL7 v2）对接的三甲医院体检中心。

**④ FHIR（HL7 v4，RESTful + JSON/XML）**

工作原理：HL7 最新标准，RESTful API + 资源模型（Patient、Observation、DiagnosticReport、ImagingStudy），JSON/XML 表示。

```
体检系统 ──FHIR RESTful──> FHIR Server
         POST /Observation        （检查结果）
         POST /DiagnosticReport   （诊断报告）
         GET  /Patient/{id}       （居民信息）
```

| 优点 | 缺点 |
|---|---|
| 现代 RESTful API，开发者友好 | 国内 LIS/PACS 普遍不支持，需适配层 |
| JSON/XML 双格式，可读性好 | 资源模型复杂，学习曲线陡 |
| 资源粒度细，可组合查询 | 实现成本高（资源关联、引用解析） |
| 国际标准，未来方向 | 国内 FHIR 实践案例少 |

适用场景：新建系统、跨机构数据交换、互联网医疗、国际对接。

**四种模式对比矩阵**：

| 维度 | 点对点 | 集成引擎 | HL7 v2 | FHIR |
|---|---|---|---|---|
| 接入成本 | 低（单点） | 中（建引擎） | 中（学标准） | 高（资源模型） |
| 扩展成本 | 高（M×N） | 低（M+N） | 低（标准接口） | 低（标准接口） |
| 标准化 | 无 | 引擎层做 | 强（医疗事实标准） | 强（国际标准） |
| 国内支持 | 都支持 | 主流三甲有 | 主流 LIS/HIS 支持 | 少见 |
| 调试难度 | 易 | 中 | 中 | 中 |
| 适合场景 | 单机构 | 多系统集成 | 三甲对接 | 新建/跨机构 |

#### 2. 公卫体检场景的选型决策

**结论：以集成引擎为骨架 + 设备网关做设备适配 + 国标 JSON 直报公卫系统（不走 HL7/FHIR）**

**理由**：

1. **公卫体检的"标准"不是 HL7/FHIR，是国家公卫规范**
   - 公卫系统对接用省级公卫系统规定的 XML/JSON 格式 + WebService 接口
   - 每个省格式不同，不能用 FHIR 统一
   - 这部分走"国标适配层"，不走医疗标准

2. **设备数据采集与 LIS/PACS 性质完全不同**
   - 设备数据采集：**协议层异构**（串口/网口/蓝牙/SDK），数据本身简单（血压值、心电波形）
   - LIS/PACS 对接：**协议层标准**（HL7/DICOM），数据本身复杂（检验项目、影像）
   - 设备数据走"设备网关 + 协议适配器"，不走集成引擎
   - 集成引擎适合消息路由，不适合做设备协议解析

3. **乡镇卫生院场景下，往往没有 LIS/PACS**
   - 体检系统直接接设备，设备结果落本地库
   - 没有 LIS 可对接，集成引擎无用武之地
   - 主检报告直接基于本地数据生成

4. **三甲医院场景下，已有 LIS/PACS**
   - 体检系统通过集成引擎对接 LIS（拿检验结果）+ PACS（拿影像报告）
   - 此时集成引擎减少适配代码

**最终架构**：

```
设备层：
  设备网关（自研）
    ├─ 串口适配器（血压计、身高体重仪）
    ├─ 网口适配器（生化仪、血球仪）
    ├─ 蓝牙适配器（便携心电）
    ├─ SDK 适配器（B 超、骨密度）
    └─ 标准化输出（统一 JSON 格式）

医疗系统对接层：
  集成引擎 Mirth Connect（仅三甲医院场景启用）
    ├─ HL7 v2 通道：LIS 检验结果接收（ORU^R01）
    ├─ DICOM 通道：PACS 影像接收（C-STORE）
    ├─ Worklist 通道：体检项目推送到设备（Modality Work List）
    └─ HTTP 通道：HIS 就诊信息查询

公卫上报层：
  国标适配层（自研，不走集成引擎）
    ├─ 公卫 XML/JSON 打包器（按省份差异化）
    ├─ WebService 客户端
    ├─ XMLDSig 签名
    └─ 上报状态机（上报中/失败/已上报）
```

**关键设计**：
- 设备网关与集成引擎解耦：设备数据先落本地，再由集成引擎按需推给 LIS/PACS
- 公卫上报独立成层：不混入医疗标准对接，避免标准污染
- 乡镇场景禁用集成引擎：减少部署成本，只用设备网关 + 国标适配

#### 3. 小机构 vs 三甲医院的差异化设计

**小机构（乡镇卫生院/社区中心）**：

| 特点 | 集成方案 |
|---|---|
| 无 LIS/PACS | 体检系统直接接设备，结果落本地 |
| 网络 4G/卫星 | 设备网关断网缓存，网络恢复批量上传 |
| IT 运维弱 | 不部署集成引擎，全 SaaS |
| 设备品牌杂 | 设备网关协议适配器覆盖主流品牌 |
| 体检量大（批量） | 检中执行支持批量操作（100 人血压一次录完） |

**三甲医院体检中心**：

| 特点 | 集成方案 |
|---|---|
| 已有完整 LIS/PACS/HIS | 部署集成引擎 Mirth Connect，标准对接 |
| 网络稳定（院内万兆） | 实时同步，无需断网缓存 |
| IT 运维强 | 集成引擎独立部署，体检系统通过引擎对接 |
| 设备品牌统一（采购规模化） | 设备网关协议适配器精简 |
| 体检量分散（个人预约为主） | 检中执行单兵作战，强调实时性 |

**SaaS 化部署的约束**：

| 维度 | SaaS（多机构共享） | 私有化（单机构部署） |
|---|---|---|
| 部署模式 | 共享一套核心，按 org_id 隔离 | 每机构一套 |
| 集成引擎 | 不能共享（每机构对接的 LIS/PACS 不同） | 必须部署 |
| 设备网关 | 不能共享（设备品牌/型号各机构不同） | 必须部署 |
| 公卫上报 | 各省不同，可共享核心 + 配置差异化 | 单机构部署 |
| 升级节奏 | 平台统一升级 | 各机构独立升级 |
| 数据归属 | 平台托管（合规要求确认） | 机构自持 |

**SaaS 化下的混合架构**：

```
SaaS 核心层（云端，多机构共享）：
  - 预约登记、报到分流、检中执行、主检报告、随访工单
  - 公卫上报（按省份差异化配置）
  - 居民端、机构管理后台

机构私有部署层（机构机房）：
  - 设备网关（对接本机构设备）
  - 集成引擎 Mirth Connect（对接本机构 LIS/PACS）
  - 数据同步代理（与云端 SaaS 核心同步）

云端 ↔ 机构：MQTT/HTTPS 长连接，断网缓存重传
```

**关键约束**：
- 设备网关必须本地部署：设备通过局域网/串口连接，不能跨公网
- 集成引擎必须本地部署：LIS/PACS 在院内网络，跨公网有合规与延迟问题
- 业务核心可以 SaaS：预约、报告、上报等不依赖院内设备的模块可以共享

#### 4. 集成模式演进路径

**演进路径：点对点 → 集成引擎 → 集成引擎 + FHIR 标准化**

**阶段一：点对点（1-3 个机构，单系统对接）**

```
体检系统 ──点对点──> LIS（厂家A）
         ──点对点──> PACS（厂家B）
```

- 适用：初创期，机构少，对接的系统少
- 痛点：每接一个机构都要重写适配代码
- 演进信号：机构数 > 5 个，对接的 LIS/PACS 厂家 > 3 个

**阶段二：引入集成引擎（5+ 个机构，多厂家对接）**

```
体检系统 ──> 集成引擎 Mirth Connect
              ├─ Channel 1: LIS-A 厂家（HL7 v2）
              ├─ Channel 2: LIS-B 厂家（私有 API）
              ├─ Channel 3: PACS-A 厂家（DICOM）
              └─ Channel 4: PACS-B 厂家（HTTP API）
```

- 引入集成引擎屏蔽厂家差异
- 体检系统只对接引擎的标准接口
- 引擎层做协议转换、消息路由
- 演进信号：开始接 FHIR 系统、跨机构数据交换需求

**阶段三：集成引擎 + FHIR 标准化（区域平台、跨机构交换）**

```
体检系统 ──> 集成引擎 ──> FHIR Server（标准化）
                          ├─ FHIR 接口对外（区域平台、互联网医院）
                          ├─ HL7 v2 适配（旧 LIS）
                          ├─ DICOM 适配（旧 PACS）
                          └─ 国标适配（公卫上报）
```

- 引擎层增加 FHIR Server，对外提供 FHIR 标准接口
- 内部系统逐步迁移到 FHIR，旧系统保留 HL7 v2 适配
- 跨机构数据交换走 FHIR

**过渡期混合架构**：

| 数据流 | 协议 | 说明 |
|---|---|---|
| 体检系统 ↔ 引擎 | FHIR | 新建系统走标准 |
| 引擎 ↔ 旧 LIS | HL7 v2 | 适配旧系统 |
| 引擎 ↔ 新 LIS | FHIR | 新系统直接标准对接 |
| 引擎 ↔ PACS | DICOM | 影像标准不变 |
| 引擎 ↔ 公卫系统 | 国标 XML/JSON | 不走医疗标准 |
| 引擎 ↔ 居民健康档案 | FHIR | 跨机构交换 |

**演进关键决策点**：
- **不要一上来就上 FHIR**：国内 LIS/PACS 普遍不支持，强行 FHIR 会增加无谓的适配层
- **集成引擎越早引入越好**：5 个机构是临界点，超过后点对点维护成本爆炸
- **国标适配层独立**：公卫上报与医疗标准对接解耦，避免标准污染
- **设备网关永不混入集成引擎**：设备协议层与消息层性质不同，强行混入会让引擎臃肿
