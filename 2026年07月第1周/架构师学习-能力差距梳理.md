## 一、智慧体检业务架构（Day01）

### 差距1.1：FHIR 资源模型实践不足
> Day1发现

- **现状**：知道 FHIR 是 HL7 最新标准（RESTful + JSON/XML + 资源模型），但未实践过资源关联、引用解析
- **架构师水平**：能基于 Patient/Observation/DiagnosticReport/ImagingStudy 资源建模体检业务，实践过 FHIR Server，能对外提供 FHIR 标准接口
- **补足方向**：
  - 学习 FHIR 资源模型（Patient/Observation/DiagnosticReport/ImagingStudy/Encounter）
  - 用 HAPI FHIR 实践一个 FHIR Server Demo
  - 学习 FHIR 资源关联（Reference）、搜索（Search Parameter）、包含（Include）机制

### 差距1.2：DICOM 协议细节不熟
> Day1发现

- **现状**：知道 DICOM 是影像标准，C-STORE/C-FIND/C-MOVE 概念知道，但协议细节（DIMSE、SOP Class、Transfer Syntax）不熟
- **架构师水平**：能讲清 DICOM 三大服务类（Storage/Query-Retrieve/Worklist），实践过 dcm4che/Orthanc PACS 对接
- **补足方向**：
  - 学习 DICOM 协议（DIMSE-C/DIMSE-N、SOP Class UID、Transfer Syntax）
  - 实践 Orthanc 开源 PACS 部署与对接
  - 学习 Modality Work List（MWL）在体检场景的应用

### 差距1.3：Mirth Connect 工程实践不足
> Day1发现

- **现状**：知道 Mirth Connect 是开源医疗集成引擎，能讲清 Channel/Transformer/Connector 模型，但未写过生产级通道
- **架构师水平**：能用 Mirth JavaScript 转换器写复杂消息转换（HL7 v2 → FHIR、ASTM → HL7 ORU^R01），实践过高可用部署、消息堆积治理
- **补足方向**：
  - 部署 Mirth Connect，写一个 HL7 v2 ORU^R01 → FHIR Observation 转换通道
  - 学习 Mirth 部署架构（Cluster 模式、消息库分库）
  - 阅读 Mirth 官方文档与社区案例

### 差距1.4：公卫国标未深读
> Day1发现

- **现状**：知道有《国家基本公共卫生服务规范》第三版，但未深读上报格式与字段要求
- **架构师水平**：能讲清公卫体检上报的国标 XML/JSON 字段（人群分类、检查项、异常项、健康评估）、WebService 接口规范、各省差异化适配
- **补足方向**：
  - 深读《国家基本公共卫生服务规范》第三版"健康档案"与"老年人健康管理"章节
  - 学习各省级公卫系统对接规范（如江苏、浙江、广东差异）
  - 复盘啄木鸟云健康公卫上报的省份适配实现

### 差距1.5：商业体检模型不清
> Day1发现

- **现状**：仅公卫体检视角，商业体检（美年大健康、爱康国宾）业务模型与架构差异不清
- **架构师水平**：能对比公卫体检 vs 商业体检 vs 入职体检 vs 从业人员健康证体检的业务模型、监管要求、收费模式、架构差异
- **补足方向**：
  - 学习美年大健康、爱康国宾的业务模型（套餐设计、加项、加价、报告交付）
  - 学习入职体检、从业人员健康证体检的监管要求（健康证需上传监管系统）
  - 调研商业体检 SaaS 系统架构（如北京量康、杭州健培）

### 差距1.6：SaaS 多机构隔离的混合架构实战不足
> Day1发现，延续第5周差距1.2（异地多活与容灾）

- **现状**：能设计云端 SaaS + 机构私有部署的混合架构，但设备网关/集成引擎的私有部署与云端协同（断网缓存、数据同步、版本管理）实战经验不足
- **架构师水平**：实践过多机构 SaaS + 私有部署混合架构，能设计云端↔机构的可靠数据同步通道（MQTT/HTTPS + 断网缓存 + 冲突解决）
- **补足方向**：
  - 复盘啄木鸟云健康多机构部署架构，识别云端↔机构的同步痛点
  - 学习 MQTT 协议在医疗场景的应用（如远程医疗、家庭医生签约）
  - 学习多端数据同步冲突解决（CRDT、最后写入胜出、版本向量）

### 差距1.7：设备协议适配的工程化不足
> Day1发现

- **现状**：知道设备协议五花八门（串口/网口/蓝牙/SDK），但协议适配器的工程化（动态加载、热插拔、版本管理、远程升级）经验不足
- **架构师水平**：设计可扩展的设备网关框架，支持 SPI 化的协议适配器插件化加载，远程升级不中断业务
- **补足方向**：
  - 学习 Java SPI 机制在设备网关中的应用
  - 学习 OSGi / 插件化框架（如 pf4j）在医疗设备适配中的应用
  - 复盘啄木鸟云健康设备网关的协议适配器设计

## 二、HIS/EMR 核心架构（Day02）

### 差距2.1：CDA 文档模型实践不足
> Day2发现

- **现状**：知道 CDA 是 HL7 v3 的文档模型（Header + Body + Section + Entry），但未实践过 CDA 模板设计、文档生成、文档解析
- **架构师水平**：能基于 CDA 模板设计体检报告（Header + 检查项/检验项/影像/主检结论各 Section），实践过 CDA 生成与解析，能对接 IHE XDS 文档共享
- **补足方向**：
  - 学习 CDA 文档模型（Header/Body/Section/Entry 四层结构）
  - 实践一个体检报告 CDA 模板（Java + HAPI FHIR CDA 模块）
  - 学习 IHE XDS.b 文档共享标准（Document Repository / Document Registry）

### 差距2.2：EMPI 概率匹配算法不熟
> Day2发现

- **现状**：知道 EMPI 的概念与确定性匹配（身份证主键），但概率匹配算法（Fellegi-Sunter）的具体权重计算、阈值调优不熟
- **架构师水平**：能实现简化版 EMPI（确定性 + 概率匹配），实践过人工审核队列，能针对新生儿/外籍/隐私脱敏场景做差异化策略
- **补足方向**：
  - 学习 Fellegi-Sunter 算法（m-probability / u-probability / 权重计算）
  - 实践简化版 EMPI（Java + ElasticSearch）
  - 学习 IHE PIX（患者交叉引用）标准

### 差距2.3：三甲医院 HIS/EMR 实战经验不足
> Day2发现

- **现状**：知道主流 HIS 厂家（卫宁、东软、创业慧康），但未参与过三甲医院 HIS/EMR 的实际部署、对接、交付
- **架构师水平**：能讲清三甲医院 HIS/EMR 的部署架构（数据库选型、集成引擎、医院信息科对接流程、病案室归档流程），实践过多厂家 HIS 适配
- **补足方向**：
  - 调研卫宁、东软、创业慧康的三甲医院部署案例
  - 学习医院信息科的工作流程（需求评审、接口对接、上线验收）
  - 学习医院病案室的归档流程（出院病历 7 天内归档、病案审核）

### 差距2.4：电子病历归档工程实现不熟
> Day2发现

- **现状**：知道归档要求（门诊 15 年、住院 30 年、不可篡改），但归档库的技术选型（WORM 存储、电子签名、时间戳、区块链存证）工程实现不熟
- **架构师水平**：能设计归档库方案（关系库 + WORM + 电子签名 + 时间戳），实践过归档库与操作库的同步、归档后的修改留痕
- **补足方向**：
  - 学习 WORM 存储（一次性写入多次读取）的工程实现
  - 学习电子签名（CA 证书）与时间戳（国家授时中心）的对接
  - 学习区块链存证在医疗病历的应用案例

### 差距2.5：医疗数据合规落地不足
> Day2发现，延续差距D（金融监管合规→医疗监管合规）

- **现状**：知道数据安全法/个保法/人口健康信息管理办法的框架，但具体到 EMPI 跨机构数据共享的患者授权机制、数据脱敏规则的工程实现不熟
- **架构师水平**：能设计患者授权机制（授权书 + 授权范围 + 授权撤销 + 审计日志），能落地数据脱敏规则（按用途动态选择脱敏级别）
- **补足方向**：
  - 学习《健康医疗数据安全指南》的分类分级与脱敏规则
  - 学习患者授权机制的设计（授权书模板、授权流转、授权撤销）
  - 学习医疗数据出境的合规要求

### 差距2.6：区域卫生信息平台实践不足
> Day2发现

- **现状**：知道区域卫生信息平台是跨机构数据聚合的中枢，但未实践过平台的对接（CDC 同步、跨机构调阅、居民健康档案聚合）
- **架构师水平**：能讲清区域平台的数据汇聚模式（CDC/文档共享/接口调用），实践过跨机构调阅与数据冲突解决
- **补足方向**：
  - 学习上海/福建/浙江区域卫生信息平台的建设案例
  - 学习 IHE PIX/XDS 标准在国内的实践
  - 复盘啄木鸟云健康与省级公卫系统的对接，识别跨机构调阅的痛点

### 差距2.7：医院信息化销售/交付/运维模式不清
> Day2发现

- **现状**：作为产品方（啄木鸟云健康）熟悉 SaaS 模式，但医院信息化的销售（招标/投标）、交付（实施/培训）、运维（驻场/远程）模式不清
- **架构师水平**：能讲清医院信息化的销售周期（3-12 个月）、交付流程（需求/部署/培训/验收）、运维模式（SLA/驻场/远程），理解医疗信息化的商业逻辑
- **补足方向**：
  - 学习医院信息化项目的招投标流程
  - 学习医院信息化项目的交付流程（需求调研/部署/培训/验收）
  - 学习医院信息化项目的运维模式（SLA/驻场/远程/工单）

## 三、HL7/FHIR 标准与医疗信息互操作性（Day03）

### 差距3.1：FHIR 资源建模与 Server 实战不足
> Day3发现，延续差距1.1

- **现状**：能讲清 FHIR 四代标准对比与资源模型（Patient/Encounter/Observation/DiagnosticReport），能用 Extension 表达国标字段，但未实践过 HAPI FHIR Server 部署、Search Parameter 调优、Subscription 推送、Transaction Bundle 批量写入
- **架构师水平**：能基于 HAPI FHIR 部署生产级 FHIR Server，设计体检业务 Profile 与 IG，能调优 PostgreSQL JSONB 索引、缓存策略、多租户隔离
- **补足方向**：
  - 部署 HAPI FHIR Server（PostgreSQL + Redis），实践体检业务资源建模
  - 学习 FHIR IG 编写，定义体检业务 Profile 与 Extension
  - 学习 Search Parameter 调优（索引设计、链式/反向链、复合参数）
  - 实践 Subscription（rest-hook）推送体检报告生成事件
  - 学习 HAPI FHIR 性能优化（PostgreSQL JSONB GIN 索引、缓存、分页）

### 差距3.2：HL7 v2 消息解析与 Mirth Connect 工程实战不足
> Day3发现，延续差距1.3

- **现状**：能讲清 HL7 v2 消息结构（MSH/PID/PV1/OBR/OBX）、Mirth Connect Channel 模型（Source/Transformer/Destination）、字段映射（PID-3 -> Patient.identifier、OBX-5 -> Observation.valueQuantity），但未写过生产级 Mirth 通道，未实践 Mirth 集群部署与高可用
- **架构师水平**：能用 Mirth JavaScript Transformer 写复杂消息转换（HL7 v2 -> FHIR、ASTM -> HL7 ORU^R01），实践过 Mirth 集群部署、消息堆积治理、死信队列
- **补足方向**：
  - 部署 Mirth Connect，写一个 HL7 v2 ORU^R01 -> FHIR Observation 转换通道
  - 学习 Mirth 集群部署（Cluster 模式、消息库分库）
  - 实践消息堆积治理（异步处理 + 多线程消费 + 限流 + 死信队列）
  - 学习 Mirth 与 Rhapsody/Corepoint 商业集成引擎的对比

### 差距3.3：SMART on FHIR 与 OAuth2 授权落地不足
> Day3发现

- **现状**：能讲清 OAuth2 三种授权（患者/医生/机构）与 SMART on FHIR 流程框架，但未实践过 SMART on FHIR 的完整授权流程（launch 上下文、scope 校验、token 验签）
- **架构师水平**：能基于 SMART on FHIR 实现完整授权流程，落地患者授权留痕、医生授权医患关系校验、机构授权合同范围限制
- **补足方向**：
  - 学习 SMART on FHIR 规范（launch/patient、launch/practitioner、patient/.*.read 等 scope）
  - 实践 OAuth2 + OpenID Connect 完整流程（授权码模式、PKCE、token 刷新）
  - 学习 SMART on FHIR 与国内合规（个保法单独同意、数据安全法分级评估）
  - 学习国内互联网医院授权实践案例

### 差距3.4：国内标准生态深度不足
> Day3发现

- **现状**：能讲清国际标准（HL7/FHIR/IHE/SNOMED/LOINC/ICD）与国标（WS 445、共享文档规范、公卫规范）的层级关系，但对国内 FHIR 落地案例（阿里健康、东软、微脉、平安好医生）的深度调研不足，对中文 FHIR IG 现状不清楚
- **架构师水平**：能讲清国内 FHIR 落地的"标杆案例 + 标准现状 + 推进路径"，能针对国内合规要求设计 FHIR IG
- **补足方向**：
  - 调研阿里健康 FHIR、东软 FHIR、微脉、平安好医生的实践案例
  - 学习《医院信息互联互通标准化成熟度测评》对 FHIR 的要求
  - 学习 OpenEHR 中文社区在推动中文 FHIR IG 的进展
  - 复盘啄木鸟云健康公卫上报现状，规划 FHIR 引入路径

### 差距3.5：FHIR 与国标兼容的工程实现不熟
> Day3发现

- **现状**：能讲清"双轨架构"思路（领域事件驱动 + FHIR/国标适配器并行），但 FHIR Extension 表达国标字段的具体实现、FHIR 资源与国标 XML 的双向映射规则、对账机制的工程实现不熟
- **架构师水平**：能实现 FHIR 资源 ↔ 国标 XML 的双向映射器，能设计对账机制（FHIR Server 资源 vs 国标上报记录），能处理国标上报失败反馈到 FHIR 状态
- **补足方向**：
  - 实践 FHIR Extension 定义（StructureDefinition）与国标字段映射
  - 学习 FHIR ConceptMap 资源做术语映射（LOINC ↔ 国标检验代码）
  - 实现国标上报状态反馈到 FHIR（DiagnosticReport.status + Extension）
  - 设计双轨对账机制（定时任务 + 告警 + 人工介入）

### 差距3.6：FHIR Server 多租户与高并发实战不足
> Day3发现，延续第5周差距1.2（异地多活与容灾）

- **现状**：能讲清多租户隔离三种方案（数据库级/表级/Schema 级）与高并发优化策略（横向扩展、读写分离、缓存、批量接口），但未实践过 HAPI FHIR 多租户部署、百万级 QPS 的性能调优
- **架构师水平**：能部署 HAPI FHIR 多租户集群，能扛住百万级日调用（API Gateway 限流 + Redis 缓存 + PG 读写分离 + 异步索引），能设计监控告警体系
- **补足方向**：
  - 学习 HAPI FHIR 多租户实现（TenantInterceptor + X-Tenant-Id）
  - 实践 HAPI FHIR 集群部署（无状态 + LB + PG 主从 + Redis 集群）
  - 学习 FHIR Server 性能调优（Search Parameter 索引、JSONB GIN、缓存策略）
  - 学习 FHIR Server 监控（Prometheus + Grafana + ELK）

### 差距3.7：FHIR 版本控制与并发实战不足
> Day3发现

- **现状**：能讲清 ETag + If-Match 乐观锁机制、Transaction Bundle 原子性、历史版本查询，但未实践过多端修改冲突的解决策略
- **架构师水平**：能设计多端修改冲突解决方案（业务规则仲裁 + 客户端重试 + 服务端合并），能在体检批量上传场景应用 Transaction Bundle
- **补足方向**：
  - 实践 HAPI FHIR 的 ETag + If-Match 乐观锁
  - 学习 FHIR Transaction vs Batch 的差异与适用场景
  - 设计体检场景的多端冲突解决策略（医生修改 vs 设备上传 vs 审核医生）
  - 学习 FHIR 历史版本查询在审计场景的应用

## 四、DICOM 医学影像标准与 PACS 系统架构（Day04）

### 差距4.1：DICOM 协议体系深度不足
> Day4发现，延续差距1.2

- **现状**：能讲清 DICOM 文件结构（Preamble + DICM + Data Elements）、Tag/VR/Length/Value 编码、Transfer Syntax 概念、DIMSE-C 五大命令（C-STORE/C-FIND/C-MOVE/C-GET/C-ECHO）、三大服务类（Storage/Query-Retrieve/MWL），但协议细节（Presentation Context 协商的失败处理、Storage Commitment 的状态机、DIMSE-N 命令的实际使用、Association 的并发控制）不熟
- **架构师水平**：能讲清 DICOM 协议栈全貌（DUL + DIMSE + SOP Class），实践过 Storage Commitment 落地，能针对设备厂家（GE/飞利浦/西门子/东软）的 DICOM 实现差异做适配，能自研轻量 DICOM SCP/SCU
- **补足方向**：
  - 深读 DICOM 官方文档（PS3.7/PS3.8 协议部分、PS3.4 服务类部分）
  - 实践 Storage Commitment 的完整流程（N-ACTION + N-EVENT-REPORT）
  - 用 dcm4che-dicombs 实践 DIMSE-N 命令
