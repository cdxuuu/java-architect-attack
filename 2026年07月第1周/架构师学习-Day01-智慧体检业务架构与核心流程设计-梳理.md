## 一、智慧体检系统在高可用架构中的定位

### 1.1 核心一句话
> **智慧体检 = 公卫监管 + 设备集成 + 报告流转 + 数据上报的"四重耦合"系统**
>
> 与电商、支付不同，智慧体检的工程约束是"批量、断网、监管、异构"——架构师要的是**让村医在 4G 信号下 5 分钟内完成 100 人血压采集**，而不是"理论上的高可用"。

### 1.2 智慧体检的四大特征

| 特征 | 含义 | 架构影响 |
| --- | --- | --- |
| 批量上门 | 单日单点 200~500 人，体检车开进乡镇 | 设备断网缓存、批量操作、检中并发 |
| 强监管上报 | 必须按国标打包上传省级公卫系统 | 国标适配层独立、上报状态机、补报重试 |
| 多机构 SaaS | 一套系统服务几十家乡镇卫生院 | 租户隔离、分库分表、配置差异化 |
| 设备异构 | 血压/心电/B超/生化品牌五花八门，协议有串口/网口/蓝牙/SDK | 设备网关协议适配器、标准化输出 |

### 1.3 智慧体检的多层架构

```
居民端/医生端/机构管理后台
   ↓ HTTPS / 网关
接入层（API 网关：鉴权、租户路由、限流、灰度）
   ↓
业务层（预约登记/报到分流/检中执行/主检报告/随访工单/套餐排班）
   ↓
集成层（LIS 适配 / PACS 适配 / HIS 对接 / 健康档案）
   ↓
设备层（设备网关：串口/网口/蓝牙/SDK 适配器）
   ↓
上报层（公卫上报国标 XML/JSON / 医保对接 / 行业监管）
   ↓
支撑层（MQ / XXL-Job / 配置中心 / 文件存储 / ES / 监控告警）
```

## 二、核心模块边界：架构师必须能画出来

### 2.1 十二大核心模块职责矩阵

| 模块 | 职责 | 不做什么 |
| --- | --- | --- |
| 预约登记 | 居民预约、人群资格校验、套餐匹配、预约号生成 | 不做检中操作 |
| 报到分流 | 现场签到、身份证识别、条码打印、检查项目分发 | 不持有体检结果 |
| 检中执行 | 各科室检查执行、结果录入、异常项标记 | 不调设备（交给设备网关） |
| 设备网关 | 设备协议适配、原始报文解析、断网缓存重传 | 不持有业务逻辑 |
| LIS 适配 | 检验仪器结果接收（ASTM/HL7 ORU^R01）、条码双向通讯 | 不做临床解释 |
| PACS 适配 | 影像接收（DICOM Store）、Worklist 推送、影像调阅 | 不存影像 |
| 主检报告 | 主检医生审核、异常项汇总、报告生成、推送 | 不修改原始数据 |
| 公卫上报 | 国标格式打包、签名、定时批量上报、回执管理 | 不修改业务数据 |
| 随访工单 | 异常项触发随访、工单生成、随访结果回写 | 不替代医生诊断 |
| 套餐排班 | 体检套餐管理、人群-套餐映射、体检车排班 | 不持有居民档案 |
| 居民端 | 预约、报告查询、随访填写、健康教育推送 | 全部走网关 |
| 机构管理后台 | 机构信息、报告模板、设备绑定、人员权限 | 不查其他机构数据 |

### 2.2 同步链路 vs 异步链路

```
同步链路（居民/医生在等）：
  预约 → 报到（身份证识别 + 条码打印）→ 检中录入（设备结果弹窗确认）
  → 危急值弹窗告警 + 短信通知主检医生

异步链路（后台处理）：
  设备出结果 → 设备网关（断网缓存）→ MQ → LIS 适配 → 检中执行落库
  所有项目完成 → MQ → 主检报告生成（自动汇总 + 医生审核）→ PDF
              → MQ → 居民端推送（小程序订阅消息）
              → MQ → 公卫上报任务入队列

定时任务（凌晨）：
  公卫上报批处理 → 国标 XML 打包 → 省级公卫系统 → 回执落库 → 失败补报
  → 异常项 → 随访工单生成 → 推送村医
```

**异步的六大理由**：
1. 设备数据回流天然异步（生化仪 5~20 分钟）
2. 报告生成依赖多源数据汇聚（事件计数 + 状态机）
3. 公卫上报天然批量（接口 QPS 1~2，本地攒批）
4. 居民端推送不可阻塞业务（微信接口可能慢）
5. 危急值告警必须可靠（MQ + 多通道兜底）
6. 同步路径仅保留预约、报到、检中录入

## 三、业务订单 vs 体检流程实例：必须拆分的理由

### 3.1 关系模型

```
批量上门体检任务 TaskBatch20260706-XX镇-老年体检
   ├─ 体检流程实例1 老人张三（预约→报到→检中→报告→上报）
   ├─ 体检流程实例2 老人李四
   └─ ... 300 人
```

### 3.2 拆分四大理由

| # | 理由 |
| --- | --- |
| 1 | 职责分离：体检任务关心"调度"，流程实例关心"个体执行" |
| 2 | 批量场景的统计聚合：完成率、异常率与单人体检查询是不同访问模式 |
| 3 | 生命周期不同：任务当周完成，流程实例永久保留（公卫档案） |
| 4 | 多业务线复用：流程实例既能服务公卫体检，也能服务商业体检、入职体检、健康证体检 |

### 3.3 批量上门 vs 日常门诊的订单模型差异

| 维度 | 批量上门体检 | 日常门诊体检 |
| --- | --- | --- |
| 触发方式 | 村医代约/集体组织 | 居民自主预约/医生开单 |
| 预约粒度 | 按批次（300人/天） | 按个人（按号源） |
| 报到方式 | 集中报到（身份证批量识别） | 一人一报到 |
| 套餐选择 | 人群固定套餐 | 居民可选/医生开单 |
| 收费 | 免费（公卫项目） | 付费（医保/自费） |
| 上报要求 | 必须上报公卫系统 | 不强制上报 |
| 网络条件 | 4G/卫星，可能断网 | 院内局域网稳定 |
| 模型差异 | TaskBatch + ProcessInstance 两层 | ProcessInstance 一层 |

### 3.4 多机构 SaaS 租户隔离

| 数据类型 | 隔离方式 | 理由 |
| --- | --- | --- |
| 业务数据 | 强隔离：分库分表，按 org_id 分片 | 监管要求独立 + 数据量差异大 |
| 居民健康档案 | 强隔离：按 org_id 分片，身份证全局唯一索引跨片查询 | 流动就医需跨片查档案 |
| 设备配置/套餐模板 | 弱隔离：共享表 + org_id 字段 | 数据量小，便于平台统一维护 |
| 用户权限/机构人员 | 强隔离：分库 | 安全合规要求 |
| 报告模板 | 中隔离：共享表 + 模板继承（平台标准 + 机构覆盖） | 减少重复维护 |
| 字典数据 | 共享：全平台一份 | 行业标准 |

**分片键选择**：
- 首选 `org_id`：99% 查询是机构维度
- 次选 `id_card_hash % N`：仅用于跨机构查档案
- 绝对不用 `cuid`/`resident_id`：会导致同一居民多次体检记录散落不同分片

## 四、公卫体检状态机：架构师必须能默写

### 4.1 完整状态流转图

```
                  ┌─────────┐
                  │ BOOKED  │ 待检
                  └────┬────┘
                       │ 现场报到
                       ▼
                  ┌─────────────┐
                  │ CHECKED_IN  │ 已报到
                  └────┬────────┘
                       │ 第一个科室开始
                       ▼
                  ┌─────────────┐
        ┌────────│  IN_EXAM    │────────┐
        │        │  检中        │        │
        │        └────┬────────┘        │
   居民离场未做完  所有项目完成       主动取消
        │             │                 │
        ▼             ▼                 ▼
   ┌─────────┐  ┌──────────────┐   ┌──────────┐
   │ PARTIAL │  │PENDING_REVIEW│   │CANCELLED │
   │ 漏项挂起 │  │ 待主检       │   │ 已取消   │
   └────┬────┘  └──────┬───────┘   └──────────┘
        │              │ 主检医生开始
        │              ▼
        │         ┌──────────┐
        │         │REVIEWING │ 主检中
        │         └────┬─────┘
        │     ┌────────┼────────┐
        │  发现漏项  审核通过   审核驳回
        │     │        │        │
        └─────┘   ┌──────────┐  回到 IN_EXAM
                  │ REPORTED │  已出报告
                  └────┬─────┘
                       │ 定时上报公卫
                       ▼
                  ┌──────────────────┐
                  │REPORTED_TO_PH    │
                  └────┬─────────────┘
                       │ 上报回执成功
                       ▼
                  ┌──────────┐
                  │ CLOSED   │ 终态
                  └──────────┘
```

### 4.2 三处危险流转

**① IN_EXAM → PENDING_REVIEW vs IN_EXAM → PARTIAL 并发竞争**

四重防护：
```sql
-- 1. 乐观锁 + 状态前置校验
UPDATE process_instance SET status=PENDING_REVIEW, version=version+1 
WHERE id=? AND status=IN_EXAM AND version=?;
-- 影响行数=0 则放弃
```
- 乐观锁（version）
- 状态机白名单（PARTIAL 不能从 PENDING_REVIEW 流转来）
- 业务前置校验（必检项计数达标 / 居民确实离场）
- 设备结果回流幂等（标本条码 + 检查项目 唯一索引）

**② REPORTED → REVIEWING 报告回退**
- 必须校验"未上报公卫"才能回退
- 保留原报告快照（report_snapshot 表），不直接覆盖
- 回退后置 REVIEWING，修改后生成新版本，版本号 +1

**③ CLOSED → REVIEWING 已上报后撤销重报**
- 必须走特殊审批流程（人工审核）
- 必须先调公卫系统"撤销上报"接口
- 撤销失败时兜底走"补充报告"流程，不能强行覆盖公卫侧已确认数据

### 4.3 漏项处理：PARTIAL 不是终态

```
IN_EXAM
   ├─ 居民离场 + 必检项未完成 → PARTIAL（漏项挂起）
   │                              │ 居民回来补检
   │                              ▼
   │                           IN_EXAM（继续检）
   └─ 所有必检项完成 → PENDING_REVIEW

主检审核阶段发现漏项：
REVIEWING
   ├─ 必检项漏 → PARTIAL（挂起等补检）
   └─ 可选项漏（居民放弃）→ 标记"放弃原因" → REPORTED

已出报告后补检：
REPORTED → REVIEWING（重新审核）→ REPORTED（V2 补充报告）

已上报公卫后补检：
REPORTED_TO_PH → CLOSED → REVIEWING（特殊审批）→ REPORTED → REPORTED_TO_PH（新版本）
```

**关键设计**：
- 必检项 vs 可选项区分：必检项漏必须补，可选项漏可标记"放弃"
- 补检走原流程：结果落到原流程实例，不新建实例
- 补充报告机制：V1 保留，V2 标注"补充"
- 漏项检测定时任务：每 10 分钟扫描 IN_EXAM 超 4 小时未操作的实例，自动转 PARTIAL

### 4.4 公卫上报失败与补报

**上报子状态机**：
```
REPORTED
   │ 凌晨定时任务触发
   ▼
REPORTED_TO_PH
   ├─ UPLOADING（上报中）
   ├─ UPLOAD_FAILED（失败，自动重试指数退避）
   └─ UPLOADED（等回执）
       │
       │ 回执查询（每小时）
       ▼
   ├─ SUCCESS → CLOSED
   ├─ FAIL（数据错）→ 回 REPORTED + 人工介入
   └─ PENDING → 继续等
```

**失败处理矩阵**：

| 失败类型 | 处理 |
| --- | --- |
| 网络超时 / 5xx | 自动重试，指数退避（5min/15min/1h/4h/24h），最多 5 次 |
| 签名错 / 证书过期 | 立即告警，不重试，人工更新证书 |
| 数据格式错 | 标记错误原因，回 REPORTED，人工修改后重新触发 |
| 重复上报 | 幂等返回成功，不报错 |
| 公卫侧已上报 | 标记"重复上报"，记录公卫侧原流水号，停止重试 |

**幂等键**：`process_instance_id + report_version + org_id`

## 五、LIS/PACS/RIS 集成模式选型

### 5.1 四种主流模式对比

| 维度 | 点对点 | 集成引擎 | HL7 v2 | FHIR |
| --- | --- | --- | --- | --- |
| 接入成本 | 低 | 中 | 中 | 高 |
| 扩展成本 | 高（M×N） | 低（M+N） | 低 | 低 |
| 标准化 | 无 | 引擎层做 | 强（医疗事实标准） | 强（国际标准） |
| 国内支持 | 都支持 | 主流三甲有 | 主流 LIS/HIS 支持 | 少见 |
| 适合场景 | 单机构 | 多系统集成 | 三甲对接 | 新建/跨机构 |

### 5.2 公卫体检场景的选型决策

**结论：集成引擎为骨架 + 设备网关做设备适配 + 国标 JSON 直报公卫（不走 HL7/FHIR）**

**理由**：
1. 公卫体检的"标准"是国家公卫规范，不是 HL7/FHIR
2. 设备数据采集是协议层异构（串口/网口/蓝牙/SDK），不走集成引擎
3. 乡镇卫生院往往没有 LIS/PACS，集成引擎无用武之地
4. 三甲医院场景下，集成引擎减少适配代码

**最终架构**：
```
设备层：
  设备网关（自研）— 串口/网口/蓝牙/SDK 适配器 → 统一 JSON

医疗系统对接层（仅三甲场景启用）：
  集成引擎 Mirth Connect
    ├─ HL7 v2 通道：LIS 检验结果接收
    ├─ DICOM 通道：PACS 影像接收
    ├─ Worklist 通道：体检项目推送到设备
    └─ HTTP 通道：HIS 就诊信息查询

公卫上报层（自研，不走集成引擎）：
  国标适配层 — 公卫 XML/JSON 打包 + WebService + XMLDSig 签名 + 上报状态机
```

### 5.3 小机构 vs 三甲医院的差异化设计

| 维度 | 小机构（乡镇/社区） | 三甲医院体检中心 |
| --- | --- | --- |
| LIS/PACS | 无，体检系统直接接设备 | 已有完整 LIS/PACS/HIS |
| 网络 | 4G/卫星，断网缓存 | 院内万兆，实时同步 |
| IT 运维 | 弱，全 SaaS | 强，集成引擎独立部署 |
| 设备品牌 | 杂，协议适配器全覆盖 | 统一，适配器精简 |
| 体检量 | 批量（200~500人/天） | 分散（个人预约） |
| 集成模式 | 设备网关 + 国标适配 | 集成引擎 + HL7/DICOM |

### 5.4 SaaS 化下的混合架构

```
SaaS 核心层（云端，多机构共享）：
  预约/报到/检中/主检/随访/公卫上报/居民端/管理后台

机构私有部署层（机构机房）：
  设备网关（对接本机构设备）
  集成引擎 Mirth Connect（对接本机构 LIS/PACS）
  数据同步代理（与云端 SaaS 同步）

云端 ↔ 机构：MQTT/HTTPS 长连接，断网缓存重传
```

**关键约束**：
- 设备网关必须本地部署（设备局域网/串口连接，不能跨公网）
- 集成引擎必须本地部署（LIS/PACS 在院内，跨公网有合规与延迟问题）
- 业务核心可以 SaaS（不依赖院内设备的模块共享）

### 5.5 集成模式演进路径

```
阶段一：点对点（1-3 个机构）
   体检系统 ──> LIS-A / PACS-A
   痛点：每接一机构重写适配代码

阶段二：引入集成引擎（5+ 机构）
   体检系统 ──> Mirth Connect ──> LIS-A / LIS-B / PACS-A / PACS-B
   引擎屏蔽厂家差异

阶段三：集成引擎 + FHIR（区域平台）
   体检系统 ──> 集成引擎 ──> FHIR Server（标准化）
                       ├─ FHIR 对外
                       ├─ HL7 v2 适配（旧 LIS）
                       ├─ DICOM 适配（旧 PACS）
                       └─ 国标适配（公卫上报）
```

**演进决策**：
- 不要一上来上 FHIR（国内 LIS/PACS 普遍不支持）
- 集成引擎 5 个机构临界点引入，超过后点对点维护爆炸
- 国标适配层独立，避免标准污染
- 设备网关永不混入集成引擎（协议层与消息层性质不同）

## 六、面试高频提问速答

**Q1：智慧体检系统为什么必须拆分体检任务和体检流程实例？**
A：职责分离（任务管调度，实例管个体执行）、批量场景统计聚合需求、生命周期不同（任务当周完成，实例永久保留）、多业务线复用（公卫/商业/入职/健康证）。批量上门体检一人 300 人是任务-实例两层，日常门诊体检实例单层即可。

**Q2：体检流程的 IN_EXAM → PENDING_REVIEW 与 IN_EXAM → PARTIAL 并发竞争怎么防？**
A：四重防护——乐观锁（version）+ 状态机白名单 + 业务前置校验（必检项计数达标 / 居民确实离场）+ 设备结果回流幂等（标本条码+项目 唯一索引）。核心是数据库层用条件 UPDATE 保证只生效一次。

**Q3：体检漏项如何处理？**
A：PARTIAL 是中间态不是终态，居民回来补检可恢复。必检项漏必须补，可选项漏可标记"放弃原因"通过。补检走原流程，结果落原实例。已出报告后补检生成"补充报告"（V2），原报告（V1）保留。已上报公卫后补检走"撤销重报"特殊审批。

**Q4：公卫上报失败如何处理？**
A：失败分类——网络超时自动重试（指数退避 5 次）；签名错告警不重试；数据格式错回 REPORTED 人工介入；重复上报幂等返回成功。幂等键 = `process_instance_id + report_version + org_id`。已上报成功的报告发现错误走"撤销重报"，撤销失败兜底"补充报告"。

**Q5：智慧体检系统对接 LIS/PACS 选哪种集成模式？**
A：看场景。小机构无 LIS/PACS，体检系统直接接设备，设备网关 + 国标适配即可。三甲医院已有 LIS/PACS，引入集成引擎（Mirth Connect）做协议适配，体检系统只对接引擎标准接口。不要一上来上 FHIR，国内 LIS/PACS 普遍不支持。

**Q6：SaaS 化部署下，设备网关和集成引擎应该云端还是本地？**
A：必须本地。设备通过局域网/串口连接，跨公网有延迟与合规问题。LIS/PACS 在院内网络，跨公网同样有合规问题。SaaS 核心层（不依赖院内设备的预约/报告/上报）可以共享云端，机构私有层（设备网关 + 集成引擎）必须本地，云端↔机构走 MQTT/HTTPS 长连接断网缓存重传。
