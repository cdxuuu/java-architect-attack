# Day 3：分布式锁 — 架构师视角梳理

## 问题全景

分布式环境下，多个进程/线程需要互斥访问共享资源。架构师的核心判断：**先问"问题本质是互斥还是原子操作"，再选方案。选错了方向，方案再精妙也是浪费。**

---

## 一、Redis 锁 — 从裸用到框架，每一步都是踩坑换来的

### 你的业务实践

```
内部管理系统 → SET NX EX（裸实现）
  问题：锁续期、不可重入、误删

ToC 核心业务 → Redisson（框架实现）
  解决：看门狗续期、可重入、Lua 原子操作
```

这个分层本身是对的——内部系统并发低、容忍度高，裸 SETNX 够用；核心业务必须上框架。但裸实现的坑要能说清，面试会追问。

### SET NX EX 的三个经典问题

```
SET lock_key unique_value NX EX 30
  NX — 互斥
  EX 30 — 防死锁
  unique_value — 标识持有者
```

| 问题 | 场景 | 后果 | 解法 |
|------|------|------|------|
| 锁过期释放 | 业务执行超 30s | 其他线程拿到锁，并发执行 | 看门狗续期 |
| 误删别人的锁 | A 的锁过期，B 加锁，A 释放了 B 的锁 | 两个线程同时持有 | Lua 原子判断再删 |
| 不可重入 | 同一线程嵌套调用 | 自己锁自己，死锁 | Redisson 可重入锁 |

**误删的完整过程（面试要能画出来）：**

```
线程A 加锁，value=A，过期 10s
  ↓ 业务执行超过 10s
锁自动过期
  ↓
线程B 加锁，value=B
  ↓
线程A 执行完，DEL key → 删了 B 的锁
  ↓
线程C 也加锁成功 → A 和 C 并发执行
```

**Lua 解锁脚本：**

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
end
return 0
```

为什么必须 Lua？GET + DEL 是两次网络请求，中间可能插进别的操作。Lua 在 Redis 单线程中原子执行，不可打断。

### 看门狗 — 续期频率不是"看剩余时间"，是固定间隔

你的理解是"后台线程定时查看锁的剩余时间来续期"，更精确的逻辑：

```
lockWatchdogTimeout = 30s（默认）
续期间隔 = 30s / 3 = 10s（固定，不看剩余时间）
每次续期 → 重置过期时间为 30s
```

不是"剩余时间少了才续"，是无脑每 10 秒续一次。原因：如果改成"剩余时间 < 10s 才续"，续期请求网络抖动时就可能来不及，锁已经过期了。固定间隔更可靠。

```
时间线：
0s    加锁成功，过期时间=30s，看门狗启动
10s   续期 → 过期时间重置为 30s
20s   续期 → 过期时间重置为 30s
25s   业务完成，主动释放锁，看门狗停止
```

### 看门狗失效的场景 — 面试必问

| 场景 | 后果 | 是否正确行为 |
|------|------|------------|
| 持有锁的进程崩溃 | 看门狗也死了，30s 后锁自动释放 | ✓ 正确，防止死锁 |
| GC 暂停（STW） | 看门狗暂停续期，锁过期，其他线程拿到锁 | ✗ 并发执行 |
| 网络分区 | 看门狗无法续期，锁过期，其他节点拿到锁 | ✗ 并发执行 |
| 手动设了 leaseTime | 看门狗不启动，锁到期直接释放 | ✗ 业务超过 leaseTime 就出问题 |

**leaseTime 的坑：**

```java
// 看门狗启动 ✓
lock.lock();

// 看门狗不启动 ✗ — 60s 后强制释放，业务没完就丢锁
lock.lock(60, TimeUnit.SECONDS);
```

Redisson 的设计：设了 leaseTime 就认为你自己管理过期时间，看门狗不参与。不少团队踩过这个坑。

**架构师视角：** 看门狗解决的是"业务没完锁过期"，解决不了"进程活着但和 Redis 断连"。后者的本质是——Redis 锁不是强一致锁，永远存在极小概率的并发窗口。

## 二、主从切换丢锁 — 知道问题在哪，更要明白为什么市面上的解法不流行

### 丢锁过程

```
1. 客户端A 在 Master 加锁成功
2. Master 还没同步到 Slave
3. Master 宕机，Slave 提升为新 Master
4. 客户端B 在新 Master 加锁成功
5. A 和 B 同时持有锁
```

### Redlock — 理论上的解法

5 个独立 Redis 实例（不是主从），加锁时向所有实例发 SET NX：

```
1. 记录开始时间 T1
2. 向 5 个实例加锁
3. 成功数 N >= 3（多数派）且总耗时 < 锁过期时间 → 成功
4. 失败 → 向所有实例解锁
```

### 为什么生产很少用？你说的对——成本和复杂度

你的判断完全正确。补充架构师视角的原因：

| 不用的原因 | 说明 |
|-----------|------|
| 运维成本 | 5 个独立实例，不能复用现有主从集群 |
| 复杂度 | 加锁 5 次网络往返，延迟翻 5 倍 |
| 仍然不完美 | Martin 证明了即使 Redlock 也有 GC 暂停问题 |
| 有更简单的替代 | fencing token 或业务层幂等，成本更低 |

### Martin vs Antirez — 面试必考的争论

**Martin 的核心批评：分布式锁不是绝对互斥。**

```
客户端A 拿到锁，fencing token = 1
客户端A GC 暂停 5s，锁过期
客户端B 拿到锁，fencing token = 2
客户端A 恢复，以为还持有锁，继续写 → 并发
```

Martin 提的解法——**fencing token**：
- 锁服务每次加锁返回单调递增的 token
- 资源服务只接受更大的 token
- 本质：不信任锁，让资源端自己防

```
客户端A 写入，token=1 → 资源服务记录 max_token=1
客户端B 写入，token=2 → 资源服务检查 2 > 1，接受，max_token=2
客户端A 再写入，token=1 → 资源服务检查 1 < 2，拒绝
```

**Antirez 的反驳：** GC 问题 ZK 也有，不是 Redlock 独有。

**架构师该站哪边？** 不站队，吸收两边的观点：

- Martin 的核心观点是对的：**不要把分布式锁当绝对互斥量用，业务层必须自己做幂等和防重**
- Antirez 的实践也没错：大部分场景下 Redis 锁够用，概率性并发极低
- fencing token 思路很好，但需要资源服务配合改造，不是所有场景都能改

**面试答法：** 先说两方观点，再说自己的判断——"我认同 Martin 的核心观点，分布式锁不是绝对互斥，业务层必须做幂等兜底。但生产中 Redlock 确实少用，因为成本高且仍不完美，更实际的做法是 Redis 锁 + 业务幂等双重保障。"

## 三、秒杀选锁 — 面试最大的坑：问题本身就错了

### 我的回答和纠正

我最初回答"秒杀选 Redis 锁（高性能）"，被纠正为：**秒杀不该用分布式锁**。

这不是细节错误，是方向性错误。架构师的核心能力就是判断问题的本质。

### 问题本质判断

```
互斥访问（同一时刻只能一个进程执行某段逻辑）→ 分布式锁
  例：选主、分布式任务调度、防重复提交

原子操作（读-改-写一步完成）→ 不要用锁，用原子操作
  例：秒杀扣库存、余额扣减、计数器递增
```

秒杀扣库存是"读库存 → 判断 → 扣减"，本质是原子操作，不是互斥访问。用锁是把原子操作降级成了串行化。

### 秒杀全链路分层设计

```
层1：前端限流 — 按钮置灰、验证码、滑动拼图
  ↓ 拦截 90% 请求

层2：网关限流 — 令牌桶/漏桶
  ↓ 再拦 90%

层3：Redis Lua 原子扣库存 — 不用锁！
  ↓ Lua 单线程执行，天然串行

层4：MQ 异步下单 — 削峰填谷
  ↓

层5：订单服务消费 MQ → 这里才需要锁
```

**层3 的 Redis Lua 扣减：**

```lua
local stock = redis.call("get", KEYS[1])
if not stock or tonumber(stock) <= 0 then
    return -1
end
redis.call("decr", KEYS[1])
return tonumber(stock) - 1
```

**对比：**

| 方案 | 流程 | 吞吐量 |
|------|------|--------|
| 分布式锁 | 加锁→读→判断→扣减→解锁 | ~1-2 万 QPS（锁竞争） |
| Lua 原子操作 | 判断+扣减一步完成 | ~10 万 QPS（无等待） |

### 层5 — 订单处理环节才真正需要锁

你的实战经验是对的：**秒杀不是全链路都不用锁，是不同环节选不同方案。**

```
场景1：同一用户重复消费 MQ → Redis 锁防重复创建订单
场景2：同一商品库存确认 → 数据库乐观锁 / 唯一键
场景3：账户扣款 → TCC 冻结（本质也是锁）
```

**面试完整回答：**

> 秒杀扣库存不用分布式锁，用 Redis Lua 原子操作。但后续订单处理环节会用到——防重复下单用 Redis 锁，扣款用 TCC 冻结，库存最终确认用数据库乐观锁。不同环节选不同方案，不是一刀切。

## 四、Redis 锁 vs ZK 锁 — 不是选更好的，是选适合业务容忍度的

| 维度 | Redis 锁 | ZK 锁 |
|------|---------|-------|
| 一致性 | AP，极小概率并发 | CP，强一致 |
| 性能 | 10 万+ QPS | 1-2 万 QPS |
| 实现方式 | SET NX EX + 看门狗 | 临时顺序节点 + Watch |
| 锁获取 | 非公平，抢到就行 | 公平，按节点顺序排队 |
| 宕机恢复 | 锁可能丢失 | 锁不会丢失（CP 保证） |
| 适用场景 | 允许极小概率并发的业务 | 绝对不能并发的场景（选主） |

**架构师选择逻辑：**

```
业务能容忍极小概率并发吗？
├─ 能（大部分业务场景）→ Redis 锁，性能优先
│   └─ 但业务层必须做幂等兜底（Martin 的观点）
└─ 不能（选主、分布式锁）→ ZK 锁，一致性优先
    └─ 代价是吞吐量低，不适合高并发场景
```

## 架构师总结

| 层次 | 核心判断 |
|------|---------|
| 判断问题本质 | 互斥访问用锁，原子操作用原子命令，不要混用 |
| 选锁实现 | 允许极小概率并发 → Redis；绝对不能并发 → ZK |
| Redis 锁的边界 | 看门狗解决续期不解决断连；leaseTime 设了看门狗不启动；解锁必须 Lua |
| Redlock 的现实 | 理论上更安全，生产上很少用——成本高、仍不完美、不如业务幂等实在 |
| 秒杀的正确思路 | 分层：Lua 原子扣库存 + MQ 异步 + 订单处理环节才用锁 |
| 终极原则 | 分布式锁不是绝对互斥，业务层幂等是必须的兜底 |

**一句话：分布式锁的选择不是"哪个更好"，而是"业务能容忍什么"。架构师的职责是认清容忍度，选代价最小的方案，然后用幂等兜住那极小概率的并发窗口。**

## 和前两天的串联

```
Day 1 CAP → Day 2 分布式事务 → Day 3 分布式锁

CAP 选型决定了系统的一致性模型
  → CP 系统（ZK/etcd）适合做强一致锁
  → AP 系统（Redis）适合做高性能锁

  → 冻结金额 = 加锁（Try）
  → 确认扣款 = 释放锁（Confirm）
  → 解冻 = 释放锁（Cancel）

  → Day 2 的幂等三件套（唯一键 + 状态机 + 乐观锁）
  → 就是锁失效后的最后一道防线
```
