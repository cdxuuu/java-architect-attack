# Day 2：分布式事务 — 架构师视角梳理

## 问题全景

微服务拆分后，一次业务操作横跨多个服务、多个数据库，本地事务兜不住了。
架构师的核心判断：**选哪种方案，取决于业务能容忍什么。**

---

## 一、2PC — 为什么面试要问一个"没人用"的东西？

**问的不是你知不知道 2PC，而是你能不能说清"为什么不用"。**

2PC 的问题不是技术实现有缺陷，而是它的**前提假设和微服务架构冲突**：

| 2PC 的假设 | 微服务的现实 |
|-----------|------------|
| 事务持续时间短（ms~s） | 业务事务可能 15~30 分钟（订单超时） |
| 参与者可控、数量少 | 服务独立部署、迭代节奏不同 |
| 可以跨库加锁 | 服务自治，不允许跨库锁 |
| 协调者可靠 | 任何节点都可能挂 |

**业务关联：** 你做咖啡机对接时，用户下单到出咖啡可能几秒，但支付确认可能要等第三方回调，时间不可控。用 2PC 等于把所有资源锁住等回调，系统直接卡死。

**架构师视角：** 2PC 不是"不能用"，而是"不该在跨服务边界用"。同库内多表事务它仍然是最简单的方案。选技术方案要看边界——服务内用本地事务，服务间用分布式事务方案，这是架构上的分界线。

---

## 二、TCC vs Saga — 不是选哪个更好，是选哪个代价你付得起

### TCC — 你已经在用的模式

你的钱包+咖啡机场景就是标准 TCC：

```
Try:     冻结金额        ← 资源隔离，别人动不了
Confirm: 扣款            ← 咖啡机成功，冻结变真扣
Cancel:  解冻            ← 咖啡机失败，释放冻结
```

**TCC 的代价：**
- 每个服务写三个接口（Try/Confirm/Cancel），业务侵入大
- 空回滚 + 悬挂问题，必须用状态机防（你已经踩过了）
- 冻结资源有成本——冻了的钱不能被别的交易用，高并发下冻结池是瓶颈

**TCC 的价值：** 不一致窗口短且**业务不可见**。冻结期间用户看不到余额变化，体验干净。

### Saga — 没有冻结，直接做

```
T1: 扣款          ← 钱直接真扣了
T2: 咖啡机出咖啡

失败时：
C2: 标记失败
C1: 退款          ← 不是解冻，是往回加钱
```

**Saga 的代价：** 中间状态**业务可见**——扣了款但还没出咖啡，用户看到余额少了，可能重复下单。需要业务层用状态遮挡（订单=处理中，前端不展示余额变化）。

**Saga 的价值：** 侵入小，每个服务只写正向+补偿。业务流程长、参与者多时（电商下单涉及订单/库存/支付/积分/优惠券/物流），TCC 的三接口成本扛不住。

### 架构师的选择逻辑

```
业务对资金一致性要求高？
  ├─ 是 → TCC（冻结隔离，窗口不可见）
  └─ 否 → 参与者多吗？
           ├─ 多 → Saga（侵入小，好维护）
           └─ 少 → TCC 也可以，但不是必须
```

**关键认知：TCC 和 Saga 都是最终一致，不是强一致。** TCC 不是比 Saga "更一致"，而是不一致窗口更短、更不可见。面试时被追问"TCC 是强一致吗"，直接说不是——Confirm 阶段依然可能部分失败，靠重试达到最终一致。

| 维度 | TCC | Saga |
|------|-----|------|
| 一致性模型 | 最终一致 | 最终一致 |
| 不一致窗口 | 短，业务不可见（冻结态隔离） | 长，业务可见（中间态暴露） |
| 业务侵入 | 3 个接口 + 状态机防倒挂 | 2 个接口，侵入小 |
| 资源成本 | 冻结占用资源，高并发有瓶颈 | 无冻结，但需业务状态遮挡 |
| 典型业务 | 支付、钱包、资金类 | 电商下单、工单流转、长流程 |

## 三、Saga 编排方式 — 本质是"流程归谁管"

### 编排式（Orchestration）— 有指挥官

```
         ┌──────────┐
         │ 协调者    │
         └──────────┘
          │    │    │
          ▼    ▼    ▼
       订单  库存  支付
```

协调者持有全流程状态，决定下一步调谁。参与者不关心上下文，只暴露正向+补偿接口。

**好处：** 流程改了改协调者一个地方，参与者的代码不用动。协调者代码就是活文档，新人一看就知道业务全流程。

**代价：** 协调者是单点，必须持久化状态做故障恢复。

### 协调式（Choreography）— 事件链驱动

```
订单 ──事件──► 库存 ──事件──► 支付
  ▲                                │
  └──────────补偿事件──────────────┘
```

每个服务监听上游事件，自己决定下一步。没有中心节点。

**好处：** 没有单点，部署简单。

**代价：** 流程逻辑分散在各服务里。3 个服务还看得懂，7 个服务没人能拼出全流程。加一个步骤要改多个服务的监听逻辑，牵一发动全身。

### 架构师的选择

| 判断条件 | 选择 | 理由 |
|---------|------|------|
| 参与者 > 3 或流程经常变 | 编排式 | 流程集中管理，改一处 |
| 参与者少且稳定 | 协调式 | 简单，不引入额外组件 |
| 团队有 DDD 实践 | 编排式 | 协调者天然对应一个聚合根 |
| 跨团队协作 | 编排式 | 接口契约清晰，避免事件格式耦合 |

**业务关联：** 你的咖啡机场景只有钱包+咖啡机两步，协调式够了。但想想如果以后加积分抵扣、优惠券核销、出票通知——协调式的每个服务都要加事件监听，改一个流程改 N 个服务。这时候就该切编排式了。

**架构师原则：** 当前够用就简单做，但要知道扩展的拐点在哪。2-3 个参与者用协调式，超过就用编排式，这不是技术偏好，是维护成本的数学题。

## 四、异步消息可靠性 — 每一跳都可能丢，每一步都要防

异步支付是生产环境最常见的模式。你提到的三层防护是对的，架构师视角补全如下：

### 三个环节，三个防丢点

```
生产者                Broker              消费者
  │                     │                   │
  ├─ 事务消息            ├─ 同步刷盘          ├─ 手动提交
  │  Half Message       │  (防宕机丢)        │  (业务完再确认)
  ├─ 本地消息表          ├─ 主从同步复制      ├─ 失败重试
  │  (重试落库兜底)      │  (防单点丢)        │  (指数退避)
  └─ Job 补偿           └─                  ├─ 死信队列
     (极端情况最终兜底)                       ├─ (告警+运营后台)
                                             └─ 人工重试/补偿
```

**每一跳丢消息的后果和处理：**

| 丢在哪 | 后果 | 防线 |
|-------|------|------|
| 生产者发不到 Broker | 订单创建了，支付没触发 | 事务消息 + 本地消息表 + Job 轮询 |
| Broker 自身丢了 | 消息进了但宕机丢了 | 同步刷盘 + 主从同步复制 |
| 消费者处理失败 | 消息到了但业务没成功 | 手动提交 + 重试 + 死信队列 |

### 幂等 — 可靠性的副产品

可靠性做好了（重试、Job 补偿），必然产生重复消费。幂等不是可选项，是**必须项**。

```
幂等三件套：

1. 唯一键 / Redis SETNX     ← 去重，挡第一次之后的重复请求
2. 乐观锁（状态机）          ← 防并发覆盖，如 status=待支付 才能扣款
3. 两者配合                  ← 唯一键挡水平重复，状态机挡状态倒退
```

**业务关联：** 你的钱包系统里，同一笔扣款消息如果被消费两次，没有幂等就会扣两次钱。用 order_id 做唯一键 + 状态机（只允许 待支付→已支付），双重保险。

### 顺序 — 被忽略的第三问

"冻结成功"事件必须在"扣款"事件之前被消费。RocketMQ 的解法：

```
同一订单的消息 → 相同 orderId 做 hash → 路到同一 Queue → 单线程顺序消费
```

顺序反了会怎样？消费端看到"扣款"时冻结记录还不存在，幂等校验失败，消息进重试，延迟增加。

### 架构师视角的闭环

消息可靠性不只是技术问题，是**运维闭环**：

```
技术层：事务消息 + 刷盘 + 手动提交 + 幂等
    ↓ 依然可能出问题
运维层：死信队列 + 告警 + 运营后台（人工重试/补偿）
    ↓ 需要度量
度量层：消息积压监控、消费延迟监控、死信队列堆积量
```

**面试时把闭环说出来：** "技术方案做到 99.99%，剩下 0.01% 靠运维兜底——死信队列 + 告警 + 运营后台人工补偿，同时监控消息积压和消费延迟，确保问题能在 SLA 内被发现和处理。"

---

## 架构师总结

| 层次 | 核心判断 |
|------|---------|
| 选模型 | 2PC 不适合跨服务；TCC 适合资金类（冻结隔离）；Saga 适合长流程（侵入小） |
| 看一致 | TCC 和 Saga 都是最终一致，区别是不一致窗口的业务可见性 |
| 编流程 | 参与者少用协调式，多用编排式，判断标准是维护成本 |
| 保可靠 | 生产者防丢 + Broker 防丢 + 消费者防丢，每一跳都要防线 |
| 做幂等 | 可靠性的必然产物，唯一键 + 状态机双保险 |
| 闭环 | 技术方案 + 运维兜底 + 监控度量，三层缺一不可 |

**一句话：分布式事务没有银弹，架构师的职责是认清业务能容忍什么，选代价最小的方案，然后把 0.01% 的异常用运维闭环兜住。**
