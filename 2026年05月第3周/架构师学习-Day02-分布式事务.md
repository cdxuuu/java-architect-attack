# Day 2：分布式事务

## 题目

你的电商系统拆成了微服务：订单服务、库存服务、支付服务各自有独立数据库。

用户下单的流程是：
1. 订单服务创建订单
2. 库存服务扣减库存
3. 支付服务处理付款

问题来了：第 2 步库存扣减成功，但第 3 步支付超时失败。订单创建了、库存扣了、钱没付。

请回答：
1. 2PC 能解决这个问题吗？为什么微服务架构下几乎没人用 2PC？
2. TCC 和 Saga 是什么？用下单场景分别画出它们的补偿流程。
3. Saga 有两种编排方式：编排式（Orchestration）和协调式（Choreography），各有什么优劣？你会选哪种？
4. 如果支付不是同步调用，而是通过消息队列异步完成的，怎么保证"订单创建 → 库存扣减 → 支付完成"这个流程的可靠性？消息丢了怎么办？重复消费怎么办？

---

## 答案

### 1. 2PC 为什么不适合微服务？

2PC（两阶段提交）的流程：
- **Prepare**：协调者通知所有参与者准备提交，参与者写 Undo/Redo 日志并锁定资源
- **Commit/Rollback**：协调者根据所有参与者的反馈决定提交或回滚

**不适合微服务的原因：**

| 问题 | 说明 |
|------|------|
| 同步阻塞 | Prepare 阶段所有参与者锁住资源直到 Commit，持锁时间 = 最慢参与者的响应时间 |
| 单点故障 | 协调者挂了，所有参与者永久阻塞在 Prepare 状态 |
| 跨服务锁 | 微服务各自独立数据库，2PC 要求跨库加锁，违背服务自治原则 |
| 性能灾难 | 任何一步慢都会拖垮整条链路，高并发下吞吐量断崖式下降 |

> 一句话：2PC 本质是"大家一起等"，微服务的目标是"大家各干各的"，方向完全相反。

2PC 适合的场景：同库内多表事务（不需要跨网络），或者参与者很少且可控的内部系统。

### 2. TCC vs Saga 的补偿流程

**TCC（Try-Confirm-Cancel）：**

```
下单流程：
  Try:    订单服务→创建订单(状态:待确认)  库存服务→预扣库存  支付服务→冻结金额
  Confirm: 订单服务→订单状态→已确认       库存服务→扣减确认  支付服务→扣款确认
  Cancel:  订单服务→订单状态→已取消       库存服务→释放预扣  支付服务→解冻金额
```

支付超时的 Cancel 流程：
```
1. 支付服务 Cancel → 解冻金额
2. 库存服务 Cancel → 释放预扣库存
3. 订单服务 Cancel → 订单标记已取消
```

**Saga（正向+补偿）：**

```
下单流程（正向）：
  T1: 订单服务→创建订单
  T2: 库存服务→扣减库存
  T3: 支付服务→处理付款

支付超时的补偿流程（反向回滚）：
  C3: 支付服务→标记支付失败（幂等，可能本来就没成功）
  C2: 库存服务→归还库存
  C1: 订单服务→订单标记已取消
```

**核心区别：**

| 维度 | TCC | Saga |
|------|-----|------|
| 资源占用 | Try 阶段就冻结资源，Confirm/Cancel 释放 | 每步直接提交，补偿时反向操作 |
| 一致性 | Try 成功后 Confirm 几乎不会失败，一致性更强 | 中间状态对外可见，是最终一致 |
| 业务侵入 | 每个服务要写 Try/Confirm/Cancel 三个方法 | 每个服务要写正向 + 补偿两个方法 |
| 适用场景 | 资金类、一致性要求高 | 业务流程长、参与者多 |

> TCC 是"先占座再确认"，Saga 是"直接做，错了反着来"。

### 3. Saga 两种编排方式

**编排式（Orchestration）：**
```
                    ┌──────────┐
     ┌─────────────►│ Saga     │◄─────────────┐
     │              │ 协调者   │               │
     │              └──────────┘               │
  创建订单          指挥各服务              报告结果
     │              │    │    │               │
     ▼              ▼    ▼    ▼               ▼
  订单服务       库存服务  支付服务         通知服务
```

- 中心化：一个协调者（Orchestrator）决定下一步调谁
- 流程逻辑集中在协调者，参与者只需提供正向+补偿接口
- 协调者成为单点，需要持久化状态做故障恢复

**协调式（Choreography）：**
```
  订单服务 ──事件──► 库存服务 ──事件──► 支付服务
     ▲                                       │
     └─────────────补偿事件──────────────────┘
```

- 去中心化：每个服务监听事件，自己决定下一步
- 没有单点，但流程分散在各服务里，很难看清全貌
- 事件耦合严重，加一个步骤要改多个服务的监听逻辑

**选择建议：**

| 场景 | 推荐 | 原因 |
|------|------|------|
| 流程参与者 > 3 | 编排式 | 集中管理，不然没人看得懂全流程 |
| 流程经常变 | 编排式 | 改协调者一个地方，不用改 N 个服务 |
| 参与者少且稳定 | 协调式 | 简单，不需要额外组件 |
| 团队对流程可视化有要求 | 编排式 | 协调者就是活文档 |

> 面试建议：先说两种方式是什么，再根据场景选，别上来就站队。

### 4. 异步支付如何保证可靠性？

异步场景下的三个核心问题：消息丢了、重复消费、顺序错乱。

**① 消息丢了怎么办？**

**本地消息表方案：**
```
1. 订单服务在同一个事务中：创建订单 + 写入消息表（状态:待发送）
2. 后台线程轮询消息表，发送到 MQ
3. 发送成功 → 更新消息状态为已发送
4. 消费端（支付服务）消费成功 → 回调确认
5. 超时未确认 → 重新发送（幂等保证安全）
```

**更优方案：事务消息（RocketMQ 支持）：**
```
1. 发送半消息（对消费者不可见）
2. 本地事务执行
3. 根据本地事务结果：提交消息（可见）或回滚消息（删除）
4. 如果步骤 3 失败，Broker 回查本地事务状态
```

**② 重复消费怎么办？—— 幂等**

```
支付服务消费消息时：
1. 用 order_id 作为幂等键
2. 查询支付记录：已支付 → 直接返回成功（不重复扣款）
3. 未支付 → 执行支付逻辑
```

幂等实现方式：
- 数据库唯一键（order_id）
- Redis SETNX
- 状态机（只允许单向状态流转）

**③ 顺序问题**

订单创建必须在库存扣减之前。方案：
- 同一订单的事件路由到同一分区（相同 order_id → 同一 Queue/Partition）
- 单分区内保证顺序消费

**完整异步流程：**
```
订单服务                              库存服务              支付服务
   │                                    │                    │
   ├─ 创建订单 ──MQ──► 扣减库存 ──MQ──► 处理付款           │
   │  (事务消息)       (幂等消费)      (幂等消费)           │
   │                                    │                    │
   │              ◄── MQ ──────────────┘│                    │
   │              支付结果回调           │                    │
   │              (更新订单状态)          │                    │
```

## 今日要点

- 2PC 同步阻塞 + 单点故障，不适合微服务跨库事务
- TCC 先冻结再确认，一致性高但侵入大；Saga 直接提交再补偿，最终一致但侵入小
- Saga 编排式适合复杂流程，协调式适合简单场景
- 异步可靠性三板斧：本地消息表/事务消息（防丢）+ 幂等（防重）+ 分区顺序（防乱）
