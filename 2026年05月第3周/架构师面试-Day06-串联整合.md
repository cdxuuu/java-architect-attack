## 场景 1：钱包 + 咖啡机对接（TCC 分布式事务 + CAP 选型）

### 业务流程

用户用钱包余额购买咖啡：
```
1. 用户在咖啡机扫码下单
2. 钱包服务扣减余额
3. 咖啡机服务锁定咖啡库存
4. 咖啡机制作完成，订单完成
```

### CAP 选型决策

| 维度 | 选择 | 原因 |
|------|------|------|
| CAP 模型 | **AP 为主，关键流程 CP** | 咖啡机需要高可用（不能因为网络分区不接单），但钱包余额需要强一致 |
| 注册中心 | **etcd（CP）** | 服务发现需要强一致，避免服务调用到已下线节点 |
| 订单状态同步 | **异步 + 最终一致** | 咖啡机制作完成后异步回调更新订单状态 |

### TCC 实现

```
Try 阶段（资源冻结）：
  ├─ 钱包服务：冻结订单金额（余额 - 冻结金额 = 可用余额）
  ├─ 咖啡机服务：锁定对应咖啡库存
  └─ 订单服务：创建订单（状态：待确认）

Confirm 阶段（确认提交）：
  ├─ 钱包服务：扣减冻结金额
  ├─ 咖啡机服务：扣减库存
  └─ 订单服务：订单状态→已完成

Cancel 阶段（回滚）：
  ├─ 钱包服务：解冻金额
  ├─ 咖啡机服务：释放库存
  └─ 订单服务：订单状态→已取消
```

**为什么选 TCC 不选 Saga？**
- 钱包余额属于资金类操作，一致性要求高
- TCC 的 Try 阶段就冻结资源，中间状态不可见，避免用户看到余额扣了但咖啡没出的尴尬
- Saga 中间状态对外可见，用户可能投诉"我钱扣了，咖啡呢？"

## 场景 2：内部运营管理后台（SETNX 分布式锁 + ZK 选主）

### 业务场景

运营后台需要执行定时任务：
```
1. 每日凌晨 2 点生成对账单
2. 每 5 分钟同步咖啡机状态
3. 库存预警通知
```

### 分布式锁选型

| 场景 | 方案 | 原因 |
|------|------|------|
| 定时任务防重复执行 | **Redis SETNX + 过期时间** | 简单高效，定时任务短时持有锁，过期自动释放 |
| 报表生成服务选主 | **ZooKeeper（CP）** | 绝对不能两个节点同时生成报表，ZK 强一致更可靠 |

### SETNX 实现

```java
// 定时任务分布式锁
public boolean tryLock(String taskId, int expireSeconds) {
    String lockKey = "lock:task:" + taskId;
    String requestId = UUID.randomUUID().toString();
    
    // SET NX EX 原子操作
    Boolean success = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, requestId, expireSeconds, TimeUnit.SECONDS);
    
    if (success) {
        // 存储 requestId 用于解锁验证
        lockContext.set(lockKey, requestId);
        return true;
    }
    return false;
}

// 解锁用 Lua 脚本保证原子性
public void unlock(String lockKey) {
    String script = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                    "return redis.call('del', KEYS[1]) else return 0 end";
    
    String requestId = lockContext.get(lockKey);
    redisTemplate.execute(new DefaultRedisScript<>(script, Long.class), 
                         Collections.singletonList(lockKey), requestId);
}
```

**为什么不用 Redisson？**
- 内部后台 QPS 不高，SETNX 足够
- 运营团队不需要引入额外依赖
- 定时任务执行时间可预估，不需要看门狗续期

## 场景 3：ToC 核心服务（Redisson 分布式锁 + 两层缓存）

### 业务场景

用户访问小程序：
```
1. 查看附近咖啡机列表
2. 查看咖啡商品详情
3. 查看钱包余额
4. 下单购买
```

### 架构设计

| 组件 | 选型 | 原因 |
|------|------|------|
| 分布式锁 | **Redisson** | ToC 业务执行时间不可预估，需要看门狗续期 |
| 缓存架构 | **本地缓存（Caffeine）+ Redis** | 咖啡机列表、商品详情读多写少，本地缓存加速 |
| 余额查询 | **只查 Redis，不查本地缓存** | 余额一致性要求高，不能用本地缓存 |

### Redisson 看门狗应用

```java
// 下单操作，执行时间不可预估
public void placeOrder(Long userId, Long productId) {
    String lockKey = "lock:user:" + userId;
    RLock lock = redissonClient.getLock(lockKey);
    
    try {
        // leaseTime = -1，启动看门狗
        lock.lock(-1, TimeUnit.SECONDS);
        
        // 执行业务：扣余额、锁库存、创建订单
        // 可能耗时较长（调用咖啡机、调用支付）
        doPlaceOrder(userId, productId);
        
    } finally {
        lock.unlock(); // 释放锁，看门狗停止
    }
}
```

**看门狗在这里解决的问题：**
- 下单可能需要调用咖啡机（网络耗时）、调用支付（第三方耗时）
- 无法预估执行时间，设短了锁过期，设长了宕机恢复慢
- 看门狗自动续期，业务执行完主动释放

### 两层缓存设计

```
读咖啡商品详情：
  ├─ 查本地缓存（Caffeine，5s TTL）→ hit → 直接返回
  ├─ miss → 查 Redis → hit → 写本地缓存 + 返回
  └─ miss → 查 DB → 写 Redis + 写本地缓存 + 返回

更新咖啡商品信息：
  ├─ 更新 DB
  ├─ 删除 Redis 缓存
  └─ 通过 MQ PubSub 通知所有节点删除本地缓存
```

**为什么本地缓存设 5s TTL？**
- 商品详情变化不频繁，5s 不一致可接受
- 即使 PubSub 通知失败，最多 5s 自动恢复

## 场景 4：秒杀系统（缓存架构 + 原子操作，不用分布式锁）

### 业务场景

限量咖啡秒杀：
```
1. 周一早 8 点，1000 杯限量咖啡，10 万人同时抢购
2. 要求：不能超卖，系统不能崩
```

### 架构分层设计

```
层 1：前端限流
  ├─ 按钮置灰（秒杀开始前不可点）
  ├─ 滑动验证码（防脚本）
  └─ 拦截 90% 请求

层 2：网关限流
  ├─ 令牌桶算法，单机 QPS 限 1000
  └─ 再拦 90%

层 3：Redis 原子扣减（核心！不用分布式锁）
  ├─ Lua 脚本：判断库存 + 扣减一步完成
  └─ Redis 单线程天然串行

层 4：MQ 异步下单
  ├─ 扣减成功发 MQ
  └─ 订单服务异步消费创建订单

层 5：数据库兜底
  └─ 库存表乐观锁（UPDATE ... WHERE stock > 0）
```

### Redis Lua 原子扣减

```lua
-- KEYS[1]: 库存 Key
-- ARGV[1]: 扣减数量
local stock = redis.call("get", KEYS[1])
if not stock or tonumber(stock) < tonumber(ARGV[1]) then
    return -1  -- 库存不足
end
redis.call("decrby", KEYS[1], ARGV[1])
return tonumber(stock) - tonumber(ARGV[1])  -- 返回剩余库存
```

**为什么不用分布式锁？**
- 10 万 QPS 抢一把锁，锁本身就是瓶颈
- Redis 单线程执行 Lua，天然保证原子性
- 原子操作比锁快 100 倍以上

### 缓存防雪崩、防击穿、防穿透

| 问题 | 解法 |
|------|------|
| **缓存雪崩** | TTL 加随机值：`base + random(0~300s)`，避免同时过期 |
| **缓存击穿** | 热点 Key 永不过期，更新时主动覆盖 |
| **缓存穿透** | 布隆过滤器拦截不存在的商品 ID |

## 全链路串联：从 CAP 到实战

### 一个请求的旅程

```
用户下单买咖啡（秒杀场景）：

1. CAP 选型先定调
   ├─ 秒杀服务：AP（高可用优先，允许短暂不一致）
   ├─ 钱包服务：CP（余额强一致）
   └─ 注册中心：etcd（CP，服务发现强一致）

2. 请求到达网关
   ├─ 网关限流（令牌桶）
   └─ 布隆过滤器拦截无效请求

3. 查商品详情
   ├─ 本地缓存（Caffeine）→ 命中
   └─ 直接返回，不到 Redis

4. 库存扣减
   ├─ Redis Lua 原子操作（不用分布式锁）
   └─ 成功 → 发 MQ

5. 订单创建
   ├─ Redisson 分布式锁（用户级别，防止重复下单）
   ├─ 看门狗自动续期（调用支付可能耗时）
   ├─ TCC 分布式事务（钱包 + 咖啡机）
   └─ 订单完成

6. 缓存更新
   ├─ 更新 DB
   ├─ 删除 Redis 缓存
   └─ Canal 订阅 Binlog 异步删缓存兜底
```

## 今日要点

### 1. 不要为了用技术而用技术

| 场景 | 你可能想选 | 实际应该选 | 原因 |
|------|-----------|-----------|------|
| 秒杀扣库存 | 分布式锁 | Redis Lua 原子操作 | 锁是瓶颈，原子操作更快 |
| 内部定时任务 | Redisson | SETNX + 过期时间 | 简单够用，不需要复杂依赖 |
| 余额查询 | 本地缓存 + Redis | 只查 Redis | 一致性要求高，不能用本地缓存 |

### 2. CAP 不是纸上谈兵

- **CP**：etcd 注册中心、钱包余额、ZooKeeper 选主
- **AP**：秒杀服务、咖啡机状态同步、商品详情
- **没有 CA**：所有分布式系统都要面对分区问题

### 3. 分布式事务 vs 分布式锁 vs 缓存

| 技术 | 解决的问题 | 典型场景 |
|------|-----------|---------|
| 分布式事务 | 跨服务数据一致性 | 钱包 + 咖啡机对接 |
| 分布式锁 | 并发互斥 | 定时任务防重复、用户防重复下单 |
| 缓存 | 性能加速 | 商品详情、咖啡机列表 |

### 4. 架构师的决策思路

```
1. 先看业务场景（读多写少？一致性要求？QPS？）
2. 再选 CAP 模型（CP 还是 AP？）
3. 然后选技术方案（TCC 还是 Saga？Redisson 还是 SETNX？）
4. 最后考虑兜底方案（Binlog 异步删缓存、数据库乐观锁）
```
