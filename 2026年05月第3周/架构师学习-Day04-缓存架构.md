# Day 4：缓存架构

## 题目

你的电商系统做了 Redis 缓存加速，商品详情页的 QPS 从 500 涨到了 5 万。但高峰期开始出问题：

- 某个爆款商品详情页突然打不开，DB 被打满
- 一个不存在的商品 ID 被疯狂请求，缓存没拦住，全打到数据库
- 大促开始瞬间，大量缓存同时过期，数据库连接池瞬间耗尽

请回答：
1. 缓存和数据库的一致性怎么保证？先更新数据库再删缓存，还是先删缓存再更新数据库？为什么？
2. 缓存穿透、击穿、雪崩分别是什么？各自的解法是什么？它们之间的区别是什么？
3. 热点 Key 问题怎么发现、怎么处理？如果某个 Key 的 QPS 是 10 万，单节点 Redis 扛不住怎么办？
4. 你的系统用了本地缓存（Caffeine）+ 分布式缓存（Redis）的两层架构，这两层缓存的一致性怎么保证？什么场景下该用本地缓存，什么场景下只用 Redis？

---

## 答案

### 1. 缓存与数据库一致性

**两种策略对比：**

| 策略 | 流程 | 问题 |
|------|------|------|
| 先删缓存再更新DB | 1. DEL cache 2. UPDATE db | 并发读把旧值写回缓存 |
| 先更新DB再删缓存 | 1. UPDATE db 2. DEL cache | 极小概率读到旧缓存 |

**先删缓存的问题（并发读回写）：**

```
线程A: DEL cache
线程B: READ cache → miss → READ db(旧值) → SET cache(旧值)
线程A: UPDATE db
→ 缓存里是旧值，数据库是新值，不一致直到缓存过期
```

**先更新DB再删缓存的问题（极小概率不一致）：**

```
线程A: READ cache → miss → READ db(旧值)
线程B: UPDATE db → DEL cache
线程A: SET cache(旧值)
→ 缓存里是旧值，但这个窗口极小（读DB+SET cache 必须比 UPDATE+DEL 更慢才会发生）
```

**结论：先更新DB再删缓存是主流选择。** 因为它的不一致窗口极小，而先删缓存的并发回写问题在高并发下很容易发生。

**但先更新DB再删缓存仍然可能失败（删缓存失败怎么办）：**

| 方案 | 做法 | 代价 |
|------|------|------|
| 延迟双删 | 更新DB → 删缓存 → 延迟N毫秒 → 再删缓存 | 延迟时间不好估计，治标不治本 |
| 消息队列重试 | 删缓存失败 → 发MQ → 消费者重试删除 | 引入MQ，业务侵入 |
| 订阅Binlog | Canal监听MySQL Binlog → 异步删缓存 | 架构复杂，但最可靠 |

**架构师选择：**

```
一致性要求一般 → 先更新DB再删缓存 + 短TTL兜底
一致性要求高   → 先更新DB再删缓存 + Canal订阅Binlog异步删
允许短暂不一致 → 先更新DB再删缓存 + 延迟双删（简单但有窗口）
```

### 2. 缓存穿透、击穿、雪崩

**缓存穿透 — 请求的数据根本不存在**

```
请求商品ID=999999 → cache miss → DB miss → 返回空
→ 恶意请求大量不存在的ID → 全部穿透到DB
```

| 解法 | 做法 | 优劣 |
|------|------|------|
| 缓存空值 | 查不到也缓存，设短TTL（60s） | 简单，但浪费内存（攻击者换ID就失效） |
| 布隆过滤器 | 请求前先过BF，不存在直接拒绝 | 内存省，但有误判率（哈希冲突） |
| 请求入口校验 | ID格式/范围校验，非法请求直接拒 | 最前置，防恶意请求 |

**缓存击穿 — 热点Key过期瞬间**

```
爆款商品缓存过期 → 大量并发请求同时miss → 全部打到DB
→ 单个Key的过期瞬间流量洪峰
```

| 解法 | 做法 | 优劣 |
|------|------|------|
| 互斥锁 | 缓存miss时用SET NX抢锁，只有拿到锁的线程查DB | 简单，但锁是瓶颈 |
| 逻辑过期 | 缓存不设TTL，存逻辑过期时间，过期后异步更新 | 无阻塞，但可能返回旧数据 |
| 热点永不过期 | 热点Key不设过期时间，更新时主动覆盖 | 最简单，但需要主动更新机制 |

**缓存雪崩 — 大量Key同时过期**

```
大批缓存同一时间设了相同TTL → 同时过期 → DB瞬间被打满
→ 不是单个Key的问题，是一批Key同时失效
```

| 解法 | 做法 | 优劣 |
|------|------|------|
| TTL加随机值 | TTL = base + random(0~300s) | 最简单有效，打散过期时间 |
| 多级缓存 | L1本地缓存 + L2 Redis + L3 DB | 层层过滤，但一致性更复杂 |
| 熔断降级 | DB连接池耗尽时返回兜底数据 | 保系统不死，但业务受损 |

**三者区别：**

| | 穿透 | 击穿 | 雪崩 |
|---|------|------|------|
| 本质 | 数据不存在 | 热点Key过期 | 大量Key同时过期 |
| 触发 | 恶意请求/bug | 爆款商品缓存到期 | 批量缓存同时失效 |
| 影响 | 每次都穿透 | 一瞬间 | 一瞬间 |
| 核心解法 | 布隆过滤器 | 互斥锁/逻辑过期 | TTL随机化 |

### 3. 热点Key

**发现问题：**

| 方法 | 做法 |
|------|------|
| Redis MONITOR | 实时监控命令（生产慎用，消耗性能） |
| BIGKEY扫描 | redis-cli --hotkeys（Redis 4.0+） |
| 客户端统计 | 在应用层统计Key访问频次 |
| 代理层统计 | Twemproxy/Codis 等代理层聚合统计 |

**处理热点Key：**

| 方案 | 做法 | 适用场景 |
|------|------|---------|
| 本地缓存 | 热点数据放JVM缓存（Caffeine） | 读多写少，容忍短暂不一致 |
| 读写分离 | 热点Key的读分散到从节点 | 读多写少 |
| 分片拆分 | hot_key → hot_key_1, hot_key_2... 取模分散 | 极高QPS单Key |
| 集群扩容 | 增加Redis节点 | 整体热点多 |

**QPS 10万的单Key：**

```
单Redis节点QPS约10万 → 临界值，一个节点扛不住

方案1：本地缓存（最优先）
  → 90%请求在应用层拦截，不到Redis
  → Caffeine 设 5s 过期，5s内一致性可接受

方案2：Key拆分
  → product:123 → product:123:1 ~ product:123:9
  → 读时随机选一个，写时全量更新
  → 10个Key分散到不同节点，每个扛1万

方案3：读写分离 + 集群
  → 从节点分担读流量
  → 需要评估从节点延迟
```

### 4. 本地缓存 + 分布式缓存的两层架构

**一致性保证：**

```
写操作：
  1. 更新DB
  2. 删Redis
  3. 删本地缓存（或等TTL过期）

读操作：
  1. 查本地缓存 → hit → 返回
  2. miss → 查Redis → hit → 写本地缓存 + 返回
  3. miss → 查DB → 写Redis + 写本地缓存 + 返回
```

**两层缓存不一致的根源：** 应用节点A更新了数据，删了自己本地缓存和Redis，但节点B的本地缓存还是旧值。

| 解法 | 做法 | 代价 |
|------|------|------|
| 短TTL | 本地缓存TTL设5~10s | 5~10s不一致窗口 |
| 广播通知 | 更新时通过MQ/PubSub通知所有节点删本地缓存 | 实时性好，但增加复杂度 |
| 版本号 | 本地缓存存版本号，对比Redis版本号 | 多一次Redis查询 |

**什么场景用本地缓存：**

| 场景 | 选择 | 原因 |
|------|------|------|
| 字典/配置数据 | 本地缓存 | 几乎不变，TTL可以很长 |
| 用户session | Redis | 多节点共享，本地缓存不一致 |
| 热点商品详情 | 本地 + Redis | 读多写少，容忍短时不一致 |
| 实时库存 | 只用Redis | 一致性要求高，不能用本地缓存 |
| 排行榜 | 只用Redis | 变化频繁，需要全局一致 |

**架构师原则：**

```
一致性要求高 → 只用Redis，不用本地缓存
读多写少 + 容忍秒级不一致 → 本地缓存 + Redis
几乎不变的数据 → 本地缓存为主
```

---

## 今日要点

- 缓存一致性：先更新DB再删缓存（不一致窗口最小）；删缓存失败靠Binlog异步补偿
- 穿透（数据不存在）→ 布隆过滤器；击穿（热点过期）→ 互斥锁/逻辑过期；雪崩（批量过期）→ TTL随机化
- 热点Key：本地缓存优先拦截 + Key拆分分散流量 + 读写分离
- 两层缓存：本地缓存适合读多写少、容忍秒级不一致的场景；实时性要求高的只用Redis
