# Day 4：可观测性与微服务进阶 — 架构师角度梳理

## 核心定位

Day4的主题可以一句话概括：**从"出了问题能发现"到"出了问题能快速定位"，再到"出问题前就有防御"**。可观测性解决发现问题，分布式ID/部署策略/Service Mesh/安全解决不同维度的工程问题，但共同点是——架构师要能说清"为什么选这个方案"。

---

## 可观测性：不是装三个组件，是定位故障的SOP

```
三位一体的核心逻辑：

  Metrics → what（系统出了什么问题）
  Tracing → where（卡在哪个服务）
  Logging → why（根因是什么）

  面试时不要只说"我们用Prometheus+SkyWalking+ELK"
  要说"监控发现错误率飙升 → 链路追踪定位到库存服务 → 日志确认是慢查询锁表"
  这才是架构师的表达方式——用流程说话，不是用组件清单说话
```

### Metrics告警设计的本质

```
告警不是监控的附属品，是可观测性到行动的桥梁

  好的告警：
    "库存服务错误率从0.1%升至10%，影响订单下单，可能原因：MySQL慢查询"
    → 有数据(10%) + 有影响(订单下单) + 有方向(慢查询)

  差的告警：
    "inventory_error_rate > 0.05"
    → 没有影响面、没有方向、值班不知道该不该处理

  架构师设计告警的标准：
    每条告警必须回答三个问题：
      1. 什么出了问题？（指标+阈值+当前值）
      2. 影响了什么业务？（关联的核心链路）
      3. 可能是什么原因？（常见根因方向）
```

## 分布式ID：选型看两个维度

```
两个维度决定方案：
  1. 性能要求：高QPS需要本地生成，低QPS可以依赖外部
  2. 递增要求：严格递增还是趋势递增

  面试时不要只说"我们用Snowflake"
  要说"订单号用Leaf号段因为需要趋势递增+高可用，
       支付流水用数据库自增因为金融合规要求严格递增，
       TraceId用UUID因为不需要递增"
```

### Snowflake的工程细节

```
面试追问的三层防线：

  L1：64位结构？
    1位符号 + 41位时间戳(69年) + 10位机器(1024台) + 12位序列(4096/ms)

  L2：时钟回拨？
    <5ms等待 / <100ms续编 / >100ms拒绝+告警 / Leaf号段兜底

  L3：为什么不用UUID？
    UUID无序 → 索引性能差（B+树页分裂）
    Snowflake趋势递增 → 索引友好
    但UUID适合TraceId/临时标识等非DB主键场景
```

### 号段模式的精妙设计

```
号段模式的核心不是"从DB取一段ID"，而是"双Buffer预加载"

  单Buffer的问题：
    号段用完 → 去DB取新号段 → DB慢/挂了 → 发号中断
    依赖DB的可用性 → 不是真正的高可用

  双Buffer的解法：
    Buffer1(当前使用): [1, 1000]  ← 正在发号
    Buffer2(预加载):   [1001, 2000] ← 已提前加载好

    Buffer1消耗到20%时 → 异步加载Buffer2
    Buffer1用完 → 无缝切换到Buffer2
    切换过程零延迟

    DB挂了？
    Buffer2里有1000个号 → 按QPS 1000算 → 能撑1秒
    号段步长设10000 → 能撑10秒
    步长设100000 → 能撑100秒
    架构师根据DB恢复时间（通常5-10分钟）调整步长
```

## 部署策略：不是选一个，是按风险等级组合

```
部署策略和Day2降级三层架构思维一脉相承：

  Day2降级：
    L1自动(覆盖90%) → L2配置(覆盖9%) → L3人工(覆盖1%)

  Day4部署：
    日常金丝雀(90%的发布) → 紧急滚动(9%的发布) → 重大变更流量镜像(1%的发布)

  共同思维：分层+按场景选策略，不是一刀切
```

### 有状态服务部署：扩展-收缩模式

```
数据库变更是最容易出事故的部署环节

  核心原则：永远不要一步完成破坏性变更

  扩展阶段（加新）：
    加新列 → 代码双写新旧列 → 迁移老数据 → 代码改读新列
    所有步骤都是加法，不会丢数据

  收缩阶段（删旧）：
    确认新列数据完整 → 停写旧列 → 观察一周 → 删旧列
    每一步都有回退空间

  和Day2熔断器的状态机思维一致：
    CLOSED → HALF_OPEN → OPEN
    每一步都有明确的进入条件和回退路径
```

## Service Mesh：基础设施下沉

```
Service Mesh的选型本质：治理能力放哪一层？

  SDK层（Dubbo/Spring Cloud）：
    优点：零额外延迟、功能丰富（Sentinel热点参数限流）
    缺点：语言绑定、升级困难、业务耦合

  Sidecar层（Istio + Envoy）：
    优点：语言无关、业务零侵入、统一管控
    缺点：+1-3ms延迟、资源开销、运维复杂

  架构师决策：不是二选一，是看场景
    Java为主的团队 → SDK够用，别为了Mesh而Mesh
    多语言团队 → Mesh是刚需
    渐进式 → 先SDK后Mesh，混合过渡

  和Day3 gRPC vs RESTful的选型逻辑一致：
    不是"哪个更好"，是"哪个更适合当前场景"
```

### Service Mesh和前序知识的映射

```
每个Day2-Day3的能力在Service Mesh中都有对应实现：

  Day2能力              SDK实现              Mesh实现
  ─────────────────────────────────────────────────────
  限流                  Sentinel FlowRule    EnvoyFilter+RateLimit
  熔断                  Sentinel DegradeRule DestinationRule+OutlierDetection
  降级                  @SentinelResource    VirtualService重写
  隔离                  信号量/线程池         Sidecar天然隔离(独立进程)

  Day3能力              SDK实现              Mesh实现
  ─────────────────────────────────────────────────────
  链路追踪              SkyWalking Agent     Envoy自动注入Header
  灰度路由              Nacos元数据+LoadBalancer VirtualService权重路由
  配置中心              Nacos Config         istiod xDS推送
  认证鉴权              Spring Security      Istio RBAC/ABAC

  核心差异：实现位置变了（进程内→Sidecar），能力没变
  架构师要理解的是能力本身（限流/熔断/降级），不是实现方式
```

## 微服务安全：四层模型

```
安全的本质：从不信任到逐步验证

  L1 网关：你谁？（认证）
  L2 服务间：你真的是你吗？（mTLS）
  L3 鉴权：你能做什么？（RBAC/ABAC）
  L4 数据：你能看什么？（脱敏/加密/隔离）

  每一层解决一个问题，四层叠加=纵深防御
  和Day2五层漏斗的思路一致：每一层挡一类攻击
```

### JWT的工程取舍

```
JWT的选择不是"好不好"的问题，是"你能不能接受它的缺点"

  JWT的三个硬伤：
    1. 无法主动踢人 → 短有效期15分钟+版本号校验
    2. Payload明文 → 只放userId+role，不放敏感数据
    3. Token体积大 → 1KB vs SessionId几十字节

  什么时候用Session：
    需要主动踢人（后台管理系统）
    Token传输带宽敏感（移动端弱网）
    不想处理RefreshToken逻辑

  什么时候用JWT：
    微服务网关验证（无状态=网关不依赖Redis）
    跨域SSO（多系统单点登录）
    移动端App（没有Cookie）

  架构师决策：
    对外API → JWT（无状态+跨域）
    内部管理系统 → Session（主动踢人+简单）
    两者可以共存，不是二选一
```
