# Day 6：微服务主题串联整合 — 从拆分到实战的完整闭环

## 场景一：秒杀系统全链路微服务设计 — 五层漏斗+三层限流+分布式事务

### 场景描述
```
秒杀系统，10万QPS，5秒售罄

涉及环节：
  1. 网关入口：认证+限流+路由
  2. 秒杀服务：Redis预扣+售罄标记
  3. 订单服务：创建订单
  4. 库存服务：扣减库存
  5. 支付服务：支付处理

要求：
  - 10万QPS不能打挂任何服务
  - 库存不能超卖
  - 订单不能丢失
  - 5秒售罄后剩余流量零DB压力
```

### 架构师方案：五层漏斗 + 分布式事务 + 全链路保障
```
                        10万QPS
                           │
                    ┌──────▼──────┐
                    │  L1 CDN/Nginx│  挡90%静态请求
                    │  静态资源缓存│  → 剩1万QPS
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  L2 Gateway │  令牌桶限流1万QPS
                    │  JWT认证    │  → 令牌桶允许短暂突发
                    │  三层限流   │  单IP滑动窗口100 QPS/min
                    └──────┬──────┘  → 剩5000 QPS
                           │
                    ┌──────▼──────┐
                    │  L3 秒杀服务│  Redis预扣+售罄标记
                    │  Lua原子扣减│  → 扣到0设soldout标记
                    │  售罄标记   │  → 售罄后直接返回，0 DB压力
                    └──────┬──────┘  → 剩500 QPS（有效购买）
                           │
                    ┌──────▼──────┐
                    │  L4 熔断降级│  Sentinel熔断保护
                    │  服务层限流 │  滑动窗口多维度统计
                    │  热点参数   │  单商品ID限流100 QPS
                    └──────┬──────┘  → 剩300 QPS
                           │
                    ┌──────▼──────┐
                    │  L5 DB兜底  │  最终一致性写DB
                    │  消息异步化 │  MQ削峰 → 100 TPS消费
                    └─────────────┘

  分布式事务设计（订单→库存→支付）：
  ┌──────────────────────────────────────────────────────┐
  │  方案选择：Saga编排（不是TCC）                         │
  │                                                       │
  │  为什么不用TCC？                                      │
  │    秒杀场景不需要Try阶段冻结——库存扣到0直接售罄标记   │
  │    允许短暂脏读——库存多扣后补偿恢复，用户无感        │
  │    TCC开发成本4x，秒杀不需要强隔离                    │
  │                                                       │
  │  Saga正向流程：                                       │
  │  创建订单(T1)→扣库存(T2)→支付(T3)                    │
  │                                                       │
  │  Saga补偿流程（支付失败）：                            │
  │  取消订单(C1)←恢复库存(C2)←退款(C3)                  │
  │                                                       │
  │  关键：库存用Redis预扣+Lua原子操作保证不超卖          │
  │  Saga补偿只处理业务层面的回退，不处理并发安全问题     │
  └──────────────────────────────────────────────────────┘
```

### 本周知识点串联
| 知识点 | 在这个场景中的应用 |
| --- | --- |
| **拆分策略（Day1）** | 按业务能力拆分：秒杀/订单/库存/支付独立服务，各自独立扩缩容 |
| **注册发现（Day1）** | Nacos AP模式，秒杀服务多实例注册，网关动态路由 |
| **五层漏斗（Day2）** | CDN→网关→Redis预扣→熔断降级→DB，每层挡90%+ |
| **三层限流（Day5）** | 网关令牌桶→服务滑动窗口→接口热点参数，每层算法不同根因不同 |
| **售罄标记（Day2）** | Redis soldout标记，5秒售罄后剩余55秒DB压力归零 |
| **超时重试熔断铁三角（Day3）** | 网关30s>服务10s>下游3s×2次，熔断OPEN不重试直接降级 |
| **分布式事务（Day5）** | 秒杀用Saga不用TCC，一致性够用就行 |
| **容量规划（Day5）** | 四步法：压测基线→流量预估→容量计算→弹性扩缩 |
| **故障演练（Day5）** | 大促前全链路压测+故障注入，验证五层漏斗有效性 |

### 架构师思考：秒杀系统为什么是微服务架构的最佳验证场景？
```
秒杀系统集中了微服务架构的所有核心挑战：

  1. 高并发 → 限流/熔断/降级/隔离（Day2）
  2. 分布式 → 事务选型/服务通信/注册发现（Day1/Day3/Day5）
  3. 可观测 → 链路追踪/监控告警/故障定位（Day4）
  4. 弹性 → 容量规划/自动扩缩/成本优化（Day5）
  5. 安全 → 网关认证/防刷/热点限流（Day4/Day5）

  面试核心句：
  "秒杀是微服务架构的试金石——如果五层漏斗、三层限流、分布式事务、
   全链路压测都搞不定，说明微服务治理体系还没建好"
```

## 场景二：保险商城微服务化改造 — 从单体到微服务的渐进式演进

### 场景描述
```
平安健康险保险商城，当前是单体应用

当前痛点：
  1. 投保流程3-5秒，用户体验差
  2. 核保接口变更频繁，影响整个系统发布
  3. 保费计算CPU密集，影响其他模块性能
  4. 团队15人，各模块耦合严重，互相等发布

目标：
  渐进式拆分，不影响现有业务运行
```

### 架构师方案：三阶段演进 + Day1-Day5知识全应用
```
阶段1：单体→服务化（0-6个月）— 先拆最痛的

  ┌──────────────────────────────────────────────────────┐
  │  拆分顺序（Day1拆分策略+Day5演进路线）：              │
  │                                                       │
  │  第1个拆：核保服务                                    │
  │    理由：变更最频繁（保险产品经常调整核保规则）       │
  │    方式：Spring Cloud + Nacos注册                     │
  │                                                       │
  │  第2个拆：保费计算服务                                │
  │    理由：CPU密集，影响其他模块（资源需求最大）        │
  │    方式：独立部署，可单独扩容                         │
  │                                                       │
  │  先不拆的：保单管理、用户管理（变更少、耦合高）       │
  │                                                       │
  │  基建同步建设（Day3基础设施优先级）：                  │
  │    1. Nacos注册+配置中心 — 没有就别拆                 │
  │    2. Gateway网关 — 统一入口                          │
  │    3. SkyWalking链路追踪 — 没有排障就是盲人           │
  └──────────────────────────────────────────────────────┘

阶段2：服务化→微服务化（6-18个月）— 补治理体系

  ┌──────────────────────────────────────────────────────┐
  │  Day2-Day4的治理能力全部要补上：                      │
  │                                                       │
  │  限流熔断（Day2）：                                   │
  │    保费计算服务 → Sentinel慢调用比例熔断              │
  │    为什么？保费计算经常慢查询，不是直接挂             │
  │    配置：RT阈值500ms，比例50%，timeWindow 10s         │
  │                                                       │
  │  降级策略（Day2）：                                   │
  │    保费计算降级 → 用缓存报价（允许过期）              │
  │    核保降级 → 异步化+返回排队凭证（允许延迟）         │
  │    决策矩阵：实时性×数据过期容忍度                    │
  │                                                       │
  │  可观测性（Day4）：                                   │
  │    Metrics→Tracing→Logging三位一体                    │
  │    保费计算RT P99 > 3s → 告警                        │
  │    核保服务错误率 > 1% → 告警                         │
  │                                                       │
  │  安全（Day4）：                                       │
  │    网关JWT认证 + 服务间mTLS + RBAC鉴权               │
  │    保单数据脱敏（身份证/手机号/健康信息）             │
  └──────────────────────────────────────────────────────┘

阶段3：微服务→云原生（18-36个月）— 自动化+弹性

  ┌──────────────────────────────────────────────────────┐
  │  容量规划（Day5四步法）：                              │
  │                                                       │
  │  基线：保费计算单机QPS 500，安全水位400               │
  │  预估：投保高峰QPS 2000（日常4倍）                    │
  │  计算：2000/400×1.3 ≈ 7台                             │
  │  扩容：HPA自动扩缩，日常3台→高峰7台                   │
  │                                                       │
  │  部署策略（Day4）：                                    │
  │    日常金丝雀(1%→5%→10%→100%)                        │
  │    核保规则变更 → 金丝雀验证 → 全量发布               │
  │                                                       │
  │  故障演练（Day5）：                                    │
  │    每月1次：kill保费计算实例，验证熔断降级             │
  │    每季度1次：全链路故障注入，验证应急预案             │
  └──────────────────────────────────────────────────────┘
```

### 架构师思考：保险商城为什么要分阶段拆，不能一步到位？
```
一步到位的反模式（Day5四大反模式之一）：

  ❌ 一次性拆10个微服务
     结果：没有基建支撑，排障靠猜，服务间调用混乱

  ✅ 先拆1-2个最痛的 → 补基建 → 验证有效 → 再拆下一个

  保险商城的分阶段策略：
    Phase1 拆核保：验证注册发现+网关+链路追踪是否就绪
    Phase2 拆保费计算：验证熔断降级+容量规划是否有效
    Phase3 拆支付：验证分布式事务+安全四层模型是否可靠

    每拆一个服务，都是对治理体系的验证
    前一个验证不通过，不拆下一个

  面试核心句：
  "保险商城的微服务化不是一步到位，是分阶段验证。
   每拆一个服务都是对治理体系的压力测试，
   验证不通过就先补基建，不急着拆下一个"
```

## 场景三：医疗在线问诊高可用架构 — 容错设计+链路追踪+故障应急

### 场景描述
```
医疗在线问诊平台，核心链路：患者发起问诊→匹配医生→建立会话→消息收发

关键要求：
  1. 问诊不能断（患者和医生在沟通中，服务挂了=断诊）
  2. 匹配要快（患者等3秒以上就流失）
  3. 消息不能丢（医嘱、处方是关键医疗数据）
  4. 高峰期：早8-10点、晚7-9点，QPS是平时的5倍

依赖的第三方：
  - IM推送服务（自研，但和问诊是不同团队）
  - 处方服务（医院HIS系统对接，稳定性不可控）
  - 支付服务（问诊付费）
```

### 架构师方案：容错+可观测+应急三位一体
```
服务拆分（Day1）：
  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 问诊服务 │  │ 匹配服务 │  │ 会话服务 │  │ 处方服务 │
  │ (核心)   │  │ (核心)   │  │ (核心)   │  │ (非核心) │
  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
       │              │              │              │
       └──────────────┴──────────────┴──────────────┘
                         │
                    ┌────▼─────┐
                    │ IM推送   │ ← 不同团队，当第三方处理
                    │ 服务     │
                    └──────────┘

容错设计（Day2）— 隔离策略：
  ┌──────────────────────────────────────────────────────┐
  │  处方服务（HIS对接，最不稳定）                        │
  │                                                       │
  │  隔离方案：线程池隔离 + 超时控制                      │
  │  CompletableFuture.supplyAsync(调用HIS, hisPool)      │
  │    .orTimeout(5, SECONDS)                             │
  │    .exceptionally(ex -> 降级返回)                     │
  │                                                       │
  │  为什么线程池隔离不用信号量？                          │
  │    HIS是外部系统 → 完全不可控 → 需要彻底隔离         │
  │    信号量只限制并发，不隔离线程 → HIS慢会拖Tomcat    │
  │                                                       │
  │  熔断配置：                                           │
  │    策略：异常比例（HIS经常直接报错，不是慢）          │
  │    比例：50%，最小请求：5，熔断时间：30s              │
  │    为什么30s？HIS故障恢复通常需要30秒以上             │
  │                                                       │
  │  降级方案：                                           │
  │    处方→返回"处方生成中，稍后查看"                    │
  │    决策矩阵定位：低实时+不可过期 → 异步化             │
  │    异步化：处方请求写入MQ → HIS恢复后补开             │
  ├──────────────────────────────────────────────────────┤
  │  IM推送服务（自研但不同团队）                         │
  │                                                       │
  │  隔离方案：信号量隔离（Day2）                         │
  │  为什么信号量不用线程池？                              │
  │    IM推送是内部服务 → 可控性高 → 信号量轻量够用      │
  │    并发限制50 → 超出排队而非新建线程                  │
  │                                                       │
  │  熔断配置：                                           │
  │    策略：慢调用比例（IM经常是慢不是挂）               │
  │    RT阈值：200ms，比例：60%，熔断时间：10s            │
  │                                                       │
  │  降级方案：                                           │
  │    IM推送失败 → 本地消息表兜底（Day5消息最终一致性）  │
  │    定时任务扫描未推送消息 → 重试推送                  │
  └──────────────────────────────────────────────────────┘

链路追踪（Day3/Day4）— 排障核心：
  ┌──────────────────────────────────────────────────────┐
  │  三种传播场景（Day3）：                                │
  │                                                       │
  │  HTTP传播：网关→问诊服务→匹配服务                     │
  │    → W3C TraceContext Header                          │
  │                                                       │
  │  MQ传播：问诊服务→MQ→处方服务                         │
  │    → Message Property携带TraceId                      │
  │                                                       │
  │  异步线程：问诊服务内线程池调用IM                      │
  │    → TransmittableThreadLocal传递                     │
  │                                                       │
  │  采样策略：头部1% + 尾部100%（异常必采）              │
  │  为什么？正常流量1%够分析，异常100%不丢排障线索       │
  └──────────────────────────────────────────────────────┘

故障应急（Day5）— 5步SOP：
  ┌──────────────────────────────────────────────────────┐
  │  场景：HIS处方服务宕机                                │
  │                                                       │
  │  Step1 止血（0-5min）：                               │
  │    决策树：                                           │
  │      能回滚？→ HIS是外部系统，回滚不了                │
  │      能降级？→ YES！处方异步化，返回"生成中"          │
  │    止血动作：熔断处方调用 → 异步化降级                │
  │                                                       │
  │  Step2 通告（5-10min）：                               │
  │    故障群：处方服务降级，问诊不受影响                  │
  │                                                       │
  │  Step3 定位（10-20min）：                              │
  │    Metrics→处方服务错误率飙升                          │
  │    Tracing→定位到HIS接口超时                           │
  │    Logging→HIS返回Connection Refused                  │
  │                                                       │
  │  Step4 修复（20-30min）：                              │
  │    联系HIS团队确认恢复 → 关闭熔断 → 验证正常         │
  │                                                       │
  │  Step5 复盘（24-48h）：                                │
  │    5Whys：为什么HIS挂了→为什么没有更早发现→改进措施  │
  └──────────────────────────────────────────────────────┘
```

### 本周知识点串联
| 知识点 | 在这个场景中的应用 |
| --- | --- |
| **服务拆分（Day1）** | 问诊/匹配/会话/处方按业务能力拆分，处方非核心可降级 |
| **线程池vs信号量隔离（Day2）** | HIS外部→线程池隔离彻底隔离，IM内部→信号量轻量够用 |
| **熔断策略匹配（Day2）** | HIS→异常比例（经常报错），IM→慢调用比例（经常慢） |
| **降级决策矩阵（Day2）** | 处方→异步化（低实时+不可过期），IM→本地消息表兜底 |
| **超时控制（Day2/Day3）** | HIS 5s超时+CompletableFuture.orTimeout，灵魂是超时 |
| **链路追踪传播（Day3）** | HTTP Header/MQ Property/TTL三种场景三种方案 |
| **采样策略（Day3）** | 头部1%+尾部100%，异常必采不丢排障线索 |
| **可观测性SOP（Day4）** | Metrics发现→Tracing定位→Logging确认根因 |
| **消息最终一致性（Day5）** | IM推送失败→本地消息表→异步补推 |
| **故障5步SOP（Day5）** | 止血→通告→定位→修复→复盘，5分钟MTTR |

### 架构师思考：医疗问诊的核心设计原则是什么？
```
医疗问诊的核心原则：问诊不能断

  这和电商的"宁可慢也不能错"不同：
    电商：数据一致性 > 可用性（超卖比宕机更严重）
    医疗：可用性 > 数据一致性（断诊比数据延迟更严重）

  体现在架构上：
    1. 核心链路（问诊+匹配+会话）→ AP优先，可用性第一
    2. 非核心链路（处方）→ 异步化降级，不影响问诊
    3. 消息可靠性 → 本地消息表兜底，不丢医嘱
    4. 容错设计 → 第三方调用必须隔离+超时+降级三件套

  面试核心句：
  "医疗问诊的核心是'不能断诊'，所以核心链路AP优先，
   非核心链路全部异步化降级，第三方调用必须隔离+超时+降级三件套。
   降级方案不是'返回错误'，是'异步处理+稍后通知'，用户无感"
```

## 场景四：蛋壳钱包支付微服务安全与治理 — 安全四层+分布式事务+容量规划

### 场景描述
```
蛋壳钱包，核心链路：充值→转账→提现

安全要求：
  1. 资金不能错（TCC强一致，Day5已覆盖）
  2. 支付接口不能被盗刷
  3. 内部服务间不能被横向渗透
  4. 用户敏感数据（身份证/银行卡）要脱敏加密

容量要求：
  1. 充值QPS峰值 5000（发工资日）
  2. 转账QPS日常 2000，峰值 8000（红包场景）
  3. 提现QPS日常 500（低频）
```

```

  │  │                                               │    │

  │  │    L2 服务滑动窗口（多维度统计RT/异常率）      │    │

  │  └──────────────────────────────────────────────┘    │

分布式事务（Day5）— TCC三个陷阱：
  ┌──────────────────────────────────────────────────────┐
  │  转账场景：A转100给B                                  │
  │                                                       │

  │                                                       │

### 架构师思考：支付系统的安全为什么是四层而不是一层？
```
单层安全的致命漏洞：

  只有网关JWT认证 → 拿到Token就能调任何服务
    攻击场景：拿到Token后调用提现服务的内部接口
    结果：资金被盗

  四层防御的价值：
    L1 网关认证：挡住未认证请求（95%攻击）
    L2 mTLS：挡住伪造服务调用（4%攻击）
    L3 RBAC：挡住越权操作（0.9%攻击）
    L4 数据加密：即使DB被拖库也无法还原明文（0.1%攻击）

  这和Day2五层漏斗的思维一致：
    每层挡一部分，漏斗越往下流量越小
    安全也是漏斗：外层挡大部分，内层保底线

  面试核心句：
  "支付系统安全是四层纵深防御，不是网关加个JWT就完了。
   网关管你是谁，mTLS管服务身份，RBAC管你能做什么，
   数据加密管数据本身的安全。零信任原则：进了内网也不安全"
```

## 架构师总结：微服务的"六维决策模型"

### 维度一：拆不拆 → 四条件判断
```
Day5架构演进的核心判断：

  条件1：团队规模 ≥ 5人
  条件2：部署频率 ≥ 每天1次
  条件3：有独立运维能力
  条件4：有基建投入预算

  4个全满足 → 微服务
  2-3个满足 → 选定服务化
  0-1个满足 → 模块化单体

  记住：微服务不是目的，是手段
  痛点是什么→拆了能不能解决→基建跟不跟得上
```

### 维度二：怎么拆 → 拆分策略选型
```
Day1拆分策略 + Day5演进路线：

  按业务能力拆（推荐）：
    秒杀/订单/库存/支付 — 自然边界清晰
    保险：核保/保费/保单/理赔 — 业务领域明确

  按子域拆（DDD）：
    适合业务复杂、团队大的场景

  反模式：按技术层拆（Controller/Service/DAO各一个服务）
    结果：一个业务改三个服务，比单体还痛苦

  拆分顺序：最痛的先拆，不是最简单的先拆
    ① 变更最频繁
    ② 故障率最高
    ③ 资源需求最大
```

### 维度三：怎么治 → 限流/熔断/降级/隔离
```
Day2服务治理的四件套：

  限流 → 多少请求能进来
    算法选型根因：入口令牌桶（有突发）/中间滑动窗口（多维统计）/出口漏桶（匀速）

  熔断 → 下游挂了怎么办
    策略匹配故障模式：慢调用比例→变慢、异常比例→偶尔挂、异常数→必挂

  降级 → 服务不可用时怎么兜底
    决策矩阵：实时性×数据过期容忍度→兜底值/缓存/异步化/备链路

  隔离 → 故障不要扩散
    内部服务→信号量（轻量够用）
    外部服务→线程池+超时（彻底隔离，灵魂是超时）
```

### 维度四：怎么建 → 基础设施优先级
```
Day3核心基础设施，按优先级：

  P0 必须先有（没有就别拆）：
    ① 注册发现（Nacos） — 服务互相找不到=拆了白拆
    ② 配置中心（Nacos） — 配置散落=改一个配置重启所有
    ③ 网关（Gateway） — 没有统一入口=客户端直连服务

  P1 拆完必须有（没有就别想高可用）：
    ④ 链路追踪（SkyWalking） — 没有排障=盲人摸象
    ⑤ 限流熔断（Sentinel） — 没有治理=一个挂全挂

  P2 进阶能力：
    ⑥ 监控告警（Prometheus+Grafana）
    ⑦ Service Mesh（Istio）
```

### 维度五：怎么观测 → 可观测性三位一体
```
Day4可观测性SOP：

  Metrics → 发现异常(what)
    四种类型：Counter算QPS/Gauge看状态/Histogram算分位数

  Tracing → 定位卡点(where)
    三种传播：HTTP→Header/MQ→Property/线程池→TTL
    采样组合：头部1%+尾部100%

  Logging → 确认根因(why)
    结构化日志+TraceId关联

  告警三条红线：
    1. 每条P0必须有人响应
    2. 同一故障不超过3条
    3. 每条带上下文
```

### 维度六：怎么战 → 容量+应急+事务三闭环
```
Day5实战综合的三个闭环：

  容量闭环：基线→预估→计算→扩容
    数字有根因：70-80%安全水位、1.3冗余、3-5倍峰值

  应急闭环：止血→通告→定位→修复→复盘
    止血优先级：回滚>降级>限流>重启
    5分钟MTTR目标

  事务闭环：选型→实现→兜底
    三维选型：一致性×业务特征×团队成本
    兜底：事务控制表+对账脚本+人工介入
```

"我从六个维度理解微服务架构：

   部署≥每天1次、有运维能力、有基建预算。我之前保险商城项目

2. 怎么拆：按业务能力拆，最痛的先拆。秒杀系统按业务能力拆成
   秒杀/订单/库存/支付四个服务，每个独立扩缩容。

3. 怎么治：限流/熔断/降级/隔离四件套。关键是要匹配——
   慢调用比例熔断对应下游变慢，异常数熔断对应下游必挂；
   外部服务用线程池+超时隔离，内部服务用信号量够用。

4. 怎么建基建：注册发现和网关是P0，没有就别拆。
   我项目用Nacos做注册和配置，Gateway做网关，SkyWalking做链路追踪。

5. 怎么观测：Metrics发现→Tracing定位→Logging确认根因。
   采样策略是头部1%+尾部100%，异常必采不丢排障线索。

6. 怎么战：容量四步法闭环+应急5步SOP+事务三维选型。
   容量规划安全水位70-80%，冗余1.3；故障止血决策树是
   回滚>降级>限流>重启，5分钟MTTR；分布式事务看一致性×
   业务特征×团队成本，金融用TCC因为钱不能错。

一句话总结：微服务不是技术选型，是工程体系——拆、治、建、观、战，
缺一不可。"
```
