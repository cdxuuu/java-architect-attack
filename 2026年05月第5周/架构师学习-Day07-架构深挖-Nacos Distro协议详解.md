# Day 7：架构深挖 - Nacos Distro 协议详解 — 从协议设计反推技术选型

## 引言

Day3我们学了Nacos的注册发现和配置中心，但理解停留在"读全量写部分"的口号。差距八和九明确指出：Distro协议说不出完整流程，临时vs持久实例的核心区别搞混，注册中心的UDP推拉和配置中心的Long Polling混为一谈。

今天从架构师视角深挖Distro协议，训练一种核心能力：**从协议设计反推技术选型**——为什么Nacos AP模式选了Distro而不是Raft？为什么注册中心用UDP推送而配置中心用Long Polling？每个设计决策背后都是数据特性在驱动。

## 1. Distro协议的核心定位

### 1.1 Nacos的两套一致性协议

```
Nacos同时支持AP和CP，用两套不同的协议：

  ┌─────────────────────────────────────────────────────────┐
  │                  Nacos Server                           │
  │                                                         │
  │  ┌──────────────────┐    ┌──────────────────────────┐   │
  │  │  临时实例(AP)     │    │  持久实例(CP)             │   │
  │  │  Distro协议      │    │  Raft协议(JRaft)         │   │
  │  │                  │    │                          │   │
  │  │  特点：          │    │  特点：                   │   │
  │  │  • 高可用优先    │    │  • 强一致优先             │   │
  │  │  • 最终一致      │    │  • Leader写+Follower读   │   │
  │  │  • 每节点可写    │    │  • 只有Leader可写        │   │
  │  │  • 对等架构      │    │  • 主从架构               │   │
  └──────────────────┘    └──────────────────────────┘   │
  │                                                         │
  │  数据特性决定协议选择：                                  │
  │    注册数据：允许短暂不一致 → AP(Distro)                │
  │    元数据/持久服务：不能丢 → CP(Raft)                   │
  └─────────────────────────────────────────────────────────┘

  面试核心句：
  "Nacos不是AP或CP二选一，是临时实例用Distro做AP、
   持久实例用Raft做CP，数据特性决定协议选择"
```

### 1.2 Distro vs Raft 的设计哲学差异

```
Raft的设计哲学：强一致 → Leader写 → 所有节点数据一致后才返回

  客户端 → Leader写 → 同步Follower → 多数确认 → 返回成功
  优点：读任何节点都是最新数据
  缺点：Leader单点写瓶颈，Leader挂了整个集群不可写（选举期间）

Distro的设计哲学：高可用 → 每节点可写 → 异步同步最终一致

  客户端 → 任意节点写 → 立即返回 → 异步同步其他节点
  优点：任何节点挂了不影响写入，无选举开销
  缺点：短暂窗口内不同节点数据可能不一致

  为什么注册中心选Distro？
    ① 注册数据允许短暂不一致（服务A看到3个实例，服务B看到2个，
       最多多一次失败重试，不影响业务正确性）
    ② 注册中心可用性比一致性更重要（注册中心挂了=所有服务发现不了=全局故障）
    ③ 注册数据变更频率高（上下线频繁），Leader写会成为瓶颈
```

## 2. Distro协议核心机制

### 2.1 Hash分区 — 每个节点负责一部分

```
Distro的第一个设计：职责划分

  假设3个Nacos节点：N1, N2, N3
  Service列表：order-service, user-service, pay-service,
               inventory-service, notify-service, logistics-service

  分区算法：hashCode(serviceName) % nodeCount

  ┌─────────────────────────────────────────────────────────┐
  │  N1 负责：order-service, notify-service                  │
  │  N2 负责：user-service, logistics-service                │
  │  N3 负责：pay-service, inventory-service                 │
  │                                                          │
  │  关键理解：                                              │
  │  "负责" = 该节点是这个service的权威数据源               │
  │  写请求必须到负责节点 → 保证同一service的写入顺序       │
  │  读请求任意节点 → 返回本地缓存（可能短暂过期）          │
  └─────────────────────────────────────────────────────────┘

  分区的好处：
    1. 写入分散：不是所有写请求打到一个Leader，每节点处理1/N的写
    2. 故障隔离：N1挂了只影响它负责的service，其他service不受影响
    3. 扩展性好：加节点自动重新分区，写能力线性增长

  面试追问："写请求打到非负责节点怎么办？"
    → 转发到负责节点（不是拒绝，是转发）
    → 客户端无感知，Nacos Server内部做路由
```

### 2.2 写流程 — 转发+本地写+异步同步

```
完整写流程（以注册order-service为例）：

  ┌─────────────────────────────────────────────────────────┐
  │  Step1：客户端注册请求                                   │
  │                                                          │
  │  ServiceA实例 → POST /nacos/v1/ns/instance               │
  │                → 任意Nacos节点（比如打到N2）             │
  └─────────────────────────────────────────────────────────┘
                    │
                    ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Step2：N2判断是否负责                                  │
  │                                                          │
  │  hashCode("order-service") % 3 → N1负责                 │
  │  N2不是负责节点 → 转发请求到N1                          │
  │                                                          │
  │  转发方式：HTTP内部调用                                  │
  │  POST http://N1:8848/nacos/v1/ns/distro/datum           │
  └─────────────────────────────────────────────────────────┘
                    │
                    ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Step3：N1（负责节点）处理写请求                         │
  │                                                          │
  │  ① 写入本地注册表（内存+磁盘）                          │
  │  ② 更新本地索引                                         │
  │  ③ 立即返回客户端成功                                   │
  │     ← 注意：不等其他节点同步就返回！                     │
  └─────────────────────────────────────────────────────────┘
                    │
                    ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Step4：异步同步其他节点（fire-and-forget）              │
  │                                                          │
  │  N1 ──→ N2：同步order-service的增量数据                  │
  │  N1 ──→ N3：同步order-service的增量数据                  │
  │                                                          │
  │  关键：fire-and-forget                                   │
  │    • 异步发送，不等对方确认                               │
  │    • 不阻塞写请求返回                                    │
  │    • 最终一致，不是强一致                                 │
  │                                                          │
  │  同步失败怎么办？                                        │
  │    • 重试（最多3次）                                     │
  │    • 定期全量校验兜底（每5秒一次）                       │
  └─────────────────────────────────────────────────────────┘

  面试画这个图就够了：
  客户端 → 任意节点 → 判断负责 → 转发到负责节点 → 本地写+返回
                                                    → 异步同步其他节点
```

### 2.3 读流程 — 就近读取+定期刷新

```
读流程（以发现order-service实例为例）：

  ┌─────────────────────────────────────────────────────────┐
  │  客户端 → GET /nacos/v1/ns/instance/list                 │
  │          → 任意Nacos节点（比如N2）                       │
  │                                                          │
  │  N2直接返回本地缓存中的order-service实例列表             │
  │  不需要问负责节点N1！                                    │
  │                                                          │
  │  这就是"读全量"：每个节点都有全量数据的本地缓存         │
  └─────────────────────────────────────────────────────────┘

  读到的数据可能过期吗？
    是的！N2的order-service数据是异步同步过来的
    最大延迟 = 同步间隔（默认1秒） + 网络延迟

  注册中心能接受这个延迟吗？
    能！服务发现本身就是"最终一致"语义：
      • 客户端本地也有缓存（30秒刷新一次）
      • 多1秒的Nacos Server侧延迟，叠加客户端缓存
      • 最坏情况：服务下线后31秒才感知到
      • 客户端本身就有重试机制，一次调用失败换实例重试

  面试核心句：
  "Distro读全量的设计基于一个前提：注册数据允许短暂不一致。
   客户端本身就有缓存+重试，Nacos Server侧1秒的同步延迟
   在整个链路中是可接受的"
```

### 2.4 新节点加入 — 全量同步+增量同步

```
新节点N4加入集群：

  ┌─────────────────────────────────────────────────────────┐
  │  Phase1：全量同步（启动时）                              │
  │                                                          │
  │  N4启动 → 向任意节点（比如N1）请求全量数据              │
  │  GET /nacos/v1/ns/distro/datum?all=true                  │
  │  N1返回所有service的完整实例数据                         │
  │  N4写入本地注册表                                        │
  │                                                          │
  │  为什么需要全量同步？                                    │
  │    新节点没有任何数据，增量同步无法开始                  │
  │    必须先有一个完整的基线                                │
  └─────────────────────────────────────────────────────────┘
                    │
                    ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Phase2：增量同步（运行时）                              │
  │                                                          │
  │  全量同步完成后，切换到增量同步：                        │
  │    • 负责节点变更时，异步推送增量数据到N4                │
  │    • 定期校验：每5秒，N4与负责节点比对数据版本          │
  │    • 发现不一致 → 请求缺失的增量数据                    │
  │                                                          │
  │  增量同步的三种触发方式：                                │
  │    1. 事件驱动：负责节点数据变更→推送                    │
  │    2. 定期拉取：每5秒校验一次                            │
  │    3. 全量兜底：增量校验不一致→全量重新同步             │
  └─────────────────────────────────────────────────────────┘

  面试追问："全量同步期间，新节点能处理读请求吗？"
    能！读本地缓存，只是数据可能不完整
    新节点的注册表在同步过程中逐步填充
    客户端重试机制兜底，不完整不会导致业务故障
```

## 3. Distro协议的一致性保障

### 3.1 三层一致性保障机制

```
Distro不是"不管一致性"，是"尽力保证+兜底校验"：

  ┌─────────────────────────────────────────────────────────┐
  │  L1 实时异步同步（毫秒级）                              │
  │                                                          │
  │  写请求 → 负责节点写入 → 异步推送变更到其他节点         │
  │  覆盖场景：正常情况下的数据变更                          │
  │  延迟：1-10ms（同机房）                                 │
  └─────────────────────────────────────────────────────────┘
                    │
                    ▼
  ┌─────────────────────────────────────────────────────────┐
  │  L2 定期增量校验（5秒级）                               │
  │                                                          │
  │  每个节点每5秒与负责节点比对数据版本号                  │
  │  发现版本不一致 → 请求增量数据                          │
  │  覆盖场景：L1异步推送丢失（网络抖动/节点短暂不可达）   │
  │  延迟：最多5秒                                          │
  └─────────────────────────────────────────────────────────┘
                    │
                    ▼
  ┌─────────────────────────────────────────────────────────┐
  │  L3 全量校验兜底（30秒级）                              │
  │                                                          │
  │  每个节点每30秒与负责节点做全量数据比对                 │
  │  发现大量不一致 → 触发全量重新同步                      │
  │  覆盖场景：长时间网络分区后的数据修复                   │
  │  延迟：最多30秒                                         │
  └─────────────────────────────────────────────────────────┘

  这和Day2五层漏斗的思维一脉相承：
    每层覆盖不同场景，越往下越重但越可靠
    正常情况L1就够，异常情况有L2/L3兜底
```

### 3.2 与Eureka同步机制的对比

```
Distro vs Eureka的同步模型差异：

  ┌─────────────────────┬───────────────────┬───────────────────┐
  │                     │  Eureka           │  Nacos Distro     │
  ├─────────────────────┼───────────────────┼───────────────────┤
  │  写扩散方式         │  同步复制          │  异步fire-and-    │
  │                     │  (Peer-to-Peer)   │  forget           │
  │                     │  写N-1个节点      │  只同步N-1个节点  │
  │                     │  阻塞等待         │  不阻塞           │
  ├─────────────────────┼───────────────────┼───────────────────┤
  │  写入延迟           │  O(N)同步延迟     │  O(1)本地写+      │
  │                     │  节点多时慢       │  异步同步          │
  ├─────────────────────┼───────────────────┼───────────────────┤
  │  负责节点           │  无概念           │  Hash分区          │
  │                     │  任意节点可写     │  写请求路由到      │
  │                     │  全部同步         │  负责节点          │
  ├─────────────────────┼───────────────────┼───────────────────┤
  │  数据一致性         │  同步复制+定期    │  异步同步+增量    │
  │                     │  对比修复         │  校验+全量兜底    │
  ├─────────────────────┼───────────────────┼───────────────────┤
  │  新节点加入         │  全量拉取         │  全量拉取+增量    │
  │                     │  无增量机制       │  同步             │
  ├─────────────────────┼───────────────────┼───────────────────┤
  │  写入顺序保证       │  无（时间戳覆盖）│  同一service由    │
  │                     │                   │  同一负责节点写   │
  └─────────────────────┴───────────────────┴───────────────────┘

  Distro比Eureka的核心改进：
    1. 异步同步不阻塞写入 → 写性能更好
    2. Hash分区保证写入顺序 → 同一service的数据不会乱序
    3. 三层校验机制 → 一致性保障比Eureka更强
    4. 增量同步 → 网络开销更小

  面试话术：
  "Eureka是同步复制写扩散，Nacos Distro是异步fire-and-forget。
   Eureka任意节点可写→同一服务数据可能乱序（时间戳覆盖解决），
   Distro Hash分区→同一服务由负责节点统一写入→顺序保证。
   写入性能Distro更好，一致性保障Distro也更强"
```

## 4. 临时实例 vs 持久实例 — 数据特性决定一切

### 4.1 核心区别：健康检查发起方

```
差距九的核心：临时vs持久实例不是协议区别，是健康检查发起方区别

  ┌─────────────────────────────────────────────────────────┐
  │  临时实例（ephemeral）                                   │
  │                                                          │
  │  健康检查：客户端主动心跳上报                            │
  │  ┌──────────────────────────────────────────────┐       │
  │  │  Service实例 → 心跳 → Nacos Server           │       │
  │  │                                              │       │
  │  │  心跳间隔：5秒                                │       │
  │  │  不健康标记：15秒未收到心跳                   │       │
  │  │  剔除实例：30秒未收到心跳                     │       │
  │  │                                              │       │
  │  │  核心语义：客户端证明自己活着                 │       │
  │  │  不发心跳 = 我死了，请把我删掉               │       │
  │  └──────────────────────────────────────────────┘       │
  │                                                          │
  │  协议：Distro (AP)                                       │
  │  为什么AP？客户端随时可能下线，注册数据本身是临时的      │
  │  临时数据允许短暂不一致 → AP够用                         │
  └─────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │  持久实例（persistent）                                  │
  │                                                          │
  │  健康检查：服务端主动探测                                │
  │  ┌──────────────────────────────────────────────┐       │
  │  │  Nacos Server → TCP/HTTP探测 → Service实例   │       │
  │  │                                              │       │
  │  │  探测间隔：20秒（可配置）                     │       │
  │  │  不健康标记：探测失败                         │       │
  │  │  不剔除！只标记为不健康                       │       │
  │  │                                              │       │
  │  │  核心语义：服务端验证对方活着                 │       │
  │  │  探测失败 ≠ 实例该删，只代表暂时不可达       │       │
  │  └──────────────────────────────────────────────┘       │
  │                                                          │
  │  协议：Raft (CP)                                         │
  │  为什么CP？持久实例不能丢，注册数据是持久的              │
  │  持久数据不允许不一致 → CP必须                           │
  └─────────────────────────────────────────────────────────┘

  一句话记忆：
    临时实例：客户端证明自己活着，不发心跳=死了=删掉
    持久实例：服务端验证对方活着，探测失败≠死了=只标记不健康
```

### 4.2 从数据特性推导技术选型

```
差距九的核心教训：数据特性决定技术选型

  ┌──────────────┬──────────────────┬──────────────────┐
  │  数据特性     │  注册数据         │  配置数据         │
  ├──────────────┼──────────────────┼──────────────────┤
  │  一致性要求   │  允许短暂不一致   │  不能丢不能错     │
  │  变更频率     │  高（上下线频繁） │  低（偶尔改配置） │
  │  数据量       │  小（实例列表）   │  小（KV配置）     │
  │  允许丢失     │  临时实例可以     │  绝对不行         │
  ├──────────────┼──────────────────┼──────────────────┤
  │  → 选型       │  AP + UDP推送     │  CP + Long       │
  │              │  + 客户端心跳     │  Polling          │
  └──────────────┴──────────────────┴──────────────────┘

  注册中心：UDP推送 + 6秒拉取兜底
    为什么用UDP？
      ① 注册数据允许丢（客户端有缓存+重试）
      ② UDP轻量，不需要建立连接
      ③ 推送失败不影响服务端
    为什么还要拉取兜底？
      UDP不可靠 → 推送可能丢 → 定期拉取保证最终一致
      拉取间隔6秒 → 最多6秒延迟

  配置中心：Long Polling 29.5秒
    为什么不用UDP推送？
      配置数据不能丢 → UDP不可靠
      配置变更必须精确送达 → 需要可靠通道
    为什么用Long Polling？
      ① 配置变更频率低 → 推送模式大部分时间空转
      ② Long Polling = 客户端发请求，服务端hold住，有变更再返回
      ③ 29.5秒超时 → 客户端立即重连 → 变更秒级送达
      ④ 可靠：HTTP协议，不会丢

  面试核心句：
  "注册数据允许短暂不一致所以用轻量UDP推送+拉取兜底，
   配置数据不能丢所以用可靠Long Polling。
   不是技术选型决定架构，是数据特性决定技术选型"
```

## 5. Distro协议的工程细节

### 5.1 数据同步的版本控制

```
Distro用版本号做增量同步：

  每个Service的数据有一个版本号（timestamp）：

  ┌─────────────────────────────────────────────────────────┐
  │  N1（负责节点）：                                        │
  │    order-service → 版本 1001, 实例列表 [A, B, C]        │
  │    user-service  → 版本 1002, 实例列表 [D, E]           │
  │                                                          │
  │  N2（非负责节点）：                                      │
  │    order-service → 版本 1001, 实例列表 [A, B, C]  ← 一致│
  │    user-service  → 版本 1000, 实例列表 [D]        ← 过期│
  └─────────────────────────────────────────────────────────┘

  增量校验流程：
    N2每5秒 → 向N1请求版本列表
    比对本地版本 → 发现user-service版本1000 < 1001
    → 请求N1发送user-service的增量数据
    → 更新本地版本为1001

  全量校验流程：
    N2每30秒 → 向N1请求全量版本列表
    比对所有版本 → 如果差异超过阈值
    → 触发全量重新同步
```

### 5.2 节点故障处理

```
场景：负责节点N1挂了

  ┌─────────────────────────────────────────────────────────┐
  │  临时实例场景：                                          │
  │                                                          │
  │  N1负责order-service → N1挂了                            │
  │                                                          │
  │  影响：                                                  │
  │    ① order-service的新注册请求无法处理                   │
  │       → 客户端会重试，可能打到N2/N3                     │
  │       → N2/N3转发到N1 → 转发失败 → 返回错误             │
  │       → 客户端重试 → 最终N1恢复或重新分区               │
  │                                                          │
  │    ② 已有的order-service数据还在其他节点的缓存中        │
  │       → 读请求仍然可以服务                              │
  │       → 只是无法更新（新注册/注销）                      │
  │                                                          │
  │  恢复：                                                  │
  │    N1恢复 → 重新加入集群 → 全量同步                     │
  │    N1永久不可用 → 集群自动重新Hash分区                  │
  │       → order-service的负责节点变为N2或N3               │
  │       → 已有的注册数据靠其他节点缓存恢复                │
  │       → 心跳超时的实例会被正常剔除                      │
  ├─────────────────────────────────────────────────────────┤
  │  持久实例场景：                                          │
  │                                                          │
  │  N1是Raft Leader → N1挂了                               │
  │                                                          │
  │  影响：                                                  │
  │    集群进入选举状态 → 选举新Leader（约1秒）             │
  │    选举期间不可写（Raft保证）                            │
  │    选举完成后恢复正常                                    │
  └─────────────────────────────────────────────────────────┘
```

生产环境调优参数：

  ┌──────────────────────┬───────────┬───────────────────────┐

  ├──────────────────────┼───────────┼───────────────────────┤

## 6. 架构师视角：从Distro协议推导的设计原则

### 6.1 原则一：数据特性驱动协议选择

```
这是Day3差距"数据特性→技术选型"的核心体现：

  注册数据（临时实例）：
    特性：临时、允许短暂不一致、变更频繁
    → AP + Distro + 客户端心跳 + UDP推送

  注册数据（持久实例）：
    特性：持久、不能丢、变更少
    → CP + Raft + 服务端探测

  配置数据：
    特性：不能丢不能错、变更频率低
    → CP + Long Polling + 历史版本

  推广到其他系统的选型：

  ┌──────────────┬──────────────┬──────────────────────┐
  │  数据特性     │  协议/方案    │  例子                │
  ├──────────────┼──────────────┼──────────────────────┤
  │  临时+允许丢  │  AP+异步     │  服务注册/会话数据   │
  │  持久+不能丢  │  CP+同步     │  配置/元数据/订单    │
  │  海量+可丢失  │  AP+批处理   │  日志/指标/埋点      │
  │  金融+不能错  │  CP+TCC      │  转账/支付/资金      │
  └──────────────┴──────────────┴──────────────────────┘
```

### 6.2 原则二：写入顺序由分区保证

```
Distro用Hash分区的核心价值：同一service的写入有序

  没有分区（Eureka模式）：
    N1收到order-service注册实例A → 时间戳100
    N2收到order-service注册实例B → 时间戳101
    同步到N3：N1先到（时间戳100），N2后到（时间戳101）
    → 顺序正确（碰巧）

    但如果N2的网络比N1快：
    → N2先到N3（时间戳101），N1后到N3（时间戳100）
    → N3最终以时间戳100覆盖101？还是101覆盖100？
    → Eureka用时间戳覆盖（后到覆盖先到），但语义上可能不正确

  有分区（Distro模式）：
    order-service的所有写请求 → 路由到N1（负责节点）
    N1本地有序写入 → 异步同步到其他节点
    → 其他节点收到的数据一定是按N1的写入顺序
    → 顺序保证，不需要时间戳覆盖

  推广：
    任何需要写入顺序的系统，都应该由单点（分区）负责写入
    Kafka的Partition Leader、MySQL的主从复制、Redis的主从同步
    → 同一数据由单一权威节点写入，是保证顺序的通用方案
```

### 6.3 原则三：多层兜底比单层强一致更实用

```
Distro的三层一致性保障 vs Raft的强一致：

  Raft强一致：
    优点：读任何节点都是最新数据
    缺点：Leader写瓶颈、选举期间不可用、多数节点不可达则不可用

  Distro三层兜底：
    L1 异步同步 → 覆盖99%场景（正常情况）
    L2 增量校验 → 覆盖0.9%场景（推送丢失）
    L3 全量校验 → 覆盖0.1%场景（网络分区后恢复）

  哪个更适合注册中心？
    Distro！因为：
    ① 注册中心可用性 >> 一致性（挂了=全局故障）
    ② 三层兜底已经够用（最多5秒不一致）
    ③ 客户端本身就有缓存+重试，服务端1秒延迟可接受

  推广到其他系统：
    大部分互联网系统更适合"多层兜底"而不是"强一致"：
      • 缓存：L1本地缓存 + L2 Redis + L3 DB
      • 消息：L1实时推送 + L2重试 + L3死信+对账
      • 安全：L1网关认证 + L2 mTLS + L3 RBAC + L4数据加密
      • 限流：L1 CDN + L2网关 + L3服务 + L4接口 + L5 DB

    核心思想：每层覆盖不同场景，越往下越可靠
    这和Day2五层漏斗、Day4安全四层模型完全一致
```

## 7. 面试实战

### 7.1 面试官问："Nacos的AP模式怎么保证数据一致性？"

```
❌ 开发级回答：
  "Nacos用Distro协议，读全量写部分。"

❌ 高级开发级回答：
  "Distro协议，读全量写部分，最终一致性。"

✅ 架构师级回答：
  "Distro协议的核心是Hash分区+异步复制，分三层保证最终一致性：

   第一层，Hash分区保证写入顺序。每个Service由一个负责节点写入，
   hashCode(serviceName)%nodeCount确定负责节点。写请求打到非负责节点
   会自动转发。这样同一Service的写入顺序由单一节点保证，避免乱序。

   第二层，fire-and-forget异步同步。负责节点写入后立即返回，
   异步推送变更到其他节点，不阻塞写入。同步失败会重试3次。

   第三层，定期校验兜底。增量校验每5秒比对版本号，全量校验每30秒
   比对完整数据。发现不一致就拉取修复。

   这三层覆盖不同场景：L1覆盖正常情况1-10ms延迟，L2覆盖推送丢失
   最多5秒延迟，L3覆盖网络分区恢复最多30秒延迟。

   和Eureka对比：Eureka是同步复制写扩散，Distro是异步fire-and-forget，
   Distro写入性能更好。Eureka没有分区概念所以同一服务数据可能乱序，
   Distro Hash分区保证写入顺序。

   核心选型逻辑：注册数据允许短暂不一致所以用AP+Distro，
   配置数据不能丢所以用CP+Long Polling，
   数据特性决定协议选择。"
```

### 7.2 面试官问："Nacos临时实例和持久实例有什么区别？"

```
❌ 开发级回答：
  "临时实例用AP，持久实例用CP。"

❌ 高级开发级回答：
  "临时实例客户端心跳，持久实例服务端探测。"

✅ 架构师级回答：
  "核心区别不是协议，是健康检查发起方——谁来判断实例是否存活。

   临时实例：客户端证明自己活着。每5秒发心跳，15秒不健康，30秒剔除。
   不发心跳=我死了=请把我删掉。协议用Distro(AP)，因为临时数据允许
   短暂不一致。

   持久实例：服务端验证对方活着。服务端每20秒主动探测，探测失败只标记
   不健康不剔除。探测失败≠实例该删，只代表暂时不可达。协议用Raft(CP)，
   因为持久数据不允许丢。

   这个设计体现了'数据特性决定技术选型'：
   临时实例的数据本身就是临时的→允许丢→AP+客户端心跳
   持久实例的数据是持久的→不能丢→CP+服务端探测

   延伸到注册中心和配置中心的通信模型也是同样的逻辑：
   注册中心用UDP推送+6秒拉取兜底（注册数据允许短暂不一致）
   配置中心用Long Polling 29.5秒（配置数据不能丢）
   不是随意选择，是数据特性驱动。"
```

### 7.3 面试官问："为什么Nacos不全部用Raft？"

```
✅ 架构师级回答：
  "两个原因：性能和可用性。

   性能：Raft只有一个Leader写，所有写请求串行化。注册数据变更频繁
   （上下线、心跳更新），如果全部走Raft，Leader会成为写瓶颈。
   Distro的Hash分区让每节点处理1/N的写入，写能力线性扩展。

   可用性：Raft Leader挂了需要选举，选举期间集群不可写。注册中心的
   可用性比一致性更重要——注册中心挂了=所有服务发现不了=全局故障。
   Distro没有Leader概念，任何节点挂了只影响它负责的1/N的service，
   其他service不受影响。

   核心判断：注册数据允许短暂不一致（客户端有缓存+重试），
   但注册中心不能挂（全局影响）。所以AP比CP更适合注册中心。

   Nacos的做法是按数据特性分协议：临时实例AP+持久实例CP，
   不是一刀切，是每种数据用最适合的方案。"
```

## 8. 总结

### Distro协议的架构师级理解

```
Distro协议的本质：
  Hash分区 → 写入有序
  异步同步 → 写入高性能
  三层兜底 → 最终一致性
  对等架构 → 高可用

  不是"随便写写然后同步"，是精心设计的分布式协议

从Distro推导的三个设计原则：
  ① 数据特性驱动协议选择（临时→AP，持久→CP）
  ② 单一分区保证写入顺序（同一数据由单一权威节点写入）
  ③ 多层兜底比单层强一致更实用（L1实时+L2增量+L3全量）

这三个原则不仅适用于Nacos，适用于所有分布式系统设计：
  Kafka的Partition → 原则②
  Redis的主从同步 → 原则②
  任何缓存架构 → 原则③
  任何数据选型 → 原则①
```
