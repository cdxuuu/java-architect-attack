# Day 1：微服务基础 — 架构师角度梳理

## 核心定位

微服务不是目的，是**应对复杂度的手段**。架构师要回答的不是"要不要微服务"，而是"当前阶段的复杂度是否值得微服务的成本"。

---

## 拆分策略：架构师的核心手艺

### 拆分的本质是减耦合，不是切模块

```
错误理解：拆 = 按功能切成小项目
正确理解：拆 = 让变化隔离、故障隔离、团队隔离

判断拆得好不好的唯一标准：
  - 改一个需求，需要改几个服务？（1个最好，3个及格，5个+说明拆错了）
  - 一个服务挂了，影响多大范围？（挂1个不影响其他=拆对了）
  - 两个团队能独立发版吗？（能=拆对了）
```

### 三种拆法不是三选一，而是叠加使用

```
第一刀：按业务域（DDD限界上下文）
  → 确定大边界：订单域、支付域、用户域

第二刀：按数据所有权
  → 确认数据归属：账户表只归账户服务，订单服务不能直连

第三刀：按变化频率
  → 微调粒度：核心下单稳定+活动规则多变 → 活动引擎独立服务
```

### 过度拆分的本质：分布式复杂度 > 业务复杂度

```
判断公式：
  拆分收益 = 独立部署 + 故障隔离 + 团队自治
  拆分成本 = 分布式事务 + 调用链RT + 运维复杂度 + 调试难度

  当 成本 > 收益 时，不该拆

  典型信号：
    - 一个需求改5个服务 → 拆错了
    - 排一个bug看4个服务日志 → 拆错了
    - 100个服务只有10个人维护 → 拆错了
    - CRUD服务（无业务逻辑）独立部署 → 拆错了
```

## 注册发现：CAP选择是架构价值观

### AP vs CP不是技术选型，是业务价值观

```
注册中心选AP：
  价值观 → 可用性比一致性重要
  业务含义 → "服务发现不可用"是比"读到旧数据"更严重的故障
  数学证明 → 注册中心不可用 = 100%请求受影响；读到旧数据 = 1次重试

配置中心选CP：
  价值观 → 一致性比可用性重要
  业务含义 → "配置不一致"比"配置不可读"更危险
  真实案例 → 数据库密码配错了，所有连接失败；配对了但不同节点不一致，更难发现

Nacos的设计哲学：
  注册发现默认AP + 配置管理默认CP = 同一个产品，两套价值观
  这不是偷懒，是对业务语义的精准理解
```

### 注册中心宕机不等于服务宕机

```
架构师必须想清楚的问题：注册中心全挂了，系统还能用吗？

  能用多久？
    - Nacos客户端本地内存缓存 → 无限长（直到服务上下线）
    - Nacos客户端本地磁盘缓存 → Consumer重启也能恢复

  什么时候不能用？
    - 新服务上线 → 注册中心挂了，无法通知Consumer
    - 服务下线 → 注册中心挂了，Consumer还在调已下线实例 → 报错

  降级SOP：
    1. 0-5分钟：靠本地缓存，业务无感知
    2. 5-30分钟：人工切换备用注册中心集群
    3. 30分钟+：静态配置文件降级（运维手动维护服务列表）
```

## 通信方式：性能不是唯一维度

### 选型的三维决策

```
         性能
          │
          │    RPC（高性能区）
          │   ╱
          │  ╱
          │ ╱
          │╱──────────────── 耦合度
         ╱│
        ╱ │
       ╱  │ HTTP（低耦合区）
      ╱   │
          │
     MQ（异步区）

  RPC：高性能 + 高耦合 → 内部高频调用
  HTTP：低性能 + 低耦合 → 对外API + BFF层
  MQ：不看性能 + 最松耦合 → 异步解耦 + 削峰

  常见错误：全链路用一种方式
  正确做法：混合使用，每段按需选择
```

### Dubbo调用不经过注册中心

```
这是面试高频细节题：

  Consumer从注册中心拿到Provider列表 → 缓存到本地
  后续所有调用 → 直连Provider，不再经过注册中心

  意味着：
    1. 注册中心挂了，正在运行的调用不受影响
    2. 注册中心的性能瓶颈不会成为调用链路的瓶颈
    3. 但Provider列表更新依赖注册中心推送

  和MQ的对比：
    MQ：生产者→Broker→消费者，消息经过Broker
    RPC：Consumer→Provider，不经过Registry
    这是同步和异步通信的根本区别
```

## 容错设计：分层防御的工程思维

### 四种容错不是四个方案，是一套体系

```
调用链路：
  A → B → C

  每一跳都有可能出问题，架构师要做的是：

  第一层：超时（底线）
    A调B超时3秒 → 快速失败，不阻塞线程
    问题：超时了还是占用了3秒

  第二层：熔断（主动保护）
    B连续失败50% → 熔断 → A直接快速失败，不等3秒
    问题：熔断期间正常请求也被拒

  第三层：隔离（切断传播）
    A调B用独立线程池 → B慢只影响调B的线程，不影响调C
    问题：线程池开销

  第四层：降级（牺牲非核心）
    B熔断后 → 返回缓存数据/走备链路/返回兜底响应
    问题：数据不实时

  四层叠起来 = 分层防御
  任何单层都有漏洞，叠多层才安全
```

### 限流算法的本质：谁在保护谁

```
令牌桶 = 保护自己（允许合理的突发）
  网关用令牌桶 → "我有10000个令牌，谁抢到谁进"
  适合入口：流量进来前先挡一道

漏桶 = 保护下游（强制匀速消费）
  MQ消费端用漏桶 → "不管来多少，我只按1000 QPS消费"
  适合出口：保护脆弱的下游

滑动窗口 = 精确计量（统计最近1秒的真实QPS）
  Sentinel用滑动窗口 → "最近1秒QPS是950，没超限"
  适合中间层：精确限流

架构师决策：
  - 网关层 → 令牌桶（允许正常突发，超限才拒）
  - 服务间 → 滑动窗口（精确控制QPS）
  - 下游保护 → 漏桶（匀速消费，保护DB/外部API）
```
