# Day 5：微服务实战综合 — 分布式事务选型·API网关全貌·容量规划·故障演练·架构演进（练习整理）

## 题目一：微服务分布式事务 — TCC/Saga/消息最终一致性怎么选

### 三种分布式事务方案的本质区别

```
分布式事务的本质：跨服务如何保证数据一致性

  TCC (Try-Confirm-Cancel)          Saga (编排/协调)            消息最终一致性
  ┌──────────────────────┐       ┌──────────────────────┐     ┌──────────────────────┐
  │ 预留→确认/取消        │       │ 正向→补偿             │     │ 本地事务→消息→消费     │
  │                      │       │                      │     │                      │
  │ Try:    冻结资源     │       │ T1→T2→T3→T4          │     │ 1. 本地表写入         │ │
  │ Confirm: 提交资源    │       │  ↓ 失败              │     │ 2. 发MQ消息           │
  │ Cancel: 释放资源     │       │ C4→C3→C2→C1          │     │ 3. 消费者幂等处理     │
  │                      │       │                      │     │                      │
  │ 强一致性             │       │ 最终一致性            │     │ 最终一致性            │
  │ 侵入性最强           │       │ 侵入性中等            │     │ 侵入性最低            │
  └──────────────────────┘       └──────────────────────┘     └──────────────────────┘

  一致性强度：TCC > Saga > 消息最终一致性
  侵入程度：  TCC > Saga > 消息最终一致性
  实现复杂度：TCC > Saga > 消息最终一致性
```

### 选型决策框架

```
三个维度决定选型：

  维度1：一致性要求
    ├─ 强一致（金融转账、资金扣减）        → TCC
    ├─ 最终一致 + 可补偿（订单流程）        → Saga
    └─ 最终一致 + 可异步（通知、日志）      → 消息最终一致性

  维度2：业务特征
    ├─ 隔离性要求高（不能超卖/超扣）        → TCC（Try阶段预留资源）
    ├─ 长流程多步骤（下单→库存→支付→物流） → Saga（编排清晰、补偿明确）
    └─ 单向通知（支付成功→通知发货）        → 消息最终一致性

  维度3：团队与成本
    ├─ 团队强 + 金融场景 + 可投入          → TCC
    ├─ 团队中等 + 电商场景 + 可补偿        → Saga
    └─ 团队一般 + 非核心链路 + 低成本      → 消息最终一致性

  面试核心句："不是选最好的一致性，而是选刚好够用的。金融场景用TCC因为钱不能错，
            电商下单用Saga因为可以补偿退货，支付回调用消息因为通知不需要同步"
```

### TCC实战 — 蛋壳钱包转账场景

```
场景：A账户转100元到B账户

  ┌─ Try阶段 ─────────────────────────────────────────┐
  │  A账户：余额1000 → 冻结100 → 可用余额900           │
  │  B账户：不操作                                      │
  │                                                     │
  │  SQL：UPDATE account SET frozen = frozen + 100      │
  │       WHERE id = A AND balance - frozen >= 100      │
  │                                                     │
  │  关键：Try是"预留"不是"扣减"，余额不变，冻结增加     │
  └─────────────────────────────────────────────────────┘
            │
            ├── 全部Try成功 ──→ Confirm
            │   ┌─ Confirm阶段 ─────────────────────────┐
            │   │  A账户：余额900+冻结100 → 余额900      │
            │   │        冻结100 → 冻结0                  │
            │   │  B账户：余额500 → 余额600               │
            │   │                                         │
            │   │  SQL A：UPDATE account SET              │
            │   │        balance = balance - frozen,      │
            │   │        frozen = 0 WHERE id = A          │
            │   │  SQL B：UPDATE account SET              │
            │   │        balance = balance + 100          │
            │   │        WHERE id = B                     │
            │   │                                         │
            │   │  关键：Confirm必须幂等，网络重试不能重复扣 │
            │   └─────────────────────────────────────────┘
            │
            └── 任一Try失败 ──→ Cancel
                ┌─ Cancel阶段 ─────────────────────────┐
                │  A账户：冻结100 → 冻结0               │
                │  B账户：不操作                          │
                │                                        │
                │  SQL：UPDATE account SET               │
                │       frozen = frozen - 100            │
                │       WHERE id = A                     │
                │                                        │
                │  关键：Cancel也必须幂等                  │
                └────────────────────────────────────────┘

  TCC的三大陷阱：
    1. 空回滚：Try没到但收到Cancel → 检查Try是否执行过
    2. 幂等：Confirm/Cancel可能重试 → 全局事务ID+状态表去重
    3. 悬挂：Cancel比Try先到 → Cancel记录事务ID，Try检查是否已Cancel
```

### Saga编排模式 — 电商下单场景

```
场景：下单→扣库存→扣优惠券→支付

  Saga编排器（State Machine）：

  正向流程：
  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
  │ 创建订单│───→│ 扣库存  │───→│ 扣优惠券│───→│ 支付    │
  │ T1     │    │ T2     │    │ T3     │    │ T4     │
  └────────┘    └────────┘    └────────┘    └────────┘

  补偿流程（T4支付失败）：
  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
  │ 取消订单│←───│ 恢复库存│←───│ 恢复优惠券│←───│ 退款    │
  │ C1     │    │ C2     │    │ C3     │    │ C4     │
  └────────┘    └────────┘    └────────┘    └────────┘

  Saga vs TCC核心区别：
    TCC：Try预留 → 隔离性靠冻结字段保证
    Saga：直接提交 → 隔离性靠补偿保证（脏读风险）

  Saga的脏读问题：
    T2扣库存成功 → 此时其他人看到库存少了
    T3扣优惠券失败 → C2恢复库存
    中间状态：库存被短暂多扣了

  解法：
    1. 业务容忍：电商场景可接受短暂脏读（库存多扣→补偿恢复）
    2. 语义锁：库存加"预扣"状态，查询时过滤
```

### 消息最终一致性 — 支付成功通知发货

```
场景：支付服务支付成功 → 通知订单服务发货

  方案1：本地消息表（推荐）
  ┌─────────────────────────────────────────────────┐
  │  支付服务                                         │
  │                                                   │
  │  1. 开启本地事务                                   │
  │  2. UPDATE payment SET status = 'PAID'            │
  │  3. INSERT message_table (id, topic, content)     │
  │     -- 同一事务，保证支付和消息原子性              │
  │  4. COMMIT                                        │
  │                                                   │
  │  5. 异步线程：扫描message_table → 发MQ → 标记已发  │
  └─────────────────────────────────────────────────┘
          │ MQ
          ▼
  ┌─────────────────────────────────────────────────┐
  │  订单服务                                         │
  │                                                   │
  │  1. 消费MQ消息                                    │
  │  2. 查询支付状态（幂等校验）                       │
  │  3. UPDATE order SET status = 'SHIPPING'          │
  │  4. ACK                                          │
  └─────────────────────────────────────────────────┘

  方案2：RocketMQ事务消息
  ┌─────────────────────────────────────────────────┐
  │  1. 发送Half消息（对消费者不可见）                 │
  │  2. 执行本地事务（支付状态更新）                   │
  │  3. COMMIT → 消息对消费者可见                     │
  │     ROLLBACK → 消息删除                           │
  │  4. 回查：本地事务超时未确认 → Broker查本地事务状态 │
  └─────────────────────────────────────────────────┘

  两种方案对比：
    本地消息表：通用性强（任何MQ都行），多一次DB写入，需定时扫描
    事务消息：  依赖RocketMQ，无需额外表，Broker自带回查
```

### 架构师级回答模板

```
"我们项目根据业务特征选了不同方案：
  - 蛋壳钱包转账用TCC：金融场景强一致，Try阶段冻结资金保证隔离性
  - 电商下单用Saga编排：长流程可补偿，库存短暂脏读可接受
  - 支付回调发货用消息最终一致性：单向通知不需要同步等待

  三个方案的取舍逻辑：
  一致性够用就行，不是越强越好——TCC的侵入性导致开发成本是Saga的2倍、
  是消息方案的4倍。金融场景值得这个成本，电商不需要。"
```

## 题目二：API网关架构全貌 — 从请求入口到服务调用的完整设计

### API网关的六大职责

```
客户端请求
    │
    ▼
┌─────────────────────────────────────────────────────┐
│                    API Gateway                       │
│                                                      │
│  ① 协议转换    HTTP↔gRPC、WebSocket→MQ              │
│  ② 认证鉴权    JWT校验/限流/黑名单                    │
│  ③ 限流熔断    令牌桶+Sentinel熔断                    │
│  ④ 灰度路由    按Header/权重/标签路由                 │
│  ⑤ 聚合裁剪    BFF模式多服务聚合、响应裁剪            │
│  ⑥ 日志监控    请求日志/访问指标/链路追踪起点         │
│                                                      │
└─────────────────────────────────────────────────────┘
    │
    ▼
  微服务集群
```

### 网关限流的三层设计

```
三层限流：从粗到细

  L1 网关层 — 全局限流
  ┌───────────────────────────────────────┐
  │  令牌桶：全局限流 10000 QPS           │
  │  滑动窗口：单IP限流 100 QPS/min      │
  │  黑名单：恶意IP直接拒绝               │
  │                                       │
  │  为什么用令牌桶？                     │
  │  入口流量有突发（秒杀开抢），          │
  │  令牌桶允许短暂突发 > 平均速率        │
  └───────────────────────────────────────┘
    │ 通过的流量
    ▼
  L2 服务层 — 服务级限流
  ┌───────────────────────────────────────┐
  │  Sentinel滑动窗口                     │
  │  订单服务：2000 QPS                   │
  │  支付服务：1000 QPS                   │
  │  库存服务：3000 QPS                   │
  │                                       │
  │  为什么用滑动窗口？                   │
  │  服务层需要统计RT/异常率，             │
  │  滑动窗口是多维度统计引擎             │
  └───────────────────────────────────────┘
    │ 通过的流量
    ▼
  L3 接口层 — 热点参数限流
  ┌───────────────────────────────────────┐
  │  Sentinel热点参数限流                  │
  │  单商品ID：100 QPS（防羊毛党）        │
  │  单用户ID：5 QPS（防刷单）            │
  │                                       │
  │  为什么需要热点参数？                 │
  │  前两层限不住"对同一个热点商品的攻击" │
  │  秒杀商品 itemId=123 → 10万请求全打到 │
  │  库存服务的这一个接口上               │
  └───────────────────────────────────────┘

  流量漏斗：
  10万QPS → L1剩1万 → L2剩3000 → L3剩100
  每层挡90%+，最终到DB的请求可控
```

### 网关认证方案 — JWT + RefreshToken

```
认证流程：

  客户端                    网关                    认证服务
    │                       │                        │
    │ 1. POST /login        │                        │
    │ ──────────────────────→ ──────────────────────→ │
    │                       │                        │
    │ 2. 返回AccessToken(15min) + RefreshToken(7天)   │
    │ ←────────────────────── ←────────────────────── │
    │                       │                        │
    │ 3. 请求 + AccessToken │                        │
    │ ──────────────────────→                        │
    │                       │                        │
    │    4. 校验JWT          │                        │
    │       ├─ 有效 → 放行   │                        │
    │       ├─ 过期 → 401   │                        │
    │       └─ 版本号检查     │                        │
    │          (支持踢人)    │                        │
    │                       │                        │
    │ 5. AccessToken过期     │                        │
    │    用RefreshToken换新  │                        │
    │ ──────────────────────→ ──────────────────────→ │
    │                       │     校验RefreshToken    │
    │ ←────────────────────── ←────────────────────── │
    │   新AccessToken       │                        │

  JWT主动踢人方案：
    Payload加version字段，用户改密码/登出时version+1
    网关校验时查Redis中的version，不匹配则拒绝
    无需等待Token自然过期，秒级生效

  网关层认证 vs 服务层认证：
    网关：统一入口认证（JWT校验+限流），挡住未认证请求
    服务：mTLS服务间认证 + RBAC细粒度鉴权
    不重复：网关管"你是谁"，服务管"你能做什么"
```

### BFF聚合模式

```
场景：移动端首页需要 用户信息+推荐商品+购物车数量

  无BFF：
  客户端 → 网关 → 用户服务  (200ms)
  客户端 → 网关 → 商品服务  (300ms)
  客户端 → 网关 → 购物车服务 (100ms)
  总耗时：600ms + 3次网络往返

  有BFF：
  客户端 → 网关/BFF ──→ 用户服务    (200ms)
                   ├──→ 商品服务    (300ms)  ← 并发调用
                   └──→ 购物车服务  (100ms)
  总耗时：300ms + 1次网络往返

  BFF两种形态：
    1. 网关内聚合：Spring Cloud Gateway + 自定义Filter
       适合简单聚合（2-3个服务）
    2. 独立BFF服务：Node.js/GraphQL
       适合复杂聚合（多个端、大量字段裁剪）

  架构师决策：
    ≤3个服务聚合 → 网关Filter内做，少维护一个服务
    >3个服务 / 多端差异大 → 独立BFF，网关只做路由
```

### 网关高可用架构

```
                    ┌──── Nginx/LB ────┐
                    │                   │
              ┌─────▼─────┐     ┌──────▼────┐
              │ Gateway-1 │     │ Gateway-2 │
              │  (Active)  │     │  (Active)  │
              └─────┬─────┘     └──────┬────┘
                    │                   │
              ┌─────▼───────────────────▼────┐
              │        Nacos 服务发现         │
              └──────────────────────────────┘

  高可用三重保障：
    1. 多实例无状态部署（网关不存会话，随便切）
    2. Nacos健康检查自动剔除故障节点
    3. Nginx上游健康检查做最外层兜底

  限流数据存Redis：
    单机限流：内存（快但不一致，网关挂了限流失效）
    集群限流：Redis + Lua脚本（一致但多1ms延迟）
    架构师选择：核心链路（秒杀）用Redis集群限流，
               普通接口用单机限流 + 熔断兜底
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 网关做路由转发和JWT认证 |
| 高级开发 | 网关有限流、认证、灰度功能 |
| 架构师 | 六大职责的完整设计 + 三层限流漏斗 + JWT踢人方案 + BFF聚合决策 + 高可用三重保障。每个设计决策都能说清"为什么"——入口用令牌桶因为有突发、服务层用滑动窗口因为要多维统计、核心链路用Redis集群限流因为不能漏 |

**差距**：网关的六大职责可能能说出3-4个，但三层限流的"每层为什么用不同算法"这个根因大概率说不清。BFF聚合决策也容易忽视。要记住：**网关不是"加个Spring Cloud Gateway"，是流量治理的第一道防线**。

## 题目三：微服务容量规划 — 从压测到扩容的完整方法论

### 容量规划的四步法

```
Step1：基线建立 — 系统能承载多少？

  压测工具：JMeter / Gatling / Wrk
  压测策略：阶梯加压（不是一上来就10万QPS）

  ┌────────────────────────────────────────┐
  │  QPS                                    │
  │  15000 ┤                      ╱ ← 系统崩溃 │
  │  12000 ┤                ╱─────           │
  │  10000 ┤           ╱──← 安全水位线      │
  │   8000 ┤       ╱──│     (70-80%峰值)   │
  │   5000 ┤   ╱───│                       │
  │   2000 ┤──│    │                        │
  │      0 ┼──┴────┴──────────────── RT     │
  │        50ms  200ms  1s    5s    10s     │
  │                                          │
  │  关键数据点：                             │
  │  - 最大稳定QPS：10000（RT<200ms）        │
  │  - 安全水位线：7000-8000（70-80%）       │
  │  - 破裂点：12000（RT开始飙升）           │
  └────────────────────────────────────────┘

  每个服务的基线：
  ┌──────────┬───────────┬──────────┬──────────┐
  │ 服务     │ 单机QPS   │ 安全水位 │ 破裂点   │
  ├──────────┼───────────┼──────────┼──────────┤
  │ 网关     │ 15000     │ 12000    │ 20000    │
  │ 订单服务 │ 2000      │ 1600     │ 3000     │
  │ 库存服务 │ 3000      │ 2400     │ 4500     │
  │ 支付服务 │ 1000      │ 800      │ 1500     │
  └──────────┴───────────┴──────────┴──────────┘

Step2：流量预估 — 未来会有多少？

  流量来源分析：
    常规流量：日均QPS = DAU × 每用户请求量 / 86400
    峰值流量：日均 × 峰值倍数（电商3-5倍，秒杀100倍）

  例子：电商日活100万
    日均QPS = 100万 × 20次 / 86400 ≈ 230 QPS
    峰值QPS = 230 × 5 = 1150 QPS
    秒杀峰值 = 230 × 100 = 23000 QPS

Step3：容量计算 — 需要多少实例？

  公式：实例数 = 目标QPS / 单机安全水位 × 冗余系数

  例子：订单服务应对峰值1150 QPS
    实例数 = 1150 / 1600 × 1.3 ≈ 1 → 最少2台（高可用）
    秒杀场景：1150 / 1600 × 1.3 + 售罄标记 = 2台够了

  冗余系数为什么是1.3？
    不是1.5也不是1.1——
    1.5浪费资源（多50%实例空转）
    1.1扛不住突发（一个实例挂了就过载）
    1.3：一台挂了还有30%余量 → 标准实践

Step4：扩容预案 — 什么时候扩？怎么扩？

  扩容触发条件：
    CPU > 70% 持续5分钟 → HPA自动扩
    QPS > 安全水位线80%  → 预警，准备扩容
    大促前3天            → 预扩容到预估峰值

  扩容方式：
    纵向扩容：升配置（2核→4核），快但有限
    横向扩容：加实例，无上限但需要无状态设计
    波峰波谷：K8s HPA + CronHPA
      白天8台 → 晚上3台 → 大促20台
```

### 全链路压测方法论

```
单服务压测 vs 全链路压测：

  单服务压测（日常）：
    隔离环境 → 只压一个服务 → 找出单机基线
    问题：不知道服务间调用链路的瓶颈

  全链路压测（大促前必做）：
    影子库 + 染色标记 → 压测流量走完整链路 → 真实瓶颈

  全链路压测的三个关键：
    1. 数据隔离：压测流量写影子库，不影响线上数据
    2. 链路染色：HTTP Header压测标记，全链路透传
    3. 监控分离：压测指标和线上指标分开看

  全链路压测发现的典型瓶颈：
    ┌──────────┬──────────────┬─────────────────────┐
    │ 瓶颈位置 │ 现象         │ 解法                │
    ├──────────┼──────────────┼─────────────────────┤
    │ 网关     │ QPS上不去    │ 增加网关实例/优化Filter │
    │ 服务A    │ RT飙升       │ 优化慢查询/加缓存     │
    │ Redis    │ 连接池满     │ 增大连接池/分片       │
    │ MySQL    │ 慢查询       │ 加索引/读写分离       │
    │ 线程池   │ 任务堆积     │ 调大线程池/异步化     │
    └──────────┴──────────────┴─────────────────────┘
```

### 容量规划与成本优化

```
架构师的容量账本：

  每月成本 = 实例数 × 单价 × 730小时

  例子：订单服务
    常态：2台 × 4核8G × ¥0.5/时 = ¥730/月
    峰值：5台 × 4核8G × ¥0.5/时 = ¥1825/月
    弹性：常态2台 + 峰值自动扩到5台 = ¥730 + 按需

  成本优化三招：
    1. 弹性伸缩：常态低配+峰值自动扩，省60%成本
    2. 请求分类：核心链路高配+非核心低配，资源倾斜
    3. 缓存优化：命中率从80%→95%，后端容量需求降75%
       （10000 QPS × 20%穿透 = 2000到DB，
        10000 QPS × 5%穿透 = 500到DB）
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 做过JMeter压测，单机能扛2000 QPS |
| 高级开发 | 有容量评估意识，知道要留冗余 |
| 架构师 | 四步法闭环（基线→预估→计算→扩容），全链路压测三要素（数据隔离+链路染色+监控分离），每个数字有依据——安全水位70-80%、冗余系数1.3、峰值倍数按业务定。成本意识：弹性伸缩省60%、缓存命中率提升减少后端容量75% |

**差距**：容量规划很容易只停留在"我压测过"，但缺少完整的四步闭环。尤其是"安全水位线为什么是70-80%"和"冗余系数为什么是1.3"这两个数字的根因。面试时说"单机QPS 2000"不够，要说"安全水位1600，部署2台冗余1.3，可承载峰值2000 QPS"。

## 题目四：故障演练与应急体系 — Chaos Engineering实战与故障预案

### 故障演练的价值

```
为什么需要故障演练？

  真实故障的成本：
    ┌────────────────────────────────────────┐
    │  P0故障：损失10万/分钟                  │
    │  平均恢复时间(MTTR)：30分钟             │
    │  单次故障损失：300万                    │
    │                                        │
    │  故障演练后：                           │
    │  MTTR：30分钟 → 5分钟                  │
    │  单次故障损失：300万 → 50万             │
    │  演练成本：0.5人天/月                   │
    │  ROI：300万差价 / 演练成本 = 极高       │
    └────────────────────────────────────────┘

  没演练过的应急预案 = 废纸
  因为：
    1. 预案写了没人看 → 真出事找不到
    2. 步骤过时了 → 代码改了预案没更新
    3. 依赖的工具挂了 → 应急脚本依赖S3，S3也挂了
```

### Chaos Engineering实战方法

```
故障演练四步法：

  Step1：定义稳态假设
    "订单服务在2000 QPS下RT<200ms，错误率<0.1%"
    如果没有明确的稳态指标，演练无意义

  Step2：设计故障注入
    ┌─────────────────────────────────────────────┐
    │  故障类型          │ 注入方式                │
    ├─────────────────────┼───────────────────────┤
    │  服务宕机          │ kill -9 进程            │
    │  网络延迟          │ tc netem delay 500ms    │
    │  网络丢包          │ tc netem loss 30%       │
    │  DNS故障           │ 修改/etc/hosts          │
    │  Redis挂了         │ iptables屏蔽6379端口    │
    │  MySQL慢查询       │ SELECT SLEEP(5)         │
    │  磁盘满            │ dd if=/dev/zero of=big  │
    │  CPU飙升           │ stress-ng --cpu 8       │
    └─────────────────────┴───────────────────────┘

  Step3：观察系统反应
    可观测性三位一体（Day4）在这里派上用场：
    Metrics：错误率/RT/QPS变化
    Tracing：请求走向是否如预期
    Logging：是否有预期之外的异常

  Step4：验证与修复
    ├─ 系统自动恢复了？→ 好，限流/熔断生效
    ├─ 系统降级了但没挂？→ 好，降级策略生效
    ├─ 系统挂了但恢复快？→ 好，预案有效
    └─ 系统挂了且恢复慢？→ 找问题，修预案，再演练

  演练频率：
    核心链路：每月1次
    全链路演练：每季度1次
    新服务上线：上线后2周内演练
```

### 故障预案体系

```
故障分级与响应：

  ┌────────┬──────────────────┬───────────┬─────────────┐
  │ 等级   │ 定义             │ 响应时间  │ 通知方式     │
  ├────────┼──────────────────┼───────────┼─────────────┤
  │ P0     │ 核心链路完全不可用│ 5分钟介入 │ 电话+短信    │
  │ P1     │ 核心链路部分降级  │ 15分钟    │ 短信+IM      │
  │ P2     │ 非核心功能异常    │ 1小时     │ IM           │
  │ P3     │ 体验问题/优化建议 │ 下个工作日│ 工单         │
  └────────┴──────────────────┴───────────┴─────────────┘

  故障处理SOP（5步法）：
  ┌──────────────────────────────────────────────┐
  │  1. 止血（0-5分钟）                           │
  │     限流/降级/重启/回滚                        │
  │     目标：先恢复服务，不管根因                  │
  │                                               │
  │  2. 通告（5-10分钟）                           │
  │     故障群通告：影响面+当前状态+预计恢复时间     │
  │                                               │
  │  3. 定位（10-20分钟）                          │
  │     Metrics→Tracing→Logging（Day4三位一体SOP） │
  │                                               │
  │  4. 修复（20-30分钟）                          │
  │     临时修复 → 确认恢复 → 持续观察             │
  │                                               │
  │  5. 复盘（故障后24-48小时）                    │
  │     5Whys分析 + 改进措施 + 跟进到完成          │
  └──────────────────────────────────────────────┘

  止血的优先级决策树：
    能回滚？ → 回滚（最快，5分钟内恢复）
    不能回滚？ → 能降级？ → 降级（关闭非核心功能）
    不能降级？ → 能限流？ → 限流（保核心用户）
    不能限流？ → 重启（赌一把）
```

### 典型故障预案模板

```
预案：MySQL主库宕机

  触发条件：MySQL主库心跳超时3次 / 主库进程不存在

  止血步骤：
  ┌───────────────────────────────────────────┐
  │  1. 确认主库状态（0-1分钟）                │
  │     mysql -h master -e "SELECT 1"          │
  │                                           │
  │  2. 决策：能否快速恢复？（1-3分钟）        │
  │     ├─ 进程还在/网络抖动 → 等待恢复       │
  │     └─ 进程没了/磁盘坏了 → 主从切换       │
  │                                           │
  │  3. 主从切换（3-5分钟）                    │
  │     a. 确认从库同步延迟 < 5秒             │
  │     b. 停止从库复制：STOP SLAVE           │
  │     c. 提升从库为主：RESET SLAVE ALL      │
  │     d. 修改应用连接指向新主库              │
  │     e. 验证：新主库可读写                  │
  │                                           │
  │  4. 通告恢复（5分钟）                      │
  │     故障群通告：MySQL已切换，请验证业务    │
  └───────────────────────────────────────────┘

  预防措施：
    - 半同步复制（至少1个从库确认才返回）
    - MHA/ORC自动故障转移
    - 定期演练主从切换（每月1次）
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 出了问题看日志排查 |
| 高级开发 | 有监控告警，出问题能定位 |
| 架构师 | 故障演练四步法（稳态假设→注入→观察→验证），故障处理5步SOP（止血→通告→定位→修复→复盘），每步有明确时间要求。止血决策树：能回滚→回滚，不能回滚→降级→限流→重启。P0故障5分钟MTTR目标，300万损失降到50万 |

**差距**：故障演练和应急预案很容易停留在"我知道要做"，但缺少结构化方法。面试时被问"线上出问题了你怎么处理"，要说5步SOP而不是"看监控看日志"。核心句：**"先止血再治病"**——5分钟内必须恢复服务，根因后面再查。

## 题目五：微服务架构演进路线 — 从单体到云原生的渐进式演进

### 架构演进的三个阶段

```
阶段1：单体 → 服务化（0-6个月）

  ┌──────────────────────────────────┐
  │         单体应用                  │
  │  ┌─────┐ ┌─────┐ ┌─────┐       │
  │  │用户 │ │订单 │ │商品 │       │
  │  │模块 │ │模块 │ │模块 │       │
  │  └──┬──┘ └──┬──┘ └──┬──┘       │
  │     └────────┼────────┘         │
  │            ┌─▼──┐               │
  │            │ DB │               │
  │            └────┘               │
  └──────────────────────────────────┘
         │
         │ 第一步：先拆最痛的
         ▼
  ┌──────────────────────────────────┐
  │  哪个模块最痛就先拆哪个：         │

  │  不要一上来就全拆！               │
  │  单体 → 先拆1-2个服务 → 验证     │

  └──────────────────────────────────┘

阶段2：服务化 → 微服务化（6-18个月）

  ┌─────────────────────────────────────────────┐

  │                                               │
  │  先有基建再拆，不是拆了再补基建                │
  └─────────────────────────────────────────────┘

阶段3：微服务 → 云原生（18-36个月）

  ┌─────────────────────────────────────────────┐
  │  云原生不只是"上K8s"：                       │
  │                                               │

  │  ┌───────┐    ┌────────┐    ┌──────────┐     │

  │  │自动构建│    │声明式  │    │智能运维  │     │

  │                                               │
```

### 演进中的反模式

```
❌ 反模式1：一步到位

  单体 → 直接拆20个微服务
  结果：没有基建支撑，排障困难，服务间调用混乱
  正确：先拆1-2个验证 → 补基建 → 再拆

❌ 反模式2：为了拆而拆

  "我们都用微服务了" → 3个人的团队拆了10个服务
  结果：维护成本 >> 开发效率提升
  正确：团队人数 < 5人不建议微服务，模块化单体更合适
  康威定律：系统架构 ≈ 组织沟通结构

❌ 反模式3：只拆不管

  拆了服务但没加限流/熔断/监控
  结果：一个服务挂了连锁故障，比单体还惨
  正确：先有Day2-Day4的治理体系，再拆服务

❌ 反模式4：数据不拆

  服务拆了但共享同一个DB
  结果：DB成为单点，服务独立性是假的
  正确：每个服务有独立DB（至少独立Schema），数据通过API交互
```

### 架构演进决策框架

```
要不要微服务？看四个条件：

  ┌──────────────────────────────────────────────┐
  │  条件1：团队规模 ≥ 5人（否则模块化单体足够）  │
  │  条件2：部署频率 ≥ 每天1次（否则拆了没收益）  │
  │  条件3：有独立运维能力（否则拆了更痛苦）      │
  │  条件4：有基建投入预算（否则治理跟不上）      │
  │                                               │
  │  4个条件全满足 → 微服务                        │
  │  满足2-3个 → 选定服务化（部分拆分）           │
  │  满足0-1个 → 模块化单体                       │
  └──────────────────────────────────────────────┘

  面试核心句："微服务不是目的，是解决特定问题的手段。
  单体扛不住的痛点是什么？拆了能不能解决？基建跟不跟得上？
  三个问题都想清楚了再动手。"
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 现在都用微服务了 |
| 高级开发 | 微服务有注册中心、网关、配置中心 |
| 架构师 | 三阶段演进路线（单体→服务化→云原生），每个阶段做什么、不做什么。四大反模式（一步到位/为拆而拆/只拆不管/数据不拆）。决策框架：团队规模×部署频率×运维能力×基建预算。康威定律：系统架构≈组织沟通结构。核心判断：微服务不是目的，是手段 |

**差距**：面试中很容易说"我们用微服务"，但被追问"为什么要从单体迁到微服务"时说不清根因。架构师要能说清：**痛点是什么→拆了能不能解决→基建跟不跟得上**，而不是"大家都用微服务"。
