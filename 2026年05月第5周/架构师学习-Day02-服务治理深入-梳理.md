# Day 2：服务治理深入 — 架构师角度梳理

## 核心定位

服务治理不是"加个Sentinel注解"，是**在故障发生前预埋防御，故障发生时自动降级，故障恢复后自动探测**。架构师要回答的不是"要不要限流"，而是"每个位置用什么算法、什么参数、什么降级策略"。

## 限流算法：选择依据是谁在保护谁

### Sentinel滑动窗口的本质：统计引擎，不是限流引擎

```
常见误解：Sentinel用滑动窗口做限流
正确理解：Sentinel用滑动窗口做统计，用规则做限流，两者是独立模块

  滑动窗口（LeapArray）的职责：
    - 统计最近1秒的pass/block/exception/rt
    - 提供给限流/熔断/系统保护做判断依据
    - 用环形数组 + 时间窗口过期重置实现"滑动"

  限流/熔断的职责：
    - 拿到统计数据 → 判断是否超阈值 → 决定放行还是拒绝
    - 限流模式：直接拒绝 / 排队等待 / 预热
    - 熔断模式：慢调用比例 / 异常比例 / 异常数

  为什么不直接用令牌桶？
    令牌桶只能统计"令牌数"，无法统计"RT"、"异常率"等多维度
    滑动窗口天然多维度 → 一个统计引擎服务多种规则
```

### 限流算法的本质差异：突发容忍度

```
突发容忍度从高到低：

  令牌桶（允许突发）
    → 桶里攒了令牌，瞬间消费完
    → 适合入口：正常流量允许短时突发
    → 网关层限流首选

  滑动窗口（精确计量）
    → 统计真实QPS，超限拒绝
    → 适合中间层：精确控制每秒调用量
    → Sentinel默认方式

  漏桶（拒绝突发）
    → 不管来多少，匀速出
    → 适合出口：保护脆弱下游
    → MQ消费端本质就是漏桶

  固定窗口（有2倍突发漏洞）
    → 不推荐用于生产
    → 只在精度要求极低的场景使用

  架构师决策框架：
    入口 → 令牌桶（容忍突发）
    中间 → 滑动窗口（精确控制）
    出口 → 漏桶（保护下游）
```

## 熔断器：不是开关，是状态机

### 三种熔断策略对应三种故障模式

```
故障模式              熔断策略          典型场景
─────────────────────────────────────────────────
下游变慢但没挂        慢调用比例        DB慢查询、GC停顿
下游偶发异常          异常比例          第三方API不稳定
下游必挂              异常数            服务OOM/进程挂了

为什么不全用异常比例？
  - 下游偶尔超时但最终成功 → RT长但不算异常 → 慢调用比例更准
  - 下游直接挂了 → 每次都异常 → 异常数比比例更灵敏（不用等分母积累）
  - 架构师要按故障模式选策略，不是一刀切
```

### 熔断参数的架构师级理解

```
timeWindow（熔断时长）的决策逻辑：

  问：下游恢复要多久？
    几秒 → timeWindow = 10s（默认值够用）
    需要重启 → timeWindow = 30-60s
    需要人工介入 → timeWindow = 60s + 告警

  问：降级对业务影响多大？
    影响小（推荐/通知）→ timeWindow 可以长
    影响大（支付/下单）→ timeWindow 要短，但HALF_OPEN探测要谨慎

  minRequestAmount（最小请求数）的决策逻辑：

    QPS < 10的服务 → minRequestAmount = 5
    QPS > 100的服务 → minRequestAmount = 10-20
    防止低QPS时1-2次偶然失败就熔断（误杀）

  slowRatioThreshold（慢调用比例）的决策逻辑：

    核心服务 → 0.3-0.5（30%-50%慢了就熔断，保护性强）
    非核心服务 → 0.5-0.8（容忍度更高）
```

## 降级策略：不是返回默认值，是业务连续性保障

### 降级策略的选择矩阵

```
                    实时性要求
                 低           高
              ┌────────┬────────┐
         允许  │ 兜底值  │ 缓存   │
  数据     │        │        │
  过期     ├────────┼────────┤
         不允许 │ 异步化  │ 备链路 │
              └────────┴────────┘

  架构师决策路径：
    1. 这个数据能接受过期吗？
       能 → 兜底值或缓存
       不能 → 异步化或备链路

    2. 用户需要立刻看到结果吗？
       需要 → 缓存或备链路
       不需要 → 兜底值或异步化

    3. 值得建备链路吗？
       核心业务 → 值得（支付通道A/B切换）
       非核心 → 不值得（推荐"暂无推荐"就行）
```

### 降级三层架构的设计原则

```
L1 自动降级（代码层，毫秒级）
  原则：覆盖90%常见故障，零人工介入
  实现：@SentinelResource fallback
  条件：降级逻辑是确定性的（缓存/兜底值）

L2 配置开关（Nacos配置，秒级）
  原则：覆盖9%需要人工判断的场景
  实现：@Value配置开关 + Nacos动态推送
  条件：降级逻辑需要人工决策（切备库、改TTL）

L3 人工预案（SOP文档，分钟级）
  原则：覆盖1%极端场景
  实现：运维手册 + 值班SOP
  条件：需要人工操作（DBA切库、运维扩容）

  三层的本质区别：
    L1：代码替你做决策 → 快但不够灵活
    L2：配置替你做决策 → 灵活但需要人触发
    L3：人做决策 → 最灵活但最慢

    不是替代关系，是叠加：L1自动降级了，L2/L3还可能需要介入
```

## 隔离策略：性能与安全的平衡

### 为什么Sentinel不用线程池隔离

```
Hystrix线程池隔离的开销：
  每个Command一个线程池
  100个Command = 100个线程池
  每个线程池默认10个线程 = 1000个线程
  线程上下文切换 + 队列内存 ≈ 每次调用额外1-3ms RT

  在高并发场景下：
    1万QPS × 3ms额外开销 = 30秒的CPU时间/秒
    而且ThreadLocal传递有问题（TransmittableThreadLocal补救）

Sentinel的替代方案：
  信号量隔离（线程数限流） + 熔断器
  - 信号量：限制并发数，零额外线程
  - 熔断器：异常时快速失败，不需要线程中断
  - 组合效果 ≈ 线程池隔离，但开销小一个数量级

  什么时候还是需要线程池隔离？
    - 调用外部不可控服务（第三方API），RT完全不可预测
    - 此时用CompletableFuture + 自定义线程池，而不是Hystrix
```

### 线程数限流 vs QPS限流的选择

```
核心公式：并发线程数 ≈ QPS × 平均RT

  场景A：RT=10ms，QPS=100
    并发线程 ≈ 100 × 0.01 = 1
    用QPS限流就行，线程数几乎不会成为瓶颈

  场景B：RT=1s，QPS=50
    并发线程 ≈ 50 × 1 = 50
    看似QPS不高，但50个线程卡着 → 需要线程数限流

  场景C：RT波动大，平时10ms，故障时5s
    平时并发 ≈ 1，故障时并发 ≈ 250
    QPS限流看不出问题 → 线程数限流能在并发飙升时触发

  架构师决策：
    RT稳定且低（< 100ms）→ QPS限流
    RT高或波动大 → 线程数限流
    不确定 → 两者都配，线程数限流做兜底
```

## 秒杀系统容错：五层漏斗

### 漏斗模型的核心思想

```
每一层的职责不是"限流"，而是"过滤无效请求"

  L1 CDN/WAF：过滤恶意流量（90%）
    判断标准：不是正常用户 → 拒绝
    特点：不消耗后端资源

  L2 网关：过滤溢出流量（90%）
    判断标准：超出系统容量 → 拒绝
    特点：消耗极少量网关资源

  L3 服务层：过滤无效业务请求（90%）
    判断标准：不满足业务条件 → 拒绝
    特点：消耗Redis资源，不消耗DB资源

  L4 调用链路：异常时降级保护（90%）
    判断标准：下游不可用 → 降级
    特点：消耗MQ资源，不消耗核心服务资源

  L5 数据层：最后兜底
    判断标准：全部正常 → 放行
    特点：消耗DB资源，但请求量已极小

  每层挡90%的结果：
    10万QPS → 1万 → 1000 → 100 → 10 → 1
    最终DB只需要处理极少量真实有效的库存扣减
```

### 售罄标记是最强优化

```
为什么售罄标记比限流更重要？

  限流的问题：
    - 限流阈值设多少？设少了浪费流量，设多了打挂系统
    - 售罄后还在限流范围内 → 请求继续进来 → 继续消耗资源

  售罄标记的优势：
    - 库存扣到0 → 设置Redis标记
    - 后续请求 → 检查标记 → 直接返回"已售罄"
    - 从"限流"变成"直接拒绝"，资源消耗趋近于零

  工程细节：
    // 设置售罄标记（在Redis库存扣减后）
    if (remainStock <= 0) {
        redis.set("seckill:soldout:" + itemId, "1", "EX", 3600);
    }

    // 检查售罄（在网关或服务入口最前面）
    // 用pipeline批量检查，减少RT
    if ("1".equals(redis.get("seckill:soldout:" + itemId))) {
        return SeckillResult.soldOut();
    }

  售罄标记 + 限流的关系：
    限流 = 活动期间的流量控制
    售罄 = 活动结束的快速退出
    两者配合：限流保证活动期间系统不崩，售罄保证活动后零开销
```
