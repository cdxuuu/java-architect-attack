# Day 1：微服务基础 — 拆分策略、注册发现、通信方式（练习整理）

## 题目一：微服务怎么拆？拆分策略与过度拆分的代价

### 三大拆分维度

| 维度 | 核心思路 | 适用场景 | 典型案例 |
|------|---------|---------|---------|
| **按业务域拆** | DDD限界上下文，一个业务域一个服务 | 业务边界清晰的系统 | 保险商城：保单域、理赔域、用户域 |
| **按数据拆** | 谁的数据谁管，跨服务不直连DB | 数据隔离要求高 | 钱包系统：账户服务独占账户表 |
| **按变化频率拆** | 变化快的和变化慢的分开 | 迭代节奏差异大 | 秒杀：核心下单稳定，活动规则多变 |

### 拆分粒度的权衡

```
粒度太粗                          粒度适中                        粒度太细
┌──────────────┐            ┌──────────────┐            ┌──────────────┐
│  单体应用     │            │  合理微服务   │            │  微服务地狱   │
│  10万行代码   │    ───→    │  每个服务     │    ───→    │  100+服务     │
│  一体部署     │            │  1-5万行      │            │  CRUD拆3个    │
│              │            │  独立部署     │            │  调用链深5层   │
└──────────────┘            └──────────────┘            └──────────────┘
   部署慢                       平衡点                      运维爆炸
   牵一发动全身                  独立迭代                     分布式问题x100
   技术栈绑定                   故障隔离                     调试困难
```

### 过度拆分的六大代价

| 代价 | 具体表现 | 真实案例 |
|------|---------|---------|
| **调用链过长** | 一次请求跨5+服务，RT叠加 | 查个订单要调用用户/商品/库存/价格/优惠5个服务 |
| **分布式事务爆炸** | 本来一个本地事务，拆成3个服务要做分布式事务 | 下单=库存+订单+支付，TCC成本翻3倍 |
| **运维成本飙升** | 100个服务=100套部署+监控+告警 | 每次发版要协调10+服务 |
| **数据一致性难** | 跨服务join不存在了，数据冗余+最终一致 | 用户地址改了，订单里的地址要异步同步 |
| **调试困难** | 一个bug跨3个服务，日志分散 | 排障要从3个服务的日志里拼链路 |
| **团队边界模糊** | 一个需求要改5个服务，涉及3个团队 | 跨团队协调比写代码还慢 |

### 架构师决策框架：该不该拆？

```
决策 = f(业务复杂度, 团队规模, 变化频率, 运维能力)

不拆的条件（满足任一即不拆）：
  - 团队 < 10人
  - 业务域边界模糊
  - 运维没有容器化/自动化能力
  - 系统变化频率低（半年迭代一次）

拆的条件（满足越多越该拆）：
  - 团队 > 20人，多团队协作
  - 业务域清晰，各域独立迭代
  - 不同模块变化频率差异大
  - 已有DevOps/容器化基础设施
```

> 你的蛋壳钱包系统，如果只有5个人开发，整个钱包一个服务就够了；如果分账户/风控/清算三个团队，拆成三个服务是合理的。

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 按功能模块拆，一个模块一个服务 |
| 高级开发 | 按业务域拆（DDD），要考虑数据独立性 |
| 架构师 | 拆分是成本-收益决策。按业务域+数据+变化频率三维度拆，关键是不拆的时候也要说清为什么。10人以下团队优先合而不是拆，过度拆分比单体危害更大 |

**补足方向**：练习回答"你的秒杀系统是怎么拆的？为什么不能更细/更粗？"——能说出粒度决策的数字和依据。

## 题目二：服务注册发现 — CP还是AP？Nacos vs Eureka vs ZooKeeper

### 注册中心的核心职责

```
服务A（Provider）                    服务B（Consumer）
     │                                    │
     │  ① 注册（IP:Port/元数据）           │
     ├──────────→ 注册中心 ←──────────────┤
     │              │      ② 订阅          │
     │              │  ③ 推送/拉取服务列表   │
     │              ├────────────────────→│
     │              │                     │
     │  ④ 心跳保活   │                     │
     ├────────────→│                     │
     │              │  ⑤ 剔除不健康实例     │
     │              │                     │
     │  ←══════════╪═══════════════════→│
     │        ⑥ 服务间直接调用              │
```

### 三大注册中心对比

| 维度 | Eureka | ZooKeeper | Nacos |
|------|--------|-----------|-------|
| **CAP模型** | AP | CP | AP/CP可切换 |
| **一致性** | 最终一致（Peer-to-Peer复制） | 强一致（ZAB协议） | AP:最终一致 / CP:Raft协议 |
| **健康检查** | 客户端心跳（30s），90s未心跳剔除 | 会话保持（临时节点），超时删节点 | AP:客户端心跳 / CP:服务端主动探测 |
| **可用性** | 高（Peer-to-Peere，任一节点可服务） | 低（Leader挂了要选举，30-120s不可用） | AP模式高 / CP模式依赖Raft选举 |
| **典型问题** | 可能读到旧数据（新注册实例延迟可见） | Leader选举期间不可用 | CP模式选举期间不可用 |
| **适用场景** | AWS环境，对可用性要求高 | 强一致场景（分布式锁、配置） | 国内微服务首选，AP/CP按需切换 |

### CP vs AP 的选择 — 映射到上周学的CAP

```
回顾第3周：CAP不可能同时满足

注册中心场景：
  - 网络分区是常态（机房光纤被挖、交换机故障）
  - 分区时，CP = 拒绝注册/发现请求 → 服务无法上下线 → 调用链路可能中断
  - 分区时，AP = 允许注册/发现（数据可能旧） → 可能调到已下线实例 → 但大部分请求正常

架构师决策：
  - 注册中心首选AP → 可用性 > 强一致
  - 原因：读到旧实例列表最多重试一次，但注册中心不可用 = 全局故障
  - 例外：配置中心需要CP（配置不一致比不可用更危险）

  这就是为什么Nacos把注册发现默认AP、配置管理默认CP
```

### Nacos AP/CP切换的工程细节

```java
// Nacos 临时实例 → AP模式（默认）
// 临时实例：客户端心跳保活，不可用时自动剔除
Instance instance = new Instance();
instance.setIp("192.168.1.100");
instance.setPort(8080);
instance.setEphemeral(true);  // 临时实例 → AP
namingService.registerInstance("order-service", instance);

// Nacos 持久实例 → CP模式
// 持久实例：服务端主动探测，必须手动注销
instance.setEphemeral(false);  // 持久实例 → CP
namingService.registerInstance("order-service", instance);
```

| 实例类型 | 一致性 | 健康检查 | 适用场景 |
|---------|--------|---------|---------|
| 临时实例（ephemeral=true） | AP | 客户端心跳 | 微服务实例（随时上下线） |
| 持久实例（ephemeral=false） | CP | 服务端探测 | 数据库、缓存等基础设施 |

### 注册中心宕机怎么办？

```
注册中心不可用 ≠ 服务不可用！

  Consumer本地缓存：
    - 拉取的服务列表缓存在内存
    - 注册中心挂了，Consumer用缓存继续调用
    - 问题：新实例上线/下线不可感知

  架构师的三层防御：
    L1: 本地缓存 → 注册中心挂了短时间不影响
    L2: 本地磁盘持久化 → Consumer重启也能从磁盘加载缓存
    L3: 降级策略 → 注册中心长时间不可用，人工更新配置/切换备用集群

  关键数字：
    - Nacos本地缓存默认30秒刷新
    - 磁盘缓存路径：{naming}/cache/
    - 降级阈值：注册中心不可用 > 5分钟，启动降级预案
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 用Nacos做注册中心，服务自动注册发现 |
| 高级开发 | Nacos支持AP和CP，微服务用AP，知道Eureka是AP、ZK是CP |
| 架构师 | 注册中心选AP，因为可用性 > 强一致（分区时旧数据最多重试，不可用则全局故障）。Nacos临时实例AP/持久实例CP的设计对应微服务vs基础设施。注册中心挂了≠服务挂了，三层防御：内存缓存→磁盘持久化→降级预案。5分钟是降级决策阈值 |

**补足方向**：练习回答"注册中心全挂了，你的秒杀系统还能下单吗？"——要说出本地缓存兜底+降级预案。

## 题目三：微服务通信方式 — HTTP vs RPC vs MQ，什么时候用什么？

### 三种通信方式对比

| 维度 | HTTP（REST/Feign） | RPC（Dubbo/gRPC） | MQ（异步） |
|------|-------------------|-------------------|-----------|
| **协议** | HTTP/1.1, HTTP/2 | 自定义TCP协议 | 各MQ自有协议 |
| **序列化** | JSON（文本） | Hessian/Protobuf（二进制） | 各MQ自有 |
| **性能** | 低（文本序列化+HTTP开销） | 高（二进制+长连接） | 异步，不看RT |
| **耦合度** | 松（接口契约） | 紧（接口+SDK） | 最松（消息Schema） |
| **跨语言** | 天然支持 | 需要各语言SDK | 天然支持 |
| **调试** | cURL/Postman直接调 | 需要专用工具 | 查消息轨迹 |
| **适用** | 外部API、BFF层 | 内部高频调用 | 异步解耦、削峰 |

### 性能数字对比

```
同环境压测参考值（1KB数据）：

  HTTP + JSON：     ~5,000 QPS/连接，  RT 2-5ms
  Dubbo + Hessian： ~20,000 QPS/连接， RT 0.5-1ms
  gRPC + Protobuf： ~30,000 QPS/连接， RT 0.3-0.8ms

  性能差距核心原因：
    1. 序列化：JSON反射+文本 vs Protobuf编译期生成+二进制
    2. 协议：HTTP每次请求头几百字节 vs TCP自定义协议头几十字节
    3. 连接：HTTP短连接/连接池 vs RPC长连接复用
```

### 架构师决策框架

```
              需要同步返回结果？
                    │
              ┌─────┴─────┐
              │是         │否
              ↓           ↓
         调用频率？     用MQ（异步解耦）
              │
        ┌─────┴─────┐
        │高          │低
        ↓            ↓
    内部服务？     外部API？
        │            │
   ┌────┴────┐  ┌────┴────┐
   │是       │  │是       │
   ↓         ↓  ↓         ↓
 用RPC    用HTTP  用HTTP   用HTTP
(Dubbo)  (Feign) (REST)  (REST)


实际项目中混合使用：
  秒杀系统：
    用户 → API网关（HTTP） → 秒杀服务
    秒杀服务 → 订单服务（RPC/Dubbo，高频内部调用）
    秒杀服务 → MQ → 积分/通知（异步解耦）
```

### Dubbo核心架构

```
Consumer                    Registry                    Provider
   │                          │                          │
   │  ① subscribe             │           ② register     │
   ├─────────────────────────→│←─────────────────────────┤
   │                          │                          │
   │  ③ notify（服务列表）     │                          │
   │←─────────────────────────┤                          │
   │                          │                          │
   │  ④ invoke（直连调用，不经过注册中心）                  │
   ├────────────────────────────────────────────────────→│
   │                          │                          │
   │  ⑤ monitor（调用统计）   │                          │
   ├──────────────────────────┼──────────────────────────┤
   │                    Monitor                           │
```

关键点：**调用不经过注册中心**！注册中心只负责地址推送，Consumer拿到地址后直连Provider。

### Feign vs Dubbo 的选型实战

| 场景 | 选型 | 原因 |
|------|------|------|
| 内部服务高频调用 | Dubbo | 性能4倍+，长连接复用 |
| 对外开放API | HTTP/Feign | 跨语言、可调试、防火墙友好 |
| BFF层→后端服务 | 都行 | 看团队技术栈，Spring Cloud选Feign，Dubbo生态选Dubbo |
| 新项目从零开始 | Dubbo 3.x | Triple协议兼容HTTP，同时支持REST和RPC |

> Dubbo 3.x 的 Triple �协议基于HTTP/2，既支持标准REST调用，又支持RPC高性能调用，不再是非此即彼的选择。

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 用Feign调其他服务，简单方便 |
| 高级开发 | Feign走HTTP性能差，Dubbo走RPC性能好，按需选择 |
| 架构师 | 通信方式是成本-性能-耦合度的三维决策。HTTP松耦合+可调试+跨语言，RPC高性能+紧耦合+内部专用，MQ异步解耦+最终一致。调用不经过注册中心是关键细节。Dubbo 3.x Triple协议不再是非此即彼。项目里混合使用：外部HTTP + 内部RPC + 异步MQ |

**补足方向**：画一遍你秒杀系统的完整调用链路，标注每一段用什么通信方式+为什么。面试时能指着图说"这用RPC因为高频+内部，这用MQ因为异步解耦"。

## 题目四：服务雪崩与容错设计 — 从一次线上故障说起

### 什么是服务雪崩？

```
正常状态：
  A → B → C → D    每个服务RT 50ms，总RT 200ms

D变慢（DB慢查询）：
  A → B → C → D    D RT从50ms涨到5s
       ↑ 线程阻塞     ↑ 线程阻塞    ↑ 根因

雪崩：
  A → B → C → D    D不可用
  ↑ 线程池耗尽        ↑ 线程池耗尽   ↑ 线程池耗尽
  ↑ 新请求被拒        ↑ 新请求被拒   ↑ 新请求被拒
  ↑
  整条链路崩溃，所有服务不可用

本质：同步调用 + 线程池有限 + 级联故障 = 雪崩
```

### 四种容错模式

| 模式 | 做法 | 适用场景 | 代价 |
|------|------|---------|------|
| **超时控制** | 设置合理的调用超时 | 所有同步调用 | 超时设太短误杀，设太长拖垮自己 |
| **熔断** | 失败率超阈值→断路器打开→快速失败 | 下游不可用时保护自己 | 熔断期间正常请求也被拒绝 |
| **隔离** | 每个下游独立线程池/信号量 | 防止一个下游拖垮所有调用 | 线程池开销、资源浪费 |
| **降级** | 失败后返回兜底数据/走备链路 | 非核心功能牺牲体验保核心 | 数据可能不实时/不完整 |

### 熔断器的三种状态

```
                 失败率 ≥ 阈值（默认50%）
    ┌────────┐ ────────────────────────→ ┌────────┐
    │  CLOSED │                           │  OPEN   │
    │ （正常） │ ←──────────────────────── │ （熔断） │
    └────────┘    探测成功（半开状态下）     └────────┘
         ↑                                    │
         │         超时后（默认5s）             │
         │         ┌────────┐                 │
         └─────────│HALF-OPEN│←───────────────┘
          探测成功  │ （半开） │  失败率仍高
                   └────────┘  → 回到OPEN

  关键参数（Sentinel默认值）：
    - 慢调用比例阈值：50%
    - 熔断时长：10s
    - 最小请求数：5
    - 统计时长：1000ms
```

### Sentinel核心概念

```java
// 1. 流控规则 — 限流
FlowRule flowRule = new FlowRule("orderService")
    .setCount(1000)          // QPS阈值
    .setGrade(RuleConstant.FLOW_GRADE_QPS)
    .setLimitApp("default");
FlowRuleManager.loadRules(Collections.singletonList(flowRule));

// 2. 熔断规则 — 慢调用比例
DegradeRule degradeRule = new DegradeRule("orderService")
    .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())
    .setCount(0.5)           // 慢调用比例阈值50%
    .setSlowRatioThreshold(0.5)
    .setTimeWindow(10)       // 熔断时长10秒
    .setMinRequestAmount(5)  // 最小请求数
    .setStatIntervalMs(1000);// 统计时长1秒
DegradeRuleManager.loadRules(Collections.singletonList(degradeRule));

// 3. 降级处理
@SentinelResource(value = "getOrder", fallback = "getOrderFallback")
public Order getOrder(Long orderId) {
    return orderService.getById(orderId);
}

// 降级方法：返回兜底数据
public Order getOrderFallback(Long orderId) {
    Order order = new Order();
    order.setId(orderId);
    order.setStatus("降级中，请稍后重试");
    return order;  // 或走缓存/走备库
}
```

### 限流算法对比

| 算法 | 原理 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **固定窗口** | 1秒内计数，超限拒绝 | 简单 | 临界点2倍突发 | 精度要求低 |
| **滑动窗口** | 细分窗口滑动计数 | 解决临界突发 | 内存稍高 | 精度要求高（Sentinel默认） |
| **令牌桶** | 固定速率放令牌，请求取令牌 | 允许一定突发 | 实现稍复杂 | 允许突发的场景（API网关） |
| **漏桶** | 固定速率出水，超量排队/拒绝 | 流量绝对均匀 | 不能利用突发 | 严格匀速（MQ消费端） |

```
令牌桶 vs 漏桶的本质区别：

  令牌桶：允许突发（桶里有令牌就能瞬间处理）
    → 适合网关：正常流量允许突发，超限才限流
    → 秒杀场景：前100个请求瞬间通过，后续限流

  漏桶：强制匀速（不管来多少，出口恒速）
    → 适合下游保护：不管上游多猛，下游按能力消费
    → MQ消费端：本质就是漏桶
```

### 秒杀系统中的容错实战

```
用户请求 → [网关限流] → 秒杀服务 → [Sentinel熔断] → 订单服务 → DB

  网关层（令牌桶）：
    - QPS限制10000（根据压测结果）
    - 超限直接返回"活动太火爆"

  秒杀服务层（滑动窗口）：
    - 单用户QPS限制5（防刷）
    - 商品维度QPS限制（热点参数限流）

  调用订单服务（熔断+降级）：
    - 订单服务RT > 500ms比例 > 50% → 熔断10秒
    - 降级：返回"排队中，请稍后查看" + 消息入MQ异步处理

  三层防御关系：网关挡90% → 服务层挡9% → 熔断降级兜底1%
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 加个超时，失败了重试 |
| 高级开发 | 用Sentinel做限流熔断，降级返回默认值 |
| 架构师 | 容错是分层防御体系：超时是底线，熔断是主动保护，隔离是切断传播，降级是牺牲非核心保核心。令牌桶允许突发（网关），漏桶强制匀速（下游保护），选择依据是谁在保护谁。秒杀系统三层限流：网关挡90%、服务挡9%、熔断兜底1%。关键数字：熔断阈值50%、统计窗口1秒、最小请求数5 |

**补足方向**：画出你的秒杀系统完整容错链路图，标注每层限流算法+阈值+降级策略。面试时能说"我的秒杀从网关到DB有5层防御"。
