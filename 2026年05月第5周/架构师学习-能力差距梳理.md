# 架构师面试能力差距梳理 — 第5周服务治理

## 当前定位：Java高级开发 → 目标：架构师

### 第5周Day2问答表现基准：60/100
- 服务治理的整体概念有，但细节深度不够
- 实战经验（线程池隔离第三方调用）是亮点
- 核心短板：熔断策略不熟、漏斗结构思维缺失、售罄标记未掌握

## 差距一：熔断策略对应故障模式 — 知识空白
> Day2发现

### 现状
完全不了解Sentinel三种熔断策略（慢调用比例/异常比例/异常数），无法区分适用场景。

### 架构师为什么需要
熔断是容错的核心机制，面试必问。不知道三种策略的区别 = 没有深度使用过Sentinel。面试官一追问"你线上熔断规则怎么配的"，答不上来直接减分。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 加个熔断，失败了走降级 |
| 高级开发 | 熔断器有CLOSED/OPEN/HALF_OPEN三种状态 |
| 架构师 | 三种策略对应三种故障模式：慢调用比例→下游变慢但没挂（DB慢查询）、异常比例→偶发异常（第三方API不稳定）、异常数→必挂（服务OOM）。参数调优：timeWindow看下游恢复时间（业务10s/支付30s/DB故障60s），minRequestAmount防误杀取5-10 |

### 怎么补
1. **记住对应关系**：慢调用管"变慢"、异常比例管"偶尔挂"、异常数管"必挂"
2. **写三组规则配置**：针对你项目中的订单服务、支付服务、库存服务各写一组熔断规则，标注每个参数为什么这么设
3. **面试话术**："我线上配的是慢调用比例熔断，因为订单服务更多是慢查询而不是直接挂掉，RT阈值500ms、比例50%、熔断10秒"

## 差距二：漏斗结构思维缺失 — 罗列式回答
> Day2发现

### 现状
答容错时罗列了CDN、网关、MQ、缓存、限流、监控等措施，但没有"每层挡多少"的定量结构。答五层防御时说成"前端静态加速+网关负载均衡+MQ削峰+限流+监控"，像在报菜名。

### 架构师为什么需要
面试官听容错设计，想听的是**层次感和数字**，不是措施清单。"五层漏斗，每层挡90%，从10万QPS漏到DB只有1次写入"比"我们有限流、熔断、降级"强十倍。漏斗模型体现的是对系统全链路的量化掌控。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 我们有限流和降级 |
| 高级开发 | 网关限流+服务层熔断+MQ削峰，多层防御 |
| 架构师 | 五层漏斗：L1 CDN挡90%恶意流量→L2网关令牌桶挡90%溢出→L3服务层Redis预扣挡90%无效→L4熔断降级挡90%异常→L5 DB兜底。10万QPS→1万→1000→100→10→1。每层限流算法不同：令牌桶(入口)→滑动窗口(中间)→熔断(保护)→漏桶(出口) |

### 怎么补
1. **每次答容错题，先画漏斗图**：标出每层的通过量，用数字说话
2. **记住"每层挡90%"的经验值**：不是精确数字，但体现架构师的量化思维
3. **面试时主动说**："从容错角度看是漏斗，从算法角度看每层不同：入口令牌桶、中间滑动窗口、出口漏桶"

## 差距三：售罄标记机制未掌握
> Day2发现

### 现状
提到了Lua+Redis防超卖，但没想到"卖完之后怎么办"。售罄后的流量处理是比限流更强力的优化，完全没想到。

### 架构师为什么需要
防超卖是业务正确性，售罄标记是高可用+业务正确性的交汇点。秒杀5秒卖完，剩余55秒请求全部直接返回——**DB压力归零**。这是面试中的亮点回答，比"限流1万QPS"更有说服力。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | Lua脚本防超卖 |
| 高级开发 | Redis预扣库存+Lua原子操作+数据库对账 |
| 架构师 | 防超卖是业务正确性，售罄标记是性能优化。库存扣到0时设Redis soldout标记，后续请求直接返回"已售罄"，零DB压力。秒杀5秒卖完→剩余55秒DB写入量归零。防超卖+售罄标记 = 正确性+性能的双重保障 |

### 怎么补
1. **画时间线**：10:00开抢→10:00:05售罄→10:00:05设置标记→10:01:00全部直接返回。对比有无标记的DB压力曲线
2. **面试话术**："我们最强优化不是限流，是售罄标记。卖完后DB压力直接归零，比限流更彻底"
3. **补充细节**：售罄标记要设过期时间（EX 3600），避免活动结束后标记残留

## 差距四：滑动窗口选型根因不清晰
> Day2发现

### 现状
答"Sentinel为什么选滑动窗口"时说了"实现简单、轻量级"，但没说到根本原因——**多维度统计**。

### 架构师为什么需要
"实现简单"是表面原因，面试官一追问"令牌桶实现也不复杂啊，为什么不选？"就答不上来了。根本原因是：令牌桶只能统计"令牌数"一个维度，但熔断器需要异常比例、慢调用比例、RT等多维度数据。滑动窗口是**统计引擎**，一个LeapArray同时服务限流/熔断/系统保护三种规则。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 滑动窗口更精确 |
| 高级开发 | 滑动窗口比固定窗口没有2倍突发问题，实现简单 |
| 架构师 | Sentinel选滑动窗口的根本原因是多维度统计。令牌桶/漏桶只有单维度（令牌数），无法支撑熔断器的异常比例、慢调用比例判断。滑动窗口是统计引擎，一个LeapArray服务限流/熔断/系统保护三种规则。两者不是替代而是配合：排队等待模式下Sentinel用漏桶效果做匀速 |

### 怎么补
1. **记住核心句**："令牌桶统计不了RT和异常率，但熔断器需要这些数据"
2. **理解LeapArray结构**：环形数组 + 窗口过期重置 + LongAdder高并发统计，面试时能说出这个结构
3. **配合关系**：滑动窗口负责统计+判断，排队等待模式用漏桶做匀速效果，不是替代

## 差距五：降级策略缺决策框架
> Day2发现

### 现状
能说出四种降级策略的适用场景，但是罗列式的，不是二维决策框架。选"异步化"时没有说明决策路径。

### 架构师为什么需要
罗列=背书，二维矩阵=决策。面试官想听的是"你怎么选的"，不是"你知道几种"。二维矩阵（实时性×数据过期容忍度）体现的是结构化决策能力。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 降级返回默认值 |
| 高级开发 | 非核心用兜底值，核心用备链路 |
| 架构师 | 降级策略是二维矩阵：实时性×数据过期容忍度。兜底值(低实时+可过期)、缓存(高实时+可过期)、异步化(低实时+不可过期)、备链路(高实时+不可过期)。核心链路绝不用兜底值——宁可慢也不能骗用户。秒杀订单降级走异步化+返回排队凭证，因为允许延迟但不允许假数据 |

### 怎么补
1. **记住二维矩阵**：实时性(高/低) × 数据过期(允许/不允许) = 四种策略
2. **每遇到降级场景，先定位到矩阵的哪个象限**，再选策略
3. **面试话术**："核心链路降级先看两件事：用户要不要立刻看到结果？数据能不能接受过期？"

## 差距六：隔离策略缺超时意识
> Day2发现

### 现状
答"第三方支付API怎么做隔离"时说了策略+工厂+线程池隔离，实战经验对。但没提到超时控制——**没有超时的隔离只是"不拖Tomcat"，线程池资源还是会被耗尽**。

### 架构师为什么需要
线程池隔离的核心不是"分开线程"，而是"可控超时"。面试官问隔离，你答线程池，追问"超时怎么设"，答不上就白答了。CompletableFuture.orTimeout(3, SECONDS) 才是隔离的灵魂。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 调第三方用线程池隔离 |
| 高级开发 | 第三方调用用独立线程池，防止拖垮主线程池 |
| 架构师 | 线程池隔离的灵魂是超时控制，不是独立线程。CompletableFuture+独立线程池+orTimeout(3s)三位一体。没有超时的隔离只是"不拖Tomcat"，线程池资源还是会被耗尽。内部服务用Sentinel信号量+熔断（轻量够用），外部服务自己做CompletableFuture+线程池+超时（彻底隔离） |

### 怎么补
1. **记住核心句**："超时是隔离的灵魂"
2. **写一段代码**：CompletableFuture.supplyAsync(外部调用, externalPool).orTimeout(3, SECONDS)，写一遍就记住了
3. **面试时主动提**："我们第三方调用是线程池隔离+3秒超时+降级三件套"

## 差距七：数字直觉仍不足
> Day2发现，延续第4周差距一

### 现状
答容错时全程无量级数字——网关限流多少QPS？熔断阈值多少？售罄后DB压力降多少？都没有说。

### 第4周已识别，第5周仍未改善

| 缺失数字 | 应该说出的 |
|---------|-----------|
| 网关限流 | 令牌桶1万QPS |
| 单用户限流 | 滑动窗口5 QPS |
| 熔断阈值 | 慢调用比例50%，RT 500ms，timeWindow 10s |
| minRequestAmount | 5（防误杀） |
| 售罄后DB压力 | 归零（从5秒卖完到活动结束） |
| 信号量隔离并发数 | 线程数限流50 |
| 线程数vs QPS公式 | 并发线程数 ≈ QPS × RT |

### 怎么补
1. **每组规则配置带数字**：写熔断规则时不只写策略，带上count/timeWindow/minRequestAmount
2. **回答时先给数字再解释**："网关限流1万QPS，因为压测单机承载上限是1.2万"
3. **建立数字直觉**：不需要精确，但要有量级——"万级QPS""毫秒级RT""秒级熔断"
