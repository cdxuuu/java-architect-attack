# Day 5：微服务实战综合 — 架构师角度梳理

## 核心定位

Day5的主题一句话概括：**微服务不是技术选型，是工程体系——拆、治、建、观、战，缺一不可**。分布式事务选型体现"一致性够用就行"的务实思维，网关全貌体现"流量治理第一道防线"的系统设计，容量规划体现"量化决策"的工程方法，故障演练体现"先止血再治病"的应急哲学，架构演进体现"不是目的而是手段"的架构判断。

## 分布式事务：一致性够用就行

```
选型不是选最好的，是选刚好够用的

  三个维度：一致性要求 × 业务特征 × 团队成本

  ┌──────────────────┬──────────┬───────────┬──────────┐
  │                  │ TCC      │ Saga      │ 消息最终  │
  ├──────────────────┼──────────┼───────────┼──────────┤
  │ 一致性           │ 强一致   │ 最终一致   │ 最终一致  │
  │ 隔离性           │ 冻结资源 │ 脏读风险   │ 无隔离    │
  │ 侵入性           │ 最高     │ 中等       │ 最低      │
  │ 开发成本         │ 4x       │ 2x        │ 1x       │
  │ 典型场景         │ 金融转账 │ 电商下单   │ 支付通知  │
  └──────────────────┴──────────┴───────────┴──────────┘

  面试时不要只说"我们用Seata"
  要说"金融用TCC因为钱不能错，电商用Saga因为可以补偿退货，通知用消息因为不需要同步"
```

### TCC的工程细节

```
三个陷阱必须知道：
  1. 空回滚：Try没到但收到Cancel → 检查Try是否执行过
  2. 幂等：  Confirm/Cancel可能重试 → 全局事务ID+状态表去重
  3. 悬挂：  Cancel比Try先到 → Cancel记录事务ID，Try检查是否已Cancel

  这三个问题不解决，TCC上线必出bug
  解决方案统一：事务控制表 + 全局事务ID
    INSERT INTO tx_control (tx_id, status) VALUES (?, 'trying')
    Try执行前先查status，Cancel执行前先查status
```

### Saga的核心权衡

```
Saga vs TCC的核心区别不是"正向+补偿"vs"预留+确认"

  核心区别是隔离性策略：
    TCC：靠冻结字段做业务隔离（Try阶段余额不变，冻结增加）
    Saga：靠补偿做逻辑隔离（直接提交，脏读后补偿恢复）

  Saga脏读的应对：
    业务容忍：电商库存多扣→补偿恢复，用户无感
    语义锁：  库存加"预扣"状态，查询时过滤预扣量
    不适合金融：A转100给B，中间状态B查到多了100，不能接受
```

## API网关：流量治理的第一道防线

```
六大职责不是罗列，是有优先级的防线：

  ① 认证鉴权 — 第一道门，没身份就别进来
  ② 限流熔断 — 第二道门，进来了也要排队
  ③ 灰度路由 — 第三道门，不同人走不同路
  ④ 协议转换 — 桥梁，内外协议不同时翻译
  ⑤ 聚合裁剪 — 优化，减少客户端往返
  ⑥ 日志监控 — 基础，所有请求可追溯

  三层限流算法不同的根因：
    L1 网关用令牌桶 → 入口有突发流量，令牌桶允许短暂超均值
    L2 服务用滑动窗口 → 需要多维度统计（RT/异常率/慢调用）
    L3 接口用热点参数 → 同一热点商品10万请求，前两层限不住
```

### 网关认证的关键决策

```
JWT踢人方案：
  Payload加version字段 + Redis存当前version
  改密码/登出时version+1 → 网关校验时查Redis比对
  无需等Token过期，秒级生效

  网关层 vs 服务层认证的分工：
    网关：管"你是谁"（JWT校验 + 限流）
    服务：管"你能做什么"（RBAC细粒度鉴权 + mTLS）
    不重复做，各有侧重
```

### BFF决策

```
≤3个服务聚合 → 网关Filter内做
  优点：少维护一个服务
  例子：首页需要用户信息+推荐商品+购物车数量

>3个服务 / 多端差异大 → 独立BFF
  优点：端定制化 + GraphQL按需查询
  例子：App端、H5端、小程序端首页数据差异大
```

## 容量规划：量化决策

```
四步法闭环：

  基线建立 → 流量预估 → 容量计算 → 扩容预案
     │            │           │           │
  压测阶梯加压  DAU×倍数    QPS/水位×1.3  HPA+CronHPA
  找安全水位    分常规/峰值  最少2台高可用  波峰波谷弹性

  三个关键数字的根因：
    安全水位70-80%：100%运行=0余量，一个抖动就过载
    冗余系数1.3：一台挂了还有30%余量，1.5浪费1.1扛不住
    峰值倍数3-5x：电商日常3倍/大促5倍/秒杀100倍，看业务特征

  全链路压测三要素（大促前必做）：
    1. 数据隔离：影子库，压测流量不影响线上
    2. 链路染色：HTTP Header标记，全链路透传
    3. 监控分离：压测指标和线上分开看
```

## 故障演练：先止血再治病

```
故障处理的黄金法则：5分钟内恢复服务，根因后面再查

  5步SOP：
  ┌─────────────────────────────────────┐
  │  1. 止血（0-5min）                  │
  │     决策树：回滚 > 降级 > 限流 > 重启│
  │                                     │
  │  2. 通告（5-10min）                 │
  │     影响面 + 当前状态 + 预计恢复     │
  │                                     │
  │  3. 定位（10-20min）                │
  │     Metrics→Tracing→Logging        │
  │                                     │
  │  4. 修复（20-30min）                │
  │     临时修复→确认恢复→持续观察       │
  │                                     │
  │  5. 复盘（24-48h后）                │
  │     5Whys + 改进 + 跟进             │
  └─────────────────────────────────────┘

  止血优先级决策树：
    能回滚？→ 回滚（最快5分钟恢复）
    不能回滚？→ 能降级？→ 降级
    不能降级？→ 能限流？→ 限流
    不能限流？→ 重启

  演练的四步法：
    稳态假设 → 故障注入 → 观察反应 → 验证修复
    没有明确的稳态指标 = 演练无意义

  核心信念：没有演练过的预案 = 废纸
```

## 架构演进：不是目的而是手段

```
三阶段不是时间线，是能力线：

  阶段1 单体→服务化：拆最痛的1-2个模块验证
  阶段2 服务化→微服务：补基建，治理跟上
  阶段3 微服务→云原生：自动化+弹性+智能运维

  四个条件判断要不要微服务：
    团队规模 ≥ 5人
    部署频率 ≥ 每天1次
    有独立运维能力
    有基建投入预算
    → 全满足才考虑，否则模块化单体更务实

  四大反模式：
    一步到位：没有基建支撑就全拆
    为拆而拆：3人团队10个服务（康威定律：架构≈组织）
    只拆不管：拆了没加限流熔断监控
    数据不拆：服务独立但共享DB

  面试核心句：
  "微服务不是目的，是解决特定问题的手段。
  痛点是什么→拆了能不能解决→基建跟不跟得上，三个问题想清楚再动手。"
```
