# Day 3：微服务核心基础设施 — 架构师角度梳理

## 核心定位

微服务基础设施不是"装个组件就能用"，是**每个组件都要回答：为什么选这个方案、数据怎么流转、异常怎么兜底**。架构师要能说清每个组件的设计哲学和选型根因，而不是背配置。

## 注册中心：Distro协议的本质是职责划分

```
常见误解：Nacos AP就是Eureka的翻版
正确理解：Distro协议的核心创新是"职责划分 + 异步同步"

  Eureka的做法：
    所有节点平等 → 任意节点可写 → 同步给所有peer
    问题：写扩散 → 3节点3次同步，5节点5次同步

  Distro的做法：
    Hash分区 → 每个节点只负责一部分service
    写只到负责节点 → 异步fire-and-forget同步给其他节点
    读任意节点 → 因为每个节点都有全量数据

  性能对比：
    Eureka：写1次 + 同步N-1次（N=节点数）
    Distro：写1次 + 异步同步N-1次（fire-and-forget，不阻塞写）

  Distro vs Raft（Nacos内部两种协议的对比）：
    Distro (AP)：
      写入 → 不等同步完成就返回 → 快但可能短暂不一致
      适合：微服务实例注册（频繁上下线，短暂不一致可接受）

    Raft (CP)：
      写入 → 等过半节点确认 → 慢但强一致
      适合：持久实例（数据库地址，不能丢不能错）
```

### 临时vs持久：谁做健康检查是核心

```
面试必考句："临时实例客户端心跳，持久实例服务端探测"

  为什么不是通信协议的区别？
    Nacos 2.x临时和持久实例都用gRPC通信
    区别在于：谁来判断"这个实例还活着"

  临时实例——客户端证明自己活着：
    5秒心跳 → 15秒标不健康 → 30秒剔除
    类比：员工每天打卡 → 3天没打卡 → 离职

  持久实例——服务端验证对方活着：
    服务端主动TCP/HTTP探测 → 不健康只标记不剔除
    类比：公司查岗 → 发现不在岗 → 记录但不辞退

  protectThreshold的设计哲学：
    Eureka：心跳<85% → 全保留（包括挂掉的）→ 一刀切
    Nacos：不健康比例>阈值 → 保留全部；未超过 → 正常剔除
    架构师可调：高可用场景取0.5，强一致场景取0
```

### 注册中心vs配置中心：为什么通信模型不同

```
核心洞察：数据特性决定通信模型

  注册数据特性：
    变更频率：中（实例上下线）
    一致性要求：弱（秒级延迟可接受）
    数据量：大（数千实例）
    → 选UDP推送+拉取兜底：轻量+容许丢失+拉取修正

  配置数据特性：
    变更频率：低（几天一次）
    一致性要求：强（配置丢失=行为不一致）
    数据量：小（几百个配置项）
    → 选Long Polling：可靠+不丢+秒级感知

  面试回答框架：
    "注册中心用UDP推拉结合，因为注册数据允许短暂不一致；
     配置中心用Long Polling，因为配置变更必须可靠送达。
     两个场景的通信模型由数据特性决定，不是随意选择。"
```

## 配置中心：Long Polling的本质是客户端主动拉

```
常见误解：Long Polling是服务端推送
正确理解：Long Polling是"客户端拉，服务端延迟响应"

  和纯拉取的区别：
    纯拉取：客户端每30秒问一次"有变更吗？"
    Long Polling：客户端问"有变更吗？没有就等，有了再告诉我"
    区别：纯拉取有固定延迟，Long Polling变更即时感知

  和WebSocket的区别：
    WebSocket：服务端主动推 → 双向长连接 → 需维护连接状态
    Long Polling：客户端主动拉 → 每次请求独立 → HTTP无状态

  为什么Long Polling更适合配置中心：
    1. 配置变更极低频（几天一次）→ WebSocket大部分时间空闲
    2. 万级客户端 → WebSocket连接管理复杂
    3. HTTP天然穿透代理/防火墙 → 运维简单
    4. 无状态 → 服务端重启客户端自动重连
```

### 配置安全的五层防御

```
L1 变更审批 → 防误操作
  配置变更需要审批（生产环境）
  谁改的、为什么改、什么时候改 → 可追溯

L2 灰度发布 → 防爆炸半径
  先1%实例 → 观察监控 → 再10% → 再100%
  Nacos Beta发布（指定灰度IP）是最简单的实现

L3 快速回滚 → 防长时间故障
  Nacos一键回滚 + Long Polling秒级推送
  正在执行的请求用旧配置 → 可接受

L4 配置校验 → 防格式错误
  JSON Schema校验 → 防止配置格式错误导致启动失败
  枚举值校验 → 防止配了非法值

L5 监控告警 → 防隐性故障
  配置变更后自动触发监控巡检
  5分钟内错误率上升 → 自动告警

  和Day2降级三层架构的对应：
    L1审批 = L3人工预案（人做决策）
    L2灰度 = L2配置开关（配置控制范围）
    L3回滚 = L1自动降级（自动执行）
    L4校验 = 预防层（Day2没覆盖的）
    L5监控 = 观测层（Day4会深入）
```

## 服务网关：过滤器链的本质是责任链

```
常见误解：网关就是做路由转发
正确理解：网关是微服务的统一入口，承担路由+鉴权+限流+监控+灰度五重职责

  过滤器链的设计哲学：
    洋葱模型：前置处理→代理转发→后置处理
    和Dubbo Filter链、Spring Interceptor链是同一模式

  order的设计原则：
    order越小 → 前置越先执行 → 后置越后执行
    这意味着：order小的Filter"包住"order大的Filter
    所以：日志(order=-1000)包住限流(order=-500)包住鉴权(order=0)
    日志能看到完整的请求耗时，包括限流和鉴权的耗时
```

### 灰度路由的架构师决策

```
不是"选标签还是选比例"，是"先标签再比例"

  灰度发布的标准流程：
    Phase 1：标签灰度（内部用户/测试账号）
      → 验证功能正确性
      → 影响面可控（只有内部用户）

    Phase 2：比例灰度（1% → 10% → 50%）
      → 验证性能和稳定性
      → 逐步扩大影响面

    Phase 3：全量发布
      → 确认稳定后全量

  两个阶段用不同的路由策略：
    Phase 1 → 按用户标签（精准+可控）
    Phase 2 → 按流量比例（自动+渐进）
    不是二选一，是流程的先后步骤
```

## 链路追踪：不是装个Agent就完了

```
常见误解：装个SkyWalking Agent就有链路追踪了
正确理解：Agent只解决同步HTTP调用的自动传播，异步场景需要手动处理

  三种传播场景的难度递增：
    1. HTTP同步调用 → Agent自动注入Header → 零代码
    2. 线程池异步 → TransmittableThreadLocal → 加个依赖+装饰线程池
    3. MQ异步 → 手动传播Message Property → 每个生产者/消费者都要改

  架构师要做的：
    不是让每个开发自己处理传播
    而是封装统一SDK：
      TraceUtils.asyncRun(() -> { ... });  // 线程池自动传播
      TraceMessage.wrap(msg);              // MQ自动传播
    开发无感知，架构师保证全链路TraceId不断
```

### 采样策略：正常1% + 异常100%

```
头部采样的本质：入口赌博
  请求进来时不知道会不会出错 → 随机决定采不采
  结果：99%的异常Trace被丢弃 → 排障时找不到关键链路

尾部采样的本质：事后决策
  请求完成后看结果 → 异常的保留，正常的丢弃
  结果：异常Trace 100%保留 → 排障效率高

  两者不是替代关系，是组合：
    头部采样1% → 覆盖正常流量（用于性能分析）
    尾部采样 → 保留所有异常/慢请求（用于故障排查）
    组合效果：正常请求1%采样 + 异常请求100%保留

  成本分析：
    纯头部1%：1万QPS → 100条Trace/秒
    头1%+尾异常：1万QPS → 100条正常+异常条数（通常<10条）→ ~110条/秒
    纯尾部：1万QPS → 暂存1万条/秒 → 内存压力大

    结论：组合方案成本几乎不增，排障价值大增
```

## 微服务通信：超时+重试+熔断的铁三角

```
超时+重试+熔断不是三个独立配置，是联动系统

  联动规则：
    熔断OPEN → 不超时不重试，直接降级（毫秒级）
    熔断HALF_OPEN → 放一个探测，不重试
    熔断CLOSED → 正常超时+重试逻辑

  熔断策略决定重试策略：
    慢调用比例熔断 → 下游慢但没挂 → 重试可能成功（换实例）
    异常比例熔断 → 下游偶发异常 → 重试1次可能成功
    异常数熔断 → 下游必挂 → 重试只会加重负担 → 不重试

  超时嵌套原则：
    外层超时 > 内层超时之和
    为什么？给内层留出重试+降级的时间

    网关30秒 > 服务10秒 > 下游3秒×2次重试=6秒
    服务层10秒内有4秒余量做降级处理

  重试策略的优先级：
    1. 先判断幂等性：非幂等不重试
    2. 再看熔断状态：OPEN不重试
    3. 最后看重试次数：≤2次+指数退避
```

### gRPC vs RESTful：不是性能问题

```
常见误解：gRPC比RESTful快，所以选gRPC
正确理解：选型是五维决策，性能只是其中一维

  架构师决策路径：
    1. 谁调？（前端/外部系统/内部服务）
       前端/外部 → RESTful
       内部 → gRPC

    2. 调多频繁？（QPS和延迟要求）
       高频低延迟 → gRPC
       低频可接受50ms+ → RESTful

    3. 团队能力？（多语言/Protobuf经验）
       只会JSON → RESTful
       有Protobuf经验 → gRPC

    4. 生态约束？（网关/浏览器/服务网格）
       需要浏览器 → RESTful
       Istio服务网格 → gRPC原生支持

    5. 演进路径？（新旧系统混合）
       老系统RESTful → 核心链路先改gRPC → 逐步迁移

  最常见的组合模式：
    内部服务 → gRPC（性能优先）
    对外API → RESTful（通用性优先）
    两者之间 → gRPC-Gateway做协议转换
```
