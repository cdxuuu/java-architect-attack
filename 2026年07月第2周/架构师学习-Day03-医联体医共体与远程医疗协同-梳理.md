## 一、医联体/医共体协同在高可用架构中的定位

### 1.1 核心一句话

> **医联体/医共体 = 多机构数据归属与共享 + EMPI 跨机构主索引 + 业务流程跨机构编排 + 网络异构与等保隔离 + 数据标准多样性治理**
>
> 与单一医院信息化的根本差异：数据归属跨机构（采集/归属/调阅三方）、EMPI 跨机构匹配（串档风险）、网络异构（医院内网/政务外网/互联网）、业务流程跨机构编排（双向转诊/远程会诊）、数据标准多样（多 HIS 厂商、术语映射）。

### 1.2 医联体的四种组织形态

| 形态 | 龙头机构 | 成员机构 | 当前政策密度 |
| --- | --- | --- | --- |
| 城市医疗集团 | 三甲医院 | 城区二级 + 社区中心 | 中 |
| 县域医共体 | 县级医院 | 乡镇卫生院 + 村卫生室 | **高（紧密型试点扩面）** |
| 专科联盟 | 国家级/省级专科中心 | 跨区域专科机构 | 中 |
| 远程医疗协作网 | 三甲医院 | 基层/边远地区 | 中 |

### 1.3 紧密型医共体"五统一"

| 统一 | 政策含义 | 信息化工程映射 |
| --- | --- | --- |
| 统一法人 | 医共体内法人统一 | 机构主数据统一标识 |
| 统一账户 | 财务账户统一 | 医保结算统一账户 |
| 统一人员管理 | 人员统一调配 | 医师多点执业工程化 |
| 统一药品耗材 | 统一采购配送 | 药品目录统一、库存共享 |
| 统一信息平台 | 信息系统统一 | **一套 HIS vs 多套 HIS + 数据中台** |

### 1.4 区域卫生信息平台的多层架构

```
用户层：患者端 + 医师工作台 + 管理后台 + 监管大屏
  ↓
接入层：统一身份认证（医联体 SSO）+ API 网关 + 跨网安全网关
  ↓
业务层：EMPI + 健康档案调阅 + 双向转诊 + 远程会诊 + 影像云 + 检查检验互认 + 处方流转 + 慢病随访
  ↓
数据中台：健康档案中心 + 数据采集交换 + 数据治理 + 主数据管理
  ↓
数据源层：医联体内多家机构系统（HIS/EMR/PACS/LIS/RIS/公卫/体检）
  ↓
监管层：卫健委上报 + 医保局上报 + 操作审计
  ↓
支撑层：标准字典 + 术语映射 + 隐私授权 + KMS
```

### 1.5 "统一信息平台"两种路径的选型权衡

| 维度 | 路径 A：一套 HIS | 路径 B：多套 HIS + 数据中台（推荐） |
| --- | --- | --- |
| 改造成本 | 高（替换原有 HIS） | 低（不动 HIS，新建中台） |
| 业务匹配 | 难（县医院 vs 乡镇差异大） | 好（各机构按需选型） |
| 厂商灵活 | 差（厂商锁定） | 好（数据中台自主可控） |
| 演进路径 | 一次性 | 平滑演进 |
| 数据治理 | 简单（天然统一） | 复杂（多源异构治理） |

## 二、EMPI 主索引

### 2.1 EMPI 的核心问题

同一患者在医联体内多家机构有不同的就诊卡号，必须做主索引匹配，把"张三@县医院 cardId=A"和"张三@乡镇卫生院 cardId=B"识别为同一个患者。

### 2.2 三层匹配算法

| 层 | 算法 | 输入 | 阈值 | 覆盖率/准确率 |
| --- | --- | --- | --- | --- |
| 层 1 | 确定性匹配 | 身份证号 / 医保电子凭证 ID | 完全一致 | 覆盖率低 / 准确率高 |
| 层 2 | 概率匹配（Fellegi-Sunter） | 姓名 + 性别 + 出生日期 + 手机号 + 地址 | p > 0.95 匹配 / 0.5-0.95 待确认 | 覆盖率中 / 准确率中 |
| 层 3 | ML 匹配（XGBoost/DNN） | 层 2 相似度特征 + 关联特征 | p > 0.9 匹配 | 覆盖率高 / 准确率高 |

### 2.3 EMPI 全链路

```
建索引（首次就诊）→ 查索引（跨机构调阅）→ 更新索引（信息变更）→ 错配修复（串档修复）
```

### 2.4 串档风险防范

| 防范措施 | 工程实现 |
| --- | --- |
| 人工确认 | 概率匹配 0.5-0.95 区间必须人工确认 |
| 双向验证 | A→B 与 B→A 双向匹配 |
| 可撤销 | 关联操作可撤销，同步撤销派生数据 |
| 操作留痕 | 不可变审计表，监管可追溯 |
| 风险告警 | 短期关联多张新卡 → 告警 |
| 主索引画像 | 关联卡数、机构数、首次/最近关联时间 |

### 2.5 EMPI 与各标识的关系

```
EMPI Master（区域主索引）
  ├─ 身份证号（国家级，唯一）
  ├─ 医保电子凭证 ID（国家级，唯一）
  ├─ 机构 A 就诊卡号 cardId_A（机构级）
  ├─ 机构 B 就诊卡号 cardId_B（机构级）
  └─ 健康档案 ID（区域级，由 EMPI 派生）
```

## 三、健康档案调阅

### 3.1 六大档

| 档案类型 | 数据来源 |
| --- | --- |
| 基本健康档案 | 公卫系统 |
| 慢病档案 | 公卫系统、慢病随访 |
| 就诊档案 | 各机构 HIS/EMR |
| 检查检验档案 | 各机构 LIS/PACS |
| 用药档案 | 各机构 HIS、处方流转 |
| 免疫接种档案 | 公卫系统 |

### 3.2 跨机构调阅授权链路

```
1. 调阅人身份核验（执业证书 + 所属机构 + 在岗状态）
2. 调阅目的登记（接诊/会诊/随访/转诊）
3. 患者授权核验（知情同意 + 范围 + 时限）
4. 调阅机构资质核验（医联体白名单 + 诊疗关系）
5. 调阅范围控制（按目的自动限制范围）
6. 调阅操作执行（实时调阅 + 自动脱敏 + 30 分钟会话）
7. 调阅操作留痕（不可变审计表 + 患者透明化查询）
```

### 3.3 最小必要原则

| 措施 | 工程实现 |
| --- | --- |
| 按需调阅 | 按调阅目的返回必要字段 |
| 限时授权 | 30 分钟会话有效 |
| 自动脱敏 | 非必要字段脱敏 |
| 敏感档案单独授权 | 精神科/传染科区段单独加密 |
| 调阅频率限制 | 同医师同患者 1 小时 ≤ 5 次 |
| 异常调阅告警 | 日均调阅 > 100 / 调阅非本人接诊患者 → 告警 |

## 四、数据采集与交换

### 4.1 四种采集方式

| 方式 | 适用场景 | 实时性 |
| --- | --- | --- |
| 前置机 | 历史数据迁移、批量上报 | T+1 |
| ETL | 历史数据汇聚 | T+1 |
| API 实时上报 | 关键业务事件 | 实时 |
| 消息队列 | 高频实时数据 | 实时 |

**推荐混合模式**：API 实时上报（关键事件）+ MQ 兜底 + 前置机/ETL（历史）+ 准实时对账（每 5 分钟）。

### 4.2 数据交换标准选型

| 标准 | 场景 | 选型建议 |
| --- | --- | --- |
| HL7 V2 | HIS/LIS/RIS 间 | 机构内部（兼容厂商） |
| FHIR R4 | 现代 RESTful API | 机构→区域平台（新建） |
| WS/T 445 | 国家卫健委数据元 | 卫健委上报 |
| CDA | 临床文档交换 | 出院小结、门诊病历 |

### 4.3 数据治理三层

1. **术语映射**：厂商私有码 → 标准码（ICD-10/11、药品本位码、LOINC）
2. **单位统一**：如血糖 mmol/L vs mg/dL 统一
3. **值域标准化**：性别、婚姻、民族等 GB/T 标准

### 4.4 数据质量评分维度

完整性 + 准确性 + 一致性 + 时效性 + 唯一性

## 五、双向转诊

### 5.1 上转 vs 下转差异

| 维度 | 上转 | 下转 |
| --- | --- | --- |
| 触发方 | 基层医师 | 上级医师 |
| 转诊原因 | 基层能力不足 | 稳定后回基层随访 |
| 绿色通道 | 有 | 一般无 |
| 数据流向 | 基层→上级 | 上级→基层 |
| 考核指标 | 上转率 | 下转率（分级诊疗核心） |

### 5.2 转诊单关键字段

转诊类型、原因、转出/转入机构、患者、诊断（ICD）、临床信息、用药、检查申请、优先级、状态。

### 5.3 转诊回执

```
PENDING → ACCEPTED（接收，分配医师/号源/床位）
       → REJECTED（拒收，必须填原因）
       → RESCHEDULED（改期，必须提供替代时间）
```

### 5.4 绿色通道

| 步骤 | 实现 |
| --- | --- |
| 号源预留 | 上级为基层预留号源（标记"医联体上转专用"） |
| 检查 slot 预留 | 上级预留检查 slot |
| 床位预留 | 急诊上转 2 小时内安排住院 |
| 接单 SLA | 急诊 15min / 择期 2h / 检查 4h |
| 超时升级 | 15min 未接 → 升级科主任 → 升级医务科 → 转其他医联体 |

### 5.5 绿色通道考核

接单时长 + 接单率（≥ 95%）+ 拒收率（≤ 5%）+ 实际就诊率（≥ 90%）。

## 六、远程会诊

### 6.1 四种远程会诊模式

| 模式 | 数据类型 | 数据量 | 实时性 | 架构重点 |
| --- | --- | --- | --- | --- |
| 远程临床会诊 | 音视频 + 病历 | 中 | 实时 | WebRTC + SFU + 录制 |
| 远程影像会诊 | DICOM 影像 | 大（GB） | 准实时 | DICOM 云存储 + 云 PACS |
| 远程心电会诊 | 心电数据 | 小（KB-MB） | 准实时 | 心电上传 + 报告回传 |
| 远程病理会诊 | WSI 数字切片 | 大（GB） | 准实时 | 全切片扫描 + 云存储 |

### 6.2 远程临床会诊链路

```
预约 → 资料预审 → 音视频会诊（WebRTC + SFU + 屏幕共享 + 录制）→ 报告 → 签章 → 归档
```

### 6.3 WebRTC + SFU 技术栈

| 组件 | 选型 |
| --- | --- |
| WebRTC | 原生 |
| SFU | Mediasoup / Jitsi Videobridge |
| STUN/TURN | coturn |
| 录制 | Mediasoup RecordingObserver |
| 屏幕共享 | getDisplayMedia |
| 信令 | WebSocket |

### 6.4 录制存档合规

- SHA256 上链防篡改
- KMS 国密 SM4 加密
- 保留 ≥ 15 年
- 调阅需"调阅人 + 调阅目的"双因子
- 患者可查询"会诊录像访问记录"

### 6.5 报告法律定位

| 报告类型 | 法律定位 | 责任划分 |
| --- | --- | --- |
| 远程临床会诊 | 咨询意见 | 申请医师主责，会诊专家连带（按过错） |
| 远程影像/心电/病理 | 独立诊断报告 | 会诊专家主责（报告错误） |

### 6.6 异常处理与 SLA

| 异常 | SLA |
| --- | --- |
| 转诊申请失败 | 30min 内解决 |
| 转诊拒收 | 即时 |
| 会诊专家缺席 | 4h 前通知 |
| 转诊途中病情变化 | 即时 |
| 转诊后未就诊 | 24h 反馈 |
| 会诊报告超时 | 24h 升级科主任 / 48h 升级医务科 |
| 数据同步失败 | 1h 内解决 |
| 音视频故障 | 15min 内决策（切备用/电话/改期） |

## 七、影像云

### 7.1 多层架构

```
用户层：医师工作台 + 患者端
  ↓
接入层：DICOM 网关（C-STORE/C-FIND/C-MOVE）+ FHIR ImagingStudy + Web API
  ↓
业务层：影像接收 + 索引 + 调阅 + 处理 + 云胶片 + 共享
  ↓
存储层：元数据（PG）+ 对象存储（OSS/S3）+ 索引（ES）+ CDN
  ↓
数据源层：医联体内各机构 PACS
```

### 7.2 热-温-冷分层存储

| 层级 | 时间 | 介质 | SLA | 成本 |
| --- | --- | --- | --- | --- |
| 热 | 3 个月 | SSD 后端 | < 1s | 高 |
| 温 | 3m-1y | HDD 后端 | 1-5s | 中 |
| 冷 | 1y-15y | 归档存储 | 分钟级 | 低 |

### 7.3 云 PACS vs 传统 PACS

| 维度 | 传统 PACS | 云 PACS |
| --- | --- | --- |
| 部署 | 医院内网 | 云端 |
| 存储 | 本地 | 云对象存储 |
| 调阅 | 内网客户端 | Web + 移动 |
| 跨机构 | 难 | 易 |
| 扩展性 | 受限 | 弹性 |
| 成本 | 一次投入 | 按需付费 |

### 7.4 云胶片工程实现

```
DICOM → 转 JPEG/WebP（关键切片 + 缩略图）→ 生成二维码（studyId + 授权 token）
→ 患者扫码 → H5 调阅 → 身份核验 → 快速预览 + 完整加载 → 调阅留痕
```

### 7.5 DICOM 高效传输

| 技术 | 实现 |
| --- | --- |
| 分块上传 | 10 MB/块，并行 |
| 断点续传 | 中断后从断点继续 |
| 并行传输 | 4-8 线程并行 |
| 压缩 | JPEG 2000 Lossless（无损） / JPEG Lossy（预览） |
| 协议 | DIMSE C-STORE / STOW-RS / WADO-RS |

### 7.6 分层加载策略

```
缩略图（< 200ms）→ 关键切片（< 1s）→ 完整序列（按需流式）→ 高清诊断（按需）
```

### 7.7 网络优化

CDN 加速缩略图/关键切片 + 多区域就近访问 + HTTP/2 多路复用 + QUIC 弱网优化 + 列表预加载。

## 八、医联体数据治理

### 8.1 三方数据归属

| 角色 | 含义 | 数据权利 |
| --- | --- | --- |
| 采集机构 | 原始采集机构 | 数据所有权 |
| 归属机构 | 患者归属机构 | 数据归属权（决定共享范围） |
| 调阅机构 | 申请调阅机构 | 数据使用权 |

### 8.2 患者授权四要素

| 要素 | 工程 |
| --- | --- |
| 知情同意 | 电子签章知情同意书 |
| 范围授权 | 全部/部分/排除敏感 |
| 时限授权 | 一次性/7d/30d/1y/长期 |
| 撤回授权 | 随时撤回 + 通知调阅机构 |

### 8.3 数据共享三种模式

| 模式 | 适用场景 |
| --- | --- |
| 原始数据共享 | 医联体内（信任高，诊疗必需） |
| 摘要数据共享 | 跨医联体（信任一般，需知情） |
| 脱敏数据共享 | 科研/统计/监管（无需知情） |

### 8.4 主要术语标准

| 标准 | 场景 | 国内推广 |
| --- | --- | --- |
| ICD-10 | 疾病诊断 | 主流（强制） |
| ICD-11 | 疾病诊断（新版） | 推广中（2025 起逐步替换） |
| 药品本位码 | 药品 | 强制 |
| LOINC | 检验项目 | 推广中 |
| SNOMED CT | 临床综合术语 | 试点 |
| ATC | 药品分类 | 主流 |

### 8.5 术语映射质量评分

覆盖率 + 准确率 + 冲突率 + 单向率。

### 8.6 ICD-10 → ICD-11 迁移期

- 双码上报（同时上报 ICD-10 和 ICD-11）
- 术语映射表维护双向映射
- 历史数据保留 ICD-10，可查询映射到 ICD-11

## 九、医防融合

### 9.1 核心目标

把"医疗"（治已病）和"预防"（治未病）结合，从"以治病为中心"转向"以健康为中心"。

### 9.2 高血压完整链路

```
1. 门诊就诊（HIS 诊断 ICD:I10）
2. 慢病建档（事件触发自动建档）
3. 随访计划生成（自动生成计划）
4. 随访执行（基层医师/村医）
5. 健康干预（控制不达标 → 上转）
6. 数据闭环（汇聚到健康档案）
```

### 9.3 数据触发机制

```
临床事件（Kafka）→ 规则引擎（Drools/LiteFlow）→ 慢病建档 → 随访计划 → 随访执行 → 健康干预
```

### 9.4 医防融合挑战与对策

| 挑战 | 对策 |
| --- | --- |
| 数据标准不统一 | 数据中台术语映射 |
| 数据归属不同 | 三方归属，明确数据权利 |
| 数据时效不同 | 实时 + 批量 + 合并 |
| 业务流程不同 | 流程编排引擎 |
| 数据质量参差 | 质量评分 + 治理工单 |
| 隐私合规 | 授权 + 留痕 + 透明化 |
