## 背景

2026 年，互联网医院已从"政策驱动"走向"业务深水区"。国家政策线：

- **2018 年**：国务院《关于促进"互联网+医疗健康"发展的意见》、国家卫健委《互联网诊疗管理办法（试行）》《互联网医院管理办法（试行）》《远程医疗服务管理规范（试行）》三大文件出台，明确互联网医院的准入、执业、监管框架。
- **2020 年**：新冠疫情催化，互联网诊疗纳入医保支付，互联网医院进入快车道。
- **2022 年**：国家卫健委《互联网诊疗监管细则（试行）》出台，明确"复诊认定、处方监管、医师资质、数据安全"四条红线。
- **2023-2025 年**：国家互联网诊疗监管中心上线，全国互联网医院纳入统一监管；互联网+医保线上结算在 30+ 省市落地。

截至 2025 年底，全国互联网医院数量突破 **3000 家**，但**真正盈利的不超过 5%**。问题不在"建系统"，而在"跑通业务闭环 + 合规监管 + 医保对接 + 处方流转 + 服务体验"五个齿轮的同时啮合。

互联网医院有两种形态：

1. **实体医院互联网化**：实体医院自建互联网医院（如北京协和、上海华山），主要服务本院复诊患者。
2. **依托实体医院的第三方平台**：微医、好大夫在线、京东健康、阿里健康、平安好医生等平台型互联网医院，依托合作实体医院执业。

啄木鸟云健康的体检客户中，部分三甲医院健康管理中心正在自建互联网医院，把"体检异常项 -> 互联网医院在线问诊 -> 慢病随访"打通。本周 Day01 聚焦**互联网医院核心架构与在线问诊业务设计**，为后续电子处方（Day02）、医联体/远程医疗（Day03）、CDSS/AI 辅诊（Day04）、医疗大数据（Day05）、串联整合（Day06）、架构深挖（Day07）打基础。

互联网医院与第1周的衔接点：

- **第1周 Day01 智慧体检**：体检异常项 -> 互联网医院在线问诊 -> 慢病随访 -> 公卫上报
- **第1周 Day02 HIS/EMR**：互联网医院调阅实体医院 EMR 做复诊认定
- **第1周 Day03 HL7/FHIR**：互联网医院与 HIS/LIS 数据交换走 FHIR
- **第1周 Day05 医保对接**：互联网+医保线上结算（互联网医院延伸场景）

互联网医院的工程约束（与电商、IM、支付系统的根本差异）：

1. **强合规监管**：仅复诊不得首诊、麻醉精神药品不得在线开、处方需执业医师 + 主任医师双签、所有诊疗行为 24h 内上报监管平台。
2. **医师资源稀缺**：医生不是 7×24 在线，接单时延 SLA 通常 30 分钟内，视频问诊需医患双端预约。
3. **三流合一**：信息流（处方）+ 资金流（支付）+ 物流（药品配送）+ 票据流（医保结算凭证）必须可追溯对账。
4. **数据互通壁垒**：互联网医院平台 vs 实体医院 HIS/EMR 数据双向同步，需要解决 EMPI、术语映射、数据归属。
5. **隐私合规**：等保三级、个人信息保护法、医疗数据敏感等级最高，问诊记录保存期限 ≥ 15 年。

## 题目一（架构设计题）：互联网医院核心架构与模块拆分

请你设计一套支撑**第三方平台型互联网医院**（如微医、京东健康）的整体架构，并回答以下问题：

1. 你会将互联网医院拆分为哪些核心模块/子系统？至少覆盖：在线问诊（图文/电话/视频）、电子处方、药品配送、检查检验开单、慢病管理、医师资质核验、医保对接、监管上报、医患 IM、视频通话。每个模块的职责边界是什么？哪些模块是"业务核心"，哪些是"基础设施"？
2. 模块之间的调用链路是怎样的？同步链路（创建问诊、开处方、医保预结算）与异步链路（问诊结束触发处方流转、药品配送状态回流、监管上报、慢病随访触发）分别如何走？为什么问诊结束 -> 处方审核 -> 药品配送这条链必须异步？
3. 互联网医院的"问诊订单"（一次问诊交易）和"诊疗事件"（这次问诊的全过程记录：主诉、现病史、既往史、体格检查、辅助检查、诊断、处方、医嘱）是什么关系？为什么需要做这层拆分？图文问诊（异步、可中断）与视频问诊（同步、不可中断）在订单模型上有什么差异？
4. 第三方平台型互联网医院通常对接多家实体医院（执业挂靠），如何做"医院实体 + 科室 + 医师"的三级数据隔离？哪些数据必须按实体医院强隔离（如电子病历调阅、处方签章），哪些可以平台共享（如医师资质库、药品库）？
5. 互联网医院的"医师资质核验"为什么必须前置？如何与国家卫健委医师执业注册信息查询接口对接？医师多点执业在系统中如何表达？

#### 2. 同步与异步链路

**典型同步链路**（患者前台操作，RT < 1s）：

```
患者发起图文问诊
   ↓ 同步
1. 创建问诊订单（Consult 服务）
2. 校验医师资质（DoctorCredential 服务）
3. 校验复诊认定（EMR 调阅服务，判断 6 个月内是否有该科室线下就诊记录）
4. 锁定号源（Schedule 服务）
5. 拉起 IM 会话（IM 服务）
6. 返回问诊订单号 + IM 会话 ID
   ↓
患者开始发消息，医师在 30 分钟内接诊
```

**典型异步链路**（问诊结束后触发，秒级 ~ 分钟级）：

```
医师点击"结束问诊"
   ↓
1. 问诊订单状态置为"已结束"（Consult 服务）
2. 触发"开方"事件（如果医师开了处方）
   ↓ 异步 MQ
3. 处方进入"待审核"队列（RxReview 服务）
4. 药师 30 分钟内审方
   ↓ 处方审核通过
5. 处方流转至药品配送模块（DrugShip 服务）
6. 处方同步至医保预结算（Insurance 服务）
7. 处方上报监管（RegulatorReporter 服务）
   ↓
8. 药企/药店接单配送
9. 配送状态回流（已发货 -> 配送中 -> 已签收）
   ↓
10. 触发慢病随访计划（CDM 服务，如果是慢病患者）
11. 触发公卫上报（如果是公卫体检延伸场景，参考第1周 Day01）
```

**为什么问诊 -> 处方 -> 配送链必须异步**：

- **医师接诊 SLA 30 分钟**：同步等待会让患者前台等待 30 分钟，体验崩坏。
- **药师审方 SLA 30 分钟**：药师不是 7×24 在线，需要排队。同步等待会让医师前台卡住。
- **药品配送 SLA 数小时~数天**：跨城配送根本无法同步等待。
- **监管上报 SLA 24 小时**：监管平台接口可能限流、抖动，必须异步重试。
- **失败补偿需求**：处方审核失败、配送失败、医保结算失败都需要独立的补偿机制，同步链路无法承载。

异步的实现：MQ（RocketMQ/Kafka）+ 状态机 + 定时对账。每个步骤的入站消息幂等（基于处方 ID + 步骤 ID）。

#### 3. 问诊订单 vs 诊疗事件

**问诊订单 ConsultOrder**（交易契约，对账依据）：

| 字段 | 类型 | 说明 |
|------|------|------|
| orderId | String | 订单唯一 ID（雪花算法） |
| patientId | String | 患者 ID |
| doctorId | String | 医师 ID |
| hospitalId | String | 执业挂靠的实体医院 ID |
| deptCode | String | 科室代码 |
| consultType | Enum | 图文 / 电话 / 视频 |
| status | Enum | 待支付/待接诊/问诊中/已结束/已退款 |
| amount | Decimal | 订单金额 |
| payTime | DateTime | 支付时间 |
| startTime | DateTime | 接诊时间 |
| endTime | DateTime | 结束时间 |
| refundStatus | Enum | 退款状态 |

**诊疗事件 ConsultEvent**（临床记录，监管 + 病历）：

| 字段 | 类型 | 说明 |
|------|------|------|
| eventId | String | 事件 ID |
| orderId | String | 关联订单 ID（多对一） |
| chiefComplaint | Text | 主诉 |
| presentIllness | Text | 现病史 |
| pastHistory | Text | 既往史 |
| physicalExam | Text | 体格检查 |
| auxiliaryExam | Text | 辅助检查 |
| diagnosis | List<ICD10> | 诊断（ICD-10/11） |
| prescriptions | List<RxId> | 处方 ID 列表 |
| doctorSign | Bytes | 医师电子签名 |
| eventTime | DateTime | 事件记录时间 |

**为什么做这层拆分**：

1. **关注点分离**：订单关注交易（金额、状态、退款），诊疗事件关注临床（主诉、诊断、处方）。
2. **监管要求**：诊疗事件必须保存 ≥ 15 年（医疗病历保存期限），订单只需保存 5 年（电商交易存档要求）。
3. **数据归属**：订单归属互联网医院平台，诊疗事件归属实体医院（医师执业挂靠的医院）的 EMR。
4. **隐私等级**：订单可对账务/运营人员可见，诊疗事件仅对医师/药师/患者本人可见。
5. **一次问诊可能产生多次事件**：医师可能"接诊 -> 沟通 -> 修改诊断 -> 开方 -> 结束"，每个动作产生一个事件，但订单只有一个。

**图文问诊 vs 视频问诊在订单模型上的差异**：

| 维度 | 图文问诊 | 视频问诊 |
|------|---------|---------|
| 时效 | 异步，可中断（医师 24 小时内多次回复） | 同步，不可中断（连续 15-30 分钟） |
| 状态机 | 待支付 -> 待接诊 -> 问诊中 -> 已结束 | 待支付 -> 预约成功 -> 待呼叫 -> 视频中 -> 已结束 |
| 计费 | 按订单计费（一次图文问诊固定费用） | 按时长计费（视频分钟数） |
| IM 会话 | 必须（消息持久化） | 可选（视频后可补录医嘱） |
| 录制存档 | 不需要 | 必须（监管要求视频问诊全程录制存档 ≥ 15 年） |
| 接诊 SLA | 30 分钟内首次回复 | 准时呼叫（预约时间 ± 5 分钟） |
| 退款规则 | 医师未接诊可全额退款 | 医师未呼叫可全额退款，呼叫后按分钟计费 |

#### 4. 三级数据隔离（医院 + 科室 + 医师）

第三方平台型互联网医院通常对接多家实体医院（如微医对接 5000+ 家医院），数据隔离的关键：

**必须按实体医院强隔离的数据**：

- **电子病历调阅**：患者在本医院 EMR 的就诊记录只能由本医院的医师调阅。FHIR API 必须按 hospitalId 做租户过滤。
- **处方签章**：处方签章的 CA 证书由本医院医师持有，签章后的处方归属本医院。
- **诊疗事件**：诊疗事件归属实体医院的 EMR，平台仅做"代理存储 + 同步回写"。
- **HIS 对接凭证**：每家医院的 HIS 接口凭证（appKey/appSecret）独立管理。
- **监管上报归属**：每家医院独立上报国家监管平台（用本医院的医疗机构执业许可证编码）。

**可以平台共享的数据**：

- **医师资质库**：医师资质核验通过后，平台共享（同一医师可在多家医院多点执业）。
- **药品库**：国家医保药品目录全国统一，平台共享。
- **患者主索引**：患者用身份证 + 手机号注册，平台共享（但患者授权每家医院调阅其档案需独立授权）。
- **术语映射**：ICD-10/11、SNOMED CT、ATC 编码全局共享。

**分片键选择**：

- **订单表 consult_order**：分片键 `hospitalId + patientId`，按医院分库，按患者分表。
- **诊疗事件表 consult_event**：分片键 `hospitalId + orderId`，与订单同库。
- **处方表 prescription**：分片键 `hospitalId + patientId`，与订单同库（处方需 join 订单）。
- **配送表 drug_shipment**：分片键 `patientId`（按患者分库，跨医院聚合配送需求少见）。
- **医师资质表 doctor_credential**：分片键 `doctorId`（单库，平台共享）。

#### 5. 医师资质核验前置

**为什么必须前置**：

- **监管红线**：未取得医师执业证书的人员不得在互联网医院执业。一旦违规开方，互联网医院许可证会被吊销。
- **多点执业合规**：医师必须在执业注册的医院开展互联网诊疗，平台不能"借证开方"。
- **处方签章依赖**：CA 证书绑定医师执业证号，签章前必须核验执业证有效性。
- **科室匹配**：医师只能执业注册范围内的科室（如内科医师不能开外科处方）。

**对接国家卫健委医师执业注册信息查询接口**：

```

1. 医师提交：身份证、医师资格证、医师执业证、职称证、学历证
2. 平台调用国家卫健委"医师执业注册信息查询"接口（https://医师执业查询.国家卫健委.gov.cn）
```

### 作答区

#### 1. 图文问诊消息可靠投递

**完整链路图**：

```
患者端 APP                      服务端                       医师端 PC
   │                              │                              │
   │ 1. 发送消息（含客户端ID）    │                              │
   │ ───────────────────────────> │                              │
   │                              │ 2. 落库（msgId、orderId、     │
   │                              │    senderId、content、ctime） │
   │                              │ 3. 投递 MQ（topic=consult_msg）│
   │                              │ 4. 返回 msgId + 服务端 ctime  │
   │ <─────────────────────────── │                              │
   │ 5. 客户端 ACK                │                              │
   │ ───────────────────────────> │                              │
   │                              │ 6. MQ 消费 -> 推送服务        │
   │                              │ 7. 查询医师在线长连接         │
   │                              │ 8. 通过长连接推送消息         │
   │                              │ ───────────────────────────> │
   │                              │ 9. 医师端 ACK                │
   │                              │ <─────────────────────────── │
   │                              │ 10. 更新消息状态为"已送达"    │
   │                              │                              │
   │                              │ 11. 医师点击查看              │
   │                              │ 12. 医师端发送"已读"上报      │
   │                              │ <─────────────────────────── │
   │                              │ 13. 更新消息状态为"已读"      │
   │                              │ 14. 推送"已读"回执给患者      │
   │                              │ ───────────────────────────> │
```

**可靠性保障**：

| 步骤 | 风险 | 保障措施 |
|------|------|---------|
| 患者端 -> 服务端 | 网络抖动、客户端超时 | 客户端本地消息表 + 失败重试（指数退避）+ 客户端 ID 去重 |
| 服务端落库 | DB 写入失败 | 主从同步 + 半同步复制；写入失败回滚 MQ |
| 服务端 -> MQ | MQ 投递失败 | 同步发送 + 失败重试；MQ 生产者确认机制（publisher confirm） |
| MQ -> 推送服务 | MQ 消费失败 | 消费失败重试 N 次 + 死信队列；幂等消费（msgId 去重） |
| 推送服务 -> 医师长连接 | 长连接断开 | 离线消息入库 + 医师上线后拉取；多端推送（PC + PAD + 手机） |
| 医师端 ACK | ACK 丢失 | 服务端定时扫描"未 ACK"消息，二次推送 |

**消息三状态**：

| 状态 | 含义 | 实现 |
|------|------|------|
| 已发送 sent | 客户端发送成功，服务端已落库 | 服务端落库 + 客户端 ACK |
| 已送达 delivered | 服务端推送到医师端，医师端 ACK | 长连接 ACK + 状态机更新 |
| 已读 read | 医师打开会话查看 | 医师端"已读"上报 + 批量更新 |

**幂等与去重**：

- 客户端消息 ID（UUID v7 + 患者ID + 时间戳），服务端用 Redis SETNX 做幂等校验，TTL 24h。
- MQ 消费幂等：msgId 作为去重键，Redis SETNX + DB 唯一索引（msgId + orderId）双保险。
- 消息顺序：单问诊单队列（MQ 分区键 = orderId），保证同一问诊的消息顺序消费。

**顺序保障**：

- 患者端按发送顺序携带 seq（递增整数），服务端按 seq 落库（msgId 全局唯一，seq 问诊内递增）。
- 医师端按 seq 排序展示。
- 重试消息携带原 seq，服务端去重时按 seq + senderId + orderId 三元组去重。

#### 2. 视频问诊的信令与媒体流

**协议分层**：

| 层 | 协议 | 用途 |
|----|------|------|
| 信令层 | WebSocket + 自定义 JSON 协议（或 SIP） | 呼叫邀请、接听、拒绝、挂断、媒体协商（SDP） |
| 媒体层 | WebRTC（SRTP 加密）+ UDP | 音视频实时传输 |
| 媒体转发 | SFU（Selective Forwarding Unit）| 多方通话、媒体路由 |
| NAT 穿透 | STUN/TURN (ICE) | 内网穿透、中继 |
| 录制存档 | MediaServer 录制 + MP4 落盘 | 合规存档 |

**为什么信令与媒体流必须分离**：

- **可靠 vs 实时**：信令必须可靠（TCP/WebSocket），媒体流必须实时（UDP，允许丢包）。
- **网络路径不同**：信令走 HTTPS/WSS 长连接（443 端口），媒体流走 UDP（SRTP，3478/5349 端口）。
- **故障域隔离**：信令服务故障不能影响正在进行的视频通话（媒体流继续走 SFU 转发）。
- **扩展性不同**：信令服务可水平扩展（无状态），SFU 媒体服务单机有上限（CPU/带宽），需要分房间路由。

**NAT 穿透在医疗场景的特殊性**：

1. **医院内网严格**：医院 HIS/EMR 网络通常与互联网物理隔离，视频问诊需走"互联网出口 + DMZ"，防火墙策略严格。STUN 直连成功率低，必须部署 TURN 中继。
2. **4G/5G 切换**：医师可能在院内 Wi-Fi 切 4G/5G，ICE 重协商必须支持网络切换不中断。
3. **等保三级合规**：所有媒体流必须端到端加密（SRTP + DTLS），TURN 中继服务器不能解密媒体流（SRTP 转发不解密）。
4. **录制存档合规**：监管要求视频问诊全程录制，录制文件需加水印（医师姓名 + 患者姓名 + 时间），存储 ≥ 15 年， hashes 防篡改。

**视频录制存档合规**：

```
录制流程：
1. SFU 服务端录制（不依赖客户端，避免客户端篡改）
2. 录制文件加水印：医师姓名 + 患者姓名 + 身份证后4位 + 时间戳
3. 录制文件计算 SHA256 hash，存入区块链/可信日志
4. 录制文件加密存储（AES-256），密钥由 KMS 管理
5. 录制文件元数据：orderId、doctorId、patientId、startTime、endTime、duration、hash
6. 上传对象存储（OSS），权限：仅监管审计岗可读
7. 保存期限：≥ 15 年（与病历保存期限一致）
```

#### 3. 问诊会话状态机

**完整状态机**：

```
                  ┌─────────────┐
                  │  待支付     │
                  │  PENDING_PAY│
                  └──────┬──────┘
                         │ 患者支付成功
                         ▼
                  ┌─────────────┐
                  │  待接诊     │  超时 30 分钟未接诊
                  │  PENDING    │ ──────────────────┐
                  └──────┬──────┘                   │
                         │ 医师接诊                  ▼
                         ▼                  ┌─────────────┐
                  ┌─────────────┐           │  已退款     │
                  │  问诊中     │           │  REFUNDED   │
                  │  CONSULTING │           └─────────────┘
                  └──────┬──────┘
                         │ 医师结束 / 患者结束 / 系统超时
                         ▼
                  ┌─────────────┐
                  │  已结束     │
                  │  FINISHED   │
                  └─────────────┘

异常状态分支：
- 待支付 -> 患者取消 -> 已关闭 CLOSED
- 待接诊 -> 医师离线超 30 分钟 -> 自动退款 -> 已退款
- 待接诊 -> 患者取消 -> 退款（扣 5% 手续费）-> 已退款
- 问诊中 -> 医师离线超 5 分钟 -> 系统强制结束 -> 已结束（标记"医师离线"）
- 问诊中 -> 患者投诉 -> 人工介入 -> 已退款 / 已结束
```

**每个状态的进入/退出/超时**：

| 状态 | 进入条件 | 退出条件 | 超时 |
|------|---------|---------|------|
| 待支付 | 创建订单 | 支付成功 -> 待接诊；患者取消 -> 已关闭 | 15 分钟未支付自动关闭 |
| 待接诊 | 支付成功 | 医师接诊 -> 问诊中；超时 -> 已退款 | 30 分钟未接诊自动退款 |
| 问诊中 | 医师接诊 | 医师结束 -> 已结束；患者结束 -> 已结束 | 24 小时未结束自动关闭（图文）；30 分钟自动结束（视频） |
| 已结束 | 医师/患者/系统结束 | 终态 | - |
| 已退款 | 退款成功 | 终态 | - |
| 已关闭 | 患者取消未支付 | 终态 | - |

**状态机实现**：

- **方案 A：状态模式 + 事件驱动**（推荐）
  - 每个状态实现 `handle(event)` 方法，事件驱动状态流转
  - 优点：状态流转逻辑集中，易测试
  - 缺点：状态多时代码膨胀
- **方案 B：数据库状态字段 + 定时扫描**
  - 订单表 status 字段，定时任务扫描超时订单
  - 优点：实现简单
  - 缺点：状态流转逻辑分散，易出 bug
- **医疗场景推荐：方案 A + 定时扫描兜底**
  - 主流程用状态模式（接诊、结束、退款）
  - 定时扫描兜底超时（待接诊 30 分钟、问诊中 24 小时）
  - 状态变更必须走"事件 -> 状态机 -> 落库"链路，禁止直接 update status

#### 4. 长连接方案选型

**自研 Netty + WebSocket vs 第三方 IM SDK**：

| 维度 | 自研 Netty + WebSocket | 第三方 IM SDK |
|------|----------------------|---------------|
| 开发成本 | 高（需解决长连接、心跳、重连、消息可靠投递、多端同步） | 低（SDK 集成 + API 调用） |
| 定制能力 | 强（医疗合规、消息审计、录制存档可定制） | 弱（受 SDK 能力限制） |
| 运维成本 | 高（需运维长连接网关集群、监控、扩容） | 低（SLA 由厂商保障） |
| 数据归属 | 自有数据库，数据合规可控 | 第三方数据库，数据出境/合规风险 |
| 单价成本 | 固定人力成本 | 按 DAU/MAU/消息量计费，规模化后贵 |
| 医疗合规 | 可定制满足等保三级、监管存档 | 部分厂商不满足医疗合规 |

**医疗场景下自研长连接网关的特殊问题**：

1. **多端同步**：医师 PC + PAD + 手机同时登录，消息需多端推送。每条消息推送时查询医师所有在线连接，逐个推送。
2. **消息合规存档**：所有医患沟通消息必须存档 ≥ 15 年，包括撤回的消息（撤回不删除，标记"已撤回"+ 撤回原因）。
3. **医师多端登录互斥**：同一医师同一类型端只能一处登录（如 PC 只能一处），但 PC + PAD + 手机可同时在线。在线状态变更需同步推送。
4. **消息撤回与审计**：医师撤回消息需 24h 内可撤回，撤回后患者端显示"医师撤回了一条消息"。撤回消息仍保留审计。
5. **敏感词过滤**：医患沟通内容需敏感词过滤（如违规广告、辱骂），命中后拦截 + 人工审核。
6. **会话隔离**：不同问诊订单的消息严格隔离，禁止跨订单推送。

**自研长连接网关架构**：

```
患者端/医师端
   │ WebSocket / WSS
   ▼
┌────────────────────────────────────┐
│  长连接网关集群（Netty）           │
│  - 鉴权（JWT）                     │
│  - 心跳保活（30s）                 │
│  - 连接路由（按 userId 路由到节点）│
└────────────────────────────────────┘
   │ 内部 RPC / MQ
   ▼
┌────────────────────────────────────┐
│  消息推送服务（Push Service）       │
│  - 查询用户在线连接                 │
│  - 多端推送                         │
│  - 离线消息入未读队列               │
└────────────────────────────────────┘
   │
   ▼
┌────────────────────────────────────┐
│  消息存储（MySQL + Redis）         │
│  - 消息表（持久化 ≥ 15 年）         │
│  - 未读数（Redis hash）            │
│  - 在线状态（Redis set）           │
└────────────────────────────────────┘
```

#### 5. 离线消息与未读消息

**医师下班后患者发的消息**：

1. 消息照常落库（医师离线不影响患者发送）。
2. 推送服务查询医师在线状态，发现离线，转 APNS/FCM/厂商通道推送（"您有 3 条新问诊消息"）。
3. 消息进入医师未读队列（Redis hash: `unread:{doctorId}:{orderId}` -> count）。

**医师次日登录后拉取未读**：

```
医师登录流程：
1. 长连接建立，鉴权通过
2. 客户端调用 getUnreadCount() 接口
3. 服务端返回：每个问诊订单的未读数 + 总未读数
4. 医师点击某订单进入会话
5. 客户端调用 getMessages(orderId, lastMsgId, pageSize) 拉取历史消息
6. 服务端分页返回 + 同时将该订单未读数清零
7. 服务端推送"已读"回执给患者端
```

**多端未读数一致**：

- **服务端为权威源**：未读数存储在 Redis hash，多端拉取同一份数据。
- **多端"已读"同步**：医师在 PC 端读了消息 -> PC 端上报"已读" -> 服务端清零未读数 -> 推送"未读数变更"给 PAD/手机 -> PAD/手机未读数同步清零。
- **会话级未读数**：`unread:{userId}:{orderId}` -> count，按问诊订单隔离。
- **总未读数**：`unread:{userId}:total` -> count，所有订单未读之和。
- **多端登录时**：每端拉取一次未读数，保证初始一致。

### 与架构师水平的差距及补足方向

**差距1：医疗 IM 的合规存档深度不足**

- **现状**：能设计消息可靠投递链路，但对医疗 IM 的合规存档（撤回消息、修改消息、敏感词、医生语音转文字、患者照片脱敏）工程化经验不足。
- **架构师水平**：能讲清医疗 IM 与普通 IM 的差异（撤回消息仍存档、敏感词命中后人工审核流、患者隐私照片 OCR 脱敏、医患沟通全部纳入病历附件），能设计满足等保三级与医疗监管的存档方案。
- **补足方向**：
  - 精读《互联网诊疗监管细则》关于"问诊记录存档"的条款
  - 学习医疗 IM 合规存档方案（如韵达医疗、海泰医疗的方案）
  - 复盘啄木鸟云健康体检咨询 IM 的存档实现，识别合规短板

**差距2：WebRTC 在医疗场景的工程化经验不足**

- **现状**：知道 WebRTC + SFU + STUN/TURN，但医疗场景的录制存档合规、等保三级网络隔离、医师多端切换的工程实战不足。
- **架构师水平**：能讲清"医院内网 + 互联网 + 医师 4G 切换 + 患者家庭 Wi-Fi"四场景下的 ICE 重协商、TURN 中继部署、录制文件 SHA256 上链、密钥管理（KMS）。
- **补足方向**：
  - 学习医疗视频问诊的开源方案（如 Jitsi Meet、Mediasoup 的医疗定制案例）
  - 学习等保三级对音视频通信的加密要求（SRTP + DTLS + 密钥轮换）
  - 学习 KMS（如阿里云 KMS、HashiCorp Vault）在录制密钥管理中的应用

**差距3：长连接网关的水平扩展与一致性不熟**

- **现状**：知道长连接网关集群，但对"用户连接路由、跨节点推送、连接迁移（节点下线）、灰度发布"的工程实现不熟。
- **架构师水平**：能设计基于一致性哈希 + Redis 路由表的长连接网关集群，节点下线时连接平滑迁移，灰度发布时按医师 ID 灰度。
- **补足方向**：
  - 学习 Netty 长连接网关集群方案（如 Dubbo、Spring Cloud Gateway 的连接管理）
  - 学习"用户 -> 节点"路由表的设计（Redis hash + 一致性哈希）
  - 学习长连接网关的灰度发布（按 userId 路由到新版本节点）

### 作答区

#### 1. 复诊认定

**复诊认定的本质**：监管要求"互联网医院不得对首诊患者开展互联网诊疗活动"。复诊 = 患者在 6 个月内（部分地区 3 个月）有该科室线下就诊记录。

**3 种复诊认定策略对比**：

| 策略 | 实现 | 优点 | 缺点 | 可靠性 | 合规性 |
|------|------|------|------|--------|--------|
| HIS 就诊记录调阅 | 调阅实体医院 HIS 的就诊记录 | 数据权威、自动判定 | 依赖 HIS 接口、跨医院不可用 | 高 | 高 |
| 电子病历调阅 | 通过区域卫生信息平台调阅 EMR | 跨医院可用 | 平台覆盖不全 | 中 | 高 |
| 患者自述 + 医师确认 | 患者勾选"6 个月内线下就诊" + 医师在问诊中确认 | 实现简单 | 患者可能撒谎、医师可能放水 | 低 | 低（部分省份不允许） |

**6 个月内合作实体医院就诊记录的复诊认定**：

```
患者在互联网医院发起问诊（科室：心内科）
   ↓
1. 互联网医院调用合作实体医院 HIS 接口（FHIR Encounter 资源）
   GET /Encounter?patient={patientId}&date=ge2025-07-13&date=le2026-01-13
   &class=ambulatory
   &department=CARD
   ↓
2. HIS 返回该患者在心内科的就诊记录列表
   ↓
3. 互联网医院校验：记录数 > 0 -> 复诊认定通过
   ↓
4. 复诊认定结果落库（patientId、hospitalId、deptCode、认定时间、就诊记录 ID 列表）
   ↓
5. 允许问诊继续；否则拦截问诊，提示"该科室需线下首诊"
```

**跨医院复诊认定**：

- **方案 1：区域卫生信息平台调阅**（如省级健康信息平台）
  - 患者授权后，平台调阅该患者在该省所有医院的就诊记录
  - 优点：覆盖广
  - 缺点：依赖平台覆盖率，部分省份未覆盖基层医院
- **方案 2：互联网医院间复诊认定互认**
  - 患者在 A 互联网医院的复诊认定结果，B 互联网医院可调阅
  - 优点：互联网医院间打通
  - 缺点：行业互认机制尚未建立
- **方案 3：医保结算记录反推**
  - 通过医保结算记录反推患者 6 个月内的就诊历史
  - 优点：覆盖所有医保定点医院
  - 缺点：医保数据访问需医保局授权
- **当前主流方案**：方案 1 + 方案 3 组合（区域卫生信息平台 + 医保结算记录）

#### 2. 处方流转的"四流合一"

**四流合一的核心**：每流有唯一流水号，跨系统可追溯对账。

| 流 | 流水号 | 关键字段 |
|----|--------|---------|
| 信息流 | rxId（处方号） | rxId、orderId、doctorId、patientId、hospitalId、drugCode、quantity、用法用量、签章 |
| 资金流 | payId（支付号） | payId、rxId、amount、payType（自费/医保）、payTime、payChannel |
| 物流 | shipId（配送号） | shipId、rxId、pharmacyId、courierId、shipStatus、deliveryTime、signTime |
| 票据流 | invoiceId（票据号） | invoiceId、rxId、payId、insuranceSettlementId、invoiceType（医疗票据/电子发票） |

**处方全流程追溯链路**：

```
开立（rxId 生成）
   ↓ traceId: rx_open
   - 字段：rxId, orderId, doctorId, patientId, hospitalId, drugItems, openTime
   - 状态：DRAFT
   - 追溯：doctorSign（医师电子签名）

审核（药师审方）
   ↓ traceId: rx_review
   - 字段：rxId, pharmacistId, reviewResult, reviewTime
   - 状态：DRAFT -> REVIEWED / REJECTED
   - 追溯：pharmacistSign（药师电子签名）

流转（处方流转平台）
   ↓ traceId: rx_transfer
   - 字段：rxId, pharmacyId, transferTime
   - 状态：REVIEWED -> TRANSFERRED
   - 追溯：transferLog（流转日志）

配送（药企/药店配送）
   ↓ traceId: rx_ship
   - 字段：rxId, shipId, courierId, shipTime, status（已发货/配送中/已签收/已退回）
   - 状态：TRANSFERRED -> SHIPPED -> DELIVERED
   - 追溯：shipLog（配送轨迹）

签收（患者签收）
   ↓ traceId: rx_sign
   - 字段：rxId, signTime, signPerson（本人/代签）
   - 状态：DELIVERED -> SIGNED
   - 追溯：signPhoto（签收照片）、signGeo（签收地理位置）

退药（特殊情况退药）
   ↓ traceId: rx_return
   - 字段：rxId, returnReason, returnTime, refundAmount
   - 状态：SIGNED -> RETURNED
   - 追溯：returnLog（退药日志）、refundLog（退款日志）
```

**追溯 ID 设计**：

- 主键 ID：rxId（雪花算法，全局唯一）
- 关联 ID：orderId（问诊订单）、payId（支付）、shipId（配送）、invoiceId（票据）
- 追踪 ID：traceId（每个节点的链路追踪 ID，用于全链路日志聚合）
- 状态机 ID：rxStatus（DRAFT/REVIEWED/TRANSFERRED/SHIPPED/DELIVERED/SIGNED/RETURNED）
- 每个节点的 traceId 通过日志关联（如 SkyWalking traceId），全链路可追溯。

**对账机制**：

- **日终对账**：每天凌晨对账，校验信息流（处方数）= 资金流（支付数）= 物流（配送数）= 票据流（票据数）。
- **异常告警**：四流不齐的处方进入"异常队列"，人工介入。
- **监管追溯**：监管平台可通过 rxId 一键追溯全流程。

#### 3. 医师电子签名与处方签章

**法律效力**：

- 《中华人民共和国电子签名法》（2005 年实施，2019 年修订）：可靠的电子签名与手写签名/盖章具有同等法律效力。
- 《卫生系统电子认证服务管理办法》（2010 年）：卫生健康领域电子认证服务规范。
- 《互联网诊疗监管细则（试行）》（2022 年）：处方需医师电子签名，电子签名需 CA 证书认证。

**CA 证书流程**：

```
颁发：
1. 医师提交身份证、执业证、资格证至互联网医院
2. 互联网医院核验 -> 调用 CA 机构（CFCA/北京CA/上海CA）申请证书
3. CA 机构核验身份 -> 颁发证书（含公钥、私钥、有效期、医师信息）
4. 证书私钥存储在 USBKey 或软件证书库

续期：
1. 证书到期前 30 天提醒
2. 医师提交续期申请 -> CA 机构重新核验 -> 颁发新证书
3. 旧证书标记"已过期"

吊销：
1. 医师离职/违规 -> 互联网医院申请吊销
2. CA 机构将证书加入 CRL（吊销列表）
3. 后续签名验证时查询 CRL，吊销证书签名无效
```

**处方签章的"数字签名 + 时间戳 + CA 证书"**：

```
处方签章流程：
1. 医师开方 -> 处方内容（rxId、药品、用法用量）序列化为 JSON
2. 医师 USBKey/软件证书库对 JSON 计算 SHA256 -> 用私钥加密 -> 签名值
3. 调用时间戳服务（TSA）-> 获取时间戳 token（含签名时间 + TSA 签名）
4. 处方 + 签名值 + 时间戳 + CA 证书 -> 落库

防篡改验证：
1. 取处方 JSON -> 计算 SHA256
2. 用 CA 证书公钥解密签名值 -> 得到 SHA256
3. 比对两个 SHA256 -> 一致则未被篡改
4. 校验时间戳 -> 确认签名时间在证书有效期内
5. 校验 CA 证书是否在 CRL（吊销列表）-> 未吊销则有效
```

**医师离职后已签处方处理**：

- **已签处方不可撤销**：处方一旦签发且患者已签收，法律效力持续。
- **离职后吊销证书**：CA 证书加入 CRL，离职后医师无法再开新方。
- **已签处方继续有效**：历史处方的法律效力不因医师离职而失效。
- **病历归档**：医师离职前所有处方/诊疗事件归档至实体医院 EMR。

#### 4. 互联网诊疗监管上报

**国家互联网诊疗监管中心要求上报的数据**：

| 数据类别 | 字段 | 上报频率 |
|---------|------|---------|
| 机构信息 | 机构许可证、执业范围、地址、负责人 | 注册时一次性 + 变更时 |
| 医师信息 | 医师执业证、资格证、多点执业、科室 | 入驻时 + 变更时 |
| 患者信息 | 身份证、手机号、就诊卡号 | 首次就诊时 + 变更时 |
| 问诊记录 | orderId、医师、患者、起止时间、问诊类型、诊断 | 实时（问诊结束后 1h 内） |
| 处方信息 | rxId、药品、用法用量、签章、审核药师 | 实时（处方签发后 1h 内） |
| 不良反应 | 患者用药后不良反应事件 | 24h 内 |
| 投诉举报 | 患者投诉、医师违规 | 24h 内 |

**数据格式**：

- 国家互联网诊疗监管中心统一接口规范（基于 FHIR 资源模型 + 国标字段扩展）
- 数据格式：JSON（部分历史接口仍用 XML）
- 传输协议：HTTPS + 双向 TLS 认证

**上报失败兜底**：

1. **本地暂存**：上报失败的数据进入"待上报"队列，本地 DB 暂存。
2. **指数退避重试**：1min、5min、30min、2h、24h 重试 5 次。
3. **死信告警**：5 次失败后进入死信队列，告警人工介入。
4. **次日批量补报**：定时任务每日凌晨补报前一天失败的数据。
5. **监管追溯**：所有上报记录（成功/失败）保留 ≥ 15 年，监管审计时可追溯。

**中药饮片处方特殊管理**：

| 禁忌类型 | 实现 |
|---------|------|
| 十八反 | 药品库标记"反"药对，开方时校验配伍 |
| 十九畏 | 药品库标记"畏"药对，开方时校验配伍 |
| 剂量上限 | 每味药有"日常用量上限"，开方时校验 |
| 妊娠禁用 | 妊娠期患者标记，开方时校验禁忌 |
| 毒性中药 | 如附子、川乌、马钱子，需特殊审批 |

实现：

```java
public class TcmRxValidator {
    public void validate(TcmPrescription rx) {
        // 1. 十八反十九畏校验
        for (DrugPair pair : rx.getDrugPairs()) {
            if (incompatibilityTable.isIncompatible(pair.getDrug1(), pair.getDrug2())) {
                throw new RxException("配伍禁忌：" + pair.getDrug1() + "与" + pair.getDrug2() + "相反");
            }
        }
        // 2. 剂量上限校验
        for (TcmDrug drug : rx.getDrugs()) {
            if (drug.getDosage() > drug.getMaxDosage()) {
                throw new RxException(drug.getName() + "剂量超上限：" + drug.getDosage() + " > " + drug.getMaxDosage());
            }
        }
        // 3. 妊娠禁用校验
        if (rx.getPatient().isPregnant()) {
            for (TcmDrug drug : rx.getDrugs()) {
                if (drug.isPregnancyContraindicated()) {
                    throw new RxException(drug.getName() + "妊娠期禁用");
                }
            }
        }
        // 4. 毒性中药审批校验
        for (TcmDrug drug : rx.getDrugs()) {
            if (drug.isToxic() && !rx.hasSpecialApproval()) {
                throw new RxException(drug.getName() + "为毒性中药，需特殊审批");
            }
        }
    }
}
```

### 与架构师水平的差距及补足方向

**差距1：医疗合规的"工程化 + 可审计"深度不足**

- **现状**：知道监管要求，但工程化映射（如复诊认定的多策略组合、处方权限的三道防线、监管上报的失败兜底）不够深。
- **架构师水平**：能讲清每个合规要求的"代码层 + 数据层 + 流程层 + 审计层"四层映射，能把合规做成可观测、可审计、可回溯的工程能力。
- **补足方向**：
  - 精读《互联网诊疗监管细则（试行）》全文，逐条列出工程化映射点
  - 学习医疗合规的"可审计"设计（操作日志、数据变更日志、不可变审计表）
  - 复盘啄木鸟云健康合作医院的互联网医院合规实现

**差距2：处方签章的密码学底层不熟**

- **现状**：知道数字签名 + 时间戳 + CA 证书，但底层密码学（RSA vs SM2、SHA256 vs SM3、PKCS#7 签名格式、PAdES 长效签名）不熟。
- **架构师水平**：能讲清国密算法（SM2/SM3/SM4）在医疗电子签名中的应用、PAdES 长效签名（应对证书过期后的验签）、密钥管理（HSM 硬件安全模块）。
- **补足方向**：
  - 学习国密算法 SM2/SM3/SM4（医疗行业强制使用国密）
  - 学习 PAdES（PDF 高级电子签名）应对证书过期
  - 学习 HSM（硬件安全模块）在密钥管理中的应用
  - 学习 CFCA/北京 CA 的医疗电子签名 SDK

**差距3：中药处方管理不熟**

- **现状**：知道十八反十九畏，但中药饮片处方管理的工程化（配伍禁忌表、剂量上限、毒性中药分级、妊娠禁用）经验不足。
- **架构师水平**：能设计完整的中药处方审核引擎（配伍 + 剂量 + 毒性 + 妊娠 + 特殊人群），能与药典委员会的中药数据库对接。
- **补足方向**：
  - 学习《中华人民共和国药典》中药饮片部分
  - 学习中药处方审核引擎的开源方案（如中药智能审核系统）
  - 调研合作中医医院的中药处方审核实现

**差距4：监管上报的工程化深度不足**

- **现状**：知道要上报，但上报的"实时 + 日批 + 月批"组合、失败兜底、追溯链路设计经验不足。
- **架构师水平**：能设计高可用的监管上报通道（实时 + 兜底 + 死信 + 补报 + 追溯），能应对监管平台接口变更（版本兼容）。
- **补足方向**：
  - 学习国家互联网诊疗监管中心的上报接口规范
  - 学习监管上报的版本兼容设计（多版本接口共存）
  - 学习监管审计的"不可变审计表"设计（append-only + 哈希链）
