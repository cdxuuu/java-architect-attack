## 一、互联网医院核心架构（Day01）

### 差距1.1：互联网医院监管细则的工程化映射不深
> Day1发现

- **现状**：知道《互联网诊疗监管细则（试行）》规定了"复诊认定、处方监管、医师资质、数据安全"四条红线，但工程化映射（如何用代码强制约束复诊认定、如何用状态机约束处方权限）不够深
- **架构师水平**：能讲清每条监管细则对应到代码的哪个校验点、哪个状态机约束、哪个数据流拦截器，能把"合规"做成可观测、可审计、可回溯的工程能力
- **补足方向**：
  - 精读《互联网诊疗监管细则（试行）》全文，逐条列出工程化映射点
  - 学习国家互联网诊疗监管中心的上报接口规范（数据格式、字段、频率）
  - 复盘啄木鸟云健康合作医院的互联网医院执业校验实现

### 差距1.2：第三方平台型 vs 实体医院型互联网医院架构差异不清
> Day1发现

- **现状**：知道有两种形态，但架构差异（数据归属、监管上报主体、医保结算主体、处方签章主体）模糊
- **架构师水平**：能对比两种形态在"医师执业挂靠、处方签章、医保结算、监管上报、EMR 归属"五个维度的架构差异，能根据业务规模选择合适形态
- **补足方向**：
  - 调研微医、京东健康、好大夫在线的平台型互联网医院架构（公开资料、年报、技术博客）
  - 调研北京协和、上海华山等实体医院互联网医院架构
  - 学习"实体医院 + 第三方平台"合作模式的法律协议与系统分工

### 差距1.3：互联网医院的商业模型与运营指标不熟
> Day1发现

- **现状**：偏技术视角，对互联网医院的商业模型（医师服务费、药品 GMV、医保占比、复购率）不熟
- **架构师水平**：能从业务模型倒推技术架构（如药品 GMV 占比高 -> 药品配送模块要支持千万级订单/日，医师服务费占比高 -> 问诊 SLA 必须严格）
- **补足方向**：
  - 阅读京东健康、阿里健康、平安好医生的年报，拆解收入结构
  - 学习互联网医院的运营指标（医师接诊率、患者满意度、处方转化率、复购率）
  - 与运营团队复盘啄木鸟云健康合作医院的互联网医院运营数据

### 差距1.4：跨医院复诊认定的工程实战不足
> Day1发现

- **现状**：知道复诊认定的策略（HIS 调阅、区域平台、患者自述），但跨医院复诊认定的工程实战（区域卫生信息平台调阅、医保结算记录反推）经验不足
- **架构师水平**：能设计多策略组合的复诊认定方案（HIS 调阅 + 区域平台 + 医保记录），能处理"患者首次在互联网医院但 6 个月内在合作实体医院有就诊记录"等边界 case
- **补足方向**：
  - 学习省级健康信息平台的接入规范与覆盖率
  - 学习医保结算记录反推就诊历史的合规路径
  - 调研互联网医院间复诊认定互认的行业进展

### 差距1.5：医师多点执业的工程化表达与签章归属不熟
> Day1发现，延续第1周差距2.5（医疗数据归属与跨机构授权）

- **现状**：知道医师多点执业概念，但工程化表达（医师 -> 医院 -> 科室三级挂靠关系、处方签章归属当前执业医院）不熟
- **架构师水平**：能设计医师多点执业的数据模型（practiceSites 数组 + isPrimary 标记），每次开方使用当前执业挂靠医院的 CA 证书，处方归属当前医院
- **补足方向**：
  - 学习国家卫健委多点执业备案系统对接
  - 学习 CA 证书在多点执业场景下的颁发与吊销流程
  - 复盘啄木鸟云健康合作医院的多点执业管理实现

## 二、医患 IM 与视频问诊（Day01）

### 差距2.1：医疗 IM 的合规存档深度不足
> Day1发现

- **现状**：能设计消息可靠投递链路，但对医疗 IM 的合规存档（撤回消息、修改消息、敏感词、医生语音转文字、患者照片脱敏）工程化经验不足
- **架构师水平**：能讲清医疗 IM 与普通 IM 的差异（撤回消息仍存档、敏感词命中后人工审核流、患者隐私照片 OCR 脱敏、医患沟通全部纳入病历附件），能设计满足等保三级与医疗监管的存档方案
- **补足方向**：
  - 精读《互联网诊疗监管细则》关于"问诊记录存档"的条款
  - 学习医疗 IM 合规存档方案（如韵达医疗、海泰医疗的方案）
  - 复盘啄木鸟云健康体检咨询 IM 的存档实现，识别合规短板

### 差距2.2：WebRTC 在医疗场景的工程化经验不足
> Day1发现

- **现状**：知道 WebRTC + SFU + STUN/TURN，但医疗场景的录制存档合规、等保三级网络隔离、医师多端切换的工程实战不足
- **架构师水平**：能讲清"医院内网 + 互联网 + 医师 4G 切换 + 患者家庭 Wi-Fi"四场景下的 ICE 重协商、TURN 中继部署、录制文件 SHA256 上链、密钥管理（KMS）
- **补足方向**：
  - 学习医疗视频问诊的开源方案（如 Jitsi Meet、Mediasoup 的医疗定制案例）
  - 学习等保三级对音视频通信的加密要求（SRTP + DTLS + 密钥轮换）
  - 学习 KMS（如阿里云 KMS、HashiCorp Vault）在录制密钥管理中的应用

### 差距2.3：长连接网关的水平扩展与一致性不熟
> Day1发现，延续第5周差距2.3（微服务长连接网关）

- **现状**：知道长连接网关集群，但对"用户连接路由、跨节点推送、连接迁移（节点下线）、灰度发布"的工程实现不熟
- **架构师水平**：能设计基于一致性哈希 + Redis 路由表的长连接网关集群，节点下线时连接平滑迁移，灰度发布时按医师 ID 灰度
- **补足方向**：
  - 学习 Netty 长连接网关集群方案（如 Dubbo、Spring Cloud Gateway 的连接管理）
  - 学习"用户 -> 节点"路由表的设计（Redis hash + 一致性哈希）
  - 学习长连接网关的灰度发布（按 userId 路由到新版本节点）

### 差距2.4：消息多端同步与未读数一致性不熟
> Day1发现

- **现状**：知道多端推送，但多端"已读"同步、未读数一致性（PC + PAD + 手机同时登录）的工程实现细节不熟
- **架构师水平**：能设计服务端权威未读数 + 多端"已读"事件广播的方案，保证多端未读数最终一致
- **补足方向**：
  - 学习主流 IM 的多端同步方案（如微信、钉钉的技术博客）
  - 学习操作日志（OpLog）在多端同步中的应用
  - 学习 CRDT 在协同编辑场景的参考价值

## 三、合规监管与处方签章（Day01）

### 差距3.1：医疗合规的"工程化 + 可审计"深度不足
> Day1发现，延续第1周差距5.X（医保合规）

- **现状**：知道监管要求，但工程化映射（如复诊认定的多策略组合、处方权限的三道防线、监管上报的失败兜底）不够深
- **架构师水平**：能讲清每个合规要求的"代码层 + 数据层 + 流程层 + 审计层"四层映射，能把合规做成可观测、可审计、可回溯的工程能力
- **补足方向**：
  - 精读《互联网诊疗监管细则（试行）》全文，逐条列出工程化映射点
  - 学习医疗合规的"可审计"设计（操作日志、数据变更日志、不可变审计表）
  - 复盘啄木鸟云健康合作医院的互联网医院合规实现

### 差距3.2：处方签章的密码学底层不熟
> Day1发现

- **现状**：知道数字签名 + 时间戳 + CA 证书，但底层密码学（RSA vs SM2、SHA256 vs SM3、PKCS#7 签名格式、PAdES 长效签名）不熟
- **架构师水平**：能讲清国密算法（SM2/SM3/SM4）在医疗电子签名中的应用、PAdES 长效签名（应对证书过期后的验签）、密钥管理（HSM 硬件安全模块）
- **补足方向**：
  - 学习国密算法 SM2/SM3/SM4（医疗行业强制使用国密）
  - 学习 PAdES（PDF 高级电子签名）应对证书过期
  - 学习 HSM（硬件安全模块）在密钥管理中的应用
  - 学习 CFCA/北京 CA 的医疗电子签名 SDK

### 差距3.3：中药处方管理不熟
> Day1发现

- **现状**：知道十八反十九畏，但中药饮片处方管理的工程化（配伍禁忌表、剂量上限、毒性中药分级、妊娠禁用）经验不足
- **架构师水平**：能设计完整的中药处方审核引擎（配伍 + 剂量 + 毒性 + 妊娠 + 特殊人群），能与药典委员会的中药数据库对接
- **补足方向**：
  - 学习《中华人民共和国药典》中药饮片部分
  - 学习中药处方审核引擎的开源方案（如中药智能审核系统）
  - 调研合作中医医院的中药处方审核实现

### 差距3.4：监管上报的工程化深度不足
> Day1发现，延续第1周差距5.X（医保监管上报）

- **现状**：知道要上报，但上报的"实时 + 日批 + 月批"组合、失败兜底、追溯链路设计经验不足
- **架构师水平**：能设计高可用的监管上报通道（实时 + 兜底 + 死信 + 补报 + 追溯），能应对监管平台接口变更（版本兼容）
- **补足方向**：
  - 学习国家互联网诊疗监管中心的上报接口规范
  - 学习监管上报的版本兼容设计（多版本接口共存）
  - 学习监管审计的"不可变审计表"设计（append-only + 哈希链）

### 差距3.5：处方流转"四流合一"的对账与异常处理不熟
> Day1发现

- **现状**：知道信息流/资金流/物流/票据流四流合一的概念，但日终对账的工程实现（四流不齐的异常处理、跨系统追溯、监管追溯）经验不足
- **架构师水平**：能设计完整的四流对账方案（日终对账 + 异常队列 + 人工介入 + 监管追溯），能处理"处方已开但未支付"、"药品已配送但未签收"等边界 case
- **补足方向**：
  - 学习电商/支付行业的对账方案（如支付宝、美团的对账系统）
  - 学习处方流转平台的对账案例（如京东健康、阿里健康）
  - 学习分布式事务在四流合一中的应用（Saga、TCC）

## 四、电子处方流转平台架构（Day02）

### 差距4.1：处方流转政策与"双轨对接"的工程化深度不足
> Day2发现，延续第1周差距5.X（医保监管上报）

- **现状**：知道处方外流政策与双轨制概念，但"互联网医院自建处方流转平台 + 省级医保电子处方中心"双轨对接的工程实现（统一接收、统一流转、统一追溯、统一对账）深度不足
- **架构师水平**：能设计双轨对接架构，处理"自建平台处方 -> 上传省级电子处方中心 -> 患者在医保定点药店调取"的完整链路，能应对双轨下的对账与监管
- **补足方向**：
  - 精读国家医保局《关于推进电子处方流转服务的指导意见》全文
  - 调研 20+ 省份的省级电子处方中心接口规范（如广东、四川、浙江）
  - 学习京东健康、阿里健康与省级电子处方中心对接的公开案例

### 差距4.2：拆单策略与多仓库存的高并发实战不足
> Day2发现

- **现状**：知道需要拆单，但"多药店拆单 + 多仓调度 + 多配送方协同"的工程实现细节（拆单算法、库存满足度评估、配送聚合、退药拆分）经验不足
- **架构师水平**：能设计完整的拆单引擎（多策略组合 + 库存实时同步 + 拆单后聚合配送 + 退药拆分），能处理"5 种药拆 3 个子订单 + 1 个药品改方"等复杂 case；能设计支撑万级 QPS 的多仓库存系统（Redis 预扣 + DB 兜底 + 对账修复），处理批号效期管理、近效期预警、回库校验
- **补足方向**：
  - 学习电商行业的拆单引擎（京东、淘宝的拆单算法）
  - 学习药品零售行业的拆单案例（叮当快药、美团买药）
  - 学习 GSP《药品经营质量管理规范》对库存管理的要求
  - 学习批号效期管理在药品库存中的应用

### 差距4.3：药品配送三种模式的统一抽象与冷链工程实战不足
> Day2发现

- **现状**：知道三种配送模式，但"合作药店自配送 + 第三方配送 + 医院药房配送"的统一抽象、运单号回传、轨迹同步、签收照片/地理位置/温度监控的工程实战不足
- **架构师水平**：能设计统一的 Shipment 实体抽象三种模式，处理冷链药品的温度监控（5min 回传、超温告警、开箱温度、超 8℃ 拒收），应对配送失败/退药的逆向物流
- **补足方向**：
  - 学习主流配送方（顺丰、京东物流、达达）的医疗冷链能力
  - 学习药品 GSP 对冷链配送的要求
  - 复盘啄木鸟云健康合作药店的配送实现

## 六、处方流转合规与药事服务（Day02）
