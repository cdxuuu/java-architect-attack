## 一、医疗大数据在医疗信息化中的定位

### 1.1 核心一句话

医疗大数据平台是把"业务系统沉淀的多源异构数据"通过"汇聚 + 治理 + 标准化 + 隐私计算 + 服务化"转化为"可科研、可监管、可商业化、可训练 AI"的数据资产，本质是"数据驱动"而非"流程驱动"。

### 1.2 医疗大数据发展四阶段

| 阶段 | 时期 | 核心 | 代表 |
| --- | --- | --- | --- |
| 数据导出期 | 2010-2018 | HIS 导出 + Excel 分析 | 各医院自建数据导出工具 |
| CDR 落地期 | 2018-2022 | CDR + 数据治理 + 互联互通 | 医惠、嘉和美康、东软 CDR |
| 隐私计算期 | 2022-2024 | 多中心隐私计算 + 真实世界研究 | 蚂蚁链摩斯、FATE、翼方健众 |
| 大模型数据期 | 2024- | CDR + CDM + 隐私计算 + 大模型数据管线 | 头部三甲 + 大模型厂商 |

2025 年后趋势是"CDR + CDM + 隐私计算 + RWE + 大模型数据"一体化数据平台，五层各有工程挑战，但有统一的数据治理与合规框架。

### 1.3 医疗大数据的核心能力

- **临床调阅**：患者 360 视图，跨机构就诊、检验、影像、病历一键调阅
- **科研队列**：基于 OMOP CDM 标准化，跨中心队列筛选
- **真实世界研究**：观察性数据的因果推断，产出药品审评证据
- **监管上报**：医保控费、药监 RWE、公卫上报的统一数据源
- **AI 训练数据供给**：CDR 数据经脱敏、标注、对齐后供给大模型训练

### 1.4 监管与商业路径

- **法律基础**：《个人信息保护法》《数据安全法》《人遗条例》《病历管理规定》
- **数据资产化**：财政部 2024 年暂行规定，数据资源可计入资产
- **RWE 监管路径**：CDE 2020 年《真实世界证据支持药物研发与审评的指导原则》
- **数据交易所**：北京国际大数据交易所、上海数据交易所、深圳数据交易所

### 1.5 医疗大数据的工程约束

1. 数据驱动而非流程驱动
2. 多源异构数据治理是首要难题
3. OMOP CDM 标准化是跨中心协同的基础
4. 隐私计算是"数据可用不可见"的工程实现
5. 科研队列设计是临床医生与数据科学家的"翻译层"
6. 真实世界研究的因果推断是核心方法论
7. 大模型训练数据管线是新挑战
8. 合规与伦理是"硬约束"
9. 数据质量是科研可信度的基础
10. 数据资产化与价值变现是商业可持续的关键

## 二、医疗大数据平台架构

### 2.1 整体分层

```
业务应用层（科研队列/RWE/AI 训练数据/监管上报/患者 360/BI）
        ↓
数据服务层（REST/FHIR/GraphQL、SQL 即席、队列 API、订阅 API、统一鉴权）
        ↓
数据资产层（CDR + CDM + 术语映射 + EMPI）

数据治理层（质量评分 + 血缘追踪 + 资产目录 + Owner + 脱敏 + 变更审批 + 工单闭环）
        ↓
数据汇聚层（CDC + HL7/FHIR 消息 + 批量 ETL + 跨机构 FHIR API）

数据源层（HIS/EMR/LIS/PACS/RIS/互联网医院/体检/医联体/医保/公卫/IoT）

### 2.2 模块分类

- **数据核心**：CDR、CDM、术语映射、EMPI、数据资产目录

## 三、CDR 临床数据中心

### 3.1 多模态存储栈

| 数据类型 | 存储 | 选型理由 |
| --- | --- | --- |
| 结构化（就诊/医嘱/处方/检验/医保） | PostgreSQL + Citus | 强事务、复杂 SQL、JSONB、水平分片 |
| OLAP 分析 | ClickHouse | 列式、向量化、10亿行秒查 |
| 病历文本 | MongoDB + ES | 文档模型、全文检索 |
| 影像与病理 | MinIO + DICOMweb | S3 兼容、EC 纠删码、OHIF Viewer |
| 时序（监护/IoT） | InfluxDB / TimescaleDB | 高吞吐、时间窗口聚合 |
| 图（EMPI/术语/关联） | Neo4j | 多跳查询 P95 < 100ms |
| 数据湖 | Apache Iceberg | 时间旅行、ACID、Schema 演进 |

**为什么不用单一数据库**：医疗数据多模态特征（结构化 + 文本 + 影像 + 时序 + 图）决定"多专多能"存储栈是必然。Polyglot Persistence 是医疗大数据平台标配。

### 3.2 EMPI 多源 ID 映射

```
EMPI 中心：global_patient_id (UUID v7) + demographic + source_ids[] + confidence + audit_log

多源映射：HIS_ID / EMR_ID / LIS_ID / PACS_ID / 体检_ID / 互联网医院_ID / 医联体_ID / 医保_ID / 公卫_ID
        -> global_patient_id

三层匹配算法：
- L1 确定性匹配（身份证 hash 完全匹配，99%+）
- L2 概率匹配（Fellegi-Sunter 模型，70-99%）
- L3 机器学习匹配（XGBoost + 特征工程，70-95%）

错配修复：人工双复核 + 派生数据迁移 + 通知机构 + 串档风险画像
串档风险防范：异常合并/拆分告警 + 双人复核 + 审计日志
```

### 3.3 数据同步策略

| 同步方式 | 延迟 | 实现 | 场景 |
| --- | --- | --- | --- |
| 实时 CDC | < 1s | Debezium + Kafka | 业务库变更 |
| 准实时消息 | < 5s | HL7 v2 / FHIR Subscription | 业务事件 |
| 批量 ETL | T+1 | Spark + Airflow | 全量对账与回算 |
| 跨机构 FHIR API | < 1min | FHIR Bulk Data Access | 跨机构数据同步 |
| 影像异步上传 | < 5min | DICOM STOW-RS | 大文件归档 |

**CDC + 消息 + ETL 三层并用**：
- CDC 只能捕获库变更，业务事件需消息
- 消息只能流式增量，历史回算需 ETL
- ETL 批量无法满足实时调阅，需 CDC

## 四、数据治理体系

### 4.1 数据质量 6 维度评分

```
完整性 Completeness：字段非空率、必填完整率、关联记录完整率
准确性 Accuracy：值域合规率、参考范围合规率、业务规则合规率
一致性 Consistency：跨系统一致率、跨时间一致率、主数据一致率
时效性 Timeliness：上报延迟分布、同步延迟分布、数据新鲜度
合规性 Validity：术语编码合规率、数据元规范率、格式合规率
唯一性 Uniqueness：EMPI 重复率、就诊号重复率、检验号重复率

综合评分 = Σ(维度分 × 权重)
- 95+ 优秀、90-95 良好、80-90 合格、< 80 不合格
```

### 4.2 数据血缘追踪

```
血缘模型：Source -> Process -> Sink（含字段级血缘）
工具：Apache Atlas + OpenLineage
应用：
- 术语映射变更影响评估
- CDM 升级影响评估
- 数据质量根因分析
- 合规审计
```

### 4.3 数据资产目录

```
DataHub 资产元数据：
- 域、Owner、来源、SLA、Schema、质量评分、关联资产、标签
检索：全文、标签、Owner、质量分
应用：数据探索、数据申请、影响评估、合规审计
```

### 4.4 数据 Owner 机制

- 业务 Owner：科室主任，对数据业务含义负责
- 技术 Owner：信息科，对数据技术实现负责
- 数据 Steward：数据管家，对数据质量与合规负责
- 三方协同：周例会 + 月评审 + 季度回顾

### 4.5 数据治理闭环

```
前置校验（接入时）-> 实时监控 -> 告警工单 -> 根因分析 -> 修复治理 -> 验证闭环 -> 复盘改进 -> 知识沉淀
```

### 5.3 术语映射引擎

```
L1 精确匹配（SOURCE_TO_CONCEPT_MAP，命中率 80-90%）
   ↓ 未命中
L2 模糊匹配（同义词 + 编辑距离 + BERT 嵌入向量相似度，5-10%）
   ↓ 未命中
L3 人工审核（术语专家 + 临床医师，1-3%）
   ↓ 未命中
L4 新概念申请（OHDSI 社区或国家卫健委）

质量评估：精确匹配率、模糊匹配置信度分布、人工审核通过率、未映射率
版本管理：VOCABULARY_VERSION、mapping_version、cdm_version 三轨
```

### 5.4 CDM 版本升级

```
策略：滚动升级 + 双版本共存（6 个月）+ 影子对比 + 切流 + 回滚预案（12 个月）

字段兼容：
- 新加字段：向后兼容
- 删除字段：评估下游，必要时保留为 derived
- 类型变更：必须 ETL 转换 + 校验

概念兼容：
- 概念 ID 跨版本稳定
- 废弃概念保留 invalid_reason = 'D'
- 替代关系记入 CONCEPT_RELATIONSHIP
```

## 六、大模型训练数据管线

```
Step 1 数据抽取：从 CDR 按疾病/药品/检查/操作多维度筛选，多模态同步抽取
Step 2 去重：SimHash（文本）+ 像素哈希（影像）+ EMPI（患者），保留 70-80%
Step 3 清洗：脱敏（NER + 哈希）+ 低质量过滤 + 异常值过滤，保留 50-60%
Step 4 标注：人工 + LLM 辅助 + 多专家交叉，百万级样本
Step 5 对齐：SFT（指令微调）+ RLHF（医师偏好）+ DPO（直接偏好优化）
Step 6 合规授权：患者授权 + IRB 审批 + HGR 审批 + PIA + DSA + 数据授权范围
Step 7 版本管理：DVC + Git LFS + 数据集快照 + Datasheet + Model Card

管线编排：Airflow / DolphinScheduler
- 月度全量管线（新数据 + 重新标注 + 重新对齐）
- 周度增量管线（新数据 + 自动标注 + 人工抽检）
- 监控：每步量级、质量、耗时、成本
- 告警：失败/超时/数据异常
```

## 七、多中心隐私计算

### 7.1 三条技术路径对比

| 维度 | 联邦学习 FL | 安全多方计算 MPC | 可信执行环境 TEE |
| --- | --- | --- | --- |
| 核心原理 | 数据不动模型动 | 多方计算协议（SPDZ、Garbled Circuit） | 可信硬件（Intel SGX、ARM TrustZone） |
| 性能 | 高 | 低（10-100x 慢） | 高（接近明文） |
| 精度 | 接近明文（< 1-3% 损失） | 完全等价明文 | 完全等价明文 |
| 数据范围 | 横向/纵向 | 任意（复杂度爆炸） | 任意（enclave 有限） |
| 部署成本 | 中 | 高 | 中 |
| 典型应用 | 影像 AI、慢病预测 | 跨机构统计、联合定价 | 安全查询、模型推理 |
| 攻击面 | 梯度反演、成员推断 | 协议漏洞、侧信道 | 硬件漏洞（SGX Foreshadow） |
| 代表平台 | 蚂蚁链摩斯、FATE、NVIDIA FLARE | 华控清交、富数、Sharemind | Azure CC、Oasis Labs |

### 7.2 选型决策框架

```
是否训练深度模型？
  - 是 -> FL 为主（性能高，但需 DP 防梯度反演）
  - 否 -> 跨机构统计/匹配？
    - 是 -> MPC（精确等价明文）
    - 否 -> 模型推理/安全查询？
      - 是 -> TEE（性能接近明文）

是否需要跨中心同一患者识别？
  - 是 -> PSI（OPRF-PSI 性能最优）
  - 否 -> 直接 FL/MPC

是否涉及敏感数据出境？
  - 是 -> 数据本地化 + FL（数据不出域）+ TEE 模型推理
  - 否 -> 标准隐私计算平台
```

### 7.3 联邦学习的两个维度

**中心参数服务器 vs 去中心化 gossip**：
- 中心：O(N) 通信、SPOF、强一致、N < 100
- 去中心化：O(N²) 通信、无 SPOF、最终一致、N < 30

**横向 vs 纵向**：
- 横向：按样本分区（不同中心有不同患者），通信低，隐私强
- 纵向：按特征分区（不同中心有同一患者不同特征），需 PSI + 中间结果加密

## 八、科研队列设计

### 8.1 OHDSI CohortExpressions

```
{
  "entryEvent": "primaryCriteria（如 drug_exposure AND condition_occurrence_in_365d）",
  "inclusionRules": [
    "年龄 ≥ 18 岁",
    "首次用药前 365 天有 II 型糖尿病",
    "首次用药前 365 天无胰腺癌"
  ],
  "endStrategy": "drug_exposure_end_date + offset",
  "censoringCriteria": ["胰腺癌发生", "DEATH"],
  "cohortEras": {"eraGapDays": 30}
}
```

### 8.2 入组与排除标准

```
入组：暴露组（用药 + 诊断）、对照组（诊断 + 不用药）
排除：1 型糖尿病、胰腺癌病史、终末期肾病、妊娠期糖尿病、胰腺炎病史
```

### 8.3 PSM 协变量选择

```
人口学：年龄、性别、BMI、吸烟、饮酒、民族、教育、社会经济
合并症：Charlson 评分、高血压、冠心病、心衰、卒中、CKD、脂肪肝、肝硬化、胰腺炎
用药史：其他降糖药、降脂药、降压药、抗血小板、化疗药
检验值：HbA1c、空腹血糖、血脂、肝功能、胰腺功能、肿瘤标志物
影像特征：胰腺 CT/MRI、既往胰腺占位
医疗机构：医院级别、就诊频率、体检频率

匹配方法：最近邻 + 卡钳（PS 差 ≤ 0.2 SD）+ 无放回
匹配质量：SMD < 0.1（每协变量）+ PS 分布直方图 + 共变量平衡表
```

## 九、RWE 因果推断

### 9.1 因果推断方法对比

| 方法 | 原理 | 适用场景 | 局限 |
| --- | --- | --- | --- |
| PSM | 用 PS 模型匹配用药 vs 对照 | 治疗分配可观测 | 不能处理不可测混杂 |
| IPTW | 用 PS 倒数作权重构造虚拟随机分组 | 队列研究、生存分析 | 极端权重不稳定 |
| IV | 用"影响治疗不影响结局"的变量 | 不可测混杂存在 | 强工具变量难找 |
| DiD | 处理组前后变化 vs 对照组前后变化 | 政策评估 | 平行趋势假设 |
| 合成控制 | 加权对照组模拟反事实 | 单一处理单位 | 权重不稳定 |
| RDD | 利用治疗分配阈值 | 阈值明确的治疗 | 局部估计，外推不可靠 |

### 9.2 主分析 + 敏感性分析

```
主分析：PSM + Kaplan-Meier + Log-rank + Cox HR (95% CI)

敏感性分析：
- IPTW 估计 ATE，与 PSM 对比一致性
- E-value：未测混杂需多大才能消除效应（> 2 稳健）
- 阴性对照：与暴露无关结局（如骨折）应无效应
- 阳性对照：已知效应暴露（如吸烟 -> 肺癌）应有效应
```

### 9.3 不可测混杂处理

```
1. 工具变量 IV：找"影响治疗不影响结局"的变量（如医生处方偏好）
2. 阴性对照校正：阴性对照效应 = 不可测混杂代理，从主分析扣除
3. E-value 敏感性：未测混杂需多大才能消除效应
4. 多数据库三角验证：多库重复研究，混杂结构不同
5. 与 RCT 对照（如有可能）：用 RCT 验证 RWE 结论
```

## 十、跨中心 EMPI 一致性

### 10.1 PSI 隐私集合求交

```
目标：N 方在不暴露各自身份集合前提下，求出共同身份集合

主流协议：
- Hash-PSI：简单但易受字典攻击
- RSA-PSI：经典，性能中等
- BF-PSI：通信量小
- HE-PSI：精度高，性能低
- OPRF-PSI：主流，性能高（GB 级集合秒级）

工程实现：libPSI、Google's PSI、Microsoft's SEAL-PSI
```

### 10.2 时间线对齐（不交换身份）

```
基于匿名标识的联邦事件流：
1. PSI 后获取 global_patient_id 集合
2. 各方按 global_patient_id + event_date + event_type 上传匿名事件
3. TEE 内聚合每个 global_patient_id 的事件流
4. TEE 内计算间隔（如起始日期 - 诊断日期）
5. 结果加密返回研究方

关键工程点：
- 数据最小化（只传研究需要的字段）
- TEE 内聚合防推断
- 计算结果加 DP 噪声防小样本推断
- 全程审计日志

## 十一、数据安全与审计

### 11.1 多参与方权限模型

```
权限原则：
- 最小必要（只访问研究需要的字段）
- 患者身份不可见（PSI 后用匿名标识）
- 原始数据不出域（FL 只传梯度/参数）
- 全程审计（操作留痕，可追溯）

权限矩阵：数据方（自己数据 + 本地梯度） / 研究方（聚合结果） / CRO（聚合结果 + 梯度审计） / 监管方（聚合结果 + 全部审计） / 牵头方（本地数据 + 全部梯度 + 全部审计）
```

### 11.2 梯度反演攻击防护

```
攻击：从梯度反推训练数据（深层 NN 易受攻击）

防护：
1. DP-SGD / DP-FedAvg：梯度加噪 + 范数裁剪 + 隐私预算（ε=1-10）
2. 安全聚合：秘密共享下聚合，中心只见聚合梯度
3. 梯度压缩：稀疏化 + 量化 + 加噪
4. 大 batch size（> 32）提高反演难度
5. 审计与监控：异常梯度告警
```

### 11.3 研究链路合规留痕

```
研究设计：Protocol + IRB Approval + HGR Approval + DUA + 知情同意书模板
数据准备：本地准备日志 + 脱敏审计 + PIA + DSA
PSI 与训练：PSI 日志 + 联邦训练日志 + 安全聚合日志 + DP 预算消耗
结果发布：发布日志 + 小样本脱敏（k ≥ 5）+ 报告存档 + 论文审稿
监管申报：RWE 证据包（方案 + 数据 + 代码 + 结果 + 敏感性）+ 现场核查

审计日志要求：不可变（append-only + 哈希链）+ 保存 ≥ 15 年 + 第三方审计接口 + 双因子认证
```

### 13.1 出境合规要求

```
《数据出境安全评估办法》触发条件：

- 中国人类遗传资源出境需 HGR 审批
- 重要家系、特定地区严禁出境
- 国际合作需"国际合作许可"
```

```
数据在境内（隐私计算平台）

模型部署在境内，参数从境外更新
合规路径：境外算法入境评估（CFDA 算法备案）+ 数据不出境
```

## 十四、数据资产化风险治理

### 14.1 四类风险

```
合规风险：未取得授权、超出范围、未申报出境、未审批人遗
安全风险：泄露、篡改、丢失、供应链
商业风险：评估虚高、模式不可持续、客户违约、价格战
声誉风险：患者投诉、媒体曝光、监管处罚、行业抵制
```

### 14.2 应急响应流程

```
0-1h：发现与确认（监控告警、用户反馈）
1-4h：应急处置（隔离、止血、保留证据）
4-24h：影响评估（数据范围、敏感度、影响人数）
24-72h：监管上报（网信办、卫健委、公安）+ 用户通知
72h-7d：原因调查、修复、整改
7-30d：复盘改进、第三方安全评估、合规审计
30+d：长期监控、声誉修复
```

### 14.3 三道防线

```
第一道（业务部门）：流程内嵌合规校验 + 业务人员合规培训 + 业务系统日志
第二道（合规部门）：合规评估（PIA + DSA + HGR）+ 合同审查 + 季度审计
第三道（内部审计与监管）：年度审计 + 外部审计 + 监管对接
```
