## 一、CDSS 在医疗智能化中的定位

### 1.1 核心一句话

CDSS（Clinical Decision Support System）是把"医学知识工程化 + 实时推理"嵌入到临床决策节点的系统，本质是"知识驱动 + 数据驱动"的双轮--规则保合规、AI 提准确率、大模型提自然度。

### 1.2 CDSS 发展三阶段

| 阶段 | 时期 | 核心 | 代表 |
| --- | --- | --- | --- |
| 规则型 CDSS | 2010-2020 | 规则引擎 + 知识库 | 东软、卫宁、嘉和美康 |
| AI 辅诊型 CDSS | 2020-2024 | 医学影像 AI + 病历 NLP | 联影智能、推想、依图、深睿 |
| 大模型 CDSS | 2024- | 医学多模态大模型 | 商汤大医、京东京医千询、阿里医知仁 |

2025 年后的趋势是"三层架构"--规则保合规（无需证）、AI 提准确率（III 类证）、大模型提自然度（III 类证）。三层各有监管路径。

### 1.3 CDSS 的核心能力

- **事前提醒**（开立医嘱前）：基于患者画像推荐用药/检查
- **事中拦截**（开立医嘱时）：用药审查、过敏、相互作用、剂量、禁忌
- **事后审计**（开立医嘱后）：病历质控、不合理医嘱回顾、临床路径偏离

### 1.4 监管与商业路径

- **NMPA 分类**：非辅助决策类（CADe）按 I/II 类、辅助决策类（CADx）按 III 类医疗器械管理
- **CDSS 临床应用管理规范（2025）**：规则库覆盖率 ≥ 80%、自动审方率 ≥ 70%、规则可解释、误报漏报年度回顾
- **医保收费破冰（2024）**：肺结节 CT、糖尿病视网膜病变眼底、骨折 X 光三类 AI 辅诊可单独收费

### 1.5 CDSS 的工程约束

1. 知识驱动而非流程驱动
2. 可解释性是第一约束
3. 规则与模型双轮驱动
4. 警报疲劳是首要工程难题
5. 多模型多版本治理
6. 监管双轨（规则无需证、AI 需 III 类证）
7. 数据闭环驱动持续优化

## 二、CDSS 平台架构

### 2.1 七层架构

```
临床端 -> 接入层 -> 业务应用层 -> 知识推理层 -> 知识资产层 -> 数据接入层 -> 治理层
```

| 层 | 核心模块 | 分类 |
| --- | --- | --- |
| 业务应用层 | 用药审查、辅助诊断、风险预警、临床路径、病历质控、检验建议、医保审查、反馈闭环 | 业务应用 |
| 知识推理层 | 规则引擎、推理引擎、模型推理、知识图谱 | 知识核心 |
| 知识资产层 | 规则库、知识图谱、模型库、指南库 | 知识核心 |
| 数据接入层 | EMR 实时、检验 LIS、影像 PACS、医保接口 | 基础设施 |
| 治理层 | 版本治理、性能监控、审计合规、反馈闭环 | 基础设施 |

### 2.2 分类原则

- **知识核心**：CDSS 区别于普通业务系统的本质。规则引擎、推理引擎、知识图谱、模型库、规则库、指南库必须自研或深度定制。
- **业务应用**：消费"知识核心"的能力，对临床问题做封装。一个业务应用可同时调用规则 + 推理 + 模型 + 图谱。
- **基础设施**：与普通业务系统共性部分。医疗特化点：CDS Hooks 接入、DICOM 解析、监管证管理、等保三级审计。

## 三、规则引擎

### 3.1 规则引擎核心架构

四层结构：

```
规则编辑层（业务规则/决策表/决策树）
  -> 规则编译层（ANTLR 语法解析 + 字节码编译）
  -> 规则执行层（Rete 算法 + 工作内存 + 冲突解决）
  -> 规则输出层（Info/Warning/Hard Stop/Override）
```

### 3.2 规则建模三层抽象

| 抽象层 | 适用 | 优势 | 劣势 |
| --- | --- | --- | --- |
| 业务规则（IF-THEN） | 简单条件 | 易读、医学专家可编辑 | 复杂逻辑难表达 |
| 决策表 | 多条件组合 | 易审计、易穷举 | 仅适合条件-动作型 |
| 决策树 | 流程型 | 流程清晰、对接临床路径 | 不适合并行规则 |

**取舍原则**：优先决策表 > 流程型用决策树 > 简单条件用业务规则 > 复杂规则用决策树内嵌决策表。

### 3.3 规则版本化

- **版本号**：MAJOR.MINOR.PATCH
- **生命周期**：DRAFT -> REVIEW -> TEST -> GRAY -> PROD -> DEPRECATED -> ARCHIVED
- **灰度策略**：按科室 / 按医师 / 按比例 / 按场景
- **A/B 测试**：同规则两版本 50/50 分流，评估命中率/接受率/不良事件
- **回滚**：保留 3 个历史版本热备，触发条件：误报率/漏报率/接受率超阈值、不良事件

### 3.4 规则冲突解决

- **优先级（Salience）**：0-100，值大优先
- **LIFO**：同优先级后触发先输出
- **警报聚合**：同医嘱多规则聚合为复合提醒
- **警报抑制**：24h 内同患者同规则只提醒 1 次
- **白名单**：医师可申请，需主任审批 + 药学部备案

### 3.5 规则 DSL 选型

| DSL | 选型建议 |
| --- | --- |
| Arden Syntax | 不选（生态弱） |
| CDS Hooks | 必选（EMR 集成入口，仅触发与展示） |
| FHIR Clinical Reasoning | 选（知识互操作标准，不在生产引擎执行） |
| Drools | 选（底层执行引擎） |
| 自定义医学 DSL | 选（顶层编辑语言，编译到 Drools） |

**自定义 DSL 平衡三目标**：医学专家可读（when/then + 标准编码 + 临床概念）+ 程序员可维护（编译到 Drools DRL）+ 机器可执行（Rete 算法索引）。

### 3.6 药物相互作用规则示例

```
rule "WarfarinAmiodaroneInteraction"
when
    MedicationRequest(drug.code == "B01AA03" && status == "active")  // 华法林
    MedicationRequest(drug.code == "C01BD01" && status == "active"   // 胺碘酮
                      && occurred within 30 days)
then
    Card(
        level = HARD_STOP,
        summary = "华法林与胺碘酮联用风险极高",
        detail = "胺碘酮抑制 CYP2C9，使华法林代谢减慢，INR 显著升高，出血风险增加 2-3 倍",
        source = "Lexicomp 药物相互作用数据库 等级 D",
        suggestions = [...],
        overrideReasons = [...]
    )
end
```

## 四、CDS Hooks 集成

### 4.1 11 个 Hook 映射

| Hook | 触发节点 | 同步要求 |
| --- | --- | --- |
| patient-view | 患者档案打开 | < 500ms |
| medication-prescribe | 药品开立 | < 500ms |
| order-select | 医嘱选择 | < 500ms |
| order-sign | 医嘱签名 | < 500ms |
| order-review | 医嘱复核 | < 5s（异步） |
| discharge-plan-create | 出院计划 | < 5s（异步） |
| encounter-start | 就诊开始 | < 500ms |
| encounter-discharge | 就诊结束 | < 5s（异步） |
| appointment-book | 预约挂号 | < 500ms |
| diagnostic-report-create | 检验报告 | < 5s（异步） |
| imaging-study-create | 影像生成 | < 30s（异步） |

### 4.2 延迟分解（同步 < 500ms）

| 阶段 | 耗时 |
| --- | --- |
| EMR -> CDS 网关 | 20ms |
| CDS 网关 -> 规则引擎 | 30ms |
| 规则引擎执行（200-500 条规则） | 200ms |
| 知识图谱查询 | 100ms |
| 返回链路 | 50ms |
| 合计 | ~400ms |

### 4.3 EMR 不支持 CDS Hooks 的三层兜底

1. **集成引擎适配**（Mirth Connect / Rhapsody）：HL7 V2 转 CDS Hooks 调用
2. **数据库触发器 + 轮询**：触发器写消息队列，CDSS 消费后写"医嘱审查表"，EMR 轮询展示
3. **前端插件**：浏览器扩展监听 EMR 前端事件（仅 Web 版，合规风险）

## 五、警报疲劳治理

### 5.1 五位一体治理体系

```
1. 警报优先级（Priority）
   - Hard Stop / Warning / Info
   - 动态调整：根据 override rate 反向调整

2. 警报抑制（Suppression）
   - 时间抑制（24h 1 次）
   - 患者抑制（白名单）
   - 医师抑制（科室白名单）
   - 状态抑制（临终关怀）

3. 警报聚合（Aggregation）
   - 同医嘱聚合
   - 同患者聚合
   - 时间窗聚合（5 分钟内合并）

4. 医师反馈（Feedback）
   - Card 后附"是否有帮助"按钮
   - Override 必填结构化原因
   - 反馈进入迭代闭环

5. 规则迭代（Iteration）
   - 月度回顾：override > 80% 规则下线/优化
   - 季度回顾：漏报案例补强规则
   - 年度回顾：规则库整体评估
```

### 5.2 自动审方率 30% -> 70% 工程路径

| 阶段 | 周期 | 目标 | 关键工程 | 风险与缓解 |
| --- | --- | --- | --- | --- |
| 阶段 1 | 3 月 | 30% -> 50% | 低风险处方"自动审过"规则 | 误判高风险为低风险 -> 试点校准 |
| 阶段 2 | 3 月 | 50% -> 65% | 药师反馈闭环 + 规则迭代 | 新规则引入新误报 -> 灰度 2 周 |
| 阶段 3 | 6 月 | 65% -> 70%+ | AI 辅助审方（二筛） | AI 误判 -> AI 仅二筛，不做最终决策 |

**避免漏报保障**：自动审过基于"强证据 + 强规则"；灰度场景送人工；7 天内可追溯审计；不良事件触发回溯。

## 六、互联网医院 CDSS 辅助链路

### 6.1 8 个 CDSS 介入节点

```
1. 问诊前画像 -> 患者风险画像卡片
2. 上传材料分析 -> OCR 报告关键信息提取
3. 接诊前提醒 -> 鉴别诊断建议
4. 问诊中辅助 -> 实时 NLP + 动态鉴别诊断
5. 处方前置审查 -> 用药审查 + 医保限制
6. 药师审方辅助 -> 审方要点 + 类似案例
7. 处方后审计 -> 7 天/30 天回顾
8. 随访规则触发 -> 慢病随访路径
```

### 6.2 患者上下文一致性

- **预加载**：患者发起问诊时并行拉取
- **缓存**：体检报告 1h、EMR 30min、用药史 5min
- **一致性**：跨机构数据最终一致（5 分钟），关键决策强制刷新
- **授权**：首次问诊一次性授权，存证上链，token 24h，可撤销

### 6.3 体检异常项 -> 慢病随访路径

知识图谱建模：

```
节点：异常项 / 疾病 / 检查检验 / 随访路径 / 风险因素
边：提示 / 升级 / 推荐 / 监测 / 触发
```

**多异常项组合风险**：肺结节 + 吸烟（≥20 包年）+ 年龄 ≥ 50 -> 升级到"肺癌高风险筛查"路径，触发年度 LDCT。

### 6.4 处方前置审查七大维度

| 维度 | 规则来源 |
| --- | --- |
| 剂量 | 药品说明书、临床指南 |
| 频次 | 药品说明书 |
| 相互作用 | Lexicomp、Micromedex |
| 过敏 | 患者过敏史、交叉过敏 |
| 适应症 | 药品说明书、医保目录 |
| 医保限制 | 国家医保目录、省医保目录 |
| 特殊人群 | 妊娠期、哺乳期、儿童、老年、肾功能不全 |

### 6.5 医保限适应症审查

结构化编码：

```
DrugRestriction (
    drugCode = "曲妥珠单抗",
    restrictions = [
        IndicationRestriction(icd10=["C50.*"], biomarker="HER2 阳性"),
        IndicationRestriction(icd10=["C16.*"], biomarker="HER2 阳性")
    ]
)
```

规则执行：处方诊断不在医保限定 ICD-10 范围内 -> Hard Stop Card，要求自费或补充诊断。

### 6.6 AI 辅助审方模型

**输入特征**：处方 + 患者特征 + 病史 + 病历 NLP
**输出**：适宜性评分 + 不适宜原因（多标签）
**可解释性**：Attention + SHAP + 类似案例 + 循证依据

**与规则引擎三层协同**：

```
Layer 1：规则引擎（硬规则）-> 通过 / 阻断
Layer 2：AI 模型（灰度场景二筛）-> 适宜 / 不适宜
Layer 3：药师人工审（兜底）-> 最终决策
```

**III 类证判断**：AI 仅做"二筛"、不输出最终决策 -> 不需 III 类证；AI 直接输出"应改为某药某剂量" -> 需 III 类证。

### 6.7 互联网医院 CDSS 合规特殊性

| 维度 | 院内 CDSS | 互联网医院 CDSS |
| --- | --- | --- |
| 监管 | CDSS 规范 | + 互联网诊疗监管细则 |
| 数据 | 院内完整 | 跨机构，需授权 |
| 医师 | 全院 | 互联网医院执业备案 |
| 患者 | 全院 | 复诊为主 |
| 实时性 | 院内低延迟 | 互联网延迟波动 |
| 合规 | 等保三级 | + 互联网医院专项 |

**三者平衡**：合规 > 准确率 > 延迟。例外：紧急抢救延迟优先，事后补合规审查。

## 七、AI 辅诊可解释性三层目标

### 7.1 三层目标

| 层 | 受众 | 诉求 | 可解释性 |
| --- | --- | --- | --- |
| 医师信任 | 临床医师 | 为什么这么判断？能否反驳？ | 临床证据、可视化、鉴别诊断、类似病例 |
| 监管审计 | 卫健委/NMPA/伦理 | 决策依据？可追溯？合规？ | 决策日志、版本、数据溯源、性能 |
| 患者知情 | 患者及家属 | 为什么这么诊断？AI 起什么作用？ | 通俗语言、风险说明、替代方案 |

### 7.2 SHAP/Attention 的局限

- **SHAP**：医师看不懂"Shapley 值"；解释"模型怎么想"而非"医学上为什么"；缺乏临床证据。
- **Attention**：权重不等于因果解释；医师看到"关注左上角"也不知道为什么；缺乏临床语义。

**根本问题**：技术可解释性 ≠ 医学可解释性。SHAP/Attention 解决"模型如何工作"，医师/监管/患者关心"医学上为什么这么做"。

### 7.3 肺结节辅诊三层可解释性

```
医师信任层：
- 可视化标注（结节位置 + 边界 + 测量值）
- 临床特征（大小、形态、密度、边缘）
- 鉴别诊断对比（与既往 CT 对比）
- 类似病例（历史相似结节的病理结果）
- 风险分层（Fleischner / Lung-RADS）
- 推荐动作（复查周期、活检建议）

监管审计层：
- 决策日志（模型版本、输入哈希、输出、时间戳）
- 训练数据溯源（规模、来源、标注质量）
- 性能指标（敏感性、特异性、AUC、PPV、NPV，按子群体）
- 失效案例（已知失效场景）
- 监管证号

患者知情层：
- 通俗语言（"肺部小结节，约 8mm"）
- 风险说明（35% 恶性概率，65% 良性概率）
- AI 介入说明（AI 辅助识别，最终判断由医师负责）
- 替代方案（3 月复查、立即活检、PET-CT）
- 决策共享（医师 + 患者共同决定）
```

## 八、模型漂移与应对

### 8.1 漂移类型

| 类型 | 定义 | 本案例 |
| --- | --- | --- |
| 数据漂移 | 输入分布变，标签关系不变 | 多厂商眼底相机图像分布不同 |
| 概念漂移 | 标签与输入关系变 | 疾病定义/诊断标准改变 |
| 分布漂移 | 数据漂移 + 概念漂移统称 | - |

### 8.2 监控手段

- **输入分布**：图像特征统计（分辨率/亮度/对比度），KL 散度或 PSI > 0.2 告警
- **预测分布**：阳性率、置信度分布，PSI > 0.2 告警
- **性能监控**：滚动 AUC（30 天窗口），下降 > 5% 告警

### 8.3 应对手段

**预防（铺开前）**：
- 多中心数据训练（A/B/C/D 厂商）
- 数据增强（模拟不同分辨率）
- 域自适应（域不变特征）
- 灰度铺开（试点 -> 全院）

**应对（铺开后）**：
- 紧急回退（降级为"仅辅助提示"）
- 增量训练（新厂商数据）
- 域适配微调
- 多模型路由（按厂商路由到子模型）

### 8.4 全院铺开预检清单

多中心验证 / 性能公平性 / 边界 case / 部署环境 / 灰度计划 / 监控阈值 / 回退预案 / 医师培训 / 患者告知。

## 九、早期预警模型提前量工程化

### 9.1 AKI 预警"事后诸葛亮"根因

1. 预测窗口过短（仅 24h）
2. 特征时间窗过长（48h，错过新事件）
3. 阈值偏高（避免误报，敏感性不足）
4. 批量运行（错过实时事件）

### 9.2 提前量工程化

| 维度 | 设计 |
| --- | --- |
| 预测窗口 | 6h / 12h / 24h / 48h 多档 |
| 特征时间窗 | 短窗（6h）+ 中窗（24h）+ 长窗（72h）融合 |
| 模型架构 | 时序模型（LSTM/Transformer）+ 静态特征融合 |
| 阈值调优 | 6h 高敏感性（> 90%），48h 高特异性（> 90%） |
| 事件驱动 | 监听电子病历事件流，关键事件立即触发推理 |

### 9.3 避免"早预警低阳性率"

- **分级预警**：红/黄/蓝三级，医师只看红色
- **动态阈值**：按患者基础风险调整
- **预警聚合**：同患者 24h 多次预警合并
- **预警解释**：附"为什么"（如"肌酐 24h 上升 0.5 mg/dL"）

## 十、漏报事件责任与工程改进

### 10.1 责任归属

| 主体 | 责任 |
| --- | --- |
| 医院 | 医疗质量与患者安全总责；选择/部署/运维 CDSS |
| CDSS 厂商 | 产品性能与合规；不良事件报告与产品迭代 |
| 模型提供方 | 模型性能；训练数据真实性；失效场景告知 |
| 医师 | 最终医疗决策；不完全依赖 CDSS；人工兜底 |

**法律框架**：《医疗机构管理条例》（医院负总责）+《医疗器械监督管理条例》（厂商/模型方按医疗器械管理）+《医师法》（医师对决策负责）。实际案例：通常医院先行赔偿，再向厂商/模型方追偿。

### 10.2 RCA 方法（5-Why）

```
Why 1：为何患者死亡？-> 脓毒症恶化未及时干预
Why 2：为何未及时干预？-> 未早期识别脓毒症
Why 3：为何未早期识别？-> CDSS 未预警
Why 4：为何 CDSS 未预警？-> 模型对该患者特征预测概率低于阈值
Why 5：为何预测概率低？-> 训练数据中该亚组（老年合并糖尿病）样本不足
```

**RCA 输出**：直接原因（模型亚组性能不足）+ 间接原因（数据不平衡、亚组性能未评估）+ 系统原因（上线前未做亚组公平性评估）。

### 10.3 工程改进

1. **模型迭代**：漏报案例数据加入训练集，重新训练
2. **规则补强**：漏报场景用规则做"双轨预警"
3. **双轨预警**：模型（数据驱动）+ 规则（指南驱动）并行
4. **人工兜底**：ICU 关键场景保留人工预警
5. **漏报监控**：月度回顾"CDSS 未预警但实际发生的事件"

### 10.4 监管层面

- AI 辅诊属 III 类医疗器械，漏报可能构成"医疗器械不良事件"
- 医院 5 个工作日内向 NMPA 上报
- 若漏报属"模型已知失效场景"（说明书已告知），可能不构成不良事件
- 若属"模型未知失效场景"，构成不良事件需上报，NMPA 可能要求召回或修改说明书

## 十一、多模型统一治理

### 11.1 分散治理的问题

- 模型注册混乱（5 模型分散，无统一注册）
- 性能监控割裂（指标定义不一致）
- 迭代节奏不齐（流程不统一）
- 监管证管理分散（证到期未及时续证）

### 11.2 统一治理架构

```
Model Registry（注册中心）
  - 模型元数据 / 训练数据 / 性能指标 / 监管证
Model Lifecycle（生命周期）
  - Development -> Validation -> Deployment -> Monitoring -> Retirement
Model Monitoring（性能监控）
  - 实时性能 / 漂移检测 / 公平性 / 漏报误报
Model Audit（审计合规）
  - 决策日志 / 变更日志 / 监管上报 / 不良事件
```

### 11.3 模型生命周期

| 阶段 | 准出标准 |
| --- | --- |
| Development | 性能达标 + 公平性达标 |
| Validation | 外部多中心验证 + 公平性评估 |
| Deployment | 灰度期性能达标 + 培训完成 + 监管证在有效期内 |
| Monitoring | 性能/漂移/公平性/不良事件持续监控 |
| Retirement | 通知医师 + 替代方案 + 模型归档 + 审计日志 |

### 11.4 医疗 MLOps 特化点

| 维度 | 互联网 MLOps | 医疗 MLOps |
| --- | --- | --- |
| 数据合规 | 自由使用 | 患者隐私授权、伦理审批 |
| 模型验证 | 离线指标 | 外部多中心 + 公平性 + 伦理 |
| 监管证 | 无 | NMPA III 类证 |
| 部署节奏 | 周/日迭代 | 月小版本、年大版本 |
| 性能监控 | 业务指标 | 临床指标（AUC/敏感性/漏报率/不良事件） |
| 公平性 | 可选 | 必选（亚组性能监控） |
| 可解释性 | 可选 | 必选（三层目标） |
| 回退 | 业务损失 | 患者安全风险 |
| 审计 | 业务日志 | 不可变审计日志 + 监管上报 |

### 11.5 迭代节奏

- **月小版本**：训练数据增量更新，参数更新，内部验证 + 灰度，无需重新注册（轻微变更）
- **年大版本**：训练数据大更新或架构变化，外部多中心验证，NMPA 重新注册或变更注册（重大变更）
