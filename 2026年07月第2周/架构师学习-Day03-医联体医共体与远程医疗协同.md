## 背景

Day01 完成"互联网医院核心架构 + 在线问诊"，Day02 完成"电子处方流转 + 药事服务 + 互联网+医保结算"。前两天的视角是"单一互联网医院 + 它的合作药店/配送方/医保平台"。Day03 把视角从单一机构拉到"区域协同"——医联体/医共体内的多机构协同、区域卫生信息平台、远程医疗、双向转诊、影像云。

2026 年，医联体/医共体建设已经走过三轮：

- **2015-2017 年**：城市医联体、县域医共体、专科联盟、远程医疗协作网四种形态起步。原国家卫计委《关于推进医疗联合体建设和发展的指导意见》明确四种组织形态。
- **2018-2020 年**：分级诊疗制度落地，医联体考核指标发布。国家卫健委《医疗联合体综合绩效考核工作方案（试行）》将"双向转诊率、下转人次占比、远程医疗占比"纳入考核。
- **2021-2023 年**：县域医共体"医保总额付费、结余留用、合理超支分担"落地，医共体内部"人财物统一管理"开始试点。国家卫健委《"千县工程"县医院综合能力提升工作方案》要求 1000+ 县医院达到三级医院能力。
- **2024-2025 年**：紧密型医联体试点扩面，27 个省份 700+ 个县开展紧密型县域医共体试点；国家卫健委《关于推进紧密型县域医共体建设的通知》明确"统一法人、统一账户、统一人员管理、统一药品耗材、统一信息平台"五统一。

行业格局：

- **城市医疗集团**：以三甲医院为龙头，覆盖城区二级医院 + 社区卫生服务中心，如北京医院医疗集团、上海瑞金医疗集团。
- **县域医共体**：以县级医院为龙头，覆盖乡镇卫生院 + 村卫生室，如浙江德清医共体、安徽天长医共体。当前是政策最密集的赛道。
- **专科联盟**：以国家级/省级专科中心为龙头，跨区域联合，如国家儿童医学中心专科联盟、国家肿瘤区域医疗中心。
- **远程医疗协作网**：以三甲医院为核心，对基层/边远地区开展远程会诊、远程影像、远程心电，如解放军总医院远程医疗协作网。

啄木鸟云健康的体检客户中，部分县域医共体客户在尝试"乡镇卫生院体检异常项 -> 县级医院复查 -> 上转市级医院 -> 稳定后下转乡镇随访"的协同链路；部分城市医疗集团客户在尝试"集团内健康档案调阅 + 检查检验结果互认"。Day03 聚焦**区域卫生信息平台 + EMPI 主索引 + 双向转诊 + 远程会诊 + 影像云 + 医联体数据治理**，承接 Day01/Day02 的"单一机构视角"，向"区域协同视角"延伸。

Day03 与 Day01/Day02 的衔接点：

- **Day01 互联网医院执业挂靠医院**：Day03 延伸到"互联网医院 + 实体医院 + 社区中心"的医联体内多点执业。
- **Day01 在线问诊**：Day03 延伸到"基层首诊 + 上级远程会诊"的远程医疗协同。
- **Day02 处方流转**：Day03 延伸到"医联体内长处方流转 + 延伸处方在基层药店取药"。
- **第1周 Day03 HIS/EMR**：Day03 延伸到"医联体内 EMR 互认 + 检查检验结果互认"。
- **第1周 Day04 HL7/FHIR**：Day03 延伸到"区域信息平台 FHIR 互操作"。

Day03 的工程约束（与单一医院信息化的根本差异）：

1. **多机构数据归属与共享授权**：每条数据有"采集机构 + 归属机构 + 调阅机构"三方，授权链路跨机构。
2. **EMPI 主索引的跨机构匹配**：同一患者在医联体内多家机构有不同的就诊卡号，必须做主索引匹配，且匹配错误风险高（错配会串档）。
3. **网络异构与等保隔离**：医联体内机构可能跑在不同网络（医院内网、政务外网、互联网），跨网数据交换需走安全网关。
4. **业务流程跨机构编排**：双向转诊、远程会诊涉及多家机构的 HIS/EMR/PACS/RIS 协同，必须做流程编排与异常兜底。
5. **数据标准的多样性**：医联体内机构可能用不同 HIS 厂商（东软、卫宁、东华、创业慧康），数据标准不统一，必须做术语映射与数据治理。

#### 2. EMPI 主索引的匹配算法与全链路

**EMPI 的核心问题**：同一患者在医联体内多家机构有不同的就诊卡号，必须做主索引匹配，把"张三@县医院 cardId=A"和"张三@乡镇卫生院 cardId=B"识别为同一个患者。

**匹配算法的三层组合**：

**层 1：确定性匹配（Deterministic Matching）**

- 规则：身份证号 / 医保电子凭证 ID 完全一致 -> 直接匹配
- 优点：精度高（身份证号唯一）
- 缺点：覆盖率低（基层就诊时部分患者未带身份证、用临时卡）
- 工程实现：哈希索引（身份证号哈希 -> patientMasterId）

**层 2：概率匹配（Probabilistic Matching）**

- 算法：Fellegi-Sunter 概率匹配模型
- 输入：姓名、性别、出生日期、手机号、地址
- 输出：匹配概率 p，p > 0.95 视为同一个人，0.5 < p < 0.95 待人工确认，p < 0.5 视为不同
- 工程实现：
  - 姓名相似度：编辑距离 + 拼音相似度（处理"张三" vs "张叁"）
  - 出生日期相似度：完全一致 / 年月一致 / 年一致
  - 性别相似度：完全一致
  - 手机号相似度：完全一致 / 后 4 位一致
  - 地址相似度：行政区划 + 街道一致

**层 3：机器学习匹配（ML Matching）**

- 算法：XGBoost / DNN 二分类
- 输入：层 2 的相似度特征 + 其他特征（同机构就诊史、同家庭住址、同监护人）
- 输出：匹配概率
- 工程实现：
  - 训练数据：人工标注的"匹配/不匹配"样本
  - 在线推理：实时调用模型
  - 离线评估：每周评估模型准确率，避免漂移

**三层组合的工程实现**：

```java
public class EmpiMatcher {
    public MatchResult match(Patient patient) {
        // 层 1：确定性匹配（身份证号 / 医保电子凭证 ID）
        if (patient.getIdCard() != null) {
            PatientMaster master = empiDao.findByIdCardHash(patient.getIdCardHash());
            if (master != null) {
                return MatchResult.exact(master);
            }
        }
        if (patient.getInsuranceId() != null) {
            PatientMaster master = empiDao.findByInsuranceId(patient.getInsuranceId());
            if (master != null) {
                return MatchResult.exact(master);
            }
        }

        // 层 2：概率匹配
        List<PatientMaster> candidates = empiDao.findCandidates(
            patient.getName(), patient.getBirthDate(), patient.getPhone());
        MatchResult bestResult = MatchResult.noMatch();
        for (PatientMaster candidate : candidates) {
            double score = probabilisticMatcher.score(patient, candidate);
            if (score > 0.95) {
                return MatchResult.exact(candidate);
            }
            if (score > 0.5 && score > bestResult.getScore()) {
                bestResult = MatchResult.uncertain(candidate, score);
            }
        }

        // 层 3：ML 匹配（对层 2 待确认的样本）
        if (bestResult.isUncertain()) {
            double mlScore = mlMatcher.predict(patient, bestResult.getCandidate());
            if (mlScore > 0.9) {
                return MatchResult.mlMatch(bestResult.getCandidate(), mlScore);
            }
        }

        // 无匹配 -> 新建主索引
        if (bestResult.isNoMatch()) {
            return MatchResult.createNew();
        }

        // 待人工确认
        return bestResult;
    }
}
```

**EMPI 全链路**：

```
1. 建索引（首次就诊）
   - 患者在机构 A 首次就诊，机构 A 给出 cardId_A
   - 上报到 EMPI：身份证号 + 姓名 + 出生日期 + 性别 + 手机号 + cardId_A
   - EMPI 层 1/2/3 匹配：
     - 命中已有 master：关联 cardId_A 到该 master
     - 未命中：新建 master，关联 cardId_A

2. 查索引（跨机构调阅）
   - 患者在机构 B 就诊，机构 B 给出 cardId_B
   - 上报到 EMPI：身份证号 + cardId_B
   - EMPI 查询：返回 masterId
   - 机构 B 通过 masterId 调阅区域健康档案

3. 更新索引（患者信息变更）
   - 患者改手机号、改姓名（如结婚改名）
   - 触发 EMPI 更新：标记旧信息失效，新增新信息
   - 不删除旧信息（保留变更历史，便于追溯）

4. 错配修复（串档修复）
   - 发现错配：机构 A 的张三被错误关联到机构 B 张三的 master
   - 修复流程：
     a. 标记错配记录，发起错配工单
     b. 人工审核（双人工复核 + 主管药师/医师审批）
     c. 拆分 master：旧 master 拆为两个 master
     d. 数据迁移：把错误关联的就诊记录、健康档案迁移到新 master
     e. 通知所有曾调阅过该 master 的机构（数据已被迁移）
     f. 全程审计留痕
```

**串档风险的防范**：

| 防范措施 | 工程实现 |
|---------|---------|
| 人工确认 | 概率匹配 0.5-0.95 区间必须人工确认，不能自动关联 |
| 双向验证 | 关联时同时校验"机构 A 的张三 -> 机构 B 的张三"和"机构 B 的张三 -> 机构 A 的张三"双向匹配 |
| 可撤销 | 关联操作可撤销，撤销时同步撤销所有派生数据（如合并的健康档案） |
| 操作留痕 | 所有 EMPI 关联操作记录到不可变审计表，监管可追溯 |
| 风险告警 | 同一 master 短期内关联多张新卡 -> 告警人工审核 |
| 主索引画像 | 每个 master 维护"关联卡数、关联机构数、首次关联时间、最近关联时间"，异常画像告警 |

**EMPI 与各标识的关系**：

```
EMPI Master（区域主索引）
  ├─ 身份证号（国家级，唯一）
  ├─ 医保电子凭证 ID（国家级，唯一）
  ├─ 机构 A 就诊卡号 cardId_A（机构级）
  ├─ 机构 B 就诊卡号 cardId_B（机构级）
  ├─ 机构 C 就诊卡号 cardId_C（机构级）
  └─ 健康档案 ID（区域级，由 EMPI 派生）
```

- **身份证号 / 医保电子凭证 ID**：国家级唯一标识，是 EMPI 的"金标准"。
- **机构就诊卡号**：机构级标识，必须关联到 EMPI master 才能跨机构使用。
- **健康档案 ID**：区域级标识，由 EMPI master 派生，用于跨机构档案调阅。

#### 3. 健康档案调阅的授权链路

**健康档案的"六大档"**：

| 档案类型 | 数据来源 | 关键字段 |
|---------|---------|---------|
| 基本健康档案 | 公卫系统、基层医疗机构 | 个人信息、家族史、过敏史、生活习惯 |
| 慢病档案 | 公卫系统、慢病随访 | 高血压/糖尿病/慢阻肺等慢病记录、随访记录 |
| 就诊档案 | 各机构 HIS/EMR | 门诊病历、住院病历、诊断、手术 |
| 检查检验档案 | 各机构 LIS/PACS | 化验结果、影像报告、心电报告 |
| 用药档案 | 各机构 HIS、处方流转平台 | 处方记录、用药记录、不良反应 |
| 免疫接种档案 | 公卫系统、接种单位 | 疫苗接种记录、接种反应 |

**档案组织方式**：

- 按"患者 master ID + 档案类型 + 时间戳"组织。
- 每个档案文档（EhrDocument）包含多个区段（EhrSection）。
- 区段可单独授权（如精神科档案区段单独加密、单独授权）。

**跨机构调阅的授权链路**：

```
医师（机构 B）发起调阅
   ↓
1. 调阅人身份核验
   - 医师执业证书核验
   - 医师所属机构核验
   - 当前是否在岗（值班表核验）
   ↓
2. 调阅目的登记
   - 必须填写调阅目的（"接诊患者"、"会诊"、"随访"、"转诊"）
   - 调阅目的与患者当前就诊关联
   ↓
3. 患者授权核验
   - 患者"知情同意"状态（就诊时签署的知情同意书）
   - 患者授权范围（全部档案 / 部分档案 / 仅本次就诊相关）
   - 患者授权时限（一次性 / 7 天 / 30 天 / 长期）
   ↓
4. 调阅机构资质核验
   - 调阅机构是否在医联体内（医联体白名单）
   - 调阅机构与患者是否有"诊疗关系"（近 6 个月内有就诊记录）
   ↓
5. 调阅范围控制
   - 最小必要原则：根据调阅目的自动限制调阅范围
     - 接诊患者 -> 调阅近 6 个月就诊档案 + 检查检验档案 + 用药档案
     - 会诊 -> 调阅会诊相关专科档案 + 近 1 年就诊档案
     - 随访 -> 调阅慢病档案 + 近 3 个月用药档案
     - 转诊 -> 调阅全部档案
   ↓
6. 调阅操作执行
   - 实时调阅：从档案中心实时查询返回
   - 自动脱敏：非必要字段自动脱敏（如他人姓名、住址详细门牌号）
   - 限时会话：调阅会话 30 分钟有效，超时重新发起
   ↓
7. 调阅操作留痕
   - 调阅人、调阅时间、调阅目的、调阅范围、调阅 IP 全部记录
   - 不可变审计表 + 哈希链
   - 患者可在 APP 查询"谁调阅过我的档案"（透明化）
```

**最小必要原则的工程化**：

| 措施 | 工程实现 |
|------|---------|
| 按需调阅 | 不返回全量档案，按调阅目的返回必要字段 |
| 限时授权 | 调阅会话 30 分钟有效，超时重新发起 |
| 自动脱敏 | 非必要字段脱敏（如他人姓名 -> "张**"） |
| 精神科/传染科单独授权 | 敏感档案区段单独加密，单独授权 |
| 调阅频率限制 | 同一医师对同一患者 1 小时内调阅 ≤ 5 次 |
| 异常调阅告警 | 医师日均调阅 > 100 次 / 调阅非本人接诊患者 -> 告警 |

#### 4. 数据采集与交换

**数据采集方式对比**：

| 方式 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 前置机 | 在机构内部署前置机，定时 ETL 上报 | 机构改动小、安全可控 | 实时性差（T+1）、前置机维护成本 | 历史数据迁移、批量数据上报 |
| ETL | 数据中台定时从机构库 ETL 抽取 | 标准化、可批量 | 实时性差、对机构库压力大 | 历史数据汇聚 |
| API 实时上报 | 机构业务系统通过 API 实时上报 | 实时性高 | 机构业务系统改造大 | 关键业务事件（就诊、处方、检验） |
| 消息队列 | 机构业务系统发 MQ，数据中台订阅 | 实时性高、解耦 | 机构需具备 MQ 能力 | 高频实时数据（检验结果、影像报告） |

**推荐方案：混合模式**：

- **关键业务事件**（就诊、处方、检验申请、检验结果、影像报告、转诊、会诊）：API 实时上报 + MQ 兜底
- **历史数据迁移**：前置机 + ETL 批量
- **日常增量同步**：API 实时上报 + 准实时对账（每 5 分钟比对源系统与中台数据）

**数据交换标准选型**：

| 标准 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| HL7 V2 | HIS/LIS/RIS 间消息交互 | 国内主流、厂商支持好 | 字段松散、扩展性差 |
| HL7 V3/CDA | 临床文档交换 | 结构化、语义清晰 | 复杂、推广慢 |
| FHIR | 现代 RESTful API | 现代、易扩展、国际主流 | 国内推广中 |
| WS/T 445 | 国家卫健委数据元标准 | 国家强制 | 偏数据元，无 API 规范 |

**推荐方案**：

- **机构内部**（HIS/LIS/RIS 间）：HL7 V2（兼容现有厂商）
- **机构 -> 区域平台**：FHIR R4（新建接口走 FHIR，存量接口逐步迁移）
- **数据上报卫健委**：WS/T 445（国家强制）
- **CDA 文档**：出院小结、门诊病历等"文档型"数据走 CDA

**数据治理的三个层面**：

1. **术语映射**（Term Mapping）：
   - 厂商私有码 -> 标准码（如 ICD-10、药品本位码、LOINC）
   - 例：机构 A 的"高血压"诊断码 "I10-A" -> ICD-10 "I10"
   - 维护术语映射表，定期更新（ICD 每年更新）

2. **单位统一**：
   - 不同机构单位不同（如血糖 mmol/L vs mg/dL）
   - 在数据采集时统一转换为标准单位

3. **值域标准化**：
   - 性别：男/女/未知（GB/T 2261.1）
   - 婚姻状况：未婚/已婚/离异/丧偶（GB/T 2261.2）
   - 民族：56 民族标准码（GB/T 3304）

**数据质量评分**：


#### 5. 紧密型医共体"五统一"的信息化工程映射

**五统一的政策要求**：

| 统一 | 政策含义 | 信息化工程映射 |
|------|---------|--------------|
| 统一法人 | 医共体内机构法人统一 | 机构主数据统一标识（orgId 跨机构一致） |
| 统一账户 | 医共体内财务账户统一 | 医保结算统一账户、结算资金统一归集 |
| 统一人员管理 | 医共体内人员统一调配 | 医师/护士执业注册跨机构共享、多点执业工程化 |
| 统一药品耗材 | 医共体内药品耗材统一采购、统一配送 | 药品目录统一、库存共享、处方流转延伸至基层 |
| 统一信息平台 | 医共体内信息系统统一 | **关键问题**：是"一套 HIS"还是"多套 HIS + 统一数据中台" |

**"统一信息平台"的两种路径**：

**路径 A：一套 HIS（彻底统一）**

- 所有机构用同一套云 HIS（县医院 + 乡镇卫生院 + 村卫生室）
- 优点：数据天然统一、协同零障碍、运维成本低
- 缺点：
  - 县医院 HIS 复杂度高，乡镇/村用不上（"杀鸡用牛刀"）
  - 改造成本高（替换原有 HIS，县医院业务停摆风险大）
  - 厂商锁定（一旦选定厂商，难以更换）
  - 县医院临床专科需求 vs 乡镇公卫需求的差异难以平衡

**路径 B：多套 HIS + 统一数据中台（推荐）**

- 县医院用三甲 HIS（东软/卫宁/东华），乡镇/村用轻量级云 HIS，村卫生室用公卫系统
- 各机构 HIS 通过数据中台统一：数据采集 -> 标准化 -> 汇聚到健康档案中心
- 协同业务（双向转诊、远程会诊、影像云、处方流转）走统一的协同业务层
- 优点：
  - 各机构 HIS 选型匹配业务复杂度
  - 改造成本低（不动 HIS，新建数据中台 + 协同业务层）
  - 厂商灵活（数据中台是自有的，可对接任意厂商 HIS）
- 缺点：
  - 数据治理复杂（多源异构数据标准化）
  - 协同业务需做接口适配
  - 实时性受限（依赖数据上报时效）

**推荐选型：路径 B（多套 HIS + 统一数据中台）**，理由：

1. **改造成本**：紧密型医共体考核紧迫，路径 B 改造成本低、上线快。
2. **业务匹配**：县医院与乡镇/村的业务差异大，一套 HIS 难以匹配。
3. **演进路径**：路径 B 可向路径 A 平滑演进（数据中台先建，后续逐步替换机构 HIS）。
4. **厂商灵活**：自有数据中台可对接任意厂商，避免厂商锁定。

**路径 B 的工程约束**：

- 数据中台必须自研或自主可控（不能被单一厂商绑定）
- 数据标准必须统一（FHIR + WS/T 445）
- 协同业务层必须支持多机构多角色（机构 A 医师 + 机构 B 医师 + 患者）
- 接口适配层必须解耦（机构 HIS 升级不影响协同业务）

### 与架构师水平的差距及补足方向

- **现状**：知道 EMPI 概念和确定性匹配，但"概率匹配（Fellegi-Sunter）+ ML 匹配"的工程实战、错配修复的工程化、串档风险防范经验不足。

## 题目二（场景设计题）：双向转诊与远程会诊协同

### 作答区

#### 1. 双向转诊的完整业务流程

**上转 vs 下转的差异**：

| 维度 | 上转（基层 -> 上级） | 下转（上级 -> 基层） |
|------|---------------------|---------------------|
| 触发方 | 基层医师评估后发起 | 上级医师评估稳定后发起 |
| 转诊原因 | 基层能力不足、需进一步检查/手术/住院 | 稳定后回基层随访、康复、慢病管理 |
| 转诊类型 | 急诊上转、择期上转、检查上转 | 康复下转、慢病下转、随访下转 |
| 绿色通道 | 有（优先就诊/检查/住院） | 一般无 |
| 数据流向 | 基层 -> 上级（病历、检查检验） | 上级 -> 基层（出院小结、用药、随访计划） |
| 接收方义务 | 上级必须接诊（医联体协议） | 基层必须接诊（医联体协议） |
| 转诊考核 | 上转率（基层转出比例） | 下转率（上级转下比例，分级诊疗核心指标） |

#### 2. 双向转诊的"绿色通道"

**绿色通道的工程化**：

```
基层医师发起上转（急诊/择期）
   ↓
转诊单进入"绿色通道"队列
   ↓
1. 号源预留
   - 上级医院为医联体基层预留号源（如心内科每天预留 10 个号给基层上转）
   - 预留号源在 HIS 中标记"医联体上转专用"，普通挂号不可占用
   ↓
2. 检查 slot 预留
   - 上级医院为上转患者预留检查 slot（如 CT 每天 14:00-16:00 留 5 个 slot）
   - 上转患者优先安排检查
   ↓
3. 床位预留
   - 上级医院为急诊上转预留床位（如心内科 CCU 每天留 2 张床）
   - 急诊上转 2 小时内安排住院
   ↓
4. 接单 SLA
   - 急诊上转：上级医院 15 分钟内接单
   - 择期上转：上级医院 2 小时内接单
   - 检查上转：上级医院 4 小时内接单
   ↓
5. 超时兜底
   - 急诊上转 15 分钟未接单 -> 自动升级到科主任
   - 择期上转 2 小时未接单 -> 自动升级到医务科
   - 升级后仍未接单 -> 启动医联体协议兜底（转到其他医联体上级医院）
```

**号源/slot/床位预留的工程实现**：

```sql
-- 上级医院号源表
CREATE TABLE outpatient_quota (
    quota_id BIGINT PRIMARY KEY,
    org_id VARCHAR(32),
    dept_code VARCHAR(32),
    doctor_id VARCHAR(32),
    clinic_date DATE,
    clinic_period ENUM('MORNING', 'AFTERNOON'),
    total_quota INT,                    -- 总号源
    general_quota INT,                  -- 普通号源
    referral_reserved_quota INT,        -- 转诊预留号源
    referral_used_quota INT DEFAULT 0,  -- 转诊已用号源
    INDEX idx_org_dept_date (org_id, dept_code, clinic_date)
);

-- 转诊接单时扣减预留号源
UPDATE outpatient_quota
SET referral_used_quota = referral_used_quota + 1
WHERE org_id = #{toOrgId}
  AND dept_code = #{deptCode}
  AND clinic_date = #{clinicDate}
  AND clinic_period = #{clinicPeriod}
  AND referral_used_quota < referral_reserved_quota;
-- affected rows = 0 表示预留号源已用完
```

**绿色通道的考核**：

- 接单时长（急诊 15 分钟 / 择期 2 小时）
- 接单率（应 ≥ 95%）
- 拒收率（应 ≤ 5%，且必须有合理原因）
- 上转患者实际就诊率（应 ≥ 90%，未就诊需追溯原因）

#### 3. 远程会诊的几种模式

| 模式 | 数据类型 | 数据量 | 实时性 | 架构重点 |
|------|---------|--------|--------|---------|
| 远程临床会诊 | 音视频 + 病历 | 中（音视频流） | 实时 | WebRTC + SFU + 录制存档 |
| 远程影像会诊 | DICOM 影像 | 大（GB 级） | 准实时 | DICOM 云存储 + 云 PACS |
| 远程心电会诊 | 心电数据 | 小（KB-MB） | 准实时 | 心电数据上传 + 报告回传 |
| 远程病理会诊 | 病理切片数字化 | 大（GB 级） | 准实时 | 全切片扫描（WSI）+ 云存储 |

**远程临床会诊的音视频协同**：

```
申请机构医师发起会诊
   ↓
1. 会诊预约
   - 选择会诊专家（上级医院专家）
   - 预约会诊时间
   - 上传会诊资料（病历、检查检验、影像）
   ↓
2. 会诊资料预审
   - 专家提前 1 天预审资料
   - 专家可发起"会诊补充资料"请求
   ↓
3. 会诊开始（音视频协同）
   - WebRTC 建立 P2P 连接（申请医师 + 专家 + 患者）
   - SFU 服务器中转（多端会议）
   - 屏幕共享（影像调阅、病历调阅）
   - 录制存档（音视频 + 屏幕共享）
   ↓
4. 会诊讨论
   - 申请医师汇报病史
   - 专家查阅影像、病历
   - 双方讨论诊断与治疗方案
   ↓
5. 会诊报告
   - 专家撰写会诊报告（诊断意见、治疗建议）
   - 专家电子签章 + 会诊机构签章
   - 报告回传申请机构
   ↓
6. 会诊报告归档
   - 报告归档到患者健康档案
   - 会诊录像归档（保留 ≥ 15 年）
```

**WebRTC + SFU 的工程实现**：

| 组件 | 选型 | 说明 |
|------|------|------|
| WebRTC | 原生 WebRTC | 浏览器 + 移动端通用 |
| SFU 服务器 | Mediasoup / Jitsi Videobridge | 多端会议，转发音视频流 |
| STUN/TURN | coturn | NAT 穿透 |
| 录制 | Mediasoup RecordingObserver | 录制音视频流到文件 |
| 屏幕共享 | WebRTC getDisplayMedia | 共享桌面 / 应用窗口 |
| 信令 | WebSocket | 会话建立、媒体协商 |

**录制存档的合规要求**：

- 录制文件 SHA256 上链（防篡改）
- 录制密钥管理（KMS，国密 SM4 加密）
- 录制文件存储 ≥ 15 年
- 录制文件访问需"调阅人 + 调阅目的"双因子核验
- 患者可在 APP 查询"会诊录像访问记录"（透明化）

**远程影像会诊的架构**：

```
申请机构医师发起影像会诊
   ↓
1. 申请机构上传 DICOM 影像到影像云
   - 通过 DICOM Store 接口（DIMSE C-STORE）
   - 影像云返回 studyId
   ↓
2. 申请单关联 studyId
   - 会诊申请单关联 studyId、申请机构、申请医师
   ↓
3. 会诊专家调阅影像
   - 专家通过云 PACS 调阅影像
   - 云 PACS 提供测量、标注、重建工具
   ↓
4. 专家撰写影像会诊报告
   - 报告含：影像所见、诊断意见、建议
   - 专家签章
   ↓
5. 报告回传申请机构
   - 报告回传到申请机构的 RIS/HIS
   - 报告归档到患者健康档案
```

**远程心电会诊的架构**：

```
1. 基层医疗机构采集心电数据（12 导联心电图机）
   ↓
2. 心电数据上传到心电云平台
   - 标准格式：SCP-ECG / DICOM-ECG / HL7 aECG
   - 数据量小（KB-MB），上传快
   ↓
3. 心电云平台通知会诊专家
   ↓
4. 专家在线判读
   - 心电云平台提供波形显示、测量、对比工具
   ↓
5. 专家出具心电报告
   - 报告回传申请机构
   - 报告归档
```

**远程病理会诊的架构**：

```
1. 申请机构制备病理切片
   ↓
2. 全切片扫描（Whole Slide Imaging, WSI）
   - 玻片扫描仪（如 3DHistech、Hamamatsu）扫描成数字切片
   - 单切片数据量：GB 级
   ↓
3. 数字切片上传到病理云平台
   - 分块上传、断点续传
   - 存储：对象存储（如 OSS、S3）
   ↓
4. 专家在线判读
   - 病理云平台提供放大、缩小、标注、对比工具
   ↓
5. 专家出具病理报告
   - 报告回传申请机构
```

#### 4. 远程会诊的报告与责任归属

**报告的法律定位**：

| 报告类型 | 法律定位 | 法律依据 |
|---------|---------|---------|
| 远程临床会诊报告 | 咨询意见 | 《医师法》《远程医疗服务管理规范》 |
| 远程影像会诊报告 | 诊断报告（独立） | 《放射诊疗管理规定》 |
| 远程心电会诊报告 | 诊断报告（独立） | 同上 |
| 远程病理会诊报告 | 诊断报告（独立） | 《病理诊断中心基本标准（试行）》 |

**关键区别**：

- **远程临床会诊**：会诊专家提供"咨询意见"，**不替代**申请医师的诊疗决定。申请医师综合会诊意见后做出最终诊疗决定。
- **远程影像/心电/病理会诊**：会诊专家出具"独立诊断报告"，**具有独立法律效力**。专家对报告内容负责，申请机构可直接采信。

**报告签章的工程实现**：

```
会诊报告生成
   ↓
1. 会诊专家电子签章
   - 专家 CA 证书 + 国密 SM2 签名
   - 时间戳服务（防抵赖）
   ↓
2. 会诊机构签章
   - 会诊机构（上级医院）CA 证书 + 机构签章
   - 证明专家执业关系
   ↓
3. 报告归档
   - 签章后的报告归档到患者健康档案
   - 报告哈希上链（防篡改）
   ↓
4. 报告回传申请机构
   - 报告回传到申请机构 HIS/RIS
   - 申请机构可调阅、引用
```

**会诊报告的采纳**：

- **远程临床会诊**：申请医师可采纳 / 部分采纳 / 不采纳，但**必须将会诊意见记录在病历中**。
- **远程影像/心电/病理会诊**：申请医师可直接采信报告，作为诊疗依据。

**责任划分**：

| 场景 | 责任划分 |
|------|---------|
| 临床会诊意见被采纳，诊疗正确 | 申请医师主责，会诊专家连带 |
| 临床会诊意见被采纳，诊疗错误 | 申请医师主责，会诊专家连带（按过错比例） |
| 临床会诊意见未被采纳 | 申请医师独立承担 |
| 影像/心电/病理报告错误 | 会诊专家主责（报告错误） |
| 影像/心电/病理报告正确，临床应用错误 | 申请医师主责 |

#### 5. 异常处理的 SLA 与兜底机制

**异常场景与处理**：

| 异常场景 | 处理流程 | SLA |
|---------|---------|-----|
| 转诊申请失败（接收方系统故障） | 自动重试 3 次 -> 人工介入 -> 通知基层医师改投其他机构 | 30 分钟内解决 |
| 转诊接收方拒收 | 必须填写拒收原因 -> 通知基层医师 -> 基层医师改投或继续观察 | 即时 |
| 会诊专家缺席（临时请假） | 提前 4 小时通知申请机构 -> 改期或更换专家 -> 通知患者 | 4 小时前通知 |
| 转诊途中患者病情变化 | 基层医师评估 -> 启动急诊上转（如已在上转流程中，升级优先级） -> 通知上级医院急诊科 | 即时 |
| 转诊后患者未实际就诊 | 上级医院 24 小时内反馈"未就诊" -> 通知基层医师 -> 基层医师联系患者追溯原因 | 24 小时 |
| 会诊报告超时未出 | 24 小时未出报告 -> 升级到科主任 -> 48 小时未出 -> 升级到医务科 | 24/48 小时 |
| 转诊数据同步失败 | 自动重试 -> 人工补传 -> 数据对账兜底 | 1 小时内解决 |
| 远程会诊音视频故障 | 切换备用 SFU -> 切换电话会诊 -> 改期 | 15 分钟内决策 |

**异常兜底机制**：

1. **重试机制**：所有跨机构调用必须有重试机制（指数退避 + 最大重试次数）。
2. **降级机制**：远程会诊音视频故障 -> 降级为电话会诊 -> 降级为图文会诊。
3. **兜底机制**：转诊接收方故障 -> 转到医联体内其他机构 -> 仍然故障 -> 转到医联体外机构（需患者同意）。
4. **人工介入**：异常队列监控 + 人工介入 SLA（30 分钟内介入）。
5. **数据对账**：转诊数据同步失败 -> T+1 对账 -> 补传失败数据。
6. **告警机制**：异常率超阈值告警（如转诊失败率 > 5% -> 告警医联体管理办）。

#### 2. DICOM 影像的高效传输

**DICOM 影像的特点**：

- 单次检查数据量大：CT 检查 100-500 MB，MRI 检查 200-1000 MB，乳腺钼靶 1-2 GB
- 文件数多：单次检查 100-1000 个 DICOM 文件（每个文件 1 切片）
- 跨网络传输：医联体内机构 -> 影像云，跨网络环境

**高效传输的工程实现**：

1. **分块上传**：大文件分块（如 10 MB/块），并行上传
2. **断点续传**：上传中断后从断点继续
3. **并行传输**：多线程并行上传（如 4-8 线程）
4. **压缩算法**：
   - 无损压缩：JPEG 2000 Lossless（压缩比 2:1，医疗强制无损）
   - 有损压缩：JPEG Lossy（压缩比 10:1，用于缩略图/预览）
5. **DICOM 原生协议**：DIMSE C-STORE（医院 PACS 标准协议）
6. **HTTP/WebSocket 传输**：跨网络环境（如互联网医院场景）

**分块上传的工程实现**：

```java
public class DicomUploader {
    public String upload(DicomStudy study) {
        String uploadId = initMultipartUpload(study);

        // 分块并行上传
        List<DicomInstance> instances = study.getInstances();
        instances.parallelStream().forEach(instance -> {
            byte[] data = instance.getData();
            int chunkSize = 10 * 1024 * 1024;  // 10 MB
            int chunkCount = (data.length + chunkSize - 1) / chunkSize;

            for (int i = 0; i < chunkCount; i++) {
                int start = i * chunkSize;
                int end = Math.min(start + chunkSize, data.length);
                byte[] chunk = Arrays.copyOfRange(data, start, end);

                uploadChunkWithRetry(uploadId, i, chunk);
            }
        });

        completeMultipartUpload(uploadId);
        return study.getStudyId();
    }

    private void uploadChunkWithRetry(String uploadId, int partNum, byte[] chunk) {
        int maxRetry = 3;
        for (int i = 0; i < maxRetry; i++) {
            try {
                uploadChunk(uploadId, partNum, chunk);
                return;
            } catch (Exception e) {
                if (i == maxRetry - 1) throw e;
                sleep((long) Math.pow(2, i) * 1000);  // 指数退避
            }
        }
    }
}
```

#### 3. 医联体数据治理的"数据归属与共享授权"

**三方数据归属关系**：

| 角色 | 含义 | 数据权利 |
|------|------|---------|
| 采集机构 | 数据的原始采集机构 | 数据所有权（在 EMR/HIS 中保存原始数据） |
| 归属机构 | 患者归属的医联体机构（通常是首诊机构或签约机构） | 数据归属权（决定数据共享范围） |
| 调阅机构 | 申请调阅数据的机构 | 数据使用权（在授权范围内使用） |

**典型场景**：

- 患者在县医院首诊 -> 县医院是采集机构 + 归属机构
- 患者上转市级医院 -> 市级医院是采集机构（新数据）+ 调阅机构（调阅县医院数据）
- 患者下转乡镇卫生院随访 -> 乡镇卫生院是采集机构（随访数据）+ 调阅机构（调阅县医院+市级医院数据）

**患者授权的工程化**：

| 授权要素 | 工程实现 |
|---------|---------|
| 知情同意 | 患者就诊时签署"知情同意书"（电子签章），同意数据在医联体内共享 |
| 范围授权 | 患者可限定共享范围（全部档案 / 仅基本档案 / 仅慢病档案 / 排除精神科等敏感档案） |
| 时限授权 | 患者可限定共享时限（一次性 / 7 天 / 30 天 / 1 年 / 长期） |
| 撤回授权 | 患者可随时撤回授权，撤回后调阅机构不可再调阅（已调阅数据需评估是否删除） |
| 授权记录 | 所有授权操作记录到不可变审计表，患者可在 APP 查询"授权历史" |

**授权服务的工程实现**：

```java
public class ConsentService {
    public Consent grantConsent(Patient patient, ConsentRequest request) {
        Consent consent = new Consent();
        consent.setPatientId(patient.getMasterId());
        consent.setScope(request.getScope());  // 全部档案 / 部分 / 排除
        consent.setDuration(request.getDuration());  // 一次性 / 7d / 30d / 1y / 长期
        consent.setPermittedOrgs(request.getPermittedOrgs());  // 授权机构列表
        consent.setPermittedPurposes(request.getPermittedPurposes());  // 授权目的
        consent.setGrantedAt(Instant.now());
        consent.setExpiresAt(calculateExpiry(request.getDuration()));

        consentDao.save(consent);
        auditLog.record("CONSENT_GRANTED", consent);
        return consent;
    }

    public boolean checkAccess(AccessRequest request) {
        Consent consent = consentDao.findActiveConsent(
            request.getPatientId(),
            request.getAccessOrgId()
        );

        if (consent == null || consent.isExpired()) {
            return false;
        }

        if (!consent.getPermittedOrgs().contains(request.getAccessOrgId())) {
            return false;
        }

        if (!consent.getPermittedPurposes().contains(request.getPurpose())) {
            return false;
        }

        if (!consent.getScope().includes(request.getRequestedScope())) {
            return false;
        }

        return true;
    }

    public void revokeConsent(Long consentId) {
        Consent consent = consentDao.findById(consentId);
        consent.setRevokedAt(Instant.now());
        consentDao.save(consent);
        auditLog.record("CONSENT_REVOKED", consent);

        // 通知所有曾调阅过该授权的机构
        notifyRevokedAccess(consent);
    }
}
```

**数据共享的三种模式**：

| 模式 | 含义 | 适用场景 |
|------|------|---------|
| 原始数据共享 | 共享完整原始数据 | 医联体内调阅（影像、检查检验、病历） |
| 摘要数据共享 | 共享数据摘要（如诊断、关键指标） | 跨医联体调阅（如区域健康档案摘要） |
| 脱敏数据共享 | 共享脱敏后的数据（去除身份证号、姓名等） | 科研、统计、监管上报 |

**适用场景**：

- **原始数据共享**：医联体内（信任关系高、临床诊疗必需）
- **摘要数据共享**：跨医联体（信任关系一般、需知情）
- **脱敏数据共享**：科研、统计、监管（无需知情、隐私保护）

**ICD-10 -> ICD-11 迁移期的双标准共存**：

```
迁移期（2025-2028 年）：
- 机构 HIS 同时支持 ICD-10 和 ICD-11
- 诊断编码同时上报 ICD-10 和 ICD-11（双码上报）
- 术语映射表维护 ICD-10 <-> ICD-11 双向映射

迁移后（2028+）：
- ICD-10 退场，仅 ICD-11
- 历史数据保留 ICD-10 编码，可查询映射到 ICD-11
```

**术语映射质量评分**：

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| 覆盖率 | 私有码被映射到标准码的比例 | 已映射私有码数 / 总私有码数 |
| 准确率 | 映射正确的比例 | 人工抽样正确数 / 抽样总数 |
| 冲突率 | 同一私有码映射到多个标准码的比例 | 冲突私有码数 / 总私有码数 |
| 单向率 | 私有码 -> 标准码单向映射（无反向映射） | 单向映射数 / 总映射数 |

```java
public class TermMappingQualityScorer {
    public MappingQuality score(String sourceSystem, String standard) {
        MappingQuality q = new MappingQuality();

        // 覆盖率
        int total = termDao.countSourceCodes(sourceSystem);
        int mapped = termDao.countMappedCodes(sourceSystem, standard);
        q.setCoverage((double) mapped / total);

        // 冲突率
        int conflict = termDao.countConflictMappings(sourceSystem, standard);
        q.setConflictRate((double) conflict / mapped);

        // 准确率（人工抽样）
        q.setAccuracy(manualReviewService.sampleAccuracy(sourceSystem, standard));

        return q;
    }
}
```

#### 5. 医防融合的工程化

**医防融合的核心**：把"医疗"（治已病）和"预防"（治未病）结合起来，从"以治病为中心"转向"以健康为中心"。

**临床数据与公卫数据的融合**：

| 数据源 | 数据类型 | 系统 |
|--------|---------|------|
| 临床数据 | 门诊病历、住院病历、检查检验、处方、影像 | HIS/EMR/LIS/PACS |
| 公卫数据 | 健康档案、慢病随访、免疫接种、传染病报告 | 公卫系统、慢病随访系统 |

**融合的工程挑战**：

- 数据标准不统一（临床用 ICD，公卫用国家公卫数据集）
- 数据归属不同（临床归医院，公卫归疾控/基层）
- 数据时效不同（临床实时，公卫批量）
- 业务流程不同（临床以诊疗为中心，公卫以人群为中心）

**以"高血压"为例的医防融合链路**：

```
1. 门诊就诊（临床）
   - 患者在县医院心内科就诊，血压 160/95 mmHg
   - HIS 记录：诊断"高血压 ICD:I10"，处方"氨氯地平 5mg qd"
   ↓ 触发
2. 慢病建档（公卫）
   - 数据中台监听 HIS 诊断事件
   - 诊断 ICD:I10 -> 触发"高血压慢病建档"规则
   - 自动创建慢病档案（公卫系统）
   - 档案关联：患者 masterId、首诊机构、首诊日期、首诊血压
   ↓ 触发
3. 随访计划生成（公卫）
   - 慢病建档完成 -> 触发"高血压随访计划"规则
   - 随访计划：第 1 个月每周随访 1 次，第 2-3 个月每 2 周 1 次，3 个月后每月 1 次
   - 随访方式：电话/上门/门诊
   - 随访执行机构：患者归属的乡镇卫生院
   ↓ 触发
4. 随访执行（公卫 + 临床）
   - 乡镇卫生院医师/村医执行随访
   - 随访记录：血压、用药、不良反应、生活方式
   - 随访数据上报到数据中台
   ↓ 触发
5. 健康干预（公卫 + 临床）
   - 血压控制不达标（如 3 次随访血压 > 140/90）-> 触发"上转"规则
   - 上转县医院调整治疗方案
   - 调整后下转基层继续随访
   ↓ 循环
6. 数据闭环
   - 所有数据汇聚到患者健康档案
   - 医师可调阅"就诊 + 慢病 + 随访 + 干预"完整链路
   - 患者可在 APP 查看自己的慢病管理情况
```

**数据触发机制的工程实现**：

```java
public class ClinicalEventTrigger {
    @KafkaListener(topic = "clinical_events")
    public void onClinicalEvent(ClinicalEvent event) {
        // 1. 解析临床事件
        if (event.getType() == EventType.DIAGNOSIS) {
            String icdCode = event.getDiagnosisCode();

            // 2. 触发慢病建档规则
            if (chronicDiseaseRules.isChronic(icdCode)) {
                ChronicDisease chronic = chronicDiseaseRules.map(icdCode);

                // 检查是否已建档
                if (!chronicArchiveService.exists(event.getPatientId(), chronic)) {
                    // 自动建档
                    ChronicArchive archive = ChronicArchive.builder()
                        .patientId(event.getPatientId())
                        .chronicType(chronic)
                        .firstDiagnosisDate(event.getEventDate())
                        .firstDiagnosisOrg(event.getOrgId())
                        .build();
                    chronicArchiveService.create(archive);

                    // 触发随访计划
                    followUpService.generatePlan(archive);
                }
            }
        }

        // 3. 触发其他规则（如传染病报告、肿瘤登记）
        if (notifiableDiseaseRules.isNotifiable(icdCode)) {
            reportService.triggerReport(event);
        }
    }
}
```

**医防融合的挑战与对策**：

| 挑战 | 对策 |
|------|------|
| 数据标准不统一 | 数据中台做术语映射、数据元标准化 |
| 数据归属不同 | 区分"采集机构、归属机构、调阅机构"，明确数据权利 |
| 数据时效不同 | 临床实时上报 + 公卫批量同步，数据中台做实时合并 |
| 业务流程不同 | 业务流程编排引擎，跨机构跨系统编排 |
| 数据质量参差 | 数据质量评分 + 治理工单 + 闭环修复 |
| 隐私合规 | 患者授权 + 调阅留痕 + 透明化 |
