## 一、处方流转在高可用架构中的定位

### 1.1 核心一句话

> **处方流转 = 处方驱动的下单 + 强监管的流通 + 医保结算的耦合 + 多仓多店的调度 + 药事服务的合规**
>
> 与电商购药、药店 O2O 的根本差异：处方驱动（非用户选商品）、强监管流通（每步留痕 ≥ 15 年）、医保结算强耦合（医保 + 自费混合支付）、多仓多店调度（区域仓 + 城市仓 + 门店仓 + 合作药店）、药事服务合规（药师审方是法定环节）。

### 1.2 处方流转的五大特征

| 特征 | 含义 | 架构影响 |
| --- | --- | --- |
| 处方驱动 | 用户只能选药店/配送方式，不能选商品 | 处方接收 + 路由 + 拆单引擎 |
| 强监管流通 | 接收/审核/调剂/配送/签收/退药全留痕 | 状态机 + 不可变审计 + 监管上报 |
| 医保结算耦合 | 医保 + 自费混合支付，医保目录限定 | 医保接口对接 + 预结算/结算 + 凭证回传 |
| 多仓多店调度 | 区域仓 + 城市仓 + 门店仓 + 合作药店 | 库存统一抽象 + 多仓调度算法 |
| 药事服务合规 | 药师审方 + 调剂 + 用药指导法定环节 | 审方规则引擎 + 药师工作台 |

## 三、处方流转状态机

### 3.1 完整状态机

```
DRAFT 草稿
  ↓ 处方提交流转 + 医师签章
REVIEWED 已审核（药师审方通过 + 药师签章）
  ↓ 处方路由选定药店
ROUTED 已路由
  ↓ 药店接单 + 库存预扣
ACCEPTED 已接单
  ↓ 调剂完成 + 复核通过 + 出库
DISPENSED 已调剂
  ↓ 配送员取货 + 运单号回传
SHIPPED 已发货
  ↓ 配送员送达
DELIVERED 已送达
  ↓ 签收确认（照片 + 地理位置 + 时间戳）
SIGNED 已签收（终态）
  ↓ 退药申请通过 + 药品回收 + 退款
RETURNED 已退药（终态）

异常终态：
- VOID 已作废：草稿超时未提交
- REJECTED 已退方：药师审方不通过
- CANCELED 已退单：药店接单后因库存/调剂原因退单
- TIMEOUT 已超时：配送超时未签收且无法联系患者
```

### 3.2 每个状态的进入/退出/超时

| 状态 | 进入条件 | 退出条件 | 超时 |
| --- | --- | --- | --- |
| DRAFT | 互联网医院开方 | 提交流转 + 医师签章 -> REVIEWED | 24h 未提交 -> VOID |
| REVIEWED | 药师审方通过 | 路由选定药店 -> ROUTED | 30min 未审 -> REJECTED |
| ROUTED | 处方路由选定药店 | 药店接单 -> ACCEPTED | 10min 未接单 -> 转其他药店 |
| ACCEPTED | 药店接单 + 库存预扣 | 调剂完成 -> DISPENSED | 30min 未调剂 -> CANCELED |
| DISPENSED | 调剂出库 | 配送员取货 -> SHIPPED | - |
| SHIPPED | 已发货 | 送达 -> DELIVERED | 48h 未送达 -> 告警 |
| DELIVERED | 配送员送达 | 签收 -> SIGNED | 24h 未签收 -> 自动签收 |
| SIGNED | 患者签收 | 申请退药 -> RETURNED | - |
| RETURNED | 退药完成 | 终态 | - |

### 3.3 状态机实现

- **方案 A：状态模式 + 事件驱动**（推荐）
- **方案 B：DB 状态字段 + 定时扫描兜底**
- **处方流转推荐：A + 定时扫描兜底**
  - 主流程用状态模式
  - 定时扫描兜底超时
  - 所有状态变更落 `rx_status_log`（append-only），监管可追溯

### 3.4 为什么必须异步

| 步骤 | SLA | 同步等待问题 |
| --- | --- | --- |
| 药师审方 | 30 分钟内 | 患者前台卡 30 分钟 |
| 药店接单 | 5-10 分钟 | 患者前台卡几分钟 |
| 药品调剂 | 10-30 分钟 | 同步等待不可接受 |
| 药品配送 | 数小时~数天 | 跨城配送无法同步 |
| 医保结算 | 1-5 分钟 | 患者前台卡几分钟 |
| 监管上报 | 24 小时 | 监管接口限流、抖动 |

实现：MQ + 状态机 + 定时对账 + 入站消息幂等（处方 ID + 步骤 ID）。

## 四、配送三种模式对比

| 维度 | 合作药店自配送 | 第三方配送 | 医院药房配送 |
| --- | --- | --- | --- |
| 配送范围 | 门店周边 3-5 km | 全国 | 医院周边 3-5 km 或院内自提 |
| 时效 | 30 分钟-2 小时 | 当日达/次日达 | 1-4 小时 |
| 运单号 | 药店自建 | 配送方系统 | 医院药房自建 |
| 温度监控 | 部分 | 完整冷链 | 部分 |
| 签收 | 配送员 + 患者确认 | 配送员 + 患者确认 | 患者到院签字 |
| 退药复杂度 | 低 | 高 | 中 |

### 4.1 统一抽象：Shipment 实体

通过 `shipmentType` 区分（PHARMACY_SELF / THIRD_PARTY / HOSPITAL_PHARMACY），统一字段：shipId、orderId、rxId、carrierId、trackingNo、pickupTime、shipTime、deliveryTime、signTime、signPhoto、signGeo、signPerson、trackEvents、coldChain。

### 4.2 轨迹同步

- 配送方 webhook 回调推送轨迹
- 我方定时拉取兜底（每 10 分钟）
- 冷链药品每 5 分钟回传温度，超温告警

### 4.3 签收合规要求

- 签收照片：药品包装 + 患者面部（脱敏）
- 地理位置：精确到楼栋
- 签收时间戳：精确到秒
- 签收人：本人（生物识别）/ 代签人（关系证明 + 短信验证码）
- 冷链药品：同步记录开箱温度，超 8℃ 拒收

## 五、多药品 + 多药店拆单

### 5.1 拆单策略

| 策略 | 描述 | 优先级 |
| --- | --- | --- |
| 单药店优先 | 选能一次性满足所有药品的最近药店 | 1（推荐） |
| 多药店拆单 | 单药店无法满足时按距离 + 库存满足度拆 | 2 |
| 处方改方 | 所有药店缺某药时反馈医师改方 | 3（兜底） |

### 5.2 拆单算法

```
1. 查询所有合作药店库存
2. 策略 1：找单药店满足 -> 优先选最近的
3. 策略 2：多药店拆单 -> 每次选能满足最多剩余药品的最近药店
4. 策略 3：所有药店都无法满足 -> 反馈医师改方
```

### 5.3 拆单后对账

- 一张处方 -> 多子订单 -> 多配送单 -> 多签收单
- 处方维度状态聚合：所有子订单签收完成 -> 处方 SIGNED
- 医保结算按处方维度（不是子订单），确保医保金额一致
- 退药可按子订单退，也可整单退

## 六、多仓库存模型

### 6.1 多仓结构

```
区域仓（Region）-> 城市仓（City）-> 门店仓（Store）-> 合作药店（Partner）
```

| 仓类型 | 服务范围 | 库存深度 | 时效 |
| --- | --- | --- | --- |
| 区域仓 | 多省 | 数月用量 | 次日达 |
| 城市仓 | 单城市 | 数周用量 | 当日达 |
| 门店仓 | 单门店 | 数日用量 | 30 分钟-2 小时 |
| 合作药店 | 单药店 | 实时查询 | O2O |

### 6.2 库存表设计

```sql
drug_stock(stock_id, drug_code, warehouse_id, warehouse_type,
           batch_no, expiry_date,
           total_qty, locked_qty, available_qty AS (total_qty - locked_qty))
stock_log(log_id, stock_id, op_type[LOCK|UNLOCK|OUTBOUND|INBOUND|RETURN],
          op_qty, ref_type, ref_id, op_time)  -- append-only
```

### 6.3 库存扣减时机

| 时机 | 优点 | 缺点 |
| --- | --- | --- |
| 下单预扣（locked_qty+） | 防超卖、用户友好 | 库存被锁，未支付订单占用 |
| 出库实扣（total_qty-） | 真实反映库存 | 下单到出库期间可能超卖 |
| 签收确认扣 | 用户视角简单 | 超卖风险高 |

**推荐方案：下单预扣 + 出库实扣 + 签收确认**

```
1. 下单：locked_qty += N（预扣）  -- 超时未支付/未出库 -> 释放
2. 出库：locked_qty -= N, total_qty -= N（实扣）
3. 签收：仅记录状态，不扣库存
4. 退药：total_qty += N（回库，需校验批号、效期、包装）
```

### 6.4 防超卖

```sql
UPDATE drug_stock
SET locked_qty = locked_qty + #{qty}
WHERE drug_code = #{drugCode}
  AND warehouse_id = #{warehouseId}
  AND available_qty >= #{qty};
-- affected rows = 0 表示库存不足
```

- Redis 预扣（Lua 脚本保证原子性）+ 异步落 DB
- DB 兜底（事务 + 行锁）
- 定时对账 Redis vs DB vs 实物

### 6.5 多仓调度优先级

1. 就近原则：距患者最近的门店仓
2. 库存满足度：单仓不足跨仓组合
3. 冷链能力：冷链药品必须有冷链能力
4. 医保定点：医保药品必须医保定点药店
5. 时效要求：紧急用药选最近门店仓 + 自配送

### 6.6 库存同步实时性

| 仓类型 | 同步频率 |
| --- | --- |
| 门店仓/合作药店 | 实时（API + Redis 缓存 1 分钟） |
| 城市仓 | 准实时（每 5 分钟） |
| 区域仓 | 准实时（每 15 分钟） |
| 兜底 | 下单时直接查源系统 |

## 七、互联网+医保线上结算

### 7.1 完整链路

```
1. 激活医保电子凭证（医保码，有效期 1 年）
2. 医保身份核验（参保地、参保类型、账户余额）
3. 医师开方（药品标注医保编码）
4. 处方流转到合作药店
5. 医保预结算（计算医保金额 + 自付金额 + 限定范围校验）
6. 患者确认支付（医保 + 自费混合）
7. 医保结算 + 自费支付
8. 医保结算凭证回传
9. 药店出库配送
10. 患者签收
11. 日终对账
```

### 7.2 关键接口规范

| 接口 | SLA | 失败兜底 |
| --- | --- | --- |
| 医保电子凭证激活 | 5s | 走实体医保卡 |
| 医保身份核验 | 1s | 提示自费 |
| 医保目录查询 | 500ms | 缓存兜底 |
| 医保预结算 | 3s | 提示自费 |
| 医保结算 | 5s | 释放库存，订单失败 |
| 凭证回传 | 5s | 重试 + 补报 |
| 拒付通知 | 异步 | 触发退款链路 |

### 7.3 预结算 vs 结算

| 维度 | 预结算 | 结算 |
| --- | --- | --- |
| 时机 | 处方路由后，患者确认支付前 | 患者确认支付后 |
| 作用 | 计算医保 + 自付金额 | 实际扣款，生成结算凭证 |
| 资金 | 不扣款 | 医保 + 个人扣款 |
| 失败影响 | 可重新预结算 | 需退款链路 |
| 时效 | 30 分钟内有效 | 不可撤销（除非退药） |

### 7.4 互联网+医保关键约束

- **医保电子凭证强制**：必须用医保码，不能用实体卡
- **统筹账户 vs 个人账户**：门诊统筹、门诊慢病、门诊特殊病报销政策不同
- **限定支付范围**：部分药品有"限二线用药、限住院"等限定
- **医保目录版本**：每年调整，需动态加载
- **医保接口限流**：QPS 有限，需"批量预结算 + 异步结算"

## 八、医保结算与配送的协同

### 8.1 核心原则：先医保结算后配送

**为什么必须先结算后配送**：

1. 医保资金安全：未结算就配送可能造成医保基金损失
2. 合规要求：医保定点药店出库前必须完成医保结算
3. 业务流程：药店调剂出库前必须知道收款金额
4. 退药复杂度：先配送后结算的话，配送失败后退款链路复杂

### 8.2 协同链路

```
处方审核通过 -> 处方路由选定药店 -> 药店接单 + 库存预扣
-> 医保预结算 -> 患者确认支付 -> 医保结算 + 自费支付
-> 药店调剂出库 -> 配送 -> 签收 -> 医保结算凭证回传
```

### 8.3 配送失败/退药的医保回退

| 场景 | 处理 |
| --- | --- |
| 医保结算成功 + 出库前取消 | 医保退款（全额）+ 自费退款（全额）+ 释放库存 |
| 医保结算成功 + 配送失败 | 药品回收 + 医保退款（全额）+ 自费退款（全额） |
| 医保结算成功 + 签收后退货 | 药品回收 + 医保退款（按比例）+ 自费退款（按比例） |
| 医保结算成功 + 患者拒收 | 药品回药店 + 医保退款（全额）+ 自费退款（全额） |

### 8.4 资金倒挂风险

- **定义**：医保结算成功但配送失败，医保已支付但患者未收到药
- **防范**：
  - 配送失败后立即触发医保退款
  - 资金倒挂超 24h 未处理 -> 告警人工介入
  - 日终对账校验"医保结算数 = 签收数 + 已退款数"

### 8.5 退药的医保退款链路

```
患者申请退药 -> 药师审核 -> 药店回收药品（total_qty += N）
-> 医保退款 InsAdapt.refund(settlementId, 退药金额)
-> 自费退款（第三方支付退款）
-> 退药完成 -> 医保退款凭证回传 -> 日终对账
```

## 九、四流合一的对账

### 9.1 对账口径

- **主键**：rxId（处方号）
- **维度**：每日凌晨对账，按 rxId 维度校验四流齐全
- **辅助键**：payId、shipId、settlementId、invoiceId

### 9.2 对账数据源

| 流 | 数据源 | 关键字段 |
| --- | --- | --- |
| 信息流 | prescription | rxId、orderId、status |
| 资金流 | payment | payId、rxId、amount、payType |
| 物流 | shipment + sign_record | shipId、rxId、status、signTime |
| 票据流 | insurance_settlement | settlementId、rxId、payId、invoiceId |

### 9.3 对账规则

```
1. 处方维度：每张处方必须有 1 条支付 + 1 条配送 + 1 条签收 + 1 条医保结算
2. 金额对账：处方金额 = 支付金额 = 医保支付 + 自费支付
3. 时间对账：处方开立 < 支付 < 配送 < 签收 < 票据回传
4. 状态对账：处方状态 = SIGNED 必须有签收；RETURNED 必须有退药 + 退款
```

### 9.4 异常类型与处理

| 异常类型 | 描述 | 处理 |
| --- | --- | --- |
| 有处方无支付 | 处方已开但未支付 | 24h 后作废处方，释放库存 |
| 有支付无配送 | 支付成功但未配送 | 告警人工介入 |
| 有配送无签收 | 配送中但未签收 | 48h 后告警，联系配送方 |
| 有签收无票据 | 签收但凭证未回传 | 重试回传 + 补报 |
| 四流不齐 | 任意流缺失 | 异常队列，人工介入 |
| 金额不一致 | 四流金额不匹配 | 异常队列，资金对账 |
| 时间逆序 | 时间戳逻辑错误 | 异常队列，日志追溯 |

### 9.5 对账兜底

- T+1 对账：每日凌晨对账前一日数据
- T+7 复核：每周复核未结清异常单
- 月报：每月生成对账月报
- 监管追溯：所有对账记录保留 ≥ 15 年

## 十、医保拒付处理

### 10.1 拒付原因

| 类型 | 描述 | 示例 |
| --- | --- | --- |
| 处方不合规 | 处方违反医保规定 | 超量开药、超适应症 |
| 患者非本人 | 医保卡冒用 | 患者医保卡给家人开药 |
| 药品串换 | 实际售药与处方不符 | 处方开 A 药实际给 B 药 |
| 二次转让 | 患者将医保药品转卖 | 套保行为 |
| 虚假处方 | 处方本身虚假 | 假冒医师开方 |
| 限定支付范围 | 不符合限定支付 | 门诊开限住院药品 |

### 10.2 拒付处理流程

```
1. 接收拒付单（拒付单号 + rxId + 原因 + 金额）
2. 冻结该处方的退药/退款（防重复退款）
3. 资金追回：
   - 医保基金部分：医保局直接扣减
   - 个人自付部分：从患者医保账户扣减 / 线下追回
4. 标记处方为"已拒付"
5. 上报监管（互联网诊疗监管 + 医保监管）
6. 触发风控告警：医师/药店/患者行为审计
7. 资金倒挂处理：
   - 已签收 + 拒付：医保基金损失，向患者追回
   - 未签收 + 拒付：撤销配送，全额追回
```

### 10.3 拒付后个人自付转换

- 医保拒付 -> 原医保支付部分需患者自付
- 系统生成"自付补缴单"推送患者
- 患者支付后处方恢复"已签收（自费）"
- 患者拒补缴 -> 标记"待追回"，进入风控黑名单

### 10.4 拒付退款链路

```
1. 已签收 + 患者配合补缴 -> 处方恢复
2. 已签收 + 患者拒补缴 -> 线下法务追回
3. 未签收 + 拒付 -> 撤销配送，全额追回
4. 已退药 + 拒付 -> 仅追回已退款部分
```

## 十三、药师审方

### 13.1 法律依据

《医疗机构处方审核规范》（2018）：药师必须对处方进行合法性、规范性、安全性审核，未审核不得调剂。

### 13.2 审方内容

| 维度 | 审核内容 | 示例 |
| --- | --- | --- |
| 合法性 | 医师资质、处方权限、特殊药品 | 麻醉药不得在线开 |
| 规范性 | 处方格式、药品名称、用法用量、签名 | 药品须用通用名 |
| 安全性 | 配伍禁忌、相互作用、剂量、过敏史、特殊人群 | A+B 相互作用 |

### 13.3 审方规则引擎

```
1. 合法性：
   - 医师资质（执业证、资格证、多点执业、科室匹配）
   - 处方权限（特殊药品需特殊处方权）
   - 互联网医院禁止药品（麻醉、精神、毒性、放射、易制毒、终止妊娠）

2. 规范性：
   - 处方格式（前记、正文、后记）
   - 药品名称（须用通用名）
   - 用法用量（符合规范）

3. 安全性：
   - 配伍禁忌（中药十八反十九畏）
   - 药物相互作用（西药 CONTRAINDICATED / SEVERE）
   - 剂量上限
   - 过敏史
   - 特殊人群（妊娠、儿童、老年）
```

### 13.4 SLA 30 分钟保障

1. **自动审方**：规则引擎自动审 80%，秒级出结果
2. **人工审方**：需人工审的 20%，药师人工审
3. **药师排班**：7×24 值班，每班至少 2 名（含 1 名主管药师）
4. **超时告警**：25min 黄色 / 30min 红色 + 升级
5. **超时兜底**：30min 未审 -> 自动退方（不能"默认通过"）

### 13.5 审方退回处理

| 类型 | 处理 |
| --- | --- |
| 退回医师 | 医师修改 -> 重新签章 -> 重新审方 |
| 拒方 | 处方作废，记录医师违规，上报监管 |
| 转主管药师 | 复杂用药转主管药师审 |

| 终止妊娠药品 | 《母婴保健法》 |

三道防线：前端选药校验 + 后端提交校验 + 药师审方校验。

### 14.2 特药全程追溯（冷链 + 批号 + 效期）

```
1. 处方开立：rxId 关联药品追溯码（20 位）
2. 药品出库：扫码记录批号、效期、出库时间、出库人
3. 冷链配送：每 5 分钟回传温度，超温告警
4. 患者签收：扫码确认，记录开箱温度
5. 用药反馈：不良反应 24h 内上报
6. 全程追溯：药品追溯码每环节采集，医保局一键查询
```

### 14.3 药品追溯码体系

- **20 位编码**：前 8 厂商 + 4 药品 + 4 规格 + 4 序列号
- **采集时机**：生产出库、流通出库、零售出库、患者签收
- **校验时机**：每环节扫码校验上一环节码，防伪防串货
- **回传医保**：零售出库回传追溯码到医保电子处方中心

### 14.4 冷链药品工程实现

```json
{
  "coldChain": {
    "required": true,
    "temperatureRange": [2, 8],
    "temperatureLogs": [...],
    "temperatureExcursion": false
  },
  "batchNo": "B20260601",
  "expiryDate": "2027-06-01"
}
```

- 超温 > 8℃ 持续 5min -> 自动告警 + 药品冻结 + 转人工
- 签收时开箱温度超温 -> 自动拒收 + 回收 + 全额退款

## 十五、退药与退款

### 15.1 退药条件

| 条件 | 是否可退 |
| --- | --- |
| 药品质量问题 | 可退 |
| 药品未拆封 + 包装完整 | 可退（7 天内） |
| 药品已拆封 | 不可退 |
| 特殊管理药品 | 不可退 |
| 冷链药品拆零 | 不可退 |
| 近效期药品（< 6 个月） | 不可退 |
| 患者签收超过 7 天 | 不可退 |

### 15.2 退药流程

```
1. 系统校验退药条件
2. 药师审核（包装照片 + 批号效期）
3. 药品回收（配送员上门 / 患者到店）
4. 药店验收（包装、批号、效期、追溯码）
5. 验收通过 -> 药品回库（total_qty += N）
6. 退款：医保退款 + 自费退款
7. 退款凭证回传
8. 处方状态 -> 已退药
9. 监管上报
```

### 15.3 退款资金流对账

| 流 | 退款流水号 | 关键字段 |
| --- | --- | --- |
| 信息流 | rxId | 退药数量、原因、时间 |
| 资金流 | refundPayId | 原 payId、退款金额、方式 |
| 物流 | returnShipId | 回收时间、验收结果 |
| 票据流 | refundInvoiceId | 原 settlementId、退款金额 |

### 15.4 退药合规要点

1. **药品回收验收**：校验包装、批号、效期、追溯码，防"假退药"
2. **回库登记**：批号效期重新登记，单独标记"退货药品"
3. **再销售限制**：退货药品再销售须明示
4. **监管上报**：退药全流程上报，防"虚假退药"套保

## 十六、处方流转防欺诈

### 16.1 四大欺诈类型

| 类型 | 描述 | 示例 |
| --- | --- | --- |
| 套保 | 开药后转卖套现 | 患者开高价药转卖黑市 |
| 串换药品 | 处方 A 实际给 B | 处方开阿胶实际给保健品 |
| 虚假处方 | 假冒医师开方 | 假冒医师资质 |
| 二次转让 | 患者将医保药转卖 | 患者开药转卖给药贩 |

### 16.2 风控规则（医师/药店/患者/处方四层）

```
1. 医师层面：
   - 开方量异常（日均 > 同科室均值 3 倍）
   - 药品集中度高（某药 > 50%）
   - 异常时段开方（凌晨 2-5 点）

2. 药店层面：
   - 单品销售占比过高（> 50%）
   - 单医师处方占比过高（> 50%）
   - 销售量异常（日均 > 同规模 3 倍）

3. 患者层面：
   - 跨机构开同种药（7 天内多机构开同种）
   - 开药频率异常（月开 > 30 天用量）
   - 异地开药（参保地 ≠ 开方地）

4. 处方层面：
   - 金额异常（> 同类均价 3 倍）
   - 药品组合异常（套保组合）
   - 长期处方异常（长期处方开高价药）
```

### 16.3 欺诈识别后处置

| 风险等级 | 处置 |
| --- | --- |
| LOW | 通过，记录日志，纳入画像 |
| MEDIUM | 人工审核，药师重点核查 |
| HIGH | 拦截，风控团队 24h 内出结果 |
| CRITICAL | 拦截 + 上报监管（医保 + 互联网诊疗） |

### 16.4 可信审计设计

1. **不可变审计表**（append-only）：
   - `audit_log(id, op_type, rx_id, operator_id, op_time, op_data, prev_hash, curr_hash)`
   - 哈希链：`curr_hash = SHA256(prev_hash + op_data)`，防篡改
2. **操作日志**：所有用户操作记录，保留 ≥ 15 年
3. **数据变更日志**：变更前后值，追溯修改历史
4. **监管审计接口**：医保局、卫健委一键查询，访问留痕

### 16.5 哈希链校验

```
record 1: prev_hash=0 curr_hash=SHA256("0" + op_data_1)
record 2: prev_hash=curr_hash_1 curr_hash=SHA256(curr_hash_1 + op_data_2)
record 3: prev_hash=curr_hash_2 curr_hash=SHA256(curr_hash_2 + op_data_3)

## 十九、面试速记口诀

```
处方流转五特征：处方驱动、强监管、医保耦合、多仓多店、药事合规
业务架构十模块：接审路订库、配签退对医
协同样四件套：药师审方、用药指导、药店接单、配送交接
集成层四对接：互联网医院、药店、配送方、医保
监管层三必做：电子处方中心上报、处方追溯、电子签名
支撑层五基础：药品库、药店库、医保目录、配伍禁忌、用药知识

流转状态机：DRAFT -> REVIEWED -> ROUTED -> ACCEPTED -> DISPENSED
         -> SHIPPED -> DELIVERED -> SIGNED -> RETURNED
异常终态：VOID（草稿超时）/ REJECTED（退方）/ CANCELED（退单）/ TIMEOUT（超时）

为什么必须异步：审方 30min + 接单 5min + 调剂 30min + 配送数小时 + 结算 5min + 上报 24h

配送三模式：药店自配（30min-2h）、第三方（当日/次日）、医院药房（1-4h）
统一抽象 Shipment：shipId + carrierId + trackingNo + signPhoto + signGeo + coldChain
签收四要素：照片 + 地理位置 + 时间戳 + 签收人（本人/代签）
冷链四要求：5min 回传温度、超温告警、开箱温度、超 8℃ 拒收

拆单三策略：单药店优先、多药店拆单、处方改方
多仓四层：区域仓（数月）-> 城市仓（数周）-> 门店仓（数日）-> 合作药店（实时）
库存扣减三步：下单预扣 + 出库实扣 + 签收确认
防超卖三件：Redis 预扣 + DB 行锁 + 定时对账

医保结算链：激活凭证 -> 身份核验 -> 开方 -> 预结算 -> 患者确认
         -> 结算 + 自费 -> 凭证回传 -> 配送 -> 签收 -> 对账
预结算 vs 结算：计算 vs 扣款，30min 有效 vs 不可撤销
协同原则：先医保结算后配送（资金安全 + 合规 + 业务流程 + 退药复杂度）

四流合一：rxId 信息流、payId 资金流、shipId 物流、settlementId 票据流
对账四维度：处方维度 + 金额维度 + 时间维度 + 状态维度
异常七类：有处方无支付/有支付无配送/有配送无签收/有签收无票据/四流不齐/金额不一致/时间逆序

拒付六因：处方不合规/患者非本人/串换药品/二次转让/虚假处方/限定支付范围
拒付流程：冻结 -> 资金追回 -> 标记已拒付 -> 上报监管 -> 风控告警 -> 资金倒挂处理

跨省医保：参保地政策 + 就医地目录，国家平台中转
试点五省市：京津冀、长三角、西南 5 省

双轨制：自建平台（合作药店）+ 省级电子处方中心（医保定点药店）
统一监管上报、统一对账

审方三维度：合法性 + 规范性 + 安全性
SLA 30min 保障：自动审方 80% + 人工审方 20% + 药师 7×24 排班
超时兜底：30min 未审 -> 自动退方（不能默认通过）

退药条件：质量可退、未拆封 7 天可退、拆封不可退、特管药不可退、冷链拆零不可退
退药流程：校验 -> 药师审核 -> 回收 -> 验收 -> 退款 -> 凭证回传 -> 上报
