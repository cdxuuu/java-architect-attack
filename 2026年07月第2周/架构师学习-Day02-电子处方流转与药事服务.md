## 背景

Day01 完成了"互联网医院核心架构 + 在线问诊 + 合规监管 + 四流合一"的概览，其中"四流合一"是 30 分钟的速讲。Day02 把"处方流转 + 药事服务 + 互联网+医保结算 + 药品配送 + 多仓库存"这条链路做深。

2026 年，处方流转的政策线已经走完三轮：

- **2017 年**：原国家卫计委《关于进一步改革完善药品生产流通使用政策的若干意见》首次提出"门诊患者自主选择在医疗机构或零售药店购药"，处方外流开闸。
- **2021 年**：国家医保局《关于积极推进'互联网+'医疗服务医保支付工作的指导意见》，明确互联网医院可纳入医保协议管理，互联网+医保线上结算开始落地。
- **2023 年**：国家医保局《关于开展医保基金监管专项整治工作的通知》将"虚假处方、串换药品、二次转让"列为重点打击对象，处方流转的"防欺诈"成为合规底线。
- **2024-2025 年**：国家医保局"电子处方中心"在 20+ 省份试点上线，作为处方流转的官方枢纽（医保定点机构处方的统一接收、流转、追溯），与互联网医院自建处方流转平台形成"双轨"。

行业格局：

- **互联网医院自建处方流转平台**：京东健康、阿里健康、平安好医生等，对接自家药房或合作连锁药店。
- **省级医保电子处方中心**：医保局统一建设，连接医保定点医院 + 医保定点药店，处方在医保平台流转，资金走医保结算。
- **第三方处方流转平台**：如微信医保支付、支付宝医保电子凭证，作为"通道方"连接医院、药店、医保。

啄木鸟云健康的体检客户中，部分三甲医院健康管理中心在尝试"体检异常项 -> 互联网医院开方 -> 患者选择就近药店自提或配送"的链路。Day02 聚焦**电子处方流转平台 + 药品配送 + 互联网+医保结算 + 药事服务**，承接 Day01 的"四流合一"概念，向下深挖到工程实现。

Day02 与 Day01 的衔接点：

- **Day01 题目三 - 四流合一**：Day02 深挖四流的对账、异常处理、追溯链路。
- **Day01 题目三 - 电子签名**：Day02 延伸到处方流转过程中的"药师审方签章、药店调剂签章"。
- **Day01 题目三 - 监管上报**：Day02 延伸到处方流转的"国家医保电子处方中心上报"。
- **第1周 Day05 医保对接**：Day02 延伸到"互联网+医保线上结算"的具体实现。

Day02 的工程约束（与普通电商购药、药店 O2O 的根本差异）：

1. **处方驱动的下单**：不是用户选商品下单，而是处方驱动药品 + 数量 + 用法，用户只能选药店/配送方式。
2. **强监管的流通**：每张处方的接收、审核、调剂、配送、签收、退药都必须留痕，可被医保局追溯 ≥ 15 年。
3. **医保结算的强耦合**：医保目录药品的支付必须走医保结算，自费药品走自费支付，部分药品可"医保+自费"混合支付。
4. **多仓多店的库存调度**：区域仓 + 城市仓 + 门店仓 + 合作药店库存，必须实时扣减防超卖。
5. **药事服务的合规**：药师审方、调剂、用药指导是法定环节，不能省略，必须在系统中体现。

## 题目一（架构设计题）：处方流转平台整体架构与药品配送全链路

请你设计一套支撑**第三方平台型互联网医院 + 多家合作药店 + 多仓配送**的处方流转平台，回答以下问题：

1. 你会将处方流转平台拆分为哪些核心模块/子系统？至少覆盖：处方接收、处方审核（药师审方）、处方路由（药店选择）、药品库存、订单管理、配送管理、签收管理、退药管理、对账管理、监管上报。每个模块的职责边界是什么？哪些是"业务核心"，哪些是"基础设施"？
2. 处方从互联网医院到药店的"流转链路"是同步还是异步？为什么？流转过程中的状态机是怎样的（DRAFT -> REVIEWED -> ROUTED -> ACCEPTED -> DISPENSED -> SHIPPED -> DELIVERED -> SIGNED -> RETURNED）？每个状态的进入/退出条件、超时是什么？
3. 药品配送的三种模式（合作药店自配送、第三方配送如顺丰/京东物流、医院药房配送）在架构上有什么差异？配送单的"运单号回传、轨迹同步、签收照片、地理位置、温度监控（冷链药品）"如何统一抽象？
4. 处方"多药品 + 多药店"的拆单逻辑：一张处方开了 5 种药，3 种在合作药店 A 有货、2 种在合作药店 B 有货，如何拆单？如果患者希望"一次配送"如何聚合？如果某种药在所有合作药店都缺货，如何处理（处方改方 / 等待补货 / 转医院药房）？
5. 药品库存与多仓调度：区域仓 + 城市仓 + 门店仓 + 合作药店库存的"多仓库存"模型如何设计？库存扣减时机（下单预扣、出库实扣、签收确认）如何选择？如何防止超卖？库存同步的实时性如何保障（Redis 缓存 + DB 兜底 + 定时对账）？

#### 2. 处方流转链路的同步异步与状态机

**为什么必须异步**：

| 步骤 | SLA | 同步等待问题 |
|------|-----|-------------|
| 药师审方 | 30 分钟内 | 患者前台卡 30 分钟，体验崩坏 |
| 药店接单 | 5-10 分钟 | 患者前台卡几分钟 |
| 药品调剂 | 10-30 分钟（看门店繁忙度） | 同步等待不可接受 |
| 药品配送 | 数小时~数天 | 跨城配送无法同步 |
| 医保结算 | 1-5 分钟（医保接口抖动可能更长） | 患者前台卡几分钟 |
| 监管上报 | 24 小时 | 监管接口限流、抖动 |
| 退药处理 | 数小时 | 同步等待不可接受 |

**异步的实现**：MQ（RocketMQ/Kafka）+ 状态机 + 定时对账 + 入站消息幂等（处方 ID + 步骤 ID）。

**处方流转状态机**（处方维度的状态流转，订单维度可能进一步拆分）：

```
                  ┌─────────────┐
                  │  DRAFT      │  互联网医院已开方，未提交流转
                  │  草稿       │  → 超时 24h 未提交 → 已作废
                  └──────┬──────┘
                         │ 处方提交流转 + 医师签章
                         ▼
                  ┌─────────────┐
                  │  REVIEWED   │  药师审方通过 + 药师签章
                  │  已审核     │  → 超时 30min 未审 → 退方
                  └──────┬──────┘  → 审方不通过 → 已退方
                         │ 处方路由选定药店
                         ▼
                  ┌─────────────┐
                  │  ROUTED     │  已路由到药店，等待药店接单
                  │  已路由     │  → 超时 10min 未接单 → 转其他药店
                  └──────┬──────┘
                         │ 药店接单 + 库存预扣
                         ▼
                  ┌─────────────┐
                  │  ACCEPTED   │  药店已接单，等待调剂
                  │  已接单     │  → 超时 30min 未调剂 → 退单
                  └──────┬──────┘
                         │ 调剂完成 + 复核通过 + 出库
                         ▼
                  ┌─────────────┐
                  │  DISPENSED  │  已调剂出库，等待配送交接
                  │  已调剂     │
                  └──────┬──────┘
                         │ 配送员取货 + 运单号回传
                         ▼
                  ┌─────────────┐
                  │  SHIPPED    │  已发货，配送中
                  │  已发货     │  → 超时 48h 未签收 → 告警
                  └──────┬──────┘
                         │ 配送员送达 + 患者签收
                         ▼
                  ┌─────────────┐
                  │  DELIVERED  │  已送达，等待签收确认
                  │  已送达     │  → 超时 24h 未签收 → 自动签收
                  └──────┬──────┘
                         │ 签收确认（照片 + 地理位置 + 时间戳）
                         ▼
                  ┌─────────────┐
                  │  SIGNED     │  已签收（终态）
                  │  已签收     │  → 患者申请退药 → 退药流程
                  └──────┬──────┘
                         │ 退药申请通过 + 药品回收 + 退款
                         ▼
                  ┌─────────────┐
                  │  RETURNED   │  已退药（终态）
                  │  已退药     │
                  └─────────────┘

异常终态：
- 已作废 VOID：草稿超时未提交
- 已退方 REJECTED：药师审方不通过
- 已退单 CANCELED：药店接单后因库存不足/无法调剂等原因退单
- 已超时 TIMEOUT：配送超时未签收且无法联系患者
```

**每个状态的进入/退出/超时**：

| 状态 | 进入条件 | 退出条件 | 超时 |
|------|---------|---------|------|
| DRAFT | 互联网医院开方 | 提交流转 + 医师签章 -> REVIEWED | 24h 未提交 -> VOID |
| REVIEWED | 药师审方通过 | 路由选定药店 -> ROUTED | 30min 未审 -> REJECTED |
| ROUTED | 处方路由选定药店 | 药店接单 -> ACCEPTED | 10min 未接单 -> 转其他药店 |
| ACCEPTED | 药店接单 + 库存预扣 | 调剂完成 -> DISPENSED | 30min 未调剂 -> CANCELED |
| DISPENSED | 调剂出库 | 配送员取货 -> SHIPPED | - |
| SHIPPED | 已发货 | 送达 -> DELIVERED | 48h 未送达 -> 告警 |
| DELIVERED | 配送员送达 | 签收 -> SIGNED | 24h 未签收 -> 自动签收 |
| SIGNED | 患者签收 | 申请退药 -> RETURNED | - |
| RETURNED | 退药完成 | 终态 | - |

**状态机实现**：

- **方案 A：状态模式 + 事件驱动**（推荐）
  - 每个状态实现 `handle(event)` 方法，事件驱动状态流转
  - 优点：状态流转逻辑集中、易测试、易审计
- **方案 B：DB 状态字段 + 定时扫描兜底**
  - 状态变更走"事件 -> 状态机 -> 落库"链路，定时任务扫描超时状态
- **处方流转推荐：A + 定时扫描兜底**
  - 主流程用状态模式（接单、调剂、配送、签收）

#### 4. 多药品 + 多药店的拆单逻辑

**典型场景**：

```
医师开方 5 种药：
- 药品 A：阿托伐他汀钙片（医保乙类）
- 药品 B：苯磺酸氨氯地平片（医保甲类）
- 药品 C：缬沙坦胶囊（医保乙类）
- 药品 D：阿司匹林肠溶片（医保甲类）
- 药品 E：依折麦布片（自费）

合作药店库存情况：
- 药店 A（距患者 1.5 km）：A、B、C、D 有货
- 药店 B（距患者 3.0 km）：A、B、D、E 有货
- 药店 C（距患者 5.0 km）：A、B、C、D、E 有货（最全）
```

**拆单策略**：

**策略 1：单药店优先（推荐）**

- 优先选择能一次性满足所有药品的最近药店（药店 C）。
- 优点：一次配送、患者体验好、对账简单。
- 缺点：可能选择较远的药店，配送时效稍长。

**策略 2：多药店拆单（次选）**

- 当单药店无法满足时，按"距离优先 + 库存满足度"拆单。
- 拆单后每个子订单独立配送，患者可能分多次签收。
- 优点：覆盖率高。
- 缺点：多次配送、患者体验下降、对账复杂。

**策略 3：处方改方（最后兜底）**

- 当所有合作药店都缺某药品时，反馈给医师考虑改方（同疗效替代药品）。
- 改方必须医师重新开方 + 重新签章 + 重新审方。
- 不能由药师或药店自行替换（违反处方管理办法）。

**拆单的工程实现**：

```java
public class OrderSplitter {
    public List<SubOrder> split(Prescription rx, Patient patient) {
        // 1. 查询所有合作药店的库存
        List<PharmacyStock> allStock = stockService.queryStock(rx.getDrugItems(), patient.getLocation());

        // 2. 策略 1：找单药店满足
        Optional<Pharmacy> fullMatchPharmacy = allStock.stream()
            .filter(s -> s.canFulfillAll(rx.getDrugItems()))
            .min(Comparator.comparing(s -> s.distanceTo(patient.getLocation())));

        if (fullMatchPharmacy.isPresent()) {
            return List.of(new SubOrder(rx, fullMatchPharmacy.get(), rx.getDrugItems()));
        }

        // 3. 策略 2：多药店拆单（按药品分组）
        Map<Pharmacy, List<DrugItem>> splitMap = new HashMap<>();
        List<DrugItem> remaining = new ArrayList<>(rx.getDrugItems());

        while (!remaining.isEmpty()) {
            // 每次选能满足最多剩余药品的最近药店
            Pharmacy best = allStock.stream()
                .filter(s -> s.canFulfillAny(remaining))
                .max(Comparator.comparing(s -> s.countFulfillable(remaining)))
                .min(Comparator.comparing(s -> s.distanceTo(patient.getLocation())))
                .orElseThrow(() -> new RxException("无药店可满足"));

            List<DrugItem> fulfilled = best.getFulfillableItems(remaining);
            splitMap.put(best, fulfilled);
            remaining.removeAll(fulfilled);
        }

        return splitMap.entrySet().stream()
            .map(e -> new SubOrder(rx, e.getKey(), e.getValue()))
            .collect(Collectors.toList());
    }
}
```

**拆单后的对账**：

- 一张处方 -> 多个子订单 -> 多个配送单 -> 多个签收单。
- 处方维度的状态聚合：所有子订单签收完成 -> 处方 SIGNED。
- 医保结算按处方维度（不是按子订单），确保医保金额一致。
- 退药可按子订单退，也可整单退。

#### 5. 多仓库存模型与防超卖

**多仓库存模型**：

```
区域仓（Region Warehouse）
   - 服务多个城市
   - 库存深度大（覆盖数月用量）
   - 配送时效：次日达
   ↓
城市仓（City Warehouse）
   - 服务单个城市
   - 库存深度中（覆盖数周用量）
   - 配送时效：当日达
   ↓
门店仓（Store Stock）
   - 单个药店门店
   - 库存深度小（覆盖数日用量）
   - 配送时效：30 分钟-2 小时
   ↓
合作药店库存（Partner Pharmacy）
   - 第三方合作药店
   - 通过 API 实时查询
```

**库存表设计**：

```sql
-- 药品库存表（按仓库维度）
CREATE TABLE drug_stock (
    stock_id BIGINT PRIMARY KEY,
    drug_code VARCHAR(32) NOT NULL,
    warehouse_id VARCHAR(32) NOT NULL,
    warehouse_type ENUM('REGION', 'CITY', 'STORE', 'PARTNER'),
    batch_no VARCHAR(32),  -- 批号
    expiry_date DATE,      -- 效期
    total_qty INT NOT NULL,           -- 总库存
    locked_qty INT DEFAULT 0,         -- 锁定库存（已下单未出库）
    available_qty INT GENERATED ALWAYS AS (total_qty - locked_qty) STORED,  -- 可用库存
    INDEX idx_drug_warehouse (drug_code, warehouse_id),
    INDEX idx_drug_available (drug_code, available_qty)
);

-- 库存流水表（append-only）
CREATE TABLE stock_log (
    log_id BIGINT PRIMARY KEY,
    stock_id BIGINT NOT NULL,
    op_type ENUM('LOCK', 'UNLOCK', 'OUTBOUND', 'INBOUND', 'RETURN') NOT NULL,
    op_qty INT NOT NULL,
    ref_type ENUM('ORDER', 'RETURN', 'ADJUST') NOT NULL,
    ref_id VARCHAR(32) NOT NULL,  -- 订单 ID / 退货 ID
    op_time DATETIME NOT NULL,
    INDEX idx_stock (stock_id, op_time)
);
```

**库存扣减时机选择**：

| 时机 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 下单预扣（locked_qty+） | 防止超卖、用户友好 | 库存被锁，未支付订单占用 | 互联网医院处方流转 |
| 出库实扣（total_qty-） | 真实反映库存 | 下单到出库期间可能超卖 | 必须配合预扣使用 |
| 签收确认扣 | 用户视角简单 | 超卖风险高 | 不推荐 |

**推荐方案：下单预扣 + 出库实扣 + 签收确认**：

```
1. 下单：locked_qty += N（预扣）
   - 防止其他订单占用
   - 超时未支付/未出库 -> locked_qty -= N（释放）
2. 出库：locked_qty -= N, total_qty -= N（实扣）
   - 真实扣减库存
3. 签收：仅记录状态，不扣库存
4. 退药：total_qty += N（回库，需校验批号、效期、包装完整）
```

**防超卖的工程实现**：

```sql
-- 库存预扣（乐观锁 + 行锁）
UPDATE drug_stock
SET locked_qty = locked_qty + #{qty}
WHERE drug_code = #{drugCode}
  AND warehouse_id = #{warehouseId}
  AND available_qty >= #{qty};
-- affected rows = 0 表示库存不足
```

- **Redis 预扣**（高性能）：
  - 下单时 Redis SETNX 预扣库存（Lua 脚本保证原子性）。
  - 异步落 DB（locked_qty 更新）。
  - 兜底：DB 主键约束 + 唯一索引。
- **DB 兜底**（强一致）：
  - DB 事务 + 行锁，保证不超卖。
  - 与 Redis 不一致时以 DB 为准（对账修复）。
- **定时对账**：
  - 每日凌晨对账 Redis vs DB vs 实物盘点。
  - 不一致告警 + 人工介入。

**多仓调度的优先级**：

1. **就近原则**：优先选距患者最近的门店仓。
2. **库存满足度**：单仓不足时跨仓组合。
3. **冷链能力**：冷链药品必须选有冷链能力的仓。
4. **医保定点**：医保药品必须选医保定点药店。
5. **时效要求**：紧急用药选最近的门店仓 + 自配送。

**库存同步的实时性**：

- **门店仓 / 合作药店库存**：实时同步（API 调用 + Redis 缓存 1 分钟）。
- **城市仓库存**：准实时（每 5 分钟同步）。
- **区域仓库存**：准实时（每 15 分钟同步）。
- **兜底**：下单时直接查询源系统库存，避免缓存导致超卖。

| 医保身份核验 | 凭证 ID | 参保地 + 类型 + 余额 | 1s | 提示自费 |
| 医保目录查询 | 药品编码 | 医保类别 + 报销比例 + 限定范围 | 500ms | 缓存兜底 |
| 医保预结算 | 凭证 ID + 处方 | 医保金额 + 自付金额 | 3s | 提示自费 |
| 医保结算 | 凭证 ID + 处方 + 预结算结果 | 结算 ID + 票据号 | 5s | 释放库存，订单失败 |
| 医保结算凭证回传 | 结算 ID + 配送单号 + 签收单号 | 回传成功 | 5s | 重试 + 补报 |
| 医保拒付通知 | 拒付单号 + 拒付原因 | - | 异步 | 触发退款链路 |

**互联网+医保的关键约束**：

- **医保电子凭证强制**：互联网医院医保结算必须用医保电子凭证，不能用实体卡。
- **统筹账户 vs 个人账户**：门诊统筹、门诊慢病、门诊特殊病有不同的报销政策，需按参保类型匹配。
- **限定支付范围**：部分药品有"限定支付范围"（如限二线用药、限住院），不符合的不予支付。
- **医保目录版本**：国家医保目录每年调整，需动态加载最新版本。
- **医保接口限流**：医保平台接口 QPS 有限，需做"批量预结算 + 异步结算"。

**预结算 vs 结算的区别**：

| 维度 | 预结算 | 结算 |
|------|--------|------|
| 时机 | 处方流转选定药店后，患者确认支付前 | 患者确认支付后 |
| 作用 | 计算医保支付金额 + 自付金额，供患者确认 | 实际扣款，生成结算凭证 |
| 资金 | 不扣款 | 医保账户扣款 + 个人支付扣款 |
| 失败影响 | 可重新预结算 | 需走退款链路 |
| 时效 | 30 分钟内有效（避免目录/余额变化） | 不可撤销（除非退药） |

#### 2. 医保结算与药品配送的协同

**核心原则：先医保结算后配送**。

**为什么必须先结算后配送**：

1. **医保资金安全**：医保结算确认医保基金支付，未结算就配送可能造成医保基金损失（配送后患者不付款）。
2. **合规要求**：医保定点药店出库前必须完成医保结算，否则视为"自费销售"，无法享受医保报销。
3. **业务流程**：药店调剂出库前必须知道"医保支付多少、患者自付多少"，否则无法确定收款金额。
4. **退药复杂度**：先配送后结算的话，配送失败后退款链路复杂（医保未结算无需医保退款，但患者可能已自付）。

**协同链路**：

```
处方审核通过
   ↓
处方路由选定药店
   ↓
药店接单 + 库存预扣
   ↓
医保预结算（计算医保金额 + 自付金额）
   ↓
患者确认支付
   ↓
医保结算（医保账户扣款）+ 自费支付（第三方支付扣款）
   ↓
药店调剂出库
   ↓
配送
   ↓
签收
   ↓
医保结算凭证回传（含配送单号 + 签收单号）
```

**配送失败/退药的医保回退**：

| 场景 | 处理 |
|------|------|
| 医保结算成功 + 出库前取消 | 医保退款（全额）+ 自费退款（全额）+ 释放库存 |
| 医保结算成功 + 配送失败 | 药品回收 + 医保退款（全额）+ 自费退款（全额） |
| 医保结算成功 + 签收后退货 | 药品回收（需校验包装完整 + 批号效期）+ 医保退款（按退药比例）+ 自费退款（按退药比例） |
| 医保结算成功 + 患者拒收 | 药品回药店 + 医保退款（全额）+ 自费退款（全额） |

**"资金倒挂"风险**：

- 定义：医保结算成功但配送失败，医保已支付但患者未收到药。
- 风险：医保基金损失 + 患者投诉。
- 防范：
  - 配送失败后立即触发医保退款（同步退款 + 异步对账）。
  - 资金倒挂超过 24 小时未处理 -> 告警人工介入。
  - 日终对账校验"医保结算数 = 签收数 + 已退款数"，倒挂的进异常队列。

**退药的医保退款链路**：

```
患者申请退药
   ↓
药师审核（校验药品包装、批号、效期）
   ↓
药店回收药品（total_qty += N）
   ↓
医保退款 InsAdapt.refund(settlementId, 退药金额)
   ↓ 返回退款 ID + 退款金额
自费退款（第三方支付退款）
   ↓
退药完成
   ↓
医保退款凭证回传
   ↓
日终对账：退药数 = 医保退款数 = 自费退款数
```

#### 3. "四流合一"对账的工程实现

**对账口径**：

- **主键**：以 rxId（处方号）为主键做四流对账。
- **维度**：每日凌晨对账，按 rxId 维度校验四流是否齐全。
- **辅助键**：payId、shipId、settlementId、invoiceId 作为关联键。

**对账数据源**：

| 流 | 数据源 | 关键字段 |
|----|--------|---------|
| 信息流 | 处方表 prescription | rxId、orderId、status |
| 资金流 | 支付表 payment | payId、rxId、amount、payType |
| 物流 | 配送表 shipment + 签收表 sign_record | shipId、rxId、status、signTime |
| 票据流 | 医保结算表 insurance_settlement | settlementId、rxId、payId、invoiceId |

**对账规则**：

```
1. 处方维度对账：
   - 每张处方必须有 1 条支付记录、1 条配送记录、1 条签收记录、1 条医保结算记录
   - 缺一即异常

2. 金额对账：
   - 处方金额 = 支付金额 = 医保支付金额 + 自费支付金额
   - 金额不一致进异常队列

3. 时间对账：
   - 处方开立 < 支付 < 配送 < 签收 < 票据回传
   - 时间逆序进异常队列

4. 状态对账：
   - 处方状态 = SIGNED 时，必须有签收记录
   - 处方状态 = RETURNED 时，必须有退药记录 + 退款记录
   - 状态不匹配进异常队列
```

**异常类型与处理**：

| 异常类型 | 描述 | 处理 |
|---------|------|------|
| 有处方无支付 | 处方已开但未支付 | 24h 后自动作废处方，释放预扣库存 |
| 有支付无配送 | 支付成功但未配送 | 告警人工介入，可能配送系统故障 |
| 有配送无签收 | 配送中但未签收 | 48h 后告警，联系配送方 |
| 有签收无票据 | 签收但医保结算凭证未回传 | 重试回传 + 补报 |
| 四流不齐 | 任意流缺失 | 进异常队列，人工介入 |
| 金额不一致 | 四流金额不匹配 | 进异常队列，资金对账 |
| 时间逆序 | 时间戳逻辑错误 | 进异常队列，日志追溯 |

**对账的工程实现**：

```sql
-- 日终对账 SQL（简化版）
SELECT
    p.rx_id,
    p.order_id,
    p.status AS rx_status,
    pay.pay_id,
    pay.amount AS pay_amount,
    s.ship_id,
    s.status AS ship_status,
    sig.sign_time,
    ins.settlement_id,
    ins.insurance_amount,
    ins.self_amount
FROM prescription p
LEFT JOIN payment pay ON p.rx_id = pay.rx_id
LEFT JOIN shipment s ON p.rx_id = s.rx_id
LEFT JOIN sign_record sig ON s.ship_id = sig.ship_id
LEFT JOIN insurance_settlement ins ON p.rx_id = ins.rx_id
WHERE p.open_date = #{yesterday}
  AND (
    pay.pay_id IS NULL
    OR s.ship_id IS NULL
    OR sig.sign_time IS NULL
    OR ins.settlement_id IS NULL
    OR pay.amount != (ins.insurance_amount + ins.self_amount)
  );
```

**对账的兜底**：

- **T+1 对账**：每日凌晨对账前一日数据。
- **T+7 复核**：每周复核未结清的异常单。
- **月报**：每月生成对账月报，统计四流一致率、异常处理率。
- **监管追溯**：所有对账记录保留 ≥ 15 年。

#### 4. 医保拒付的处理

**医保拒付的常见原因**：

| 类型 | 描述 | 示例 |
|------|------|------|
| 处方不合规 | 处方违反医保规定 | 超量开药、超适应症开药 |
| 患者非本人 | 医保卡冒用 | 患者医保卡给家人开药 |
| 药品串换 | 实际售药与处方不符 | 处方开 A 药，实际给 B 药 |
| 二次转让 | 患者将医保药品转卖 | 套保行为 |
| 虚假处方 | 处方本身虚假 | 假冒医师开方 |
| 限定支付范围 | 不符合限定支付 | 门诊开限住院药品 |

**拒付的处理流程**：

```
医保局事后拒付（拒付通知）
   ↓
接收拒付单（拒付单号 + rxId + 拒付原因 + 拒付金额）
   ↓
1. 冻结该处方的退药/退款（避免重复退款）
2. 触发资金追回：
   - 医保基金部分：医保局直接扣减（已拒付）
   - 个人自付部分：从患者医保账户余额扣减 / 走线下追回
3. 标记处方为"已拒付"
4. 上报监管（互联网医院监管中心 + 医保监管）
5. 触发风控告警：
   - 该医师的开方行为审计
   - 该药店的售药行为审计
   - 该患者的医保使用行为审计
6. 资金倒挂处理：
   - 医保已拒付 + 患者已签收：医保基金损失，向患者追回
   - 医保已拒付 + 患者未签收：撤销配送，全额追回
```

**拒付后个人自付的转换**：

- **场景**：医保拒付，原本医保支付的部分需由患者自付。
- **处理**：
  - 系统自动生成"自付补缴单"，推送患者支付。
  - 患者支付后，处方状态从"已拒付"恢复为"已签收"（自费）。
  - 患者拒付补缴 -> 标记为"待追回"，进入风控黑名单。

**拒付的退款链路**：

```
拒付通知
   ↓
1. 已签收 + 患者配合补缴：患者自付补缴 -> 处方恢复
2. 已签收 + 患者拒补缴：进入追回流程（线下法务）
3. 未签收 + 拒付：撤销配送 -> 全额追回
4. 已退药 + 拒付：仅追回已退款部分
```

**风控反欺诈**：

- **医师层面**：开方量异常（某医师日均开方 100+）、药品集中（某医师 90% 开同一种药）。
- **药店层面**：销售异常（某药店日均销售 1000+ 盒同种药）、单品占比过高。
- **患者层面**：开药异常（某患者月开 30 天药量但 7 天后又开）、跨机构开药（多机构开同种药）。
- **处方层面**：处方金额异常（远超同类处方均价）、药品组合异常（常见套保组合）。

**风控规则引擎**：

```java
public class FraudDetectionEngine {
    public List<FraudAlert> detect(Prescription rx) {
        List<FraudAlert> alerts = new ArrayList<>();

        // 1. 医师层面：开方量异常
        if (doctorStats.dailyRxCount(rx.getDoctorId()) > 100) {
            alerts.add(new FraudAlert("DOCTOR_HIGH_VOLUME", rx.getDoctorId()));
        }

        // 2. 药店层面：单品占比过高
        if (pharmacyStats.singleDrugRatio(rx.getPharmacyId(), rx.getDrugCode()) > 0.5) {
            alerts.add(new FraudAlert("PHARMACY_SINGLE_DRUG_HIGH", rx.getPharmacyId()));
        }

        // 3. 患者层面：跨机构开同种药
        if (patientStats.crossOrgSameDrug(rx.getPatientId(), rx.getDrugCode(), 7)) {
            alerts.add(new FraudAlert("PATIENT_CROSS_ORG_DUPLICATE", rx.getPatientId()));
        }

        // 4. 处方层面：药品组合异常
        if (fraudRuleEngine.isSuspiciousCombo(rx.getDrugItems())) {
            alerts.add(new FraudAlert("SUSPICIOUS_DRUG_COMBO", rx.getRxId()));
        }

        return alerts;
    }
}
```

#### 5. 跨省医保结算的特殊性

**跨省医保直接结算的政策线**：

- **2017 年**：跨省异地就医住院直接结算全面联通。
- **2021 年**：跨省异地就医门诊直接结算在 5 省市试点（京津冀、长三角、西南 5 省）。
- **2024-2025 年**：跨省门诊直接结算扩展到 30+ 省份，但互联网医院跨省医保结算仍在试点。

**互联网医院跨省医保结算的可行性**：

| 维度 | 省内医保 | 跨省医保 |
|------|---------|---------|
| 政策支持 | 普遍支持 | 试点中，部分省份支持 |
| 接口对接 | 省级医保平台 | 国家医保平台 + 省级医保平台双对接 |
| 报销政策 | 按参保地政策 | 按参保地政策（就医地目录） |
| 结算时效 | 实时 | 实时（部分省份延迟） |
| 拒付风险 | 较低 | 较高（参保地监管） |
| 试点范围 | 全国 | 京津冀、长三角、西南 5 省 |

**互联网医院跨省医保的特殊场景**：

1. **参保地与就医地分离**：患者参保地在 A 省，互联网医院执业挂靠医院在 B 省，如何结算？
   - 当前主流：按互联网医院执业挂靠医院所在省结算（就医地政策）。
   - 部分试点：按参保地政策结算（需国家医保平台中转）。
2. **医保目录差异**：A 省医保目录含某药品，B 省不含。患者参保 A 省、互联网医院在 B 省。
   - 当前主流：按就医地（B 省）目录，B 省不含则自费。
3. **报销比例差异**：A 省报销 80%，B 省报销 70%。
   - 当前主流：按参保地（A 省）比例。

**跨省医保结算的接口差异**：

| 维度 | 省内 | 跨省 |
|------|------|------|
| 接口 | 省级医保平台 | 国家医保平台 + 省级医保平台 |
| 协议 | 省级 HTTPS + 双向 TLS | 国家级 + 省级双层 HTTPS + 双向 TLS |
| 数据格式 | 省级规范（基于国家规范扩展） | 国家规范 |
| 时效 | 实时 | 准实时（部分延迟到 T+1） |
| 异常处理 | 省内重试 | 跨省协调，时效长 |

**互联网医院跨省医保的架构**：

```
互联网医院
   ↓
医保路由层（按参保地路由）
   ├─> 参保地 = 执业挂靠地：省内医保接口
   └─> 参保地 ≠ 执业挂靠地：跨省医保接口（国家医保平台中转）
   ↓
医保结算
   ↓
结算凭证回传（双平台回传：参保地 + 就医地）
```

**互联网医院跨省医保的当前局限**：

- 仅试点省份支持，非试点省份患者只能自费。
- 报销政策复杂（参保地 vs 就医地），患者体验差。
- 跨省拒付风险高，互联网医院需承担资金倒挂风险。
- 当前主流互联网医院（京东健康、阿里健康）仅支持省内医保，跨省医保仍在试点。

### 与架构师水平的差距及补足方向

**差距1：医保电子凭证与医保接口的工程实战不足**

- **现状**：知道医保电子凭证和医保接口，但"医保电子凭证激活、身份核验、预结算、结算、凭证回传"全链路的工程实战（接口规范、失败兜底、限流应对）经验不足。
- **架构师水平**：能讲清医保电子凭证的密码学底层（国密 SM2/SM3）、医保接口的双向 TLS + 国密加密、医保接口限流应对（异步结算 + 批量结算）、医保目录版本兼容。
- **补足方向**：
  - 学习国家医保局《医疗保障信息平台编码标准》
  - 学习医保电子凭证的激活与认证流程
  - 学习医保接口的安全规范（双向 TLS + 国密 + 签名）

**差距2：四流合一的对账与异常处理深度不足**

- **现状**：知道四流合一概念，但日终对账的工程实现（对账规则、异常类型、异常处理、资金倒挂）深度不足。
- **架构师水平**：能设计完整的四流对账方案（处方维度 + 金额维度 + 时间维度 + 状态维度），能处理"医保已结算 + 配送失败"等资金倒挂边界 case。
- **补足方向**：
  - 学习电商行业的对账系统（如支付宝、美团的对账设计）
  - 学习支付行业的资金倒挂处理（如银联、网联的对账机制）
  - 学习医疗行业的处方流转对账案例（如京东健康、阿里健康）

**差距3：跨省医保结算的政策与工程实战不足**

- **现状**：知道跨省医保直接结算概念，但互联网医院跨省医保结算的政策、接口、报销规则差异经验不足。
- **架构师水平**：能讲清跨省医保结算的"参保地政策 vs 就医地目录"差异、国家医保平台中转机制、跨省拒付风险应对。
- **补足方向**：
  - 学习国家医保局《关于跨省异地就医直接结算服务的指导意见》
  - 调研 5 省市试点（京津冀、长三角、西南 5 省）的跨省医保结算实施情况
  - 学习互联网医院跨省医保试点的最新进展

## 题目三（深度题）：处方流转的合规、安全与药事服务深度

### 作答区

#### 1. 处方外流政策与"双轨制"

**处方外流政策的法律基础**：

| 政策 | 发布时间 | 核心内容 |
|------|---------|---------|
| 《关于进一步改革完善药品生产流通使用政策的若干意见》 | 2017 | 门诊患者自主选择在医疗机构或零售药店购药 |
| 《医疗机构处方审核规范》 | 2018 | 药师必须对处方进行审核，未审核不得调剂 |
| 《长期处方管理规范（试行）》 | 2021 | 长期处方（最长 12 周）规范，慢病患者可一次开多周用药 |
| 《关于推进电子处方流转服务的指导意见》 | 2023 | 推进省级电子处方流转服务建设 |
| 《药品追溯码体系建设要求》 | 2023 | 药品全流程追溯码采集与校验 |
| 《互联网诊疗监管细则（试行）》 | 2022 | 互联网医院处方监管细则 |

**电子处方中心 vs 处方流转平台**：

| 维度 | 电子处方中心（官方） | 处方流转平台（商业） |
|------|---------------------|---------------------|
| 建设主体 | 国家医保局 / 省级医保局 | 互联网医院 / 第三方平台 |
| 法律地位 | 官方枢纽 | 商业平台 |
| 处方来源 | 医保定点医院 | 互联网医院 |
| 流转范围 | 医保定点药店 | 合作药店 |
| 资金结算 | 医保直接结算 | 商业结算 + 医保对接 |
| 监管追溯 | 官方追溯 | 商业追溯 + 上报官方 |
| 上线进度 | 20+ 省份试点 | 主流互联网医院均有 |

**国家医保电子处方中心的核心能力**：

1. **统一接收**：医保定点医院所有处方统一上传到电子处方中心。
2. **统一流转**：患者凭医保电子凭证在任一医保定点药店调取处方。
3. **统一追溯**：处方从开立到签收的全链路追溯，医保局可一键查询。
4. **统一风控**：处方异常（超量、超适应症、串换）自动识别。
5. **统一对账**：处方数 = 医保结算数 = 配送数 = 签收数，医保局自动对账。

**对接策略**：

- **医保定点药店走官方**：处方上传省级电子处方中心，患者凭医保电子凭证在医保定点药店调取。
- **合作药店走自建**：处方流转平台自有路由，药店接单。
- **统一监管上报**：两条链路都需上报互联网诊疗监管中心 + 医保电子处方中心。
- **统一对账**：两条链路的处方都纳入日终对账。

#### 2. 药师审方的工程化

**法律依据**：《医疗机构处方审核规范》（2018）要求"药师必须对处方进行合法性、规范性、安全性审核，未审核不得调剂"。

**审方内容**：

| 维度 | 审核内容 | 示例 |
|------|---------|------|
| 合法性 | 医师资质、处方权限、特殊药品 | 麻醉药不得在线开 |
| 规范性 | 处方格式、药品名称、用法用量、签名 | 药品名称须用通用名 |
| 安全性 | 配伍禁忌、药物相互作用、剂量、过敏史、特殊人群 | 处方 A 药 + B 药有相互作用 |

**审方规则引擎设计**：

```java
public class RxReviewEngine {
    public ReviewResult review(Prescription rx) {
        ReviewResult result = new ReviewResult();

        // 1. 合法性审核
        // 1.1 医师资质
        if (!doctorCredentialService.isValid(rx.getDoctorId(), rx.getHospitalId(), rx.getDeptCode())) {
            result.addError("医师资质无效");
        }
        // 1.2 处方权限（特殊药品）
        for (DrugItem item : rx.getDrugItems()) {
            if (item.isSpecial() && !doctorCredentialService.hasSpecialRxPermission(rx.getDoctorId(), item.getDrugCategory())) {
                result.addError("医师无 " + item.getDrugName() + " 处方权");
            }
        }
        // 1.3 互联网医院禁止药品
        for (DrugItem item : rx.getDrugItems()) {
            if (!item.isInternetHospitalAllowed()) {
                result.addError(item.getDrugName() + " 禁止互联网医院开具");
            }
        }

        // 2. 规范性审核
        // 2.1 处方格式
        if (!rx.isValidFormat()) {
            result.addError("处方格式不规范");
        }
        // 2.2 药品名称（须用通用名）
        for (DrugItem item : rx.getDrugItems()) {
            if (!item.isGenericName()) {
                result.addError(item.getDrugName() + " 须使用通用名");
            }
        }
        // 2.3 用法用量
        for (DrugItem item : rx.getDrugItems()) {
            if (!medicationKnowledge.isValidUsage(item.getDrugCode(), item.getUsage(), item.getDosage())) {
                result.addError(item.getDrugName() + " 用法用量不规范");
            }
        }

        // 3. 安全性审核
        // 3.1 配伍禁忌（中药十八反十九畏）
        for (DrugPair pair : rx.getDrugPairs()) {
            if (incompatibilityTable.isIncompatible(pair.getDrug1(), pair.getDrug2())) {
                result.addError("配伍禁忌：" + pair.getDrug1() + " 与 " + pair.getDrug2());
            }
        }
        // 3.2 药物相互作用（西药）
        for (DrugPair pair : rx.getDrugPairs()) {
            InteractionLevel level = interactionTable.getInteraction(pair.getDrug1(), pair.getDrug2());
            if (level == InteractionLevel.CONTRAINDICATED) {
                result.addError("药物相互作用禁忌：" + pair.getDrug1() + " 与 " + pair.getDrug2());
            } else if (level == InteractionLevel.SEVERE) {
                result.addWarning("药物相互作用严重：" + pair.getDrug1() + " 与 " + pair.getDrug2());
            }
        }
        // 3.3 剂量上限
        for (DrugItem item : rx.getDrugItems()) {
            if (item.getDosage() > item.getMaxDosage()) {
                result.addError(item.getDrugName() + " 剂量超上限");
            }
        }
        // 3.4 过敏史
        Patient patient = patientService.get(rx.getPatientId());
        for (DrugItem item : rx.getDrugItems()) {
            if (patient.getAllergies().stream().anyMatch(a -> item.isAllergicTo(a))) {
                result.addError("患者对 " + item.getDrugName() + " 过敏");
            }
        }
        // 3.5 特殊人群
        if (patient.isPregnant()) {
            for (DrugItem item : rx.getDrugItems()) {
                if (item.isPregnancyContraindicated()) {
                    result.addError(item.getDrugName() + " 妊娠期禁用");
                }
            }
        }
        if (patient.isPediatric()) {
            for (DrugItem item : rx.getDrugItems()) {
                if (!item.isPediatricAllowed()) {
                    result.addError(item.getDrugName() + " 儿童禁用");
                }
            }
        }
        if (patient.isElderly()) {
            for (DrugItem item : rx.getDrugItems()) {
                if (item.getDosage() > item.getElderlyMaxDosage()) {
                    result.addError(item.getDrugName() + " 老年人剂量超上限");
                }
            }
        }

        return result;
    }
}
```

**审方 SLA 30 分钟的保障**：

1. **自动审方**：规则引擎自动审核 80% 处方（明显合规/明显不合规），秒级出结果。
2. **人工审方**：规则引擎判定为"需人工审核"的处方（约 20%），药师人工审核。
3. **药师排班**：7×24 药师值班，每班次至少 2 名药师（含 1 名主管药师）。
4. **超时告警**：25 分钟未审 -> 黄色告警；30 分钟未审 -> 红色告警 + 升级到主管药师。
5. **超时兜底**：30 分钟未审 -> 自动退方（不能"默认通过"，违反合规）。

**审方退回的处理**：

| 退回类型 | 描述 | 处理 |
|---------|------|------|
| 退回医师 | 处方有问题，医师修改后重开 | 退回医师工作台，医师修改 -> 重新签章 -> 重新审方 |
| 拒方 | 处方严重违规，不可修改 | 处方作废，记录医师违规行为，上报监管 |
| 转主管药师 | 药师无法判定（复杂用药） | 转主管药师审方 |

#### 3. 特殊药品的全程追溯

**特殊管理药品的"线上不可开"强制**：

| 类别 | 监管依据 | 工程实现 |
|------|---------|---------|
| 麻醉药品 | 《麻醉药品和精神药品管理条例》 | 药品库标记 `narcotic: true, internetHospitalAllowed: false`，开方时强制拦截 |
| 第一类精神药品 | 同上 | 同上 |
| 第二类精神药品 | 同上 | 部分省份可开（如地西泮），需特殊审批 |
| 医疗用毒性药品 | 《医疗用毒性药品管理办法》 | 同上 |
| 放射性药品 | 《放射性药品管理办法》 | 同上 |
| 易制毒药品 | 《易制毒化学品管理条例》 | 同上 |
| 终止妊娠药品 | 《母婴保健法》 | 同上 |

**三道防线**（与 Day01 一致）：

1. 医师选药时校验（前端拦截）
2. 处方提交时全量校验（后端拦截）
3. 药师审方时再次校验（人工 + 系统双校验）

**特药（特殊管理药品，如抗肿瘤药）的全程追溯**：

特药虽然可以在互联网医院开具，但需要全程冷链 + 批号 + 效期追溯。

```
1. 处方开立：rxId 关联药品追溯码（20 位）
   ↓
2. 药品出库：扫码记录批号、效期、出库时间、出库人
   ↓
3. 冷链配送：每 5 分钟回传温度，超温告警
   ↓
4. 患者签收：扫码确认收货，记录签收时间、签收人、开箱温度
   ↓
5. 用药反馈：患者用药后不良反应上报（24h 内）
   ↓
6. 全程追溯：药品追溯码（20 位）在每环节采集，医保局可一键查询
```

**药品追溯码体系**：

- **20 位追溯码**：国家药监局统一编码规则（前 8 位厂商 + 4 位药品 + 4 位规格 + 4 位序列号）。
- **采集时机**：生产出库、流通出库、零售出库、患者签收。
- **校验时机**：每个环节扫码校验上一环节的码，防伪防串货。
- **回传医保**：零售出库时回传药品追溯码到医保电子处方中心。

#### 4. 退药与退款的合规与工程实现

**法律依据**：《药品经营质量管理规范》（GSP）规定"药品一经售出，除质量原因外，不得退换"。

**退药条件**：

| 条件 | 是否可退 | 说明 |
|------|---------|------|
| 药品质量问题 | 可退 | 厂家召回 / 药店发现质量问题 |
| 药品未拆封 + 包装完整 | 可退 | 7 天内（参考电商法规） |
| 药品已拆封 | 不可退 | 影响二次销售 |
| 特殊管理药品 | 不可退 | 麻醉药、精神药等 |
| 冷链药品拆零 | 不可退 | 冷链断链后质量无法保证 |
| 近效期药品（效期 < 6 个月）| 不可退 | 即将过期 |
| 患者签收超过 7 天 | 不可退 | 超过退货期限 |

**退药流程**：

```
患者申请退药
   ↓
1. 系统校验退药条件（药品类型、签收时间、包装状态）
   ↓ 不符合条件
   拒绝退药，告知原因
   ↓ 符合条件
2. 药师审核退药申请
   - 校验药品包装完整性（患者上传包装照片）
   - 校验药品批号、效期（与出库时一致）
   ↓ 审核通过
3. 药品回收（配送员上门取药 / 患者到店退药）
   ↓
4. 药店验收
   - 校验药品包装、批号、效期、追溯码
   - 验收通过 -> 药品回库（total_qty += N）
   - 验收不通过 -> 退药失败，告知患者
   ↓
5. 退款
   - 医保退款 InsAdapt.refund(settlementId, 退药金额)
   - 自费退款（第三方支付退款）
   ↓
6. 退款凭证回传
   - 医保退款凭证回传到医保电子处方中心
   - 自费退款凭证回传到对账系统
   ↓
7. 处方状态置为"已退药"
   ↓
8. 监管上报（退药全流程上报）
```

**退款的资金流对账**：

| 流 | 退款流水号 | 关键字段 |
|----|----------|---------|
| 信息流 | rxId | rxId、退药数量、退药原因、退药时间 |
| 资金流 | refundPayId | refundPayId、rxId、原 payId、退款金额、退款方式 |
| 物流 | returnShipId | returnShipId、rxId、回收时间、验收结果 |
| 票据流 | refundInvoiceId | refundInvoiceId、原 settlementId、退款金额、退款凭证 |

**日终对账**：

- 退药数 = 医保退款数 = 自费退款数 = 票据回传数
- 退药金额 = 医保退款金额 + 自费退款金额
- 异常进异常队列，人工介入

**退药的合规要点**：

1. **药品回收验收**：必须校验包装、批号、效期、追溯码，防止"假退药"（患者退假药）。
2. **回库登记**：回库药品必须重新登记批号、效期，单独标记"退货药品"，不可与新货混放。
3. **再销售限制**：退货药品再销售时必须明示"退货药品"（部分省份要求）。
4. **监管上报**：退药全流程上报医保电子处方中心，防止"虚假退药"套保。

#### 5. 处方流转的"防欺诈"工程化

**四大欺诈类型**：

| 类型 | 描述 | 示例 |
|------|------|------|
| 套保 | 用医保卡开药后转卖套现 | 患者开高价药后转卖黑市 |
| 串换药品 | 处方开 A 药实际给 B 药 | 处方开阿胶，实际给保健品 |
| 虚假处方 | 假冒医师开方或伪造处方 | 假冒医师资质开方 |
| 二次转让 | 患者将医保药品转卖 | 患者开药后转卖给药贩 |

**风控规则识别**：

```java
public class FraudDetectionEngine {
    public FraudRisk assess(Prescription rx) {
        FraudRisk risk = new FraudRisk();

        // 1. 医师层面
        // 1.1 开方量异常（日均开方数远超同科室均值）
        if (doctorStats.dailyRxCount(rx.getDoctorId()) > doctorStats.deptAvgRxCount(rx.getDeptCode()) * 3) {
            risk.addRule("DOCTOR_VOLUME_ANOMALY", RiskLevel.HIGH);
        }
        // 1.2 药品集中度高（某医师 90% 开同一种药）
        if (doctorStats.singleDrugRatio(rx.getDoctorId(), rx.getDrugCode()) > 0.5) {
            risk.addRule("DOCTOR_DRUG_CONCENTRATION", RiskLevel.MEDIUM);
        }
        // 1.3 异常时段开方（凌晨 2-5 点开方）
        if (rx.getOpenTime().getHour() >= 2 && rx.getOpenTime().getHour() <= 5) {
            risk.addRule("DOCTOR_OFF_HOURS_RX", RiskLevel.LOW);
        }

        // 2. 药店层面
        // 2.1 单品销售占比过高
        if (pharmacyStats.singleDrugRatio(rx.getPharmacyId(), rx.getDrugCode()) > 0.5) {
            risk.addRule("PHARMACY_DRUG_CONCENTRATION", RiskLevel.MEDIUM);
        }
        // 2.2 单医师处方占比过高
        if (pharmacyStats.singleDoctorRatio(rx.getPharmacyId(), rx.getDoctorId()) > 0.5) {
            risk.addRule("PHARMACY_DOCTOR_CONCENTRATION", RiskLevel.MEDIUM);
        }
        // 2.3 销售量异常（日均销售量远超同规模药店）
        if (pharmacyStats.dailySales(rx.getPharmacyId()) > pharmacyStats.similarPharmaciesAvg() * 3) {
            risk.addRule("PHARMACY_VOLUME_ANOMALY", RiskLevel.HIGH);
        }

        // 3. 患者层面
        // 3.1 跨机构开同种药（7 天内多机构开同种药）
        if (patientStats.crossOrgSameDrug(rx.getPatientId(), rx.getDrugCode(), 7)) {
            risk.addRule("PATIENT_CROSS_ORG_DUPLICATE", RiskLevel.HIGH);
        }
        // 3.2 开药频率异常（月开药量超 30 天用量）
        if (patientStats.monthlyDrugQuantity(rx.getPatientId(), rx.getDrugCode()) > 30) {
            risk.addRule("PATIENT_OVERUSAGE", RiskLevel.MEDIUM);
        }
        // 3.3 异地开药（参保地 A，开方地 B）
        if (!patientStats.getInsuranceState(rx.getPatientId()).equals(rx.getHospitalState())) {
            risk.addRule("PATIENT_CROSS_STATE_RX", RiskLevel.LOW);
        }

        // 4. 处方层面
        // 4.1 金额异常（远超同类处方均价）
        if (rx.getAmount() > rxStats.avgAmountForSimilarRx(rx) * 3) {
            risk.addRule("RX_AMOUNT_ANOMALY", RiskLevel.MEDIUM);
        }
        // 4.2 药品组合异常（常见套保组合）
        if (fraudRuleEngine.isSuspiciousCombo(rx.getDrugItems())) {
            risk.addRule("SUSPICIOUS_DRUG_COMBO", RiskLevel.HIGH);
        }
        // 4.3 长期处方异常（长期处方开高价药）
        if (rx.isLongTermRx() && rx.getAmount() > 5000) {
            risk.addRule("LONG_TERM_RX_HIGH_AMOUNT", RiskLevel.MEDIUM);
        }

        return risk;
    }
}
```

**欺诈识别后的处置**：

| 风险等级 | 处置 |
|---------|------|
| LOW | 通过，记录日志，纳入风控画像 |
| MEDIUM | 人工审核，药师审方时重点核查 |
| HIGH | 拦截，转风控团队人工审核，24h 内出结果 |
| CRITICAL | 拦截 + 上报监管（医保监管 + 互联网诊疗监管） |

**可信审计的设计**：

1. **不可变审计表**（append-only）：
   - 所有处方流转的关键操作（开方、审方、流转、配送、签收、退药）都记录到不可变审计表。
   - 表设计：`audit_log`（id, op_type, rx_id, operator_id, op_time, op_data, prev_hash, curr_hash）
   - 哈希链：每条记录的 `curr_hash = SHA256(prev_hash + op_data)`，防篡改。
2. **操作日志**：
   - 所有用户操作（登录、查询、修改、删除）记录到操作日志。
   - 日志保留 ≥ 15 年。
3. **数据变更日志**：
   - 所有数据变更（处方状态变更、库存变更、订单状态变更）记录变更前后值。
   - 用于追溯数据修改历史。
4. **监管审计接口**：
   - 医保局、卫健委可通过审计接口一键查询处方全流程。
   - 接口权限：仅监管审计岗可访问，所有访问记录留痕。

**哈希链示意**：

```
record 1: op_data=... prev_hash=0 curr_hash=SHA256("0" + op_data_1)
record 2: op_data=... prev_hash=curr_hash_1 curr_hash=SHA256(curr_hash_1 + op_data_2)
record 3: op_data=... prev_hash=curr_hash_2 curr_hash=SHA256(curr_hash_2 + op_data_3)
...

校验：
- 重新计算每条记录的 curr_hash，与存储的 curr_hash 比对
- 任一记录被篡改 -> 后续所有记录的 curr_hash 都不匹配
- 监管审计时可一键校验全链路完整性
```
