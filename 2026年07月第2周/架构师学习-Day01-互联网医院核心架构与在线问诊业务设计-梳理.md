## 一、互联网医院在高可用架构中的定位

### 1.1 核心一句话

> **互联网医院 = 合规监管 + 医师资源 + 三流合一 + 数据互通的医疗业务系统**
>
> 与电商、IM、支付系统的根本差异：强合规监管（仅复诊、麻醉精神药品禁开、24h 上报监管）+ 医师资源稀缺（接诊 SLA 30 分钟）+ 三流合一（信息流/资金流/物流/票据流可追溯）+ 数据互通壁垒（与实体医院 HIS/EMR 双向同步）。

### 1.2 互联网医院的四大特征

| 特征 | 含义 | 架构影响 |
| --- | --- | --- |
| 强合规监管 | 仅复诊不得首诊、麻醉精神药品不得在线开、24h 内上报监管 | 复诊认定前置、处方权限三道防线、监管上报通道 |
| 医师资源稀缺 | 医师非 7×24 在线，接诊 SLA 30 分钟 | 异步问诊、消息可靠投递、医师排班/号源管理 |
| 三流合一 | 信息流/资金流/物流/票据流必须可追溯对账 | 四类流水号设计、日终对账、异常队列 |
| 数据互通壁垒 | 互联网医院 vs 实体医院 HIS/EMR 双向同步 | FHIR 接口、EMPI、术语映射、数据归属分层 |

### 1.3 互联网医院的多层架构

```
┌──────────────────────────────────────────────────────────────────┐
│  患者端（APP/H5/小程序）+ 医师端（PC/PAD）+ 药师端 + 运营后台    │
└──────────────────────────────────────────────────────────────────┘
                 │ HTTPS / 网关
                 ▼
┌──────────────────────────────────────────────────────────────────┐
│  接入层：API 网关（鉴权、限流、灰度、医院路由）                   │
│        + 长连接网关（WebSocket/IM、视频信令）                     │
└──────────────────────────────────────────────────────────────────┘
                 │
                 ▼
┌──────────────────────────────────────────────────────────────────┐
│  业务层：在线问诊 Consult、电子处方 eRx、药品配送 DrugShip        │
│          检查检验 ExamOrder、慢病管理 CDM、健康档案 EHR           │
│          医保对接 Insurance                                       │
└──────────────────────────────────────────────────────────────────┘
                 │                              │
                 ▼                              ▼
┌──────────────────────────────────┐  ┌──────────────────────────┐
│  协同样层：医患 IM、视频通话     │  │  集成层：HIS/LIS/药企    │

                 ▼
```

### 1.4 互联网医院的两种形态

| 形态 | 实体医院互联网化 | 第三方平台型 |
| --- | --- | --- |
| 典型代表 | 北京协和、上海华山 | 微医、京东健康、阿里健康、平安好医生 |
| 执业挂靠 | 本院医师 | 多家合作实体医院医师 |
| 监管上报主体 | 实体医院 | 平台 + 合作医院分别上报 |
| 医保结算主体 | 实体医院 | 合作医院 |
| 处方签章主体 | 实体医院 CA | 合作医院 CA |
| EMR 归属 | 实体医院 EMR | 实体医院 EMR（平台代理同步） |
| 数据隔离 | 单医院内 | 多医院三级隔离（医院 + 科室 + 医师） |

## 三、同步与异步链路

### 3.1 同步链路（患者前台，RT < 1s）

```
患者发起图文问诊
1. 创建问诊订单（Consult 服务）
2. 校验医师资质（DoctorCredential 服务）
3. 校验复诊认定（EMR 调阅服务，6 个月内本科室线下就诊记录）
4. 锁定号源（Schedule 服务）
5. 拉起 IM 会话（IM 服务）
6. 返回问诊订单号 + IM 会话 ID
```

### 3.2 异步链路（问诊结束触发，秒 ~ 分钟级）

```
医师点击"结束问诊"
1. 问诊订单状态置为"已结束"
2. 触发"开方"事件
   ↓ MQ 异步
3. 处方进入"待审核"队列
4. 药师 30 分钟内审方
   ↓ 审核通过
5. 处方流转至药品配送模块
6. 处方同步至医保预结算
7. 处方上报监管
   ↓
8. 药企/药店接单配送
9. 配送状态回流（已发货 -> 配送中 -> 已签收）
   ↓
10. 触发慢病随访计划
11. 触发公卫上报（如果是公卫体检延伸场景）
```

### 3.3 为什么必须异步

| 步骤 | SLA | 同步等待问题 |
| --- | --- | --- |
| 医师接诊 | 30 分钟 | 患者前台等 30 分钟，体验崩坏 |
| 药师审方 | 30 分钟 | 医师前台卡住 |
| 药品配送 | 数小时~数天 | 跨城配送无法同步等待 |
| 监管上报 | 24 小时 | 监管接口限流、抖动 |
| 失败补偿 | 不定 | 同步链路无法承载独立补偿 |

实现：MQ（RocketMQ/Kafka）+ 状态机 + 定时对账 + 入站消息幂等（处方 ID + 步骤 ID）。

## 四、问诊订单 vs 诊疗事件

### 4.1 拆分必要性

| 维度 | 问诊订单 ConsultOrder | 诊疗事件 ConsultEvent |
| --- | --- | --- |
| 关注点 | 交易（金额、状态、退款） | 临床（主诉、诊断、处方） |
| 监管要求 | 保存 5 年（电商交易存档） | 保存 ≥ 15 年（医疗病历） |
| 数据归属 | 互联网医院平台 | 实体医院 EMR |
| 隐私等级 | 财务/运营可见 | 仅医师/药师/患者本人 |
| 数量关系 | 一对一 | 一次问诊可能多事件 |

### 4.2 图文问诊 vs 视频问诊

| 维度 | 图文问诊 | 视频问诊 |
| --- | --- | --- |
| 时效 | 异步，可中断 | 同步，不可中断 |
| 状态机 | 待支付 -> 待接诊 -> 问诊中 -> 已结束 | 待支付 -> 预约 -> 待呼叫 -> 视频中 -> 已结束 |
| 计费 | 按订单固定费用 | 按时长计费 |
| IM 会话 | 必须 | 可选 |
| 录制存档 | 不需要 | 必须（≥ 15 年，加水印 + SHA256 上链） |
| 接诊 SLA | 30 分钟内首次回复 | 准时呼叫（预约 ± 5 分钟） |
| 退款规则 | 未接诊全额退款 | 未呼叫全额退款，呼叫后按分钟计 |

## 五、三级数据隔离（医院 + 科室 + 医师）

### 5.1 必须按实体医院强隔离的数据

- **电子病历调阅**：FHIR API 按 hospitalId 租户过滤
- **处方签章**：CA 证书由本医院医师持有，处方归属本医院
- **诊疗事件**：归属实体医院 EMR，平台代理存储 + 同步回写
- **HIS 对接凭证**：每家医院 appKey/appSecret 独立
- **监管上报归属**：每家医院独立上报（用本医院执业许可证编码）

### 5.2 可以平台共享的数据

- **医师资质库**：核验通过后平台共享（多点执业）
- **药品库**：国家医保药品目录全国统一
- **患者主索引**：身份证 + 手机号注册，授权调阅需独立授权
- **术语映射**：ICD-10/11、SNOMED CT、ATC 全局共享

### 5.3 分片键选择

| 表 | 分片键 | 理由 |
| --- | --- | --- |
| consult_order | hospitalId + patientId | 按医院分库，按患者分表 |
| consult_event | hospitalId + orderId | 与订单同库 |
| prescription | hospitalId + patientId | 与订单同库（处方 join 订单） |
| drug_shipment | patientId | 按患者分库（跨医院配送少见） |
| doctor_credential | doctorId | 单库，平台共享 |

## 六、医师资质核验前置

### 6.1 为什么必须前置

- **监管红线**：未取得执业证书不得执业，违规开方吊销许可证
- **多点执业合规**：医师必须在执业注册医院开展互联网诊疗
- **处方签章依赖**：CA 证书绑定执业证号
- **科室匹配**：医师只能执业注册范围内的科室

### 6.2 医师入驻流程

```
1. 医师提交身份证、医师资格证、医师执业证、职称证、学历证
2. 平台调用国家卫健委"医师执业注册信息查询"接口
   输入：医师姓名 + 身份证号
   输出：执业证书编号、执业范围、执业地点、多点执业记录
3. 平台核验提交的执业证与国家接口数据一致
4. 平台调用 CA 机构（CFCA/北京 CA）签发医师电子签名证书
5. 平台分配"执业挂靠关系"（医师 -> 医院科室）
6. 医师通过执业校验后才能接诊
```

### 6.3 多点执业表达

```json
{
  "doctorId": "D20260001",
  "credentials": {
    "medicalLicense": "11010120210001",
    "practiceLicense": "11010120210001",
    "practiceScope": "内科专业"
  },
  "practiceSites": [
    {"hospitalId": "H001", "deptCode": "CARD", "isPrimary": true},
    {"hospitalId": "H002", "deptCode": "CARD", "isPrimary": false}
  ]
}
```

每次开方使用"当前执业挂靠医院"的 CA 证书，处方归属当前医院。

## 七、图文问诊消息可靠投递

### 7.1 完整链路

```
患者端 -> 服务端 -> MQ -> 推送服务 -> 医师长连接 -> 医师端 ACK -> 状态更新
                              ↓
                       离线消息入未读队列 -> APNS/FCM 推送
```

### 7.2 可靠性保障

| 步骤 | 风险 | 保障 |
| --- | --- | --- |
| 患者端 -> 服务端 | 网络抖动 | 客户端本地消息表 + 重试 + 客户端 ID 去重 |
| 服务端落库 | DB 写入失败 | 主从同步 + 半同步复制 |
| 服务端 -> MQ | 投递失败 | 同步发送 + publisher confirm |
| MQ -> 推送服务 | 消费失败 | 重试 N 次 + 死信队列 + msgId 幂等 |
| 推送 -> 长连接 | 长连接断开 | 离线消息入库 + 上线拉取 + 多端推送 |
| 医师端 ACK | ACK 丢失 | 定时扫描未 ACK 二次推送 |

### 7.3 消息三状态

| 状态 | 实现 |
| --- | --- |
| 已发送 sent | 服务端落库 + 客户端 ACK |
| 已送达 delivered | 长连接 ACK + 状态机更新 |
| 已读 read | 医师端"已读"上报 + 批量更新 |

### 7.4 幂等与顺序

- 客户端 msgId（UUID v7 + 患者ID + 时间戳），Redis SETNX 幂等，TTL 24h
- MQ 消费幂等：msgId 去重 + DB 唯一索引（msgId + orderId）
- 消息顺序：单问诊单队列（MQ 分区键 = orderId），seq 问诊内递增

## 八、视频问诊信令与媒体流

### 8.1 协议分层

| 层 | 协议 | 用途 |
| --- | --- | --- |
| 信令层 | WebSocket + 自定义 JSON（或 SIP） | 呼叫邀请/接听/拒绝/挂断/SDP 协商 |
| 媒体层 | WebRTC（SRTP）+ UDP | 音视频实时传输 |
| 媒体转发 | SFU（Selective Forwarding Unit） | 多方通话、媒体路由 |
| NAT 穿透 | STUN/TURN (ICE) | 内网穿透、中继 |
| 录制存档 | MediaServer 录制 + MP4 落盘 | 合规存档 |

### 8.2 信令与媒体流分离的原因

- **可靠 vs 实时**：信令 TCP，媒体 UDP
- **网络路径不同**：信令 443，媒体 3478/5349
- **故障域隔离**：信令故障不影响进行中的视频
- **扩展性不同**：信令无状态可水平扩展，SFU 单机有上限

### 8.3 NAT 穿透在医疗场景的特殊性

1. 医院内网严格（HIS/EMR 物理隔离），STUN 直连成功率低，必须部署 TURN 中继
2. 4G/5G 切换需 ICE 重协商支持网络切换不中断
3. 等保三级要求 SRTP + DTLS 端到端加密，TURN 不解密媒体流
4. 录制加水印 + SHA256 上链 + 保存 ≥ 15 年

### 8.4 录制存档合规

```
1. SFU 服务端录制（不依赖客户端，防篡改）
2. 加水印：医师姓名 + 患者姓名 + 身份证后4位 + 时间戳
3. SHA256 hash 上链（区块链/可信日志）
4. AES-256 加密，密钥由 KMS 管理
5. 元数据：orderId、doctorId、patientId、startTime、endTime、duration、hash
6. 上传 OSS，权限：仅监管审计岗可读
7. 保存期限：≥ 15 年
```

## 九、问诊会话状态机

### 9.1 完整状态机

```
            ┌─────────────┐
            │  待支付     │
            │  PENDING_PAY│  超时 15 分钟未支付 -> 已关闭
            └──────┬──────┘
                   │ 支付成功
                   ▼
            ┌─────────────┐
            │  待接诊     │  超时 30 分钟未接诊 -> 已退款
            │  PENDING    │  患者取消 -> 已退款（扣 5% 手续费）
            └──────┬──────┘
                   │ 医师接诊
                   ▼
            ┌─────────────┐
            │  问诊中     │  超时 24h（图文）/ 30 分钟（视频） -> 已结束
            │  CONSULTING │  医师离线超 5 分钟 -> 系统强制结束
            └──────┬──────┘
                   │ 医师/患者/系统结束
                   ▼
            ┌─────────────┐
            │  已结束     │
            │  FINISHED   │
            └─────────────┘
```

### 9.2 状态机实现

- **方案 A：状态模式 + 事件驱动**（推荐）：状态流转逻辑集中、易测试
- **方案 B：DB 状态字段 + 定时扫描**：实现简单、逻辑分散易出 bug
- **医疗场景推荐：A + 定时扫描兜底**
  - 主流程状态模式（接诊、结束、退款）
  - 定时扫描兜底超时（待接诊 30 分钟、问诊中 24h）
  - 状态变更必须走"事件 -> 状态机 -> 落库"链路，禁止直接 update status

## 十、长连接方案选型

### 10.1 自研 Netty + WebSocket vs 第三方 IM SDK

| 维度 | 自研 | 第三方 SDK |
| --- | --- | --- |
| 开发成本 | 高 | 低 |
| 定制能力 | 强（医疗合规、审计、存档） | 弱 |
| 运维成本 | 高 | 低 |
| 数据归属 | 自有数据库，合规可控 | 第三方，出境/合规风险 |
| 单价成本 | 固定人力 | 按 DAU/MAU/消息量计费 |
| 医疗合规 | 可定制满足等保三级 | 部分厂商不满足 |

### 10.2 医疗场景自研长连接的特殊问题

1. **多端同步**：PC + PAD + 手机同时登录，消息多端推送
2. **合规存档**：所有医患消息存档 ≥ 15 年，撤回消息保留审计
3. **多端登录互斥**：同类型端只能一处登录（如 PC 只能一处），不同类型可同时
4. **消息撤回审计**：24h 内可撤回，撤回消息仍保留审计
5. **敏感词过滤**：违规广告、辱骂命中后人工审核
6. **会话隔离**：不同问诊订单严格隔离，禁止跨订单推送

## 十一、离线消息与未读消息

### 11.1 医师离线时患者消息处理

1. 消息照常落库（医师离线不影响患者发送）
2. 推送服务查医师在线状态，离线转 APNS/FCM/厂商通道
3. 消息进入未读队列（Redis hash: `unread:{doctorId}:{orderId}` -> count）

### 11.2 医师次日登录拉取未读

```
1. 长连接建立，鉴权通过
2. 客户端调用 getUnreadCount()
3. 服务端返回：每订单未读数 + 总未读数
4. 医师点击订单进入会话
5. 客户端调用 getMessages(orderId, lastMsgId, pageSize)
6. 服务端分页返回 + 该订单未读数清零
7. 推送"已读"回执给患者端
```

### 11.3 多端未读数一致

- **服务端为权威源**：未读数 Redis hash，多端拉取同一份数据
- **多端"已读"同步**：PC 端读 -> 服务端清零 -> 推送"未读数变更"给 PAD/手机
- **会话级未读**：`unread:{userId}:{orderId}` -> count
- **总未读**：`unread:{userId}:total` -> count

## 十三、处方流转的"四流合一"

### 13.1 四类流水号

| 流 | 流水号 | 关键字段 |
| --- | --- | --- |
| 信息流 | rxId（处方号） | rxId、orderId、doctorId、patientId、hospitalId、drugCode、quantity、用法用量、签章 |
| 资金流 | payId（支付号） | payId、rxId、amount、payType（自费/医保）、payTime、payChannel |
| 物流 | shipId（配送号） | shipId、rxId、pharmacyId、courierId、shipStatus、deliveryTime、signTime |
| 票据流 | invoiceId（票据号） | invoiceId、rxId、payId、insuranceSettlementId、invoiceType |

### 13.2 处方全流程追溯

```
开立 rx_open         DRAFT
   ↓ 医师电子签名
审核 rx_review        DRAFT -> REVIEWED / REJECTED
   ↓ 药师电子签名
流转 rx_transfer      REVIEWED -> TRANSFERRED
   ↓ 流转日志
配送 rx_ship          TRANSFERRED -> SHIPPED -> DELIVERED
   ↓ 配送轨迹
签收 rx_sign          DELIVERED -> SIGNED
   ↓ 签收照片 + 地理位置
退药 rx_return        SIGNED -> RETURNED
   ↓ 退药日志 + 退款日志
```

### 13.3 对账机制

- **日终对账**：每天凌晨校验四流处方数 = 支付数 = 配送数 = 票据数
- **异常告警**：四流不齐进入"异常队列"，人工介入
- **监管追溯**：监管平台通过 rxId 一键追溯全流程

## 十四、医师电子签名与处方签章

### 14.1 法律效力

- 《电子签名法》（2005 实施，2019 修订）：可靠电子签名与手写签名同等法律效力
- 《卫生系统电子认证服务管理办法》（2010）：卫生健康领域电子认证规范
- 《互联网诊疗监管细则》（2022）：处方需医师电子签名 + CA 证书认证

### 14.2 CA 证书流程

```
颁发：医师提交证件 -> 平台核验 -> CA 机构（CFCA/北京 CA）颁发
续期：到期前 30 天提醒 -> 重新核验 -> 颁发新证书
吊销：医师离职/违规 -> 申请吊销 -> 加入 CRL（吊销列表）-> 后续签名无效
```

### 14.3 处方签章防篡改

```
1. 处方 JSON 序列化
2. SHA256 计算
3. 医师私钥加密 -> 签名值
4. 调用 TSA 获取时间戳 token
5. 处方 + 签名值 + 时间戳 + CA 证书 -> 落库

验证：
1. 处方 JSON -> SHA256
2. CA 证书公钥解密签名值 -> SHA256
3. 比对两个 SHA256
4. 校验时间戳（签名时间在证书有效期内）
5. 校验 CA 证书未在 CRL
```

### 14.4 医师离职后已签处方

- 已签处方不可撤销（法律效力持续）
- 离职后吊销证书（无法再开新方）
- 历史处方继续有效
- 病历归档至实体医院 EMR

- JSON（部分历史接口仍用 XML）

```
4. 次日批量补报：每日凌晨补报前一天失败数据
5. 监管追溯：所有上报记录保留 ≥ 15 年
```
