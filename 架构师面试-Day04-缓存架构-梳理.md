# Day 4：缓存架构 — 架构师视角梳理

## 问题全景

缓存是高并发系统的第一道防线，但每加一层缓存就多一层一致性代价。架构师的核心判断：**缓存方案的选择不是"快不快"，而是"不一致的代价你付不付得起"。**

---

## 一、缓存一致性 — 延迟双删不是银弹，Canal 才是兜底

### 先删缓存 vs 先更新DB

你的回答是"先更新DB再删缓存 + 延迟双删"，方向对，但要分清层次。

**先删缓存的问题——并发读回写：**

```
线程A: DEL cache
线程B: READ cache → miss → READ db(旧值) → SET cache(旧值)  ← 旧值回写
线程A: UPDATE db
→ 缓存是旧值，DB是新值，直到缓存过期才一致
```

**先更新DB再删缓存的问题——极小概率不一致：**

```
线程A: READ cache → miss → READ db(旧值)
线程B: UPDATE db → DEL cache
线程A: SET cache(旧值)
→ 不一致窗口极小（读DB+SET 必须比 UPDATE+DEL 更慢才触发）
```

**结论：先更新DB再删缓存是主流选择。** 并发回写是大概率事件，读比写慢是极小概率事件。

### 延迟双删——能用但不精确

```
更新DB → 删缓存 → 延迟N毫秒 → 再删缓存
```

问题在于"延迟多久"：太短拦不住回写，太长不一致窗口大。延迟时间取决于并发读请求的耗时，这个值在高并发下波动很大。

**架构师视角：** 延迟双删是权宜之计，不是最终方案。面试可以说，但不要当成首选。

### 方案的层次——你把三层混在一起了

| 层次 | 方案 | 可靠性 | 复杂度 | 定位 |
|------|------|--------|--------|------|
| 基础 | 先更新DB再删缓存 + 短TTL | 一般 | 最低 | 日常方案 |
| 增强 | 延迟双删 | 较好 | 低 | 补充手段 |
| 可靠 | Canal监听Binlog异步删 | 高 | 中 | 最终一致性保障 |
| 兜底 | 删缓存失败 → MQ重试 | 高 | 中 | 失败补偿 |

Canal 不是延迟双删的"延迟实现"，是**独立的可靠性保障**。不管你用什么策略删缓存，Binlog 监听都能兜住"删失败"的情况。

**面试答法：** "先更新DB再删缓存，不一致窗口最小。删缓存失败靠 Canal 监听 Binlog 异步补偿兜底。延迟双删也能用但延迟时间不精确。短TTL是最后一道安全网。"

## 二、穿透/击穿/雪崩 — 三个概念分清，解法更要分清场景

### 你对概念的理解是对的

| | 穿透 | 击穿 | 雪崩 |
|---|------|------|------|
| 本质 | 数据不存在 | 热点Key过期 | 大量Key同时过期 |
| 触发 | 恶意请求/bug | 爆款商品缓存到期 | 批量缓存同时失效 |
| DB中有数据吗 | 没有 | 有 | 有 |

### 击穿——锁住之后，没抢到锁的请求怎么办？

你说了用锁，但只说了一半：

```
缓存miss → SET NX 抢锁
  ├─ 抢到 → 查DB → 写缓存 → 释放锁
  └─ 没抢到 → ？    ← 你没说这里
```

两种策略的选择才是架构师要说的：

| 策略 | 没抢到锁怎么办 | 适用场景 |
|------|---------------|---------|
| 互斥锁 | 等待重试，拿到新数据 | 一致性要求高 |
| 逻辑过期 | 返回旧数据，后台异步更新 | 可用性要求高 |

**逻辑过期的精妙之处：** 缓存不设 TTL，存一个逻辑过期时间。读到过期数据不阻塞，异步线程去更新，当前请求返回旧数据。代价是短暂返回旧数据，但系统永远不会被阻塞。

```
一致性要求高 → 互斥锁（等就等，必须拿新数据）
可用性要求高 → 逻辑过期（旧数据先顶着，后台更新）
```

### 雪崩——你漏了熔断降级

TTL随机化和锁回补都是**预防**，但DB已经打满的时候需要**保命**：

```
DB连接池耗尽 → 熔断 → 返回兜底数据/降级页面
→ 先活下来，再查原因
```

架构师优先级：**保系统不死 > 数据最新 > 用户体验完美**

### 穿透——缺了最前置的防线

缓存空值和布隆过滤器都是在缓存层做的。架构师应该往前看一步：

```
层1：入口校验 — 非法请求直接拒（成本最低）
层2：布隆过滤器 — 不存在的Key拦截（省内存）
层3：缓存空值 — 兜底（短TTL，防换ID攻击）
```

入口校验最便宜——商品ID=999999999，格式/范围校验直接拒绝，连Redis都不用查。

### Flink 检测热点的归属

你提到雪崩场景用 Flink 检测突发热点，思路对但归属要分清：这是**热点Key问题**的解法，不是雪崩的核心解法。雪崩的核心是批量Key同时过期，TTL加随机值就解决了。Flink 是应对"冷数据突然变热"，属于热点Key范畴。

## 三、热点Key — 本地缓存才是第一优先级

### 你说的都对，但漏了最有效的一招

你提到了预热和Key拆分，但**本地缓存**应该是第一道防线：

```
QPS 10万的请求
  → 本地缓存拦截 90% → 只剩 1万到Redis
  → Redis 单节点轻松扛住
```

Caffeine 设 5s 过期，不经过网络，QPS 上限是应用本身的上限，比任何 Redis 方案快一个数量级。

### Key拆分——读分片，写全量

你提到了Key拆分，面试追问"怎么拆"时要说细：

```
product:123 → product:123:1 ~ product:123:9

读：random(1,9) 选一个读
写：9个Key全部更新

效果：1个Key 10万QPS → 9个Key 每个~1.1万QPS
```

代价：写操作从改1个Key变成改9个，写放大。只适合读远大于写的场景。

### 你问的 Nacos Distro 对比 — 很好的架构思维

这个问题说明你在用对比思维理解方案，方向完全正确。

| | Nacos Distro | 热点Key拆分 |
|---|---|---|
| 瓶颈 | 写（注册/注销） | 读（查询） |
| 设计 | 写分摊到不同节点，读本地化 | 读分摊到不同Key，写放大 |
| 数据关系 | 不同节点负责不同数据 | 不同Key存的是同一份数据副本 |
| 一致性 | 异步同步，最终一致 | 写时全量更新，强一致 |

**本质：** Nacos 的瓶颈是写，热点Key的瓶颈是读，所以方案是反过来的。架构师判断设计方向，取决于瓶颈在哪。

### 完整的优先级排序

```
层1：本地缓存 → 拦截大部分流量，成本最低
层2：预热     → 可预判的热点提前加载
层3：Key拆分  → 单Key QPS 极高时分散流量
层4：读写分离 → 从节点分担读压力
层5：Flink检测 → 应对突发热点
```

## 四、多级缓存 — 不一致的根源是"删不到别人的本地缓存"

### 核心难点

DB+Redis 的不一致靠 Canal/MQ 兜底。但本地缓存的不一致不一样——你删了自己节点的本地缓存和 Redis，**别的节点的本地缓存你删不到**：

```
节点A: 更新DB → 删Redis → 删本地缓存 ✓
节点B: 本地缓存还是旧值 → 返回旧数据
```

### 解法

| 方案 | 做法 | 不一致窗口 | 复杂度 |
|------|------|-----------|--------|
| 短TTL | 本地缓存设 5~10s 过期 | 5~10s | 最低 |
| MQ广播 | 更新时发MQ，所有节点消费后删本地缓存 | 毫秒级 | 中 |
| Redis PubSub | 更新时发通知，各节点收到后删本地缓存 | 毫秒级 | 中 |

生产中最常用的是短TTL，简单够用。MQ广播实时性好但增加复杂度——每加一个缓存Key都要确保广播覆盖到。

### 什么场景用本地缓存

```
能用的场景：
  - 字典/配置   → 几乎不变，TTL可以很长
  - 商品详情   → 读多写少，容忍5s旧数据
  - 热点Key    → 拦截流量，保护Redis

不能用的场景：
  - 实时库存   → 必须精确，不能用本地缓存
  - 用户session → 多节点共享，本地缓存不一致
  - 订单状态   → 用户对状态变化敏感
```

**业务关联：** 咖啡机商品详情页可以用本地缓存（菜单几小时才变一次），钱包余额绝对不能用（钱的事不能有延迟）。

**判断原则：一致性要求高的只用Redis，容忍秒级不一致的加本地缓存。**

## 架构师总结

| 层次 | 核心判断 |
|------|---------|
| 一致性策略 | 先更新DB再删缓存；延迟双删不精确；Canal是可靠兜底 |
| 穿透 | 入口校验 > 布隆过滤器 > 缓存空值，三层递进 |
| 击穿 | 一致性高用互斥锁，可用性高用逻辑过期 |
| 雪崩 | TTL随机化预防 + 熔断降级保命 |
| 热点Key | 本地缓存优先 > 预热 > Key拆分 > 读写分离 > 流计算检测 |
| 多级缓存 | 不一致根源是"删不到别人的本地缓存"；短TTL是最实用的解法 |
| 本地缓存 | 容忍秒级不一致 → 加；一致性要求高 → 只用Redis |
| 方案设计方向 | 瓶颈在读 → 读分片写全量；瓶颈在写 → 写分片读全量 |

**一句话：缓存的每一层加速都以一致性为代价，架构师的职责是认清业务能容忍多少不一致，选代价最小的方案，然后用TTL和异步补偿兜住那不一致的窗口。**

```
  → Day 2 的幂等三件套（唯一键 + 状态机 + 乐观锁）

  → 幂等和校验是分布式系统最后一道防线
```
