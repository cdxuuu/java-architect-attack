# 2026年06月第2周：Redis 专题能力差距梳理

## 一、缓存异常治理表达仍偏定义化

> Day3发现

### 当前表现

在回答缓存穿透、击穿、雪崩时，能够说出基本定义：

```text
击穿：Redis缓存过期导致请求到了DB
雪崩：批量Key同时过期导致批量请求到DB
穿透：Redis和DB里都没有相应的数据
```

也能抓住核心区别：

```text
1. 数据库里有没有对应数据
2. Redis过期的是单个热点Key还是大量Key
```

这说明基础概念已经建立。

### 与架构师水平的差距

架构师面试不能只停留在定义层面，还需要能推演完整故障链路：

```text
缓存失效
  ↓
Redis miss
  ↓
并发请求回源DB
  ↓
DB连接池打满
  ↓
慢查询堆积
  ↓
应用线程阻塞
  ↓
接口超时
  ↓
上游重试
  ↓
系统雪崩
```

当前回答中对“流量如何从 Redis miss 放大到 DB 故障”的链路描述不足，容易被面试官认为只是背概念。

### 补足方向

后续回答缓存类问题时，统一使用三段式：

```text
1. 现象定义：发生了什么
2. 链路推演：请求怎么一步步打到DB并放大
3. 治理组合：从入口、缓存、DB、降级多层防护
```

示例表达：

```text
缓存击穿不是简单的Key过期，而是热点Key过期后，10万QPS同时Redis miss，大量线程同时查DB，导致DB连接池打满，应用线程阻塞，最终接口超时。所以治理重点不是让每个请求都查到最新数据，而是限制回源并发，只允许一个线程重建缓存，其他请求返回旧值或降级。
```

## 二、治理方案还偏单点，缺少组合防线

> Day3发现

### 当前表现

能够给出一些正确方案：

```text
击穿：Key不过期、加锁
雪崩：过期时间分散、加锁
穿透：网关防攻击、布隆过滤器、空值缓存
```

这些方案方向正确。

### 与架构师水平的差距

架构师设计更关注系统性防线，而不是单个技术点。

例如缓存击穿不能只说“加锁”，完整方案应包括：

```text
热点Key识别
  ↓
活动前预热
  ↓
逻辑过期
  ↓
异步刷新
  ↓
本地缓存兜底
  ↓
DB回源限流
  ↓
熔断降级
  ↓
监控告警
```

缓存雪崩也不能只说“TTL随机”，还要考虑：

```text
Redis整体故障
Redis连接池耗尽
网络抖动
Redis恢复后的缓存冷启动
DB保护和线程池隔离
```

### 补足方向

回答治理方案时，按“四层防线”组织：

```text
第一层：入口层
  参数校验、网关限流、防刷、黑名单

第二层：缓存层
  布隆过滤器、空值缓存、随机TTL、逻辑过期、预热、本地缓存

第三层：回源层
  互斥锁、回源限流、线程池隔离、异步重建

第四层：降级层
  熔断、返回旧值、默认值、快速失败、告警补偿
```

## 三、方案副作用和适用边界表达不足

> Day3发现

### 当前表现

能说出“用什么方案”，但对方案副作用还没有系统表达。

例如：

```text
布隆过滤器
空值缓存
加锁
Key不过期
TTL分散
```

这些方案都被提到，但没有展开它们的风险和边界。

### 与架构师水平的差距

架构师面试中，面试官更关注候选人是否知道技术方案的代价。

| 方案 | 解决问题 | 副作用 |
|------|----------|--------|
| 布隆过滤器 | 穿透 | 有误判，删除困难，需要初始化和同步 |
| 空值缓存 | 穿透 | 占用Redis空间，TTL过长会挡住新增数据 |
| 互斥锁 | 击穿 | 增加延迟，锁竞争，锁超时，线程等待堆积 |
| 逻辑过期 | 击穿 | 可能返回旧值，异步刷新复杂 |
| TTL随机值 | 雪崩 | 只能解决同批过期，不能解决Redis整体故障 |
| 本地缓存 | 击穿/雪崩 | 多节点一致性差，更新延迟 |

### 补足方向

以后每说一个方案，都补一句：

```text
这个方案解决什么问题，但代价是什么，适合什么业务。
```

示例：

```text
逻辑过期适合秒杀商品详情、首页配置这类读多写少的热点数据，因为它可以避免热点Key物理失效导致请求集中回源。但它的代价是可能返回短暂旧值，所以不能用于支付余额、订单状态强一致判断这类场景。
```

## 四、缓存一致性问题需要形成固定回答框架

> Day3发现

### 当前表现

本次回答主要集中在穿透、击穿、雪崩，对缓存和 MySQL 一致性还需要继续强化。

### 与架构师水平的差距

缓存一致性是 Redis 面试必问题，不能只说“删缓存”或“延迟双删”，而要能完整回答：

```text
1. 先删缓存再更新DB的问题
2. 先更新DB再删缓存的问题
3. 为什么主流推荐先更新DB再删缓存
4. 删除缓存失败怎么补偿
5. 为什么删除缓存而不是更新缓存
6. MQ/binlog如何保证最终一致
7. 强一致场景如何绕过缓存
```

尤其要说清楚：

```text
Redis缓存通常只能保证最终一致，强一致判断必须回到DB或权威服务。
```

### 补足方向

固定模板：

```text
更新类操作我会优先更新DB，再删除缓存。因为先删缓存再更新DB时，并发读可能把旧值重新写回缓存；而先更新DB再删缓存，只要删除成功，后续读会回源拿到新值。它的问题是删除缓存可能失败，所以需要同步重试、MQ异步重试、死信队列、补偿任务或binlog订阅来保证最终一致。对于强一致场景，比如支付、扣库存、预约提交，不能依赖Redis缓存做最终判断，必须查DB主库或权威库存/号源服务。
```

## 五、业务场景取舍还需要继续强化

> Day3发现

### 当前表现

回答中已经能结合“大促预热”“热点Key”“恶意请求”等场景，但还需要进一步把方案映射到具体业务：

```text
保险商城
医疗问诊
秒杀系统
IM在线客服
```

### 与架构师水平的差距

架构师不是泛泛说“加缓存、加锁、加布隆过滤器”，而是要按业务语义判断：

```text
哪些数据可以旧？
哪些数据不能旧？
哪些请求可以降级？
哪些请求必须强一致？
哪些链路要保护DB？
```

例如：

```text
医生详情可以缓存旧值，但预约提交必须查权威号源。
秒杀库存展示可以旧，但扣库存必须原子扣减。
保单详情展示可以最终一致，但用户刚修改地址后要保证读己之写。
IM会话列表可以旧，但消息发送不能丢。
```

### 补足方向

以后回答架构题时，每个技术方案都落到业务语义：

```text
展示读：缓存优先，可最终一致，可返回旧值
交易读：权威存储优先，不能依赖缓存
写操作：DB/状态机为准，缓存删除和补偿
高并发热点：逻辑过期、本地缓存、限流降级
```

## 六、秒杀系统回答已经具备主链路，但异常闭环还需要模板化

> Day6发现

### 当前表现

本次回答高并发秒杀系统时，已经能抓住主链路关键点：

```text
前端 CDN 静态加速
网关限流、防刷、Token 校验
Redis + Lua 预扣库存
MQ 削峰填谷
DB 扣减真实库存
对账系统兜底修复
```

这说明对秒杀系统的整体架构已经有实战化认知，不再停留在“Redis 扣库存”这个单点。

### 与架构师水平的差距

差距主要不在方案方向，而在表达的完整性和边界清晰度。

需要继续强化三点：

```text
1. MQ 失败要区分生产端失败、消费端失败、ACK 失败、重复投递
2. Redis 幂等适合作为性能拦截，但 DB 唯一索引仍然要作为最终正确性兜底
3. Redis 和 DB 不一致时，不能简单说以谁为准，而要基于订单、库存流水、MQ 在途、预扣记录做链路级对账
```

尤其是幂等设计，不能因为 Redis 性能好就弱化 DB 唯一约束。架构师表达要区分：

```text
Redis 幂等 Key：提升性能，挡住绝大多数重复请求
DB 唯一索引：保证最终正确性，兜住 Redis 异常、Key 丢失、重复消费等极端情况
```

### 补足方向

后续回答秒杀、订单、支付这类高并发交易题时，使用固定闭环：

```text
1. 主链路：入口削峰 → Redis 预扣 → MQ 异步 → DB 落库
2. 正确性：Lua 原子性 → DB 条件更新 → 唯一索引 → 幂等状态机
3. 失败链路：发送失败、消费失败、ACK失败、重复消费、订单超时、Redis故障
4. 补偿链路：本地消息表、重试、死信队列、定时任务、对账系统
5. 一致性边界：哪些强一致，哪些最终一致
```

推荐表达：

```text
秒杀系统不是单纯依赖 Redis 保证正确性，而是 Redis 负责高并发预扣，MQ 负责异步削峰，DB 负责最终事实，唯一索引和条件更新兜底正确性，对账系统负责把异常链路收敛到最终一致。
```

下一次遇到 Redis 或秒杀系统题，可以优先使用这个总框架：

```text
先判断是什么问题：穿透 / 击穿 / 雪崩 / 一致性 / 库存并发

```
