
# Day 4 架构师梳理：Redis性能优化与问题排查

## 面试核心定位

Redis问题排查是架构师面试的高频考点，面试官不只是问"怎么排查慢查询"，更关注：

```
1. 系统化的排查思路：不是乱试，而是有步骤地定位
2. 量化判断标准：不是"碎片率高"，而是"碎片率&gt;1.5需要关注"
3. 成本意识：不是"用最好的硬件"，而是"用最小成本解决问题"
4. 预防为主：不是"出问题再救火"，而是"从设计上避免问题"
```

很多高级开发能背出慢查询日志怎么看，但被问"碎片率1.8怎么处理"、"连接数满了应急方案是什么"时就答不上来。

## 一、慢查询排查的架构师思维

### 不是"找慢查询"，而是"建防线"

```
普通开发：
  出问题 → 看慢查询日志 → 发现是KEYS命令 → 禁用KEYS

架构师：
  1. 事前预防：在代码审查中禁止KEYS/FLUSHDB，用rename-command禁用
  2. 事中监控：慢查询告警（&gt;5ms）、QPS监控、CPU监控
  3. 事后分析：慢查询日志定期分析，找出Top N慢查询持续优化
  4. 应急方案：出现慢查询时，怎么快速杀掉阻塞命令，怎么降级

区别：
  普通开发是"救火队员"，架构师是"防火体系设计师"
```

### 慢查询的三层预防

```
第一层：命令禁用
  rename-command KEYS ""
  rename-command FLUSHDB ""
  rename-command FLUSHALL ""
  → 从根源上禁用危险命令

第二层：代码规范
  禁止使用KEYS/HGETALL/SMEMBERS/LRANGE 0 -1
  大集合用SCAN/HSCAN/SSCAN/ZSCAN迭代
  → 代码审查中检查

第三层：监控告警
  慢查询&gt;5ms告警
  CPU&gt;80%告警
  QPS突增突降告警
  → 提前发现问题
```

### INFO命令的架构师解读

```
普通开发看INFO：
  看一眼CPU、内存，没发现明显问题，就不知道看什么了

架构师看INFO：
  1. 先看cluster_state是不是ok（如果是Cluster）
  2. 看used_memory_rss/used_memory算碎片率
  3. 看hit_rate = keyspace_hits/(keyspace_hits+keyspace_misses)
  4. 看instantaneous_ops_per_sec，结合QPS评估
  5. 看replication的lag值，主从延迟有没有问题
  6. 看blocked_clients，有没有阻塞命令
  7. 看expired_keys/evicted_keys，有没有大量过期淘汰

区别：
  普通开发是"看指标"，架构师是"读故事"——通过指标推演发生了什么
```

## 二、内存碎片的工程权衡

### 不是"碎片率1.8就要重启"

```
碎片率判断标准：
  &lt; 1.0：不正常 → 有swap，内存不够用了
  1.0 ~ 1.5：健康 → 正常范围
  1.5 ~ 2.0：关注 → 监控，不需要紧急处理
  &gt; 2.0：严重 → 需要处理

架构师决策：
  碎片率1.8，但内存还有50%可用 → 不需要立即处理
  碎片率1.8，但内存只剩10% → 需要处理
  碎片率1.8，且还在持续增长 → 需要处理

不是只看一个指标，而是看"指标趋势 + 业务影响"
```

### 内存碎片的成本矩阵

```
方案              效果        业务影响        成本
─────────────────────────────────────────────────────────
MEMORY PURGE      ★★☆        无              低
定期重启          ★★★        需要停机       高
主从切换          ★★★        短暂不可用      中
避免频繁更新      ★★☆        需修改代码      中
BigKey拆分        ★★★        需修改代码      高
换SSD             ★☆☆        无              极高

架构师选择：
  优先MEMORY PURGE（低成本、无影响）
  不行的话，低峰期定期重启
  不能停机的话，主从切换
  从根源上避免频繁更新和BigKey

不是选"最好的方案"，而是选"ROI最高的方案"
```

### 内存优化的ROI排序

```
投入产出比从高到低：

1. 给Key设置过期时间 ★★★★★
  → 0成本，效果最好
  → 永久Key是最大的内存浪费

2. Key名尽量短 ★★★★☆
  → 0成本，改个命名规范
  → user:info:100 → u:i:100

3. 小集合用ziplist ★★★☆☆
  → 改个配置，代码不用动
  → hash-max-ziplist-entries 512

4. Hash代替String存对象 ★★☆☆☆
  → 需要改代码
  → 但效果显著

5. 冷数据归档 ★★☆☆☆
  → 需要改代码，还有外部存储成本
  → 但长期收益大

6. 升级硬件 ★☆☆☆☆
  → 成本最高，不建议上来就用
  → 最后不得已的选择

架构师不是"上来就加机器"，而是"从0成本的优化做起，一步步来"
```

## 三、连接池问题的系统化治理

### 连接泄漏不是"加个finally"那么简单

```
普通开发：
  try { ... }
  finally { jedis.close(); }
  → 加个finally，完事

架构师：
  1. 代码规范：try-with-resources是标配
  2. 监控告警：连接池等待线程数&gt;0告警
  3. 兜底策略：testWhileIdle定期检测空闲连接
  4. 应急方案：连接数满了，先杀掉空闲连接

连接泄漏的完整治理：
  代码规范 + 监控告警 + 兜底策略 + 应急方案
  四层防线，不是只靠finally
```

### 连接池配置的架构师原则

```
原则1：连接数不是越多越好
  单应用连接池MaxTotal 50-500足够
  → 太多连接反而会降低性能（Redis单线程，太多连接竞争）

原则2：MaxIdle = MaxTotal * 0.5
  MinIdle = MaxTotal * 0.1
  → 避免频繁创建销毁连接

原则3：一定要设置超时
  maxWaitMillis: 获取连接等待超时
  connectTimeout: 连接超时
  timeout: 命令超时
  → 避免无限等待拖垮应用

原则4：一定要监控
  活跃连接数、空闲连接数、等待线程数
  → 提前发现泄漏和瓶颈

原则5：生产环境不要用连接池以外的方式
  不要每次创建新连接
  → 连接复用才是正道
```

### 连接数满的应急方案

```
第一步：先止血
  CLIENT KILL addr idle&gt;300
  → 杀掉空闲超过5分钟的连接
  config set maxclients 20000
  → 临时调大maxclients

第二步：查原因
  CLIENT LIST看哪些连接活跃
  看应用日志有没有连接泄漏
  看连接池监控，等待线程数是不是很高

第三步：根治
  修复代码中的连接泄漏
  调整连接池配置
  加监控告警

架构师原则：
  先止血，再查因，后根治
  → 顺序不能乱，先保证业务可用
```

## 四、问题排查的系统化思路

### 不是"猜"，而是"逐步定位"

```
普通开发排查问题：
  是不是网络问题？重启试试？是不是版本bug？
  → 靠猜，没有章法

架构师排查问题：
  1. 确认范围：是不是Redis的问题？还是应用的问题？还是网络的问题？
  2. 收集信息：看应用日志、Redis监控、慢查询日志
  3. 假设验证：根据信息提出假设，逐一验证
  4. 定位根因：找到问题根源
  5. 根治预防：不仅修复，还要从设计上预防

区别：
  普通开发是"试错"，架构师是"科学方法"
```

### 问题排查的通用框架

```
任何Redis问题都可以用这个框架：

第一步：是不是Redis的问题？
  redis-cli ping看延迟
  redis-cli执行简单命令
  redis-benchmark压测
  → 先确认是不是Redis本身的问题

第二步：如果是，看什么类型的问题？
  慢查询 → SLOWLOG GET
  内存 → INFO memory
  连接 → INFO clients
  复制 → INFO replication
  集群 → CLUSTER INFO/NODES

第三步：如果不是，看应用的问题？
  连接池配置
  序列化方式
  网络
  代码逻辑

第四步：如果都不是，看业务的问题？
  流量突增
  Hot Key
  数据量突增

这个框架可以覆盖90%的Redis问题
```

### 架构师的问题排查Checklist

```
每次排查问题，按这个Checklist走一遍：

□ 1. 看应用日志：有没有超时、异常
□ 2. 看Redis监控：CPU/内存/QPS/命中率/慢查询
□ 3. 看慢查询日志：Top 10慢查询是什么
□ 4. 看INFO命令：逐一检查memory/clients/replication/stats
□ 5. 看连接池监控：活跃/空闲/等待线程数
□ 6. 看网络：ping延迟/iperf带宽
□ 7. 看业务：流量是不是突增/数据量是不是突增

架构师不是"上来就调参数"，而是"先把情况摸清楚"
```

## 五、与架构师水平的差距（Day4自检）

| 维度 | 普通开发 | 高级开发 | 架构师 |
|------|----------|----------|--------|
| 慢查询 | 知道看SLOWLOG | 能找出慢查询并修复 | 有完整的"预防-监控-分析-应急"体系 |
| 内存碎片 | 没概念，出问题就重启 | 知道碎片率，会用MEMORY PURGE | 知道成本矩阵，能根据ROI选方案 |
| 连接池 | 会配置连接池 | 知道避免连接泄漏 | 有"规范-监控-兜底-应急"四层防线 |

| 优化思维 | 上来就加机器 | 会改配置、改代码 | 从0成本优化做起，看ROI决策 |

### 一句话差距

**高级开发是"会用工具排查问题"，架构师是"有系统化的排查框架和成本意识，不仅能解决问题，还能预防问题"。**
