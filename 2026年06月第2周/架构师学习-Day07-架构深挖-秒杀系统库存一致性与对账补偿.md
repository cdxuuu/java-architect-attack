## 二、题目：秒杀系统库存一致性与对账补偿设计

你负责设计一个高并发秒杀系统，核心链路如下：

```text
用户请求
  ↓
网关限流、防刷、Token 校验
  ↓
秒杀服务
  ↓
Redis + Lua 预扣库存，记录用户已抢购标记
  ↓
发送 MQ
  ↓
订单服务消费消息
  ↓
创建订单
  ↓
DB 条件扣减真实库存
  ↓
用户支付 / 超时关闭
```

活动规模如下：

```text
10 秒内 100 万请求
库存 1 万件
每个用户限购 1 件
Redis 负责前置库存预扣
DB 库存是最终事实
MQ 负责异步削峰
允许最终一致，但不能超卖
```

现在要求你设计：

```text
秒杀系统的库存一致性与对账补偿机制。
```

### 4. 订单创建成功，但 DB 扣库存失败怎么办？

场景：

```text
Redis 预扣成功
MQ 消费成功

```

例如：

```sql

set stock = stock - 1
where sku_id = ?
  and activity_id = ?
  and stock > 0;
```

请回答：

```text

此时订单状态如何处理？

用户抢购标记是否删除？

```

## 六、作答提醒

这道题的核心不是单点技术，而是架构链路。

如果只回答：

```text
Redis 扣库存，MQ 异步下单，DB 兜底，对账补偿
```

还不够。

架构师级别的表达需要进一步讲清楚：

```text
Redis 扣的是资格库存，不是最终库存；
MQ 承载的是异步下单过程态；
订单状态表达用户交易生命周期；
DB 库存是最终库存事实；
支付状态决定订单最终是否生效；
对账系统负责把过程态收敛到最终态。
```

最终要能回答清楚一句话：

```text
秒杀系统不是靠某一个组件保证一致性，而是靠 Redis 前置削峰、MQ 异步削峰、订单幂等、DB 条件扣减、唯一索引、超时关闭和对账补偿共同保证最终一致，并由 DB 条件扣减作为防止超卖的最后防线。
```
