# 专题：高并发秒杀系统架构设计

## 一、题目

你负责设计一个秒杀系统，活动开始后预计：

```text
10 秒内涌入 100 万请求
商品库存 1 万件
用户只能购买 1 件
```

系统要求：

```text
防止超卖
防止重复下单
防止接口被刷
下单成功后异步生成订单并扣减库存
系统尽量保证高可用
允许少量用户看到“排队中”
```

需要回答：

```text
1. 整体架构怎么设计？
2. 如何做流量削峰？
3. 如何保证不超卖？
4. Redis 库存扣减和数据库库存扣减如何配合？
5. MQ 消费失败、重复消费怎么处理？
6. 用户重复点击下单如何防重？
7. 如果 Redis 挂了，系统如何降级？
8. 秒杀结束后，如何对账修复库存和订单状态？
```

加分项：

```text
画出核心链路流程
写出 Redis Lua 扣库存的伪代码
说明哪些地方用最终一致性，哪些地方必须强一致性
```

---

## 二、你的回答

整体架构设计：

```text
前端页面，CDN 静态加速，倒计时和置灰防止乱点；
网关层，防攻击措施，比如 token 校验、网段拦截等；
库存使用 Redis + Lua 脚本预扣方案，利用 Redis 的效率和 Lua 的原子性；
库存和订单服务交互采用 MQ 方式削峰填谷，需要做防丢和幂等手段；
实际库存由 DB 扣减；
兜底需要对账系统，校验 Redis 剩余 + 在途 = DB，对 Redis 库存进行补偿，对重复扣减进行排除等。
```

MQ 失败和订单超时补充：

```text
MQ 消息失败场景，在生产者首先有消息重试，比如 RocketMQ 重试 3 次；
如果还是失败，就落本地消息记录表，同时邮件通知；
消息表中的数据靠 Job 或人工拉起。

既然做了重试，就要做幂等防止重复消费，比如 Redis 唯一键、DB 唯一键；
更倾向 Redis，因为在现有 Redis 架构下可靠性可以接受，同时效率更高，而 DB 的请求瓶颈难以解决。

订单超时问题有几种方案：
延迟消息，小业务量可以用，因为底层通过独立 Queue 实现，不适合业务量较大场景；
定时任务比较推荐，单独字段存储到期时间，定时任务查询。

至于 Redis 和 DB 不一致场景，倾向以 DB 为主，但是要有完整链路对比，不存在简单以谁为准。
```

## 四、标准答案

## 1. 整体架构设计

秒杀系统的核心目标是：

```text
抗高并发
防止超卖
防止重复下单
保护数据库
异步削峰
最终一致
异常可补偿
```

整体链路：

```text
用户
 ↓
前端页面 / App
 ↓
CDN 静态资源加速
 ↓
网关层：限流、防刷、Token 校验、黑名单、风控
 ↓
秒杀服务
 ↓
Redis + Lua：库存预扣 + 用户去重
 ↓
MQ：削峰填谷
 ↓
订单服务
 ↓
DB：创建订单 + 扣减真实库存
 ↓
支付服务
 ↓
超时关闭订单 / 库存回补
 ↓
对账补偿系统
```

架构职责拆分：

```text
前端：减少无效点击
CDN：静态资源加速，降低源站压力
网关：限流、防刷、鉴权、风控
Redis：承接热点库存，做高并发预扣
Lua：保证库存判断、扣减、用户去重原子执行
MQ：异步削峰，保护订单服务和数据库
订单服务：幂等消费，创建订单
DB：真实库存扣减和订单持久化
对账系统：发现异常并补偿
```

## 2. 如何做流量削峰

秒杀削峰要分层处理。

### 2.1 前端削峰

```text
页面静态化
CDN 缓存静态资源
活动开始前按钮置灰
倒计时控制入口
点击后按钮置灰，防止重复提交
验证码 / 滑块验证
```

前端不能保证安全，但可以减少大量无效请求。

### 2.2 网关削峰

```text
用户维度限流
IP 维度限流
设备维度限流
秒杀 Token 校验
接口签名校验
黑名单拦截
令牌桶 / 漏桶限流
风控识别异常请求
```

明显异常的请求直接拒绝。

### 2.3 服务端削峰

```text
Redis 承接热点库存
Lua 脚本原子扣减
预扣成功才进入 MQ
MQ 异步下单
订单服务按自身能力消费
数据库限流保护
```

核心原则：

```text
不要让所有请求直接访问数据库。
```

## 3. 如何保证不超卖

防超卖主要有两层。

### 3.1 Redis Lua 预扣库存

秒杀开始前，将活动库存预热到 Redis：

```text
seckill:stock:{activityId}:{skuId} = 10000
```

用户请求进来后，通过 Lua 脚本完成：

```text
判断用户是否已经购买
判断库存是否大于 0
扣减 Redis 库存
记录用户购买标记
```

Lua 脚本伪代码：

```lua
local stockKey = KEYS[1]
local userKey = KEYS[2]

if redis.call("exists", userKey) == 1 then
    return 2
end

local stock = tonumber(redis.call("get", stockKey))
if stock == nil or stock <= 0 then
    return 0
end

redis.call("decr", stockKey)
redis.call("setex", userKey, 86400, 1)

return 1
```

返回值：

```text
1：预扣成功，进入 MQ 下单流程
0：库存不足
2：重复下单
```

### 3.2 DB 条件扣减兜底

数据库扣库存时不能先查再扣，而要使用条件更新：

```sql
update sku_stock
set stock = stock - 1
where sku_id = ?
  and activity_id = ?
  and stock > 0;
```

判断影响行数：

```text
影响行数 = 1：扣减成功
影响行数 = 0：库存不足或已经被扣完
```

总结：

```text
Redis 负责高并发预扣；
DB 负责最终真实库存扣减；
DB 条件更新是防超卖最终兜底。
```

---

## 4. Redis 库存和 DB 库存如何配合

秒杀开始前：

```text
DB 初始库存：10000
Redis 秒杀库存：10000
```

秒杀请求进入后：

```text
Redis Lua 预扣成功
 ↓
发送 MQ 消息
 ↓
订单服务消费消息
 ↓
创建订单
 ↓
DB 条件扣减库存
 ↓
记录库存流水
```

Redis 和 DB 不是强同步关系。

更合理的理解是：

```text
Redis 是前置库存闸门，负责挡高并发；
DB 是最终事实存储，负责真实库存；
MQ 是异步衔接；
库存流水和对账系统负责最终修复。
```

如果 Redis 预扣成功，但后续失败：

```text
MQ 发送失败：本地消息表补偿投递，或回补 Redis 库存
订单创建失败：订单置失败，回补 Redis 库存
DB 扣减失败：订单置失败，回补 Redis 库存
```

---

## 5. MQ 发送失败、消费失败和重复消费如何处理

MQ 异常要分生产端和消费端。

### 5.1 生产端发送失败

Redis 预扣成功后，需要发送 MQ 消息。

如果发送失败：

```text
先依赖 MQ Producer 自身重试机制
例如 RocketMQ 同步发送失败后重试 3 次
```

如果多次重试仍失败：

```text
消息落本地消息表
状态标记为待发送
记录失败原因和重试次数
发送邮件、企微或钉钉告警
由定时任务扫描消息表重新投递
超过最大重试次数后人工处理
```

更严谨的做法：

```text
本地消息表和业务操作放在同一个本地事务里提交。
```

### 5.2 消费端处理失败

订单服务消费 MQ 时，如果处理失败：

```text
不 ACK 消息
依赖 MQ 重新投递
多次失败后进入死信队列
死信队列由补偿任务或人工处理
```

### 5.3 重复消费如何幂等

MQ 通常是至少一次投递，所以消费者必须幂等。

幂等分两层：

```text
第一层：Redis 幂等 Key
  以 requestId 或 activityId + userId 为 Key
  使用 SETNX 拦截重复消费

第二层：DB 唯一索引
  订单表 unique(activity_id, user_id)
  兜住 Redis Key 丢失、重复投递、异常重放等极端情况
```

推荐表达：

```text
Redis 幂等 Key 是性能拦截，DB 唯一索引是正确性兜底。
```

## 6. 用户重复点击下单如何防重

防重复下单需要多层防护。

### 6.1 前端防重

```text
点击后按钮置灰
请求未返回前禁止再次提交
倒计时结束前不可点击
```

### 6.2 网关防重

```text
用户维度限流
接口 Token 校验
短时间重复请求拦截
```

### 6.3 Redis 防重

Redis 中记录用户参与标记：

```text
seckill:user:{activityId}:{userId}
```

Lua 脚本中先判断该 Key 是否存在。

如果存在，直接返回重复下单。

### 6.4 DB 防重

订单表增加唯一约束：

```text
unique(activity_id, user_id)
```

总结：

```text
前端防误点；
网关防刷；
Redis 防高并发重复请求；
DB 唯一索引做最终兜底。
```

## 7. 订单超时未支付如何处理

秒杀成功后，订单通常处于待支付状态。

如果超时未支付，需要关闭订单并回补库存。

### 7.1 延迟消息方案

订单创建成功后，发送一条延迟消息。

例如：

```text
15 分钟后检查订单状态
```

延迟消息到期后：

```text
如果订单仍然是待支付，则关闭订单并回补库存；
如果订单已经支付，则忽略。
```

优点：

```text
实现简单
实时性较好
```

缺点：

```text
大规模订单场景下，MQ 延迟消息存储和调度压力较大。
```

适合中小规模场景。

### 7.2 定时任务扫描方案

订单表增加字段：

```text
expire_time
order_status
```

建立索引：

```sql
create index idx_order_status_expire_time
on seckill_order(order_status, expire_time);
```

定时任务分片扫描：

```sql
select *
from seckill_order
where order_status = 'WAIT_PAY'
  and expire_time <= now()
limit 1000;
```

然后批量关闭订单并回补库存。

该方案优点：

```text
可分片
可限速
可重试
可批量处理
便于人工干预
```

大规模场景下更推荐定时任务或时间轮方案。

## 8. Redis 挂了系统如何降级

如果 Redis 挂了，不能让请求直接打 DB。

否则数据库会被瞬间打垮。

降级策略：

```text
秒杀入口熔断
返回“活动火爆，请稍后重试”
返回“排队中”
网关限流进一步收紧
只允许少量白名单流量进入
```

如果系统中有本地库存快照，也只能用于展示或快速失败，不建议用于真实扣减。

原因：

```text
本地缓存无法保证多实例之间库存一致性，容易导致超卖。
```

降级原则：

```text
宁可少卖，也不能超卖；
宁可让用户排队，也不能打爆数据库。
```

## 9. 秒杀结束后如何对账和补偿

秒杀结束后，需要通过对账系统校验数据是否一致。

活动进行中：

```text
Redis 是高并发库存闸门；
DB 是最终持久化依据。
```

活动结束后：

```text
以 DB 订单、库存流水作为事实依据；
结合 Redis 剩余库存、MQ 在途消息、预扣记录做完整对账。
```

常见对账数据：

```text
初始库存
Redis 剩余库存
成功订单数
待支付订单数
已取消订单数
MQ 在途消息
预扣成功但未落库记录
库存流水
```

对账公式可以表示为：

```text
初始库存
= Redis 剩余库存
+ 已成功订单数
+ 待支付有效订单数
+ MQ 在途消息数
+ 预扣成功但未落库数量
```

异常处理：

| 异常场景 | 补偿方式 |
|----------|----------|
| Redis 预扣成功，但 MQ 发送失败 | 根据本地消息表重新投递 MQ，或回补 Redis 库存并删除用户标记 |
| MQ 重复投递 | Redis 幂等 Key 拦截，DB 唯一索引兜底，重复消息直接返回消费成功 |
| 订单创建成功，但 DB 扣库存失败 | 订单置失败，回补 Redis 库存，记录异常流水 |
| 订单超时未支付 | 关闭订单，回补 Redis 和 DB 库存，记录库存流水 |
| Redis 库存比预期多 | 检查是否存在订单已成功但 Redis 未扣减或重复回补 |
| Redis 库存比预期少 | 检查是否存在预扣成功但未落库、回补失败或重复扣减 |

## 10. 哪些地方强一致，哪些地方最终一致

### 10.1 必须强一致的地方

```text
Redis Lua 扣库存和用户去重必须原子
DB 扣库存必须使用条件更新
订单表 activity_id + user_id 必须唯一
支付回调更新订单状态必须幂等
库存流水不能重复
```

这些地方影响核心正确性：

```text
不能超卖
不能重复下单
不能重复支付
不能重复扣库存
```

### 10.2 可以最终一致的地方

```text
Redis 库存和 DB 库存同步
订单创建和用户请求返回
MQ 消息投递和消费
库存回补
订单超时关闭
对账补偿
```

这些场景允许短时间不一致，但必须通过：

```text
MQ 重试
本地消息表
死信队列
补偿任务
对账系统
人工处理
```

保证最终收敛。

## 五、面试总结版

可以用下面这段作为最终回答：

```text
秒杀系统的核心设计是分层削峰、防超卖、防重复和最终一致性。

前端通过页面静态化、CDN、按钮置灰和倒计时减少无效请求；
网关通过限流、防刷、Token 校验、黑名单和风控拦截异常流量；
秒杀服务不直接访问数据库，而是通过 Redis Lua 脚本完成库存预扣和用户去重；
预扣成功后发送 MQ，由订单服务异步消费消息、创建订单并扣减数据库库存；
数据库通过条件更新防止超卖，通过唯一索引防止重复下单；
MQ 生产端通过重试、本地消息表和告警保证消息不丢，消费端通过幂等和死信队列保证可恢复；
订单超时未支付可以通过延迟消息或定时任务关闭订单并回补库存；
最后通过对账系统比对 Redis、MQ、订单、库存流水和 DB 数据，发现异常后按类型补偿。

整体原则是：前面尽量挡流量，中间用 Redis 和 MQ 抗并发，后面用 DB 和对账保证最终正确。
```

## 六、能力差距与补足方向

### 1. 当前已经具备的能力

```text
能从前端、网关、Redis、MQ、DB、对账系统完整描述秒杀链路；
能意识到 Redis + Lua 是高并发库存预扣核心；
能意识到 MQ 需要防丢和幂等；
能结合 RocketMQ 重试、本地消息表、告警和 Job 补偿；
能区分延迟消息和定时任务处理订单超时的适用场景；
能意识到 Redis 和 DB 不一致不能简单粗暴覆盖。
```

### 2. 与架构师水平的差距

```text
1. 幂等设计需要明确 Redis 是性能层，DB 唯一索引是最终正确性层
2. MQ 异常要按生产失败、消费失败、ACK失败、重复投递拆开讲
3. Redis 故障降级要明确不能直打 DB，要优先保护系统
4. 对账公式要包含初始库存、Redis 剩余、成功订单、待支付订单、MQ 在途、预扣未落库等维度
5. 强一致和最终一致边界要主动说明
```

### 3. 补足方向

后续回答秒杀、支付、订单这类交易型架构题，固定使用以下框架：

```text
第一层：主链路
入口削峰 → Redis 预扣 → MQ 异步 → DB 落库

第二层：正确性
Lua 原子性 → DB 条件更新 → 唯一索引 → 幂等状态机

第三层：异常链路
MQ 发送失败 → 消费失败 → ACK失败 → 重复消费 → Redis故障 → 订单超时

第四层：补偿链路
本地消息表 → 重试 → 死信队列 → 定时任务 → 对账系统 → 人工处理

第五层：一致性边界
强一致保护核心正确性，最终一致处理异步链路和库存回补
```

推荐记忆句：

```text
Redis 负责快，MQ 负责缓冲，DB 负责准，对账负责最终收敛。
```
