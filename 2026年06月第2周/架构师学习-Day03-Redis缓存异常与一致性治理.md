# Day 3：Redis缓存异常与一致性治理 — 穿透/击穿/雪崩/一致性（练习整理）

## 题目一：缓存穿透、缓存击穿、缓存雪崩分别是什么？怎么治理？

### 场景

你的系统大量使用 Redis 缓存：

- 保险商城商品详情缓存
- 医疗问诊医生详情缓存
- 秒杀活动商品库存缓存
- IM 在线客服会话列表缓存
- 内部管理系统配置缓存

请回答：

1. 缓存穿透、缓存击穿、缓存雪崩分别是什么？
2. 三者的核心区别是什么？
3. 分别在什么业务场景下容易发生？
4. 分别有哪些治理方案？
5. 布隆过滤器、空值缓存、互斥锁、逻辑过期、随机 TTL 分别适合解决什么问题？
6. 如果是保险商城商品详情页、医疗问诊医生详情页、秒杀商品详情页，你会分别怎么设计？

---

## 你的回答

### 1.1 三者定义

击穿：Redis 缓存过期导致请求到了 DB。

雪崩：批量 Key 同时过期导致批量请求到 DB，可以算是多个击穿会引发雪崩。

穿透：Redis 和 DB 里都没有相应的数据。

### 1.2 核心区别

核心区别是：

1. 数据库里有没有对应的数据：穿透是 Redis 和 DB 都没有，击穿和雪崩通常 DB 里有。
2. Redis 过期的量：击穿是单个或少量热点 Key 过期，雪崩是大量 Key 同时过期。

### 1.3 典型场景

击穿场景：多出现在热点 Key 设置过期时间，然后超时后出现的。

雪崩场景：批量热点 Key 同时过期，比如大促时预热的数据过期时间太一致。

穿透场景：恶意请求。

### 1.4 治理方案

知道了场景，相应的方案也就有了：

- 击穿：Key 不过期，加锁，锁成功后取 DB 数据回填 Redis。
- 雪崩：过期时间分散，加锁。
- 穿透：网关防攻击，布隆过滤器，临时记录 null 到 Redis。

## 架构师点评

你这版回答方向正确，已经抓住了三类缓存异常的基本定义和主要差异：

```text
穿透：查不存在的数据，Redis和DB都没有
击穿：单个热点Key过期，大量请求瞬间回源DB
雪崩：大量Key同时过期，或Redis整体不可用，大量请求整体回源DB
```

其中“多个击穿会引发雪崩”这个理解是对的。

但架构师面试中不能只停留在定义层面，还要补足四个维度：

```text
1. 请求链路：流量是怎么从Redis miss放大到DB连接池打满的
2. 方案组合：不是一个方案解决所有问题，而是多层防线
3. 副作用：每个治理方案都有成本和适用边界
4. 业务取舍：不同业务对旧数据、延迟、强一致的容忍度不同
```

## 标准答案

## 一、缓存穿透

### 1. 定义

缓存穿透是指请求查询的数据在 Redis 和数据库中都不存在，导致每次请求都无法命中缓存，最终每次都查询数据库。

```text
请求 doctorId=999999999
    ↓
查询 Redis
    ↓
Redis miss
    ↓
查询 MySQL
    ↓
MySQL 也不存在
    ↓
不写缓存
    ↓
下一次相同请求继续打 DB
```

如果攻击者不断构造不存在的 ID：

```text
doctorId=900000001
skuId=900000002
orderId=900000003
...
```

Redis 无法拦截，DB 会持续承压。

### 2. 本质

缓存穿透的本质是：

> **缓存体系没有保存“不存在”这个结果，导致不存在的数据请求持续穿过缓存层。**

### 3. 常见场景

| 场景 | 示例 |
|------|------|
| 恶意攻击 | 随机扫不存在的商品 ID、医生 ID、订单 ID |
| 爬虫扫描 | 按 ID 递增爬取不存在的数据 |
| 参数异常 | `id=-1`、`id=0`、超大 ID、格式错误 |
| 数据删除后仍被访问 | 已下架商品、已删除医生、已取消活动 |
| 第三方调用错误 | 外部系统传了错误业务 ID |

### 4. 治理方案

#### 方案一：参数校验

第一道防线是参数校验，不要让明显非法的请求进入 Redis 和 DB。

```text
非法参数示例：
  id <= 0
  id 超过业务最大范围
  id 格式不符合雪花ID/UUID规范
  枚举值不存在
```

适用场景：

```text
所有入口请求，尤其是网关层、Controller层、RPC入口层
```

优点：

```text
成本最低，拦截最靠前
```

缺点：

```text
只能拦截格式非法，不能判断业务上是否存在
```

#### 方案二：布隆过滤器

布隆过滤器用来判断一个数据是否“可能存在”。

```text
请求 doctorId=999999999
    ↓
先查布隆过滤器
    ↓
布隆过滤器判断：一定不存在
    ↓
直接返回空，不查 Redis，不查 DB
```

布隆过滤器的结论有两种：

```text
一定不存在：可以直接拦截
可能存在：继续查 Redis / DB
```

适用场景：

```text
数据量大、读多写少、ID集合相对稳定的场景
```

例如：

- 商品 ID
- 医生 ID
- 用户 ID
- 门店 ID
- 活动 ID

优点：

```text
内存占用小，可以在大量非法ID进入DB前提前拦截
```

缺点：

```text
1. 有误判：不存在的数据可能被判断为可能存在
2. 删除困难：普通布隆过滤器不支持准确删除
3. 需要初始化和增量同步
4. 不能替代DB，只能减少无效回源
```

#### 方案三：空值缓存

如果 DB 查不到，也把空结果写入 Redis。

```text
GET doctor:999999999
    ↓
Redis miss
    ↓
查 DB，不存在
    ↓
SET doctor:999999999 null EX 60
    ↓
下次请求命中 null，直接返回空
```

适用场景：

```text
少量不存在的热点Key、删除后短时间仍被访问的数据
```

优点：

```text
实现简单，对重复访问的不存在Key很有效
```

缺点：

```text
1. 大量随机Key会占用Redis内存
2. TTL不能太长，否则真实数据新增后可能被旧null挡住
3. 需要和布隆过滤器、限流组合使用
```

#### 方案四：网关限流与防刷

针对明显异常来源做限流、封禁、验证码或黑名单。

```text
同一IP每秒请求1000个不存在ID
    ↓
网关限流
    ↓
异常User-Agent拦截
    ↓
账号/IP维度黑名单
```

适用场景：

```text
恶意攻击、爬虫扫描、异常调用方
```

## 二、缓存击穿

### 1. 定义

缓存击穿是指某个热点 Key 在过期瞬间，大量并发请求同时发现 Redis 未命中，于是同时回源数据库。

```text
热点Key：seckill:sku:1001:stock
平时QPS：10万

Key过期
    ↓
10万QPS同时访问Redis
    ↓
Redis miss
    ↓
大量线程同时查MySQL
    ↓
DB连接池打满
    ↓
接口超时
```

### 2. 本质

缓存击穿的本质是：

> **热点 Key 失效后，缓存重建没有并发控制，导致大量请求同时回源。**

### 3. 常见场景

| 场景 | 示例 |
|------|------|
| 秒杀活动 | 秒杀商品详情、库存展示、活动配置 |
| 首页热点 | 首页 Banner、推荐配置、频道页配置 |
| 医疗问诊 | 热门医生详情、专家排班 |
| 保险商城 | 热门产品详情、活动价格 |
| IM 在线客服 | 大客户会话列表、热门工单配置 |

### 4. 治理方案

#### 方案一：互斥锁重建缓存

只允许一个线程回源 DB 并重建缓存，其他线程等待、降级或返回旧值。

```text
请求发现Redis miss
    ↓
尝试获取锁：SET lock:sku:1001 NX EX 5
    ↓
获取锁成功：查DB → 回填Redis → 释放锁
    ↓
获取锁失败：短暂等待 / 返回旧值 / 降级
```

优点：

```text
可以保证只有少量请求回源DB，实现相对简单
```

缺点：

```text
1. 加锁会增加请求延迟
2. 大量线程等待锁可能打满应用线程池
3. 锁超时时间不好设置
4. 锁线程挂掉可能导致短时间无法重建
5. 锁失效过早可能产生多个线程同时回源
```

适用场景：

```text
数据一致性要求较高、可以接受短暂等待的业务
```

例如：

- 保单详情
- 用户账户信息
- 医生排班详情

#### 方案二：逻辑过期

Key 物理上不设置过期时间，而是在 Value 中保存逻辑过期时间。

```json
{
  "data": {
    "skuId": 1001,
    "stock": 1000
  },
  "expireTime": "2026-06-10 10:00:00"
}
```

请求流程：

```text
读取Redis
    ↓
命中数据
    ↓
判断逻辑过期时间
    ↓
未过期：直接返回
    ↓
已过期：尝试加锁异步刷新
    ↓
当前请求先返回旧值
```

优点：

```text
1. 热点Key不会物理失效
2. 请求不会集中打到DB
3. 高并发场景下延迟稳定
4. 适合热点数据保护
```

缺点：

```text
1. 可能返回短暂旧数据
2. 需要后台刷新或异步刷新机制
3. 代码复杂度高于普通TTL
4. 不适合强一致业务
```

适用场景：

```text
读多写少、允许短暂旧值的热点数据
```

例如：

- 秒杀活动详情
- 商品详情展示
- 首页配置
- 热门医生介绍

#### 方案三：热点 Key 预热

活动开始前提前加载热点数据到 Redis。

```text
活动配置发布
    ↓
识别热点商品/活动/医生
    ↓
提前查询DB
    ↓
写入Redis
    ↓
分散TTL或逻辑过期
```

适用场景：

```text
大促、秒杀、运营活动、已知热点场景
```

#### 方案四：多级缓存

本地缓存 + Redis + DB。

```text
请求
  ↓
本地缓存 Caffeine
  ↓ miss
Redis
  ↓ miss
MySQL
```

优点：

```text
1. 抗热点能力强
2. 降低Redis压力
3. Redis异常时可短时间兜底
```

缺点：

```text
1. 多节点本地缓存一致性更差
2. 更新和失效机制更复杂
3. 只能用于允许短暂旧数据的场景
```

## 三、缓存雪崩

### 1. 定义

缓存雪崩是指大量缓存 Key 在同一时间失效，或者 Redis 整体不可用，导致大量请求绕过缓存直接打到数据库。

```text
大量Key同时过期
    ↓
Redis大面积miss
    ↓
请求批量回源DB
    ↓
DB连接池打满
    ↓
慢查询堆积
    ↓
应用线程阻塞
    ↓
服务整体不可用
```

雪崩不只是大量 Key 过期，还包括 Redis 整体故障：

```text
Redis主节点宕机
Redis Cluster大面积不可用
Redis连接池耗尽
网络抖动导致访问Redis失败
```

### 2. 本质

缓存雪崩的本质是：

> **缓存层在一段时间内整体失去削峰能力，导致 DB 承接了远超自身容量的流量。**

### 3. 常见场景

| 场景 | 示例 |
|------|------|
| 批量预热 TTL 一致 | 大促前统一预热 100 万商品，全部设置 2 小时过期 |
| 定时任务批量写缓存 | 每天凌晨刷新配置，TTL 都是 24 小时 |
| Redis 故障 | 主从切换、集群不可用、网络抖动 |
| Redis 连接池耗尽 | 应用连接池配置过小或 Redis 慢操作堆积 |
| 缓存服务重启 | 大量缓存丢失，重启后冷启动 |

### 4. 治理方案

#### 方案一：TTL 加随机值

避免大量 Key 同一时刻失效。

```text
原始TTL：
  所有商品缓存都设置 7200秒

优化后：
  7200 + random(0, 1800) 秒
```

适用场景：

```text
批量缓存、预热数据、定时任务写入的数据
```

优点：

```text
实现简单，可以有效打散过期时间
```

缺点：

```text
只能缓解同批过期，不能解决Redis整体故障
```

#### 方案二：Redis 高可用

通过主从、哨兵、Cluster 提高 Redis 可用性。

```text
单机Redis
  ↓
主从复制
  ↓
Sentinel自动故障转移
  ↓
Cluster分片扩容
```

注意：

```text
Redis高可用只能缩短不可用时间，不能保证永远不雪崩。
```

#### 方案三：限流、熔断、降级

Redis 不可用时，不能让所有请求都查 DB。

```text
Redis异常
    ↓
应用检测到Redis错误率升高
    ↓
开启熔断
    ↓
非核心接口直接降级
    ↓
核心接口限流查DB
    ↓
保护DB连接池
```

#### 方案四：本地缓存兜底

对于允许短暂旧值的热点数据，可以使用本地缓存兜底。

```text
Redis不可用
    ↓
读取本地缓存旧值
    ↓
返回页面展示
    ↓
后台持续探测Redis恢复
```

适用场景：

```text
商品详情、活动配置、医生介绍、首页配置
```

不适用场景：

```text
支付余额、库存扣减、订单状态强一致判断
```

#### 方案五：缓存恢复后限速重建

Redis 恢复后不能让所有请求同时重建缓存。

```text
Redis恢复
    ↓
分批预热热点Key
    ↓
限制DB回源并发
    ↓
缓存写入加随机TTL
    ↓
逐步恢复正常流量
```

## 四、三者核心区别

| 维度 | 缓存穿透 | 缓存击穿 | 缓存雪崩 |
|------|----------|----------|----------|
| 数据是否存在 | DB 中不存在 | DB 中存在 | DB 中通常存在 |
| Redis 状态 | 查不到 | 单个热点 Key 过期 | 大量 Key 过期或 Redis 故障 |
| 流量特点 | 大量无效请求 | 单点热点高并发 | 大面积请求同时回源 |
| 典型原因 | 恶意请求、非法参数 | 热点 Key 失效 | TTL 集中、Redis 不可用 |
| 核心风险 | DB 被无效查询打满 | DB 被热点 Key 打满 | DB 被整体流量打爆 |
| 核心方案 | 参数校验、布隆过滤器、空值缓存 | 互斥锁、逻辑过期、预热 | 随机 TTL、高可用、限流熔断降级 |

一句话区分：

```text
穿透：查了一个根本不存在的东西。
击穿：一个特别热的东西刚好过期了。
雪崩：一大片缓存同时失效，或者Redis整体失效了。
```

## 五、常见治理手段适用范围

| 手段 | 主要解决 | 适用场景 | 副作用 |
|------|----------|----------|--------|
| 参数校验 | 穿透 | 非法 ID、非法参数 | 只能拦格式错误，不能判断业务存在性 |
| 布隆过滤器 | 穿透 | 大量不存在 ID 攻击 | 有误判，删除困难，需要初始化同步 |
| 空值缓存 | 穿透 | 少量不存在热点 Key | 占 Redis 空间，TTL 过长会挡住新数据 |
| 互斥锁 | 击穿 | 单个热点 Key 失效 | 增加延迟，锁竞争，锁超时风险 |
| 逻辑过期 | 击穿 | 读多写少热点数据 | 可能返回短暂旧值，复杂度较高 |
| 随机 TTL | 雪崩 | 批量 Key 同时写入 | 不能解决 Redis 整体故障 |
| 本地缓存 | 击穿/雪崩 | 允许短暂旧值的热点读 | 多节点一致性差 |
| 限流熔断降级 | 雪崩 | Redis 异常、DB 保护 | 牺牲部分可用性或体验 |
| 缓存预热 | 击穿/雪崩 | 大促、秒杀、活动 | 预热失败或 TTL 一致仍有风险 |

## 六、业务场景设计

### 1. 保险商城商品详情页

特点：

```text
读多写少
允许短暂旧值
商品下架、价格、权益说明需要及时更新
不是强一致交易判断点
```

推荐设计：

```text
1. 商品ID参数校验
2. 布隆过滤器拦截不存在商品ID
3. Redis缓存商品详情
4. 空值缓存已删除或不存在商品，TTL 1-5分钟
5. 热门商品使用逻辑过期
6. 商品变更后先更新DB，再删除缓存
7. 删除缓存失败通过MQ重试或binlog订阅补偿
8. 首页/活动商品提前预热
9. TTL增加随机值
```

取舍：

```text
商品详情展示可以短暂旧值，但下单时必须重新校验商品状态、价格、库存和权益。
```

### 2. 医疗问诊医生详情页

特点：

```text
医生介绍、科室信息读多写少
医生排班、号源状态一致性要求更高
非法医生ID可能被频繁扫描
```

推荐设计：

```text
1. doctorId 参数校验
2. 布隆过滤器维护有效医生ID集合
3. 不存在医生ID写空值缓存，短TTL
4. 医生基础信息使用Redis缓存 + 随机TTL
5. 热门医生使用逻辑过期
6. 排班和号源状态不能完全依赖旧缓存，下单/预约时查权威数据
7. 医生停诊、下架时更新DB后删除缓存
8. 删除失败通过MQ或binlog补偿
```

取舍：

```text
医生简介可以旧一点，但排班可预约状态不能只信缓存，预约提交时必须做强校验。
```

### 3. 秒杀商品详情页

特点：

```text
极高并发
热点明确
活动开始时间明确
展示信息允许短暂旧值
库存扣减必须强约束
```

推荐设计：

```text
1. 活动开始前预热商品详情、活动配置、库存展示Key
2. 热点Key使用逻辑过期，不依赖物理TTL
3. 本地缓存 + Redis 多级缓存承接读流量
4. 库存展示可以短暂旧值
5. 实际扣库存走Redis原子扣减/Lua脚本 + DB/MQ最终落库
6. 回源DB必须限流，避免缓存重建打爆DB
7. 对活动Key设置随机TTL或逻辑过期
8. Redis异常时活动详情可返回旧值，购买链路要限流或降级
```

取舍：

```text
秒杀详情页追求高可用和抗流量，可以返回旧值；真正下单链路必须依赖库存原子扣减、幂等、防超卖和异步落库兜底。
```

---

## 七、面试表达模板

可以这样回答：

```text
缓存穿透、击穿、雪崩本质上都是缓存没有挡住流量，导致请求回源数据库，但触发原因不同。

穿透是查不存在的数据，Redis和DB都没有，所以每次都会打到DB。治理上第一层做参数校验，第二层用布隆过滤器拦截一定不存在的ID，第三层对DB查不到的数据做短TTL空值缓存，最后在网关侧做限流和防刷。

击穿是单个热点Key过期，大量并发请求同时回源DB。治理上可以用互斥锁，只允许一个线程查DB并回填缓存；也可以用逻辑过期，热点Key物理上不过期，过期后由后台线程异步刷新，其他请求先返回旧值。对于秒杀商品、首页配置这类热点数据，我更倾向于逻辑过期 + 本地缓存 + 限流，因为它比同步加锁更能抗高并发。

雪崩是大量Key同时失效，或者Redis整体不可用，导致大量请求一起打DB。治理上要避免TTL集中失效，给过期时间加随机值；热点数据提前预热；Redis层面做主从、哨兵或Cluster高可用；应用侧要有限流、熔断、降级和本地缓存兜底，防止DB被打穿。
```

## 题目二：热点 Key 突然过期，10 万 QPS 同时打到 MySQL，怎么设计防击穿方案？

### 场景

秒杀活动首页有一个商品库存展示 Key：

```text
seckill:sku:1001:stock
```

平时 QPS 10 万。某一刻这个 Key 过期，大量请求同时回源 MySQL，导致数据库连接池打满。

请回答：

1. 你会用互斥锁还是逻辑过期？为什么？
2. 如果用互斥锁，锁粒度怎么设计？锁超时时间怎么定？
3. 如果加锁线程挂了怎么办？
4. 其他线程等待、降级、返回旧值，分别怎么取舍？
5. 如果热点 Key 非常多，怎么避免锁本身成为瓶颈？
6. 如何提前发现热点 Key？
7. 如何预热热点 Key？
8. 如何保证缓存重建不会把 MySQL 打爆？

---

## 答案

### 1. 首选逻辑过期，而不是普通物理过期

秒杀活动的库存展示 Key 属于典型热点 Key：

```text
1. QPS极高
2. 热点明确
3. 允许展示值短暂不准
4. 不能让请求集中回源DB
```

因此首选：

```text
逻辑过期 + 异步刷新 + 本地缓存 + 回源限流
```

不推荐单纯依赖物理 TTL：

```text
物理TTL到期
    ↓
Redis直接miss
    ↓
所有请求都进入缓存重建逻辑
    ↓
锁竞争、线程等待、DB压力同时出现
```

逻辑过期流程：

```text
请求读取Redis
    ↓
命中库存展示值
    ↓
判断逻辑过期时间
    ↓
未过期：直接返回
    ↓
已过期：尝试获取刷新锁
    ↓
获取锁成功：异步查DB或库存服务，刷新缓存
    ↓
获取锁失败：直接返回旧值
```

核心思想：

> **热点读请求永远不要因为缓存重建而批量阻塞，更不要批量回源 DB。**

### 2. 什么时候用互斥锁？

互斥锁适合一致性要求更高、不允许返回旧值的场景。

例如：

```text
保单详情
用户账户资料
订单详情
医生排班提交前校验
```

互斥锁流程：

```text
GET cacheKey
    ↓ miss
SET lockKey requestId NX EX 5
    ↓
成功：查DB → SET cacheKey → 删除锁
失败：短暂sleep后重试Redis，或返回降级结果
```

锁粒度：

```text
lock:cache:seckill:sku:1001:stock
```

锁粒度要和缓存 Key 对齐，不能用全局锁：

```text
错误：lock:cache:global
正确：lock:cache:{业务}:{id}
```

锁超时时间：

```text
锁TTL > P99回源耗时 + 缓存写入耗时 + 安全余量
```

例子：

```text
DB查询P99：80ms
业务组装：30ms
Redis写入：10ms
安全余量：500ms

锁TTL可以设置 1-3 秒
```

如果回源逻辑可能超过固定 TTL，可以使用 Redisson 看门狗自动续期，但要注意：

```text
1. 看门狗只能在持锁线程和进程存活时续期
2. 进程挂了，锁仍然依赖TTL释放
3. 看门狗不能解决业务幂等问题
4. 高并发热点读场景不能只靠Redisson锁硬扛
```

### 3. 加锁线程挂了怎么办？

必须给锁设置过期时间。

```text
SET lock:sku:1001 requestId NX EX 3
```

释放锁时必须校验 owner：

```text
if redis.get(lockKey) == requestId:
    redis.del(lockKey)
```

实际生产用 Lua 脚本保证原子性：

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

不能直接 `DEL lockKey`，否则可能误删别人的锁：

```text
线程A加锁，TTL=3s
线程A执行慢，锁过期
线程B获取锁
线程A执行完直接DEL
结果：线程A误删线程B的锁
```

### 4. 等待、降级、返回旧值怎么取舍？

| 策略 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| 等待锁释放 | 一致性要求较高，QPS不高 | 数据较新 | 高并发下线程容易堆积 |
| 返回旧值 | 热点展示数据 | 延迟稳定，保护DB | 可能短暂脏读 |
| 降级返回 | Redis/DB异常 | 保护系统 | 用户体验下降 |
| 快速失败 | 非核心接口 | 保护核心链路 | 可用性下降 |

秒杀库存展示推荐：

```text
返回旧值 > 降级 > 等待
```

原因：

```text
库存展示值不是最终扣减依据，允许短暂不准。
真正防超卖靠下单链路的库存扣减原子性和幂等控制。
```

### 5. 热点 Key 很多时如何避免锁瓶颈？

如果热点 Key 很多，不能简单为每个 miss 都抢锁查 DB。

组合方案：

```text
1. 热点Key提前识别和预热
2. 热点Key使用逻辑过期，减少物理miss
3. 缓存重建任务进入队列，控制并发数
4. DB回源做令牌桶限流
5. 本地缓存兜底
6. 按业务分组刷新，避免全量同时刷新
7. 对失败重建做退避重试
```

缓存重建并发控制：

```text
应用内只允许最多 N 个缓存重建任务同时查DB
超过后返回旧值或降级
```

### 6. 如何提前发现热点 Key？

可以从四个层面识别：

```text
1. 业务预判：秒杀商品、首页活动、热门医生、热门商品
2. Redis监控：hotkeys、bigkeys、命令访问频率
3. 应用埋点：按cacheKey统计QPS、miss率、回源次数
4. 网关日志：URL、商品ID、医生ID维度访问量TOP N
```

关键指标：

```text
cache_hit_rate
cache_miss_qps
db_fallback_qps
hot_key_qps
redis_command_latency
lock_wait_count
cache_rebuild_count
```

### 7. 如何预热热点 Key？

预热流程：

```text
活动发布
    ↓
运营系统生成热点数据清单
    ↓
预热任务分批查询DB
    ↓
写入Redis
    ↓
写入本地缓存或等待请求触发本地加载
    ↓
设置逻辑过期时间或随机TTL
    ↓
预热结果校验
```

注意点：

```text
1. 预热要分批，不能一次性打爆DB
2. 预热失败要告警
3. 预热后要校验命中率
4. TTL不能完全一致
5. 秒杀活动要在流量进来前完成预热
```

### 8. 如何保证缓存重建不会把 MySQL 打爆？

核心是限制回源量。

```text
1. 单Key互斥：同一Key只有一个线程回源
2. 全局限流：整个应用最多N个回源DB请求
3. 线程池隔离：缓存重建使用独立线程池
4. 熔断降级：DB慢或错误率升高时停止回源
5. 返回旧值：热点Key优先返回旧缓存
6. MQ异步重建：请求线程不直接查DB
7. 本地缓存兜底：减少Redis和DB压力
```

架构图：

```text
用户请求
  ↓
本地缓存
  ↓ miss
Redis缓存
  ↓ 逻辑过期
返回旧值 ───────┐
  ↓              │
异步刷新任务      │
  ↓              │
重建线程池限流    │
  ↓              │
MySQL查询         │
  ↓              │
回写Redis ◀──────┘
```

## 题目三：Redis 缓存和 MySQL 数据如何保证一致性？

### 场景

保险商城用户修改保单收货地址，写入 MySQL 后，Redis 里的保单详情缓存还存在旧地址。

请回答：

1. 先删缓存再更新 DB 有什么问题？
2. 先更新 DB 再删缓存有什么问题？
3. 为什么主流推荐是先更新 DB，再删除缓存？
4. 删除缓存失败怎么办？
5. 为什么不是更新缓存，而是删除缓存？
6. 延迟双删是什么？适合什么场景？有什么问题？
7. Canal / binlog 订阅删除缓存适合什么场景？
8. MQ 异步删除缓存怎么保证最终一致？
9. 如果要求强一致，Redis 缓存还能不能用？应该怎么设计？

---

## 答案

## 一、先删缓存再更新 DB 的问题

流程：

```text
写请求A：删除缓存
    ↓
读请求B：读缓存miss
    ↓
读请求B：查DB，读到旧值
    ↓
读请求B：把旧值写回缓存
    ↓
写请求A：更新DB为新值
```

结果：

```text
DB是新值，Redis是旧值。
旧值可能一直存在到TTL过期。
```

这是典型的不一致窗口。

## 二、先更新 DB 再删缓存的问题

流程：

```text
写请求A：更新DB成功
    ↓
写请求A：删除缓存
```

正常情况下，下一次读请求会 miss Redis，再从 DB 读新值并回填缓存。

主要风险：

```text
DB更新成功，但删除缓存失败。
```

结果：

```text
DB是新值，Redis仍然是旧值。
```

## 三、为什么主流推荐先更新 DB，再删除缓存？

因为它的不一致概率更低。

### 先删缓存再更新 DB 的问题更严重

在高并发读写下，旧值容易被重新写回缓存。

```text
删除缓存
    ↓
更新DB之前
    ↓
并发读请求查DB旧值
    ↓
旧值回填缓存
    ↓
DB更新完成
```

### 先更新 DB 再删缓存的不一致窗口较小

```text
更新DB
    ↓
删除缓存
```

只要删除缓存成功，后续读请求就会从 DB 读新值。

它的主要问题是删除失败，可以通过重试、MQ、binlog 订阅补偿。

结论：

```text
推荐：先更新DB，再删除缓存。
```

## 四、删除缓存失败怎么办？

不能只在业务代码里简单重试一次，要有可靠补偿机制。

### 方案一：同步重试

```text
更新DB成功
    ↓
删除Redis失败
    ↓
同步重试2-3次
```

适合：

```text
偶发网络抖动
```

不适合：

```text
Redis长时间不可用
```

### 方案二：MQ 异步重试

```text
更新DB成功
    ↓
发送缓存删除消息
    ↓
消费者删除Redis
    ↓
失败重试
    ↓
进入死信队列
    ↓
人工或任务补偿
```

注意：

```text
1. 消息要和DB更新尽量保证一致
2. 消费者删除缓存要幂等
3. 失败要有重试和死信
4. 要监控缓存删除积压
```

### 方案三：binlog 订阅

```text
业务更新MySQL
    ↓
MySQL写binlog
    ↓
Canal/Debezium订阅binlog
    ↓
解析变更表和主键
    ↓
删除对应缓存Key
```

适合：

```text
多业务系统都可能更新同一张表，需要统一缓存失效机制
```

缺点：

```text
1. 链路更长，延迟更高
2. 依赖binlog订阅组件稳定性
3. 表和缓存Key映射关系需要维护
4. 排查复杂度更高
```

## 五、为什么不是更新缓存，而是删除缓存？

因为更新缓存更容易出错。

### 问题一：并发写导致旧值覆盖新值

```text
请求A：更新DB为A1
请求B：更新DB为B1

请求B先更新缓存为B1
请求A后更新缓存为A1

结果：DB是B1，缓存是A1旧值
```

### 问题二：缓存值可能是复杂聚合结果

缓存通常不是单表字段，而是聚合视图：

```text
保单详情缓存 = 保单主表 + 收货地址 + 产品信息 + 权益信息 + 用户信息
```

写请求只改了地址，但要准确更新整个缓存，需要重新聚合，复杂且容易遗漏。

### 问题三：很多写后不一定马上读

如果每次写都更新缓存，可能浪费资源。

```text
更新100次，读1次
```

更合理的是：

```text
写时删除缓存，读时按需重建。
```

## 六、延迟双删

流程：

```text
删除缓存
    ↓
更新DB
    ↓
sleep一段时间
    ↓
再次删除缓存
```

目的：

```text
第二次删除用于清理在第一次删除后、DB更新前被并发读请求写回缓存的旧值。
```

适用场景：

```text
历史系统中已经采用先删缓存再更新DB，短期难以改造
```

问题：

```text
1. sleep时间不好确定
2. 会阻塞业务线程，通常要异步化
3. 不能彻底保证一致性
4. 增加一次Redis操作
5. 本质是降低概率，不是强一致方案
```

更推荐：

```text
先更新DB，再删除缓存 + 删除失败重试/补偿。
```

## 七、MQ 异步删除如何保证最终一致？

关键点：消息不能丢，消费要幂等，失败能补偿。

可靠链路：

```text
业务事务：更新DB
    ↓
写本地消息表 / 发送事务消息
    ↓
MQ投递缓存删除事件
    ↓
消费者删除Redis
    ↓
删除失败重试
    ↓
多次失败进入死信队列
    ↓
补偿任务扫描处理
```

消息内容：

```json
{
  "eventType": "POLICY_ADDRESS_UPDATED",
  "policyId": "P10001",
  "cacheKey": "policy:detail:P10001",
  "version": 102,
  "eventTime": "2026-06-10 10:00:00"
}
```

幂等处理：

```text
删除缓存天然幂等：删一次和删多次结果一样。
```

但如果涉及更新缓存，则必须用版本号防止乱序覆盖。

## 八、强一致场景还能不能用 Redis？

可以用，但不能把 Redis 当事实源。

强一致场景原则：

```text
1. 写路径以DB为准
2. 强一致读直接读DB或读主库
3. Redis只能做非关键展示缓存
4. 提交动作必须回源校验
5. 缓存不参与最终业务判断
```

例如保险商城修改收货地址：

```text
修改地址：写MySQL事务
    ↓
删除 policy:detail:{policyId} 缓存
    ↓
删除失败写MQ重试
    ↓
用户马上刷新详情页：
        如果需要读己之写一致性，可以直接查DB或绕过缓存一次
        如果只是普通展示，可以接受短暂最终一致
```

读己之写方案：

```text
用户刚修改成功后，在短时间内给当前用户请求加 bypassCache 标记
    ↓
下一次详情查询直接查DB
    ↓
查到新值后回填缓存
```

## 题目四：布隆过滤器如何解决缓存穿透？它有什么误判问题？

## 答案

### 1. 底层原理

布隆过滤器由一个位数组和多个哈希函数组成。

```text
初始化：
  bit数组全部为0

加入 doctorId=1001：
  hash1(1001) -> 3
  hash2(1001) -> 8
  hash3(1001) -> 15

把 bit[3]、bit[8]、bit[15] 置为1
```

查询时：

```text
查询 doctorId=1001：
  计算多个hash位置
  如果所有位置都是1：可能存在
  如果有任意一个位置是0：一定不存在
```

### 2. 为什么能判断一定不存在？

因为如果一个元素加入过布隆过滤器，它对应的多个 bit 位一定都被置为 1。

如果查询时发现任意一个 bit 位是 0，说明这个元素一定没有加入过。

### 3. 为什么不能判断一定存在？

因为不同元素可能哈希到相同位置。

```text
doctorId=1001 设置了 bit[3], bit[8], bit[15]
doctorId=2002 设置了 bit[2], bit[8], bit[20]
doctorId=3003 设置了 bit[3], bit[15], bit[20]

查询一个不存在的 doctorId=9999，刚好对应 bit[3], bit[8], bit[20]
这些位都被其他元素置为1
结果：误判为可能存在
```

所以：

```text
布隆过滤器没有漏判，但有误判。
```

### 4. 哈希函数个数和位数组大小

影响误判率的因素：

```text
1. 位数组越大，误判率越低，占用内存越多
2. 哈希函数太少，bit分布不充分，误判率高
3. 哈希函数太多，bit被置1太快，误判率也会上升
4. 数据量越接近设计容量，误判率越高
```

### 5. 误判的业务影响

误判是：

```text
不存在的数据被判断为可能存在
```

结果：

```text
请求会继续查Redis和DB
```

它不会导致存在的数据被拦截，所以不会误伤正常用户。

布隆过滤器的价值不是 100% 拦截，而是大幅减少无效 DB 查询。

### 6. 初始化

初始化流程：

```text
系统启动或定时任务
    ↓
分页扫描有效医生ID / 商品ID
    ↓
批量写入布隆过滤器
    ↓
记录版本号和构建时间
    ↓
切换为可用状态
```

注意：

```text
1. 不能一次性全表扫描打爆DB
2. 大表要分页、限速、走从库或离线任务
3. 初始化完成前需要用空值缓存和限流兜底
```

### 7. 新增数据同步

新增医生或商品后：

```text
写DB成功
    ↓
同步或异步添加ID到布隆过滤器
```

因为布隆过滤器添加是安全的，只会把更多 bit 置为 1。

### 8. 删除数据怎么办？

普通布隆过滤器不支持准确删除。

原因：

```text
某个bit位可能被多个元素共享。
如果删除一个元素时把bit置为0，可能影响其他元素判断。
```

解决方案：

```text
1. 删除业务不从布隆过滤器删除，依赖DB状态和缓存TTL
2. 定期全量重建布隆过滤器
3. 使用 Counting Bloom Filter 支持计数删除
4. 对删除频繁的数据，不适合用普通布隆过滤器做强判断
```

### 9. Redis Bitmap、Redisson BloomFilter、Guava BloomFilter

| 实现 | 适用场景 | 特点 |
|------|----------|------|
| Redis Bitmap | 分布式共享过滤器 | 多服务共享，需自己实现哈希和容量规划 |
| Redisson BloomFilter | Java 项目快速落地 | 封装好，支持预期插入量和误判率配置 |
| Guava BloomFilter | 单机本地过滤 | 性能高，但多节点不共享，重启需重建 |

### 10. 医疗问诊医生 ID 防穿透设计

```text
请求 doctorId
    ↓
参数校验：必须为正数、符合ID规则
    ↓
网关限流：IP/用户维度限制异常频率
    ↓
布隆过滤器判断
    ↓
一定不存在：直接返回医生不存在
    ↓
可能存在：查Redis医生详情
    ↓
Redis miss：查DB
    ↓
DB不存在：写空值缓存，TTL 1-5分钟
    ↓
DB存在：写医生详情缓存，TTL随机化
```

兜底：

```text
布隆过滤器不可用：
  1. 降级为参数校验 + 空值缓存 + 限流
  2. 限制DB回源并发
  3. 告警并启动重建任务
```

## 题目五：如果 Redis 整集群故障，大量请求打到数据库，如何做降级保护？

## 答案

### 1. Redis 不可用时会发生什么？

```text
请求进入应用
    ↓
访问Redis超时或连接失败
    ↓
缓存命中率骤降
    ↓
大量请求尝试回源DB
    ↓
DB连接池打满
    ↓
应用线程等待DB连接
    ↓
接口超时
    ↓
上游重试放大流量
    ↓
系统雪崩
```

### 2. 核心目标

Redis 故障时，第一目标不是保证所有请求都成功，而是：

> **保护数据库和核心链路，牺牲部分非核心体验，防止故障扩大。**

### 3. 组合防线

```text
1. Redis访问超时快速失败
2. Redis错误率升高后熔断
3. 非核心接口降级
4. 核心接口限流查DB
5. 本地缓存返回旧值
6. DB线程池隔离
7. 禁止无限重试
8. Redis恢复后分批预热
```

### 4. 哪些接口可以降级？

可以降级的接口通常是展示类、推荐类、非交易关键路径：

```text
商品详情展示
首页推荐
活动Banner
医生介绍
IM会话摘要
营销配置
排行榜
```

降级方式：

```text
返回旧值
返回默认值
隐藏模块
提示稍后再试
```

### 5. 哪些接口不能简单降级？

不能简单降级的是交易、资金、库存、状态变更类：

```text
支付
下单
扣库存
保单状态变更
问诊预约提交
资金路由决策
```

这些接口可以限流、排队、快速失败，但不能返回虚假成功。

### 6. 本地缓存兜底

适用：

```text
允许短暂旧值的热点读数据
```

例如：

```text
商品详情
医生简介
活动配置
保单详情展示
```

控制脏数据：

```text
1. 本地缓存TTL短一些
2. 设置最大容量
3. 只缓存热点Key
4. 变更时通过MQ或事件通知失效
5. 强一致动作绕过本地缓存
```

### 7. 线程池隔离

不能让 Redis 故障拖垮整个应用线程池。

```text
业务线程池
  ↓
缓存访问线程池
  ↓
DB回源线程池
  ↓
核心交易线程池
```

原则：

```text
1. 缓存重建线程池独立
2. DB回源并发受限
3. 非核心接口和核心接口隔离
4. 慢接口不能占满所有Tomcat工作线程
```

### 8. Redis 恢复后的预热

Redis 恢复后不能立即放开所有流量。

```text
Redis恢复
    ↓
健康检查连续成功
    ↓
按热点Key清单分批预热
    ↓
限制DB回源QPS
    ↓
逐步关闭降级开关
    ↓
恢复正常缓存读写
```

防止二次雪崩：

```text
1. 预热任务限速
2. TTL加随机值
3. 缓存重建限流
4. 不允许所有请求同时查DB回填缓存
```

### 9. 业务场景取舍

| 业务 | Redis故障时策略 | 原因 |
|------|----------------|------|
| 保险商城保单详情 | 可返回短暂旧值；用户刚修改后可查DB | 展示可旧，但写后读需要特殊处理 |
| IM在线客服会话列表 | 可返回本地旧列表或降级展示最近会话 | 可用性优先，消息发送链路单独保障 |
| 医疗问诊医生排班 | 展示可旧，预约提交必须查权威数据 | 排班展示和预约提交一致性要求不同 |
| 秒杀活动商品库存 | 展示可旧或降级，扣减链路限流/排队 | 展示值不是最终库存判断依据 |

## 今日总结

Redis 缓存异常治理的核心不是背三个定义，而是理解一条主线：

```text
缓存没有挡住流量
    ↓
请求回源DB
    ↓
DB连接池被打满
    ↓
应用线程阻塞
    ↓
接口超时和重试
    ↓
系统雪崩
```

架构师回答要能说清楚：

```text
1. 为什么会 miss
2. miss 后谁能回源
3. 回源并发怎么限制
4. 旧值能不能返回
5. 删除缓存失败怎么补偿
6. Redis整体不可用时如何保护DB
7. 不同业务的强一致边界在哪里
```

最重要的结论：

> **缓存是性能层，不是事实源；Redis 可以提升读性能和抗流量能力，但业务正确性必须由 DB、状态机、幂等、MQ 补偿和限流降级共同保障。**
