# Day 5 架构师梳理：Redis分布式锁与秒杀库存扣减

## 面试核心定位

Redis 分布式锁和秒杀库存扣减是 Redis 专题里最容易区分“会用”和“能设计”的题。

```text
普通开发：
  会说 SETNX、Redisson、库存放 Redis

高级开发：
  能说 SET NX PX、Lua 解锁、防误删、库存 Lua 扣减

架构师：
  能说清楚 Redis 锁的边界、主从切换风险、业务兜底、幂等状态机、对账补偿
```

面试官真正想听的不是“Redis 怎么加锁”，而是：

```text
1. 你是否理解锁在异常场景下会失效
2. 你是否知道 Redis 不能承担强一致最终正确性
3. 你是否能把 Redis、MQ、MySQL 的职责边界讲清楚
4. 你是否能给资金、订单、库存这类核心场景设计兜底
```

---

## 一、分布式锁的架构师思维

### 1.1 不要把“加锁成功”当成“业务安全”

很多回答会停在：

```text
SET key value NX PX 30000
```

这只是正确加锁的起点，不是完整答案。

架构师需要继续追问：

```text
1. value 是否唯一？
2. 解锁是否原子？
3. 锁过期了业务还没执行完怎么办？
4. Redis 主从切换会不会丢锁？
5. 这个业务允许锁偶发失效吗？
6. 如果锁失效，数据库有没有兜底？
```

核心认知：

```text
Redis 锁降低并发冲突，不保证强一致正确性。
```

### 1.2 Redis锁的四条底线

```text
第一条：加锁必须原子
  SET lockKey requestId NX PX expireMillis

第二条：解锁必须校验归属
  value 是自己的才能删

第三条：判断 + 删除必须原子
  用 Lua 脚本，不能 GET 后 DEL

第四条：核心业务必须有兜底
  DB唯一索引、条件更新、幂等表、状态机
```

如果少了第四条，回答仍然偏“中间件使用者”，不是“架构设计者”。

## 二、SETNX 裸实现为什么常被追问

### 2.1 面试官问 SETNX，本质是在问异常意识

`SETNX + EXPIRE` 的问题不是语法问题，而是异常链路问题：

```text
SETNX 成功
  ↓
应用宕机
  ↓
EXPIRE 没执行
  ↓
锁永久存在
  ↓
业务不可用
```

这类问题考察的是：

```text
你能不能发现“两条命令之间”的失败窗口。
```

架构师需要对所有“分步骤完成一致性”的方案保持警惕：

```text
1. 先查库存，再扣库存
2. 先删缓存，再更新DB
3. 先写DB，再发MQ
4. 先GET锁，再DEL锁
5. SETNX之后再EXPIRE
```

这些方案共同问题是：

```text
中间任何一步失败，都可能留下不一致状态。
```

### 2.2 一句话升级表达

普通表达：

```text
SETNX + EXPIRE 不是原子的，可能死锁。
```

架构师表达：

```text
SETNX + EXPIRE 分两步存在失败窗口，如果 SETNX 成功后应用宕机，锁没有 TTL，会造成永久死锁。所以加锁必须使用 SET key value NX PX expireMillis，在 Redis 端一次性完成互斥判断和过期时间设置。
```

## 三、Redisson不是银弹

### 3.1 Redisson解决了哪些工程问题

Redisson 相比裸写 Redis 锁，主要解决：

```text
1. 自动生成唯一锁标识
2. Lua 脚本安全解锁
3. watchdog 自动续期
4. 可重入锁
5. 公平锁、读写锁、联锁等高级封装
```

所以在普通业务里，优先 Redisson 是合理的。

### 3.2 Redisson没有解决哪些一致性问题

但 Redisson 没有把 Redis 变成 CP 系统：

```text
1. Redis主从复制仍然是异步的
2. master宕机时仍可能丢锁
3. JVM长时间STW可能导致续期失败
4. 网络分区下客户端可能误以为自己仍持锁
5. 锁失效后，业务如果没有DB兜底，仍可能出错
```

面试中要主动补一句：

```text
Redisson解决的是工程易用性和续期问题，不解决Redis异步复制带来的强一致问题。
```

这句话能明显拉开层次。

## 四、分布式锁按业务风险分级

架构师不能只问“能不能用 Redis 锁”，而要按业务风险分级。

| 场景 | 能否用 Redis 锁 | 是否需要强兜底 | 推荐方案 |
|------|----------------|----------------|----------|
| 缓存击穿重建 | 可以 | 不强制 | Redisson / SET NX PX |
| 定时任务抢占 | 可以 | 任务幂等 | Redis锁 + 任务状态 |
| 防重复点击 | 可以 | 需要 | Redis锁 + 幂等号 |
| 秒杀预扣库存 | 可以 | 必须 | Redis Lua + MQ + DB条件更新 |
| 订单创建 | 不能只依赖 | 必须 | 唯一索引 + 幂等表 |
| 账户扣款 | 不能只依赖 | 必须 | DB事务 + 状态机 + 对账 |
| 资金路由 | 不能只依赖 | 必须 | 幂等表 + 状态流转 + 补偿 |

判断标准：

```text
如果锁偶发失效只会造成重复计算或短暂延迟，可以用 Redis 锁。
如果锁偶发失效会造成资损、超卖、重复订单，就不能只靠 Redis 锁。
```

## 五、秒杀库存的职责拆分

### 5.1 秒杀不是“Redis扣库存”这么简单

完整秒杀链路：

```text
网关限流
  ↓
活动校验
  ↓
Redis Lua 判断库存 + 防重复 + 预扣
  ↓
发送 MQ
  ↓
异步创建订单
  ↓
MySQL 条件扣减库存
  ↓
待支付订单
  ↓
支付成功确认 / 超时取消释放库存
  ↓
对账补偿
```

每一层解决的问题不同：

```text
网关：挡掉明显超出系统容量的流量
Redis：快速判断资格和库存，抗高并发
MQ：削峰填谷，保护订单系统和数据库
MySQL：保存最终事实，防止超卖和重复订单
补偿：修复链路中间失败造成的不一致
```

### 5.2 Redis库存不是最终库存

面试中一定要讲清楚：

```text
Redis库存是预扣库存，用来挡流量。
MySQL库存是最终库存，用来保证正确性。
```

如果 Redis 扣减成功但后续失败，必须释放或补偿。

如果 MySQL 扣减失败，订单不能成立。

这就是“高性能”和“正确性”的分工。

## 六、为什么 Lua 是秒杀库存的核心

### 6.1 多条命令的问题

如果不用 Lua，应用层可能这样做：

```text
1. GET stock
2. 判断 stock > 0
3. SISMEMBER 判断用户是否买过
4. DECR stock
5. SADD userId
```

这些命令之间有并发窗口：

```text
库存还有1件
  ↓
用户A和用户B同时 GET stock = 1
  ↓
A、B都判断可以买
  ↓
A、B都 DECR
  ↓
库存变成 -1 或重复扣减
```

### 6.2 Lua 的价值

Lua 把多个动作放到 Redis 单线程里一次执行：

```text
判断是否重复
判断库存是否充足
扣减库存
记录用户已购买
```

价值不是“写法高级”，而是：

```text
1. 原子性：避免并发窗口
2. 性能：减少网络往返
3. 简化补偿：返回明确结果码
4. 降低DB压力：大量失败请求不进入下游
```

## 七、防超卖的三道防线

真正的防超卖不是靠单点，而是三道防线。

### 第一防线：Redis Lua 预扣

```text
作用：挡住绝大多数无库存和重复请求
特点：高性能，但不是最终事实来源
```

### 第二防线：MySQL 条件更新

```sql
update product_stock
set stock = stock - 1
where sku_id = #{skuId}
  and stock > 0;
```

```text
作用：防止库存扣成负数
特点：最终正确性兜底
```

### 第三防线：唯一索引 / 幂等表

```sql
create unique index uk_sku_user on seckill_order(sku_id, user_id);
```

```text
作用：防止同一用户重复下单
特点：应对重复请求、重复消息、接口重试
```

架构师表达：

```text
Redis负责前置拦截，MySQL负责最终裁决，MQ负责削峰，补偿任务负责修复中间态。
```

## 八、库存回补与补偿设计

### 8.1 为什么必须有补偿

分布式链路一定会出现中间失败：

```text
Redis预扣成功，但MQ发送失败
MQ发送成功，但消费者失败
订单创建成功，但MySQL扣库存失败
用户下单成功，但一直未支付
库存释放成功，但Redis回补失败
```

如果没有补偿，系统会出现：

```text
1. Redis库存越来越少，但实际订单没那么多
2. 用户被标记已购买，但订单不存在
3. MySQL库存和Redis库存长期不一致
4. 活动提前显示售罄
```

### 8.2 补偿设计关键点

```text
1. 记录预扣流水
   requestId、skuId、userId、status、expireTime

2. 状态机驱动
   RESERVED → ORDER_CREATED → PAID
   RESERVED → RELEASED
   ORDER_CREATED → CANCELED → RELEASED

3. 所有补偿动作幂等
   重复释放不能多加库存
   重复取消不能重复回补

4. 定时对账
   对比活动库存、订单数量、支付数量、释放数量
```

架构师不是假设每一步都成功，而是默认每一步都可能失败，并设计恢复路径。

## 九、和用户业务经验的结合表达

你有几个很适合拿来类比的业务背景：

```text
内部管理系统 SETNX 裸实现：
  可以讲早期方案存在 SETNX+EXPIRE 非原子、解锁误删风险，后来如何改进。

ToC 核心业务 Redisson：
  可以讲 Redisson watchdog、Lua解锁、可重入锁，但也会补 DB兜底。

完整秒杀系统经验：
  可以讲 Redis Lua 预扣、MQ异步下单、MySQL条件扣减、延迟消息释放库存。

国美e卡资金路由：
  可以强调资金类场景不能只靠 Redis 锁，必须靠幂等表、状态机、数据库约束和对账。

蛋壳钱包 TCC：
  可以说明核心交易一致性应靠 TCC/事务状态推进，Redis 锁只能做并发控制辅助。
```

这类回答能体现“不是背八股，而是知道不同业务风险下的架构取舍”。

## 十、面试高分表达模板

### 10.1 Redis分布式锁模板

```text
Redis分布式锁我会先区分使用场景。如果只是缓存重建、定时任务抢占这类短时互斥，可以用 Redis 锁，推荐 Redisson 或 SET key value NX PX expireMillis。value 必须是唯一请求标识，解锁必须用 Lua 判断 value 后再删除，避免锁过期后误删别人的锁。

如果业务执行时间不确定，Redisson watchdog 可以自动续期，避免业务没执行完锁先过期。但它也有边界，比如 JVM STW、网络抖动、Redis 主从切换都可能导致锁失效。尤其 Redis 主从复制是异步的，master 加锁后还没同步到 slave 就宕机，新 master 上可能没有锁。

所以我不会把 Redis 锁作为资金、订单、库存最终正确性的唯一保障。核心业务一定会加数据库唯一索引、条件更新、幂等表或状态机兜底。Redis 锁只是降低并发冲突，最终正确性要靠强约束保证。
```

### 10.2 秒杀库存模板

```text
秒杀库存我会分层设计。入口层先限流和校验活动资格，Redis 层用 Lua 脚本原子完成判断库存、判断用户是否重复、扣减库存、记录购买用户，挡住绝大多数并发请求。Redis 扣减成功后发送 MQ，由订单服务异步创建订单，MySQL 用 stock > 0 的条件更新做最终扣减，并用 sku_id + user_id 唯一索引防重复下单。

Redis 库存只是预扣库存，不是最终事实来源。链路中 Redis 扣减成功但 MQ 失败、订单失败、支付超时都要设计释放库存和补偿机制。未支付订单通过延迟消息关闭，只有订单状态从 WAIT_PAY 更新为 CANCELED 成功后才能释放库存，保证释放幂等。

所以这个方案的核心不是单纯用 Redis 扣库存，而是 Redis 抗并发、MQ 削峰、MySQL 兜底正确性、延迟消息和对账任务修复中间失败。
```

## 十一、常见追问清单

```text
1. SETNX + EXPIRE 为什么不安全？
2. SET NX PX 中 value 为什么不能固定写 1？
3. 解锁 Lua 脚本怎么写？
4. Redisson watchdog 多久续期一次？
5. 指定 leaseTime 后 watchdog 还会不会生效？
6. Redis 主从切换为什么可能导致锁丢失？
7. RedLock 有什么争议？
8. 秒杀库存为什么不能只用 DECR？
9. Lua 脚本扣库存怎么防重复购买？
10. Redis 扣库存成功但 MQ 发送失败怎么办？
11. MySQL 怎么防止最终超卖？
12. 未支付订单释放库存如何保证幂等？
13. Redis库存和MySQL库存不一致怎么对账？
14. 资金类业务能不能用 Redis 锁保证不重复处理？
```

如果这些追问都能答清楚，Redis 专题基本就从“会用缓存”升级到“能设计高并发一致性方案”。
