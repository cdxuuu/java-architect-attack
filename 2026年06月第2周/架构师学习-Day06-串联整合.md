## 核心考题：高并发秒杀系统架构设计

### 场景描述

```text
秒杀活动开始后，预计 10 秒内涌入 100 万请求。
商品库存 1 万件，每个用户只能购买 1 件。

要求：
- 不能超卖（库存不能多扣）
- 不能重复下单（同一用户不能生成多笔订单）
- 防止接口被刷（恶意流量拦截）
- 数据库不能被打爆
- 下单成功后异步生成订单
- 允许少量用户看到"排队中"
- 需要对账补偿系统保证最终一致
```

---

### 架构师方案：分层削峰 + 最终一致性

```text
用户
 ↓
前端 / CDN：静态化页面、按钮置灰、倒计时控制
 ↓
网关层：限流、防刷、Token 校验、黑名单、风控
 ↓
秒杀服务
 ↓
Redis + Lua：库存判断 + 库存预扣 + 用户去重
 ↓
MQ：削峰填谷
 ↓
订单服务：消息消费 + 幂等判断 + 订单创建
 ↓
DB：条件扣减库存 + 唯一索引防重
 ↓
超时关闭订单 / 库存回补
 ↓
对账补偿系统：Redis库存 + 订单 + DB库存 + MQ在途
```

### 本周知识点串联

| 知识点 | 在秒杀场景中的应用 |
|--------|-------------------|
| **String 做计数（Day1）** | `DECR` 命令做库存自减，`SET` 命令做用户购买标记 |
| **单线程 / Lua 原子性（Day1）** | Lua 脚本保证库存判断 + 扣减 + 用户去重三者原子执行 |
| **主从切换丢数据（Day2）** | Redis 主从切换可能导致库存 Key 和用户 Key 丢失，需要 DB 索引 + 条件更新兜底 |
| **缓存一致性（Day3）** | Redis 库存是前置预扣，DB 库存是最终事实，通过 MQ + 对账保证最终一致 |
| **热 Key 治理（Day4）** | 秒杀库存 Key 是典型的超高热 Key，可以用库存分片或本地缓存兜底读 |
| **内存优化（Day4）** | 用户购买标记可以用 `SETBIT` 位图或短过期时间减少内存占用 |
| **SET NX PX 锁（Day5）** | 秒杀接口本身不是用分布式锁控制并发，而是 Lua 预扣 + MQ 削峰 |
| **Redis Lua 预扣（Day5）** | 预扣成功才进 MQ，失败直接返给用户 |
| **DB 条件更新兜底（Day5）** | 即使 Redis 出现异常溢出，DB 层 `where stock > 0` 保证不超卖 |

### 一、Redis Lua 库存预扣

秒杀系统的核心并发控制入口。

秒杀开始前，将活动库存预热到 Redis：

```text
seckill:stock:{activityId}:{skuId} = 10000
```

用户请求到达秒杀服务后，执行 Lua 脚本：

```lua
local stockKey = KEYS[1]
local userKey = KEYS[2]

-- 判断用户是否已参与
if redis.call("exists", userKey) == 1 then
    return 2   -- 重复下单
end

-- 判断库存是否充足
local stock = tonumber(redis.call("get", stockKey))
if stock == nil or stock <= 0 then
    return 0   -- 库存不足
end

-- 扣减库存 + 记录用户标记
redis.call("decr", stockKey)
redis.call("setex", userKey, 86400, 1)

return 1  -- 成功
```

返回含义：

```text
1 → 预扣成功，进入 MQ 下单流程
0 → 库存已售罄
2 → 已参与过秒杀
```

Lua 脚本在 Redis 单线程中原子执行，消除了并发判断和扣减之间的竞争窗口。

### 二、MQ 异步削峰

预扣成功后的流程不能同步完成，否则数据库和订单服务会被打爆。

```text
Redis 预扣成功
  ↓
发送 MQ 消息（消息体包含 activityId、skuId、userId、requestId）
  ↓
秒杀服务立即返回"抢购成功，正在处理订单"
  ↓
订单服务按自身消费能力拉取消息
  ↓
每条消息处理：幂等判断 → 创建订单 → 扣减 DB 库存
```

#### 生产端：消息不丢

RocketMQ 同步发送，开启 confirm 回调。

如果发送失败：

```text
第 1~3 次：MQ 内置重试
后续仍失败：落本地消息表，状态 = 待发送，触发告警
补偿任务定时扫描本地消息表，重新投递
```

更严谨的做法：

```text
本地消息表的写入和业务操作在同一个本地事务中提交。
```

这样即使秒杀服务宕机，消息表记录也不会丢失。

#### 消费端：幂等

RocketMQ 是“至少一次投递”，订单服务必须处理重复消息。

```text
第一层：Redis 幂等 Key →
  以 requestId 或 activityId + userId 为 Key
  消费前 SETNX，成功则处理，失败则直接返回

第二层：DB 唯一索引 →
  订单表 unique(activity_id, user_id)
  即使 Redis 幂等 Key 丢失，DB 也防重
```

失败处理：

```text
消费处理异常 → 不 ACK → MQ 重试
重试 16 次仍失败 → 死信队列
死信队列 → 补偿任务或人工处理
```

### 三、DB 库存扣减兜底

订单服务创建订单后，扣减真实库存。

不能先查再扣，必须使用条件更新：

```sql
update sku_stock
set stock = stock - 1
where sku_id = ?
  and activity_id = ?
  and stock > 0;
```

```text
影响行数 = 1：扣减成功，订单生效
影响行数 = 0：库存不足，订单置为失败
```

DB 层的条件更新是防超卖的最终防线。

### 四、全链路防重复下单

秒杀系统最怕用户通过快速点击或脚本刷出多个订单。

分层防护：

```text
前端层：按钮置灰，请求返回前不可重复点击
网关层：用户维度限流，1 秒内相同请求拦截
Redis 层：Lua 脚本中判断用户标记 Key 是否已存在
DB 层：唯一索引 unique(activity_id, user_id)
```

任何一层的松动，都有下层兜住。

### 五、订单超时未支付

秒杀成功后的订单通常需要限时支付，超时后关闭订单并回补库存。

推荐方案：定时任务扫描

```sql
-- 建索引
create index idx_order_status_expire
on seckill_order(order_status, expire_time);

-- 扫描
select *
from seckill_order
where order_status = 'WAIT_PAY'
  and expire_time <= now()
limit 1000;
```

批量关闭订单并回补库存：

```text
订单状态更新为 CANCELED
Redis 库存 +1（回补）
DB 库存 +1（回补）
记录库存回补流水
```

### 六、Redis 故障降级

如果 Redis 在秒杀期间挂了怎么办？

绝对不能让请求直接打到 DB，否则数据库会被瞬间打垮。

降级策略：

```text
秒杀入口熔断：直接返回"活动火爆"
网关限流进一步收紧：丢弃大部分请求
允许少量白名单流量进入（如已预热的灰度用户）
```

本地缓存不能用于真实扣减：

```text
本地缓存无法保证多实例之间库存一致性
多实例并发扣减必然超卖
```

降级原则：

```text
宁可少卖，不超卖
宁可让用户排队，不打爆数据库
```

### 七、对账补偿系统

秒杀结束后，需要通过对比多端数据发现并修复不一致。

核心对账公式：

```text
初始库存
= DB 已成功扣减库存
+ Redis 剩余库存
+ 待支付有效订单数
+ MQ 在途消息数
+ 预扣成功但未落库记录
```

异常类型和补偿：

| 异常 | 处理方式 |
|------|----------|
| Redis 预扣成功，MQ 发送失败 | 本地消息表重新投递，或回补 Redis 库存 |
| MQ 重复投递 | Redis 幂等 Key + DB 唯一索引兜底 |
| 订单创建成功，DB 扣库存失败 | 订单置失败，回补 Redis 库存 |
| 超时未支付 | 关闭订单，回补 Redis + DB 库存 |
| Redis 库存比预期多 | 检查是否有预扣未落库的记录 |
| Redis 库存比预期少 | 检查是否有重复扣减或回补缺失 |

### 八、架构师思考：秒杀系统的职责拆分

#### 8.1 不要把秒杀当成一个"Redis 问题"

秒杀系统的核心不是 Redis，而是分层：

```text
Redis：高并发库存预扣，但最终正确性不依赖 Redis
MQ：异步削峰，但不保证顺序，消费端必须幂等
DB：最终事实存储，通过条件更新和唯一索引兜底
对账：异常链路最终修复，保证数据收敛
```

每个组件只负责自己擅长的部分：

```text
Redis 负责快
MQ 负责稳
DB 负责准
对账负责全
```

#### 8.2 强一致和最终一致的边界

必须强一致的地方：

```text
Redis Lua 扣库存 + 用户去重：原子执行
DB 扣库存：条件更新，不能先查再扣
订单表唯一索引：防重复订单
支付状态变更：幂等 + 状态机
```

可以最终一致的地方：

```text
Redis 库存和 DB 库存的同步
MQ 消息投递和消费
库存回补
订单超时关闭
对账补偿
```

#### 8.3 秒杀系统常见的失败认知

| 错误认知 | 正确理解 |
|----------|----------|
| Redis 扣库存就能保证不超卖 | Redis 只是预扣，DB 条件更新才是最终兜底 |
| MQ 保证不丢消息就够了 | 消费端必须幂等，MQ 是至少一次投递 |
| 唯一键去重就够了 | 唯一键是兜底，高并发下要先走 Redis 去重挡流量 |
| Redis 挂了系统还能正常秒杀 | Redis 挂了必须降级，不能直打 DB |
| 活动结束就算完了 | 活动结束后的对账才是保证不资损的关键 |
