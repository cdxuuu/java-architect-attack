# Day 3 架构师梳理：Redis缓存异常与一致性治理

## 面试核心定位

Redis 缓存异常治理是架构师面试高频主题。面试官通常不是想听“穿透、击穿、雪崩”的定义，而是想判断你是否真正理解：

```text
1. Redis miss 后流量是如何打穿 DB 的？
2. 为什么热点 Key 不能简单依赖 TTL？
3. 为什么缓存一致性通常是最终一致，而不是强一致？
4. 删除缓存失败后怎么补偿？
5. Redis 整体故障时，系统如何保护核心链路？
```

核心结论先记住：

> **缓存是性能层，不是事实源；缓存治理的核心不是让所有请求都查到最新值，而是在性能、可用性、一致性之间做业务取舍，并保护 DB 不被打穿。**

---

## 一、缓存异常的统一视角：缓存层失去削峰能力

穿透、击穿、雪崩表面上是三个问题，本质上都是缓存层没有挡住流量。

```text
用户请求
  ↓
Redis缓存层
  ↓ miss / 过期 / 故障
MySQL数据库
  ↓
连接池打满
  ↓
慢查询堆积
  ↓
应用线程阻塞
  ↓
接口超时
  ↓
上游重试
  ↓
流量进一步放大
```

架构师回答时，要从“定义”升级为“链路推演”。

### 1. 穿透：不存在的数据持续穿过缓存

```text
查不存在的数据
    ↓
Redis没有
    ↓
DB也没有
    ↓
没有缓存空结果
    ↓
下次继续查DB
```

关键点：

```text
穿透不是缓存过期，而是缓存体系没有表达“不存在”。
```

治理原则：

```text
越靠前拦截越好。
```

分层防线：

```text
网关限流/防刷
  ↓
参数校验
  ↓
布隆过滤器
  ↓
Redis空值缓存
  ↓
DB查询限流
```

### 2. 击穿：热点 Key 失效导致单点高并发回源

```text
热点Key过期
    ↓
10万QPS同时Redis miss
    ↓
大量线程同时查DB
    ↓
DB连接池打满
```

关键点：

```text
击穿的危险不在于一次 miss，而在于热点 Key 的并发放大。
```

治理原则：

```text
热点 Key 不能让所有请求同时参与缓存重建。
```

典型方案：

```text
互斥锁：保证单线程回源
逻辑过期：永远返回缓存值，异步刷新
本地缓存：挡住极端热点
预热：活动前提前加载
限流：限制DB回源并发
```

### 3. 雪崩：缓存层整体失效

```text
大量Key同时过期 / Redis整体不可用
    ↓
缓存命中率骤降
    ↓
大量请求回源DB
    ↓
DB被打爆
```

关键点：

```text
雪崩不只是 TTL 一致，也包括 Redis 集群故障、网络抖动、连接池耗尽。
```

治理原则：

```text
不能假设 Redis 永远可用；Redis 不可用时必须保护 DB。
```

典型方案：

```text
随机TTL
Redis高可用
本地缓存兜底
限流熔断降级
DB线程池隔离
恢复后分批预热
```

## 二、互斥锁 vs 逻辑过期：本质是“一致性”和“可用性”的取舍

### 互斥锁适合什么？

互斥锁的目标是：

```text
缓存 miss 后，只允许一个线程回源 DB。
```

流程：

```text
Redis miss
  ↓
SET lock NX EX
  ↓
成功：查DB并回填缓存
失败：等待 / 重试 / 降级
```

适合：

```text
1. QPS不是极端高
2. 不能接受明显旧值
3. 回源耗时可控
4. 请求可以短暂等待
```

例如：

```text
保单详情
订单详情
用户资料
医生排班详情
```

风险：

```text
1. 锁等待导致应用线程堆积
2. 锁TTL过短会并发回源
3. 锁TTL过长会阻塞重建
4. 持锁线程挂掉需要TTL兜底
5. 解锁必须校验owner，不能直接DEL
```

### 逻辑过期适合什么？

逻辑过期的目标是：

```text
热点 Key 不物理删除，请求永远能拿到一个值。
```

流程：

```text
读取Redis命中
  ↓
检查value中的expireTime
  ↓
未过期：返回
  ↓
已过期：尝试异步刷新
  ↓
当前请求返回旧值
```

适合：

```text
1. 极高并发热点读
2. 允许短暂旧值
3. 读多写少
4. 活动、商品、配置类数据
```

例如：

```text
秒杀商品详情
首页Banner
热门医生介绍
商品详情页
活动配置
```

风险：

```text
1. 可能返回旧值
2. 需要异步刷新机制
3. 刷新失败后旧值可能持续存在
4. 强一致场景不能直接使用
```

### 架构师取舍模板

```text
如果这是展示型热点数据，我优先选择逻辑过期，因为它能保证高并发下请求不阻塞、不批量回源，代价是可能短暂返回旧值。

如果这是用户强一致读数据，我会选择互斥锁或直接读DB，因为旧值可能影响业务判断，但要配合锁TTL、owner校验、回源限流和线程池隔离，避免锁等待拖垮应用。
```

## 三、布隆过滤器：解决穿透，但不是银弹

### 布隆过滤器的面试重点

不要只说“布隆过滤器可以防穿透”，要讲清楚三个结论：

```text
1. 它能判断一定不存在
2. 它不能判断一定存在，只能判断可能存在
3. 它有误判，但没有漏判
```

### 为什么没有漏判？

如果一个 ID 真实加入过布隆过滤器，它对应的多个 bit 位一定被置为 1。

查询时只要有一个 bit 位为 0，就说明这个 ID 一定没有加入过。

### 为什么有误判？

多个不同 ID 的哈希位可能重叠。

```text
不存在的ID，刚好命中了其他ID设置过的所有bit位
    ↓
布隆过滤器判断：可能存在
    ↓
继续查Redis/DB
```

误判的影响：

```text
误判只会让少量不存在数据继续查DB，不会把真实存在的数据拦掉。
```

### 布隆过滤器的工程问题

架构师要补充这些细节：

```text
1. 初始数据怎么加载？
2. 新增数据怎么同步？
3. 删除数据怎么办？
4. 误判率怎么控制？
5. 布隆过滤器不可用怎么兜底？
```

工程方案：

```text
初始化：分页扫描有效ID，批量写入布隆过滤器
新增：DB写成功后同步/异步add
删除：普通布隆过滤器不删除，定期重建或使用Counting Bloom Filter
兜底：参数校验 + 空值缓存 + 限流
```

## 四、缓存一致性：主流是“先更新 DB，再删除缓存”

### 为什么不是先删缓存再更新 DB？

并发读会把旧值写回缓存。

```text
写请求A：删除缓存
  ↓
读请求B：缓存miss
  ↓
读请求B：查DB，读到旧值
  ↓
读请求B：旧值写入缓存
  ↓
写请求A：更新DB为新值
```

结果：

```text
DB新，缓存旧。
```

### 为什么推荐先更新 DB，再删除缓存？

```text
写请求A：更新DB
  ↓
写请求A：删除缓存
  ↓
读请求B：缓存miss
  ↓
读请求B：查DB新值
  ↓
回填缓存
```

只要删除缓存成功，后续读就能读到新值。

主要风险是：

```text
DB更新成功，缓存删除失败。
```

这个风险可以补偿。

### 删除缓存失败怎么补偿？

```text
同步重试
  ↓
MQ异步重试
  ↓
死信队列
  ↓
补偿任务
  ↓
binlog订阅兜底
```

架构师表达：

> **缓存一致性不是靠一次删除保证的，而是靠“删除 + 重试 + 补偿 + 监控”把不一致窗口控制在业务可接受范围内。**

### 为什么是删除缓存，不是更新缓存？

更新缓存有三个问题：

```text
1. 并发写可能旧值覆盖新值
2. 缓存通常是聚合数据，更新逻辑复杂
3. 写多读少时更新缓存浪费资源
```

删除缓存更简单：

```text
写时删除，读时重建。
```

## 五、最终一致 vs 强一致：按业务链路拆分

Redis 缓存一致性通常是最终一致。

但业务系统里不是所有读都要求强一致。

### 展示读可以最终一致

例如：

```text
商品详情展示
医生简介
保单详情展示
活动配置
```

可以接受：

```text
短时间旧值
本地缓存旧值
逻辑过期旧值
```

### 决策读必须强校验

例如：

```text
下单
支付
扣库存
预约挂号
保单状态变更
资金路由
```

必须：

```text
查DB主库
查权威库存服务
走状态机
做幂等校验
做版本号/CAS判断
```

### 典型拆分：医生排班

```text
医生排班页面展示：
  可以读缓存，允许短暂旧值。

用户提交预约：
  必须查权威号源，做事务扣减或状态校验。
```

### 典型拆分：秒杀库存

```text
商品详情页库存展示：
  可以返回旧值，比如“库存紧张”。

下单扣库存：
  必须走Redis原子扣减/Lua脚本 + DB/MQ最终落库 + 幂等防超卖。
```

### 典型拆分：保单地址

```text
保单详情展示：
  普通用户刷新可以最终一致。

用户刚修改地址后的下一次查看：
  需要读己之写，可以绕过缓存查DB一次。
```

## 六、Redis 整体故障：降级的目标是保护 DB

Redis 集群故障时，最怕的是应用把所有请求都转发给 DB。

错误做法：

```text
Redis失败
  ↓
所有请求直接查DB
```

正确做法：

```text
Redis失败率升高
  ↓
熔断Redis访问
  ↓
非核心接口降级
  ↓
核心接口限流查DB
  ↓
允许旧值的接口读本地缓存
  ↓
保护DB连接池
```

### 降级分类

| 类型 | 策略 |
|------|------|
| 展示类 | 返回旧值、默认值、隐藏模块 |
| 查询类 | 限流查 DB，超出快速失败 |
| 交易类 | 排队、限流、直接提示稍后重试 |
| 资金类 | 不降级为成功，必须走权威存储 |

### 线程池隔离

Redis 故障时，缓存重建不能占满业务线程。

```text
Tomcat业务线程池
  ↓
缓存访问线程池
  ↓
缓存重建线程池
  ↓
DB访问线程池
```

核心原则：

```text
1. 缓存重建独立线程池
2. DB回源必须限流
3. 非核心接口不能抢核心接口资源
4. Redis超时时间要短，不能长时间阻塞
```

## 七、业务场景落地

### 保险商城

```text
商品详情/保单详情展示：
  Redis缓存 + 随机TTL + 删除缓存补偿 + 本地缓存兜底。

修改保单地址：
  先更新DB，再删除缓存；删除失败MQ重试。

用户刚修改后刷新：
  当前用户短时间绕过缓存读DB，保证读己之写。

下单/支付/状态变更：
  不依赖缓存做最终判断，必须读权威数据。
```

### 医疗问诊

```text
医生详情：
  doctorId参数校验 + 布隆过滤器 + 空值缓存。

热门医生：
  逻辑过期 + 异步刷新。

排班展示：
  可以缓存。

预约提交：
  必须查权威号源，事务扣减，防止超卖。
```

### 秒杀系统

```text
活动前：
  热点商品、活动配置、库存展示预热。

活动中：
  本地缓存 + Redis + 逻辑过期。

缓存重建：
  异步刷新 + DB回源限流。

下单链路：
  Redis Lua原子扣库存 + 幂等 + MQ削峰 + DB最终落库。
```

### IM 在线客服

```text
会话列表：
  可以本地缓存短时间兜底。

消息发送：
  不能因为缓存降级而丢消息，必须落消息存储或可靠队列。

在线状态：
  可以最终一致，短暂不准可接受。
```

## 八、面试高频追问

### 追问1：加锁能不能完全解决击穿？

不能。

```text
1. 锁会增加延迟
2. 等锁线程可能堆积
3. 锁失效可能并发回源
4. 热点Key很多时锁数量也很多
5. Redis锁本身也可能故障
```

所以要组合：

```text
逻辑过期 + 本地缓存 + 回源限流 + 异步刷新 + 监控告警
```

### 追问2：Redis 和 DB 如何做到强一致？

严格强一致很难，通常不让 Redis 参与强一致判断。

```text
强一致读：查DB主库或权威服务
高性能展示：查Redis
```

### 追问3：删除缓存失败怎么办？

不能只说“重试”。要说完整链路：

```text
同步重试 → MQ异步重试 → 死信队列 → 补偿任务 → 监控告警
```

### 追问4：布隆过滤器误判怎么办？

误判只是让不存在的数据继续查 DB，不会误伤真实存在的数据。

治理：

```text
合理设置容量和误判率
监控误判导致的DB回源
定期重建
配合空值缓存和限流
```

### 追问5：Redis 恢复后为什么不能马上放开流量？

因为缓存是冷的。

```text
Redis恢复
  ↓
缓存大量缺失
  ↓
请求同时回源DB重建
  ↓
DB再次被打爆
```

所以要：

```text
分批预热 + 回源限流 + TTL随机化 + 逐步关闭降级
```

## 九、架构师金句

```text
1. 缓存是性能层，不是事实源。

2. 穿透是不存在的数据绕过缓存，击穿是热点Key失效，雪崩是缓存层整体失效。

3. 热点Key不要简单依赖物理TTL，高并发场景优先考虑逻辑过期和异步刷新。

4. 缓存一致性不是靠一次删除保证的，而是靠删除、重试、补偿、监控共同保障最终一致。

5. 删除缓存优于更新缓存，因为缓存通常是聚合视图，更新更容易出现并发覆盖和逻辑遗漏。

6. Redis故障时，第一目标不是让所有请求都成功，而是保护DB和核心链路。

7. 展示读可以接受最终一致，交易决策读必须回到权威数据源。

8. 布隆过滤器能判断一定不存在，不能判断一定存在。

9. 空值缓存要短TTL，否则可能挡住后续新增数据。

10. 逻辑过期牺牲短暂一致性，换取热点场景下的高可用和低延迟。
```

最终记住一句话：

> **Redis 架构设计的核心不是“怎么让缓存永远正确”，而是“哪些数据可以最终一致，哪些链路必须强一致，以及缓存失效时如何保护系统不被打穿”。**
