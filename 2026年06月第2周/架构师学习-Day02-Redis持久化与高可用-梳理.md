# Day 2 架构师梳理：Redis持久化与高可用

## 面试核心定位

Redis 高可用是架构师面试的高频深挖区。面试官通常不是想听“RDB 是快照、AOF 是日志”这种定义，而是想判断你有没有真正理解：

```
1. Redis 到底能不能丢数据？
2. Redis 高可用和 MySQL 高可用有什么本质区别？
3. Sentinel 自动切换为什么仍然可能丢数据？
4. Cluster 分片解决了什么，又没解决什么？
5. 什么时候不该上 Cluster？
```

核心结论先记住：

> **Redis 高可用解决的是服务恢复，不等于数据零丢失；Redis 可以做高性能访问层，但核心业务事实源必须有 DB 或日志兜底。**

---

## 一、持久化的本质：恢复点与恢复速度的权衡

### RDB 和 AOF 不是谁替代谁

很多人会说：“AOF 更安全，所以生产用 AOF。”这句话只对了一半。

架构师要这样理解：

```
RDB解决的问题：
  如何快速恢复一个较大的数据集？

AOF解决的问题：
  如何尽量减少最近写入的数据丢失？

混合持久化解决的问题：
  如何同时兼顾恢复速度和数据新鲜度？
```

所以二者不是简单的替代关系，而是两个维度：

| 维度 | RDB | AOF |
|------|-----|-----|
| 恢复速度 | 快 | 慢，命令越多越慢 |
| 数据新鲜度 | 差，取决于快照频率 | 好，everysec 最多丢 1 秒 |
| 文件大小 | 小 | 大，需要 rewrite |
| 写入开销 | 低 | 中等，取决于 fsync |
| 备份友好度 | 高 | 一般 |
| 宕机恢复点 | 分钟级 | 秒级 |

### 持久化要按数据语义分层

Redis 里常见数据可以分成四类：

```
第一类：纯缓存数据
  例：商品详情、店铺信息、配置副本
  特点：DB 有原始数据，Redis 丢了能回源
  策略：可以不开持久化，或只开 RDB 用于快速预热

第二类：临时状态数据
  例：验证码、限流计数、临时 Token
  特点：丢了影响体验，但不影响资金或事实一致性
  策略：可不开持久化，必要时 RDB

第三类：业务加速视图
  例：资金路由评分、排行榜、库存预扣视图
  特点：Redis 是高性能视图，DB/MQ/日志有事实源
  策略：AOF everysec 或 RDB + DB 兜底

第四类：关键业务防线
  例：支付幂等、延迟任务、订单状态、资金流水
  特点：Redis 不能作为唯一事实源
  策略：DB 唯一索引/任务表/流水表兜底，Redis 只做加速层
```

### 分布式锁为什么不建议持久化？

这是面试高频陷阱。

```
错误理解：
  锁很重要，所以锁 Key 要持久化。

实际问题：
  分布式锁是“时间敏感状态”，不是业务事实。
  旧锁恢复出来，可能比丢锁更危险。

例子：
  T1：服务A获取锁 lock:order:1，TTL=30s
  T2：Redis生成AOF
  T3：Redis宕机
  T4：业务服务A也宕机，锁本该自然失效
  T5：Redis重启，从AOF恢复 lock:order:1

结果：
  旧锁被恢复，其他服务无法处理订单。
  如果 TTL 也恢复异常，可能造成长时间误互斥。

正确做法：
  ① 锁 Key 不追求持久化
  ② 锁必须有 TTL
  ③ 解锁必须校验 owner
  ④ 核心业务靠幂等和状态机兜底，不靠锁兜底
```

架构师金句：

> **锁是并发控制手段，不是业务事实源；锁丢了靠幂等兜底，旧锁恢复反而可能扩大故障。**

## 二、fork + COW：Redis 持久化最大的隐藏成本

### 为什么 BGSAVE/BGREWRITEAOF 也可能阻塞？

很多人只记得 BGSAVE 是子进程执行，不阻塞主线程。但 fork 本身会阻塞。

```
Redis内存 20GB
  ↓
fork子进程
  ↓
操作系统要复制页表，不复制物理内存
  ↓
页表复制期间主线程暂停
  ↓
可能阻塞几十毫秒到几百毫秒
```

量化参考：

| Redis 内存 | fork 阻塞参考值 |
|-----------|----------------|
| 1GB | 几毫秒 |
| 10GB | 几十毫秒 |
| 30GB | 100ms+ |
| 50GB+ | 可能数百毫秒 |

### COW 内存放大的风险

fork 后，父子进程共享物理内存页。主进程继续写入时，OS 会复制被修改的页。

```
极端场景：
  Redis 20GB 内存
  BGSAVE 期间大量写入
  10GB 热数据页被修改

内存占用：
  原始 20GB
  + COW 复制 10GB
  = 30GB

如果机器总内存 32GB：
  非常接近 OOM
```

架构师措施：

```
1. 单实例内存控制在 10-20GB 内
2. BGSAVE/BGREWRITEAOF 避开写入高峰
3. 监控 latest_fork_usec
4. 监控 used_memory_rss 和 mem_fragmentation_ratio
5. 大实例优先 Cluster 拆分，而不是单节点堆到 50GB
```

## 三、主从复制的本质：异步日志追赶

### 复制三要素

Redis 主从复制的核心不是“主发给从”，而是三个元信息：

```
replid：主库复制身份
offset：复制进度
backlog：最近写命令的环形缓冲区
```

三者关系：

```
Slave：
  “我上次跟的是 replid=abc，已经同步到 offset=10000。”

Master：
  “我还是 replid=abc，并且 offset=10001 之后的命令还在 backlog。”

结果：
  增量复制。

如果 replid 不一致，或 offset 太旧：
  全量复制。
```

### backlog 大小必须会算

架构师不能只说“调大 backlog”，要能算。

```
公式：
  repl-backlog-size >= 写入流量 × 最大断连时间 × 安全系数

例子：
  Redis写入流量：3MB/s
  允许网络抖动：120秒
  安全系数：2

  backlog >= 3 × 120 × 2 = 720MB

如果只配默认1MB：
  网络断开不到1秒就可能丢增量窗口
  从库恢复时触发全量复制
```

全量复制的风险：

```
1. Master fork 生成 RDB → 主线程短暂停顿
2. RDB 文件通过网络发送 → 占用带宽
3. Slave 清空旧数据 → 读请求不可用或读空
4. Slave 加载 RDB → CPU/磁盘压力
5. 多个 Slave 同时全量 → 雪崩式复制风暴
```

架构师金句：

> **主从复制稳定性的关键不是“能不能全量”，而是尽量避免全量；backlog 配置是防复制风暴的第一道防线。**

## 四、主从延迟：不能消灭，只能分级治理

### 主从延迟的本质

Redis 从库延迟来自三段：

```
主库生成命令 → 网络传输 → 从库单线程执行
```

任何一段慢都会导致延迟。

| 延迟来源 | 典型原因 | 治理 |
|---------|---------|------|
| 主库写入过大 | 大 Key、大 MSET、大 Lua | 拆分 Key，拆命令 |
| 网络传输慢 | 跨机房、带宽不足 | 同机房、扩带宽、避免大对象 |
| 从库执行慢 | DEL 大 Key、AOF fsync、慢命令 | UNLINK、禁慢命令、从库轻持久化 |

### 读写分离的一致性分级

架构师要先问：这个读请求能不能接受旧数据？

```
一级：强一致读
  例：支付状态、订单状态、幂等判断
  策略：读主，或直接读 DB

二级：写后短期一致
  例：用户刚修改昵称后立即查看
  策略：写后读主窗口 3-5 秒

三级：弱一致读
  例：商品详情、排行榜、统计报表
  策略：读从，允许秒级延迟

四级：最终一致异步视图
  例：资金路由评分、推荐特征
  策略：读 Redis 视图，异常时回退 DB/默认策略
```

不要试图让所有读都强一致，否则读写分离就失去意义。

## 五、Sentinel：自动切换不是零丢失

### Sentinel 解决的是控制平面问题

Sentinel 的核心价值：

```
没有 Sentinel：
  主挂了 → 人工发现 → 人工选从 → 人工改配置 → 客户端重连

有 Sentinel：
  主挂了 → 自动发现 → 自动选从 → 自动改配置 → 客户端获取新主
```

它解决的是**故障转移自动化**，不是数据一致性。

### SDOWN/ODOWN 的本质

```
SDOWN：单个 Sentinel 的本地判断
ODOWN：多个 Sentinel 的多数派共识
```

这背后的分布式系统原则：

> **单点观察不可信，多数派观察才具备决策资格。**

### Sentinel 丢数据的根因

Redis 主从默认异步复制。

```
客户端写 Master 成功
    ↓
Master 返回 OK
    ↓
写命令还没发到 Slave
    ↓
Master 宕机
    ↓
Slave 被提升为 Master
    ↓
刚才写成功的数据丢失
```

这与 Sentinel 是否工作正常无关，是复制模型决定的。

### min-replicas-to-write 的取舍

```
配置：
  min-replicas-to-write 1
  min-replicas-max-lag 10

含义：
  至少有1个从库延迟小于10秒，主库才允许写。

收益：
  降低主库孤岛写入导致的大量数据丢失。

代价：
  如果从库全断，主库拒写。
  可用性下降。
```

这就是 CAP 权衡：

```
更低 RPO → 牺牲可用性
更高可用 → 接受更大丢失窗口
```

## 六、Cluster：槽位是稳定中间层

### 为什么不直接 hash 到节点？

如果直接：

```
node = hash(key) % node_count
```

扩容时：

```
3节点 → 4节点
几乎所有 key 的路由都变化
需要大规模迁移
```

Redis Cluster 引入 slot：

```
key → slot → node
```

扩容时只变：

```
部分 slot → 新 node
```

Key 到 slot 的映射稳定，只有 slot 到 node 的映射变化。

这就是“中间层解耦”的架构思想。

### 16384 的权衡

```
槽太少：
  迁移粒度粗，负载不均

槽太多：
  集群元数据变大，Gossip 消息变大

16384：
  每个节点用 bitmap 表示槽位，只需 2KB
  对几十到几百节点足够细
  元数据传播成本可控
```

### MOVED 和 ASK 的架构意义

```
MOVED：稳定态纠错
  “这个 slot 已经永久属于别的节点”
  → 客户端更新路由表

ASK：迁移态临时纠错
  “这个 key 可能已经迁过去，但 slot 还没整体迁完”
  → 客户端只本次临时访问，不更新路由表
```

为什么需要 ASK？

```
槽迁移是 Key 级逐步迁移，不是瞬间完成。

迁移中：
  同一个 slot 的一部分 key 在旧节点
  一部分 key 在新节点

如果只有 MOVED：
  客户端会把整个 slot 都切到新节点
  但还有 key 在旧节点 → 错误

ASK 解决迁移中间态。
```

### Cluster 的四个“不解决”

架构师必须主动说：

```
1. 不解决单 Key 热点
   一个 hot key 仍然落在一个节点
   解决：本地缓存、Key 拆分、读副本

2. 不解决大 Key
   大 Hash/ZSet 仍然可能阻塞所在节点
   解决：拆 Key、SCAN、UNLINK、治理大 Key

3. 不解决强一致
   Cluster 主从仍是异步复制
   Master 挂了仍可能丢最后写入

4. 不解决跨槽事务
   MULTI/Lua/MGET 多 Key 必须同槽
   解决：Hash Tag 或业务层拆分
```

## 七、Redis 架构选型：不要迷信 Cluster

### Cluster 不是默认答案

很多人一听高可用就说 Cluster，这是误区。

```
如果业务：
  数据 5GB
  QPS 2万
  Key 操作多为多 Key
  团队运维经验一般

推荐：哨兵，不是 Cluster。

原因：
  ① 单机容量够
  ② QPS 不高
  ③ Cluster 引入跨槽限制
  ④ 运维复杂度更高
```

### 架构选型四维法

```
维度1：容量
  单实例 < 10GB：单机/哨兵
  10-20GB：哨兵谨慎使用
  > 20GB：Cluster 或业务拆分

维度2：QPS
  < 1万：单机
  1万-10万：哨兵/读写分离
  > 10万：Cluster、本地缓存、Key拆分

维度3：可用性
  可人工恢复：主从
  要自动切换：哨兵
  要自动切换 + 水平扩展：Cluster

维度4：数据可丢性
  可丢：Redis 可作为缓存
  不可丢：DB/MQ/日志必须兜底
```

### 用户业务场景映射

结合你现有业务经验，可以这样回答：

```
1. IM 在线状态
   特点：高 QPS、低延迟、允许秒级误差
   架构：哨兵/Cluster + 本地缓存 + 心跳过期
   事实源：在线状态本身是临时状态，不需要强持久化

2. 资金路由评分
   特点：实时排序、可短暂不一致、不能无兜底
   架构：ZSet + 哨兵 + AOF everysec
   兜底：MySQL 保存最终评分，Redis 只是实时排序视图

3. 支付幂等
   特点：不能重复支付，Redis 误丢会出事故
   架构：Redis SET NX 加速 + DB 唯一索引兜底
   原则：Redis 不能做唯一防线

4. 秒杀库存
   特点：高并发、写热点、最终库存不能错
   架构：Redis 预扣 + Lua 原子性 + MQ 异步落库 + DB 对账
   原则：Redis 削峰，DB 兜底
```

## 八、与架构师水平的差距（Day2自检）

| 维度 | 开发 | 高级开发 | 架构师 |
|------|------|---------|--------|
| 持久化 | RDB/AOF定义 | 知道优缺点 | 按数据语义分层选型，知道锁不应恢复旧状态 |
| fork/COW | 不知道 | 知道 fork | 能量化 fork 阻塞和 COW 内存放大风险 |
| 主从复制 | 主从同步 | 全量/增量复制 | replid+offset+backlog，能计算 backlog 大小 |
| 主从延迟 | 读主库 | 写后读主 | 按业务一致性分级治理，弱一致读从、强一致读主 |
| Sentinel | 自动切主 | SDOWN/ODOWN | 控制平面，不能保证 RPO=0，可用性和一致性权衡 |
| Cluster | 自动分片 | 16384槽 | 槽位中间层、MOVED/ASK区别、四个“不解决” |
| 架构选型 | 用最强方案 | 哨兵/Cluster区别 | 容量/QPS/可用性/可丢性四维决策 |

### 一句话差距

**高级开发知道 Redis 高可用组件怎么用，架构师知道这些组件解决什么、不解决什么，以及在业务事实源、RPO、可用性、运维复杂度之间怎么取舍。**

## 九、今日面试金句

```
1. “Redis 高可用解决的是服务恢复，不等于数据零丢失；哨兵和 Cluster 都不能消除异步复制带来的 RPO 窗口。”

2. “RDB 和 AOF 不是简单替代关系：RDB 解决恢复速度，AOF 解决恢复点新鲜度，混合持久化是在两者之间做平衡。”

3. “分布式锁不建议持久化，因为锁是时间敏感的并发控制状态，不是业务事实源；恢复旧锁可能比丢锁更危险。”

4. “主从复制稳定性的关键是避免全量复制，backlog 大小要按写入速率 × 最大断连时间 × 安全系数来算。”

5. “Cluster 的本质是 key→slot→node 的中间层解耦，解决扩缩容路由变化；但它不解决热 Key、大 Key、强一致和跨槽事务。”

6. “不要迷信 Cluster，如果单实例 8GB、QPS 3 万，哨兵比 Cluster 更合适，因为简单、限制少、运维成本低。”
```
