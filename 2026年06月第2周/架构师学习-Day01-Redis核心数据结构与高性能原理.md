# Day 1：Redis核心数据结构与高性能原理 — 从会用到底层懂（练习整理）

## 题目一：Redis核心数据结构 — 不只是Key-Value

### 场景

你的团队要选型Redis做多个功能，每个功能该用什么数据结构？

| 功能需求 | 候选方案 |
|---------|---------|
| 商品库存扣减（原子性） | String DECR |
| 用户购物车（增删查） | Hash |
| 消息队列（有序消费） | List / Stream |
| 用户标签/画像（交集推荐） | Set |
| 实时排行榜（Top100） | ZSet |
| 签到统计（二进制位） | Bitmap |
| UV统计（去重计数） | HyperLogLog |
| 附近的人（地理位置） | Geo |

请回答：
1. 每个数据结构的核心操作和时空复杂度是什么？
2. 什么场景下不能用这个数据结构？（陷阱和边界）
3. Hash和String存对象的区别是什么？为什么购物车选Hash不选String？

---

## 答案

### Redis八大数据结构速查表

```
┌──────────────┬────────────────────────┬────────────────────────┬────────────────────────┐
│  数据结构     │  核心操作               │  时间复杂度             │  典型场景               │
├──────────────┼────────────────────────┼────────────────────────┼────────────────────────┤
│  String       │  SET/GET/INCR/DECR      │  O(1)                  │  缓存、计数器、分布式锁  │
│  Hash         │  HSET/HGET/HGETALL      │  O(1)字段级            │  对象存储、购物车        │
│  List         │  LPUSH/RPOP/LRANGE      │  O(1)头尾 / O(N)范围   │  消息队列、时间线        │
│  Set          │  SADD/SMEMBERS/SINTER   │  O(1)增删 / O(N)交集   │  标签、共同关注、去重    │
│  ZSet         │  ZADD/ZRANGEBYSCORE     │  O(logN)               │  排行榜、延迟队列        │
│  Bitmap       │  SETBIT/GETBIT/BITCOUNT │  O(1)单bit / O(N)统计  │  签到、在线状态          │
│  HyperLogLog  │  PFADD/PFCOUNT          │  O(1)                  │  UV统计（误差0.81%）     │
│  Geo          │  GEOADD/GEORADIUS       │  O(logN)增 / O(N)查询  │  附近的人、位置服务      │
│  Stream       │  XADD/XREAD/XGROUP      │  O(1)生产 / O(N)消费   │  日志流、持久化消息队列  │
└──────────────┴────────────────────────┴────────────────────────┴────────────────────────┘
```

### 各数据结构的陷阱与边界

**String — 不是万能缓存**

```
陷阱1：大Value
  SET product:12345 "10MB的JSON"
  → 单Key 10MB，Redis是单线程，一次传输阻塞其他请求
  → 网卡打满，RT暴涨

  红线：单个String < 10KB，超过考虑Hash拆分或压缩

陷阱2：存复杂对象
  SET user:1001 "{name,age,address,orders...}"
  → 改一个字段要读写整个JSON
  → 序列化反序列化CPU开销

  解法：Hash存对象，字段级更新
```

**Hash — 字段数有上限**

```
Hash的底层编码转换：
  ziplist（压缩列表）：field+value总数 < 512，且每个 < 64B
  hashtable（哈希表）：超过任一阈值就转hashtable

陷阱：HGETALL大Hash
  1000个field的Hash → HGETALL一次返回全部 → 阻塞
  解法：用HSCAN分批取，或只取需要的field（HMGET）

Hash vs String存对象：
  ┌────────────────┬─────────────────┬─────────────────┐
  │                │  String(JSON)   │  Hash           │
  ├────────────────┼─────────────────┼─────────────────┤
  │  内存占用      │  高（字段名重复）│  低（field共享） │
  │  字段级更新    │  无（全量读写）  │  有（O(1)）     │
  │  过期时间      │  整体过期        │  只能整体过期   │
  │  序列化开销    │  有              │  无             │
  └────────────────┴─────────────────┴─────────────────┘

购物车选Hash的原因：
  ① 商品ID做field，数量做value → 增删改都是O(1)
  ② 不用序列化整个购物车
  ③ 内存比String(JSON)省30%+
```

**List — 不是高并发队列的最佳选择**

```
陷阱：BRPOP阻塞读
  BRPOP queue 30
  → 连接阻塞30秒，连接池被打满
  → 高并发下连接数爆炸

  解法：用LPUSH+轮询，或换Stream

陷阱2：没有消费者组
  多个消费者用BLPOP → 一个消息只能被一个消费者消费
  → 不能广播，不能ACK

  解法：需要ACK和广播 → 换Stream（Redis 5.0+）
```

**ZSet — 排行榜首选，但有写入瓶颈**

```
ZADD操作是O(logN)，N是元素个数
  10万元素的ZSet → ZADD约几十微秒
  1000万元素的ZSet → ZADD约几百微秒 → 单线程累积延迟

陷阱：超大ZSet
  全站用户积分排行榜（千万级）→ 一个ZSet太大
  解法：按时间段/分段拆分（月度榜、赛季榜）

陷阱2：ZRANGE分页深翻页
  ZRANGE rank 100000 100010 → 内部要遍历10万元素
  → O(N)不是O(logN)
  解法：排行榜只展示Top1000，后面的用户不展示排名
```

**Bitmap — 省空间但有计算开销**

```
1亿用户签到 → Bitmap只需 1亿bit = 12.5MB
String存 → 1亿个Key或一个大JSON → GB级

陷阱：BITCOUNT全量统计
  BITCOUNT sign:202506 → 遍历12.5MB → O(N)
  → 阻塞单线程几十毫秒

  解法：预聚合（每天统计结果存String），或BITCOUNT分段
```

**HyperLogLog — 牺牲精度换空间**

```
PFADD/PFCOUNT都是O(1)
1000万UV → 只需12KB内存

陷阱：不能存具体元素
  只能回答"有多少个不同元素"，不能回答"有哪些元素"
  误差约0.81%

  解法：需要精确去重 → 换Set（内存爆炸）或布隆过滤器
```

**Geo — 不是专业GIS**

```
底层是ZSet，score是geohash编码
GEORADIUS是O(N)，N是集合元素数

陷阱：元素过多
  百万级POI → GEORADIUS要遍历百万 → 阻塞
  解法：按城市/网格拆分Geo集合

不能替代：PostGIS、Elasticsearch Geo（专业空间索引）
```

**Stream — Redis的消息队列终极方案**

```
核心优势：
  ① 消费者组（Consumer Group）：同一条消息可被不同组消费
  ② ACK机制：XACK确认，支持Pending列表重试
  ③ 持久化：AOF保证消息不丢（List不持久化会丢）

陷阱：没有分区（Partition）
  Kafka有Topic+Partition实现水平扩展
  Stream只有单Key → 写入性能上限约10万/秒
  解法：按业务拆多个Stream Key
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | Redis就是Key-Value，存String |
| 高级开发 | 知道String/Hash/List/Set/ZSet五种，能说出场景 |
| 架构师 | 八种数据结构的选择有量化依据：复杂度、内存占用、阻塞风险。知道每个数据结构的底层编码转换阈值（ziplist/hashtable/intset/skiplist）、阻塞操作的红线（大Hash HGETALL/大ZSet深翻页/大Bitmap BITCOUNT）、以及什么时候该换技术栈（Geo百万级换ES、List高并发队列换Stream/Kafka） |

**补足方向**：被问"购物车用什么数据结构"，要说清"Hash，因为field级更新O(1)不需要序列化整个对象，内存比String省30%，但要注意field数超过512会转hashtable，HGETALL会阻塞所以用HSCAN或HMGET"。

## 题目二：ZSet底层实现 — 为什么能同时支持O(logN)插入和范围查询？

### 场景

你设计了一个实时积分榜，要求：
- 用户得分实时更新（频繁ZADD）
- 实时查询Top100（ZRANGE 0 99）
- 查询用户排名（ZRANK userId）

面试官问：ZSet底层是什么数据结构？为什么这三个操作都能做到O(logN)？

---

## 答案

### ZSet的底层双数据结构

```
ZSet = 跳表（skiplist）+ 哈希表（hashtable）

  ┌────────────────────────────────────────────────────────┐
  │  跳表（skiplist）— 支持范围查询和排序                   │
  │                                                         │
  │  L3:  head ──────────────────────────────→ 100         │
  │  L2:  head ──────────────→ 50 ───────────→ 100        │
  │  L1:  head ─────→ 20 ────→ 50 ────→ 80 ──→ 100       │
  │  L0:  head → 10 → 20 → 30 → 50 → 60 → 80 → 100       │
  │                                                         │
  │  查询30：从L3开始，逐层下降 → O(logN)                   │
  │  范围查询20-80：找到20，沿L0遍历 → O(logN + M)          │
  └────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────┐
  │  哈希表（hashtable）— 支持按member查score               │
  │                                                         │
  │  member → score映射：                                   │
  │  "user:1001" → 80                                       │
  │  "user:1002" → 50                                       │
  │  "user:1003" → 100                                      │
  │                                                         │
  │  ZSCORE user:1001 → O(1)直接查哈希表                    │
  └────────────────────────────────────────────────────────┘
```

### 跳表的原理与复杂度

```
跳表是平衡树的一种概率替代：

  插入过程（以50为例）：
    ① 随机决定层高（抛硬币，概率1/2升级）
    ② 平均层高 = 2，最高层约logN
    ③ 从最高层开始查找插入位置
    ④ 逐层下降，每层走若干步

  复杂度分析：
    查找：O(logN) — 每层期望走常数步，共logN层
    插入：O(logN) — 先查找位置，再修改指针
    删除：O(logN)
    范围查询：O(logN + M) — M是返回元素数

  为什么用跳表不用红黑树？
    ┌────────────────┬─────────────────┬─────────────────┐
    │                │  跳表           │  红黑树         │
    ├────────────────┼─────────────────┼─────────────────┤
    │  范围查询      │  O(logN+M)      │  O(logN+M)      │
    │  实现复杂度    │  简单（指针操作） │  复杂（旋转）   │
    │  并发锁粒度    │  可以局部锁      │  必须全局锁     │
    │  内存占用      │  稍大（多级指针） │  稍小           │
    └────────────────┴─────────────────┴─────────────────┘

  Redis选择跳表的核心原因：
    ① 实现简单，几百行代码，bug少
    ② 范围查询效率高（沿底层遍历即可）
    ③ 插入删除不需要旋转平衡
```

### 编码转换阈值

```
ZSet的底层编码：
  ziplist（压缩列表）：元素数 < 128，且每个member < 64B
  skiplist（跳表+哈希表）：超过任一阈值

为什么有ziplist？
  小ZSet用跳表浪费空间（指针开销）
  ziplist是连续内存，省空间但O(N)操作
  元素少时O(N)和O(logN)没区别，但省很多内存
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | ZSet是排序集合，底层是跳表 |
| 高级开发 | ZSet底层是跳表+哈希表，跳表支持范围查询，哈希表支持按member查score |
| 架构师 | ZSet底层是跳表（skiplist）+ 哈希表（hashtable）双结构，跳表提供O(logN)的范围查询和排序，哈希表提供O(1)的member→score映射。跳表是概率平衡结构，期望层高logN，插入不需要旋转平衡，实现比红黑树简单且并发友好。小ZSet（<128元素）用ziplist连续存储省空间。ZRANK是O(logN)因为沿跳表从头遍历计数，超大ZSet的ZRANK会变慢 → 需要分片或预估排名 |

**补足方向**：面试时主动说"跳表vs红黑树的选择"——"Redis选跳表不选红黑树，核心原因是实现简单+范围查询友好+不需要旋转平衡。跳表是概率数据结构，期望性能等于平衡树，但代码复杂度低一个数量级"。

## 题目三：Redis为什么单线程还能10万+QPS？

### 场景

面试官问：Redis是单线程的，为什么性能比多线程的MySQL还高？单线程不是只能用一个CPU核心吗？Redis 6.0又为什么引入多线程IO？

---

## 答案

### Redis高性能的四个核心原因

```
┌────────────────────────────────────────────────────────┐
│  原因1：纯内存操作                                      │
│  所有数据在内存，读写都是内存访问（纳秒级）             │
│  MySQL读数据要从磁盘加载页到Buffer Pool（微秒级）       │
│  内存访问速度 ≈ 磁盘随机IO的10万倍                     │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│  原因2：IO多路复用 — 单线程同时处理数万连接             │
│                                                         │
│  传统阻塞IO：                                            │
│    线程A read(fd1) → 阻塞等待数据 → 无法处理fd2        │
│    1万连接 → 1万个线程 → 上下文切换开销巨大            │
│                                                         │
│  Redis的IO多路复用（epoll）：                           │
│    一个线程管理1万个fd                                  │
│    epoll_wait() → 返回有数据的fd列表                    │
│    逐个处理 → 全部处理完 → 再次epoll_wait()             │
│                                                         │
│    没有阻塞，没有线程切换，一个事件循环搞定所有连接      │
│                                                         │
│  核心代码逻辑：                                          │
│    while (!stop) {                                      │
│      ready = epoll_wait(epfd, events, maxevents, timeout)│
│      for (i = 0; i < ready; i++) {                      │
│        process_event(events[i])  // 处理读/写/超时      │
│      }                                                  │
│    }                                                    │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│  原因3：避免上下文切换                                   │
│  多线程的隐藏成本：                                      │
│    ① 线程切换：保存/恢复寄存器、栈指针 → 几微秒        │
│    ② 锁竞争：多个线程抢锁 → 自旋/阻塞 → 几十微秒      │
│    ③ 缓存失效：线程切换后CPU缓存命中率下降             │
│                                                         │
│  Redis单线程：                                          │
│    没有线程切换，没有锁竞争，CPU缓存命中率高             │
│    每个请求的处理是连续的，没有中断                      │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│  原因4：高效的数据结构                                   │
│  SDS（简单动态字符串）：O(1)取长度，预分配减少扩容       │
│  跳表：O(logN)的有序操作                                │
│  压缩列表：小数据量时省内存                              │
│  整数集合：纯整数Set用数组存，省空间                     │
└────────────────────────────────────────────────────────┘
```

### 单线程的瓶颈与Redis 6.0的多线程IO

```
单线程的真正瓶颈：
  不是CPU计算，而是网络IO
  读取请求包 → 解析协议 → 执行命令 → 写响应包
  ├─ 执行命令是O(1)纳秒级 ─┤
  ├─ 网络读写是微秒级 ─────┤

  当QPS到10万+时，网络读写的累积时间占主导
  → 单线程的网络IO成为瓶颈

Redis 6.0的多线程IO：
  ┌────────────────────────────────────────────────────────┐
  │  核心设计：命令执行仍是单线程，网络IO可以多线程         │
  │                                                         │
│  多线程IO流程：                                          │
│    ① 主线程：epoll_wait → 发现有数据的连接              │
│    ② IO线程池：多个线程并行读取请求包（read+解析）      │
│    ③ 主线程：单线程执行所有命令（保证原子性）           │
│    ④ IO线程池：多个线程并行写响应包（encode+write）     │
│                                                         │
│  为什么命令执行不能多线程？                              │
│    多线程执行命令需要加锁保护共享数据结构                 │
│    锁竞争会抵消多线程收益，甚至性能更差                  │
│    所以只把不共享状态的"网络读写"多线程化                │
│                                                         │
│  效果：网络IO密集型负载提升约1倍（10万→20万QPS）        │
│  默认关闭，需配置io-threads-do-reads yes                │
└────────────────────────────────────────────────────────┘
```

### 单线程的副作用 — 大Key阻塞

```
Redis单线程的致命弱点：一个慢操作阻塞所有请求

  场景1：大Key删除
    DEL big_hash（100万个field）
    → 内部循环删除100万个元素
    → 耗时几百毫秒 → 期间所有请求排队

  场景2：大Key遍历
    HGETALL big_hash / KEYS *
    → O(N)遍历 → 阻塞

  场景3：复杂聚合
    SUNIONSTORE set1 set2 set3（千万级元素）
    → 阻塞

  解法：
    ① 删除大Key → UNLINK（异步删除，4.0+）
    ② 遍历大Key → HSCAN/SSCAN/ZSCAN（分批）
    ③ 避免生产环境用KEYS → 用SCAN
    ④ 大Key拆分 → Hash按field拆多个Key
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | Redis单线程快是因为纯内存 |
| 高级开发 | Redis用epoll做IO多路复用，单线程避免了上下文切换 |
| 架构师 | Redis高性能是四因素叠加：纯内存+IO多路复用(epoll)+无上下文切换+高效数据结构。单线程的瓶颈在网络IO不是CPU，所以6.0引入多线程IO（只并行网络读写，命令执行仍单线程保证原子性）。单线程的副作用是慢操作阻塞全局——大Key删除用UNLINK异步，遍历用SCAN分批， KEYS命令生产禁用。能说出"网络读写占微秒，命令执行占纳秒，所以多线程IO有效但多线程执行无效" |

**补足方向**：面试时主动提Redis 6.0多线程IO的设计——"命令执行仍单线程，网络读写多线程，这是权衡了原子性和性能的最优解"。

## 题目四：SDS — 为什么Redis不用C字符串？

### 场景

Redis的String底层是SDS（Simple Dynamic String），为什么不用C语言原生的`char*`？SDS相比C字符串有哪些优势？

---

## 答案

### C字符串的缺陷

```
C字符串：以'\0'结尾的字符数组

  缺陷1：获取长度O(N)
    strlen("hello") → 遍历到'\0' → O(N)
    每次STRLEN都要遍历 → 频繁操作性能差

  缺陷2：缓冲区溢出
    strcat(dst, src) → 不检查dst容量 → 越界写
    → 栈/堆破坏，安全漏洞

  缺陷3：二进制不安全
    C字符串不能存包含'\0'的数据
    图片/音频等二进制数据无法直接存储

  缺陷4：频繁扩容的内存拷贝
    每次追加都要realloc+memcpy → O(N)
    频繁追加性能极差
```

### SDS的结构设计

```
SDS结构（Redis 3.2+优化版）：

  ┌──────────┬──────────┬──────────────────────┬──────────┐
  │  flags   │  len     │  alloc               │  buf[]   │
  │  (1B)    │  (1-8B)  │  (1-8B)              │          │
  ├──────────┼──────────┼──────────────────────┼──────────┤
  │  编码类型 │  已用长度 │  分配的总容量         │  数据区  │
  └──────────┴──────────┴──────────────────────┴──────────┘

  flags决定len/alloc的位数：
    SDS_TYPE_5：len=5bit，alloc=5bit（极短字符串，已废弃主动使用）
    SDS_TYPE_8：len/alloc都是uint8_t（<256B）
    SDS_TYPE_16：len/alloc都是uint16_t（<64KB）
    SDS_TYPE_32：len/alloc都是uint32_t（<4GB）
    SDS_TYPE_64：len/alloc都是uint64_t（超大字符串）

  优势1：O(1)获取长度
    sdslen(s) → 直接读s->len → O(1)

  优势2：二进制安全
    buf[]不以'\0'为结束标志
    可以存图片、序列化数据等任意二进制
    但Redis上层协议仍兼容C字符串（末尾加'\0'方便兼容）

  优势3：预分配 + 惰性释放
    追加时：如果alloc - len > 追加长度 → 直接写，不扩容
    预分配策略：扩容到2倍（<1MB时）或1MB（>=1MB时）
    释放时：不真正free，只减len → 下次追加复用

    例子：
      原字符串 "hello"（len=5, alloc=10）
      追加 " world"（6字节）
      5+6=11 > alloc(10) → 扩容到 11*2=22
      新SDS：len=11, alloc=22
      下次追加10字节以内 → 不需要扩容

  优势4：杜绝缓冲区溢出
    sdscat(s, "xxx") → 先检查alloc-len是否够 → 不够自动扩容
    → 所有操作都是安全的
```

### 内存预分配的量化收益

```
频繁追加场景对比（追加1000次，每次10字节）：

  C字符串：
    每次strcat → 遍历找结尾O(N) + realloc拷贝O(N)
    总时间 ≈ O(N^2) → 百万次操作要几秒

  SDS：
    第1次：扩容到20B
    第2-10次：直接写，不扩容
    第11次：扩容到40B
    第12-30次：直接写
    ...
    总扩容次数 ≈ log(总长度) → 10次左右
    总时间 ≈ O(N)，比C字符串快100倍+
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | Redis自己实现了字符串，为了安全 |
| 高级开发 | SDS可以O(1)取长度，二进制安全，预分配减少扩容 |
| 架构师 | SDS相比C字符串有四大优势：O(1)取长度（C是O(N)）、二进制安全（不以'\0'结尾）、预分配+惰性释放（追加均摊O(1)，C是O(N)）、杜绝缓冲区溢出（自动扩容）。3.2+版本按长度选5种编码类型（8/16/32/64），最小化内存 overhead。预分配策略：<1MB时翻倍，>=1MB时+1MB，平衡了内存和追加性能。面试时要说出"预分配让频繁追加的均摊复杂度从O(N^2)降到O(N)" |

**补足方向**：被问"Redis字符串和Java StringBuilder的区别"——"两者都是可变字符串+预分配，但SDS还解决了C字符串的O(1)长度和二进制安全问题，而且按数据长度自适应编码（8/16/32/64bit），内存更省"。

## 题目五：实时排行榜设计 — ZSet在千万级用户场景下的应用

### 场景

你是蛋壳钱包的架构师，要设计一个"消费排行榜"功能：
- 全站用户按累计消费金额排名
- 实时展示Top100
- 用户可查看自己的排名和前后10名
- 日活用户100万，总用户1000万
- 高峰期每分钟1万次得分更新

请回答：
1. ZSet怎么设计Key？用户量1000万单ZSet会不会太大？
2. 查询Top100是O(logN + M)，N=1000万时性能如何？
3. 如果Redis挂了，排行榜数据怎么恢复？
4. 多机房部署时，排行榜怎么保证一致性？

---

## 答案

### 方案1：单ZSet（简单但不推荐）

```
Key: ranking:daily:20250608
Value: ZSet — member=用户ID, score=消费金额

问题：
  ① 1000万元素的ZSet → ZADD O(logN) ≈ 23步，但单线程累积延迟
  ② 每分钟1万次ZADD → 10000 × 几十微秒 ≈ 几百毫秒 → 占单线程时间片的10%
  ③ Redis挂了 → 1000万数据从MySQL重建 → 分钟级
  ④ 内存占用：每个元素约100字节 → 1000万 × 100B = 1GB

结论：单ZSet能跑，但风险和成本都不理想
```

### 方案2：分片ZSet（推荐）

```
按用户ID分片，每片10万人：

  Key设计：ranking:daily:20250608:shard:{user_id % 100}
  → 100个分片，每片10万用户

  查Top100：
    ① 从100个分片各取Top100 → 10000条数据
    ② 合并排序取全局Top100
    ③ 聚合时间 ≈ 100次ZRANGE + 内存归并

  查用户排名：
    ① 先定位分片：shard = user_id % 100
    ② ZRANK within shard → 片内排名
    ③ 没有全局排名！需要额外维护全局排名

  全局排名解法：
    方案A：用另一个汇总ZSet存Top10000
      → 每片贡献Top100，汇总成全局Top10000
      → 查排名只在Top10000里查，查不到说明排名很低
    方案B：不展示精确排名，只展示百分比
      → "您超过了95%的用户" → 不需要全局ZRANK
```

### 方案3：Write-Through + MySQL兜底（生产推荐）

```
┌────────────────────────────────────────────────────────┐
│  架构图：                                               │
│                                                         │
│  消费发生 → 写MySQL（事务）                            │
│        ↓                                                │
│  异步更新Redis ZSet（Canal/MQ）                        │
│        ↓                                                │
│  读请求 → 查Redis ZSet → 返回排名                      │
│        ↓                                                │
│  Redis挂了 → 降级读MySQL排序（慢但可用）               │
│                                                         │
│  MySQL表设计：                                          │
│  CREATE TABLE user_rank (                               │
│    user_id BIGINT PRIMARY KEY,                          │
│    score DECIMAL(18,2),                                 │
│    update_time TIMESTAMP,                               │
│    INDEX idx_score (score DESC)                         │
│  );                                                     │
│                                                         │
│  Redis挂了时的降级：                                    │
│    SELECT COUNT(*) FROM user_rank WHERE score > ?       │
│    → 用覆盖索引，1000万行约几百毫秒 → 可接受降级        │
└────────────────────────────────────────────────────────┘
```

### 性能量化

```
ZSet操作性能（单节点Redis，元素100万级别）：

  ZADD：约 50-100微秒
  ZRANGE 0 99：约 100-200微秒
  ZRANK：约 50-100微秒
  ZCARD：O(1)，约 1微秒

  1000万元素ZSet：
    ZADD → log2(1000万) ≈ 23层跳表 → 约 100-200微秒
    1万QPS的ZADD → 总耗时约 1-2秒/秒 → 单线程占用10-20%
    → 还有80%时间处理读请求 → 可接受

  但风险：
    一个慢操作（比如ZREVRANGE 0 999999）阻塞全局
    → 需要代码审查禁用危险命令
```

### 多机房一致性

```
方案1：单主多从（推荐）
  排行榜写入量不大（1万/分钟 ≈ 167/秒）
  单主完全可以承受
  从节点承担读请求（Top100可以容忍秒级延迟）

方案2：各机房独立计算
  每个机房有独立ZSet
  最终汇总到中心Redis
  → 复杂度高，适合写入量极大的场景

结论：排行榜类低频写场景，单主多从足够
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 用ZSet存排行榜，ZADD更新，ZRANGE查Top100 |
| 高级开发 | 用户量大要分片，按user_id取模分到多个ZSet |
| 架构师 | 千万级排行榜要三层设计：①ZSet分片降低单Key大小和重建风险；②MySQL兜底+Canal异步同步，Redis挂了降级读MySQL覆盖索引；③不展示精确全局排名（只展示百分比或TopN内精确排名），避免跨分片聚合。性能量化：1000万ZSet的ZADD约100-200微秒，1万QPS写占用单线程10-20%可接受。多机房用单主多从即可，写入量167/秒不需要多活 |

**补足方向**：被问"千万级排行榜怎么设计"，不要只说"ZSet分片"，要说完整的降级方案——"Redis+MySQL双写，Redis挂了读MySQL覆盖索引，排名只展示百分比避免全局聚合，分片按user_id取模每片10万"。

## Day1核心总结

```
Day1形成Redis基础认知框架：

  数据结构选型：
    不是"Redis存String"，是八种数据结构按场景选
    每个数据结构有编码转换阈值、阻塞红线、换技术栈边界

  底层原理：
    ZSet = 跳表+哈希表，跳表单点O(logN)范围查询，哈希表O(1)查score
    跳表是概率平衡结构，实现比红黑树简单，并发更友好

  高性能根因：
    纯内存 + epoll IO多路复用 + 无上下文切换 + 高效数据结构
    6.0多线程IO只并行网络读写，命令执行仍单线程保证原子性
    单线程的代价：慢操作阻塞全局 → UNLINK/SCAN/避免大Key

  字符串设计：
    SDS解决C字符串四大缺陷：O(1)长度、二进制安全、预分配、防溢出
    按长度自适应5种编码，最小化内存overhead

  架构设计：
    千万级ZSet要分片+MySQL兜底+降级策略
    量化思维：ZADD耗时、QPS占用单线程比例、重建时间
```
