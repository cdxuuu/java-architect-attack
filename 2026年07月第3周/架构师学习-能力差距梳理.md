## 一、K8s 核心架构与控制面（Day01）

### 差距1.1：K8s 控制面组件实战调优不足
> Day1发现

- **现状**：知道 K8s 控制面组件构成（API Server/etcd/Scheduler/Controller Manager），但 API Server、etcd 的性能调优（max-requests-inflight、APF PriorityLevel、etcd –quota-backend-bytes）只调过参数，未深度实战
- **架构师水平**：能根据集群规模（节点数、Pod 数、QPS）规划 API Server/etcd 资源，能基于监控指标（apiserver_request_duration、etcd_wal_fsync_duration）做容量评估，能设计 APF FlowSchema/PriorityLevel 分级限流
- **补足方向**：
  - 精读 K8s 官方"Scaling Kubernetes to 2,500 Nodes"系列博客
  - 学习 APF FlowSchema/PriorityLevel 配置，在生产集群验证限流效果
  - 复盘啄木鸟云健康集群（如有）的 API Server 监控指标

### 差距1.2：etcd 备份恢复与故障演练缺失
> Day1发现

- **现状**：知道 etcd 是 K8s 命根子，但没做过 etcd 备份恢复演练、脑裂演练
- **架构师水平**：能设计 etcd 备份策略（snapshot 周期 + 异地备份 + 恢复演练），能在 etcd 脑裂时快速诊断与恢复
- **补足方向**：
  - 搭 3 节点 etcd 集群，做脑裂、磁盘满、节点宕机演练
  - 学习 `etcdctl snapshot save/restore` 完整流程
  - 在生产集群配置 etcd 监控告警（fsync 延迟、leader 切换、磁盘使用率）

### 差距1.3：Leader 选举机制细节不熟
> Day1发现

- **现状**：知道 kube-scheduler / kube-controller-manager 用 Leader 选举，但基于 Lease 资源的细节（holderIdentity / leaseDuration / renewTime / acquireTime）不熟
- **架构师水平**：能讲清 Lease 资源的字段含义、续约机制、切换时间窗口（默认 5-15s），能自定义控制器复用同样的 Leader 选举机制（client-go 的 leader-election 库）
- **补足方向**：
  - 阅读 client-go leader-election 库源码
  - 自定义控制器实践 Leader 选举
  - 学习 K8s 1.20+ 的 Lease 升级（coordination.k8s.io/v1）

### 差距1.4：API Server 故障排查体系化不足
> Day1发现，延续第2周差距2.3（长连接网关水平扩展）

- **现状**：知道 API Server 慢会影响集群，但 API Server 慢的诊断（429 告警、慢请求分析、watch 滞后）不熟
- **架构师水平**：能基于 apiserver_request_duration、apiserver_storage_objects、workqueue_depth 等指标定位 API Server 瓶颈，能区分 etcd I/O / 准入 webhook / List 请求三类瓶颈
- **补足方向**：
  - 学习 K8s 控制面监控指标体系（apiserver_*、etcd_*、workqueue_*）
  - 学习 API Server audit log 分析方法
  - 复盘啄木鸟云健康集群（如有）的 API Server 历史 incidents

## 二、Pod 模型与多容器设计（Day01）

### 差距2.1：Sidecar 模式实战主要在 Istio Envoy，自定义 Sidecar 经验不足
> Day1发现

- **现状**：用过 Istio Envoy、Filebeat Sidecar，但对 Ambassador / Adapter 模式、自研 Sidecar 注入控制器不熟
- **架构师水平**：能根据业务需求设计多容器 Pod（如医疗 IM 长连接网关 + Mesh Sidecar + 日志 Sidecar），能自研 MutatingAdmissionWebhook 实现 Sidecar 自动注入
- **补足方向**：
  - 阅读 Istio Sidecar Injector 实现源码
  - 自研一个简单的 Sidecar 注入控制器（如自动注入日志采集容器）
  - 复盘啄木鸟云健康线上 Sidecar 模式使用情况，识别可优化点

### 差距2.2：探针配置缺乏统一规范
> Day1发现

- **现状**：探针配置主要复制粘贴，liveness / readiness / startup 没有清晰职责划分
- **架构师水平**：能为不同类型应用（Java/Go/Python、Web/Worker/CronJob）制定探针配置规范，能基于 Actuator/Micrometer 设计精细化健康检查
- **补足方向**：
  - 制定 Spring Boot 应用探针配置规范（liveness + readiness + startup 分离）
  - 制定 Go/Python 应用探针配置规范
  - 在啄木鸟云健康线上推广，逐步替换老配置

### 差距2.3：多容器 Pod 资源竞争调优经验不足
> Day1发现

- **现状**：知道多容器 Pod 共享 network namespace，但 CPU/IO/网络资源竞争的调优（CPU Manager static 策略、cgroup v2 io 限制）不熟
- **架构师水平**：能为多容器 Pod 设计精细化的资源配额（业务容器独占 CPU、Sidecar 限制 burst），能基于 cgroup v2 配置 IO 隔离避免日志 Sidecar 拖慢主容器
- **补足方向**：
  - 学习 K8s CPU Manager（static 策略）、Memory Manager
  - 学习 cgroup v2 io 隔离配置
  - 在测试集群验证 Sidecar 资源配额对主容器的影响

### 差距2.4：K8s 故障排查体系化不足
> Day1发现，延续第5周差距2.3（微服务长连接网关）

- **现状**：能查 `kubectl describe`、`kubectl logs`，但 CrashLoopBackOff、OOMKilled、Init:Error 等高频故障的体系化排查不熟
- **架构师水平**：能建立 K8s 故障排查手册（按现象 -> 原因 -> 诊断 -> 处理流程化），能基于 kubelet/容器运行时日志深入分析
- **补足方向**：
  - 整理 K8s 常见故障案例库（CrashLoopBackOff、ImagePullBackOff、OOMKilled、Pending、Evicted）
  - 学习 `crictl`、`ctr`、`kubectl debug` 等工具
  - 复盘历史故障（如 Pod 频繁重启、节点 NotReady）形成 SOP

### 差距2.5：K8s 1.28+ sidecarContainers 新特性不熟
> Day1发现

- **现状**：知道 K8s 1.28 引入了 sidecarContainers（Init Container + restartPolicy: Always），但生产实战没用过
- **架构师水平**：能在新版本集群中用 sidecarContainers 解决"Sidecar 必须先启动、Sidecar 随主容器退出"问题，能平滑迁移老的 Sidecar 配置
- **补足方向**：
  - 在测试集群验证 sidecarContainers 行为
  - 学习从老 Sidecar 模式迁移到 sidecarContainers 的最佳实践

## 三、与往周专题的衔接差距（Day01）

### 差距3.1：K8s 与微服务专题的衔接认知不深
> Day1发现，延续第5周差距（微服务）

- **现状**：知道 K8s 提供 DNS + ConfigMap + Ingress + Mesh，但与 Nacos/Sentinel/Seata 等独立组件的取舍不清
- **架构师水平**：能讲清"K8s 原生能力 vs 独立中间件"的边界（如 K8s Service 替代 Nacos 服务发现，K8s ConfigMap 替代 Nacos 配置中心，但 Sentinel 限流仍需独立组件或下沉到 Mesh），能根据业务阶段（小规模 vs 大规模）选择合适方案
- **补足方向**：
  - 对比 K8s ConfigMap vs Nacos 配置中心（动态推送、灰度发布、多环境）
  - 对比 K8s Service vs Nacos 服务发现（DNS vs RPC 注册、跨集群）
  - 学习 Sentinel 与 Istio Envoy 的限流能力对比

### 差距3.2：K8s 与限流降级专题的衔接认知不深
> Day1发现，延续第4周差距（限流降级）

- **现状**：知道限流可下沉到 Service Mesh，但 Sentinel vs Istio Envoy 的限流能力差异、生产实践不熟
- **架构师水平**：能讲清"应用层限流（Sentinel）vs 网关层限流（Nginx/APISIX）vs Mesh 层限流（Envoy）"三层限流的分工，能根据 SLA 与故障域设计多层级流
- **补足方向**：
  - 学习 Istio Envoy 的限流机制（Global Rate Limit + Local Rate Limit）
  - 学习 Sentinel 的"集群限流 + Token Server"与 Envoy 的"全局限流 + Redis"对比
  - 设计多层级限流方案（Mesh + 网关 + 应用）
