## 一、路由在支付系统中的定位

### 1.1 核心一句话

> **路由 = 用规则 + 画像 + 权重，从候选渠道中选出"当前最优"的那一个，并准备好失败后的 Fallback 链**

幂等解决"同一笔不重复"，对账解决"业务侧与资金侧状态对齐"，路由解决"在多个可选渠道中做最优决策"。三者构成支付系统三大核心能力。

### 1.2 路由的四大目标

| 目标 | 描述 | 衡量指标 |
|---|---|---|
| 成功率 | 选的渠道能不能成 | 路由后交易成功率 |
| 成本 | 选的渠道费率多少 | 千分之几 |
| 体验 | 选的渠道快不快 | TP99 |
| 合规 | 选的渠道是否合规 | 监管否决率 |

四个目标互相博弈：成本低的成功率可能波动大、成功率高的费率高、体验好的可能限额小。路由的本质是**多目标决策**。

### 1.3 路由 ≠ 简单 if-else

生产级路由必须解决：
- 多业务线隔离（电商/借款/还款/提现走不同渠道池）
- 多商户号负载均衡（一个渠道 N 个商户号）
- 动态权重（渠道状态实时变化）
- 故障切换（秒级感知 + 半开探活 + 渐进恢复）
- 灰度发布（新渠道按比例验证）
- 运营可配（规则热更新，无需发版）

## 二、路由分层模型（核心抽象）

### 2.1 五层漏斗

```
请求 → [L1业务准入] → [L2商户/渠道池] → [L3渠道筛选] → [L4排序] → [L5兜底] → 渠道下单
```

| 层 | 职责 | 是否可缓存 |
|---|---|---|
| L1 业务准入 | 业务线识别、风控准入、合规校验 | 可缓存（按业务+租户） |
| L2 渠道池 | 业务线→渠道池映射、灰度分流 | 可缓存 |
| L3 渠道筛选 | 硬性约束（状态/限额/币种/卡类型/维护窗） | 部分缓存 |
| L4 排序 | 软性偏好打分（成本/成功率/TP99/限额） | 实时计算 |
| L5 兜底 | 全挂了怎么办 | 实时决策 |

### 2.2 分层的三个核心收益

1. **可缓存**：L1/L2 缓存命中率 > 95%，路由决策 < 50ms
2. **可降级**：每层都有兜底（缓存兜底/默认池/全渠道随机/拒绝）
3. **可观测**：每层打决策日志，定位问题能精确到层

### 2.3 分层的常见反模式

- **不分层**：一个函数里 1000 行 if-else，无法维护
- **层过细**：拆 8 层以上，调用链过长，性能与可读性双输
- **层职责混乱**：L3 做了排序、L4 做了过滤，层间契约失效

## 三、规则引擎设计

### 3.1 规则五元组

```
规则 = 条件(condition) + 动作(action) + 优先级(priority) + 生效时间(effect) + 灰度(gray)
```

### 3.2 选型对比

| 候选 | 适用场景 | 支付路由是否推荐 |
|---|---|---|
| Drools | 复杂金融规则（税务/保险） | ❌ 太重，热更新麻烦 |
| Aviator | 表达式级，热更新友好 | ✅ **首选** |
| QLExpress | 阿里系，脚本级 | ⚠️ 可选 |
| 自研DSL | 业务特殊 | ⚠️ 成本高 |

**支付路由推荐：Aviator 做条件匹配 + 自研做规则编排**

### 3.3 规则热更新三级缓存

```
DB（持久化） → Redis（1分钟缓存） → 本地 Caffeine（10秒缓存）
```

更新链路：运营改 DB → Canal 监听 binlog → 清 Redis → MQ 广播 → 各节点清本地缓存

### 3.4 冲突裁决

- **first-match**：命中即停（支付路由用这个）
- 上线前**沙盒回放**：用历史流量对比新旧规则集决策差异
- **优先级 0 兜底规则**：保证任何请求都能匹配到一条

## 四、渠道画像与动态权重

### 4.1 画像维度

| 类别 | 字段 |
|---|---|
| 静态元数据 | 费率、单笔限额、日累计限额、币种、卡类型、维护时间窗 |
| 实时指标 | 1m/5m/15m 成功率、TP99、错误率 |
| 状态 | 熔断状态（CLOSED/HALF_OPEN/OPEN）、当日已用额度 |

### 4.2 成功率统计

- **数据链路**：下单结果 → Kafka → Flink 滚动窗口 → Redis
- **三档窗口**：1m（敏感）/5m（平衡）/15m（稳定）
- **衰减加权**：`weight = exp(-Δt/τ)`，τ=5min
- **样本量保护**：< 50 笔不更新，用历史值兜底

### 4.3 权重公式（核心）

```
Score = α·成功率分 + β·成本分 + γ·TP99分 + δ·限额分 + ε·偏好分

推荐权重：α=0.45, β=0.25, γ=0.15, δ=0.10, ε=0.05
```

**权重设计逻辑**：
- 成功率压倒成本（0.45 vs 0.25）：成功率每降 1% ≈ 成本上浮 5%
- 成本权重次之（0.25）：在成功率接近时主导
- TP99（0.15）：体验维度
- 限额（0.10）：避免选即将打满的商户号
- 偏好（0.05）：兜底微调

### 4.4 熔断器（借鉴 Sentinel 三态）

| 信号 | 阈值 | 熔断时长 | 半开策略 |
|---|---|---|---|
| 错误率 | 5m > 10% | 30s | 10% 探活流量 |
| 慢调用 | 5m TP99 > 2s 且占比 > 30% | 60s | 5% 探活流量 |
| 连续失败 | 20 次 | 30s | 1 笔探活 |

**半开探活关键点**：
- 用真实流量探活（不专门发探活请求，省成本）
- 探活成功 → **渐进恢复**（10% → 30% → 60% → 100%）
- 探活失败 → **指数退避**（30s → 60s → 120s → 300s，上限 5 分钟）
- 熔断维度：`channel_id + merchant_id`（不整个渠道熔断）

## 五、多商户号管理

### 5.1 商户号类型

| 类型 | 用途 | 选择策略 |
|---|---|---|
| EXCLUSIVE 专用 | 大客户绑定 | 固定走该号 |
| SHARED 共享 | 通用池 | 加权轮询/一致性哈希 |
| RESERVE 备用 | 应急 | 故障时切换 |

### 5.2 负载均衡策略

- **加权轮询**：`weight = remaining_amount / daily_max_amount`，额度多的多走
- **一致性哈希**：`hash(orderId) % N`，幂等重试时锁定商户号
- **混合**：默认加权轮询，幂等重试用一致性哈希

### 5.3 限额三档预警

| 阈值 | 动作 |
|---|---|
| 80% | 降低权重（限额分下降） |
| 90% | 从候选集剔除 |
| 95% | 人工介入 |

**主动切换条件**：`剩余额度 < 预计10分钟交易额 × 1.5` → 提前切，不等打满

### 5.4 切换商户号的幂等保障

- 幂等键：`out_trade_no` 全局唯一
- 预占机制：选定后写 `route_decision_log(PENDING)`，成功改 SUCCESS
- 重试逻辑：先查决策日志，有 PENDING/SUCCESS 则复用，无则重新路由
- DB 唯一索引：`uk(out_trade_no, channel_id)`

## 六、Fallback 链与兜底

### 6.1 Fallback 链设计

```
渠道A → 失败 → 渠道B → 失败 → 渠道C → 全失败 → 拒绝/备付金
```

**关键约束**：
- **同步 Fallback 最多 1 次**（避免 TP99 不可控）
- **失败必须是明确失败**（超时不进 Fallback，进查询补偿）
- **每渠道用同一 out_trade_no**（幂等保障）

### 6.2 兜底分级

| 业务 | 兜底 | 原因 |
|---|---|---|
| ToC 支付 | 拒绝 + 引导换支付方式 | 资金不能失控 |
| 还款 | 走备付金 | 还款必须成功 |
| 提现 | 拒绝 + 人工介入 | 提现不能错 |
| 内部转账 | 走备付金 | 闭环业务 |

### 6.3 路由决策本身失败的降级链

```
规则引擎异常 → 上一次成功决策（Redis缓存）
       ↓
   Redis挂 → 本地 Caffeine（10秒前快照）
       ↓
  本地缓存挂 → 硬编码默认规则（按业务线→默认渠道池）
       ↓
       全挂 → "系统繁忙"，业务侧降级
```

## 七、灰度发布

### 7.1 灰度维度选择

| 维度 | 适用 | 优劣 |
|---|---|---|
| 商户ID取模 | **首选** | 同商户全走新/全走老，便于对账 |
| 业务线 | 隔离彻底 | 流量可能不足 |
| 金额段 | 不推荐 | 灰度组与对照组差异大，难归因 |
| 百分比随机 | 不推荐 | 同商户可能新老都走，对账复杂 |

### 7.2 灰度比例阶梯

1% → 5% → 20% → 50% → 100%，每档观察 1 小时 + 至少 1 万笔样本。

### 7.3 自动回滚阈值

- 成功率比对照组低 1% → 回滚
- TP99 比对照组高 50% → 回滚
- 对账差错率 > 0.05% → 回滚
- 渠道侧主动告警 → 回滚

## 八、退款逆向路由

### 8.1 是否必须原路退回

- **默认必须**：央行 259 号文合规要求
- **例外允许换渠道**：原渠道下线/注销/长时间故障/不支持部分退

### 8.2 逆向路由策略

```
① 原渠道 + 原商户号
② 原渠道备用商户号
③ 备付金（内部记账）
④ 其他渠道出款（标记"换渠道退款"，财务单独核算）
⑤ 人工挂账（T+2 处置）
```

### 8.3 换渠道退款的资金对平

- A 渠道收单 100，B 渠道退款 100
- 本地账务：A 应收 +100，B 应付 -100，净 0
- 渠道对账：A 对账单 100 收单，B 对账单 100 退款，各自对账无差错
- 资金中心：实际资金 A +100，B -100，总和 0 ✅
- **风险**：双重手续费（A 收一次，B 收一次），需在差错流程中标记并财务归集

## 九、与对账系统的协同

### 9.1 路由输出给对账的"决策上下文"

```json
{
  "out_trade_no": "T202607021234",
  "final_channel": "WECHAT",
  "final_merchant_id": "WX_001",
  "fallback_chain": [
    {"channel": "LIANLIAN", "result": "FAIL"},
    {"channel": "WECHAT", "result": "SUCCESS", "channel_request_id": "WX_xxx"}
  ],
  "decision_time": "2026-07-02 14:00:00.123"
}
```

### 9.2 对账如何消费

- 按 `final_channel + final_merchant_id` 拉账单 → 主体对账
- 按 `fallback_chain` 全渠道拉账单 → 副表对账（识别"试单"痕迹，避免误判短单）
- 按 `decision_time` 判断跨日 → 跨日交易勾对逻辑特殊处理
- 按 `out_trade_no` 关联支付与退款决策 → 退款溯源

### 9.3 协同设计要点

- 路由层是**事实来源**（每个决策都记录）
- 对账层是**真相校验**（用渠道账单校验路由决策的结果）
- 退款层是**逆向溯源**（基于支付决策做逆向路由）

### 11.1 路由五层
L1业务准入 → L2渠道池 → L3筛选 → L4排序 → L5兜底

`Score = 0.45·成功率 + 0.25·成本 + 0.15·TP99 + 0.10·限额 + 0.05·偏好`

CLOSED → OPEN（错误率>10%/慢调用/连续失败）→ HALF_OPEN（探活10%流量）→ CLOSED
