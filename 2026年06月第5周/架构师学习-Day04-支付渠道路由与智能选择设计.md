## 背景

支付渠道路由是支付系统的"大脑"，决定了每一笔交易走哪个渠道（微信、支付宝、银联、备付金、网联通道等）、走哪个费率档位、走哪条链路。路由设计直接关系到**成功率、成本、体验、合规**四个核心指标，是架构师必须能Hold住的核心模块。

接入 20+ 渠道、日均 2000 万笔、多业务线、灰度发布、运营可配——本题从架构设计与场景实战两个维度展开，并结合国美e卡资金路由经验做差距自评。

## 题目一（架构设计题）：渠道路由整体架构与决策引擎设计

### 作答区

#### 1. 路由分层模型

**核心思想**：把路由决策拆成"漏斗式"五层，每层只解决一个问题，层间用明确的数据契约衔接。这样做的好处是：每层可独立演化、独立测试、独立缓存、独立降级。

```
请求入参
  │
  ▼
[1] 业务准入层    —— 解决"谁能走什么"（准入与隔离）
  │ 输出：业务标签 + 候选渠道池ID
  ▼
[2] 商户/渠道池层 —— 解决"在哪个池子里选"（业务线→池映射）
  │ 输出：候选渠道集合 C1
  ▼
[3] 渠道筛选层    —— 解决"哪些渠道能接"（硬性约束）
  │ 输出：候选渠道集合 C2 ⊆ C1
  ▼
[4] 排序层        —— 解决"哪个最优"（软性偏好）
  │ 输出：有序候选列表 [c*, c2, c3...]
  ▼
[5] 兜底层        —— 解决"全挂了怎么办"
  │ 输出：最终决策 + Fallback链
  ▼
渠道下单
```

**各层职责边界**：

| 层 | 职责 | 不做什么 | 输入 | 输出 |
|---|---|---|---|---|
| L1 业务准入 | 业务线识别、风控准入、合规校验（如借贷业务不能走备付金） | 不做渠道选择 | 订单 + 业务线 + 用户标签 | 业务上下文 + 渠道池ID |
| L2 商户/渠道池层 | 业务线→渠道池映射、租户隔离、灰度分流 | 不做渠道能力判断 | 业务上下文 | 候选渠道集合（粗筛后） |
| L3 渠道筛选 | 按硬性约束过滤：渠道状态（健康/熔断/维护中）、限额（单笔/日累计）、币种、卡类型、维护时间窗 | 不做成本/成功率比较 | 候选渠道集合 | 可用渠道集合 |
| L4 排序层 | 按软性偏好打分排序：成本、成功率、TP99、业务偏好 | 不做硬性过滤 | 可用渠道集合 | 有序候选列表 + Fallback链 |
| L5 兜底层 | 全部不可用时的最终决策；决策异常时的降级 | 不再做精细选择 | 排序结果 + 异常信号 | 最终渠道 or 拒绝 |

**为什么这样分层**：
- **可缓存**：L1/L2 的结果可按"业务线+租户"缓存，命中率 > 95%；L3/L4 必须实时计算
- **可降级**：每层都有兜底（缓存兜底 / 默认池 / 全渠道随机 / 拒绝）
- **可观测**：每层都打决策日志，定位问题能精确到层
- **可灰度**：灰度逻辑收敛在 L2，不污染下层

#### 2. 路由规则引擎设计

**(1) 规则数据结构**

```sql
CREATE TABLE route_rule (
  rule_id          BIGINT PRIMARY KEY,
  rule_name        VARCHAR(128),
  biz_line         VARCHAR(32),      -- 业务线：ECOMMERCE/LOAN/RECHARGE/WITHDRAW
  priority         INT,              -- 优先级，越大越先匹配
  condition_expr   TEXT,             -- 条件表达式 DSL
  action_type      VARCHAR(32),      -- CHANNEL_POOL / PIN_CHANNEL / PIN_MERCHANT / REJECT / GRAY
  action_params    JSON,             -- 动作参数
  effect_from      DATETIME,
  effect_to        DATETIME,
  status           TINYINT,          -- 0禁用 1启用 2灰度
  gray_ratio       INT,              -- 灰度比例 0-100
  gray_dimension   VARCHAR(32),      -- MERCHANT/USER/AMOUNT/PERCENT
  version          INT,              -- 乐观锁
  update_time      DATETIME,
  operator         VARCHAR(64)
);
```

规则 = **条件（condition）+ 动作（action）+ 优先级（priority）+ 生效时间（effect）+ 灰度（gray）** 五元组。

**(2) 规则引擎选型**

| 候选 | 优点 | 缺点 | 适用 |
|---|---|---|---|
| Drools | 功能完备、业界标准 | 重、学习成本高、热更新需要 KieSession 重载 | 复杂金融规则 |
| Aviator | 轻量、表达式级、热更新友好 | 只能表达式，不能写复杂流程控制 | **推荐：支付路由** |
| QLExpress | 阿里出品、支持脚本、热更新 | 社区活跃度一般 | 阿里系项目 |
| 自研DSL | 完全可控 | 研发成本高、需配套编辑器 | 业务足够特殊时 |

**选型结论：Aviator + 自研规则编排层**
- 条件匹配用 Aviator（如 `amount > 5000 && bizLine == 'LOAN' && channelProfile.successRate > 0.95`）
- 规则编排（优先级、灰度、生效时间、动作执行）自研，因为这部分是业务语义，Aviator 表达不了
- 不选 Drools 的原因：支付路由规则没那么复杂（不是税务/保险规则），Drools 的 KieSession 热更新有性能问题，Aviator 表达式预编译后纳秒级执行

**(3) 规则冲突与优先级裁决**

- **优先级字段**：priority 越大越优先，同优先级按 rule_id 升序
- **匹配模式**：first-match（命中即停）vs all-match（全部执行后合并）—— **支付路由用 first-match**，避免多规则动作冲突
- **冲突检测**：上线新规则前，用历史流量做"沙盒回放"，对比新规则集 vs 老规则集的决策差异，差异大的规则需人工评审
- **兜底规则**：priority = 0 的默认规则（如"金额<100走微信，>=100走持牌通道"），保证任何请求都能匹配到一条

**(4) 规则热更新**

- 规则存储：DB（持久化） + Redis（缓存，1分钟过期）+ 本地 Caffeine（10秒过期）
- 更新链路：运营改 DB → 发 MQ 广播 → 各节点清本地缓存 → 下次请求从 Redis 拉最新规则
- 一致性：DB → Redis 强一致（同事务写 binlog → Canal → Redis）；Redis → 本地缓存最终一致（10秒内）
- 灰度规则：通过 `gray_ratio + gray_dimension` 控制，按 `hash(merchantId) % 100 < gray_ratio` 命中灰度规则

#### 3. 渠道画像与动态权重

**(1) 渠道画像维度**

```sql
CREATE TABLE channel_metric_realtime (
  channel_id        VARCHAR(32),
  merchant_id       VARCHAR(64),      -- 渠道+商户号维度
  -- 基础元数据
  fee_rate          DECIMAL(6,4),     -- 费率 0.0038
  single_min_amount DECIMAL(18,2),    -- 单笔下限
  single_max_amount DECIMAL(18,2),    -- 单笔上限
  daily_max_amount  DECIMAL(18,2),    -- 日累计上限
  currency          VARCHAR(8),
  card_types        JSON,             -- 支持的卡类型
  maintain_windows  JSON,             -- 维护时间窗
  -- 实时指标（每30秒刷新）
  success_rate_1m   DECIMAL(5,4),     -- 近1分钟成功率
  success_rate_5m   DECIMAL(5,4),
  success_rate_15m  DECIMAL(5,4),
  tp99_1m           INT,              -- 近1分钟TP99
  tp99_5m           INT,
  error_rate_1m     DECIMAL(5,4),
  -- 状态
  circuit_state     VARCHAR(16),      -- CLOSED/HALF_OPEN/OPEN
  daily_used_amount DECIMAL(18,2),    -- 当日已用额度
  update_time       DATETIME,
  PRIMARY KEY (channel_id, merchant_id)
);
```

**(2) 成功率统计**

- **数据来源**：渠道下单结果异步回写（成功/失败/超时），写入 Kafka → Flink 滚动窗口聚合 → Redis
- **窗口选择**：1分钟（敏感）/ 5分钟（平衡）/ 15分钟（稳定）三档并存，路由时按场景选用（大促用5分钟、日常用15分钟）
- **衰减加权**：近期样本权重高，`weight = exp(-Δt/τ)`，τ=5分钟，避免老样本稀释当前故障信号
- **样本量保护**：样本数 < 50 时不更新成功率（用历史值兜底），避免冷启动抖动

**(3) 权重计算公式**

```
Score = α · 成功率分 + β · 成本分 + γ · TP99分 + δ · 限额匹配分 + ε · 业务偏好分

其中：
  成功率分 = success_rate_5m * 100              // 0~100
  成本分   = (1 - fee_rate / max_fee_rate) * 100 // 0~100，费率越低分越高
  TP99分   = (1 - tp99_5m / max_tp99) * 100      // 0~100，越快分越高
  限额匹配分 = 剩余额度比例 * 100                  // 0~100
  业务偏好分 = 规则配置的固定加分                  // 0~30

推荐权重：α=0.45, β=0.25, γ=0.15, δ=0.10, ε=0.05
```

**调参思路**：
- 成功率权重最大（0.45）：成功率每降 1%，约等于成本上浮 0.1% × 5 = 5% 的代价，所以成功率压倒成本
- 成本权重次之（0.25）：在成功率接近时，成本主导决策
- TP99（0.15）：体验维度，避免选太慢的渠道
- 限额（0.10）：避免选即将打满的商户号
- 业务偏好（0.05）：兜底微调，不强主导

权重通过配置中心动态调整，A/B 验证后全量。

**(4) 渠道熔断机制**

借鉴 Sentinel 的三态熔断：

| 信号 | 阈值 | 熔断时长 | 半开策略 |
|---|---|---|---|
| 错误率 | 5m 错误率 > 10% | 30s | 放 10% 探活流量 |
| 慢调用 | 5m TP99 > 2s 且占比 > 30% | 60s | 放 5% 探活流量 |
| 连续失败 | 连续失败 20 次 | 30s | 放 1 笔探活 |

**半开探活**：
- 探活流量从正常流量中"借"（按比例放行），不专门发探活请求（避免空跑成本）
- 探活成功 → 关闭熔断，恢复正常流量
- 探活失败 → 重新熔断，时长翻倍（指数退避，上限 5 分钟）

**关键细节**：
- 熔断器维度：`channel_id + merchant_id`，不整个渠道熔断（一个商户号挂了不连累其他商户号）
- 熔断信号来源：异步回写 + 同步快速失败探测（超时即记一次失败，不等渠道返回）

#### 4. 渠道池与商户号管理

**(1) 多商户号负载均衡**

- **专用商户号**：某商户绑定的专用号，固定走该号（适合大客户）
- **共享商户号池**：一个渠道有 N 个共享号，按策略选
  - **加权轮询**：按剩余额度加权（`weight = remaining_amount / daily_max_amount`），额度多的多走
  - **一致性哈希**：按 `hash(orderId) % N`，保证同一订单总走同一商户号（便于幂等与对账）
  - **混合策略**：默认加权轮询，幂等重试时按 `hash(orderId)` 锁定商户号

**(2) 限额预警与切换**

- **三档预警**：
  - 80% 预警：路由时降低该商户号权重（限额匹配分下降）
  - 90% 预警：从候选集中剔除（不再分配新流量）
  - 95% 预警：人工介入，确认是否紧急提额或全量切换
- **剩余额度计算**：`daily_used_amount` 实时累加（下单时预扣 + 异步回调时确认），失败回滚
- **跨日重置**：每天 00:00 异步任务重置 `daily_used_amount = 0`，按渠道时区（微信按北京时间，部分海外渠道按 UTC）

**(3) 商户号与商户绑定关系**

```sql
CREATE TABLE channel_merchant (
  id              BIGINT PRIMARY KEY,
  channel_id      VARCHAR(32),
  merchant_id     VARCHAR(64),       -- 渠道侧商户号
  bind_type       VARCHAR(16),       -- EXCLUSIVE(专用) / SHARED(共享) / RESERVE(备用/备付金)
  bind_merchant   VARCHAR(64),       -- 我方商户ID，EXCLUSIVE时必填
  status          TINYINT,
  daily_max_amount DECIMAL(18,2),
  ...
);
```

**(4) 切换商户号的幂等保障**

- **幂等键**：`out_trade_no`（我方业务订单号），全局唯一
- **预占机制**：选定商户号后，先在 `route_decision_log` 写一条 PENDING 记录（含 out_trade_no + merchant_id），渠道下单成功后改 SUCCESS
- **重试逻辑**：若超时重试，先查 `route_decision_log` 是否已有 PENDING/SUCCESS，有则复用同一商户号，无则重新路由
- **关键约束**：同一 out_trade_no 在同一渠道只能对应一个 merchant_id，DB 加唯一索引 `uk(out_trade_no, channel_id)`

#### 5. 兜底与降级策略

**(1) 全部不可用时的兜底**

按业务类型分级兜底：

| 业务 | 兜底策略 | 原因 |
|---|---|---|
| ToC 支付 | 拒绝 + 引导用户换支付方式 | 不能让资金失控 |
| 还款 | 走备付金（内部记账） | 还款必须成功，备付金兜底 |
| 提现 | 拒绝 + 人工介入 | 提现不能错 |
| 内部转账 | 走备付金 | 闭环业务 |

**(2) 路由决策本身失败**

- 规则引擎异常：用"上一次成功的路由决策"（缓存于 Redis，按业务线 + 商户维度）
- 缓存挂了：用本地 Caffeine 兜底（10秒前的快照）
- 本地缓存也没了：用"硬编码默认规则"（按业务线 → 默认渠道池）
- 全挂了：返回"系统繁忙"，业务侧降级

**(3) 灰度发布新渠道**

- **灰度维度**：商户ID取模（最常用）/ 金额段 / 业务线 / 地域
- **比例控制**：1% → 5% → 20% → 50% → 100%，每档观察 1 小时（大促期间不灰度）
- **指标对比**：成功率差、TP99差、对账差异、客诉率
- **自动回滚**：成功率比对照组低 1% 或 TP99 高 30%，自动降比例或回滚


**(2) 渠道商户号表**：见上文 channel_merchant

**(3) 路由规则表**：见上文 route_rule

**(4) 渠道实时画像表**：见上文 channel_metric_realtime

**(5) 路由决策日志表（核心，用于回溯与分析）**

```sql
CREATE TABLE route_decision_log (
  log_id            BIGINT PRIMARY KEY,
  out_trade_no      VARCHAR(64),
  biz_line          VARCHAR(32),
  amount            DECIMAL(18,2),
  -- 决策过程
  candidate_channels JSON,        -- L3输出的候选集
  scored_channels    JSON,        -- L4排序后的列表
  final_channel      VARCHAR(32),
  final_merchant_id  VARCHAR(64),
  -- 决策元数据
  matched_rule_id    BIGINT,
  decision_time      DATETIME(3),
  decision_cost_ms   INT,
  -- 上下文（给对账系统用）
  fallback_chain     JSON,        -- 实际走的fallback链
  channel_request_id VARCHAR(64), -- 渠道侧流水
  channel_resp_code  VARCHAR(16),
  channel_resp_msg   VARCHAR(256),
  final_status       VARCHAR(16), -- SUCCESS/FAIL/TIMEOUT
  INDEX idx_out_trade_no(out_trade_no),
  INDEX idx_time_decision(final_status, decision_time)
);
```

**决策日志的两大用途**：
- **回溯**：用户客诉时，凭 out_trade_no 还原完整决策链路
- **效果分析**：统计每条规则的命中数、每个渠道的实战成功率，反哺画像

#### 7. 关键流程时序

```
[下单请求]
   │
   ├─① [同步] 业务准入层（L1）：业务线识别、风控准入        ── < 5ms
   ├─② [同步] 渠道池层（L2）：查缓存 → 候选渠道池            ── < 5ms（缓存命中）
   ├─③ [同步] 渠道筛选层（L3）：硬性约束过滤                  ── < 10ms
   ├─④ [同步] 排序层（L4）：查画像 + 打分排序                 ── < 15ms
   ├─⑤ [同步] 兜底层（L5）：生成 fallback 链                  ── < 2ms
   ├─⑥ [同步] 写 route_decision_log(PENDING)                 ── < 5ms（异步落库可优化）
   ├─⑦ [同步] 调用渠道下单 API                                ── 100~500ms（取决于渠道）
   │
   │  异步回写：
   ├─⑧ [异步] 渠道下单结果回写 route_decision_log(SUCCESS/FAIL)
   ├─⑨ [异步] 上报指标到 Kafka → 画像更新
   └─⑩ [异步] 限额预扣确认 / 回滚
```

**可异步化**：
- 决策日志落库（先写本地队列，批量 flush）
- 画像指标上报
- 限额预扣可异步（但失败要回滚，需保证最终一致）

**必须同步**：
- L1~L5 决策（业务要求实时返回）
- 渠道下单调用（同步等结果）
- 幂等校验（路由前必须查重）

**性能目标**：路由决策本身（L1~L5）< 50ms，整体下单（含渠道调用）< 800ms（TP99）。

## 题目二（场景实战题）：路由故障切换与资金成本最优场景

### 作答区

#### 场景1：渠道突发故障的秒级切换

**感知信号与时间**：
- 主信号：**错误率 + 慢调用双信号**，由熔断器在 5 分钟窗口内统计
- 次信号：业务侧反馈（客服客诉激增），作为滞后补强
- **感知时间**：5 分钟窗口 + 30 秒检测周期 = **最快 30 秒感知**（错误率突升时），慢调用型故障需 1~2 分钟

**为何不能更快**：
- 1 分钟窗口抖动大（样本量小），可能误熔断
- 探测周期太短会增加误判
- 折中：1 分钟窗口 + 10 秒检测周期，错误率阈值 15% 触发，最快 10 秒感知

**飞行中交易处置**：
- 已发往微信但未返回的交易：**不主动撤单**（可能渠道侧已扣款，撤单会引发"撤单失败+对账长单"）
- 设置 5s 超时，超时后标记 TIMEOUT，进入"渠道查询补偿"流程：
  - 异步查询微信订单状态（如查询接口也超时，每 30s 重试，最多 10 次）
  - 查询 SUCCESS → 推进业务，发异步退款（如果用户已换渠道支付）
  - 查询 FAIL/未支付 → 安全失败
  - 查询始终失败 → 人工挂账
- **避免双扣**：路由切换后，新请求走新渠道；老请求走"查询补偿"，不会重新发起支付。前端用户重试时，路由层基于 `out_trade_no` 幂等查到 PENDING 状态，复用原商户号而非重新路由

**半开探活策略**：
- 熔断 30s 后进入 HALF_OPEN
- 放行 10% 流量探活（按 merchantId 取模）
- 探活 10 笔，成功率 ≥ 80% → 关闭熔断
- 探活成功率 < 80% → 重新熔断，时长翻倍（60s → 120s → 300s，上限 5 分钟）
- 探活期间继续走其他渠道

**关键风险点**：
- 熔断恢复后流量瞬间涌入，可能再次压垮渠道 → 探活成功后**渐进恢复**（10% → 30% → 60% → 100%，每 30 秒一档）

#### 场景2：单商户号日限额打满

**预警时机**：
- 80% 预警：用 800 万时触发，路由层降低该商户号权重（限额匹配分 = 0.2 * 100 = 20）
- 90% 预警：用 900 万时触发，从候选集中剔除
- 95% 预警：人工介入

本场景 980 万（98%），应已触发 90% 预警，该商户号从候选集剔除。

**路由调整策略**：
- **首选**：切换到同渠道的备用商户号（如该商户有微信商户号2），保持"微信支付"体验一致
- **次选**：切换到其他渠道（支付宝/银联），但用户体验有差异
- **决策**：优先切备用商户号（同渠道），无备用号时才切其他渠道

**前端体验保障**：
- **用户无感切换**：用户已选"微信支付"，路由层切到微信商户号2，前端无需任何变化（商户号对用户透明）
- **跨渠道切换**：仅在严重故障时切到支付宝，前端展示"支付方式临时调整，将使用支付宝完成支付"，并允许用户主动取消
- **关键约束**：一旦切换，必须在 200ms 内完成新路由决策，否则用户感知卡顿

**额度预估**：
- 当前 980 万，剩余 20 万，预计 10 分钟内进入 50 万 → 必然超限
- 提前切换：当 `剩余额度 < 预计10分钟交易额 * 1.5` 时主动切换，不等打满

#### 场景3：成本最优路由决策

**计算各渠道得分**（用权重公式，α=0.45, β=0.25, γ=0.15, δ=0.10, ε=0.05）：

假设 max_fee_rate = 0.6%（归一化用），max_tp99 = 1000ms，5000 元均未超单笔限额，剩余额度均充足。

| 渠道 | 成功率分 | 成本分 | TP99分 | 限额分 | 偏好分 | 总分 |
|---|---|---|---|---|---|---|
| 微信支付 | 99.5 | (1-0.6/0.6)*100=0 | (1-300/1000)*100=70 | 100 | 5 | 0.45*99.5+0.25*0+0.15*70+0.10*100+0.05*5 = **65.28** |
| 支付宝 | 99.3 | (1-0.55/0.6)*100=8.3 | (1-350/1000)*100=65 | 100 | 5 | 0.45*99.3+0.25*8.3+0.15*65+0.10*100+0.05*5 = **68.10** |
| 连连 | 98.5 | (1-0.38/0.6)*100=36.7 | (1-800/1000)*100=20 | 100 | 5 | 0.45*98.5+0.25*36.7+0.15*20+0.10*100+0.05*5 = **73.16** |

**最终选择：连连**（73.16 > 支付宝 68.10 > 微信 65.28）

**单纯成本最优的风险**：
- 连连费率最低（0.38%），但成功率 98.5% 比微信低 1%
- 大促期间连连可能进一步劣化，1% 失败率意味着每 100 笔有 1 笔失败，需重试或退款，客诉率上升
- TP99 800ms 比微信慢 500ms，影响用户体验

**Fallback 链设计**（风险对冲）：

```
尝试连连 → 成功则结束
   ↓ 失败
尝试微信 → 成功则结束（同时给连连上报一笔失败）
   ↓ 失败
尝试支付宝 → 成功则结束
   ↓ 全失败
拒绝支付
```

**Fallback 链的关键约束**：
- **同步尝试 vs 异步尝试**：连连失败后是否同步尝试微信？同步链导致 TP99 不可控（800+300+350=1450ms），异步链用户体验差
- **折中方案**：同步 Fallback 仅 1 次（连连 → 微信），如果微信也失败，直接返回失败，让用户重试
- **必须幂等**：每个渠道用同一 `out_trade_no`，避免多渠道都生成交易
- **失败信号**：必须是明确失败（渠道返回 FAIL），超时不进 Fallback（超时进入查询补偿流程）

#### 场景4：路由规则灰度发布

**灰度维度选择**：
- **首选：商户ID取模**（`hash(merchantId) % 100 < 5`）—— 商户维度隔离，同一商户所有交易要么全走新渠道要么全走老渠道，便于对账
- **次选：业务线**（如只在"借款还款"业务线灰度）—— 隔离彻底，但流量可能不足
- **避免：金额段**—— 灰度组与对照组差异大，难以归因
- **避免：百分比随机**—— 同一商户可能新老渠道都走，对账复杂

**指标对比基线**：
- **同分流比下的对照组**：5% 走新渠道（网联），5% 走原渠道（按同一 hash 维度），消除时间因素
- **关键指标**：
  - 成功率差异（< 0.3% 视为可接受）
  - TP99 差异（< 50ms 视为可接受）
  - 对账差错率差异（< 0.01% 视为可接受）
  - 客诉率差异（按 P95 评估）
- **观察周期**：每档至少 1 小时 + 至少 1 万笔样本

**自动回滚触发条件**：
- 成功率比对照组低 1% → 立即回滚
- TP99 比对照组高 50% → 立即回滚
- 对账差错率 > 0.05% → 立即回滚
- 渠道侧主动告警 → 立即回滚
- 回滚动作：将灰度规则的 `gray_ratio` 置 0，规则缓存 1 分钟内全网生效

#### 场景5：退款路由的"逆向寻路"

**是否必须走原渠道**：
- **默认必须**：央行 259 号文要求"原路退回"，合规要求
- **例外允许换渠道**：
  1. 原渠道已下线（如下架的持牌通道）
  2. 原渠道商户号已注销
  3. 原渠道长时间故障（> 24 小时未恢复）
  4. 原渠道不支持部分退款且业务需要部分退

**换渠道退款的资金流对平**：

举例：原支付 A 渠道收单 100 元，退款走 B 渠道出款 100 元。
- **本地账务**：
  - A 渠道应收：+100（用户支付时入账）
  - B 渠道应付：-100（退款时出账）
  - 备付金账户：净 0（业务角度）
- **渠道对账**：
  - A 渠道对账单：100 元收单（正常）
  - B 渠道对账单：100 元退款（正常）
  - 各自对账无差错
- **资金中心对账**：本地账务应收应付 = 0，但资金中心实际资金：A 渠道 +100，B 渠道 -100，总和 = 0 ✅

**关键挑战**：换渠道退款时，原支付的"手续费"留在 A 渠道（如 0.6 元），退款时 B 渠道可能再收一笔手续费（如 0.6 元），导致**双重手续费损失**。处置方案：
- 业务上接受（小概率事件，成本可控）
- 或在差错处理流程中标记"换渠道退款"，财务做单独的成本归集

**逆向路由策略**：

```
退款请求
   │
   ├─① 优先：原渠道 + 原商户号
   │     ├─ 原渠道可用 → 走原渠道退款
   │     └─ 原渠道不可用 → ↓
   │
   ├─② 次选：原渠道的备用商户号
   │     ├─ 有备用号 → 走备用号
   │     └─ 无备用号 → ↓
   │
   ├─③ 再次：备付金（内部记账）
   │     ├─ 业务允许 → 走备付金
   │     └─ 不允许 → ↓
   │
   ├─④ 兜底：换其他渠道出款
   │     ├─ 选成本最低的可用渠道
   │     └─ 标记"换渠道退款"，财务单独核算
   │
   └─⑤ 终极：人工挂账
        └─ 进入差错处理流程，T+2 人工处置
```

**关键约束**：
- 退款路由的"决策日志"必须关联原支付的路由决策日志（通过 `out_trade_no` 关联），形成完整链路
- 退款金额不能超过原支付金额（部分退需校验剩余可退金额）
- 退款必须幂等（同一 `refund_no` 全局唯一）

## 三、与 Day03 的衔接

对账（Day03）发现的差错单根因很多在路由：
- 路由切换时机不当 → 跨日切换导致对账跨日
- 商户号切换 → 渠道侧账单出现"非本商户号"交易 → 短单
- Fallback 链执行 → 同一业务单在多渠道都有"试单"痕迹 → 对账干扰

**路由层向对账层输出的"决策上下文"**：

```json
// route_decision_log 中的关键字段，供对账系统消费
{
  "out_trade_no": "T202607021234567890",
  "final_channel": "WECHAT",
  "final_merchant_id": "WX_001",
  "fallback_chain": [
    {"channel": "LIANLIAN", "merchant_id": "LL_001", "result": "FAIL", "resp_code": "E001"},
    {"channel": "WECHAT", "merchant_id": "WX_001", "result": "SUCCESS", "channel_request_id": "WX_xxx"}
  ],
  "decision_time": "2026-07-02 14:00:00.123",
  "channel_request_id": "WX_xxx"
}
```

**对账系统如何使用**：
- 按 `final_channel + final_merchant_id` 拉取渠道账单 → 主体对账
- 按 `fallback_chain` 中的所有渠道拉账单 → 副表对账（识别"试单"痕迹，避免误判为短单）
- 按 `decision_time` 判断跨日 → 跨日交易的勾对逻辑特殊处理
- 按 `out_trade_no` 关联支付与退款决策日志 → 退款溯源

**协同设计要点**：
- 路由层是"事实来源"（每个决策都记录）
- 对账层是"真相校验"（用渠道账单校验路由决策的结果）
- 退款层是"逆向溯源"（基于支付决策做逆向路由）

## 本日小结

- 路由是支付系统的"大脑"，分层（L1~L5）是核心抽象
- 规则引擎选 Aviator + 自研编排，热更新靠"DB+Redis+本地缓存"三级缓存
- 渠道画像是路由的眼睛，实时指标靠 Flink 滚动窗口 + 衰减加权
- 熔断借鉴 Sentinel 三态机，半开探活用真实流量而非探活请求
- 多商户号负载均衡按"剩余额度加权轮询 + 幂等锁定"
- 兜底分级（ToC 拒绝/还款走备付金/提现人工）
- 退款走"逆向路由"：原渠道 → 备用号 → 备付金 → 其他渠道 → 人工挂账
- 路由与对账协同：决策日志输出 fallback_chain，让对账能还原路由轨迹
