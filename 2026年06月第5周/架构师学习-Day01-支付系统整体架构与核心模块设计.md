## 背景

你所在的"蛋壳钱包"要从单一通道升级为多渠道聚合支付平台，同时承接内部国美e卡资金路由、平安健康险保险商城保费收缴、医疗在线问诊充值等多业务线。CTO 把白板推过来：先把支付系统的整体架构画清楚，讲明白一笔支付从用户点击付款到资金到账经历哪些环节、有哪些核心模块、为什么这么拆。

支付是资金链路，任何一环出错都可能造成资损、客诉、监管问题。架构师不仅要会画图，更要讲清楚每个模块的职责边界、调用关系、异常路径与选型权衡。本日聚焦于**整体架构与核心模块**，为后续幂等、对账、资金安全、退款分账打基础。

#### 1. 核心模块拆分与职责边界

整体架构（自上而下分层，业务侧 → 聚合层 → 渠道层 → 资金层 → 支撑层）：


各模块职责边界：

| 模块 | 职责 | 不做什么 |
|---|---|---|
| **收银台 Cashier** | 支付方式展示、用户交互、风控前置校验、C端Token换取、跳转/拉起 | 不直接调用渠道，不做清结算 |
| **支付网关 Gateway** | 统一API接入、签名验签、幂等、限流熔断、路由分发、流水记录 | 不维护订单状态机（交给订单中心），不直接持有渠道凭证 |
| **订单中心 PayOrder** | 支付订单CRUD、状态机管理、订单与流水一致性、订单查询 | 不做渠道调用，不做资金核算 |
| **渠道路由 Router** | 按策略匹配渠道（成本/成功率/限额/卡BIN/灰度）、降级切换、热插拔 | 不持有订单状态 |
| **渠道适配 Channel** | 对接微信/支付宝/银联/e卡，屏蔽渠道差异，统一抽象 | 不做业务路由决策 |
| **风控 RiskCtrl** | 限额（单笔/日累计/月累计）、黑名单、设备指纹、防盗刷、合规拦截 | 不直接拒付，只给拦截信号 |
| **通知中心 Notify** | 业务回调、渠道回调分发、重试补偿、MQ解耦 | 不修改订单状态，只触发事件 |
| **账户/账务** | 内部账户余额、记账（复式记账）、可用余额冻结/扣减 | 不与渠道直接通信 |
| **清结算 Settlement** | T+1结算、渠道对账文件解析、资金划拨指令、结算批次 | 不负责实时扣款 |
| **对账 Reconcile** | 长短款识别、差异挂账、差异处理（补单/退款/挂账） | 不修改业务订单 |

#### 2. 同步链路与异步链路

**同步链路（用户付款实时）**：

```
用户 → 收银台 → 支付网关 → 订单中心(创建INIT) → 风控校验
     → 路由匹配渠道 → 渠道适配器下单 → 渠道(微信/支付宝)
     → 返回支付凭证(预支付参数/prepay_id)
     → 收银台拉起SDK → 用户输密码 → 渠道扣款(异步)
     → 收银台轮询订单状态(超时2-3秒) → 未拿到结果返回"处理中"
```

**异步链路（回调、清结算、对账）**：

```
渠道扣款成功 → 渠道异步回调 → 支付网关(验签) → 通知中心
            → MQ → 订单中心(状态流转 PROCESSING→SUCCESS)
            → MQ → 业务系统(更新业务订单为已支付)
            → MQ → 账务系统(记账)

夜间 T+1：
渠道对账文件 → 对账系统 → 与本地订单/账务比对 → 差异挂账 → 人工/自动处理
```

**为什么有些链路要异步？**

- **回调必须异步**：渠道回调时机不可控（最快秒级，最慢几分钟甚至小时级），同步阻塞会拖垮网关线程池。用 MQ 削峰 + 解耦，订单中心只关心"事件已到"而非"渠道调用结果"。
- **业务通知必须异步**：业务系统（电商、保单）可能慢、可能重试，MQ 隔离避免渠道回调被业务系统拖垮导致回调超时重试。
- **清结算/对账天然异步**：T+1 批量处理，数据量大，与实时链路隔离避免影响支付。
- **账务记账异步**：复式记账涉及多表写入，同步会拖慢支付链路；用 MQ 保证最终一致即可。
- **同步路径仅保留必要的**：创建订单、调用渠道下单是同步的（用户在等），其余尽量异步。

#### 3. 渠道路由策略与可扩展引擎

**路由策略维度**：

1. **成本优先**：费率低的渠道优先（如 0.38% vs 0.6%）
2. **成功率优先**：渠道近5分钟成功率低于阈值自动降级
3. **限额过滤**：单笔限额、日累计限额（如微信单笔5万、单日20万）
4. **卡BIN路由**：信用卡按卡BIN路由到对应银行通道（费率更低）
5. **业务类型路由**：保险→指定保险支付通道，e卡→e卡专用通道
6. **灰度切换**：新渠道先按1%灰度，逐步放量
7. **降级熔断**：渠道错误率超阈值自动熔断，切备渠道
8. **地区/币种**：跨境按币种路由到对应国际通道

**可扩展策略引擎设计**（新增渠道不改代码）：

```java
// 策略接口
public interface RouteStrategy {
    RouteResult match(RouteContext ctx);   // 返回匹配的渠道 + 优先级
    int order();                            // 策略优先级
}

// 渠道配置（DB/配置中心，热生效）
RouteRule {
    channelId, bizType, amountMin, amountMax, 
    cardBinRegex, region, weight, 
    successRateThreshold, enabled
}

// 路由引擎
public class RouteEngine {
    List<RouteStrategy> strategies;  // Spring注入，按order排序
    
    public Channel route(PayRequest req) {
        RouteContext ctx = buildCtx(req);
        List<Channel> candidates = channels.stream()
            .filter(c -> matchRule(c, req))   // 规则过滤（限额、卡BIN、业务类型）
            .filter(c -> healthy(c))          // 健康度过滤（成功率、熔断状态）
            .sorted(comparing(c -> cost(c)))  // 成本排序
            .collect(toList());
        for (RouteStrategy s : strategies) {
            Channel picked = s.match(ctx).getChannel();
            if (picked != null) return picked;
        }
        return candidates.get(0);  // 默认成本最优
    }
}
```

关键点：
- **策略SPI化**：策略实现走 SPI/Spring Bean，新增策略加类即可
- **规则数据化**：限额、卡BIN、业务类型走配置中心，不写死代码
- **健康度实时计算**：基于滑动窗口统计的成功率、P99 延迟、熔断状态
- **路由日志全留痕**：每次路由决策记录候选集、命中规则、最终渠道，便于排查与回放

#### 4. 业务订单 vs 支付订单

**关系**：业务订单（电商订单/保单/充值单）与支付订单是一对多关系。

```
业务订单(电商单SO202606290001)
   ├─ 支付订单1(PayOrder-微信, 成功)
   ├─ 支付订单2(PayOrder-支付宝, 已关闭) ← 用户切渠道重试
   └─ 支付订单3(PayOrder-微信退款, REFUNDED)
```

**为什么要拆分？**

1. **职责分离**：业务订单关心"用户买了什么、收货地址"，支付订单关心"通过哪个渠道、扣了多少、状态如何"。混在一起会导致业务表频繁被支付链路更新污染。
2. **多支付方式**：用户可能先选微信失败，再切支付宝，一笔业务订单对应多笔支付订单（只有最后一笔成功）。
3. **退款/分账**：一次支付可能多次退款（部分退款），支付订单是退款的最小单元。
4. **多业务线复用**：支付能力是公共能力，不绑定电商业务，保险、问诊、e卡都能复用同一套支付订单模型。
5. **数据隔离与生命周期**：业务订单7年留存（合规），支付订单可按渠道分库分表，独立归档。

**一对多处理**：
- 业务订单记录"当前有效支付单ID"
- 同一时刻只允许一笔支付订单处于 PROCESSING，新发起需先关闭旧的（状态机保证）
- 业务订单状态由支付订单事件驱动：支付成功→业务订单已支付；退款成功→业务订单已退款

**多对多（少见，分账场景）**：
- 一笔业务订单（如保险分期）对应多笔支付订单（多期）
- 多笔业务订单合并支付（购物车一次结账）→ 一笔支付订单关联多个业务订单号（用关联表 pay_order_biz_ref 维护）

## 题目二（状态机设计题）：一笔支付订单的全生命周期与状态机设计

一笔支付订单从创建到最终完成（或失败/退款），会经历多个状态。请回答：

1. 画出支付订单的完整状态机，至少包含：INIT/待支付、处理中（PROCESSING）、成功（SUCCESS）、失败（FAILED）、关闭（CLOSED）、退款中（REFUNDING）、已退款（REFUNDED）。标注每个状态的进入条件、退出条件、允许的状态流转路径。
2. 哪些状态流转是"危险"的（一旦流转错误就可能导致资损或重复扣款）？例如：PROCESSING → SUCCESS 与 PROCESSING → FAILED 之间的并发竞争如何防止？状态机如何与幂等结合？
3. 用户在收银台点击"付款"后，应用、支付网关、渠道（微信/支付宝）之间的时序是怎样的？如果渠道回调晚于用户主动查询，如何保证状态最终一致？
4. 一个常见 Bug：用户付款成功，但订单系统一直显示"处理中"。请从架构角度列出至少 5 个可能的根因，并给出对应的排查思路与系统级补偿方案。

### 作答区

#### 1. 支付订单完整状态机

```
                  ┌─────────┐  创建支付单
                  │  INIT   │  (待支付)
                  └────┬────┘
                       │ 调用渠道下单成功 / 拉起SDK
                       ▼
                  ┌─────────────┐
        ┌────────│ PROCESSING  │────────┐
        │        │  (处理中)    │        │
        │        └─────────────┘        │
        │              │                 │
   用户超时未付    渠道回调成功      渠道回调失败/关单
   /主动关单        /查询确认         /主动关单
        │              │                 │
        ▼              ▼                 ▼
   ┌────────┐   ┌─────────┐        ┌─────────┐
   │ CLOSED │   │ SUCCESS │        │ FAILED  │
   │ (已关闭)│   │ (已支付) │        │ (已失败) │
   └────────┘   └────┬────┘        └─────────┘
                     │ 退款发起
                     ▼
                ┌────────────┐
                │ REFUNDING  │  退款中
                └─────┬──────┘
                      │ 退款回调成功
                      ▼
                ┌────────────┐
                │ REFUNDED   │  已退款
                │ (可多次部分退款累计)│
                └────────────┘
```

**状态流转条件与允许路径**：

| 当前状态 | 目标状态 | 进入条件 |
|---|---|---|
| INIT | PROCESSING | 渠道下单成功，凭证返回 |
| INIT | CLOSED | 用户超时未付（默认15-30分钟）/ 业务主动关单 |
| PROCESSING | SUCCESS | 渠道回调成功 + 验签通过 + 幂等校验通过 |
| PROCESSING | FAILED | 渠道回调明确失败（余额不足、卡片拒绝等） |
| PROCESSING | CLOSED | 主动关单（需先调渠道关单接口确认） |
| SUCCESS | REFUNDING | 业务发起退款请求 |
| REFUNDING | REFUNDED | 渠道退款回调成功 |
| REFUNDING | SUCCESS | 退款失败，回退到已支付（少见，需人工介入） |

**禁止的逆向流转**：SUCCESS → PROCESSING、REFUNDED → SUCCESS（除非全额退款失败回退，需特殊审批）、FAILED → SUCCESS（绝对禁止，必须通过重新创建支付订单解决）。

#### 2. 危险的状态流转与并发竞争

**最危险的三处**：

**① PROCESSING → SUCCESS vs PROCESSING → FAILED 并发竞争**

场景：渠道异步回调线程A要置 SUCCESS，同时本地超时关单线程B要置 FAILED。如果加锁/版本号不到位，可能出现"已退款给用户但订单显示FAILED"或"已扣款但订单显示FAILED导致重复支付"。

防护手段：
- **乐观锁 + 状态前置校验**：`UPDATE pay_order SET status=SUCCESS, version=version+1 WHERE id=? AND status=PROCESSING AND version=?`，更新影响行数=0 则说明状态已变更，放弃本次流转。
- **状态机白名单**：每个状态有"允许的下一状态集合"，禁止逆向与跳跃。状态机引擎统一拦截非法流转。
- **幂等键 + 唯一索引**：渠道流水号在 pay_order 上建唯一索引，同一笔渠道回调即使并发也只会有一次生效。
- **分布式锁**：以 pay_order_id 为 key 加 Redisson 锁（持锁30秒），同一订单串行化流转，仅作为兜底，主路径靠数据库乐观锁。

**② SUCCESS → REFUNDING 与重复退款**

防护：退款单号（业务方生成）建唯一索引；退款金额校验累计不超过原支付金额；退款发起前必须校验订单为 SUCCESS 且未被冻结。

**③ CLOSED 状态下被回调置 SUCCESS**

场景：本地超时关单了，但渠道实际已扣款成功，回调晚到。
- 必须有"关单前调渠道关单接口"的步骤，但渠道关单可能失败（用户已付款）
- 回调到达时若订单已 CLOSED，要走"退款补偿"流程而非置 SUCCESS，避免业务方误发货
- 关键设计：**CLOSED 状态接收 SUCCESS 回调 → 不置 SUCCESS，触发自动退款**

**状态机与幂等结合**：
- 状态流转的"动作"必须幂等：同一渠道流水号重复回调 → 落到同一支付订单 → 状态机判断当前已是目标状态则直接返回成功（让渠道不重试）
- 状态机表设计：`from_status, to_status, action, biz_check` 四元组，所有流转走配置化表，禁止代码里裸写 setStatus

```
   │   轮询查询    │              │              │             │            │
   │──────────────>│  查询订单    │              │             │            │
   │               │─────────────>│ queryPayOrder │            │            │
   │               │              │  状态PROCESSING          │            │
   │   返回"处理中"│              │              │             │            │
   │<──────────────│                                                          │
   │                                                                          │
   │                              渠道异步回调                                │
   │                              <─────────────────────────────────────────│
   │                              验签 → 通知中心 → MQ                        │
   │                              订单中心消费MQ → 置SUCCESS                  │
   │                              → MQ → 业务系统更新业务订单                 │
   │                              → MQ → 账务记账                             │
   │                                                                          │
   │   二次查询    │              │              │             │            │
   │──────────────>│─────────────>│              状态SUCCESS                 │
   │   返回成功    │              │              │             │            │
   │<──────────────│                                                          │
```

**回调晚于查询的最终一致保障**：

1. **主动查询补偿**：订单进入 PROCESSING 后启动延迟查询任务（30s、1min、5min、10min、30min 五次指数退避），主动调渠道查询接口拉取真实状态。即使回调丢失，最迟30min也能纠正。
2. **对账兜底**：T+1 对账时发现"渠道已成功 + 本地PROCESSING"会自动补单置 SUCCESS（同时触发后续业务流程）。
3. **回调与查询互斥幂等**：两者都通过 `渠道流水号 + 状态机前置校验` 保证只生效一次，先到的执行流转，后到的判断已是目标状态直接返回成功。
4. **MQ 重试机制**：回调入库 → MQ 消费失败会指数退避重试，进入死信队列后人工介入。

#### 4. "付款成功但订单显示处理中" Bug 排查

**根因清单（至少5个）与排查思路**：

| # | 根因 | 排查思路 | 系统级补偿 |
|---|---|---|---|
| 1 | 渠道回调未送达 / 网关验签失败丢弃 | 查网关访问日志、回调失败队列、验签失败日志 | 启用主动查询补偿任务；网关验签失败入死信队列告警 |
| 2 | 回调到达但 MQ 消费失败 / 消费者宕机 | 查 MQ 堆积、消费者重启日志、rebalance情况 | MQ 集群高可用 + 监控告警 + 死信队列人工处理 |
| 3 | 订单中心乐观锁冲突 / 状态机拒绝流转 | 查订单中心错误日志、状态流转记录 | 修复状态机配置；冲突自动重试；遗留订单走T+1对账补单 |
| 4 | 业务系统更新业务订单失败（下游慢） | 查业务系统日志、数据库锁等待 | MQ 重试 + 业务订单查询时实时反查支付订单（补偿读） |
| 5 | 渠道返回 SUCCESS 但本地网络抖动超时未收到响应 | 查渠道侧对账文件、本地未到账记录 | 主动查询任务 + T+1 对账补单 |
| 6 | 风控拦截了后续通知但未通知订单中心 | 查风控日志、通知链路 | 风控结果必须落库 + 通知中心订阅风控事件 |
| 7 | 分库分表路由错误 / 数据库主从延迟读到从库旧状态 | 查数据分布、从库延迟监控 | 强制读主库查订单状态；分片键校验 |

**系统级补偿方案**：
- **实时补偿**：主动查询任务（指数退避，覆盖 PROCESSING 状态所有订单）
- **批量补偿**：T+1 对账，每日凌晨全量比对渠道-本地-账务三流
- **应急补偿**：运维工具一键查询指定订单在渠道侧真实状态并补单
- **预防**：全链路 traceId 贯穿（收银台→网关→订单→渠道→回调），监控核心指标：回调成功率、PROCESSING 平均停留时长、对账差异率

## 题目三（选型权衡题，进阶）：单体支付 vs 微服务支付 vs 云原生支付

针对不同业务规模，支付系统架构演进路径不同：

1. 中小业务（日订单 < 10w）应该选单体还是微服务？为什么？哪些模块必须独立、哪些可以合并？
2. 大型业务（日订单百万~千万级，双11峰值 QPS 数万）为什么要做支付微服务化+单元化部署？同城双活/异地多活对支付系统的特殊要求是什么？
3. 你如何评估引入"云原生支付"（如 AWS Payment、Stripe、收钱吧聚合）vs 自研？决策维度有哪些？

### 作答区

#### 1. 中小业务（日订单 < 10w）选型

**结论：单体优先，模块化拆分但不过早微服务化。**

**推荐架构**：单一支付应用，内部按模块分包（gateway / order / channel / risk / notify / settlement），共享同一数据库（按模块分 schema）。

**理由**：
- 10w 日订单 ≈ 平均 1.2 QPS，峰值撑死 50 QPS，单机 4C8G 轻松扛住
- 微服务带来的 RPC、注册中心、链路追踪、分布式事务成本远超收益
- 团队规模小（通常 < 5人），微服务化反而增加协作成本

**必须独立的模块（即使单体也独立部署/独立数据库）**：
- **对账系统**：T+1 跑批，数据量大，独立避免影响实时支付
- **风控数据查询**：可能对接外部大数据，独立防止拖垮主链路

**可合并的模块**：收银台、网关、订单、渠道、通知 全部在一个应用里，按包隔离即可。

**演进信号**：当出现以下情况再考虑拆分——
- 单库 QPS > 3000 或数据量 > 5000w
- 某模块（如对账）跑批影响在线支付
- 团队按模块分组开发，相互合并代码冲突严重

#### 2. 大型业务（百万~千万级日订单，双11峰值 QPS 数万）的微服务化 + 单元化

**为什么要微服务化**：
- **故障隔离**：渠道模块挂了不影响订单查询；风控慢查询不拖垮支付主链路
- **独立扩容**：双11前渠道模块扩容10倍，对账模块不扩容
- **团队自治**：每服务一个团队，独立迭代发布
- **技术栈灵活**：风控用 Java + Flink，对账用 Spark，网关用 Go 都可以

**为什么要单元化部署**：
- **延迟**：跨机房 RT 几毫秒到几十毫秒，支付链路对延迟敏感（用户在等），同城单机房 RT 必须 < 1ms
- **容灾**：单机房故障整体不可用是灾难，单元化让每个单元独立承接流量
- **数据库瓶颈**：单一数据库撑不住千万级订单，必须按单元分片

**单元化关键设计**：
```
路由层（按用户ID/cuid 分流）
   ├─ 单元A（机房A）：承担 cuid % 4 ∈ {0,1}
   │    └─ 完整闭环：网关 + 订单 + 账务 + DB分片A
   ├─ 单元B（机房B）：承担 cuid % 4 ∈ {2,3}
   │    └─ 完整闭环：网关 + 订单 + 账务 + DB分片B
   └─ 中心单元：跨单元协调（对账、清结算、风控全局视图）
```

- **流量封闭**：同一用户的支付、查询、退款必须在同一单元完成
- **数据封闭**：每个单元只持有自己的 DB 分片，禁止跨单元访问业务库
- **全局唯一**：分布式 ID（雪花/Leaf）保证订单号跨单元不冲突
- **跨单元场景**：极少，仅对账、风控全局聚合走中心单元

**同城双活 / 异地多活对支付的特殊要求**：

| 要求 | 同城双活 | 异地多活 |
|---|---|---|
| 数据同步 | 强一致（RPO=0，主从同步复制） | 最终一致（单元化，跨地域异步） |
| 切换条件 | 主机房故障自动切换 | 单元故障切流到其他单元 |
| 一致性要求 | 资金强一致，账户余额不可超扣 | 单元内强一致，跨单元最终一致 |
| 对账频率 | T+1 | T+1 + 实时差异检测 |
| 监管合规 | 同城满足 | 需评估跨境数据合规 |
| **支付特殊** | 渠道回调必须路由到原始下单单元（按订单号反查路由表） | 同左，且需处理"用户在A地下单，B地查询"的跨单元读 |
| **资金安全** | 双写需幂等 + 对账兜底 | 单元间资金划拨需走清结算，不直接跨单元扣账 |

**支付系统在多活架构的难点**：
- 渠道是单一接入点，回调必须能定位到原始单元 → 路由表（订单号→单元）需全局可查
- 账户余额是强一致资源，跨单元不能直接扣减，必须通过账务中心或预扣+异步结算
- 退款必须路由到原支付订单所在单元，否则查不到原单

#### 3. 云原生支付 vs 自研的决策

**决策维度**：

| 维度 | 云原生支付（Stripe/收钱吧/AWS） | 自研 |
|---|---|---|
| **业务规模** | 中小业务、跨境业务、单一渠道为主 | 大型业务、多业务线、需要强控资金链路 |
| **资金成本** | 渠道费率 + 平台服务费 | 渠道费率 + 自研服务器/人力 |
| **人力成本** | 低（接入1-2周） | 高（持续团队维护） |
| **定制化** | 受限（风控规则、路由策略、对账口径） | 完全可控 |
| **合规与监管** | 平台兜底（PCI-DSS、金融牌照） | 自负其责（需牌照、过等保、过审计） |
| **资金安全** | 平台担保 | 自研需建完整风控、对账、补偿 |
| **数据归属** | 部分数据在平台侧 | 全部自有 |
| **故障责任** | 平台 SLA，但赔付有限 | 自主可控，责任100%自担 |
| **跨境能力** | 强（多币种、多渠道、税务） | 弱（需逐一接入国际渠道） |
| **业务连续性** | 平台出问题影响所有用户 | 自研可针对性降级 |

**自研的场景**：
- 蛋壳钱包（持牌支付机构）必须自研，资金链路不可托管
- 国美e卡资金路由（预付卡+多渠道聚合）业务逻辑复杂，第三方平台无法支持
- 大型电商/保险，资金规模大，每年渠道费率谈判和路由优化可省千万级成本

**用云原生的场景**：
- 初创公司快速上线，无支付研发团队
- 跨境电商，本地化渠道对接成本高
- 内部管理系统的非核心支付场景

**混合方案（最常见于中型公司）**：
- 自研支付核心（订单、账务、对账）+ 接入聚合支付（收钱吧）作为渠道之一
- 风控、路由、清结算自研，渠道适配层引入第三方 SDK
- 关键资金链路自研，非关键（如营销补贴发放）用第三方
