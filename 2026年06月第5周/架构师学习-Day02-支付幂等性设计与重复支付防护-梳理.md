## 一、幂等在支付系统中的定位

### 1.1 核心一句话

> **支付幂等 = 用唯一键 + 状态机 + 对账兜底，把"至少一次"的网络语义收敛为"恰好一次"的业务语义**

支付链路跨进程、跨服务、跨渠道，每一跳都可能是"至少一次"投递（HTTP 重试、MQ 重投、回调重发）。幂等的本质是把这种不确定性"折叠"成业务上的"恰好一次"。

### 1.2 幂等的三大特征

| 特征 | 含义 | 工程影响 |
| --- | --- | --- |
| 唯一键稳定 | 同一业务动作重试 N 次，幂等键完全一致 | 不依赖时间戳、UUID、IP |
| 状态可查 | 重试请求能拿到首次执行的结果 | 必须存 response_snapshot |
| 失败可重试 | 首次失败时允许重新执行 | status=FAILED 可被抢占更新 |

### 1.3 幂等 ≠ 唯一索引

唯一索引只能告诉你"重复了"，无法告诉你"上次执行到哪、结果是什么"。完整的幂等必须有 `status` 字段承载执行状态机。

## 二、全链路幂等点速记

```
业务方 ──biz_order_no+biz_system_id──> 网关
网关   ──pay_order_no────────────────> 订单中心
网关   ──out_trade_no────────────────> 渠道
渠道   ──channel_no──────────────────> 网关回调
订单中心 <──channel_no (MQ msg_key)── 通知中心
业务方 <──pay_order_no+biz_event_type─ 通知中心
账务   ──pay_order_no+account+direction─ 账务记账
```

**核心规则**：
- 跨进程边界换 key，同进程内不换
- key 落库时记录上游 key（parent_idempotent_key），全链路可追溯
- 回调链路用渠道方 key（最权威）

## 三、四种幂等方案对照速记

| 方案 | 性能 | 可靠性 | 适用 | 资金场景 |
| --- | --- | --- | --- | --- |
| Redis SETNX | 极高 | 中（丢数据） | 防连点、限流去重 | ❌ 不可单独用 |
| DB 唯一索引 | 中 | 高 | 支付下单、退款、记账 | ✅ 推荐 |
| 状态机乐观锁 | 中 | 高 | 并发状态流转 | ✅ 必备 |
| Token 领票 | 高 | 高 | 收银台、表单提交 | ✅ 推荐前置 |

**组合策略**（纵深防御）：

```
Redis 快速过滤（80% 重复请求）
   ↓
Token 业务防重（业务方领票）
   ↓
DB 唯一索引（权威记录）
   ↓
状态机乐观锁（并发兜底）
```

## 四、幂等表设计要点

### 4.1 字段速记

```sql
idempotent_record (
    idempotent_key,           -- 幂等键
    biz_type,                 -- 业务类型（区分支付/退款/记账）
    request_snapshot,         -- 原始请求
    status,                   -- 0-INIT 1-PROCESSING 2-SUCCESS 3-FAILED
    response_snapshot,        -- 首次响应（成功后存）
    expire_at,                -- 过期时间
    UNIQUE KEY (idempotent_key, biz_type)
)
```

### 4.2 status 字段的三大作用

1. **区分"执行中"与"已完成"**：并发请求拿到 PROCESSING 时返回"处理中"
2. **失败可重试**：FAILED 状态允许抢占更新为 INIT 重新执行
3. **僵尸记录恢复**：超时 PROCESSING 由定时任务扫描，调渠道查询补全状态

### 4.3 正确的幂等执行模板

```
查询 → SUCCESS 返回快照 / PROCESSING 返回处理中 / FAILED 抢占重试
     → 抢占式 INSERT（或 UPDATE FAILED→PROCESSING）
     → 执行业务
     → 成功：UPDATE SUCCESS + 存响应快照
     → 失败：UPDATE FAILED
```

## 五、重复支付的五道成因

| # | 类别 | 触发条件 |
| --- | --- | --- |
| 1 | 用户行为 | 连点付款按钮 |
| 2 | 客户端 | 客户端超时重试 |
| 3 | 网络 | 网关响应丢失 |
| 4 | 服务端 | 业务方超时重试 |
| 5 | 渠道侧 | 回调延迟 + 上游"补单"逻辑错误 |

**最隐蔽**：场景 5——业务方查询超时后**重新发起支付**而非调查询接口，绕过幂等键新建支付单。

## 六、四道防线纵深防御

| 防线 | 能拦 | 拦不住 |
| --- | --- | --- |
| 客户端防连点 | 用户主动连点 | API 直调、客户端 BUG |
| 服务端幂等表 | 所有重复请求 | 幂等键生成错误 |
| 状态机乐观锁 | 并发状态流转 | 跨表复合状态 |
| 渠道查询+T+1对账 | 回调丢失、状态滞后 | 渠道本身不一致 |

**架构师原则**：四道防线各有失效场景，必须组合。**幂等表覆盖 80%**，**状态机兜底并发**，**查询+对账补网**。

## 七、"渠道已扣款、本地仍 PROCESSING"差异单处理

### 7.1 处理流程

```
对账识别差异 → 调渠道查询二次确认 → 检查本地订单当前状态
            → 补单事务（乐观锁更新 + 多状态校验）
            → 异步触发业务通知 + 账务记账
            → 事后核对
```

### 7.2 直接补单的四大资损风险

| 风险 | 描述 | 正确做法 |
| --- | --- | --- |
| 业务订单已关单 | 补单置 SUCCESS 导致已关单被误发货 | 补单前校验业务订单状态 |
| 已存在退款 | 钱已退但订单显示已支付 | 补单前查退款记录 |
| 账务已冲正 | 同笔支付记一正一反两笔账 | 补单前校验账务记录 |
| 渠道实际已退款 | 对账文件时点是 SUCCESS，当前是退款中 | Step3 必须查当前状态 |

### 7.3 补单事务伪代码

```
BEGIN;
  1. SELECT FOR UPDATE 锁定支付订单
  2. 校验状态仍为 PROCESSING
  3. 调渠道查询确认当前 SUCCESS
  4. 校验无退款记录
  5. 校验账务未记账
  6. 校验业务订单未关单
  7. UPDATE status=SUCCESS
  8. 记录审计日志
COMMIT;
COMMIT 后异步：MQ 通知业务/账务/通知中心
```

## 八、高并发幂等表优化与新风险

| 优化 | 新风险 | 缓解 |
| --- | --- | --- |
| 分库分表 | 跨片查询、分片键设计 | 严格规范、运维工具 |
| 本地缓存 | 多实例不一致、重启丢失 | 仅作加速层、DB 权威 |
| 异步落库 | 数据丢失 | **资金场景禁止** |
| 热点优化 | 限流误杀正常请求 | 精细化限流、白名单 |

**优化组合**：

```
本地缓存（Caffeine 5min）
   ↓ 未命中
Redis（SETNX 30min）
   ↓ 未命中
DB 幂等表（分库分表，永久）
```

**架构师原则**：资金幂等"宁慢勿错"。优先水平扩展（分库分表），再用本地缓存过滤，绝不用异步落库。

## 九、跨服务分布式幂等

### 9.1 分层混合策略

- **跨进程边界换 key**：进入新执行单元生成自己的 key
- **同进程内不换 key**：避免无意义转换
- **key 血缘可追溯**：每条记录存 parent_idempotent_key
- **回调链路用渠道 key**：最权威

### 9.2 TCC vs Saga 幂等对比

| 维度 | TCC | Saga |
| --- | --- | --- |
| 一致性 | 强一致（资源预留） | 最终一致（补偿） |
| 幂等键 | tcc_tx_id + account_id | saga_id + step |
| 隔离性 | Try 冻结可见 | 无隔离 |
| 补偿 | Cancel（解冻） | 反向事务 |
| 适用 | 资金扣减、库存 | 长流程编排 |

### 9.3 TCC 三大特殊防护

| 防护 | 描述 | 实现 |
| --- | --- | --- |
| **幂等** | Try/Confirm/Cancel 各自幂等 | tcc_record 表 + 状态机 |
| **空回滚** | Try 未执行但 Cancel 到达 | Cancel 前检查 Try 是否执行 |
| **悬挂** | Cancel 先于 Try 到达 | Try 前检查 Cancel 是否已到达 |

### 9.4 TCC 状态机

```
INIT → TRYING → TRIED → CONFIRMED
                ↓
              CANCELLED
```

- Try：从 INIT/TRYING 抢占，执行冻结
- Confirm：从 TRIED 流转，实际扣账
- Cancel：从 TRIED 流转，解冻；或从 INIT 流转（空回滚）

## 十、面试速记口诀

1. **幂等三要素**：唯一键稳定、状态可查、失败可重试
2. **四道防线**：客户端、幂等表、状态机、对账
3. **幂等表三字段**：key、status、response_snapshot
4. **补单四校验**：业务订单、退款、账务、渠道当前状态
5. **TCC 三防护**：幂等、空回滚、悬挂
6. **资金幂等铁律**：Redis 不可单独用、异步落库禁止、宁慢勿错

## 十一、与 Day01 的衔接

Day01 落地的状态机里，PROCESSING→SUCCESS 的并发竞争 = Day02 的状态机乐观锁 + 幂等表组合防护。

Day02 的"幂等键"是 Day03 对账的"对账键"基础——对账本质是用渠道流水号（channel_no）做跨系统的幂等校验。

Day02 的"补单流程"是 Day04 资金安全的核心组件——任何资金动作都必须经过幂等表+状态机+对账三重校验。

Day02 的"退款幂等"是 Day05 退款分账的地基——部分退款的累计金额校验、合并支付的关联表都建立在退款幂等键设计之上。
