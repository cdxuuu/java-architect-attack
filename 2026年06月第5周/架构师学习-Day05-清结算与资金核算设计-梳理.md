## 一、清结算在支付系统中的定位

### 1.1 核心一句话

> **清结算 = 把"业务事实"翻译成"资金动作"和"会计语言"，确保钱算得清、划得对、记得准、校得平**

幂等解决"同一笔不重复"，路由解决"走哪个渠道"，对账解决"账实是否相符"，**清结算解决"钱怎么算怎么划怎么记"**。四者构成支付系统的核心闭环。

### 1.2 清结算的四大目标

| 目标 | 描述 | 衡量指标 |
|---|---|---|
| 资金安全 | 钱不能丢、不能多、不能挪 | 资金差错率 < 0.01‰ |
| 合规 | 满足央行/外管/税务监管 | 监管报送准时率 100% |
| 财务正确 | 会计账实相符、月结平衡 | 月结平衡率 100% |
| 运营效率 | 自动化结算、自动核算 | 自动化率 > 95% |

四个目标互相关联：资金安全是底线，合规是硬约束，财务正确是审计要求，运营效率是商业竞争力。

### 1.3 清结算 ≠ 简单转账

生产级清结算必须解决：
- 多角色分账（商户/平台/渠道/银行）
- 多结算周期（T+0/T+1/T+N 混合）
- 多币种（本币/外币/汇兑损益）
- 备付金合规（央行集中交存）
- 业财一体（业务事件驱动会计）
- 差错处理（长款/短款/单边账）
- 大促峰值（异步化/分桶/合并/轧差）

## 二、清结算分层模型（核心抽象）

### 2.1 四层 + 一中台

```
支付成功 → [L1清算] → [L2结算] → [L3核算] → [L4对账] → 财务总账
              │           │          │          │
              ▼           ▼          ▼          ▼
           算账          划钱        记账        校账
              
              ◉ 清结算中台（配置化 + 规则化 + 多业务线复用）
```

| 层 | 职责 | 输入 | 输出 |
|---|---|---|---|
| L1 清算 | 计算应收/应付/分账/手续费 | 支付成功事件 | 清算明细 |
| L2 结算 | 按周期聚合、生成结算单、调拨备付金 | 清算明细 | 结算单 + 备付金流水 |
| L3 核算 | 按会计准则生成会计分录、过总账 | 清算/结算事件 | 会计分录 + 总账 |
| L4 对账 | 与渠道/银行核对，识别差错 | 结算单 + 渠道账单 | 差错单 + 平账指令 |

### 2.2 分层的三个核心收益

1. **职责正交**：清算只算、结算只划、核算只记、对账只校，单层故障不污染其他层
2. **可独立演进**：会计准则变化不影响清算逻辑
3. **可监管**：央行/审计/税务各有独立报表口径，互不污染

### 2.3 分层的常见反模式

- **清算结算不分**：算账和划钱耦合，划款失败导致重算
- **核算耦合业务**：业务系统直接生成会计分录，月结难平衡
- **对账依赖清算**：对账失败影响次日清算，连锁故障

## 三、清算层设计

### 3.1 清算模型核心三问

1. **应收多少**：从用户收到的钱
2. **应付多少**：要给商户/平台/渠道的钱
3. **分账规则**：平台抽佣、多方分润、手续费扣除

### 3.2 分账规则四类

| 类型 | 适用 | 示例 |
|---|---|---|
| 简单分账 | 自营电商 | 固定比例抽佣 |
| 阶梯分账 | SaaS 平台 | 按金额阶梯抽佣 |
| 多方分账 | 平台型业务 | 美团/滴滴多账户分润 |
| 动态分账 | 复杂生态 | 基于商户画像/合同动态计算 |

### 3.3 清算幂等与冲正

- **幂等键**：`(out_trade_no, clearing_type)` 唯一索引
- **状态机**：INIT → CALCULATED → SETTLED → REVERSED
- **冲正清算**：原 out_trade_no + clearing_type=REFUND，与原 NORMAL 形成冲正对
- **退款清算**：独立通道消费，避免影响正向清算

## 四、结算层设计

### 4.1 结算周期模型

| 周期 | 适用 | 时效 | 复杂度 |
|---|---|---|---|
| T+0 实时 | 提现/借贷放款 | 秒级 | 高（需垫资） |
| T+1 普通 | 电商/自营 | 次日 | 中 |
| T+N 延迟 | 跨境/风控冻结 | N 天 | 中 |
| D+0 当日 | 大客户/VIP | 当日 | 高（需垫资） |

### 4.2 结算单生成策略

- **按商户聚合**：每商户每天一张（T+1 默认）
- **按金额阈值**：满 1w 立即结算（D+0 自动出款）
- **按业务线聚合**：电商/借贷分开
- **按渠道聚合**：与渠道账单对账清晰

### 4.3 备付金管理（合规核心）

央行 259 号文要求：非银行支付机构备付金集中交存。

```
┌─────────────────────────────────┐
│  备付金账户（央行交存账户）       │
│  实收备付金（资产）              │
│  应付备付金（负债）              │
└────────────┬────────────────────┘
             │
   ┌─────────┴─────────┐
   ▼                   ▼
商户结算账户        银行备付金往来
```

**核心科目**：
- 实收备付金（资产）—— 用户支付的资金
- 应付备付金（负债）—— 待结算给商户的资金
- 备付金手续费收入（收入）—— 渠道手续费差额
- 备付金利息（收入）—— 央行交存利息

**三大风险**：
- 挪用风险：实收 > 应付 + 自有资金，超部分不能挪用
- 流动性风险：T+0 垫资需有垫资额度控制
- 合规风险：每日向央行报送余额，超限处罚

### 4.4 T+0 实时结算的垫资设计

T+0 本质 = **平台先垫资，渠道资金到账后回补**。

```
用户支付 (T+0)
  ├─ 清算：实收备付金 +100 / 应付商户 +100
  ├─ 结算：检查垫资额度 → 检查备付金 → 出款
  └─ T+1 渠道到账 → 备付金回补 → 垫资归位
```

**垫资风控三件套**：
- 单商户日垫资额度（按信用动态调整）
- 总垫资池上限（如备付金余额 80%）
- 垫资逾期监控（T+1 未到账告警）

## 五、核算层设计

### 5.1 业财一体化的核心

> **业务事件 → 会计分录自动生成**

自动化率应 > 95%，异常才人工。凭证模板化：每种业务类型有固定分录模板。

### 5.2 核心会计科目体系

支付机构典型科目（按央行支付机构会计核算办法）：

```
资产类：
  1001 实收备付金（央行账户）
  1002 银行存款（自有资金）
  1101 垫资款（T+0 垫资）
  1201 待结算款项

负债类：
  2001 应付备付金（待付商户）
  2101 应付账款（应付商户结算款）

收入类：
  3001 手续费收入（渠道差额）
  3002 利息收入（备付金利息）

费用类：
  4001 渠道成本（付给渠道）
```

### 5.3 典型会计分录

**用户支付 100 元，渠道手续费 0.6 元，平台佣金 0.6 元**：

```
借：实收备付金           100.00
    贷：应付备付金（商户A）       98.80
    贷：手续费收入（平台）         0.60
    贷：手续费收入（渠道差额）     0.60
```

**T+1 结算给商户**：

```
借：应付备付金（商户A）    98.80
    贷：银行存款                  98.80
```

**退款冲正**（红字）：

```
借：应付备付金（商户A）   -98.80
    贷：实收备付金                -98.80
    贷：手续费收入（平台）         -0.60
    贷：手续费收入（渠道差额）     -0.60
```

### 5.4 业财事件映射

| 业务事件 | 会计动作 |
|---|---|
| 支付成功 | 借实收备付金 / 贷应付备付金 + 贷手续费收入 |
| 退款成功 | 红字冲销原分录 + 借应付备付金 / 贷实收备付金 |
| T+1 结算 | 借应付备付金 / 贷银行存款 |
| 渠道手续费 | 借渠道成本 / 贷银行存款 |
| 备付金利息 | 借实收备付金 / 贷利息收入 |
| 短款挂账 | 借其他应收款 / 贷应付备付金 |
| 长款挂账 | 借实收备付金 / 贷其他应付款 |

## 六、对账层与差错处理

### 6.1 三层对账

```
业务对账（内部）        渠道对账（外部1）       资金对账（外部2）
   │                       │                      │
   ▼                       ▼                      ▼
订单↔支付              支付明细↔渠道账单       结算单↔银行回单
支付↔清算              退款明细↔渠道退款       备付金↔央行账单
清算↔结算              ─────────────────       ─────────────────
   │                       │                      │
   └───────────────────────┴──────────────────────┘
                           │
                           ▼
                       差错处理中心
                    ┌──────┼──────┐
                    ▼      ▼      ▼
                  长款   短款   单边账
```

### 6.2 差错处置与清结算联动

| 差错类型 | 含义 | 处置 |
|---|---|---|
| 长款 | 渠道多给 | 挂"其他应付款"，待商户认领或转营业外收入 |
| 短款 | 渠道少给 | 挂"其他应收款"，向渠道追索或商户补扣 |
| 单边账 | 一方有记录一方无 | 挂账待查，超 7 天未查清转营业外收支 |

**架构师水平线**：差错自动处置率 > 95%，规则引擎驱动，人工只处理真正异常。

## 七、状态机设计

```
支付成功
   │
   ▼
[清算-INIT] ──计算──> [清算-CALCULATED]
                          │
                          ▼
                    [结算-INIT] ──聚合──> [结算-EXECUTING] ──出款──> [结算-SUCCESS]
                                                                      │
                                                                      ▼
                                                                [核算-POSTED]
                                                                      │
                                                                      ▼
                                                                [对账-MATCHED]
                                                                      │
                                                                      ▼
                                                                  [完成]

异常分支：
  清算-CALCULATED ──冲正──> 清算-REVERSED（退款触发）
  结算-EXECUTING ──失败──> 结算-FAIL（重试或人工）
  对账-MATCHED ──差错──> 对账-EXCEPTION（差错处置）
```

## 九、跨境清结算专项

### 9.1 多币种清算模型

```java
class CrossBorderClearingDetail {
    String originalCurrency;     // 原币种 CNY
    Long originalAmount;
    String settleCurrency;       // 结算币种 USD
    Long settleAmount;
    BigDecimal exchangeRate;     // 锁定汇率
    String rateType;             // SPOT/FORWARD
    Date rateLockTime;
    Long exchangeGainLoss;       // 汇兑损益（退款时）
}
```

### 9.2 汇率锁定策略

- **支付时锁定**：整个链路用支付时汇率
- **退款汇兑损益**：
  - 锁定原汇率：商户承担汇兑损益（业界主流）
  - 重算汇率：平台承担汇兑损益

### 9.3 跨境结算周期

```
T+0  本地清算完成（CNY 侧）
T+1  购汇指令发送（CNY → USD）
T+2  境外银行收汇
T+3~7 境外商户到账
```

### 9.4 合规报送

- 外汇申报：单笔 > 5w USD 需向外汇局申报
- 海关报关：跨境电商推送报关单
- 反洗钱：单笔 > 1w USD 或当日累计 > 5w USD 触发 AML 审查

### 9.5 跨境独立科目

## 十、TCC 与清结算协同

### 10.1 事件契约

TCC Confirm 成功 → 发送事件 → 清结算消费：

```java
@EventListener
public void onTccConfirm(TccConfirmEvent event) {
    // 1. 幂等校验（xid + outTradeNo）
    if (clearingRepository.existsByXid(event.getXid())) return;
    
    // 2. TCC 状态二次确认（防 UNKNOWN）
    TccStatus status = tccClient.queryStatus(event.getXid());
    if (status != CONFIRMED) {
        log.warn("TCC not confirmed, skip clearing");
        return;
    }
    
    // 3. 触发清算
    clearingService.clear(event);
}
```

### 10.2 UNKNOWN 状态防御

TCC Confirm 可能因网络超时返回 UNKNOWN，但实际已成功：
- 清结算不能直接清算，需先查询 TCC 状态机
- TCC 框架定时扫描 UNKNOWN 事务，最终一致后补发事件
- 兜底：清结算消费时二次查询 TCC 状态

### 10.3 Cancel 防御性冲正

Confirm 后又收到 Cancel（理论不应发生，但需防御）：
- 已生成清算明细 → 触发冲正清算（clearing_type=REVERSAL）
- 通知商户系统回滚业务状态

### 10.4 蛋壳钱包当年设计不足（自评）

- TCC 与清结算同步调用 → 应改事件驱动解耦
- 未做 TCC 状态二次确认 → UNKNOWN 状态有清算风险
- 应补：事件驱动 + 状态二次确认 + UNKNOWN 兜底

## 十一、大促清结算峰值四板斧

### 11.1 异步化

- 清算异步：支付成功事件 MQ 异步消费
- 结算异步：生成结算单 → 异步出款 → 回调更新
- 不阻塞支付主链路

### 11.2 分桶

- 清算分桶：按 out_trade_no hash 分 100 桶，并行处理
- 结算分批：按商户 hash 分批，避免单表锁
- 消费者独立 Group，避免互相影响

### 11.3 合并

- 小额结算单合并：< 100 元合并到次日
- 银行接口限流：令牌桶控制 TPS（如 100 TPS）
- 减少银行调用次数

### 11.4 轧差

- 同商户当日退款与正向轧差
- 例：售出 10w + 退款 8k → 实际结算 9.2w（一次划款）
- 减少资金划拨次数

### 11.5 跨日交易处理

- 切账时点：以"支付完成时间"为准，23:59:59 切账
- 跨日归 T+1：23:50 下单 + 次日 0:10 支付 → 归次日 T+1
- 切账冻结期：23:55~0:05，期间交易标记"待切账"，0:05 后批量补切

## 十五、一句话总结

> 清结算是支付系统的"账房"：清算算账、结算划钱、核算记账、对账校账。
> 备付金是合规核心，业财一体化是核算目标，T+0 是垫资艺术，跨境是多币种挑战。
> 架构师水平的清结算 = 自动化率 > 95% + 资金差错率 < 0.01‰ + 月结平衡率 100%。
