## 一、支付系统在高可用架构中的定位

### 1.1 核心一句话
> **支付系统 = 资金链路的"中枢神经"，用强一致 + 最终一致 + 全链路对账换取资金安全**
>
> 与电商、IM 不同，支付系统的容错边界是"一分钱不能错"——架构师要的是**资损率为0**，而不是"基本可用"。

### 1.2 支付系统的三大特征

| 特征 | 含义 | 架构影响 |
| --- | --- | --- |
| 强资金安全 | 任何一分钱差错都是 P0 事故 | 双重校验、幂等、对账兜底 |
| 强一致 + 最终一致混合 | 资金扣减强一致，回调通知最终一致 | 同步链路 + 异步链路分层 |
| 强监管合规 | 牌照、PCI-DSS、等保、反洗钱 | 数据隔离、审计日志、敏感信息加密 |

### 1.3 支付系统的多层架构

```
业务线（电商/保险/IM问诊/e卡/秒杀）
   ↓ 业务订单
收银台（C端交互、风控前置、Token换取）
   ↓ 支付请求
支付网关（统一API、签名验签、幂等、限流、路由分发）
   ↓
┌────────┬────────┬────────┬────────┐
订单中心  风控中心  路由引擎  通知中心
   ↓
渠道适配（微信/支付宝/银联/e卡）
   ↓ 渠道扣款 + 异步回调
资金层（账户/账务/清结算/对账）
```

## 二、核心模块边界：架构师必须能画出来

### 2.1 九大核心模块职责矩阵

| 模块 | 职责 | 不做什么 |
| --- | --- | --- |
| 收银台 | 支付方式展示、风控前置、Token换取、SDK拉起 | 不调渠道、不做清结算 |
| 支付网关 | 统一API、签名验签、幂等、限流熔断、路由分发 | 不持有订单状态机 |
| 订单中心 | 支付订单CRUD、状态机、订单查询 | 不调渠道、不做资金核算 |
| 渠道路由 | 策略匹配（成本/成功率/限额/卡BIN/灰度）、降级切换 | 不持有订单状态 |
| 渠道适配 | 屏蔽渠道差异、对接微信/支付宝/银联 | 不做业务路由决策 |
| 风控中心 | 限额、黑名单、设备指纹、防盗刷、合规拦截 | 不直接拒付，给拦截信号 |
| 通知中心 | 业务回调、渠道回调分发、MQ重试补偿 | 不修改订单状态，只触发事件 |
| 账户/账务 | 内部账户余额、复式记账、冻结/扣减 | 不与渠道直接通信 |
| 清结算 | T+1结算、对账文件解析、资金划拨指令 | 不负责实时扣款 |
| 对账 | 长短款识别、差异挂账、差异处理 | 不修改业务订单 |

### 2.2 同步链路 vs 异步链路

```
同步链路（用户在等）：
  收银台 → 网关 → 订单中心(INIT) → 风控 → 路由 → 渠道下单
        → 返回prepay_id → 拉起SDK → 用户输密码 → 渠道扣款(异步)
        → 收银台轮询2-3s → 未拿到返回"处理中"

异步链路（用户不在等）：
  渠道回调 → 网关验签 → 通知中心 → MQ
        → 订单中心(状态流转) → MQ → 业务系统(更新业务订单)
        → MQ → 账务(记账)
        → MQ → 风控(事后审计)

T+1批量：
  渠道对账文件 → 对账系统 → 比对本地订单/账务 → 差异挂账 → 处理
```

**异步的三大理由**：
1. 回调时机不可控（秒级到小时级），同步阻塞会拖垮网关线程池
2. 业务系统可能慢，MQ 隔离避免拖垮回调链路
3. 清结算/对账天然批量，与实时链路隔离避免影响支付

## 三、渠道路由引擎：可扩展是核心

### 3.1 路由策略八维度

| 维度 | 示例 |
| --- | --- |
| 成本优先 | 0.38% 费率渠道优先于 0.6% |
| 成功率优先 | 近5分钟成功率 < 95% 自动降级 |
| 限额过滤 | 单笔5万、单日20万 |
| 卡BIN路由 | 信用卡按卡BIN路由到对应银行通道 |
| 业务类型 | 保险→保险通道，e卡→e卡通道 |
| 灰度切换 | 新渠道先1%灰度，逐步放量 |
| 降级熔断 | 错误率超阈值自动熔断切备 |
| 地区/币种 | 跨境按币种路由到国际通道 |

### 3.2 可扩展引擎设计三原则

```java
// 1. 策略SPI化：新增策略加类即可，不改路由引擎
public interface RouteStrategy {
    RouteResult match(RouteContext ctx);
    int order();
}

// 2. 规则数据化：限额/卡BIN/业务类型走配置中心
RouteRule { channelId, bizType, amountMin, amountMax, 
            cardBinRegex, region, weight, enabled }

// 3. 健康度实时：基于滑动窗口的成功率/P99/熔断状态过滤
```

**关键设计点**：
- 路由决策全留痕（候选集 + 命中规则 + 最终渠道），便于回放排查
- 规则热生效（配置中心推送，不重启）
- 灰度按订单号 hash 切分，避免同一用户反复切渠道

## 四、业务订单 vs 支付订单：必须拆分的理由

### 4.1 关系模型

```
业务订单(SO202606290001)
   ├─ 支付订单1(微信, SUCCESS)        ← 当前有效
   ├─ 支付订单2(支付宝, CLOSED)       ← 用户切渠道
   └─ 支付订单3(微信退款, REFUNDED)   ← 退款记录
```

### 4.2 拆分五大理由

| # | 理由 |
| --- | --- |
| 1 | 职责分离：业务订单关心"买了什么"，支付订单关心"通过哪个渠道扣了多少" |
| 2 | 多支付方式：一笔业务订单对应多笔支付订单，只有最后一笔成功 |
| 3 | 退款/分账：支付订单是退款的最小单元，可多次部分退款 |
| 4 | 多业务线复用：支付能力是公共能力，不绑定电商业务 |
| 5 | 数据生命周期隔离：业务订单7年留存，支付订单可独立归档 |

### 4.3 一对多约束

- 同一时刻只允许一笔支付订单处于 PROCESSING
- 新发起支付需先关闭旧单（状态机保证）
- 业务订单状态由支付订单事件驱动

## 五、支付订单状态机：架构师必须能默写

### 5.1 完整状态流转图

```
                  ┌─────────┐
                  │  INIT   │ 待支付
                  └────┬────┘
                       │ 渠道下单成功
                       ▼
                  ┌─────────────┐
        ┌────────│ PROCESSING  │────────┐
        │        │  (处理中)    │        │
        │        └─────────────┘        │
   超时关单     渠道回调成功         渠道回调失败
        │              │                 │
        ▼              ▼                 ▼
   ┌────────┐   ┌─────────┐        ┌─────────┐
   │ CLOSED │   │ SUCCESS │        │ FAILED  │
   └────────┘   └────┬────┘        └─────────┘
                     │ 退款发起
                     ▼
                ┌────────────┐
                │ REFUNDING  │
                └─────┬──────┘
                      │ 退款回调成功
                      ▼
                ┌────────────┐
                │ REFUNDED   │
                └────────────┘
```

### 5.2 三处危险流转

**① PROCESSING → SUCCESS vs FAILED 并发竞争**

四重防护：
```sql
-- 1. 乐观锁 + 状态前置校验
UPDATE pay_order SET status=SUCCESS, version=version+1 
WHERE id=? AND status=PROCESSING AND version=?;
-- 影响行数=0 则放弃
```
- 乐观锁（version）
- 状态机白名单（禁止逆向与跳跃）
- 幂等键唯一索引（渠道流水号）
- 分布式锁（pay_order_id 为 key，兜底）

**② SUCCESS → REFUNDING 重复退款**
- 退款单号唯一索引
- 累计退款金额 ≤ 原支付金额
- 退款前校验订单为 SUCCESS 且未冻结

**③ CLOSED 状态收到 SUCCESS 回调**
- 关键设计：**不置 SUCCESS，触发自动退款**
- 关单前必须调渠道关单接口确认（渠道关单可能失败因用户已付款）
- 避免业务方误发货

### 5.3 回调晚于查询的最终一致

```
主动查询任务（指数退避）：
  PROCESSING 状态订单
    ├─ 30s 后查渠道
    ├─ 1min 后查
    ├─ 5min 后查
    ├─ 10min 后查
    └─ 30min 后查
  任何一次拿到 SUCCESS → 置 SUCCESS

T+1 对账兜底：
  渠道已成功 + 本地PROCESSING → 自动补单
```

幂等保证：回调与查询都通过"渠道流水号 + 状态机前置校验"保证只生效一次。

## 六、付款成功但订单显示"处理中"的根因排查

| # | 根因 | 排查 | 补偿 |
| --- | --- | --- | --- |
| 1 | 渠道回调未送达 / 验签失败 | 网关日志、回调失败队列 | 主动查询任务 + 死信告警 |
| 2 | MQ 消费失败 / 消费者宕机 | MQ 堆积、消费者重启日志 | MQ 集群高可用 + 死信处理 |
| 3 | 乐观锁冲突 / 状态机拒绝 | 订单中心错误日志 | 冲突重试 + T+1 对账补单 |
| 4 | 业务系统更新失败 | 业务系统日志 | MQ 重试 + 查询时反查支付订单 |
| 5 | 渠道超时但实际成功 | 渠道对账文件 | 主动查询 + T+1 对账 |
| 6 | 风控拦截未通知订单 | 风控日志、通知链路 | 风控结果落库 + 通知中心订阅 |
| 7 | 分库分表路由错误 / 主从延迟 | 数据分布、从库延迟 | 强制读主 + 分片键校验 |

**系统级补偿三层**：
- 实时补偿：主动查询任务（指数退避）
- 批量补偿：T+1 对账（渠道-本地-账务三流比对）
- 应急补偿：运维工具一键查询补单

## 七、单体 vs 微服务 vs 云原生：决策矩阵

### 7.1 按业务规模选型

| 规模 | 日订单 | 推荐 | 关键理由 |
| --- | --- | --- | --- |
| 中小 | < 10w | 单体（模块化分包） | 峰值 QPS < 50，微服务成本远超收益 |
| 中大 | 10w~100w | 微服务化 | 故障隔离 + 独立扩容 |
| 超大 | > 100w | 微服务 + 单元化 | 单库瓶颈 + 容灾 + 延迟 |

### 7.2 单元化关键设计

```
路由层（按 cuid 分流）
   ├─ 单元A：cuid % 4 ∈ {0,1}，完整闭环
   ├─ 单元B：cuid % 4 ∈ {2,3}，完整闭环
   └─ 中心单元：跨单元协调（对账、风控全局）
```

**四大原则**：
- 流量封闭：同一用户的支付/查询/退款在同一单元
- 数据封闭：每单元只持有自己的 DB 分片
- 全局唯一：分布式 ID 跨单元不冲突
- 跨单元极少：仅对账、风控聚合走中心

### 7.3 同城双活 vs 异地多活对支付的特殊要求

| 要求 | 同城双活 | 异地多活 |
| --- | --- | --- |
| 数据同步 | 强一致（RPO=0） | 最终一致（异步） |
| 切换 | 主机房故障自动切换 | 单元故障切流 |
| 渠道回调 | 路由到原始下单单元 | 同左 + 跨单元读处理 |
| 资金一致 | 双写幂等 + 对账 | 单元内强一致，跨单元走清结算 |
| 合规 | 同城满足 | 需评估跨境数据合规 |

**支付系统多活三大难点**：
1. 渠道回调必须定位原始单元（订单号→单元路由表）
2. 账户余额强一致资源，跨单元不能直接扣减
3. 退款必须路由到原支付订单所在单元

### 7.4 云原生 vs 自研九维决策

| 维度 | 云原生 | 自研 |
| --- | --- | --- |
| 业务规模 | 中小、跨境 | 大型、多业务线 |
| 资金成本 | 渠道费 + 平台服务费 | 渠道费 + 人力 |
| 人力成本 | 低（1-2周接入） | 高（持续团队） |
| 定制化 | 受限 | 完全可控 |
| 合规 | 平台兜底 | 自负其责 |
| 资金安全 | 平台担保 | 自建风控对账 |
| 数据归属 | 部分在平台 | 全部自有 |
| 跨境能力 | 强 | 弱 |
| 业务连续性 | 平台故障影响所有用户 | 自主降级 |

**常见混合方案**：自研支付核心 + 接入聚合支付作为渠道之一。

## 八、面试高频提问速答

**Q1：支付系统为什么必须拆分业务订单和支付订单？**
A：职责分离、多支付方式、退款单元、复用能力、数据生命周期。一笔业务订单可对应多笔支付订单（切渠道、退款），但同一时刻只有一笔 PROCESSING。

**Q2：PROCESSING → SUCCESS 和 FAILED 并发竞争怎么防？**
A：四重防护——乐观锁（version）+ 状态机白名单 + 渠道流水号唯一索引 + 分布式锁兜底。核心是数据库层用条件 UPDATE 保证只生效一次。

**Q3：渠道回调晚于用户主动查询怎么办？**
A：回调与查询都通过"渠道流水号 + 状态前置校验"幂等。先到的执行流转，后到的判断已是目标状态直接返回成功。兜底靠主动查询任务（指数退避）+ T+1 对账。

**Q4：CLOSED 状态收到渠道 SUCCESS 回调怎么处理？**
A：不置 SUCCESS，触发自动退款。因为本地已关单，业务方可能已发关单通知，置 SUCCESS 会导致误发货。

**Q5：支付系统为什么必须做单元化？**
A：单库扛不住千万级订单、单机房故障不可接受、跨机房延迟影响用户体验。单元化让每个单元独立承接流量、独立数据分片，故障隔离 + 横向扩展。

**Q6：什么情况下选云原生支付？**
A：中小业务、跨境业务、无支付研发团队、非核心支付场景。蛋壳钱包（持牌）必须自研，e卡资金路由（业务复杂）必须自研。

## 九、能力差距与补足方向

| 维度 | 现状 | 补足 |
| --- | --- | --- |
| 单元化部署 | 概念清晰，资金一致性细节不足 | 深读蚂蚁 LDC、京东支付单元化公开资料 |
| 金融合规 | 知道皮毛，缺乏实操 | PCI-DSS、等保三级、支付牌照、反洗钱 |
| 工程实践 | 能讲方案，代码级掌握不够 | 实践简化版支付网关（状态机+幂等+对账） |
| 跨境支付 | 未覆盖 | 外汇、税务、多币种结算链路 |
| 业务场景 | 蛋壳/e卡清晰，保险/问诊模糊 | 保险兼业代理、医疗预付费监管差异 |
