## 背景

清结算是支付系统的"账房"，决定了钱到底怎么算、怎么划、怎么记。如果说路由是"决定走哪条路"，那清结算就是"算账、划钱、记账"的最后一公里。这一块直接关系到**资金安全、合规、财务正确性、运营效率**四个硬指标，是架构师必须能Hold住的核心模块。

清结算的复杂度来源：
- **多角色**：商户、平台、渠道、央行、税务、备付金账户
- **多周期**：T+0 实时结算、T+1 普通结算、T+N 跨境结算
- **多状态**：待清算→清算中→待结算→结算中→已结算→已对账→已核算
- **多差错**：长款、短款、单边账、跨日交易、退款冲正
- **强合规**：央行 259 号文、备付金集中交存、反洗钱、税务发票

接入 20+ 渠道、日均 2000 万笔、多业务线（电商/借贷/充值/提现）、T+0 与 T+1 混合、跨境结算——本题从架构设计与场景实战两个维度展开，并结合蛋壳钱包 TCC 与国美e卡资金路由经验做差距自评。

## 题目一（架构设计题）：清结算整体架构与资金核算设计

### 作答区

#### 1. 清结算分层模型

**核心思想**：把清结算拆成"四层 + 一中台"，每层职责清晰、数据契约明确、可独立演进。清算是"算账"，结算是"划钱"，核算是"记账"，对账是"校账"，中台是"能力复用"。

```
支付成功事件
  │
  ▼
[1] 清算层（Clearing）  —— 解决"谁欠谁多少"
  │ 输出：清算明细（应收/应付/分账）
  ▼
[2] 结算层（Settlement）—— 解决"什么时候划、怎么划"
  │ 输出：结算指令 + 备付金流水
  ▼
[3] 核算层（Accounting）—— 解决"会计怎么记"
  │ 输出：会计分录 + 总账
  ▼
[4] 对账层（Reconciliation）—— 解决"账实是否相符"
  │ 输出：差错单 + 平账指令
  ▼
财务总账 / 监管报表

  ◉ 清结算中台（Clearing-Settlement Mid-platform）
     └─ 配置化、规则化、可复用，支撑多业务线
```

**各层职责边界**：

| 层 | 职责 | 不做什么 | 输入 | 输出 |
|---|---|---|---|---|
| L1 清算 | 计算应收/应付/分账/手续费，生成清算明细 | 不划钱、不记账 | 支付成功事件 + 渠道结算配置 | 清算明细 + 分账规则 |
| L2 结算 | 按结算周期聚合、生成结算单、调拨备付金 | 不计算金额、不记会计账 | 清算明细 + 结算周期配置 | 结算单 + 备付金流水 |
| L3 核算 | 按会计准则生成会计分录、过总账 | 不计算金额、不划钱 | 清算明细 + 结算单 | 会计分录 + 总账 |
| L4 对账 | 与渠道账单、银行流水核对，识别差错 | 不记账、不划钱 | 结算单 + 渠道账单 + 银行流水 | 差错单 + 平账指令 |

**为什么这样分层**：
- **职责正交**：清算只算、结算只划、核算只记、对账只校，单层故障不污染其他层
- **可独立演进**：核算规则变化（如新会计准则）不影响清算逻辑
- **可独立测试**：每层有独立数据契约，可单测+集成测
- **可复用**：清结算中台可支撑电商、借贷、充值、提现多业务线
- **可监管**：央行、审计、税务各有独立报表口径，互不污染

#### 2. 清算层设计

**(1) 清算模型核心抽象**

清算层要回答三个问题：
- **应收多少**：从用户收到的钱
- **应付多少**：要给商户/平台/渠道的钱
- **分账规则**：平台抽佣、多方分润、手续费扣除

```java
// 清算明细（Clearing Detail）—— 一笔支付清算后产生的所有资金条目
public class ClearingDetail {
    private String clearingNo;          // 清算单号
    private String outTradeNo;          // 原支付单号
    private String bizLine;             // 业务线
    private String channelId;           // 渠道
    private Long totalAmount;           // 支付总金额（分）
    
    private List<ClearingItem> items;   // 清算条目（多方）
    private Long settlementAmount;      // 实际结算金额 = sum(items.amount)
    private Long feeAmount;             // 手续费总额
    private String clearingType;        // NORMAL/REFUND/CHARGEBACK/SPLIT
    private Date clearingTime;
    private String clearingStatus;      // INIT/CALCULATED/SETTLED/REVERSED
}

// 清算条目（Clearing Item）—— 每一方的应收/应付
public class ClearingItem {
    private String accountId;           // 账户ID（商户/平台/渠道）
    private String accountType;         // MERCHANT/PLATFORM/CHANNEL/BANK
    private Long amount;                // 金额（分）
    private String direction;           // DEBIT/CREDIT（借/贷）
    private String feeType;             // PRINCIPAL/FEE/COMMISSION/TAX
    private String settleAccountId;     // 实际结算账户
}
```

**(2) 分账规则引擎**

清算的核心是"分账规则"。一笔支付到账后，按业务规则拆成多个清算条目：

```yaml
# 分账规则配置示例（电商业务）
split_rule:
  rule_id: SPLIT_ECOMMERCE_001
  biz_line: ECOMMERCE
  conditions:
    - amount_range: [100, 1000000]   # 1元 ~ 1万元
    - merchant_type: STANDARD
  actions:
    # 1. 商户应收（本金 - 平台佣金 - 渠道手续费）
    - account: ${merchant.settle_account}
      amount: ${amount - platform_fee - channel_fee}
      fee_type: PRINCIPAL
      direction: CREDIT
    
    # 2. 平台佣金（按费率）
    - account: PLATFORM_REVENUE
      amount: ${amount * 0.006}      # 千六佣金
      fee_type: COMMISSION
      direction: CREDIT
    
    # 3. 渠道手续费（按渠道费率）
    - account: CHANNEL_FEE_INCOME
      amount: ${amount * channel_rate}
      fee_type: FEE
      direction: CREDIT
    
    # 4. 备付金账户（用户实际付出，暂存备付金）
    - account: CLIENT_FUNDS
      amount: ${amount}
      fee_type: PRINCIPAL
      direction: DEBIT
```

**分账规则选型**：
- **简单分账**：固定比例（适合自营电商）
- **阶梯分账**：金额阶梯（适合 SaaS 平台）
- **多方分账**：多账户分润（适合平台型业务，如美团、滴滴）
- **动态分账**：基于商户画像/合同（适合复杂生态）

**(3) 清算幂等与冲正**

```sql
-- 清算明细表
CREATE TABLE clearing_detail (
  clearing_no      VARCHAR(32) PRIMARY KEY,
  out_trade_no     VARCHAR(32) NOT NULL,
  biz_line         VARCHAR(32),
  channel_id       VARCHAR(32),
  total_amount     BIGINT,
  settlement_amount BIGINT,
  fee_amount       BIGINT,
  clearing_type    VARCHAR(16),    -- NORMAL/REFUND/CHARGEBACK
  clearing_status  VARCHAR(16),    -- INIT/CALCULATED/SETTLED/REVERSED
  reverse_ref      VARCHAR(32),    -- 冲正关联的原清算单
  version          INT,            -- 乐观锁
  create_time      DATETIME,
  settle_time      DATETIME,
  UNIQUE KEY uk_trade_type (out_trade_no, clearing_type)  -- 幂等保证
);
```

**幂等设计**：
- `(out_trade_no, clearing_type)` 唯一索引 → 同一笔支付只产生一条清算明细
- 状态机驱动：INIT → CALCULATED → SETTLED → REVERSED（冲正）
- 退款清算：原 out_trade_no + clearing_type=REFUND，与原 NORMAL 清算形成冲正对

#### 3. 结算层设计

**(1) 结算周期模型**

| 结算周期 | 适用场景 | 资金时效 | 复杂度 |
|---|---|---|---|
| T+0 实时结算 | 提现、借贷放款 | 秒级 | 高（需备付金垫资） |
| T+1 普通结算 | 电商、自营 | 次日 | 中 |
| T+N 延迟结算 | 跨境、风控冻结 | N 天 | 中 |
| D+0 当日结算 | 大客户、VIP | 当日 | 高（需垫资） |

**(2) 结算单生成逻辑**

结算层把"清算明细"按"商户 + 结算周期 + 结算方式"聚合成"结算单"：

```java
public class SettlementOrder {
    private String settlementNo;       // 结算单号
    private String merchantId;         // 商户号
    private String settleAccount;      // 结算账户
    private Date settleCycle;          // 结算周期 T+1
    private Long totalAmount;          // 结算总金额
    private Long feeAmount;            // 手续费扣除
    private Long actualAmount;         // 实际到账金额
    
    private List<String> clearingNos;  // 包含的清算明细
    private String settleStatus;       // INIT/EXECUTING/SUCCESS/FAIL/PARTIAL
    private String bankTraceNo;        // 银行流水号
    private Date executeTime;
    private Date finishTime;
}
```

**结算单生成策略**：
- **按商户聚合**：每个商户每天生成一张结算单（T+1）
- **按金额阈值**：满 1 万元立即结算（D+0 自动出款）
- **按业务线聚合**：电商/借贷分开结算，便于核算
- **按渠道聚合**：渠道账单对账时按渠道拉取，对账清晰

**(3) 备付金管理**

央行 259 号文要求非银行支付机构备付金集中交存，这是清结算设计的硬约束：

```text
┌─────────────────────────────────────────┐
│     备付金账户（央行交存账户）           │
│   ┌────────────┬────────────┐           │
│   │  实收备付金  │  应付备付金  │          │
│   └────────────┴────────────┘           │
└─────────────┬───────────────────────────┘
              │
   ┌──────────┴──────────┐
   │                     │
┌──▼──────────┐    ┌──────▼──────┐
│ 商户结算账户  │    │  银行备付金  │
│  (待结算)    │    │  (往来账户)  │
└──────────────┘    └─────────────┘
```

**备付金核算核心科目**：
- 实收备付金（资产）—— 用户支付的资金
- 应付备付金（负债）—— 待结算给商户的资金
- 备付金手续费收入（收入）—— 渠道手续费差额
- 备付金利息（收入）—— 央行交存利息

**备付金风险点**：
- 挪用风险：实收 > 应付 + 自有资金，超出部分不能挪用
- 流动性风险：T+0 结算需备付金垫资，需有"垫资额度"控制
- 合规风险：每日向央行报送备付金余额，超限处罚

**(4) T+0 实时结算的垫资设计**

T+0 结算本质是"平台先垫资，渠道资金到账后回补"：

```text
用户支付成功 (T+0)
  │
  ├─→ 清算层：生成清算明细（实收备付金 +100，应付商户 +100）
  │
  ├─→ 结算层：触发 T+0 结算
  │     ├─ 检查商户垫资额度（如单日上限 100w）
  │     ├─ 检查备付金账户余额（避免挪用）
  │     └─ 调用银行出款接口，划款到商户结算卡
  │
  └─→ 渠道资金到账 (T+1)
        └─ 备付金账户回补，垫资归位
```

**垫资风控**：
- 单商户日垫资额度（如 100w，按商户信用动态调整）
- 总垫资池上限（如备付金账户余额的 80%）
- 垫资逾期监控（T+1 渠道未到账则触发告警）

#### 4. 核算层设计

**(1) 会计分录模型**

核算层把清算/结算业务翻译成会计语言，生成会计分录：

```java
public class AccountingEntry {
    private String entryId;            // 分录ID
    private String voucherNo;          // 凭证号
    private String accountCode;        // 科目代码
    private String accountName;        // 科目名称
    private Long amount;               // 金额
    private String direction;          // DEBIT/CREDIT
    private String bizType;            // 业务类型
    private String bizRefNo;           // 业务参考号
    private Date accountingTime;
    private String status;             // DRAFT/POSTED/REVERSED
}
```

**(2) 核心会计科目设计**

支付机构典型科目体系（按央行支付机构会计核算办法）：

```text
资产类：
  1001 实收备付金          —— 央行交存的备付金
  1002 银行存款            —— 自有资金银行存款
  1003 应收账款            —— 应收渠道手续费
  1101 垫资款              —— T+0 垫资
  1201 待结算款项          —— 待结算资金

负债类：
  2001 应付备付金          —— 应付商户的备付金
  2002 预收备付金          —— 已收未清算
  2101 应付账款            —— 应付商户结算款
  2201 预收手续费          —— 预收渠道手续费

收入类：
  3001 手续费收入          —— 渠道手续费差额
  3002 利息收入            —— 备付金利息
  3003 服务费收入          —— 平台服务费

费用类：
  4001 渠道成本            —— 付给渠道的手续费
  4002 业务成本            —— 业务支出
```

**(3) 会计分录生成规则**

一笔支付成功的典型分录：

```text
【用户支付 100 元，渠道手续费 0.6 元，平台佣金 0.6 元】

借：实收备付金（央行账户）        100.00
    贷：应付备付金（商户A）              98.80
    贷：手续费收入（平台）                0.60
    贷：手续费收入（渠道差额）            0.60

【T+1 结算给商户】
借：应付备付金（商户A）           98.80
    贷：银行存款（商户结算卡）            98.80

【渠道手续费结算（向渠道支付）】
借：渠道成本                       0.60
    贷：银行存款                          0.60
```

**核算自动化率**：
- 自动过账率应 > 95%（异常才人工）
- 凭证模板化：每种业务类型有固定的分录模板
- 月末结账：自动生成试算平衡表、利润表、资产负债表

**(4) 业财一体化**

核算层要做到"业财一体"，即业务事件直接驱动会计核算：

```text
业务事件               会计动作
────────────────────────────────────
支付成功         →  借实收备付金 / 贷应付备付金 + 贷手续费收入
退款成功         →  红字冲销原分录 + 借应付备付金 / 贷实收备付金
T+1 结算         →  借应付备付金 / 贷银行存款
渠道手续费       →  借渠道成本 / 贷银行存款
备付金利息       →  借实收备付金 / 贷利息收入
差错挂账         →  借其他应收款 / 贷应付备付金（短款）
                  借实收备付金 / 贷其他应付款（长款）
```

#### 5. 对账层与差错处理（承接 Day03）

清结算的对账比 Day03 的支付对账更进一步，覆盖三层：
- **业务对账**：支付系统内部数据一致性（订单/支付/清算/结算）
- **渠道对账**：与渠道方账单核对（Day03 已详细）
- **资金对账**：与银行流水核对（结算单 vs 银行回单）

```text
业务对账（内部）        渠道对账（外部1）       资金对账（外部2）
   │                       │                      │
   ▼                       ▼                      ▼
订单 ↔ 支付           支付明细 ↔ 渠道账单      结算单 ↔ 银行回单
支付 ↔ 清算           退款明细 ↔ 渠道退款      备付金流水 ↔ 央行账单
清算 ↔ 结算           ────────────────────      ────────────────────
   │                       │                      │
   └───────────────────────┴──────────────────────┘
                           │
                           ▼
                       差错处理中心
                           │
                    ┌──────┼──────┐
                    ▼      ▼      ▼
                  长款   短款   单边账
```

**差错处置与清结算联动**：
- **长款**（渠道多给）：挂"其他应付款"，待商户认领或转入营业外收入
- **短款**（渠道少给）：挂"其他应收款"，向渠道追索或商户补扣
- **单边账**：一方有记录一方无，先挂账待查，超 7 天未查清转营业外收支

#### 6. 状态机设计

```text
支付成功
   │
   ▼
[清算-INIT] ──计算──> [清算-CALCULATED]
                          │
                          ▼
                    [结算-INIT] ──聚合──> [结算-EXECUTING] ──出款──> [结算-SUCCESS]
                                                                      │
                                                                      ▼
                                                                [核算-POSTED]
                                                                      │
                                                                      ▼
                                                                [对账-MATCHED]
                                                                      │
                                                                      ▼
                                                                  [完成]

异常分支：
  清算-CALCULATED ──冲正──> 清算-REVERSED（退款触发）
  结算-EXECUTING ──失败──> 结算-FAIL（重试或人工）
  对账-MATCHED ──差错──> 对账-EXCEPTION（差错处置）
```

#### 7. 清结算中台能力抽象

清结算中台的核心是"配置化 + 规则化"，支撑多业务线快速接入：

```text
┌────────────────────────────────────────────┐
│           清结算中台                        │
│  ┌──────────────────────────────────────┐  │
│  │ 业务接入层（适配器）                   │  │
│  │  电商 / 借贷 / 充值 / 提现 / 跨境     │  │
│  └──────────────────────────────────────┘  │
│  ┌──────────────────────────────────────┐  │
│  │ 能力层（核心引擎）                    │  │
│  │  清算引擎 / 结算引擎 / 核算引擎 / 对账│  │
│  └──────────────────────────────────────┘  │
│  ┌──────────────────────────────────────┐  │
│  │ 配置层（规则与参数）                   │  │
│  │  分账规则 / 结算周期 / 科目映射 / 垫资│  │
│  └──────────────────────────────────────┘  │
│  ┌──────────────────────────────────────┐  │
│  │ 账务层（账户体系）                    │  │
│  │  备付金 / 商户结算 / 平台收入 / 银行  │  │
│  └──────────────────────────────────────┘  │
└────────────────────────────────────────────┘
```

**中台能力清单**：
- 清算规则配置化（运营可配，无需开发）
- 结算周期参数化（T+0/T+1/T+N 可选）
- 会计科目映射可配（业务事件 → 科目组合）
- 多业务线隔离（租户级配置）
- 灰度发布（新规则按商户灰度）

## 题目二（场景实战题）：大促清结算峰值与跨境结算

### 场景一：双11 大促清结算峰值处理

**业务背景**：
```text
双11 当日 5000w 笔支付，平时日均 2000w。
- 0:00~1:00 集中爆发 1000w 笔，是平时的 50 倍
- T+1 结算批次平时 5w 张，双11 当日 200w 张
- 跨日交易：23:50 下单，次日 0:10 支付，归哪天？
- 退款高峰：双11 后 7 天退款率 8%（平时 2%）
- 财务月结压力：12 月 1 日要出 11 月总账，时间紧

约束：
- 清算不能丢（资金安全底线）
- 结算不能延迟（合规要求 T+1 必出）
- 核算不能错（财务正确性）
- 跨日交易归 T+1 还是 T+2 必须明确
```

**架构师方案**：

**1. 清算峰值削峰**：
- 清算异步化：支付成功事件 MQ 异步消费，不阻塞支付主链路
- 清算分桶：按 out_trade_no hash 分 100 桶，并行处理
- 清算预热：大促前预热商户/渠道配置，避免 DB 热点
- 消费者分组：清算消费者独立 Group，避免被其他业务影响

**2. 结算批次优化**：
- 结算单分批生成：按商户 hash 分批，避免单表锁
- 结算异步化：生成结算单 → 异步执行出款 → 回调更新状态
- 银行接口限流：银行接口有 TPS 上限（如 100 TPS），用令牌桶限流
- 结算单合并：小额结算单合并（如 < 100 元合并到次日）

**3. 跨日交易处理**：
- 切账时点：以"支付完成时间"为准，统一用 23:59:59 切账
- 跨日交易归 T+1：23:50 下单 + 次日 0:10 支付 → 归次日 T+1（即 T+2 结算）
- 切账前冻结：23:55~0:05 进入"切账冻结期"，期间交易标记"待切账"，0:05 后批量补切

**4. 退款高峰处理**：
- 退款清算独立通道：退款清算消费者独立分组，避免影响正向清算
- 退款冲正分录：红字冲销原分录，保留审计痕迹
- 退款轧差：同商户当日退款与正向轧差，减少实际划款
  - 例：当日售出 10w，退款 8k → 实际结算 9.2w（一次划款）

**5. 月结优化**：
- 月结前预热：每月 1 日 0 点自动跑月结脚本，分区并行
- 月结分科目：资产/负债/收入/费用分科目并行结账
- 月结异常兜底：自动重试 3 次，失败转人工

**关键指标**：
- 清算延迟 P99 < 5s（支付成功到清算完成）
- 结算单生成延迟 P99 < 30min（切账后到结算单生成）
- 月结时长 < 2h（5000w 笔数据）
- 跨日交易识别准确率 100%

### 场景二：跨境支付清结算

**业务背景**：
```text
平台支持跨境支付，用户用人民币买海外商品，结算给海外商户美元。
- 涉及外汇结算，汇率实时波动
- 多币种清算：CNY → USD/EUR/JPY
- 跨境结算周期 T+3 ~ T+7（境外银行慢）
- 合规要求：海关报关单、外汇申报、反洗钱
- 退款涉及汇兑损益（汇率变了怎么办）
```

**架构师方案**：

**1. 多币种清算模型**：

```java
public class CrossBorderClearingDetail {
    private String clearingNo;
    private String originalCurrency;    // 原币种 CNY
    private Long originalAmount;        // 原金额
    private String settleCurrency;      // 结算币种 USD
    private Long settleAmount;          // 结算金额
    private BigDecimal exchangeRate;    // 锁定汇率
    private String rateType;            // SPOT/FORWARD
    private Date rateLockTime;          // 汇率锁定时间
    private Long exchangeGainLoss;      // 汇兑损益（退款时）
}
```

**2. 汇率锁定策略**：
- **支付时锁定**：支付成功时锁定汇率，整个链路用此汇率
- **退款时重算 vs 锁定原汇率**：
  - 锁定原汇率：商户承担汇兑损益（用户退 100 USD，按支付时汇率退 720 CNY）
  - 重算汇率：平台承担汇兑损益（用户退 100 USD，按退款时汇率退 730 CNY）
  - 业界主流：锁定原汇率，商户承担汇兑损益

**3. 跨境结算周期管理**：
- T+0：本地清算完成（CNY 侧）
- T+1：购汇指令发送（CNY → USD）
- T+2：境外银行收汇
- T+3~7：境外商户到账

**4. 合规报送**：
- 外汇申报：单笔 > 5w USD 需向外汇局申报
- 海关报关：跨境电商需推送报关单
- 反洗钱：单笔 > 1w USD 或当日累计 > 5w USD 触发 AML 审查

**5. 跨境清结算独立科目**：
```text
资产类：
  1004 境外银行存款（外币）
  1102 应收跨境款项
  1202 待结算跨境款项

负债类：
  2102 应付跨境商户款

损益类：
  3004 跨境手续费收入
  5001 汇兑损益
```

### 场景三：TCC 与清结算协同（蛋壳钱包经验复盘）

**业务背景**：
```text
蛋壳钱包 TCC 分布式事务：
- Try：冻结额度
- Confirm：扣减额度 + 生成支付成功事件
- Cancel：解冻额度

清结算需要：
- 拿到 Confirm 事件后开始清算
- Cancel 不能触发清算
- TCC 异常（如 Confirm 超时）需有兜底
```

**架构师方案**：

**1. TCC 与清结算的事件契约**：

```java
// TCC 框架在 Confirm 成功后发送事件
public class TccConfirmEvent {
    private String xid;                // TCC 全局事务ID
    private String outTradeNo;
    private Long amount;
    private Date confirmTime;
    private String confirmStatus;      // SUCCESS/TIMEOUT/UNKNOWN
}

// 清结算消费者
@EventListener
public void onTccConfirm(TccConfirmEvent event) {
    // 1. 幂等校验（xid + outTradeNo）
    if (clearingRepository.existsByXid(event.getXid())) {
        return;
    }
    
    // 2. TCC 状态确认（防止 UNKNOWN 状态触发错误清算）
    TccStatus status = tccClient.queryStatus(event.getXid());
    if (status != TccStatus.CONFIRMED) {
        log.warn("TCC not confirmed, skip clearing: xid={}", event.getXid());
        return;
    }
    
    // 3. 触发清算
    clearingService.clear(event);
}
```

**2. TCC UNKNOWN 状态处理**：
- TCC Confirm 可能因网络超时返回 UNKNOWN，但实际已成功
- 清结算不能直接清算，需先查询 TCC 状态机
- 兜底：TCC 框架定时扫描 UNKNOWN 事务，最终一致后补发事件

**3. TCC Cancel 与清结算冲正**：
- 如果 Confirm 后又收到 Cancel（理论不应发生，但需防御）
- 清结算已生成清算明细 → 触发冲正清算（clearing_type=REVERSAL）
- 通知商户系统回滚业务状态

**4. 蛋壳钱包当年设计的不足**（自评）：
- 当时 TCC 与清结算是同步调用，TCC 失败会阻塞清结算
- 应改为事件驱动，TCC 与清结算解耦
- 当时未做 TCC 状态二次确认，UNKNOWN 状态直接清算有风险

## 本日小结

- 清结算是支付系统的"账房"，四层（清算/结算/核算/对账）+ 中台是核心抽象
- 清算是"算账"，结算是"划钱"，核算是"记账"，对账是"校账"，职责正交
- 备付金是合规核心，央行 259 号文是硬约束，集中交存 + 挪用监控 + 垫资风控三件套
- 业财一体化是核算目标，业务事件 → 会计分录自动生成，月结自动化率 > 95%
- T+0 实时结算是备付金垫资，需垫资额度控制 + 流动性监控 + 渠道资金回补
- 跨境结算多币种 + 汇率锁定 + 汇兑损益 + 合规报送
- TCC 与清结算要事件驱动解耦，状态二次确认防 UNKNOWN
- 大促清结算峰值靠"异步化 + 分桶 + 合并 + 轧差"四板斧
- 差错处置与清结算联动，长款/短款/单边账自动化处置是架构师水平线
