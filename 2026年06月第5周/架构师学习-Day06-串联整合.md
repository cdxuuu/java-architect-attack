## 核心考题：电商支付系统全链路架构设计

### 业务背景

```text
某中型电商平台，业务规模：
- 日均支付 200w 笔，峰值 QPS 5000
- 大促峰值 QPS 5w，日均 2000w 笔
- 多业务线：电商（自营+第三方）、借贷、充值、提现
- 多渠道：微信、支付宝、银联、网联、备付金
- 多账户体系：商户结算账户、平台收入账户、备付金账户

历史痛点：
  - 2024 双11：支付系统雪崩，全程宕机 8 分钟，损失 2000w 交易
  - 2024 双11：渠道重复扣款 200+ 笔，客诉爆发
  - 2025 618：路由全部走某渠道，该渠道限额打满，全量失败
  - 2025 618：对账滞后 2 天，差错堆积 1w+ 笔未处理
  - 2026 春节：清结算月结不平衡，差 30w 无法结账，财务熬夜人工
  - 2026 春节：跨境支付汇率损失 5w（汇率未锁定）

目标：
  1. 全链路可用性 99.99%（年度宕机 < 53 分钟）
  2. 资金差错率 < 0.01‰
  3. 月结自动化率 > 95%
  4. 大促保障零事故
  5. 跨境支付合规 100%
```

### 设计目标

```text
1. 全链路幂等：每笔交易不重复、不丢失
2. 智能路由：成功率 > 99.5%、成本最优、故障秒级切换
3. 资金安全：备付金不挪用、月结必平衡、差错自动处置
4. 业财一体：业务事件驱动会计、月结自动化
5. 高可用：异地多活、灰度发布、故障演练常态化
6. 可观测：全链路追踪、资金安全大盘、告警秒级触达
7. SLA：P99 < 500ms、可用性 99.99%、资金差错率 < 0.01‰
```

## 架构师方案：支付系统全链路架构 + 资金安全闭环

### 一、全链路架构总览

```text
                          用户流量
                              │
                    ┌─────────▼─────────┐
        L1          │   CDN / WAF        │  ← 静态资源、CC 防护
                    └─────────┬─────────┘
                              │
                    ┌─────────▼─────────┐
        L2          │   API Gateway      │  ← 鉴权、限流、流量染色
                    │  (Spring Cloud GW) │     幂等键生成（前端+网关）
                    └─────────┬─────────┘
                              │
                    ┌─────────▼─────────┐
        L3          │   接入层            │  ← 业务线适配、协议转换
                    │  (HTTP/RPC)        │     幂等键透传、防刷
                    └─────────┬─────────┘
                              │
                    ┌─────────▼─────────┐
        L4          │   核心支付层        │  ← 支付 orchestrator
                    │  - 幂等校验         │     状态机驱动
                    │  - 风控准入         │     TCC 分布式事务
                    │  - 路由决策         │
                    │  - 渠道适配         │
                    └─────────┬─────────┘
                              │
                    ┌─────────▼─────────┐
        L5          │   渠道层            │  ← 多渠道适配器
                    │  微信/支付宝/银联   │     渠道幂等
                    └─────────┬─────────┘
                              │
                              ▼
                          渠道下单
                              │
                              ▼
                    ┌─────────────────────┐
        L6          │   账务层            │  ← 清算/结算/核算/对账
                    │  - 清算引擎         │     备付金管理
                    │  - 结算引擎         │     业财一体化
                    │  - 核算引擎         │     月结自动化
                    │  - 对账引擎         │     差错自动处置
                    └─────────────────────┘
```

### 二、全链路幂等设计（Day2 串联）

```text
[前端] 生成 out_trade_no（UUIDv7，含时间戳）
   │
   ▼
[网关] 透传 out_trade_no + 幂等校验（Redis SETNX，5s 防抖）
   │
   ▼
[接入层] 业务幂等校验（DB 唯一索引 + 状态机）
   │
   ▼
[核心层] TCC Try：幂等冻结额度（xid + out_trade_no）
   │
   ▼
[渠道层] 渠道下单：透传 out_trade_no 作为渠道请求号
   │      └─ 渠道幂等：渠道返回 SAME_REQUEST 时直接查原结果
   │
   ▼
[回调] 回调幂等：基于 out_trade_no + channel_resp_code 去重
   │
   ▼
[账务层] 清算幂等：(out_trade_no, clearing_type) 唯一索引
```

**幂等键全链路保障**：
- 前端生成：避免网关重试导致不同 out_trade_no
- 网关透传：HTTP Header `X-Idempotency-Key`，缺失则拒绝
- 业务校验：DB 唯一索引是最后防线，分布式锁只防并发不防重复
- 渠道透传：用 out_trade_no 作为渠道请求号，避免重复扣款
- 回调去重：渠道可能重发回调，必须幂等
- 清算幂等：退款清算与正向清算形成冲正对，不能重复

### 三、智能路由决策（Day4 串联）

```text
[核心层] 支付请求 → 路由决策
   │
   ▼
[L1 业务准入] 业务线识别、风控准入、合规校验
   │
   ▼
[L2 渠道池]   业务线→渠道池映射、灰度分流
   │
   ▼
[L3 渠道筛选] 硬性约束：状态/限额/币种/卡类型/维护窗
   │           └─ 渠道画像：1m/5m/15m 成功率、TP99、错误率
   │           └─ 熔断器：错误率 > 10% / TP99 > 2s / 连续失败 20 次
   ▼
[L4 排序]     软性偏好打分：
   │           Score = 0.45·成功率 + 0.25·成本 + 0.15·TP99 + 0.10·限额 + 0.05·偏好
   ▼
[L5 兜底]     全挂兜底：ToC 拒绝 / 还款走备付金 / 提现人工
   │
   ▼
[渠道下单] + Fallback 链（如：微信 → 支付宝 → 银联）
```

**路由决策日志**（供对账与清结算消费）：
```json
{
  "out_trade_no": "T202607041234567890",
  "final_channel": "WECHAT",
  "final_merchant_id": "WX_001",
  "fallback_chain": [
    {"channel": "LIANLIAN", "result": "FAIL", "resp_code": "E001"},
    {"channel": "WECHAT", "result": "SUCCESS", "channel_request_id": "WX_xxx"}
  ],
  "decision_time": "2026-07-04 14:00:00.123",
  "channel_request_id": "WX_xxx"
}
```

### 四、对账与差错处置（Day3 串联）

```text
[L6 账务层] 对账引擎
   │
   ├─ 业务对账（内部）：订单↔支付↔清算↔结算
   │   └─ 不平则告警，定位到具体环节
   │
   ├─ 渠道对账（外部1）：支付明细↔渠道账单
   │   └─ 差错类型：长款 / 短款 / 单边账
   │
   └─ 资金对账（外部2）：结算单↔银行回单 / 备付金↔央行账单
       └─ 差错类型：未到账 / 金额不符 / 跨日交易

差错处置：
  - 长款 → 挂"其他应付款"，待商户认领或转营业外收入
  - 短款 → 挂"其他应收款"，向渠道追索或商户补扣
  - 单边账 → 挂账待查，7 天未清转营业外收支

自动处置率目标：> 95%
```

**对账与路由协同**：
- 路由决策日志中的 `fallback_chain` 帮助对账识别"试单"痕迹
- 试单渠道账单需副表对账，避免误判为短款
- 跨日交易按 `decision_time` 特殊勾对

### 五、清结算核算闭环（Day5 串联）

```text
支付成功事件 → [L1 清算]
                    │
                    ▼ 计算应收/应付/分账/手续费
                [L2 结算]
                    │
                    ▼ 按周期聚合、生成结算单、调拨备付金
                [L3 核算]
                    │
                    ▼ 按会计准则生成会计分录、过总账
                [L4 对账]
                    │
                    ▼ 三层对账 + 差错处置
                财务总账 / 监管报表
```

**清结算与 TCC 协同**：
```text
TCC Try（冻结额度）
   │
   ├─ 失败 → 不触发清结算
   │
TCC Confirm（扣减额度 + 发送事件）
   │
   ├─ 事件 → 清结算消费
   │       ├─ 幂等校验（xid + out_trade_no）
   │       ├─ TCC 状态二次确认（防 UNKNOWN）
   │       └─ 触发清算
   │
TCC Cancel（解冻额度）
   │
   └─ 已 Confirm 后收到 Cancel → 触发冲正清算（REVERSAL）
```

### 六、备付金管理与合规（Day5 串联）

```text
        备付金账户（央行交存）
        ┌──────────────────────┐
        │ 实收备付金（资产）   │
        │ 应付备付金（负债）   │
        └──────────────────────┘
              │          │
   ┌──────────┘          └──────────┐
   ▼                                ▼
商户结算账户                     银行备付金往来
（应付商户款）                   （资金调拨）

合规三件套：
  1. 备付金集中交存（央行 259 号文）
  2. 挪用监控（实收 > 应付 + 自有资金，超部分不能挪用）
  3. 每日余额报送（超限处罚）

T+0 垫资风控：
  1. 单商户日垫资额度（按信用动态调整）
  2. 总垫资池上限（备付金余额 80%）
  3. 垫资逾期监控（T+1 未到账告警）
```

## 资金安全闭环设计

### 一、资金安全三大原则

```text
原则1：每一笔钱可追溯
  └─ out_trade_no 全链路透传，决策日志 + 清算明细 + 会计分录 + 对账记录可串

原则2：每一笔钱有对账
  └─ 业务对账（内部）+ 渠道对账（外部1）+ 资金对账（外部2）

原则3：每一笔钱有兜底
  └─ 路由 Fallback + 清算冲正 + 差错挂账 + 人工兜底
```

### 二、资金安全大盘

```text
资金安全大盘（实时监控）
┌──────────────────────────────────────────┐
│  实时指标                                  │
│  ────────────────────────────────────│
│  • 支付成功率（实时）                       │
│  • 清算延迟 P99                             │
│  • 备付金余额 + 垫资余额                    │
│  • 差错堆积数（按类型）                      │
│  • 月结平衡状态                             │
├──────────────────────────────────────────┤
│  告警规则                                  │
│  ────────────────────────────────────│
│  • 支付成功率 5min 内降 5% → P1 告警       │
│  • 清算延迟 P99 > 10s → P2 告警            │
│  • 备付金余额 < 阈值 → P1 告警              │
│  • 差错堆积 > 1000 笔 → P2 告警             │
│  • 月结不平衡 → P0 告警（CFO 通报）         │
└──────────────────────────────────────────┘
```

### 三、故障演练常态化

```text
演练场景（每月一次）：
  1. 渠道故障切换：模拟微信全量失败，验证 Fallback 链
  2. 清结算峰值：模拟双11 流量，验证清算延迟与结算批次
  3. 资金对账异常：注入短款/长款差错，验证自动处置
  4. TCC UNKNOWN：模拟 TCC Confirm 超时，验证状态二次确认
  5. 备付金挪用：模拟超垫资额度，验证风控拦截

演练评估：
  • 切换时间 < 30s
  • 资金差错率 < 0.01‰
  • 自动处置率 > 95%
  • 月结自动化率 > 95%
```

## 大促保障 SOP（双11 级别）

### 一、大促前（T-30 天）

```text
1. 容量规划
   ├─ 历史峰值 × 3 倍冗余
   ├─ DB 分库分表容量评估
   ├─ Redis 集群扩容
   └─ 渠道侧 TPS 配额申请

2. 压测验证
   ├─ 全链路压测（含下游渠道沙盒）
   ├─ 单服务压测（找瓶颈）
   ├─ 故障演练（注入异常）
   └─ 异地多活切换演练

3. 配置准备
   ├─ 路由规则预演（按历史流量回放）
   ├─ 限流阈值预设（网关/应用/资源三层）
   ├─ 灰度发布计划
   └─ 应急预案文档化
```

### 二、大促中（T+0）

```text
实时监控（每分钟）：
  • 支付成功率、TP99、错误率
  • 路由决策分布、Fallback 触发率
  • 清算延迟、结算批次堆积
  • 备付金余额、垫资额度
  • 差错堆积数

应急响应：
  • P1 告警 5min 内响应
  • P0 告警 1min 内响应 + 通报 CFO
  • 渠道故障 30s 内切换 Fallback
  • 清算堆积 > 10w 笔触发降级（异步化优先）
```

### 三、大促后（T+1 ~ T+7）

```text
T+1：
  • 切账对账（业务/渠道/资金三层）
  • 差错处置（自动 + 人工）
  • 清算冲正（退款高峰）
```
