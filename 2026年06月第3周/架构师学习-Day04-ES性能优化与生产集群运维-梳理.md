## 一、写入性能优化体系

### 1.1 写入流程与优化点

```latex
ES写入流程（简化版）：
  1. 客户端请求 → 协调节点
  2. 协调节点路由 → 主分片
  3. 主分片写translog + 内存buffer
  4. 主分片转发请求 → 副本分片
  5. 副本分片写translog + 内存buffer
  6. 所有分片写完 → 返回客户端
  7. 后台refresh：内存buffer → segment（1s后）
  8. 后台flush：translog → 持久化到磁盘

优化点分布：
  客户端：Bulk API批量
  协调节点：路由优化
  主/副本：translog配置、refresh间隔
  后台：segment合并策略
```

### 1.2 写入优化矩阵

| 优化手段 | 效果 | 风险 | 适用场景 |
|---------|------|------|---------|
| Bulk API 5-15MB/批 | 2-5倍提升 | 批次太大会OOM | 所有写入场景 |
| 写入时副本数设为0 | 2-3倍提升 | 期间无副本，数据可能丢 | 批量导入数据 |
| refresh_interval设为30s | 1.5-2倍提升 | 文档30s后才可见 | 日志类场景 |
| translog async模式 | 1.2-1.5倍提升 | 节点挂掉可能丢5s数据 | 日志类场景 |
| 用SSD | 5-10倍提升 | 成本高 | 所有生产环境 |

### 1.3 架构师视角：写入优化的本质

```latex
写入优化的核心矛盾：
  性能 vs 可靠性 vs 实时性

架构决策：
  1. 数据能不能丢？
     → 不能丢 → durability=request，副本数>=1
     → 可以丢少量 → durability=async，写入时副本数=0

  2. 多久需要搜索到？
     → 需要近实时 → refresh_interval=1s
     → 不需要 → refresh_interval=30s或更长

  3. 能不能分阶段优化？
     → 批量导入：先关副本、关refresh，导入完成后恢复
     → 实时写入：用异步translog、调大refresh_interval
```

## 二、查询性能优化体系

### 2.1 查询性能问题根因分析

```latex
查询慢的常见原因：
  1. 扫描文档太多
     → 没走filter缓存
     → filter顺序不对，没先过滤掉大部分文档
     → 索引设计不合理，分片太多

  2. 打分计算太多
     → 用了must而不是filter
     → 复杂的function_score
     → 查询条件太复杂

  3. 聚合计算太多
     → 聚合层级太深
     → 用了text字段聚合（fielddata）
     → 聚合的桶太多

  4. 磁盘IO问题
     → segment太多，查询要扫很多segment
     → 机械硬盘，随机读慢
```

### 2.2 查询优化思路

```latex
架构师优化思路：
  第一步：Profile，看慢在哪里
  第二步：减少扫描的文档量（filter、routing）
  第三步：减少计算量（不用打分、简化聚合）
  第四步：减少IO（SSD、force merge）
  第五步：资源扩容（增加节点、升级配置）
```

### 2.3 Filter缓存利用策略

```latex
什么时候用filter？
  → 所有只是过滤、不需要打分的条件都放filter

filter的缓存特性：
  → 自动缓存
  → 相同条件下次直接命中
  → 不影响_score

缓存策略建议：
  → 最常使用、过滤结果最稳定的条件放filter
  → 比如：tenant_id、category、date_range
  → 不要用script做filter（不能缓存）
```

## 三、集群架构规划方法论

### 3.1 集群规模估算公式

```latex
数据量估算：
  原始数据量 = 日增量 × 保留天数
  总存储量 = 原始数据量 × (1 + 副本数) × 1.5（膨胀系数）

分片数估算：
  单个分片大小 = 20-40GB（建议）
  单个索引分片数 = 该索引预计数据量 / 30GB

节点数估算：
  数据节点数 = 总存储量 / 单个节点可用存储
  主节点数 = 3（固定）
  协调节点数 = 2-3（大集群建议）
  总节点数 = 数据节点 + 主节点 + 协调节点

节点配置：
  CPU：16核+
  内存：32-64GB（heap=31GB）
  磁盘：SSD，建议单个节点2-4TB可用
  网络：万兆
```

### 3.2 分片规划原则

```latex
分片数不是越多越好：
  → 每个分片有元数据开销
  → 查询需要fan-out到所有分片
  → 分片太多会导致master压力大

原则1：分片大小20-40GB
  → 太小：分片数太多
  → 太大：恢复慢、rebalance慢

原则2：分片数不要超过节点数的3倍
  → 充分利用每个节点
  → 又不会太多

原则3：后续可以split，但不能shrink太多
  → 初期可以规划保守一点
  → 后续split API可以增加分片数（翻倍）
```

### 3.3 节点角色分离架构

```latex
生产集群建议架构：

  3个Master节点：
    → 配置：4核8G，普通磁盘
    → 作用：集群管理
    → 不处理读写请求

  N个Data节点：
    → 配置：16核32-64G，SSD
    → 作用：存储数据，处理读写
    → 可以分Hot/Warm层

  2-3个Coordinating节点（可选）：
    → 配置：8核16G
    → 作用：请求路由、结果聚合
    → 大集群建议独立

优点：
  → 角色隔离，互不影响
  → Master节点稳定，集群就稳定
  → Data节点可以独立扩容
```

## 四、监控与运维体系

### 4.1 核心监控指标体系

```latex
架构师关注的核心指标：

  集群健康度：
    → cluster.status（green/yellow/red）
    → unassigned_shards（未分配分片数）

  资源使用率：
    → JVM heap（>80%告警，>90%危险）
    → 磁盘使用率（>80%告警，>90%危险）
    → CPU使用率（>70%注意）

  性能指标：
    → 查询平均耗时（>500ms告警）
    → 写入平均耗时（>100ms告警）
    → GC频率和耗时（Old GC频繁要注意）

  业务指标：
    → QPS
    → 错误率
    → 慢查询数量
```

### 4.2 告警阈值建议

| 指标 | 警告阈值 | 危险阈值 | 处理优先级 |
|-----|---------|---------|-----------|
| cluster.status | yellow | red | P0 |
| JVM Heap | 80% | 90% | P1 |
| 磁盘使用率 | 80% | 90% | P1 |
| Old GC频率 | 5次/分 | 10次/分 | P2 |
| 查询平均耗时 | 500ms | 2s | P2 |
| 节点离线 | 任何时候 | - | P0 |
| 未分配分片 | >0 | - | P1 |

### 4.3 磁盘水位线理解

```latex
ES的三个磁盘水位线：
  low watermark：85%
    → 不再分配新分片到该节点
    → 只是警告，可以继续写入现有分片

  high watermark：90%
    → 尝试把该节点的分片移到其他节点
    → 可以继续写入，但在做搬迁准备

  flood stage：95%
    → 所有索引设为只读
    → 禁止写入，保护集群

架构师视角：
  → 80%就应该告警，准备扩容或清理
  → 不要等90%才处理
  → 定期删除旧数据是好习惯
```

## 五、故障排查与应急处理

### 5.1 集群Red状态排查流程

```latex
架构师排查步骤：

  1. 确认哪些索引Red
     → GET /_cluster/health
     → GET /_cat/indices?v&health=red

  2. 确认哪些分片未分配
     → GET /_cat/shards?v
     → 看哪些分片是UNASSIGNED

  3. 分析未分配原因
     → GET /_cluster/allocation/explain
     → 看"reason"字段

  4. 针对性处理
     → 节点离线：重启节点
     → 磁盘满：清理磁盘或扩容
     → 分片损坏：从副本恢复或重新索引

  5. 恢复后验证
     → GET /_cluster/health
     → 确认变成green/yellow
```

### 5.2 常见故障场景与处理

| 故障场景 | 表现 | 紧急处理 | 长期优化 |
|---------|------|---------|---------|
| 节点宕机 | 集群yellow/red，部分分片未分配 | 重启节点；如果无法重启，等待集群自动恢复 | 确保有足够副本；节点配置更稳定 |
| 磁盘满 | 集群只读，写入失败 | 删除旧索引；清理磁盘空间 | 配置ILM自动删除；监控磁盘使用率提前扩容 |
| OOM | 节点挂掉，日志OOM | 立即重启节点；暂时切走流量 | 优化查询/聚合；增加内存；增加节点 |
| 写入慢 | 写入队列积压，响应慢 | 临时调大refresh_interval；减少并发 | 检查GC；检查segment数量；扩容 |
| 查询慢 | 查询超时，用户投诉 | 取消慢查询；用filter优化 | Profile分析；force merge；扩容 |

### 5.3 应急预案建立

```latex
架构师应该提前准备的应急预案：

  1. 容量不足预案：
     → 什么情况要扩容（磁盘/CPU/内存）
     → 扩容流程（加节点 → 等待rebalance）

  2. 集群故障预案：
     → 集群red怎么处理
     → 数据丢失怎么恢复（从备份/从源数据重导）

  3. 性能突降预案：
     → 写入慢：调大refresh_interval，临时降副本
     → 查询慢：临时取消部分聚合，加缓存

  4. 变更回滚预案：
     → 配置变更前先记录当前值
     → 变更后观察一段时间
     → 有问题立即回滚
```

## 六、从架构师视角看ES生产环境

### 6.1 ES在系统架构中的定位

```latex
ES常见使用场景：
  1. 全文搜索（产品搜索、文章搜索）
  2. 日志分析（ELK）
  3. 时序数据分析（监控指标）
  4. 聚合统计（BI报表）

架构原则：
  → ES不是数据库，不要当数据库用
  → 不要存唯一数据，要有其他数据源
  → ES可以重建，数据来源要可靠
  → 适合读多写少，不适合高并发写入
```

### 6.2 生产环境检查清单

```latex
上线前检查：
  ✓ 节点角色分离：Master/Data/Coordinating
  ✓ 3个Master节点，防脑裂
  ✓ 副本数配置合理（至少1个）
  ✓ 分片大小合理（20-40GB）
  ✓ Heap设为31GB，不超过物理内存50%
  ✓ 用SSD
  ✓ 监控告警配置完成
  ✓ ILM策略配置（如果需要）
  ✓ 数据备份方案

日常运维检查：
  ✓ 集群状态green
  ✓ JVM Heap < 80%
  ✓ 磁盘使用率 < 80%
  ✓ GC正常（没有频繁Old GC）
  ✓ 没有慢查询/慢写入
  ✓ 分片均匀分配
```

### 6.3 能力差距与成长路径

```latex
从开发到架构师的成长：

  开发阶段：
    → 会写CRUD操作
    → 知道基本的查询语法
    → 会用Bulk API

  高级开发阶段：
    → 会做基本的性能优化（filter、source filtering）
    → 理解分片和副本
    → 会看基本的监控指标

  架构师阶段：
    → 能根据业务场景设计集群架构
    → 能系统性优化性能（写入+查询）
    → 能建立完整的监控运维体系
    → 能排查复杂故障
    → 能做容量规划和成本优化

关键转折点：
  → 从"怎么用"到"怎么设计"
  → 从"出问题再处理"到"预判问题并预防"
  → 从"只看ES"到"看ES在整个系统中的位置"
```
