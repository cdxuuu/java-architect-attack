# Day 2：读写分离与主从复制 — 架构师角度梳理

## 核心定位

Day2主题一句话概括：**读写分离是分库分表的前置方案，主从复制是基础设施。一致性强度要按场景分层（异步/半同步/MGR），写后读问题需要多层方案叠加（事务/注解/N秒/GTID），不是越严格越好，是刚好够用**。

---

## 主从复制：binlog是核心

```
完整复制流程：

  主库写binlog → IO线程接收 → 写relay log → SQL线程回放

  binlog三种格式：
    Statement → 日志小但函数歧义（NOW/UUID/RAND）
    Row       → 严格一致但日志大（金融必选）
    Mixed     → 折中但有漏判风险

  架构师选型：
    金融/订单 → Row（一致性高于一切）
    日志/统计 → Mixed（折中）
    几乎不用Statement（函数歧义=资金错误）
```

---

## 主从延迟：根因在SQL线程

```
延迟根因层次：
  L1 网络延迟（毫秒级，正常）
  L2 IO线程瓶颈（少见，顺序写）
  L3 SQL线程瓶颈（主要原因）
     5.7之前：单线程串行回放
     5.7+：group commit并行
     8.0+：WriteSet并行

五大场景的解法：
  写后读     → 强制读主（事务/注解/N秒/GTID）
  大事务     → 分批执行（while循环+sleep）
  DDL执行    → gh-ost/pt-online-schema-change
  从库压力大 → 读库分离（OLTP库vs报表库）
  硬件低配   → 从库配置≥主库

监控用心跳表（Seconds_Behind_Master在网络断开时不准）
告警阈值分级：5秒/30秒/5分钟，5分钟以上切流主库
```

## 读写分离：四种实现方式

```
  ┌──────────┬──────────────┬──────────────┐
  │  方式     │  侵入性       │  适用         │
  ├──────────┼──────────────┼──────────────┤
  │  手动     │  高           │  极致性能     │
  │  注解+AOP │  中（推荐）   │  通用         │
  │  SQL自动  │  低           │  零侵入       │
  │  Proxy   │  零（应用）   │  多语言/DBA管 │
  └──────────┴──────────────┴──────────────┘

四大关键问题：
  1. 写后读必须主库（事务内/注解/N秒/GTID）
  2. 事务必须主库（同一连接保证一致）
  3. 多从库负载均衡（轮询/权重/最小连接/最低延迟）
  4. 全部从库挂了降级主库（注意压力+告警）
```

## 一致性方案：异步/半同步/MGR

```
  ┌──────────┬──────────┬──────────┬──────────┐
  │  模式    │  性能    │  一致性  │  适用     │
  ├──────────┼──────────┼──────────┼──────────┤
  │  异步    │  最好    │  弱      │  非金融  │
  │  半同步  │  中      │  强      │  金融必选│
  │  全同步  │  最差    │  最强    │  几乎不用│
  │  MGR     │  中      │  强      │  替代MHA │
  └──────────┴──────────┴──────────┴──────────┘

  半同步的退化机制：
    从库都挂了 → 退化为异步
    rpl_semi_sync_master_timeout控制等待超时

  MGR vs 半同步：
    半同步 + MHA = 强一致 + 自动故障转移
    MGR = 内置强一致 + 内置故障转移 + 内置脑裂处理
    新项目优先MGR（8.0+较成熟）
```

## 强制读主：四种方案分层

```
  L1 事务内自动主库（框架默认）
     原理：开启事务绑定连接，整个事务同一连接
     适用：所有事务场景

  L2 注解显式@DataSource("master")
     适用：明确知道需要最新数据的关键查询

  L3 写后N秒强制
     原理：Redis记录用户写时间戳，N秒内查询走主库
     N选择：3-5秒覆盖95%场景
     适用：用户中心/订单中心

  L4 GTID跟踪
     原理：主库返回GTID，客户端比较从库GTID
     从库GTID ≥ 客户端GTID → 走从库
     适用：金融/资金极致一致

  推荐组合：
    基础：事务内自动（L1）
    补充：注解关键查询（L2）
    可选：用户中心写后3秒（L3）
    极致：资金场景GTID（L4）

  和Day2纵深防御一脉相承：多层叠加按场景选用
```

## Day2与之前知识的串联

```
  Day1分库分表前置 ← Day2读写分离（扩展读能力，挡住60-70%读压力）
  Day1垂直拆分前置 ← Day2读写分离（先做最简单的，再考虑拆库）

  Day2 binlog格式选型     ← 第5周Day3数据特性驱动选型
  Day2半同步选型          ← 第5周Day7数据特性决定协议
  Day2强制读主四方案     ← 第5周Day2五层漏斗（多层叠加按场景）

  本周核心思维：
    数据库扩展性是分层的：
      读写分离 → 分库 → 分表 → 单元化
    每一层成本递增，按瓶颈选刚好够用的
    不是上来就分库分表
```
