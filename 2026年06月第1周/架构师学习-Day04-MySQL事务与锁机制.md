# Day 4：MySQL事务与锁机制 — 隔离级别/MVCC/锁分类/死锁排查/长事务治理/高并发事务设计（练习整理）

## 题目一：事务隔离级别与MVCC

### 1.1 RR 和 RC 在快照读与当前读上的行为差异

```
快照读（普通SELECT，不加锁）：
  ┌──────────────────────────────────────────────────────────┐
  │  维度         │  RR                    │  RC              │
  ├───────────────┼────────────────────────┼──────────────────┤
  │  ReadView生成 │  事务内第一次快照读时   │  每次快照读时    │
  │               │  生成一次，后续复用     │  都生成新的      │
  ├───────────────┼────────────────────────┼──────────────────┤
  │  可见性       │  固定在首次快照时刻     │  每次读最新已提交│
  ├───────────────┼────────────────────────┼──────────────────┤
  │  不可重复读   │  ✅ 解决               │  ❌ 存在         │
  ├───────────────┼────────────────────────┼──────────────────┤
  │  幻读         │  ✅ 快照读层面解决      │  ❌ 存在         │
  └───────────────┴────────────────────────┴──────────────────┘

当前读（SELECT ... FOR UPDATE / LOCK IN SHARE MODE / UPDATE / DELETE）：
  ┌──────────────────────────────────────────────────────────┐
  │  维度         │  RR                    │  RC              │
  ├───────────────┼────────────────────────┼──────────────────┤
  │  锁粒度       │  Next-Key Lock         │  Record Lock     │
  │               │  (Record + Gap)        │  (只锁命中行)    │
  ├───────────────┼────────────────────────┼──────────────────┤
  │  间隙锁       │  ✅ 有                 │  ❌ 没有         │
  ├───────────────┼────────────────────────┼──────────────────┤
  │  幻读         │  ✅ 当前读层面解决      │  ❌ 存在         │
  └───────────────┴────────────────────────┴──────────────────┘

  关键认知：
    锁的"模式"（X/S）和"粒度"（Record/Gap/Next-Key）是正交的两个维度
    X锁 = 排它锁（FOR UPDATE）
    S锁 = 共享锁（LOCK IN SHARE MODE）
    不管X还是S，RC只锁命中行，RR还会锁间隙

  RR 防幻读是两套机制配合：
    快照读 → MVCC（ReadView固化）
    当前读 → Next-Key Lock（锁间隙阻止插入）
    只说其中一套 = 没答全
```

### 1.2 ReadView 生成时机如何导致不可重复读差异

```
ReadView 四个核心字段：
  m_ids       → 生成时刻所有活跃（未提交）事务ID列表
  min_trx_id  → m_ids中最小的事务ID（低水位）
  max_trx_id  → 系统分配给下一个事务的ID（高水位）
  creator_trx_id → 创建该ReadView的事务ID

可见性判断规则（对每行数据的版本链遍历）：
  trx_id < min_trx_id        → 可见（事务已提交）
  trx_id >= max_trx_id       → 不可见（事务在ReadView之后开始）
  min_trx_id <= trx_id < max_trx_id：
    trx_id in m_ids           → 不可见（事务未提交）
    trx_id not in m_ids       → 可见（事务已提交）

RR vs RC 的差异本质：
  RR：ReadView在事务内第一次快照读时生成 → m_ids固化 → 可见性判断不变
  RC：每次快照读都生成新ReadView → m_ids更新 → 新提交的事务变可见

  时间线示例：
    T1: BEGIN
    T2: SELECT (RR生成ReadView，m_ids=[T1])
    T3: 另一事务T3提交了修改
    T4: SELECT (RR复用ReadView，m_ids仍=[T1]，T3不可见 → 不可重复读✅解决)
    T4: SELECT (RC生成新ReadView，m_ids不含T3，T3可见 → 不可重复读❌存在)

  关键细节（面试常考）：
    RR的ReadView在「第一次快照读」时生成，不是BEGIN时
    BEGIN; ...别的提交了...; 第一次SELECT → 看不到（ReadView才生成）
    想在事务开始就锁定快照 → START TRANSACTION WITH CONSISTENT SNAPSHOT
```

### 1.3 为什么阿里建议RC / 哪些场景必须RR

```
阿里推荐RC的五个理由：

  ┌────┬──────────────────┬──────────────────────────────────────┐
  │ #  │  维度             │  RR的问题 → RC的改进                 │
  ├────┼──────────────────┼──────────────────────────────────────┤
  │ 1  │  锁范围           │  Next-Key Lock锁间隙→并发更新易死锁   │
  │    │                  │  → RC只锁命中行，并发度高              │
  ├────┼──────────────────┼──────────────────────────────────────┤
  │ 2  │  死锁概率         │  间隙锁交叉极易死锁                   │
  │    │                  │  → RC无间隙锁，死锁概率显著下降        │
  ├────┼──────────────────┼──────────────────────────────────────┤
  │ 3  │  主从复制         │  statement binlog必须RR保证一致        │
  │    │                  │  → row格式后RR优势消失，RC可用         │
  ├────┼──────────────────┼──────────────────────────────────────┤
  │ 4  │  可见性直觉       │  事务内看不到别人已提交数据，易踩坑    │
  │    │                  │  → RC每次读最新已提交，符合业务预期     │
  ├────┼──────────────────┼──────────────────────────────────────┤
  │ 5  │  半一致性读       │  RR没有                               │
  │    │  (semi-consistent)│  → RC下UPDATE遇到被锁行，读最新版本   │
  │    │                  │  判断是否匹配，不匹配直接跳过不加锁    │
  │    │                  │  大幅降低锁冲突                       │
  └────┴──────────────────┴──────────────────────────────────────┘

  半一致性读详解（RC的杀手级优势）：
    UPDATE user SET name='x' WHERE age=25  -- age无索引

    RR下：每行都加Next-Key Lock → 等价锁全表
    RC下：InnoDB对每行做「半一致性读」
      → 遇到被锁行时，读最新已提交版本
      → 如果最新版本age≠25 → 不加锁，直接跳过
      → 只有最新版本age=25才加锁等待
      → 大幅减少锁冲突

  必须留在RR的场景：

  1. 对账/报表：事务内多次查询结果必须一致
     例：月底结算，查余额→算利息→生成对账文件
     RC下中途有交易提交→余额变了→对账不平

  2. 唯一性校验+插入：
     SELECT检查是否存在 → 不存在 → INSERT
     RC下两个事务都通过校验 → 都插入 → 唯一约束冲突
     RR用Gap Lock锁住范围 → 第二个事务阻塞

     现代做法：用唯一索引兜底，不依赖隔离级别

  3. statement格式binlog同步（几乎绝迹）

  选型决策树：
    是否有跨事务一致性读需求（报表/对账）？
    ├─ 是 → RR（或单独连接用RR）
    └─ 否 → RC（绝大多数OLTP场景）

    是否大量使用SELECT ... FOR UPDATE范围锁？
    ├─ 是 → 警惕RR死锁，考虑RC+业务层去重
    └─ 否 → RR/RC都行

    主从复制格式？
    ├─ statement → 必须RR（但应升级到row）
    └─ row/mixed → RR/RC都行
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | MySQL默认RR隔离级别，RC有不可重复读问题 |
| 高级开发 | RR通过MVCC和间隙锁解决幻读，RC没有间隙锁 |
| 架构师 | 隔离级别选型是成本权衡：RR锁范围大→死锁概率高但防幻读完整，RC锁范围小→并发高但幻读要业务兜底。半一致性读是RC在并发更新场景的杀手级优势。选型决策树：OLTP用RC+唯一索引兜底、报表对账用RR+WITH CONSISTENT SNAPSHOT。能说清"RR防幻读靠两套机制配合"而不是笼统说"RR解决幻读" |

**补足方向**：面试被问"为什么改RC"，不要只说"减少死锁"，要答出五个维度（锁范围/死锁/主从/可见性直觉/半一致性读）。被追问"RC不会幻读吗"，要答"会，但唯一索引+业务兜底比间隙锁更可控"。

## 题目二：锁的分类与Next-Key Lock

### 2.1 三种行锁的区别

```
  ┌────────────────┬──────────────────┬──────────────────────────────────┐
  │  锁类型         │  锁的是什么       │  什么时候加                       │
  ├────────────────┼──────────────────┼──────────────────────────────────┤
  │  Record Lock   │  索引上的某条记录 │  唯一索引等值命中/聚簇索引命中     │
  │  (记录锁)       │                  │                                  │
  ├────────────────┼──────────────────┼──────────────────────────────────┤
  │  Gap Lock      │  索引记录之间的   │  唯一索引等值未命中/范围查询尾部   │
  │  (间隙锁)       │  间隙，不含记录   │                                  │
  ├────────────────┼──────────────────┼──────────────────────────────────┤
  │  Next-Key Lock │  Record Lock +   │  RR下默认加锁方式                 │
  │  (临键锁)       │  Gap Lock        │  普通索引命中时保持不退化          │
  │                │  = 左开右闭区间   │                                  │
  └────────────────┴──────────────────┴──────────────────────────────────┘

  关键认知修正：
    ❌ "Record Lock只在聚簇索引/主键上" → Record Lock可以出现在任何索引上
    ❌ "Gap Lock多用于插入" → Gap Lock是阻止别人插入，不是对插入加锁
    ✅ Gap Lock之间不互斥，只和Insert Intention Lock冲突
       → 这是Gap Lock导致死锁的根本原因

  锁的退化规则（RR隔离级别）：
    Next-Key Lock 在特定条件下退化：
      唯一索引等值命中 → 退化为 Record Lock（只锁命中行）
      唯一索引等值未命中 → 退化为 Gap Lock（只锁间隙）
      范围查询扫到不匹配行 → 退化为 Gap Lock（只锁尾部间隙）
      普通索引等值命中 → 不退化，保持 Next-Key Lock
```

### 2.2 主键 vs 普通索引加锁对比

```
  假设数据：主键id = 5, 10, 15, 20, 25

  ━━━ 场景A：主键索引 ━━━
  SELECT * FROM t WHERE id BETWEEN 10 AND 20 FOR UPDATE;

  扫描过程：
    id=10  Next-Key Lock (5,10]  → 唯一索引命中，退化为 Record Lock on 10
    id=15  Next-Key Lock (10,15] → 退化为 Record Lock on 15
    id=20  Next-Key Lock (15,20] → 退化为 Record Lock on 20
    id=25  扫描到不匹配          → Gap Lock (20,25)

  最终锁：Record Lock(10,15,20) + Gap Lock(20,25)

  ━━━ 场景B：普通索引(col) ━━━
  SELECT * FROM t WHERE col BETWEEN 10 AND 20 FOR UPDATE;

  扫描过程：
    col=10  Next-Key Lock (5,10]   → 不退化
    col=15  Next-Key Lock (10,15]  → 不退化
    col=20  Next-Key Lock (15,20]  → 不退化
    col=25  不匹配                  → Gap Lock (20,25)
    回表    聚簇索引加 Record Lock on 命中行的主键

  最终锁：
    普通索引：Next-Key Lock(5,10]+(10,15]+(15,20] + Gap Lock(20,25)
    聚簇索引：Record Lock on 命中行主键

  ━━━ 核心差异 ━━━

  ┌───────────┬──────────────────────┬────────────────────────────────┐
  │  维度      │  主键（唯一索引）      │  普通索引                       │
  ├───────────┼──────────────────────┼────────────────────────────────┤
  │  命中行    │  退化→Record Lock    │  不退化→Next-Key Lock           │
  │  原因      │  唯一索引确保只有一行  │  可能有重复值，必须锁间隙        │
  │           │  不需要锁间隙防插入    │  防止同值插入                    │
  ├───────────┼──────────────────────┼────────────────────────────────┤
  │  间隙范围  │  只锁尾部(20,25)      │  锁全部间隙                     │
  │           │                      │  (5,10)+(10,15)+(15,20)+(20,25)│
  ├───────────┼──────────────────────┼────────────────────────────────┤
  │  回表锁    │  不需要               │  聚簇索引加Record Lock          │
  ├───────────┼──────────────────────┼────────────────────────────────┤
  │  并发影响  │  小                   │  大（锁范围更宽）                │
  └───────────┴──────────────────────┴────────────────────────────────┘

  ━━━ 无索引列的查询（杀手题） ━━━
  UPDATE user SET name='x' WHERE age=25;  -- age无索引

  结果：聚簇索引每一行都加Next-Key Lock = 锁全表

  原因：MySQL Server层逐行读出判断，InnoDB不知道哪些行不满足
        只能对扫描到的每一行都加锁

  这就是为什么UPDATE/DELETE必须走索引！
```

### 2.3 RC下重现幻读

```
  场景：account表，id为主键，现有数据 id=1, 3, 5

  -- 事务 A                            -- 事务 B
  BEGIN;
  SELECT * FROM account
  WHERE id BETWEEN 1 AND 5;
  -- 结果: id=1, 3, 5

                                       BEGIN;
                                       INSERT INTO account VALUES(4,'D',500);
                                       COMMIT;

  -- 事务B已提交
  SELECT * FROM account
  WHERE id BETWEEN 1 AND 5;
  -- RC: 看到 id=1,3,4,5  ← 幻读！
  -- RR: 看到 id=1,3,5    ← 无幻读

  -- 更狠的：当前读
  UPDATE account SET balance=0
  WHERE id BETWEEN 1 AND 5;
  -- RC: 更新了4行（含id=4）

  SELECT * FROM account
  WHERE id BETWEEN 1 AND 5;
  -- RC: id=4的balance=0，但之前快照读没看到它
  -- "先看不到，后改到了" = 幻读

  ━━━ RR为什么不会出现？━━━

  ┌──────────┬──────────────────────────────────────────────┐
  │  读类型   │  RR怎么防                                    │
  ├──────────┼──────────────────────────────────────────────┤
  │  快照读   │  ReadView在第一次SELECT固化                  │
  │          │  id=4的事务不在可见范围内                      │
  ├──────────┼──────────────────────────────────────────────┤
  │  当前读   │  Next-Key Lock锁住(3,5)间隙                  │
  │          │  事务B的INSERT阻塞等待                        │
  └──────────┴──────────────────────────────────────────────┘

  ━━━ RC为什么防不住？━━━

  ┌──────────┬──────────────────────────────────────────────┐
  │  读类型   │  RC为什么失败                                │
  ├──────────┼──────────────────────────────────────────────┤
  │  快照读   │  每次读生成新ReadView                        │
  │          │  id=4已提交就可见                              │
  ├──────────┼──────────────────────────────────────────────┤
  │  当前读   │  只加Record Lock，不锁间隙                   │
  │          │  INSERT直接成功                               │
  └──────────┴──────────────────────────────────────────────┘
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | InnoDB有行锁和表锁，SELECT FOR UPDATE加行锁 |
| 高级开发 | 知道Record Lock/Gap Lock/Next-Key Lock，锁加在索引上 |
| 架构师 | 锁的本质是索引上的锁，加锁行为由索引类型+查询类型+隔离级别三维度决定。唯一索引等值命中退化成Record Lock是关键优化。普通索引要回表加锁。无索引=锁全表。面试一句话："唯一索引命中退化为Record Lock，普通索引不退化——根本原因是普通索引可能有重复值必须锁间隙防同值插入" |

**补足方向**：面试时被问锁，先说"锁加在索引上"，再说"索引类型决定锁的行为"，最后给"退化规则速查表"。

## 题目三：死锁排查实战

### 3.1 死锁形成过程

```
  蛋壳钱包转账场景：
    事务A：从1001转100到1002
    事务B：从1002转50到1001

  ━━━ 执行序列 ━━━

  时刻1：事务A执行
    UPDATE account SET balance=balance-100 WHERE user_id=1001;
    → 获得user_id=1001的Record Lock(X)

  时刻2：事务B执行
    UPDATE account SET balance=balance-50 WHERE user_id=1002;
    → 获得user_id=1002的Record Lock(X)

  时刻3：事务A执行
    UPDATE account SET balance=balance+100 WHERE user_id=1002;
    → 请求user_id=1002的Record Lock(X) → 被B持有 → 等待

  时刻4：事务B执行
    UPDATE account SET balance=balance+50 WHERE user_id=1001;
    → 请求user_id=1001的Record Lock(X) → 被A持有 → 等待

  ━━━ 锁等待环路 ━━━

    事务A ──持有1001锁──→ 等待1002锁 ──→ 事务B
      ↑                                    │
      └──────── 持有1002锁 ──等待1001锁 ───┘

    A等B释放，B等A释放 → 死锁！

  ━━━ 如果user_id是普通索引 ━━━
    除了Record Lock还有Gap Lock
    两个事务可能对同一间隙都加了Gap Lock（Gap Lock不互斥）
    然后都试图插入 → 产生Insert Intention Lock冲突 → 死锁
    这种死锁比主键场景更常见，也更难排查
```

### 3.2 死锁排查SOP

```
  Step 1：查看最近一次死锁信息
    SHOW ENGINE INNODB STATUS;

    关键字段：
    ┌─────────────────┬──────────────────────────────────────────┐
    │  字段             │  看什么                                  │
    ├─────────────────┼──────────────────────────────────────────┤
    │  LATEST DETECTED │  死锁发生时间                            │
    │  DEADLOCK        │                                          │
    ├─────────────────┼──────────────────────────────────────────┤
    │  Transaction 1/2 │  两个事务各自的SQL和锁信息                │
    ├─────────────────┼──────────────────────────────────────────┤
    │  HOLDS THE LOCK  │  各自持有什么锁（类型+模式+索引+行）     │
    ├─────────────────┼──────────────────────────────────────────┤
    │  WAITING FOR     │  各自等待什么锁                          │
    │  THIS LOCK TO BE │                                          │
    │  GRANTED         │                                          │
    ├─────────────────┼──────────────────────────────────────────┤
    │  WE ROLL BACK    │  InnoDB选择了哪个事务回滚（代价小的那个）│
    │  TRANSACTION     │                                          │
    └─────────────────┴──────────────────────────────────────────┘

  Step 2：查看当前锁等待
    SELECT * FROM information_schema.INNODB_LOCK_WAITS;
    -- 请求锁的事务 / 持有锁的事务 / 锁详情

    SELECT * FROM information_schema.INNODB_LOCKS;
    -- 锁的类型/模式/索引/表/行

    -- MySQL 8.0+ 使用 performance_schema
    SELECT * FROM performance_schema.data_lock_waits;
    SELECT * FROM performance_schema.data_locks;

  Step 3：查看长事务
    SELECT trx_id, trx_state, trx_started,
           trx_tables_locked, trx_rows_locked,
           TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration_sec
    FROM information_schema.INNODB_TRX
    ORDER BY trx_started;

  Step 4：分析根因
    常见死锁模式：
    ├─ AB-BA锁交叉（本题场景）→ 按固定顺序加锁
    ├─ Gap Lock + Insert Intention → RC或唯一索引
    ├─ 两个事务更新同一行的不同索引 → 索引优化
    └─ 批量更新无序 → 排序后按顺序更新

  Step 5：确认InnoDB死锁检测是否开启
    SHOW VARIABLES LIKE 'innodb_deadlock_detect';
    -- 默认ON，自动检测死锁并回滚代价小的事务
    -- 高并发场景可能考虑OFF+innodb_lock_wait_timeout控制
    -- 但一般不建议关闭
```

### 3.3 三种解决方案

```
  ━━━ 方案1：固定加锁顺序（最常用）━━━

    转账前按user_id排序，两个事务都先锁小的再锁大的
    事务A：先锁1001再锁1002
    事务B：也先锁1001再锁1002 → B等A，不会死锁

    实现：
      // 转账前排序
      Long[] ids = {fromId, toId};
      Arrays.sort(ids);
      // 按顺序加锁
      accountMapper.selectForUpdate(ids[0]);
      accountMapper.selectForUpdate(ids[1]);

    蛋壳钱包选型理由：简单有效，代码侵入小，覆盖90%场景

  ━━━ 方案2：降隔离级别为RC ━━━

    消除Gap Lock，只加Record Lock
    死锁概率大幅下降（AB-BA场景仍存在，但Gap Lock交叉死锁消失）

    蛋壳钱包选型理由：金融场景需RR保证一致性读，不适用
    但对电商库存扣减等场景推荐

  ━━━ 方案3：乐观锁 + 重试 ━━━

    不用SELECT FOR UPDATE，改用版本号：
      UPDATE account SET balance=balance-100, version=version+1
      WHERE user_id=1001 AND version=#{oldVersion};

    乐观锁没有排它锁等待 → 不会死锁
    冲突时应用层重试（3次+指数退避）

    蛋壳钱包选型理由：金融转账强一致要求高，乐观锁冲突重试
    用户体验差，适合冲突率低的场景

  ━━━ 方案对比 ━━━

  ┌──────────┬──────────┬──────────┬──────────────────────┐
  │  方案     │  死锁风险 │  一致性   │  适用场景             │
  ├──────────┼──────────┼──────────┼──────────────────────┤
  │  固定顺序 │  低       │  强       │  金融转账（蛋壳钱包） │
  │  RC隔离   │  很低     │  较弱     │  电商库存、社交       │
  │  乐观锁   │  无       │  最终     │  冲突率<5%的场景      │
  └──────────┴──────────┴──────────┴──────────────────────┘
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 两个事务互相等锁就死锁了，重启就好 |
| 高级开发 | SHOW ENGINE INNODB STATUS看死锁信息，按顺序加锁解决 |
| 架构师 | 死锁排查是五步SOP（看状态→查锁等待→查长事务→分析模式→确认配置），四种常见死锁模式对应不同解法。AB-BA固定顺序、Gap锁用RC或唯一索引、批量更新排序、乐观锁替代。选型看一致性要求和冲突率：金融强一致用固定顺序+悲观锁，电商用RC+乐观锁，冲突率低直接乐观锁。要能画出锁等待环路图 |

## 题目四：长事务的危害与治理

### 4.1 长事务对五个层面的影响

```
  ┌──────────────┬──────────────────────────────────────────────────┐
  │  层面         │  影响                                            │
  ├──────────────┼──────────────────────────────────────────────────┤
  │  undo log    │  长事务期间所有修改的旧版本都不能清除              │
  │              │  → undo log膨胀 → 磁盘暴涨                       │
  │              │                                                  │
  │              │  为什么暴涨？                                     │
  │              │  MVCC需要undo log构建版本链给其他事务读            │
  │              │  长事务不提交 → 它可能要读任意旧版本               │
  │              │  → 在它之后产生的所有undo log都不能purge           │
  │              │  例：1小时长事务，期间100万次UPDATE               │
  │              │  → 100万条undo log无法清理 → 可能GB级膨胀         │
  ├──────────────┼──────────────────────────────────────────────────┤
  │  redo log    │  长事务修改多 → redo log产生量大                  │
  │              │  → 循环写但checkpoint被长事务阻塞推进慢            │
  │              │  → redo log文件可能撑满                           │
  ├──────────────┼──────────────────────────────────────────────────┤
  │  binlog      │  长事务不提交 → binlog不完整写入                  │
  │              │  → 从库收不到完整事务 → 从库延迟                  │
  │              │  → 主从延迟可能导致读从库读到过期数据              │
  ├──────────────┼──────────────────────────────────────────────────┤
  │  锁          │  锁持有时间 = 事务持续时间                        │
  │              │  → 其他事务等待超时 → 大面积锁超时报错             │
  │              │  → 死锁检测开销随等待链增长                        │
  ├──────────────┼──────────────────────────────────────────────────┤
  │  主从复制    │  长事务提交时产生大binlog → 从库回放慢             │
  │              │  → 从库延迟                                      │
  │              │  → 读写分离场景读从库=过期数据                    │
  └──────────────┴──────────────────────────────────────────────────┘

  磁盘暴涨的根因：
    purge线程清理undo log的前提：没有活跃事务需要读该版本
    长事务不提交 → 它之前的undo log都不能purge → 累积膨胀
    不是长事务自己的修改多，是它"挡住了"其他事务的undo log清理
```

### 4.2 长事务监控

```
  ━━━ 实时监控SQL ━━━

  SELECT
    trx_id,
    trx_state,
    trx_started,
    trx_tables_locked,
    trx_rows_locked,

  -- MySQL 8.0+ 事务统计

    mysql_global_status_innodb_history_list_length   — undo log堆积
```

### 4.3 长事务根治方案

```
  ━━━ 1. 代码规范 ━━━
    ① @Transactional方法内禁止RPC调用（网络超时=事务超时）
    ② @Transactional方法内禁止耗时计算（文件IO/复杂算法）
    ③ 查询和修改尽量分离：先查后改，缩小事务范围
    ④ 事务方法命名规范：以doXxx/updateXxx开头，一目了然

  ━━━ 2. ORM配置 ━━━
    Spring + MyBatis/JPA：
      # 全局超时
      spring.datasource.hikari.connection-timeout=30000
      # 事务超时（配合@Transactional(timeout=30)）
      # MyBatis查询超时
      mybatis.configuration.default-statement-timeout=10

    @Transactional(timeout = 30)  -- 30秒超时回滚

  ━━━ 3. 连接池 ━━━
    HikariCP关键配置：
      maximum-pool-size=20          -- 不是越大越好
      connection-timeout=30000      -- 获取连接超时30s
      max-lifetime=1800000          -- 连接最大存活30分钟
      leak-detection-threshold=60000 -- 连接泄漏检测60s

    连接池耗尽的典型原因：长事务占用连接不释放

  ━━━ 4. 事务边界设计 ━━━
    反模式：大事务包含一切
      @Transactional
      public void createOrder() {
          检查库存;      // 不需要事务
          扣减库存;      // 需要事务
          创建订单;      // 需要事务
          发MQ消息;      // 不需要事务（可能超时）
          发短信通知;    // 不需要事务（可能超时）
      }

    正确：最小化事务范围
      public void createOrder() {
          检查库存;                    // 事务外
          doCreateOrder();             // 事务内：扣库存+创建订单
          发MQ消息;                    // 事务外
          发短信通知;                   // 事务外
      }

      @Transactional(timeout = 10)
      private void doCreateOrder() {
          扣减库存;
          创建订单;
      }

  ━━━ 5. 应用拆分 ━━━
    跨服务事务不要用分布式事务包住整个调用链
    而是用本地事务表+消息最终一致性：
      服务A：本地事务（写业务数据+写消息表）→ 提交
      异步线程：读消息表→发MQ→服务B消费

    这是蛋壳钱包TCC之外更轻量的方案
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 长事务占连接，要加超时 |
| 高级开发 | 长事务会锁住资源，undo log会膨胀 |
| 架构师 | 长事务影响五个层面（undo/redo/binlog/锁/主从），磁盘暴涨根因是undo log purge被阻塞。监控是四指标分级告警（持续时间/锁等待/History list length/行锁等待时间）。根治是五层：代码规范禁RPC→ORM配置超时→连接池防泄漏→事务边界最小化→应用拆分用消息表。核心原则："事务内只做必须原子的事" |

## 题目五：高并发场景下的事务设计

### 5.1 三种方案四维对比

```
  ┌──────────┬────────────┬──────────┬──────────┬──────────┐
  │  维度     │  方案A悲观锁│  方案B乐观锁│  方案C原子扣减│
  ├──────────┼────────────┼──────────┼──────────┼──────────┤
  │  TPS     │  低(100-500)│  中(1000+)│  高(3000+)│
  │          │  锁等待串行  │  无锁但冲突│  无锁行级更新│
  ├──────────┼────────────┼──────────┼──────────┤
  │  一致性   │  强一致     │  强一致    │  强一致    │
  │          │  DB保证     │  DB保证    │  DB保证    │
  ├──────────┼────────────┼──────────┼──────────┤
  │  死锁风险 │  高         │  无       │  低        │
  │          │  锁等待链    │  无锁机制  │  单行更新  │
  ├──────────┼────────────┼──────────┼──────────┤
  │  超卖风险 │  无         │  低       │  无        │
  │          │  FOR UPDATE │  version  │  stock>0  │
  │          │  串行保证    │  校验     │  WHERE条件 │
  └──────────┴────────────┴──────────┴──────────┘

  详细分析：

  方案A 悲观锁：
    优点：简单可靠，不存在超卖
    缺点：FOR UPDATE串行化→TPS低→锁等待超时→用户体验差
    适用：库存少+并发低的场景（如拍卖）

  方案B 乐观锁：
    优点：无锁等待→TPS比悲观锁高
    缺点：冲突时重试→高并发下重试率>80%→实际TPS反而低
         需要额外version字段
    适用：冲突率<5%的场景（如编辑保存）

  方案C 原子扣减：
    优点：单条SQL原子操作→无锁→TPS最高
         stock>0条件天然防超卖
    缺点：扣减失败不知道原因（卖完 vs 其他）
         需要额外流水表做对账
    适用：秒杀/抢购等高并发场景
```

### 5.2 秒杀为什么不用DB方案

```
  ━━━ 为什么不用DB扣库存？━━━

  秒杀核心矛盾：
    10万QPS → 单行库存更新 → InnoDB行锁 → 串行化 → TPS上限≈500

    10万请求进来，99995个在等锁 → 超时 → 用户看到"系统繁忙"
    实际只有5个能买到，但10万都打到DB → DB被打挂

  ━━━ Redis预扣减 + MQ异步落库 ━━━

  完整链路：

  ┌─────┐    ┌──────┐    ┌─────────┐    ┌──────────┐
  │ 用户 │───→│ 网关  │───→│ 秒杀服务 │───→│ Redis    │
  │     │    │ 限流  │    │         │    │ Lua扣减  │
  └─────┘    └──────┘    └────┬────┘    └────┬─────┘
                              │               │
                              │  扣减成功      │ stock-1
                              │               │ soldout标记
                              ▼               │
                        ┌──────────┐         │
                        │ 发MQ消息  │         │
                        │ 创建订单  │         │
                        └────┬─────┘         │
                             │               │
                             ▼               │
                        ┌──────────┐         │
                        │ 订单服务  │         │
                        │ 异步落库  │         │
                        └──────────┘         │

  Step 1：网关令牌桶限流（1万QPS → 过滤90%流量）
  Step 2：Redis Lua脚本原子扣减
    if redis.call('get', KEYS[1]) > 0 then
        redis.call('decr', KEYS[1])
        return 1  -- 扣减成功
    else
        redis.call('set', 'soldout:'..KEYS[1], 1, 'EX', 3600)
        return 0  -- 已售罄
    end
  Step 3：扣减成功 → 发MQ → 订单服务异步创建订单+扣DB库存
  Step 4：售罄标记 → 后续请求直接返回，零DB压力

  关键设计：
    - Lua脚本保证原子性（GET+DECR不被中断）
    - 售罄标记比限流更彻底（卖完后DB写入量归零）
    - MQ异步解耦（秒杀服务不依赖DB，不依赖订单服务）
```

### 5.3 Redis预扣 + DB兜底校验架构

```
  ━━━ 架构设计 ━━━

  核心原则：Redis管"快"，DB管"准"

  ┌─────────────────────────────────────────────────────────┐
  │                    写链路（扣减）                         │
  │                                                         │
  │  Redis Lua扣减 ──成功──→ MQ ──→ 订单服务 ──→ DB扣减    │
  │       │                                      │          │
  │       │                               DB扣减失败？       │
  │       │                                  │              │
  │       │                          stock≤0 → 回补Redis    │
  │       │                          重复扣减 → 幂等跳过     │
  │       │                          DB异常  → 重试3次      │
  └───────┴──────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │                    对账方案                              │
  │                                                         │
  │  定时任务（每5分钟）：                                   │
  │                                                         │
  │  Step 1：Redis查当前库存 R_stock                        │
  │  Step 2：DB查当前库存 D_stock                           │
  │  Step 3：DB查已创建订单数 D_orders                      │
  │  Step 4：校验 D_stock + D_orders = 总库存？             │
  │          R_stock = D_stock？                            │
  │                                                         │
  │  不一致处理：                                            │
  │    R_stock > D_stock → Redis多扣了（少见）              │
  │      → 以DB为准，修正Redis                              │
  │    R_stock < D_stock → DB扣多了（常见：重复扣减）        │
  │      → 查stock_log流水找出重复项                        │
  │      → 补回Redis                                       │
  │    D_stock + D_orders ≠ 总库存 → 有丢单                │
  │      → 查MQ消费记录找出未消费消息                       │
  │      → 重新投递                                        │
  │                                                         │
  │  对账表设计：                                            │
  │    reconciliation_log(                                  │
  │      id, activity_id,                                   │
  │      redis_stock, db_stock, db_orders,                  │
  │      total_expected, diff,                              │
  │      status,  -- PENDING/RESOLVED                       │
  │      created_at, resolved_at                            │
  │    )                                                    │
  └─────────────────────────────────────────────────────────┘

  ━━━ DB层兜底SQL ━━━

  -- 库存扣减（原子+防超卖）
  UPDATE goods SET stock = stock - #{count}
  WHERE id = #{goodsId} AND stock >= #{count};

  -- 影响行数=0 → 卖完了 → 回补Redis（设soldout标记）
  -- 影响行数=1 → 扣减成功

  -- 幂等校验（流水表）
  INSERT INTO stock_log (order_id, goods_id, count, status)
  VALUES (#{orderId}, #{goodsId}, #{count}, 'DEDUCTED');
  -- order_id唯一索引，重复插入失败 → 幂等

  ━━━ DBA要求"DB层保证"的架构师回答 ━━━

  "Redis是性能优化，DB是一致性兜底。两条线：
   1. 正常链路：Redis预扣→MQ→DB扣减，Redis管快DB管准
   2. 对账链路：5分钟一次，Redis库存=DB库存=总库存-已售数
   不一致以DB为准修正Redis。这样即使Redis全挂了，
   DB的stock>=0条件也能保证不超卖——只是会慢。"
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 用SELECT FOR UPDATE防超卖，或者用Redis扣库存 |
| 高级开发 | 秒杀用Redis+Lua防超卖，MQ异步落库 |
| 架构师 | 秒杀不用DB扣库存因为行锁串行化TPS上限500扛不住10万QPS。Redis预扣+MQ异步落库是标准方案，售罄标记比限流更彻底。DB兜底校验是"Redis管快DB管准"两条线：正常链路+5分钟对账链路。对账方案要覆盖三种不一致（Redis多扣/DB多扣/丢单），以DB为准。面试一句话："一致性在DB保证，性能在Redis保证，对账是桥梁" |
