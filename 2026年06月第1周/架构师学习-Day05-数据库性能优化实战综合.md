### 1.1 MySQL高可用的演进路径

```
MySQL高可用架构演进：

  ┌──────────────────────────────────────────────────────────────────────────┐
  │                                                                          │
  │  阶段1：主从复制 + VIP漂移                                               │
  │  ─────────────────────────                                               │
  │  App → VIP → Master                                                      │
  │                ├─ Slave1                                                  │
  │                └─ Slave2                                                  │
  │  故障：手动切换VIP，MTA 30分钟+                                          │
  │  数据风险：异步复制，主库宕机丢数据                                       │
  │                                                                          │
  │  阶段2：半同步复制 + MHA                                                 │
  │  ─────────────────────────                                               │
  │  App → VIP → Master                                                      │
  │                ├─ Slave1（半同步）                                        │
  │                └─ Slave2（异步）                                          │
  │  MHA Manager：监控主库，自动failover，提升最新从库                        │
  │  故障：自动切换，MTA 30秒-2分钟                                           │
  │  数据风险：半同步至少1个从库收到binlog，丢数据概率降低                     │
  │                                                                          │
  │  阶段3：MGR（MySQL Group Replication）                                   │
  │  ─────────────────────────                                               │
  │  App → MySQL Router → MGR集群                                            │
  │         Primary ↔ Secondary1 ↔ Secondary2                                │
  │  Paxos协议保证多数派写入，自动选主                                        │
  │  故障：自动切换，MTA 10-30秒                                              │
  │  数据风险：几乎不丢数据（多数派确认）                                      │
  │                                                                          │
  │  阶段4：InnoDB Cluster = MGR + MySQL Router + MySQL Shell                │
  │  ─────────────────────────                                               │
  │  App → MySQL Router（读写分离）→ MGR集群                                  │
  │  MySQL Shell：管理端，一键建集群/加节点/滚升级                             │
  │  故障：自动切换+路由自动感知，MTA < 10秒                                  │
  │  数据风险：几乎不丢数据                                                   │
  └──────────────────────────────────────────────────────────────────────────┘
```

### 1.3 选型决策树

```
数据库高可用选型决策树：

  RPO（数据丢失容忍度）是多少？
  ├── RPO=0（一帧都不能丢，金融核心）
  │   └── MGR + 半同步双保险 / InnoDB Cluster
  │       └── 团队有DBA + 能接受写性能下降15-30%？ → ✅ 用MGR
  │       └── 团队无专职DBA → 考虑云RDS高可用版（托管MGR）
  │
  ├── RPO≈0（最多丢极少量，电商核心）
  │   └── 半同步复制 + MHA
  │       └── 现有架构已经是主从？ → ✅ 加MHA成本最低
  │       └── 新项目 + MySQL 8.0？ → 考虑InnoDB Cluster
  │
  └── RPO>0（可接受少量丢失，非核心业务）
      └── 异步复制 + VIP漂移 / MHA
          └── 数据量小 + 不想折腾 → 异步复制 + VIP
          └── 想要自动切换 → 异步复制 + MHA

  关键约束：
    1. MGR要求网络延迟<10ms，跨机房部署要评估
    2. MGR多主模式有冲突检测开销，写冲突高的场景不适用
    3. MHA已停止维护（最后版本0.58），新项目优先MGR
```

### 1.4 结合你的业务场景选型

```
场景1：蛋壳钱包（金融转账）
  要求：RPO=0，一帧不能丢
  选型：InnoDB Cluster（单主模式）
  理由：
    - 转账场景数据一致性最高优先级
    - MGR多数派确认写入，主库宕机不丢数据
    - MySQL Router自动路由，应用无感知切换
    - 代价：写性能下降约20%，但金融场景可接受

场景2：秒杀系统
  要求：写QPS极高（10万+），允许极短暂不可用
  选型：主从复制 + MHA（读写分离架构）
  理由：
    - 秒杀核心压力在Redis，DB只做最终落盘
    - 写QPS需求高，MGR的多数派确认成为瓶颈
    - MHA自动切换MTA 30秒，秒杀场景可接受（售罄后降级）
    - 如果用MGR，写性能降20-30%，反而拖慢下单

场景3：啄木鸟体检系统
  要求：读多写少（体检报告查询），数据不能丢
  选型：半同步复制 + MHA（一主两从）
  理由：
    - 读多写少，读写分离效果好
    - 体检数据不能丢（医疗数据法规）
    - 半同步保证至少1从库收到binlog
    - MHA比MHA运维简单，团队没有专职DBA

场景4：IM在线客服
  要求：高可用（99.99%），消息不能丢
  选型：InnoDB Cluster
  理由：
    - 消息不能丢是硬约束
    - 需要秒级切换保证客服不中断
    - MySQL Router + MGR原生支持
    - 写量不大（聊天消息），MGR性能影响可忽略
```

```
     但换来的是不丢数据+秒级自动切换。金融场景必选，高写场景慎选。"
```

### 2.2 慢SQL分析SOP

```
收到一条慢SQL，五步排查：

  Step 1：看EXPLAIN，确认扫描行数和访问类型
    ┌──────────────────────────────────────────────────┐
    │  type列（从好到差）：                              │
    │  system > const > eq_ref > ref > range           │
    │  > index > ALL                                   │
    │                                                  │
    │  危险信号：                                       │
    │  - type=ALL + rows>10000 → 全表扫描              │
    │  - type=index + rows>10000 → 全索引扫描          │
    │  - Extra=Using filesort → 额外排序               │
    │  - Extra=Using temporary → 临时表                │
    │  - key=NULL → 没走索引                           │
    └──────────────────────────────────────────────────┘

  Step 2：看possible_keys vs key，判断索引选择
    - possible_keys有但key=NULL → 优化器没选，可能成本评估有误
    - possible_keys和key不同 → 看是否选错索引（FORCE INDEX验证）

  Step 3：看rows × filtered，估算实际扫描量
    - rows是估计值，filtered是过滤比例
    - 实际扫描 ≈ rows × filtered%
    - rows=10万 + filtered=1% → 实际只返回1000行但扫描10万 → 加条件索引

  Step 4：看Extra，确认额外操作
    - Using index → 覆盖索引，好
    - Using where → 回表过滤
    - Using filesort → 需要排序（检查ORDER BY是否走索引）
    - Using temporary → 需要临时表（GROUP BY优化）

  Step 5：验证优化效果
    - 优化前后对比EXPLAIN
    - 对比实际执行时间（EXPLAIN ANALYZE）
    - 确认优化没有引入新的问题（索引维护开销、锁范围变化）
```

### 2.3 常见慢SQL模式与解法

```
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  模式                     │  问题               │  解法                 │
  ├───────────────────────────┼─────────────────────┼──────────────────────┤
  │  SELECT * FROM t          │  全表扫描+回表       │  只查需要的列         │
  │                           │                     │  加覆盖索引           │
  ├───────────────────────────┼─────────────────────┼──────────────────────┤
  │  WHERE a=1 ORDER BY b     │  a走索引但b要排序    │  联合索引(a,b)       │
  │                           │  Using filesort     │  让ORDER BY走索引     │
  ├───────────────────────────┼─────────────────────┼──────────────────────┤
  │  WHERE a>100 ORDER BY b   │  范围查询后排序失效  │  联合索引(b,a)       │
  │                           │                     │  或ES异构索引         │
  ├───────────────────────────┼─────────────────────┼──────────────────────┤
  │  LIMIT 100000, 10         │  深分页扫描10万行    │  游标分页             │
  │                           │                     │  WHERE id>last_id     │
  ├───────────────────────────┼─────────────────────┼──────────────────────┤
  │  IN (子查询)               │  可能走全表          │  改JOIN/EXISTS       │
  ├───────────────────────────┼─────────────────────┼──────────────────────┤
  │  COUNT(*) 全表             │  InnoDB必须遍历      │  Redis计数/汇总表    │
  │                           │                     │  或MyISAM计数表       │
  ├───────────────────────────┼─────────────────────┼──────────────────────┤
  │  GROUP BY无索引            │  Using temporary    │  加联合索引           │
  │                           │                     │  或先限缩再GROUP BY   │
  ├───────────────────────────┼─────────────────────┼──────────────────────┤
  │  函数索引列 WHERE YEAR(d)  │  索引失效           │  改范围查询           │
  │                           │                     │  d >= '2026-01-01'   │
  ├───────────────────────────┼─────────────────────┼──────────────────────┤
  │  隐式类型转换              │  索引失效           │  保持类型一致         │
  │  WHERE varchar_col = 123  │                     │  WHERE col = '123'   │
  └───────────────────────────┴─────────────────────┴──────────────────────┘
```

### 2.4 结合你的业务：秒杀系统慢SQL排查

```
场景：秒杀活动开始后，订单创建接口P99从50ms飙升到3s

  Step 1：定位慢SQL
    - SkyWalking看到 order-service 的 createOrder 方法P99=3s
    - slow_query_log中找到：INSERT INTO order_table (...) VALUES (...)
    - 同时发现：SELECT stock FROM seckill_stock WHERE goods_id=? FOR UPDATE 等待时间2.8s

  Step 2：分析根因
    EXPLAIN SELECT stock FROM seckill_stock WHERE goods_id=? FOR UPDATE
    → type=ref, key=idx_goods_id, rows=1 → 索引没问题
    → SHOW ENGINE INNODB STATUS → 大量锁等待
    → 根因：FOR UPDATE行锁串行化，热点商品行锁竞争导致排队

  Step 3：优化方案
    - 短期：Redis Lua预扣库存，DB异步落盘（你已有经验）
    - 中期：库存行拆分（单商品拆成10行，随机扣减后汇总）
    - 长期：库存分片到不同库（按商品ID哈希）

  Step 4：预防
    - SQL审核：禁止秒杀链路使用FOR UPDATE
    - 告警：行锁等待>500ms立即告警
    - 限流：Redis预扣库存层限流，保护DB
```

## 题目三：大表在线变更 — pt-osc/gh-ost怎么选/大表DDL方案

### 3.1 为什么不能直接ALTER TABLE

```
大表直接ALTER TABLE的问题：

  ┌──────────────────────────────────────────────────────────────────┐
  │  MySQL 5.6之前：                                                 │
  │    ALTER TABLE → 表级锁 → 整个表不可读写 → 业务中断              │
  │                                                                  │
  │  MySQL 5.6+ Online DDL：                                        │
  │    ALGORITHM=INPLACE → 避免拷表，但仍有限制：                    │
  │    ✅ 即时操作（不锁表）：                                        │
  │      - 加列（末尾）                                               │
  │      - 删索引                                                    │
  │      - 改默认值                                                  │
  │    ⚠️ 需要重建表（短时间锁表）：                                  │
  │      - 加索引（开始和结束短暂锁表）                                │
  │      - 改列类型                                                  │
  │    ❌ 必须拷表（长时间锁表）：                                    │
  │      - 改主键                                                    │
  │      - 改字符集                                                  │
  │      - 改行格式                                                  │
  │                                                                  │
  │  Online DDL的真实问题：                                          │
  │    1. 重建表期间占用双倍磁盘（原表+新表）                        │
  │    2. binlog暴涨（全表行级变更写入binlog）                       │
  │    3. 主从延迟（从库重放全表变更）                                │
  │    4. 业务高峰期执行 → 从库延迟→读写分离读到旧数据              │
  │    5. 长事务或大查询持锁 → Online DDL被阻塞 → 反过来阻塞业务    │
  └──────────────────────────────────────────────────────────────────┘
```

### 3.2 pt-online-schema-change vs gh-ost

```
  ┌──────────────────────────────────────────────────────────────────────┐
  │  维度              │  pt-osc                        │  gh-ost         │
  ├────────────────────┼────────────────────────────────┼────────────────┤
  │  原理              │  创建新表+触发器同步增量        │  创建新表+binlog同步增量│
  │  增量数据同步      │  AFTER INSERT/UPDATE/DELETE触发器│ 解析binlog row格式│
  │  触发器开销        │  ⚠️ 有（原表每个DML都触发）    │  ✅ 无          │
  │  对主库性能影响    │  中（触发器开销+锁）            │  低（只读binlog）│
  │  主库负载          │  高                             │  低             │
  │  锁                │  需要短暂锁表建触发器           │  无触发器无额外锁│
  │  binlog格式要求    │  无                             │  必须ROW格式    │
  │  对大事务/长SQL    │  触发器可能延迟/丢失           │  binlog流式处理  │
  │  回滚              │  删触发器+删新表               │  删新表         │
  │  并发写入量大时    │  触发器跟不上→数据不一致风险   │  binlog更稳定   │
  │  使用门槛          │  低（Percona Toolkit全家桶）   │  中（需理解binlog│
  │                    │                                │  解析机制）     │
  │  开发方             │  Percona                       │  GitHub         │
  │  维护状态          │  ✅ 活跃                       │  ✅ 活跃        │
  └────────────────────┴────────────────────────────────┴────────────────┘
```

### 3.3 选型决策

```
大表在线变更选型决策树：

  并发写入量级？
  ├── 低（< 1000 TPS）→ pt-osc够了
  │   └── 简单场景，触发器开销可接受
  │
  ├── 中（1000-5000 TPS）→ 看业务敏感度
  │   ├── 能接受短暂性能波动 → pt-osc
  │   └── 不能接受 → gh-ost
  │
  └── 高（> 5000 TPS）→ gh-ost
      └── 触发器开销太大，必须用binlog方案

  补充判断：
    - binlog格式不是ROW？ → 只能用pt-osc
    - 需要暂停/恢复变更？ → gh-ost支持throttle（按负载动态调速）
    - 变更期间主库CPU>80%？ → gh-ost（无触发器开销）
    - 团队不熟悉gh-ost？ → pt-osc上手快，但高并发场景要压测

  gh-ost的杀手级特性——动态调速（throttle）：
    --max-load=Threads_running=25    # 超过25个活跃线程自动暂停
    --critical-load=Threads_running=50 # 超过50个立刻停止
    --chunk-size=1000                 # 每批拷贝行数
    --max-lag-millis=1500             # 从库延迟>1.5秒自动暂停
```

### 3.4 大表变更实操SOP

```
大表DDL五步SOP：

  Step 1：评估影响
    ├─ 表大小：SELECT DATA_LENGTH+INDEX_LENGTH FROM information_schema.TABLES
    ├─ 当前QPS：SHOW GLOBAL STATUS LIKE 'Questions'
    ├─ 预估时间：表大小 / 拷贝速度（约2GB/分钟）
    └─ 从库延迟容忍度：业务能接受多少秒延迟

  Step 2：选择工具+时间窗口
    ├─ 根据TPS和表大小选pt-osc或gh-ost
    ├─ 时间窗口：低峰期（22:00-06:00）
    └─ 如果预估>4小时 → 分多个窗口（gh-ost支持暂停续做）

  Step 3：试运行（影子表验证）
    ├─ 先在测试环境用相同数据量跑一遍
    ├─ 确认binlog暴涨量、从库延迟、CPU/IO影响
    └─ 在从库上先做一次，观察主从延迟

  Step 4：正式执行
    ├─ 开启监控：主库CPU/IO/连接数 + 从库延迟
    ├─ 执行变更（gh-ost加throttle参数）
    ├─ 观察从库延迟：< 5秒正常，> 30秒暂停
    └─ 执行完binlog回放后验证数据一致性

  Step 5：验证+收尾
    ├─ CHECKSUM TABLE 对比新旧表数据
    ├─ 确认应用连接切换到新表
    ├─ 旧表保留24小时（回滚保险）
    └─ 清理旧表+binlog

  数字直觉：
    10GB表变更 ≈ 5分钟
    100GB表变更 ≈ 50分钟
    1TB表变更 ≈ 8小时+
    gh-ost拷贝速度 ≈ 2GB/分钟（取决于磁盘IO）
```

## 题目四：数据库容量规划与架构演进 — 从单库到分库分表的完整路径

### 4.1 数据库容量规划四步法

```
容量规划四步法：基线→预估→计算→扩容

  Step 1：建立基线
    ┌──────────────────────────────────────────────────────────┐
    │  指标              │  如何获取           │  典型值         │
    ├────────────────────┼─────────────────────┼────────────────┤
    │  单机读QPS         │  sysbench压测       │  8000-10000    │
    │  单机写QPS         │  sysbench压测       │  3000-5000     │
    │  单表行数上限      │  B+树3层限制        │  ≈2000万       │
    │  单表数据文件上限   │  DDL/备份耗时       │  ≈50GB         │
    │  单实例连接数上限   │  max_connections    │  3000-5000     │
    │  单实例磁盘上限    │  RAID10+SSD         │  2-4TB         │
    └────────────────────┴─────────────────────┴────────────────┘

  Step 2：业务预估
    当前数据量 + 增长率 → 预估N个月后数据量

    例子：电商订单表
      当前：1亿行 / 100GB
      日增：500万行 / 0.5GB
      年增：18亿行 / 180GB
      1年后：19亿行 / 280GB → 超出单表限制

  Step 3：计算所需资源
    所需库数 = 预估QPS / 单库QPS
    所需表数 = 预估数据量 / 单表上限

    例子：
      预估读QPS = 50000 → 需 50000/8000 ≈ 7个读库
      预估写QPS = 15000 → 需 15000/3000 ≈ 5个写库
      预估数据量 = 19亿行 → 需 19亿/2000万 ≈ 100张表
      分4个库 × 25张表/库 = 100张表

    安全系数：×1.5（留50%余量应对突发）
      最终：4库 × 32表 = 128张表

  Step 4：扩容预案
    ├─ 短期（6个月内）：4库 × 32表 + 读写分离
    ├─ 中期（1年）：8库 × 32表（扩库不扩表）
    └─ 长期（2年+）：16库 × 32表 + 冷热分离（归档历史数据）
```

### 4.2 架构演进三阶段决策框架

```
数据库架构演进三阶段：

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  阶段1：单库优化（QPS < 5000）                                     │
  │  ─────────────────────────                                          │
  │  优化手段：                                                         │
  │    ├─ 索引优化 + SQL调优                                            │
  │    ├─ 参数调优（innodb_buffer_pool_size等）                         │
  │    ├─ 引入缓存（Redis）                                             │
  │    └─ 连接池优化（Druid/HikariCP）                                  │
  │  判断条件：单库优化后QPS仍>5000 → 进入阶段2                        │
  │                                                                     │
  │  阶段2：读写分离 + 缓存（读QPS 5000-30000）                       │
  │  ─────────────────────────                                          │
  │  优化手段：                                                         │
  │    ├─ 一主多从读写分离                                              │
  │    ├─ 缓存层（Redis + 本地缓存）                                   │
  │    ├─ 半同步复制保证数据安全                                        │
  │    └─ 强制读主解决一致性问题                                        │
  │  判断条件：写QPS>5000 或 数据量>2000万行 → 进入阶段3               │
  │                                                                     │
  │  阶段3：分库分表（写QPS>5000 或 数据量>2000万）                    │
  │  ─────────────────────────                                          │
  │  优化手段：                                                         │
  │    ├─ 垂直分库（按业务域拆分）                                      │
  │    ├─ 水平分表（按分片键哈希/范围）                                 │
  │    ├─ 分库分表中间件（ShardingSphere）                              │
  │    └─ 分布式ID（Snowflake/Leaf）                                   │
  │  判断条件：单个业务域数据量>5000万 → 冷热分离+归档                 │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘

  四大演进反模式（千万不要跳步）：
    ❌ 反模式1：QPS才2000就分库分表（运维成本暴增，没有收益）
    ❌ 反模式2：读写分离解决写瓶颈（读分离不解决写压力）
    ❌ 反模式3：缓存替代分库分表（缓存解决读不解决写+存储）
    ❌ 反模式4：上来就微服务拆DB（没有流量和团队规模支撑）
```

### 4.3 冷热分离与数据归档

```
数据归档策略：

  ┌──────────────────────────────────────────────────────────────────┐
  │  热数据         │  温数据             │  冷数据                    │
  │  (近3个月)      │  (3-12个月)         │  (1年以上)                 │
  ├─────────────────┼─────────────────────┼────────────────────────────┤
  │  MySQL主库      │  MySQL从库/归档库   │  HBase/Hive/OSS           │
  │  + Redis缓存    │  + 压缩存储         │  + Parquet列式存储        │
  ├─────────────────┼─────────────────────┼────────────────────────────┤
  │  高频访问       │  低频查询           │  报表/审计/合规            │
  │  秒级响应       │  秒-分钟级          │  分钟-小时级               │
  ├─────────────────┼─────────────────────┼────────────────────────────┤
  │  全字段索引     │  核心字段索引       │  只保留查询字段            │
  │  SSD            │  SSD/HDD            │  HDD/OSS                  │
  └─────────────────┴─────────────────────┴────────────────────────────┘

  归档方法：
    1. pt-archiver：逐行迁移+删除原表数据，锁影响小
    2. 分区表：按时间分区，ALTER TABLE DROP PARTITION秒级删
    3. 分库分表后：按表粒度归档（如按月分表，直接迁移整张历史表）

  归档的关键约束：
    - 归档期间不能影响在线查询（异步执行+限速）
    - 归档后查询要透明（中间件路由到归档库）
    - 归档数据必须可恢复（保留原始binlog）
    - 合规要求：金融数据保留5年+，医疗数据保留15年+
```

### 4.4 结合你的业务：蛋壳钱包数据库架构演进

```
蛋壳钱包架构演进回顾与反思：

  初期（验证阶段）：
    单库 → 转账+账户+流水在同一个库
    QPS < 1000，单库足够

  成长期：
    QPS到3000 → 读写分离（一主两从）
    转账走主库（强一致），查询走从库

  你用的TCC分布式事务 → 意味着到了阶段3：
    账户库、流水库、风控库已经是垂直拆分
    每个库是否需要水平拆分？
      - 账户库：按user_id哈希分4库×8表
      - 流水库：按月分表（天然时间维度，方便归档）
      - 风控库：数据量不大，暂不分

  如果重新设计，你会怎么做？
    1. 提前规划分片键：user_id做基因法，保证账户和流水同分片
    2. 流水表按月分区：ALTER TABLE DROP PARTITION秒级归档
    3. 冷热分离：3个月以上流水迁移到归档库
    4. 高可用：InnoDB Cluster（金融级必须RPO=0）

  架构师反思：
    "当时TCC解决的是分布式事务问题，但分库分表的分片键设计
     和归档策略没有提前规划，导致后期流水查询跨分片成为痛点。
     如果提前用基因法，流水和账户同分片，转账流水查询免跨库。"
```

## 题目五：数据库故障排查实战 — 连接数打满/主从延迟/磁盘暴涨

### 5.1 三大经典数据库故障排查SOP

```
故障1：连接数打满 — "Too many connections"

  排查SOP：
    1. SHOW PROCESSLIST → 看连接状态分布
       - 大量Sleep → 连接泄漏（没close）
       - 大量Query → 慢SQL占用连接
       - 大量Locked → 锁等待

    2. 分类处理：
       ├─ Sleep过多 → 检查连接池配置（maxIdle/minIdle/timeout）
       ├─ 慢SQL → Kill慢SQL + 紧急加索引/限流
       └─ 锁等待 → Kill阻塞源头事务

    3. 紧急处置：
       - 临时调大 max_connections（治标）
       - Kill不必要连接：CALL mysql.rds_kill(thread_id)
       - 限流上游应用

    4. 根治：
       - 连接池：Druid maxActive=50, maxWait=3000ms
       - 超时：wait_timeout=600（10分钟空闲断开）
       - 监控告警：连接数>80% max_connections → P2告警

  数字直觉：
    单实例默认 max_connections = 151 → 生产调到 3000-5000
    有效并发 ≈ max_connections × 0.7（留buffer）
    微服务50实例 × 连接池50 = 2500 → 接近上限


故障2：主从延迟飙升 — Seconds_Behind_Master持续增长

  排查SOP：
    1. SHOW SLAVE STATUS → 确认延迟秒数
       - Seconds_Behind_Master持续增长 → 从库追不上

    2. 定位瓶颈：
       ├─ 从库IO线程慢 → 网络带宽/主库binlog生成速度
       │   SHOW PROCESSLIST → "system user"的IO线程状态
       │
       └─ 从库SQL线程慢 → 单线程重放是瓶颈
           - 大事务 → 一个事务重放很久
           - DDL → ALTER TABLE重建表
           - 从库自身负载高（被读查询占用资源）

    3. 紧急处置：
       - 业务侧：延迟>30秒的读请求切回主库
       - 从库：Kill占资源的慢查询
       - 临时：停止从库的非必要读流量

    4. 根治：
       ├─ 多线程复制：slave_parallel_workers=4-8
       │  (MySQL 5.7+ DATABASE模式 / 8.0 WRITESET模式更优)
       ├─ 大事务拆分：单事务<1000行
       ├─ DDL走gh-ost：避免从库重放全表变更
       └─ 从库升级：SSD + 更大内存

  数字直觉：
    延迟 < 1秒 → 正常
    延迟 1-30秒 → 关注，检查大事务/DDL
    延迟 > 30秒 → 告警，读写分离可能读到旧数据
    延迟 > 300秒 → 紧急，考虑重建从库


故障3：磁盘空间暴涨

  排查SOP：
    1. 定位膨胀来源：
       SELECT file_name, ROUND(data_length/1024/1024) AS MB
       FROM information_schema.FILES ORDER BY data_length DESC;

       ┌────────────────────────────────────────────────────────┐
       │  膨胀源           │  原因               │  排查命令    │
       ├───────────────────┼─────────────────────┼──────────────┤
       │  数据文件(.ibd)   │  数据增长/碎片      │ TABLES表     │
       │  undo log         │  长事务阻塞purge    │ SHOW ENGINE  │
       │  binlog           │  没有自动清理        │ SHOW BINARY  │
       │  redo log         │  一般不会暴涨       │ SHOW ENGINE  │
       │  临时文件         │  大排序/大GROUP BY  │ 慢查询日志   │
       └───────────────────┴─────────────────────┴──────────────┘

    2. 常见根因处理：
       ├─ undo暴涨 → 找长事务：SELECT * FROM information_schema.INNODB_TRX
       │   Kill长事务 → purge线程自动回收undo
       │
       ├─ binlog暴涨 → 检查过期清理：expire_logs_days（5.7）/binlog_expire_logs_seconds（8.0）
       │   手动清理：PURGE BINARY LOGS BEFORE '2026-06-04'
       │
       └─ 数据碎片 → OPTIMIZE TABLE（会锁表，用pt-osc）
           或 ALTER TABLE t ENGINE=InnoDB（重建表回收碎片）

  数字直觉：
    undo log正常：< 1GB
    undo log告警：> 5GB（有长事务）
    undo log紧急：> 50GB（长事务+高写入）
    binlog保留：7天（生产建议）
    磁盘使用率：> 80% 告警
```

### 5.2 故障排查通用框架

```
数据库故障排查四步法（通用框架）：

  ① 现象确认（What）
    - 什么指标异常？延迟/连接数/磁盘/CPU
    - 什么时候开始的？有没有上线/活动
    - 影响范围？全部还是部分

  ② 根因定位（Why）
    - SHOW PROCESSLIST → 当前在做什么
    - SHOW ENGINE INNODB STATUS → 锁/事务/缓冲池
    - 慢查询日志 → 哪条SQL
    - 系统指标 → CPU/IO/网络

  ③ 止血优先（Stop the bleeding）
    - Kill阻塞源头（慢SQL/长事务）
    - 限流降级（保护DB不被打挂）
    - 切流量（读写分离/主库接管）

  ④ 根治防复（Fix & Prevent）
    - 修复根因（加索引/拆大事务/调参数）
    - 加监控告警（提前发现）
    - 加防护（限流/熔断/SQL审核）

  关键原则：止血 > 根治 > 优化
    不要在故障现场调优——先止血，再复盘，最后根治
```
