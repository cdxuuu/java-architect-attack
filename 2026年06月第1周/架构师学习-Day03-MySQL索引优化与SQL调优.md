# Day 3：MySQL索引优化与SQL调优 — B+树原理/索引选型/EXPLAIN实战/慢查询治理（练习整理）

## 题目一：MySQL为什么选B+树做索引 — 索引结构对比与选型

### 为什么不用其他数据结构

```
索引本质：用空间换时间，减少磁盘IO次数

  ┌──────────────────────────────────────────────────────────┐
  │  候选结构    │  磁盘IO   │  范围查询  │  结论             │
  ├──────────────┼───────────┼────────────┼───────────────────┤
  │  哈希表      │  O(1)     │  不支持     │  只适合等值查询   │
  │  二叉搜索树  │  O(logN)  │  支持       │  树太高，IO太多   │
  │  红黑树      │  O(logN)  │  支持       │  还是太高         │
  │  B树         │  O(logN)  │  支持       │  数据在非叶节点   │
  │  B+树        │  O(logN)  │  最优       │  ✅ 数据库首选    │
  └──────────────┴───────────┴────────────┴───────────────────┘

  为什么B+树胜出？三个核心优势：

  优势1：树更矮 → IO更少
    B树节点存数据 → 每个节点能存的关键字少 → 树更高
    B+树非叶节点只存关键字 → 每个节点存更多关键字 → 树更矮

    量化（InnoDB页大小16KB）：
      假设主键BIGINT(8字节)+指针(6字节)=14字节/条目
      非叶节点可存：16KB/14B ≈ 1170个指针
      假设叶子节点每行1KB：
        2层B+树：1170 × 16 = 18,720行
        3层B+树：1170 × 1170 × 16 = 21,902,400行 ≈ 2190万行
        4层B+树：1170³ × 16 = 256亿行

    结论：3层B+树可存2000万+行，3次IO就能找到任何一行

  优势2：范围查询最优
    B树：中序遍历（跨层跳跃，随机IO）
    B+树：叶子节点双向链表 → 范围查询只需找到起点，顺序扫描
    例：WHERE id BETWEEN 100 AND 500
      B+树：定位到100 → 沿链表扫到500 → 顺序IO
      B树：可能需要多次回溯父节点 → 随机IO

  优势3：查询性能稳定
    B树：数据在非叶节点也可能命中 → 不同查询IO次数不同
    B+树：所有数据都在叶子节点 → 每次查询都要走到叶子 → IO次数恒定
```

### InnoDB的B+树实现细节

```
InnoDB索引组织表：数据即索引，索引即数据

  聚簇索引（主键索引）：
    叶子节点存完整行数据 → 一棵B+树存全表
    没有显式主键时InnoDB选择策略：
      1. 有主键 → 用主键
      2. 有非空唯一索引 → 用第一个非空唯一索引
      3. 都没有 → 自动生成6字节ROW_ID（隐藏列）

  二级索引（非主键索引）：
    叶子节点存主键值 → 不存完整行数据
    查询流程：
      ① 二级索引查到主键值
      ② 回表：用主键值查聚簇索引拿到完整行
      回表代价：每行多一次B+树查找

    覆盖索引：查询列全在二级索引中 → 不需要回表
      例：SELECT name FROM user WHERE age=25
      如果有索引(age, name) → 索引中已有name → 不回表

  页结构（16KB）：
    File Header(38B) + Page Header(56B) + Infimum/Supremum(26B)
    + User Records + Free Space + Page Directory + File Trailer(8B)

    页内查找：二分查找（Page Directory槽位）
    页间查找：B+树层级查找
```

| 高级开发 | B+树比B树矮，支持范围查询，叶子节点有链表 |

## 题目二：索引优化实战 — 最左前缀/覆盖索引/索引下推

### 最左前缀原则

```
联合索引(A, B, C)的匹配规则：

  ┌─────────────────────────────────────────────────────────┐
  │  查询条件            │  索引使用情况    │  原因          │
  ├──────────────────────┼──────────────────┼────────────────┤
  │  WHERE A=1           │  ✅ 用A          │  最左前缀      │
  │  WHERE A=1 AND B=2   │  ✅ 用A+B       │  最左连续      │
  │  WHERE A=1 AND B=2   │  ✅ 用A+B+C     │  最左连续      │
  │       AND C=3        │                  │                │
  │  WHERE B=2           │  ❌ 不走索引     │  跳过A         │
  │  WHERE B=2 AND C=3   │  ❌ 不走索引     │  跳过A         │
  │  WHERE A=1 AND C=3   │  ✅ 只用A       │  B缺失，C不可用│
  └──────────────────────┴──────────────────┴────────────────┘

  索引列顺序三原则：
    ① 等值条件列在前，范围条件列在后
       索引(status, create_time)
       WHERE status=1 AND create_time > '2026-01-01'
       → status等值过滤后，create_time有序可用

    ② 区分度高的列在前
       区分度 = COUNT(DISTINCT col) / COUNT(*)
       性别区分度≈0.5 → 放后面
       用户ID区分度≈1.0 → 放前面

    ③ 查询频率高的组合优先
       80%查询用(A,B) → 把(A,B)放前面
       不要为了"万能索引"把所有列塞进去

  面试核心句：
  "联合索引的列顺序不是随便排的——等值在前范围在后、区分度高的在前、
   查询频率高的组合优先。顺序错了索引可能白建。"
```

### 覆盖索引 — 消除回表

```
回表的代价：

  SELECT * FROM order WHERE user_id=100

  索引(user_id)的执行流程：
    ① 在user_id索引树找到user_id=100的主键值 [1次IO]
    ② 回表到聚簇索引查完整行 [1次IO]
    ③ 假设满足条件的100行 → 回表100次

  覆盖索引：
    SELECT user_id, order_status FROM order WHERE user_id=100

    索引(user_id, order_status)的执行流程：
    ① 在索引树找到user_id=100 → 索引中已有order_status
    ② 不需要回表 → Extra: Using index

  实战价值：
    优化前：SELECT * FROM order WHERE user_id=100 → 回表100次
    优化后：SELECT id, user_id, status FROM order WHERE user_id=100
           + 索引(user_id, status)
           → Using index，0次回表
    性能提升：从100次IO降到1次IO（极端情况）

  架构师判断：
    不是所有查询都要覆盖索引——写频繁的表索引越多写入越慢
    只对高频查询+热点SQL做覆盖索引优化
    判断标准：慢查询Top 10 → 逐个分析 → 该加覆盖索引的加
```

### 索引下推（ICP，Index Condition Pushdown）

```
MySQL 5.6+ 的优化，减少回表次数

  场景：联合索引(name, age)
  查询：SELECT * FROM user WHERE name LIKE '张%' AND age=25

  无ICP（5.6之前）：
    ① 在name索引找到所有'张%'开头的主键（假设100条）
    ② 回表100次拿到完整行
    ③ 在完整行上过滤age=25 → 可能只剩10条
    回表次数：100次（90次白回表了）

  有ICP（5.6+）：
    ① 在索引中找到所有'张%'开头的记录
    ② 在索引中直接判断age=25（age在索引中） → 过滤到10条
    ③ 只回表10次
    回表次数：10次（减少了90%的回表）

  触发条件：
    EXPLAIN的Extra显示：Using index condition
    前提：条件列在联合索引中但未走最左前缀

  架构师认知：
    ICP是MySQL引擎层的自动优化——不需要改SQL
    但架构师要理解原理：索引中的列可以在存储引擎层提前过滤
    重点是：建联合索引时要考虑哪些列可以用于ICP
```

### 其他索引优化策略

```
  索引跳跃扫描（MySQL 8.0+）：
    联合索引(A, B)，WHERE B=2 → 8.0前不走索引
    8.0后优化器可能走Skip Scan：按A的不同值分别查B=2
    适用：A的区分度低（如性别/状态），B区分度高

  索引合并（Index Merge）：
    多个单列索引同时使用
    WHERE A=1 OR B=2 → 可能用两个索引合并结果
    Extra: Using union(idx_a, idx_b)
    不如联合索引高效，但在无法改索引时可用

  函数索引（MySQL 8.0+）：
    WHERE DATE(create_time) = '2026-06-03' → 原来不走索引
    8.0后可建函数索引：ALTER TABLE t ADD INDEX idx_date ((DATE(create_time)))
    或者改写SQL：WHERE create_time >= '2026-06-03' AND create_time < '2026-06-04'
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 建索引加速查询，联合索引要按顺序 |
| 高级开发 | 最左前缀原则，覆盖索引避免回表 |
| 架构师 | 索引优化是系统级工程不是加个索引：最左前缀三原则（等值在前/区分度高在前/高频组合优先），覆盖索引消除回表但要在查询性能和写入性能间权衡，ICP是引擎层自动优化但建索引时要考虑哪些列可利用ICP，索引不是越多越好——每多一个索引INSERT/UPDATE就多维护一棵B+树。能结合业务场景给出具体索引方案，并用EXPLAIN验证 |

**补足方向**：面试时被问"你怎么做索引优化"，要说方法论——"先慢查询Top10定位热点SQL，EXPLAIN分析执行计划，按最左前缀三原则调整联合索引顺序，对回表多的用覆盖索引，注意索引数量控制在5-7个以内"。不能只说"加索引"。

## 题目三：EXPLAIN执行计划 — 架构师必会的SQL诊断工具

### EXPLAIN各字段含义

```
EXPLAIN SELECT * FROM order WHERE user_id=100 AND status=1

  ┌──────────────┬────────────────────────────────────────────────────┐
  │  字段         │  含义与架构师关注点                                │
  ├──────────────┼────────────────────────────────────────────────────┤
  │  id           │  执行顺序：id越大越先执行                          │
  │               │  相同id从上往下执行                                │
  ├──────────────┼────────────────────────────────────────────────────┤
  │  select_type  │  查询类型：                                       │
  │               │  SIMPLE(简单)/PRIMARY(主查询)/                    │
  │               │  SUBQUERY(子查询)/DERIVED(派生表)                 │
  │               │  架构师关注：DERIVED=临时表，大数据量时性能差      │
  ├──────────────┼────────────────────────────────────────────────────┤
  │  table        │  访问哪张表                                       │
  ├──────────────┼────────────────────────────────────────────────────┤
  │  type ⭐      │  访问类型（从好到差）：                            │
  │               │  system > const > eq_ref > ref > range            │
  │               │  > index > ALL                                    │
  │               │                                                   │
  │               │  架构师红线：                                      │
  │               │    ALL（全表扫描）→ 必须优化                       │
  │               │    index（全索引扫描）→ 多数需要优化               │
  │               │    range（范围扫描）→ 通常可接受                   │
  │               │    ref（索引等值）→ 目标                           │
  │               │    const/eq_ref → 最优                            │
  ├──────────────┼────────────────────────────────────────────────────┤
  │  key          │  实际使用的索引（NULL=没走索引）                   │
  ├──────────────┼────────────────────────────────────────────────────┤
  │  key_len      │  索引使用的字节数 → 判断联合索引用了几列           │
  │               │  例：idx(a,b,c)，key_len=12                       │
  │               │  a(int4)+b(int4)+c(int4)=12 → 三列全用            │
  │               │  a(int4)=4 → 只用了a列                            │
  ├──────────────┼────────────────────────────────────────────────────┤
  │  rows         │  预估扫描行数 → 不是精确值但量级有参考价值        │
  │               │  架构师关注：rows=1000和rows=100万差距巨大        │
  ├──────────────┼────────────────────────────────────────────────────┤
  │  filtered     │  过滤比例（0-100%）                               │
  │               │  实际返回行 = rows × filtered%                    │
  │               │  filtered=0.1% → 扫1000行只返回1行 → 索引太差    │
  ├──────────────┼────────────────────────────────────────────────────┤
  │  Extra ⭐     │  额外信息（架构师重点关注）：                     │
  │               │  Using index → 覆盖索引，好                       │
  │               │  Using index condition → ICP，可接受              │
  │               │  Using where → Server层过滤，需关注               │
  │               │  Using temporary → 临时表，危险信号               │
  │               │  Using filesort → 文件排序，危险信号              │
  │               │  Using join buffer → JOIN无索引，需优化           │
  └──────────────┴────────────────────────────────────────────────────┘
```

### 架构师用EXPLAIN的SOP

```
五步诊断法：

  Step 1：看type → 是否有ALL/index
    ALL → 必须加索引或改SQL
    index → 检查是否可以用覆盖索引优化

  Step 2：看key和key_len → 索引是否正确使用
    key=NULL → 没走索引
    key_len → 联合索引用了几列
    例：idx(user_id, status, create_time)
        key_len=4 → 只用了user_id（int 4字节）
        key_len=8 → 用了user_id+status
        key_len=12 → 三列全用

  Step 3：看rows × filtered → 实际扫描量
    rows=100万, filtered=0.01% → 扫100万行返回100行 → 索引选择率极差
    rows=100, filtered=100% → 扫100行返回100行 → 索引选择率好

  Step 4：看Extra → 有无危险信号
    Using temporary → 需要优化GROUP BY/ORDER BY/DISTINCT
    Using filesort → 排序没走索引，数据量大时性能灾难
    两者同时出现 → 必须优化

  Step 5：对比优化前后
    优化前：type=ALL, rows=500万, Extra=Using where; Using filesort
    加索引后：type=ref, rows=100, Extra=Using index condition
    量化提升：扫描行数从500万→100，提升5万倍
```

### 典型EXPLAIN案例

```
案例1：全表扫描 → 加索引

  EXPLAIN SELECT * FROM order WHERE user_id=100
  ┌────┬────────┬────────┬──────┬──────────┬───────┬──────┬─────┐
  │ id │ type   │ key    │ rows │ filtered │ Extra                     │
  ├────┼────────┼────────┼──────┼──────────┼───────────────────────────┤
  │  1 │ ALL    │ NULL   │ 500万│ 0.01%    │ Using where              │
  └────┴────────┴────────┴──────┴──────────┴───────────────────────────┘

  加索引：ALTER TABLE order ADD INDEX idx_user_id(user_id)

  ┌────┬────────┬──────────────┬──────┬──────────┬─────┐
  │ id │ type   │ key          │ rows │ filtered │ Extra             │
  ├────┼────────┼──────────────┼──────┼──────────┼─────┤
  │  1 │ ref    │ idx_user_id  │ 50   │ 100%     │ NULL              │
  └────┴────────┴──────────────┴──────┴──────────┴─────┘
  提升：500万行→50行

案例2：Using filesort → 优化ORDER BY

  EXPLAIN SELECT * FROM order WHERE user_id=100 ORDER BY create_time
  ┌────┬────────┬──────────────┬──────┬──────────────────────────────┐
  │ id │ type   │ key          │ rows │ Extra                        │
  ├────┼────────┼──────────────┼──────┼──────────────────────────────┤
  │  1 │ ref    │ idx_user_id  │ 50   │ Using where; Using filesort  │
  └────┴────────┴──────────────┴──────┴──────────────────────────────┘

  优化：改为联合索引(user_id, create_time)

  ┌────┬──────┬─────────────────────┬──────┬─────────┐
  │ id │ type │ key                 │ rows │ Extra   │
  ├────┼──────┼─────────────────────┼──────┼─────────┤
  │  1 │ ref  │ idx_user_create     │ 50   │ NULL    │
  └────┴──────┴─────────────────────┴──────┴─────────┘
  filesort消失 → 排序走索引

案例3：Using temporary → 优化GROUP BY

  EXPLAIN SELECT status, COUNT(*) FROM order GROUP BY status
  ┌────┬───────┬──────┬──────┬──────────────────────────────┐
  │ id │ type  │ key  │ rows │ Extra                        │
  ├────┼───────┼──────┼──────┼──────────────────────────────┤
  │  1 │ ALL   │ NULL │ 500万│ Using temporary; Using filesort│
  └────┴───────┴──────┴──────┴──────────────────────────────┘

  优化：加索引(status)

  ┌────┬───────┬─────────────┬──────┬─────────┐
  │ id │ type  │ key         │ rows │ Extra   │
  ├────┼───────┼─────────────┼──────┼─────────┤
  │  1 │ index │ idx_status  │ 500万│ Using index │
  └────┴───────┴─────────────┴──────┴─────────┘
  temporary和filesort都消失 → 覆盖索引+索引有序
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 用EXPLAIN看看有没有走索引 |
| 高级开发 | type字段看是不是ALL，key看走了哪个索引 |
| 架构师 | EXPLAIN五步诊断法（type→key/key_len→rows×filtered→Extra→优化前后对比），能从key_len反推联合索引用了几列，Extra三个危险信号（temporary/filesort/join buffer）各对应什么问题，优化时量化提升（500万→50行），不是"看一眼"而是"系统化诊断" |

## 题目四：慢查询治理 — 从发现到根治的工程化方案

### 慢查询发现：三层监控

```
  ┌─────────────────────────────────────────────────────────┐
  │  L1 MySQL慢查询日志                                      │
  │  slow_query_log=ON                                      │
  │  long_query_time=1（超过1秒记录）                        │
  │  工具：mysqldumpslow / pt-query-digest                  │
  │  作用：发现慢SQL，但不能定位根因                          │
  │                                                          │
  │  L2 应用层监控                                           │
  │  SkyWalking/Prometheus → 接口RT分布 + SQL执行时间        │
  │  作用：关联到具体接口，知道影响面                         │
  │                                                          │
  │  L3 DBA巡检                                             │
  │  information_schema.PROCESSLIST → 当前执行中的SQL        │
  │  作用：发现正在执行的慢SQL，实时干预                      │
  └─────────────────────────────────────────────────────────┘

  pt-query-digest分析结果示例：
    Rank Query ID         Response time  Calls  R/Call
    ==== ================ ============== ====== ======
    1    0x5A1B2C3D       1200.0000     500    2.4000
    2    0x6D7E8F9A       800.0000      200    4.0000

  架构师关注：不是所有慢查询都值得优化
    优先级：执行次数多 × 影响用户多 = 优先优化
    低频管理后台的慢查询 → 不用管（非核心路径）
```

### 慢查询优化策略：六层递进

```
  ┌─────────────────────────────────────────────────────────┐
  │  L1 加索引（成本最低，效果可能最好）                      │
  │    适用：type=ALL，没有合适的索引                        │
  │    案例：WHERE user_id=100 全表扫描                      │
  │          → 加索引idx_user_id，扫描500万→50行            │
  │                                                          │
  │  L2 覆盖索引（消除回表）                                 │
  │    适用：回表次数多，Extra无Using index                   │
  │    案例：SELECT id,name FROM user WHERE age=25           │
  │          → 索引(age, name)，Using index                  │
  │                                                          │
  │  L3 SQL改写（不改业务逻辑，改写SQL）                     │
  │    常见改写：                                             │
  │      SELECT * → SELECT 具体列（减少网络+可能触发覆盖索引）│
  │      子查询 → JOIN（MySQL优化器对JOIN优化更好）          │
  │      OR → UNION ALL（OR可能导致索引失效）                │
  │      LIKE '%abc%' → LIKE 'abc%'（前缀匹配才能走索引）   │
  │      DATE(col) = '2026-06-03' → col >= '2026-06-03'    │
  │        AND col < '2026-06-04'（函数导致索引失效）        │
  │      LIMIT 1000000,10 → 游标翻页（深度分页）            │
  │                                                          │
  │  L4 反范式冗余（用空间换时间）                           │
  │    适用：跨表JOIN是瓶颈，且更新频率低                     │
  │    案例：订单列表需要用户名 → 订单表冗余user_name        │
  │    代价：用户改名要同步更新（一致性风险）                 │
  │    架构师判断：读多写少→冗余值，写多→不冗余              │
  │                                                          │
  │  L5 缓存（DB外优化）                                     │
  │    适用：读多写少，数据时效性要求不高                     │
  │    案例：商品详情缓存Redis → 命中率95%                   │
  │    和Day1五层前置方案L2一脉相承                          │
  │                                                          │
  │  L6 读写分离+分库分表（最后手段）                        │
  │    优化了SQL还是扛不住 → Day1/Day2方案                   │
  │    和Day1五层前置方案L3/L5一脉相承                       │
  └─────────────────────────────────────────────────────────┘
```

### 深度分页优化 — 面试高频考点

```
问题：LIMIT 1000000, 10 → 扫描100万+10行，丢弃100万行

  方案1：游标翻页（推荐）
    WHERE id > 上次最大id ORDER BY id LIMIT 10
    每次只扫10行，但要求排序字段有索引
    适用：下一页/上一页（不支持跳页）

  方案2：子查询延迟关联
    SELECT * FROM order o
    INNER JOIN (
      SELECT id FROM order WHERE user_id=100 ORDER BY create_time LIMIT 1000000, 10
    ) tmp ON o.id = tmp.id

    子查询走覆盖索引只取id → Using index
    外层用id回表只取10行
    扫描量：100万行索引扫描 + 10次回表（vs 原来100万次回表）

  方案3：业务限制
    最多翻100页（淘宝/京东都这么干）
    超过100页引导用户加搜索条件
    架构师认知：用户真的需要看第10万页吗？不需要就不要支持

  架构师选型：
    ≤100页 → LIMIT正常翻页
    大量翻页 → 游标翻页（产品配合）
    必须跳页 → 子查询延迟关联
    终极方案 → ES搜索引擎（Day1 L2 ES索引）
```

### COUNT优化 — 另一个面试高频

```
  COUNT(*) vs COUNT(1) vs COUNT(列名)

    COUNT(*) → InnoDB专门优化，统计行数，不取具体列
    COUNT(1) → 和COUNT(*)等价（MySQL优化器处理）
    COUNT(列名) → 统计该列非NULL的行数，要遍历判断

    架构师回答：
      "COUNT(*)和COUNT(1)没有性能差异，MySQL优化器等价处理。
       但COUNT(列名)语义不同——它统计非NULL行数。
       InnoDB的COUNT(*)要遍历二级索引（比遍历聚簇索引IO少）"

  大表COUNT的优化：
    精确总数：维护计数表 + 触发器/应用层更新
    近似总数：SHOW TABLE STATUS的Rows字段（估计值）
    业务方案：产品是否真的需要精确总数？"共约1万条"够不够？
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 开启慢查询日志，找到慢SQL加索引 |
| 高级开发 | 用EXPLAIN分析，加索引或改SQL |
| 架构师 | 慢查询治理是工程化体系：三层监控发现（慢日志+应用监控+DBA巡检）→ 六层递进优化（索引→覆盖索引→SQL改写→冗余→缓存→读写分离/分库分表）→ 深度分页三种方案（游标/延迟关联/业务限制）→ COUNT优化（精确vs近似vs业务妥协）。每层和Day1/Day2的前置方案一脉相承——低成本先行，高成本殿后。不是"加个索引"而是"系统化治理" |

**补足方向**：面试时被问"你怎么做SQL优化"，要说"先三层监控定位热点SQL，EXPLAIN五步诊断，然后按六层递进优化——索引最简单，SQL改写次之，冗余和缓存要评估代价，分库分表是最后的核弹。深度分页用游标翻页或延迟关联，COUNT要问产品是否真需要精确数"。不要只说"加索引"。

## 题目五：索引设计实战 — 结合业务场景的索引方案设计

### 场景1：订单表（高写入+多维查询）

```
表结构：order(id, user_id, merchant_id, status, create_time, amount, ...)

查询场景分析：
  ① 用户查自己的订单：WHERE user_id=? ORDER BY create_time DESC
    频率：极高（80%查询）
    索引：idx_user_create(user_id, create_time)

  ② 商户查订单：WHERE merchant_id=? AND status=?
    频率：中（15%查询）
    索引：idx_merchant_status(merchant_id, status)

  ③ 运营后台按状态+时间查：WHERE status=? AND create_time BETWEEN
    频率：低（5%查询）
    方案：不走索引也可以（非核心路径），或者走ES

索引方案：
  聚簇索引：id（自增主键，顺序写入无页分裂）
  idx_user_create(user_id, create_time) → 覆盖用户订单列表+排序
  idx_merchant_status(merchant_id, status) → 覆盖商户查询
  共3个索引（含主键）

  架构师权衡：
    不建(user_id, status, create_time)——因为80%场景只需user_id+create_time
    不建(merchant_id, status, create_time)——写入代价增大
    运营后台5%查询走全表扫+分页可接受

  反例（过度索引）：
    有人喜欢建6-8个索引"以防万一"→ INSERT每多一个索引多维护一棵B+树
    1万QPS写入 × 6个索引 = 6万次B+树维护 → 写入性能下降40%+
```

### 场景2：秒杀商品表（高并发读+低写入）

```
表结构：seckill_goods(id, goods_id, stock, start_time, end_time, status, ...)

查询场景：
  ① 查秒杀商品详情：WHERE goods_id=? AND status=1
  ② 扣库存：UPDATE seckill_goods SET stock=stock-1 WHERE goods_id=? AND stock>0
  ③ 查活动列表：WHERE status=1 AND start_time<=NOW() AND end_time>=NOW()

索引方案：
  聚簇索引：id（自增）
  idx_goods_status(goods_id, status) → 查商品详情
  idx_status_time(status, start_time, end_time) → 活动列表

  扣库存SQL的优化：
    WHERE goods_id=? AND stock>0
    走idx_goods_status → 定位到行 → 乐观锁扣减
    但stock>0是范围条件 → 无法用索引判断库存是否足够
    解法：Redis预扣库存（Day5差距三售罄标记），DB只做最终校验
```

### 场景3：用户表（读写均衡+多维度查询）

```
表结构：user(id, phone, name, age, city, status, create_time, ...)

查询场景：
  ① 手机号登录：WHERE phone=?
  ② 按城市+年龄筛选：WHERE city=? AND age BETWEEN ? AND ?
  ③ 搜索用户名：WHERE name LIKE '张%'

索引方案：
  聚簇索引：id（自增）
  uk_phone(phone) → 唯一索引，登录高频查询
  idx_city_age(city, age) → 城市筛选+年龄范围
  name查询：LIKE '张%'走索引前缀，LIKE '%张%'不行

  架构师决策：
    phone必须唯一索引（业务语义+登录性能）
    city+age联合索引：city等值在前，age范围在后
    name模糊搜索 → 如果是核心需求考虑ES全文索引
    总共3个索引（含主键），控制在5个以内

  面试关键：
    "用户表索引设计的核心是区分查询频率：
     手机号登录占60%查询 → 唯一索引必须
     城市筛选占20% → 联合索引
     名称搜索占10% → LIKE前缀或ES
     其他10% → 全表扫也行"
```

### 索引设计的架构师原则

```
  ① 控制索引数量：单表5-7个以内（含主键）
     每多一个索引，INSERT/UPDATE/DELETE多维护一棵B+树
     写入密集表 → 3-4个
     查询密集表 → 5-7个
     超过7个 → 要审视是否过度索引

  ② 索引列选择：
     区分度 > 0.1 的列才值得建索引
     性别/状态等区分度极低的列 → 单独建索引无意义
     但可以作为联合索引的一部分（ICP/覆盖索引）

  ③ 避免冗余索引：
     idx_a(a) + idx_ab(a, b) → idx_a是冗余的
     最左前缀原则：idx_ab已经覆盖了idx_a的能力

  ④ 定期清理无用索引：
     查询sys.schema_unused_indexes
     上线2个月没被使用过的索引 → 删除

  ⑤ 自增主键 vs 业务主键：
     自增主键：顺序写入，无页分裂，4-8字节
     UUID：随机写入，频繁页分裂，36字节（二级索引引用更大）
     架构师推荐：自增主键做聚簇索引 + 业务唯一键做二级索引
     如果用分布式ID → Leaf号段（趋势递增，接近顺序写入）
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 给查询条件建索引 |
| 高级开发 | 按最左前缀原则建联合索引，避免回表 |
| 架构师 | 索引设计是查询频率×写入代价×存储成本的三维权衡：先分析查询场景频率分布（80/15/5），高频查询优先覆盖，控制索引总量5-7个以内，区分度<0.1不单独建索引，定期用sys.schema_unused_indexes清理无用索引，自增主键避免页分裂，结合Day1基因法考虑分片键和索引的协同设计。不是"建索引"而是"系统设计" |

**补足方向**：面试时被问"你怎么做索引设计"，要说"先分析查询场景频率分布，高频查询用联合索引+覆盖索引一箭双雕，控制索引总量5-7个以内，每多一个索引写入多维护一棵B+树，低频查询走全表扫或ES"。不要说"给WHERE条件加索引"。
