# Day 1：分库分表基础 — 为什么分/怎么分/分片键选型/ShardingSphere入门（练习整理）

## 题目一：为什么要分库分表 — 单库的瓶颈与拆分时机

### 单库的六大瓶颈

```
MySQL单库的物理极限：

  ┌────────────────────────────────────────────────────────┐
  │  瓶颈1：连接数                                          │
  │  单MySQL实例默认max_connections=151                     │
  │  生产调到3000-5000，但有效并发约2000-3000               │
  │  微服务50个实例×每实例连接池50 = 2500连接 → 接近上限    │
  │                                                         │
  │  瓶颈2：QPS                                             │
  │  单机读QPS极限：8000-10000（简单查询）                   │
  │  单机写QPS极限：3000-5000（单行写入）                    │
  │  混合读写：有效QPS约3000-5000                            │
  │                                                         │
  │  瓶颈3：数据量                                          │
  │  单表行数 > 2000万 → B+树层级增加，查询性能下降          │
  │  单表数据文件 > 50GB → DDL/备份/恢复耗时不可接受         │
  │                                                         │
  │  瓶颈4：写入性能                                        │
  │  InnoDB写放大：一次写=写redo+undo+binlog+数据页          │
  │  自增ID锁竞争：AUTO_INCREMENT并发写瓶颈                  │
  │  索引维护开销：索引越多，写入越慢                        │
  │                                                         │
  │  瓶颈5：DDL锁表                                        │
  │  ALTER TABLE加列 → 5000万行表锁2小时                    │
  │  Online DDL也有风险（磁盘空间/主从延迟）                │
  │                                                         │
  │  瓶颈6：主从延迟                                        │
  │  单主写入压力大 → binlog传输延迟增大                    │
  │  从库延迟5秒+ → 读到过期数据                            │
  └────────────────────────────────────────────────────────┘
```

### 拆分时机的判断框架

```
不是"到了瓶颈再拆"，是"看到瓶颈趋势就拆"：

  ┌────────────────────────────────────────────────────────┐
  │  维度1：数据量                                          │
  │  单表 < 500万行    → 不需要分表                         │
  │  单表 500万-2000万 → 监控趋势，准备方案                 │
  │  单表 > 2000万行   → 必须分表                           │
  │  单库 > 500GB      → 必须分库                           │
  │                                                         │
  │  维度2：QPS                                             │
  │  单库QPS < 2000    → 不需要分库                         │
  │  单库QPS 2000-5000 → 加从库/缓存，评估分库              │
  │  单库QPS > 5000    → 必须分库                           │
  │                                                         │
  │  维度3：连接数                                          │
  │  活跃连接 < 1000    → 不需要分库                         │
  │  活跃连接 1000-3000 → 连接池优化+评估分库               │
  │  活跃连接 > 3000    → 必须分库                           │
  │                                                         │
  │  维度4：DDL影响                                        │
  │  ALTER < 10秒      → 不需要分表                         │
  │  ALTER 10秒-5分钟  → 评估分表（影响在线业务）            │
  │  ALTER > 5分钟     → 必须分表                           │
  └────────────────────────────────────────────────────────┘

  面试核心句：
  "分库分表不是上来就拆，是先试低成本方案：
   读写分离→缓存→SQL优化→归档历史数据→最后才分库分表。
   每一步的成本递增，分库分表是最后手段。"
```

### 低成本方案的优先级

```
  ┌────────────────────────────────────────────────────────┐
  │  优先级从低到高（成本从低到高）：                        │
  │                                                         │
  │  L1 SQL优化和索引 — 成本最低，效果可能立竿见影          │
  │    慢查询优化 → 加索引 → 覆盖索引 → 减少回表           │
  │    例子：加了覆盖索引，QPS从500提升到5000              │
  │                                                         │
  │  L2 缓存 — 减少DB读压力                                 │
  │    Redis缓存热点数据 → 读QPS从DB卸载到Redis            │
  │    例子：商品详情缓存命中率95%→DB读QPS降20倍           │
  │                                                         │
  │  L3 读写分离 — 读走从库，写走主库                       │
  │    1主2从 → 读能力3倍                                   │
  │    限制：主从延迟导致的数据不一致问题                    │
  │                                                         │
  │  L4 数据归档 — 减少主表数据量                            │
  │    历史订单归档到归档库/冷存储                           │
  │    例子：3个月前订单迁移 → 主表从5000万降到1000万       │
  │                                                         │
  │  L5 分库分表 — 最后手段，成本最高                        │
  │    开发成本：代码改造+跨片查询+分布式事务                │
  │    运维成本：多库多表管理+数据迁移+监控                  │
  └────────────────────────────────────────────────────────┘

  和Day2五层漏斗的思路一致：
    先用低成本的挡，挡不住再用高成本的
    分库分表是最后一道防线，不是第一选择
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 数据多了就分库分表 |
| 高级开发 | 单表超2000万行要分表，用ShardingSphere |
| 架构师 | 分库分表是最后手段。先SQL优化→缓存→读写分离→归档，都解决不了才分。拆分时机看四个维度（数据量/QPS/连接数/DDL影响），每个维度有明确的阈值。分库分表带来跨片查询/分布式事务/数据迁移三大难题，成本远高于前四种方案 |

**补足方向**：面试时被问"你们为什么要分库分表"，要说具体数字——"订单表5000万行，单表查询P99从50ms涨到500ms，读写分离+缓存已经用了，QPS还是打满主库5000，不得不分"。不能只说"数据多了"。

## 题目二：分库分表策略 — 垂直拆分 vs 水平拆分

### 垂直拆分 — 按业务拆

```
垂直分库：按业务域拆分数据库

  拆分前：
  ┌──────────────────────────────────────────────┐
  │                单库 all_in_one                │
  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐        │
  │  │用户表│ │订单表│ │商品表│ │支付表│        │
  │  │地址表│ │订单详│ │分类表│ │流水表│        │
  │  │角色表│ │物流表│ │评价表│ │退款表│        │
  │  └──────┘ └──────┘ └──────┘ └──────┘        │
  │                                               │
  │  问题：用户域和订单域在同一个库，互相影响      │
  │  DDL锁表影响全业务，连接数共享                 │
  └──────────────────────────────────────────────┘

  拆分后：
  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 用户库    │  │ 订单库    │  │ 商品库    │  │ 支付库    │
  │ user_db  │  │ order_db │  │ goods_db │  │ pay_db   │
  │          │  │          │  │          │  │          │
  │ 用户表   │  │ 订单表   │  │ 商品表   │  │ 支付表   │
  │ 地址表   │  │ 订单详情 │  │ 分类表   │  │ 流水表   │
  │ 角色表   │  │ 物流表   │  │ 评价表   │  │ 退款表   │
  └──────────┘  └──────────┘  └──────────┘  └──────────┘

  垂直分库的好处：
    ① 业务隔离：用户库DDL不影响订单库
    ② 连接独立：每个库有独立的连接池
    ③ 故障隔离：一个库挂了不影响其他业务
    ④ 权限控制：不同业务不同账号

  垂直分表：按列拆分（宽表拆窄表）

  ┌─────────────────────────────────┐
  │  拆分前：商品表（80列）          │
  │  id, name, price, stock,        │
  │  desc, images, specs,           │
  │  seo_title, seo_desc, seo_kw,   │
  │  create_time, update_time...    │
  │                                  │
  │  问题：查询列表只要基础字段，    │
  │  但desc/images这些大字段也加载了 │
  └─────────────────────────────────┘
           │
           ▼
  ┌──────────────────┐  ┌──────────────────┐
  │  商品基础表       │  │  商品详情表       │
  │  id, name, price │  │  id, desc, images│
  │  stock, status   │  │  specs, seo_*    │
  │                  │  │                  │
  │  高频查询用这个  │  │  详情页才查这个  │
  │  行小→内存效率高 │  │  行大→单独存储   │
  └──────────────────┘  └──────────────────┘
```

### 水平拆分 — 按规则分

```
水平分库分表：同一张表的数据按规则分散到多个库多个表

  分库规则：按用户ID分库
    user_id % 4 → 4个库

  分表规则：按订单ID分表
    order_id % 8 → 每个库8张表

  ┌────────────────────────────────────────────────────────┐
  │                   水平拆分后的结构                       │
  │                                                         │
  │  order_db_0              order_db_1                     │
  │  ├─ t_order_0           ├─ t_order_0                   │
  │  ├─ t_order_1           ├─ t_order_1                   │
  │  ├─ t_order_2           ├─ t_order_2                   │
  │  ├─ t_order_3           ├─ t_order_3                   │
  │  ├─ t_order_4           ├─ t_order_4                   │
  │  ├─ t_order_5           ├─ t_order_5                   │
  │  ├─ t_order_6           ├─ t_order_6                   │
  │  └─ t_order_7           └─ t_order_7                   │
  │                                                         │
  │  order_db_2              order_db_3                     │
  │  ├─ t_order_0           ├─ t_order_0                   │
  │  ├─ t_order_1           ├─ t_order_1                   │
  │  ├─ t_order_2           ├─ t_order_2                   │
  │  ├─ t_order_3           ├─ t_order_3                   │
  │  ├─ t_order_4           ├─ t_order_4                   │
  │  ├─ t_order_5           ├─ t_order_5                   │
  │  ├─ t_order_6           ├─ t_order_6                   │
  │  └─ t_order_7           └─ t_order_7                   │
  │                                                         │
  │  总计：4库 × 8表 = 32张分表                             │
  │  每张表存 5000万/32 ≈ 156万行 → 单表性能优秀           │
  └────────────────────────────────────────────────────────┘

  水平拆分的三种路由策略：

  ┌────────────────────────────────────────────────────────┐
  │  策略1：取模（Hash）                                    │
  │    order_id % 32 → 确定库和表                          │
  │    优点：数据分布均匀                                   │
  │    缺点：扩容需要数据迁移（rehash）                     │
  │    适用：数据量相对稳定的场景                           │
  │                                                         │
  │  策略2：范围（Range）                                   │
  │    order_id 1-1000万 → t_order_0                       │
  │    order_id 1000万-2000万 → t_order_1                  │
  │    优点：扩容简单，加新表就行                           │
  │    缺点：热点问题（最新数据集中在最新表）               │
  │    适用：按时间写入的场景（日志/流水）                  │
  │                                                         │
  │  策略3：一致性Hash                                     │
  │    虚拟节点 + Hash环 → 减少rehash迁移量                │
  │    优点：扩容只影响相邻节点                             │
  │    缺点：实现复杂                                       │
  │    适用：频繁扩缩容的场景                               │
  └────────────────────────────────────────────────────────┘
```

### 垂直拆分 vs 水平拆分的决策框架

```
  ┌──────────────┬──────────────────┬──────────────────────┐
  │              │  垂直拆分         │  水平拆分             │
  ├──────────────┼──────────────────┼──────────────────────┤
  │  解决的问题  │  业务耦合/       │  单表数据量/QPS瓶颈  │
  │              │  连接数共享       │                      │
  ├──────────────┼──────────────────┼──────────────────────┤
  │  拆分依据    │  业务域边界       │  数据量+QPS          │
  ├──────────────┼──────────────────┼──────────────────────┤
  │  拆分后特征  │  不同表在不同库   │  同表数据在不同库表  │
  ├──────────────┼──────────────────┼──────────────────────┤
  │  对应用影响  │  跨库join消失     │  路由/跨片查询/      │
  │              │  本地事务变分布式 │  分布式事务           │
  ├──────────────┼──────────────────┼──────────────────────┤
  │  先做哪个    │  先做（业务隔离） │  后做（性能扩展）    │
  ├──────────────┼──────────────────┼──────────────────────┤
  │  实际项目    │  先垂直拆出支付库 │  再水平拆订单表      │
  │  的顺序      │  （业务独立）     │  （5000万行扛不住）  │
  └──────────────┴──────────────────┴──────────────────────┘

  面试核心句：
  "垂直拆分解决业务边界问题，水平拆分解决性能容量问题。
   实际项目中先垂直再水平——先按业务拆库（和微服务拆分一脉相承），
   再按数据量水平分表。"
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 数据多了就分表，用user_id取模 |
| 高级开发 | 垂直分库按业务拆，水平分表按数据量拆，用ShardingSphere |
| 架构师 | 垂直拆分解决业务边界，水平拆分解决容量瓶颈，先垂直后水平。三种路由策略（取模/范围/一致性Hash）各有适用场景：取模均匀但扩容难，范围扩容简单但有热点，一致性Hash折中但复杂。选型看业务特征——稳定数据量用取模，时间序列用范围，弹性扩缩用一致性Hash |

**补足方向**：被问"你们怎么分库分表的"，要说出完整的决策路径——"先垂直拆出支付库（金融独立监管），再水平拆订单表（5000万行取模16分表），分片键选order_id因为所有核心查询都带order_id"。

## 题目三：分片键选型 — 分库分表最关键的决策

### 分片键选型的三原则

```
分片键选错了=分库分表白做

  原则1：查询覆盖度 — 尽量多的查询能定位到具体分片
  原则2：数据均匀性 — 数据不能倾斜到某几个分片
  原则3：业务相关性 — 同一业务维度的数据尽量落在同一分片

  ┌────────────────────────────────────────────────────────┐
  │  原则1的量化：查询路由命中率                            │
  │                                                         │
  │  假设订单表的查询场景：                                 │
  │    ① 按order_id查询：70%的查询                         │
  │    ② 按user_id查询：20%的查询                          │
  │    ③ 按merchant_id查询：5%的查询                       │
  │    ④ 按create_time范围查询：5%的查询                   │
  │                                                         │
  │  分片键选order_id → 70%的查询精确路由，30%广播          │
  │  分片键选user_id  → 20%的查询精确路由，80%广播         │
  │                                                         │
  │  选哪个？                                               │
  │    不一定选覆盖率最高的！要看核心链路的延迟要求         │
  │    核心链路(下单/支付)都是按order_id → 选order_id      │
  │    管理后台(按user查)可以接受广播 → 不影响核心          │
  └────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────┐
  │  原则2的量化：数据倾斜率                                │
  │                                                         │
  │  反例：按merchant_id分片                                │
  │    大商家100万订单 → 某个分片爆满                       │
  │    小商家1000订单 → 某个分片空空                        │
  │    倾斜率 1000:1 → 数据不均匀 = 没有真正分散压力       │
  │                                                         │
  │  正例：按order_id取模分片                               │
  │    order_id是自增/雪花算法 → 分布天然均匀              │
  │    倾斜率 ≈ 1:1 → 每个分片负载均衡                    │
  │                                                         │
  │  面试陷阱：                             │
  │    虽然按user_id查询多，但大用户和小用户数据量差异大    │
  │    按user_id分片→倾斜→某分片过热→又回到单库瓶颈        │
  └────────────────────────────────────────────────────────┘

  ┌────────────────────────────────────────────────────────┐
  │  原则3的应用：关联数据同分片                             │
  │                                                         │
  │  订单表和订单详情表按相同分片键分片：                   │
  │    order_id % 4 = 2 → order_db_2.t_order_2              │
  │    order_id % 4 = 2 → order_db_2.t_order_item_2         │
  │                                                         │
  │    同一个order_id的订单和详情在同一个库同一个表序号     │
  │    → 查询不需要跨库join                                 │
  │    → 事务还是本地事务                                   │
  │                                                         │
  │  这叫"绑定表"（ShardingSphere的binding-table）          │
  │  核心思想：有关联关系的数据用相同的分片策略             │
  └────────────────────────────────────────────────────────┘
```

### 分片键选型的常见场景

```
  ┌──────────────┬──────────────────┬──────────────────────┬──────────────┐
  │  业务场景     │  推荐分片键       │  原因                 │  倾斜风险    │
  ├──────────────┼──────────────────┼──────────────────────┼──────────────┤
  │  订单系统     │  order_id        │  核心链路都带order_id│  低(雪花ID)  │
  │  用户系统     │  user_id         │  所有查询带user_id   │  中(大用户)  │
  │  支付流水     │  pay_id          │  核心查询带pay_id    │  低(雪花ID)  │
  │  商品系统     │  不分表          │  商品量不大，缓存覆盖│  -           │
  │  日志流水     │  create_time     │  按时间归档查询      │  低(均匀写入)│
  │  IM消息       │  conversation_id │  同一会话同分片      │  高(热门群)  │
  └──────────────┴──────────────────┴──────────────────────┴──────────────┘

  特殊场景：多分片键

  有些系统确实需要多维度查询，怎么办？

  方案1：基因法（推荐）
    order_id中嵌入user_id的后几位（基因）
    例：order_id = 雪花ID(56bit) + user_id后4bit
    按order_id分片 → user_id的后4bit也决定了分片位置
    按user_id查询时 → 提取后4bit → 也能定位分片

    ┌───────────────────────────────────────────────────┐
    │  order_id结构：                                    │
    │  ┌────────────────────┬──────────┐                │
    │  │  雪花ID(52bit)     │基因(4bit) │                │
    │  │  时间+机器+序列    │user_id%16│                │
    │  └────────────────────┴──────────┘                │
    │                                                    │
    │  按order_id查询：直接路由                          │
    │  按user_id查询：user_id%16 → 定位到1/16的分片     │
    │  比全量广播好16倍！                                │
    └───────────────────────────────────────────────────┘

  方案2：冗余表
    按order_id分主表，再建一份按user_id分的冗余表
    冗余表只存order_id和user_id的映射关系
    按user_id查 → 先查冗余表得到order_id → 再查主表

    优点：两个维度都能精确路由
    缺点：双写一致性问题

  方案3：ES索引
    MySQL按order_id分片，ES按user_id建索引
    按user_id查 → ES得到order_id列表 → 再查MySQL

    优点：MySQL不用双写
    缺点：引入ES增加复杂度和延迟
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 按id取模分表就行了 |
| 高级开发 | 分片键要选查询最常用的字段，注意数据倾斜 |
| 架构师 | 分片键选型三原则（查询覆盖度×数据均匀性×业务相关性），量化评估：70%查询带order_id→选order_id。多维度查询用基因法（order_id嵌入user_id后4bit→按user_id查只扫1/16分片）或ES索引。绑定表保证关联数据同分片，避免跨库join |

**补足方向**：练习回答"你们订单表的分片键怎么选的"——"选order_id因为核心链路70%的查询都带order_id，order_id是雪花算法天然均匀，订单表和订单详情表绑定同一个分片键避免跨库join。按user_id查询走基因法，order_id末4位嵌入user_id基因，用户维度的查询只扫1/16分片"。

## 题目四：ShardingSphere入门 — 分库分表中间件的选型与架构

### 三大中间件对比

```
  ┌──────────────┬──────────────────┬──────────────────┬──────────────────┐
  │              │  ShardingSphere   │  MyCat            │  TDDL            │
  ├──────────────┼──────────────────┼──────────────────┼──────────────────┤
  │  架构模式    │  代理/客户端双模  │  代理模式         │  客户端模式      │
  │              │  (Proxy/JDBC)    │  (独立进程)       │  (Jar包)         │
  ├──────────────┼──────────────────┼──────────────────┼──────────────────┤
  │  性能损耗    │  JDBC:低(~5%)    │  中(~15-20%)     │  低(~5%)         │
  │              │  Proxy:中(~15%)  │  额外网络跳转    │                  │
  ├──────────────┼──────────────────┼──────────────────┼──────────────────┤
  │  运维复杂度  │  JDBC:低         │  高(独立进程     │  低              │
  │              │  Proxy:中        │   需要运维)      │                  │
  ├──────────────┼──────────────────┼──────────────────┤                  │
  │  语言支持    │  Java为主        │  语言无关         │  Java            │
  ├──────────────┼──────────────────┼──────────────────┼──────────────────┤
  │  社区活跃度  │  Apache顶级项目  │  几乎停更         │  阿里内部用      │
  │              │  活跃             │                  │  开源不活跃      │
  ├──────────────┼──────────────────┼──────────────────┼──────────────────┤
  │  推荐度      │  ★★★★★          │  ★★☆☆☆          │  ★★★☆☆          │
  └──────────────┴──────────────────┴──────────────────┴──────────────────┘

  选型结论：ShardingSphere（JDBC模式）

  为什么选JDBC模式不选Proxy模式？
    ① 性能：JDBC直连DB，无额外网络跳转，损耗<5%
    ② 运维：JDBC是jar包，随应用部署，无独立进程运维
    ③ 稳定性：无Proxy单点问题
    ④ 限制：只支持Java，但项目就是Java技术栈

  什么时候用Proxy模式？
    多语言团队 → Proxy语言无关
    DBA统一管理 → Proxy有管控界面
    不想改代码 → Proxy对应用透明
```

### ShardingSphere-JDBC的工作原理

```
  ┌────────────────────────────────────────────────────────┐
  │              ShardingSphere-JDBC 工作流程               │
  │                                                         │
  │  应用代码                                                │
  │  SELECT * FROM t_order WHERE order_id = 1001            │
  │       │                                                 │
  │       ▼                                                 │
  │  ┌──────────────────────────────────────┐               │
  │  │  ① SQL解析                           │               │
  │  │  解析出：表名=t_order, 条件=order_id=1001│            │
  │  └──────────────┬───────────────────────┘               │
  │                 ▼                                       │
  │  ┌──────────────────────────────────────┐               │
  │  │  ② 路由                              │               │
  │  │  order_id=1001, 分片算法=order_id%4  │               │
  │  │  1001 % 4 = 1 → order_db_1.t_order_1 │               │
  │  └──────────────┬───────────────────────┘               │
  │                 ▼                                       │
  │  ┌──────────────────────────────────────┐               │
  │  │  ③ SQL改写                           │               │
  │  │  SELECT * FROM t_order_1              │               │
  │  │  WHERE order_id = 1001               │               │
  │  │  （表名从t_order改为t_order_1）      │               │
  │  └──────────────┬───────────────────────┘               │
  │                 ▼                                       │
  │  ┌──────────────────────────────────────┐               │
  │  │  ④ 执行                              │               │
  │  │  发送到order_db_1执行                │               │
  │  └──────────────┬───────────────────────┘               │
  │                 ▼                                       │
  │  ┌──────────────────────────────────────┐               │
  │  │  ⑤ 归并                              │               │
  │  │  单分片查询 → 直接返回               │               │
  │  │  多分片查询 → 合并+排序+分页         │               │
  │  └──────────────────────────────────────┘               │
  └────────────────────────────────────────────────────────┘

  跨分片查询的归并策略：
    排序：每个分片各自排序 → 归并排序
    分页：每个分片取前N条 → 内存合并取前N条
    聚合：SUM→各分片求和再汇总，COUNT→各分片计数再求和
    AVG→不能直接平均，要SUM/COUNT重新计算
```

### ShardingSphere核心配置

```yaml
# 分库分表配置示例（Spring Boot + ShardingSphere-JDBC）

spring:
  shardingsphere:
    datasource:
      names: ds0,ds1
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://localhost:3306/order_db_0
        username: root
        password: root
      ds1:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://localhost:3306/order_db_1
        username: root
        password: root

    rules:
      sharding:
        tables:
          t_order:
            # 实际表：ds0.t_order_0, ds0.t_order_1, ds1.t_order_0, ds1.t_order_1
            actual-data-nodes: ds${0..1}.t_order_${0..1}
            database-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: order-db-inline
            table-strategy:
              standard:
                sharding-column: order_id
                sharding-algorithm-name: order-table-inline
        sharding-algorithms:
          order-db-inline:
            type: INLINE
            props:
              algorithm-expression: ds${user_id % 2}
          order-table-inline:
            type: INLINE
            props:
              algorithm-expression: t_order_${order_id % 2}

        # 绑定表：订单和订单详情用相同分片策略
        binding-tables:
          - t_order,t_order_item
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 用ShardingSphere分表，配个分片规则就行 |
| 高级开发 | ShardingSphere-JDBC做客户端分片，性能好，不用Proxy |
| 架构师 | 中间件选型看三个维度（性能损耗/运维复杂度/语言支持），ShardingSphere-JDBC性能损耗<5%因为无网络跳转。工作原理五步（解析→路由→改写→执行→归并），跨分片归并要注意AVG不能简单平均。绑定表配置保证关联数据同分片，避免笛卡尔积查询 |

**补足方向**：被问"ShardingSphere的工作原理"，要画出五步流程并解释每步做什么，尤其是归并阶段——跨分片的排序/分页/聚合怎么做。

## 题目五：跨分片查询 — 分库分表后最头疼的问题

### 跨分片查询的四种场景

```
场景1：不带分片键的查询 — 广播全表扫描

  SELECT * FROM t_order WHERE status = 'PAID' AND create_time > '2026-06-01'

  没有分片键 → 不知道数据在哪个分片 → 所有分片都查
  ┌────────────────────────────────────────────────────────┐
  │  t_order_0 → 查到3条                                   │
  │  t_order_1 → 查到5条                                   │
  │  t_order_2 → 查到2条                                   │
  │  t_order_3 → 查到4条                                   │
  │                     ↓ 归并                              │
  │  合并结果：14条                                         │
  │                                                         │
  │  问题：4个分片全查，RT=最慢的分片RT                     │
  │  16个分片 → 16次DB查询 → RT可能从10ms涨到100ms         │
  └────────────────────────────────────────────────────────┘

场景2：跨分片JOIN — 不可能直接join

  SELECT o.*, i.* FROM t_order o
  JOIN t_order_item i ON o.order_id = i.order_id
  WHERE o.user_id = 1001

  如果order和order_item不在同一分片 → 无法join
  ┌────────────────────────────────────────────────────────┐
  │  解法1：绑定表（推荐）                                  │
  │    order和order_item用相同分片策略 → 同分片 → 本地join  │
  │    这就是前面说的"业务相关性"原则                      │
  │                                                         │
  │  解法2：应用层组装                                      │
  │    先查order → 拿到order_id → 再查order_item           │
  │    两次查询 + 内存组装                                  │
  │    适合分片策略不同的表                                 │
  │                                                         │
  │  解法3：宽表冗余                                        │
  │    把常用join字段冗余到主表                             │
  │    order表冗余item_name → 不用join了                   │
  │    适合join字段不多的场景                               │
  └────────────────────────────────────────────────────────┘

场景3：跨分片排序 — 每个分片局部排序 + 内存归并

  SELECT * FROM t_order ORDER BY create_time DESC LIMIT 10

  ┌────────────────────────────────────────────────────────┐
  │  每个分片取top 10：                                     │
  │    t_order_0 → [10,8,6,4,2,...]                        │
  │    t_order_1 → [9,7,5,3,1,...]                         │
  │    t_order_2 → [11,3,1,...]                            │
  │    t_order_3 → [8,6,2,...]                             │
  │                                                         │
  │  内存归并排序：取全局top 10 → [11,10,9,8,8,7,6,6,5,4] │
  │                                                         │
  │  问题：深度分页                                         │
  │    LIMIT 100000, 10 → 每个分片要取100010条              │
  │    4个分片 × 100010条 = 40万条数据加载到内存            │
  │    → OOM风险                                           │
  │                                                         │
  │  解法：禁止深度分页，用游标翻页                         │
  │    WHERE create_time < 上一页最后一条的时间 LIMIT 10    │
  │    每次只取10条，不会OOM                                │
  └────────────────────────────────────────────────────────┘

场景4：跨分片聚合 — 每个分片局部聚合 + 全局二次聚合

  SELECT COUNT(*), SUM(amount), AVG(amount)
  FROM t_order WHERE status = 'PAID'

  ┌────────────────────────────────────────────────────────┐
  │  每个分片局部聚合：                                     │
  │    t_order_0 → COUNT=100, SUM=50000, AVG=500            │
  │    t_order_1 → COUNT=200, SUM=80000, AVG=400            │
  │                                                         │
  │  全局二次聚合：                                         │
  │    COUNT = 100 + 200 = 300       ← 直接求和           │
  │    SUM   = 50000 + 80000 = 130000 ← 直接求和          │
  │    AVG   = 130000 / 300 = 433.3  ← 不能平均各分片AVG! │
  │                                                         │
  │  陷阱：AVG不能(500+400)/2=450 → 错误！                 │
  │    必须用全局SUM/全局COUNT重新计算                     │
  └────────────────────────────────────────────────────────┘
```

### 跨分片查询的通用解法

```
  ┌────────────────────────────────────────────────────────┐
  │  L1 避免跨分片（最好）                                  │
  │    分片键选对 → 70%+的查询精准路由                      │
  │    绑定表 → 关联数据同分片，本地join                   │
  │    基因法 → 多维度查询也能部分路由                     │
  │                                                         │
  │  L2 异步索引（推荐）                                    │
  │    MySQL分片存储 + ES建索引                             │
  │    按非分片键查询 → 先查ES得到主键 → 再查MySQL        │
  │    适合：管理后台、多条件筛选                           │
  │                                                         │
  │  L3 宽表冗余                                            │
  │    把join字段冗余到主表，减少跨表join                  │
  │    代价：数据冗余、双写一致性                           │
  │                                                         │
  │  L4 广播表                                             │
  │    小表（配置表/字典表）每个分片都存一份                │
  │    写入时广播到所有分片，读取时本地读取                 │
  │    适合：数据量小、变更少、频繁join的表                │
  │                                                         │
  │  L5 游标翻页                                            │
  │    禁止LIMIT offset,N → 改用WHERE id > 上次最大id     │
  │    避免深度分页的内存爆炸                               │
  └────────────────────────────────────────────────────────┘

  面试核心句：
  "跨分片查询的核心策略是'能避免就避免，不能避免就缩小范围'。
   分片键选对避免70%的跨分片，ES索引解决非分片键查询，
   绑定表解决关联查询，广播表解决小表join。
   深度分页用游标翻页，聚合注意AVG不能简单平均。"
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 分表后跨表查询用代码拼 |
| 高级开发 | 跨分片查询会全表扫描，用ES做索引 |
| 架构师 | 跨分片查询五层解法（避免→ES索引→宽表冗余→广播表→游标翻页），每种场景对应不同解法：无分片键→ES索引，跨表join→绑定表/宽表，深度分页→游标翻页，聚合→SUM/COUNT重算AVG。核心思维和Day2五层漏斗一致：每层覆盖不同场景 |

**补足方向**：面试时被问"分库分表后查询怎么办"，不要只说"用ES"，要说分层策略——"核心查询靠分片键精准路由，非分片键走ES索引，关联查询用绑定表，深度分页改游标翻页"。

## Day1核心总结

```
Day1的五个知识点形成一条主线：

  为什么要分 → 单库的六大瓶颈（连接/QPS/数据量/写/DDL/主从延迟）

  用什么工具 → ShardingSphere-JDBC（五步工作原理）
```
