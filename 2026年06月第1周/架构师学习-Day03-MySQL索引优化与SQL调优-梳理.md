## B+树：为什么是数据库索引的标配

```
核心逻辑：磁盘IO是瓶颈 → 减少IO是目标 → B+树最优

  三大优势量化：
    1. 矮：3层存2190万行，3次IO定位任意行
       量化：16KB页，BIGINT主键8B+指针6B=14B/条目
         非叶节点1170个指针，叶子16行/页
         3层=1170²×16≈2190万行

    2. 范围：叶子双向链表，BETWEEN只需顺序扫描
       vs B树需要跨层回溯（随机IO）

    3. 稳定：所有数据在叶子，每次查询都是3次IO
       vs B树可能2次也可能3次（性能不可预测）

  聚簇索引 vs 二级索引：
    聚簇索引 = 数据组织方式（叶子存完整行）
    二级索引 = 查找加速（叶子存主键值，需回表）
    回表 = 二级索引的核心代价
    覆盖索引 = 消除回表的手段

  自增主键为什么优于UUID：
    自增：顺序写入→无页分裂→4-8字节
    UUID：随机写入→频繁页分裂→36字节
    二级索引引用主键→主键越大索引越大
```

## 索引优化：四把斧头

```
  ┌──────────┬──────────────────────────────────────────┐
  │  手段     │  核心价值                                │
  ├──────────┼──────────────────────────────────────────┤
  │  最左前缀 │  联合索引列顺序决定索引是否可用          │
  │           │  三原则：等值在前/区分度高在前/高频组合优先│
  ├──────────┼──────────────────────────────────────────┤
  │  覆盖索引 │  消除回表，从100次IO降到1次              │
  │           │  代价：写入多维护索引列                   │
  ├──────────┼──────────────────────────────────────────┤
  │  索引下推 │  引擎层提前过滤，减少90%回表             │
  │  (ICP)    │  前提：条件列在联合索引中                │
  ├──────────┼──────────────────────────────────────────┤
  │  索引跳跃 │  8.0+跳过前缀列，对低区分度前缀有用     │
  │  (Skip)   │  例：索引(性别,年龄)，WHERE 年龄>25     │
  └──────────┴──────────────────────────────────────────┘

  关键认知：
    索引不是越多越好——每多一个索引INSERT多维护一棵B+树
    写入密集表3-4个索引，查询密集表5-7个
    区分度<0.1的列不单独建索引
```

## EXPLAIN：五步诊断法

```
  Step 1  type → 有没有ALL/index（红灯）
  Step 2  key+key_len → 走了哪个索引、联合索引用了几列
  Step 3  rows×filtered → 实际扫描量（filtered=0.01%说明索引选择率差）
  Step 4  Extra → 三个危险信号
            Using temporary → GROUP BY/DISTINCT临时表
            Using filesort → ORDER BY没走索引
            Using join buffer → JOIN无索引
  Step 5  优化前后对比 → 量化提升

  key_len反推联合索引使用列数：
    int=4B, bigint=8B, varchar(N)=N×3+2B(utf8), date=3B, datetime=8B
    例：idx(a_int, b_int, c_int)，key_len=8 → 用了a+b
```

## 慢查询治理：六层递进

```
  ┌─────────────────────────────────────────────────────────┐
  │  L1 加索引      → 成本最低，可能从500万行→50行          │
  │  L2 覆盖索引    → 消除回表，Using index                  │
  │  L3 SQL改写     → 不改业务逻辑改写SQL                    │
  │  L4 反范式冗余  → 空间换时间，读多写少场景               │
  │  L5 缓存        → DB外优化，命中率95%                    │
  │  L6 读写分离/   → Day1/Day2方案，最后手段                │
  │     分库分表                                              │
  └─────────────────────────────────────────────────────────┘

  和Day1五层前置方案完全对齐：
    Day1 L1 SQL优化+索引  ← Day3 L1+L2+L3
    Day1 L2 缓存          ← Day3 L5
    Day1 L3 读写分离      ← Day3 L6（读分离）
    Day1 L4 数据归档      ← Day3 L3+L4
    Day1 L5 分库分表      ← Day3 L6（分片）

  深度分页三方案：
    游标翻页（推荐）→ WHERE id > 上次最大id LIMIT 10
    延迟关联        → 子查询覆盖索引取id，外层回表
    业务限制        → 最多100页，产品引导加搜索条件

  COUNT优化：
    COUNT(*) = COUNT(1)（优化器等价处理）
    精确总数：计数表 + 应用层维护
    近似总数：SHOW TABLE STATUS
    架构师问：产品真需要精确数吗？"共约1万条"够不够？
```

## 索引设计：三维权衡

```
  查询频率 × 写入代价 × 存储成本

  决策流程：
    ① 分析查询场景频率分布（80% / 15% / 5%）
    ② 高频查询优先覆盖（联合索引+覆盖索引一箭双雕）
    ③ 控制索引总量5-7个以内
    ④ 低频查询走全表扫或ES
    ⑤ 定期清理无用索引（sys.schema_unused_indexes）

  实战三场景：
    订单表（高写+多维查）：
      3个索引，用户维度覆盖80%查询，商户维度15%
      不给运营后台5%查询建索引

    秒杀商品表（高并发读+低写）：
      3个索引，核心是goods_id定位+Redis预扣
      DB索引只做最终校验

    用户表（读写均衡+多维查）：
      3个索引，phone唯一索引+city_age联合
      name模糊搜索走ES

  面试核心句：
    "索引设计不是给每个WHERE条件建索引，
     是分析查询频率分布，高频用联合索引覆盖，
     低频全表扫或走ES，控制总量5-7个以内，
     因为每个索引都是写入时的B+树维护成本"
```
