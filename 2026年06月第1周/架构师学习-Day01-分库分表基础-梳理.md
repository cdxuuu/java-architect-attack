# Day 1：分库分表基础 — 架构师角度梳理

## 核心定位

Day1的主题一句话概括：**分库分表是最后手段，不是第一选择——先穷尽低成本方案，再上分库分表；分片键决定成败，跨分片查询是核心难题**。

---

## 拆分时机：四维度判断 + 五层前置方案

```
不是"数据多了就拆"，是"先试低成本方案，都解决不了才拆"

  前置方案优先级（成本从低到高）：
  ┌─────────────────────────────────────────────────────────┐
  │  L1 SQL优化+索引     → 成本最低，可能立竿见影          │
  │  L2 缓存            → 减少DB读压力                     │
  │  L3 读写分离        → 读能力3x                          │
  │  L4 数据归档        → 减少主表数据量                    │
  │  L5 分库分表        → 最后手段，成本最高                │
  └─────────────────────────────────────────────────────────┘

  四维度判断阈值：
    数据量：单表 > 2000万行 → 必须分表
    QPS：单库 > 5000 → 必须分库
    连接数：活跃 > 3000 → 必须分库
    DDL：ALTER > 5分钟 → 必须分表
```

## 拆分策略：垂直解决边界，水平解决容量

```
  垂直拆分                     水平拆分
  ┌──────────────────────┐   ┌──────────────────────┐
  │ 按业务域拆库          │   │ 按规则拆表            │
  │ 用户库/订单库/支付库  │   │ 同表数据分散多库多表  │
  │                      │   │                      │
  │ 解决：业务耦合        │   │ 解决：数据量+QPS瓶颈  │
  │ 代价：跨库join消失    │   │ 代价：路由+跨片+事务  │
  │                      │   │                      │
  │ 先做（业务隔离）      │   │ 后做（性能扩展）      │
  └──────────────────────┘   └──────────────────────┘

  路由策略：
    取模(Hash)  → 均匀但扩容难 → 数据量稳定场景
    范围(Range) → 扩容简单但热点 → 时间序列场景
    一致性Hash  → 折中但复杂 → 弹性扩缩场景
```

## 分片键：决定分库分表成败的决策

```
分片键选型三原则：

  ① 查询覆盖度：尽量多的查询能定位到具体分片
     量化：70%查询带order_id → 选order_id

  ② 数据均匀性：数据不能倾斜到某几个分片
     反例：按merchant_id分片 → 大商家100万vs小商家1000 → 倾斜

  ③ 业务相关性：关联数据尽量同一分片
     订单+订单详情 → 绑定表，同分片策略

  多维度查询解法：
    基因法：order_id末4bit嵌入user_id基因
      → 按user_id查只扫1/16分片
    ES索引：MySQL分片存储+ES建索引
      → 非分片键查询先查ES再查MySQL
    冗余表：双写映射关系
      → 一致性问题
```

## ShardingSphere：选型与原理

```
选型：ShardingSphere-JDBC

  为什么选JDBC不选Proxy？
    性能：<5%损耗（无网络跳转）
    运维：jar包随应用部署（无独立进程）
    稳定：无Proxy单点
    限制：只支持Java（但项目就是Java）

  五步工作原理：
    ① SQL解析 → 提取表名+条件
    ② 路由    → 分片算法确定目标分片
    ③ SQL改写 → 表名替换
    ④ 执行    → 发送到目标分片
    ⑤ 归并    → 排序/分页/聚合合并

  关键配置：
    actual-data-nodes → 分片节点
    database-strategy → 分库策略
    table-strategy    → 分表策略
    binding-tables    → 绑定表
```

## 跨分片查询：五层解法

```
  L1 避免跨分片（最好）   → 分片键选对+绑定表+基因法
  L2 ES异步索引（推荐）   → 非分片键查询先查ES
  L3 宽表冗余             → 减少跨表join
  L4 广播表              → 小表每个分片一份
  L5 游标翻页             → 禁止深度分页LIMIT offset,N

  四种场景的解法：
    无分片键 → ES索引
    跨表join → 绑定表/宽表
    深度分页 → 游标翻页
    聚合     → SUM/COUNT重算AVG
```
