# Day 2：读写分离与主从复制 — binlog三种模式/主从延迟根因/半同步与MGR/强制读主（练习整理）

## 题目一：MySQL主从复制原理 — binlog三种模式与复制流程

### 主从复制的完整流程

```
MySQL主从复制是基于binlog的异步复制

  ┌──────────────────────────────────────────────────────────┐
  │                       Master主库                          │
  │                                                           │
  │  ① 客户端写入：UPDATE account SET balance=900 WHERE id=1 │
  │                            │                              │
  │                            ▼                              │
  │  ② InnoDB写入：                                           │
  │     a. 写undo log（用于回滚）                             │
  │     b. 修改内存中的数据页                                  │
  │     c. 写redo log（崩溃恢复用）                            │
  │     d. 写binlog（主从复制用）                              │
  │                            │                              │
  │                            ▼                              │
  │  ③ 提交事务：两阶段提交                                   │
  │     a. redo log prepare                                  │
  │     b. binlog write + fsync                              │
  │     c. redo log commit                                    │
  │                            │                              │
  │                            ▼ binlog dump线程              │
  │                                                           │
  └──────────────────────┬───────────────────────────────────┘
                         │ 通过网络推送binlog event
                         ▼
  ┌──────────────────────────────────────────────────────────┐
  │                       Slave从库                           │
  │                                                           │
  │  ④ IO线程：接收主库的binlog event                         │
  │              ↓                                            │
  │              写入relay log（中继日志）                    │
  │                                                           │
  │  ⑤ SQL线程：读取relay log                                 │
  │              ↓                                            │
  │              执行SQL → 更新本地数据                       │
  │                                                           │
  │  5.7之前：单SQL线程串行执行 → 主从延迟大                   │
  │  5.7之后：基于group commit的并行复制 → 延迟降低           │
  │  8.0：基于WriteSet的并行复制 → 延迟进一步降低             │
  └──────────────────────────────────────────────────────────┘
```

### binlog三种格式对比

```
binlog是主从复制的关键，格式选择直接影响一致性和性能

  ┌──────────────────────────────────────────────────────────┐
  │  Statement格式（SBR）— 记录SQL语句                       │
  │                                                           │
  │  示例：UPDATE account SET balance=balance-100 WHERE id=1 │
  │  binlog内容：原样记录SQL语句                              │
  │                                                           │
  │  优点：                                                   │
  │    日志量小（几十字节）                                    │
  │    主从带宽消耗低                                          │
  │                                                           │
  │  致命问题：非确定性函数 → 主从数据不一致                  │
  │    INSERT INTO log VALUES(NOW(), UUID(), RAND())         │
  │    主库执行时间：12:00:00.001                              │
  │    从库执行时间：12:00:00.123 → 时间不同 → 数据不一致     │
  │                                                           │
  │  适用：MySQL 5.0之前的默认                                │
  ├──────────────────────────────────────────────────────────┤
  │  Row格式（RBR）— 记录行变更                               │
  │                                                           │
  │  示例：UPDATE account SET balance=balance-100 WHERE id=1 │
  │  binlog内容：                                             │
  │    Before image: id=1, balance=1000                       │
  │    After image:  id=1, balance=900                        │
  │                                                           │
  │  优点：                                                   │
  │    严格一致：主从执行结果完全相同                          │
  │    支持闪回（pt-online-schema-change/Binlog2sql）         │
  │    无函数歧义问题                                          │
  │                                                           │
  │  问题：                                                   │
  │    日志量大：UPDATE一行=100字节，                         │
  │    UPDATE 100万行 = 100MB binlog                          │
  │    全表更新会产生海量binlog                                │
  │                                                           │
  │  适用：金融场景必选！MySQL 5.7+默认                       │
  ├──────────────────────────────────────────────────────────┤
  │  Mixed格式 — 混合模式                                     │
  │                                                           │
  │  规则：MySQL自动选择                                       │
  │    安全SQL → Statement格式                                │
  │    有歧义的SQL → Row格式                                  │
  │                                                           │
  │  优点：兼顾日志量和一致性                                  │
  │  问题：判断逻辑复杂，存在漏判风险                          │
  │                                                           │
  │  适用：折中方案，但生产推荐Row                            │
  └──────────────────────────────────────────────────────────┘

  架构师选型：金融/订单场景必须Row
    为什么？一致性高于一切
    日志量大 → 用binlog压缩、加大disk、加快网络
    不能用Statement → 函数差异导致主从不一致 = 资金错误
```

### 复制延迟的根因分析

```
为什么主从会延迟？三个层次：

  ┌──────────────────────────────────────────────────────────┐
  │  L1 网络延迟（毫秒级，正常）                              │
  │    主库写完binlog → IO线程接收                            │
  │    同机房 < 1ms，跨机房 5-20ms                            │
  │                                                           │
  │  L2 IO线程瓶颈（少见）                                    │
  │    单IO线程，写relay log                                  │
  │    一般不会瓶颈（顺序写）                                  │
  │                                                           │
  │  L3 SQL线程瓶颈（主要原因）                               │
  │    5.7之前：单SQL线程串行执行                             │
  │    主库并发写入1000 TPS → 从库串行回放                    │
  │    → 高峰期延迟可达分钟级                                  │
  │                                                           │
  │    5.7+：基于group commit的并行复制                       │
  │      同一个group commit的事务可以并行回放                 │
  │      slave_parallel_workers=N → N个worker并行             │
  │                                                           │
  │    8.0+：基于WriteSet的并行复制                           │
  │      不同行的事务可以并行（即使不在同一group）            │
  │      → 进一步降低延迟                                      │
  └──────────────────────────────────────────────────────────┘

  实际场景的延迟数字：
  ┌──────────────────┬─────────────┬─────────────────────────┐
  │  场景             │  典型延迟    │  原因                    │
  ├──────────────────┼─────────────┼─────────────────────────┤
  │  正常情况         │  10-100ms   │  网络+IO线程              │
  │  高峰期写入       │  1-5秒      │  SQL线程回放跟不上        │
  │  大事务执行       │  10秒-分钟  │  单事务SQL线程独占        │
  │  DDL执行          │  分钟-小时   │  ALTER TABLE阻塞回放      │
  │  跨机房复制       │  5-50ms     │  网络物理距离              │
  └──────────────────┴─────────────┴─────────────────────────┘
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 主从复制就是数据同步 |
| 高级开发 | binlog同步，有IO线程和SQL线程 |
| 架构师 | binlog三种格式（Statement日志小但函数歧义/Row严格一致但日志大/Mixed折中）金融必选Row。复制流程：主库写binlog→IO线程接收写relay log→SQL线程回放。延迟根因主要在SQL线程，5.7基于group commit并行/8.0基于WriteSet并行 |

**补足方向**：面试时被问"主从延迟怎么办"，不要只说"加机器"，要说根因——"延迟主要在SQL线程串行回放，5.7+用slave_parallel_workers开并行复制，8.0用WriteSet进一步降低延迟。同时避免大事务和DDL，这些会独占SQL线程"。

## 题目二：主从延迟的解决方案 — 从根因到治理

### 延迟问题的五大场景

```
场景1：写后立即读 — 用户体验问题

  用户更新资料 → 立即查询资料 → 读到旧数据
  原因：写主库后，读从库时主从还没同步

  ┌──────────────────────────────────────────────────────────┐
  │  时间线：                                                 │
  │  T0: 用户更新头像 → UPDATE → 主库执行成功                 │
  │  T1: 用户刷新页面 → SELECT → 从库还是旧头像               │
  │  T2: 用户：「我头像没改成功？」 → 又点了一次更新           │
  │                                                           │
  │  解法：强制读主（后面详细讲）                              │
  └──────────────────────────────────────────────────────────┘

场景2：大事务 — SQL线程被独占

  DELETE FROM order WHERE create_time < '2024-01-01' LIMIT 1000万

  ┌──────────────────────────────────────────────────────────┐
  │  问题：                                                   │
  │    主库10分钟执行完 → 一条binlog event 1000万行          │
  │    从库SQL线程单线程回放 → 10分钟内不能处理其他事件       │
  │    → 主从延迟10分钟                                       │
  │                                                           │
  │  解法：分批执行                                            │
  │    while True:                                            │
  │      DELETE WHERE create_time < '2024-01-01' LIMIT 1000   │
  │      sleep(0.1)                                           │
  │    一次1000行 → 1万次小事务 → 从库可并行回放              │
  └──────────────────────────────────────────────────────────┘

场景3：DDL执行 — 阻塞复制

  ALTER TABLE order ADD COLUMN new_field VARCHAR(100)

  ┌──────────────────────────────────────────────────────────┐
  │  问题：                                                   │
  │    DDL在主库执行2小时（5000万行表）                       │
  │    DDL也写binlog → 从库回放也要2小时                      │
  │    回放期间，后续所有事件等待 → 延迟2小时                 │
  │                                                           │
  │  解法：                                                   │
  │    1. 用gh-ost/pt-online-schema-change做Online DDL       │
  │       原理：建新表→拷贝数据+追增量→切换名称              │
  │       从库延迟控制在秒级                                   │
  │    2. 业务低峰期执行，避开高峰                            │
  └──────────────────────────────────────────────────────────┘

场景4：从库自身压力大 — 查询挤占资源

  ┌──────────────────────────────────────────────────────────┐
  │  问题：                                                   │
  │    从库不只是复制，还要服务读请求                          │
  │    复杂报表查询 → 占用CPU/IO → SQL线程被挤占             │
  │    → 主从延迟                                              │
  │                                                           │
  │  解法：                                                   │
  │    1. 读写分离的从库 vs 报表查询的从库分开                │
  │    2. 报表查询走专门的从库（甚至离线库）                  │
  │    3. OLTP从库不跑长查询，设置max_execution_time          │
  └──────────────────────────────────────────────────────────┘

场景5：从库硬件配置低 — 跟不上主库

  ┌──────────────────────────────────────────────────────────┐
  │  问题：                                                   │
  │    主库32核256G SSD，从库8核32G HDD                       │
  │    主库的IOPS和CPU能力远超从库                            │
  │    → 从库回放速度跟不上主库写入速度                       │
  │                                                           │
  │  解法：                                                   │
  │    从库配置必须 ≥ 主库                                     │
  │    至少：CPU相同、内存相同、磁盘IOPS相同                  │
  └──────────────────────────────────────────────────────────┘
```

### 延迟监控与告警

```
监控指标：
  ┌──────────────────────────────────────────────────────────┐
  │  Seconds_Behind_Master — 从库show slave status获取        │
  │    原理：当前时间 - SQL线程正在执行的binlog event时间戳  │
  │    优点：直观                                              │
  │    陷阱：网络断开期间值不准（被冻结）                      │
  │                                                           │
  │  心跳表方案（推荐）：                                      │
  │    主库每秒UPDATE heartbeat_table SET ts=NOW()           │
  │    从库定期SELECT ts → 计算delay = NOW() - ts            │
  │    优点：精确，网络断开能立即发现                          │
  │    缺点：需要维护心跳表                                    │
  └──────────────────────────────────────────────────────────┘

告警阈值：
  ┌──────────────────┬──────────────┬─────────────────────┐
  │  延迟级别         │  阈值         │  处理动作            │
  ├──────────────────┼──────────────┼─────────────────────┤
  │  正常             │  < 1秒        │  无操作              │
  │  关注             │  1-5秒        │  记录，分析趋势      │
  │  告警             │  5-30秒       │  IM告警，排查原因    │
  │  严重             │  30秒-5分钟   │  电话告警，介入处理  │
  │  紧急             │  > 5分钟      │  切流到主库          │
  └──────────────────┴──────────────┴─────────────────────┘
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 主从延迟就重启从库 |
| 高级开发 | 开并行复制slave_parallel_workers |
| 架构师 | 延迟根因五大场景（写后读/大事务/DDL/从库压力/硬件低配），对应五种解法（强制读主/分批/Online DDL/读库分离/硬件对齐）。监控用心跳表方案（Seconds_Behind_Master在网络断开时不准）。告警阈值分级，5分钟以上切流到主库 |

**补足方向**：被问"主从延迟怎么处理"要分场景：临时延迟监控告警等待恢复；长期延迟分析根因（大事务/DDL/硬件）；严重延迟切流到主库保业务。

## 题目三：读写分离架构 — 四种实现方式与权衡

### 四种读写分离实现方式

```
方式1：应用层路由（手动）

  ┌──────────────────────────────────────────────────────────┐
  │  原理：代码中手动指定走主库还是从库                       │
  │                                                           │
  │  @Service                                                 │
  │  public class OrderService {                              │
  │      @Resource MasterDataSource master;                  │
  │      @Resource SlaveDataSource slave;                    │
  │                                                           │
  │      public Order create(...) {                          │
  │          return masterMapper.insert(...);                 │
  │      }                                                    │
  │      public Order query(Long id) {                       │
  │          return slaveMapper.selectById(id);              │
  │      }                                                    │
  │  }                                                        │
  │                                                           │
  │  优点：精确控制，性能最好                                  │
  │  缺点：代码侵入大，每个方法都要选数据源                    │
  └──────────────────────────────────────────────────────────┘

方式2：注解+AOP（推荐）

  ┌──────────────────────────────────────────────────────────┐
  │  原理：方法上加注解，AOP切换数据源                        │
  │                                                           │
  │  @Service                                                 │
  │  public class OrderService {                              │
  │      @DataSource("master")                               │
  │      public Order create(...) { ... }                    │
  │                                                           │
  │      @DataSource("slave")                                │
  │      public Order query(Long id) { ... }                 │
  │  }                                                        │
  │                                                           │
  │  原理：                                                   │
  │    AOP拦截 → ThreadLocal保存数据源标识                   │
  │    AbstractRoutingDataSource根据ThreadLocal路由           │
  │                                                           │
  │  优点：注解清晰，代码侵入小                                │
  │  缺点：仍需要在每个方法加注解                              │
  └──────────────────────────────────────────────────────────┘

方式3：自动路由（基于SQL）

  ┌──────────────────────────────────────────────────────────┐
  │  原理：解析SQL，SELECT走从库，其他走主库                  │
  │                                                           │
  │  ShardingSphere-JDBC的读写分离配置：                      │
  │  spring.shardingsphere.rules.readwrite-splitting:        │
  │    data-sources:                                          │
  │      ms_ds:                                               │
  │        type: Static                                       │
  │        props:                                             │
  │          write-data-source-name: master                   │
  │          read-data-source-names: slave0,slave1            │
  │        load-balancer-name: round_robin                    │
  │                                                           │
  │  优点：零代码侵入                                          │
  │  缺点：                                                    │
  │    1. SELECT后立即查可能读到旧数据                        │
  │    2. 事务内必须强制主库                                   │
  └──────────────────────────────────────────────────────────┘

方式4：Proxy中间件（重场景）

  ┌──────────────────────────────────────────────────────────┐
  │  原理：应用 → Proxy（如ShardingSphere-Proxy/MaxScale）   │
  │             → Proxy路由到Master/Slave                     │
  │                                                           │
  │  优点：                                                   │
  │    应用完全无感知                                          │
  │    DBA统一管理                                            │
  │    支持多语言                                              │
  │                                                           │
  │  缺点：                                                   │
  │    多一跳网络延迟                                          │
  │    Proxy本身是瓶颈/单点                                   │
  │                                                           │
  │  适用：多语言团队、需要DBA集中管控                        │
  └──────────────────────────────────────────────────────────┘
```

### 读写分离架构的关键问题

```
问题1：主从延迟的处理

  ┌──────────────────────────────────────────────────────────┐
  │  关键原则：写后立即读必须走主库                            │
  │                                                           │
  │  场景示例：                                                │
  │    用户修改资料 → 写主库 → 立即查询资料                   │
  │    走从库 → 读到旧数据 → 用户体验差                       │
  │                                                           │
  │  解法（后面强制读主章节详细讲）：                          │
  │    1. 注解强制主库：@DataSource("master")                │
  │    2. 事务内自动主库                                       │
  │    3. 写后N秒内强制主库                                    │
  │    4. ClientSide GTID跟踪                                 │
  └──────────────────────────────────────────────────────────┘

问题2：事务必须走主库

  ┌──────────────────────────────────────────────────────────┐
  │  事务内的SELECT和UPDATE必须在同一个连接                   │
  │  如果SELECT走从库，UPDATE走主库 → 数据不一致              │
  │                                                           │
  │  解决：开启事务后所有操作走主库                            │
  │  ShardingSphere、MyBatis-Plus等都内置此规则               │
  └──────────────────────────────────────────────────────────┘

问题3：从库负载均衡

  ┌──────────────────────────────────────────────────────────┐
  │  多从库时的负载均衡策略：                                  │
  │                                                           │
  │  轮询（Round Robin）：                                    │
  │    请求1→slave0, 请求2→slave1, 请求3→slave0...           │
  │    优点：均匀                                              │
  │    缺点：不考虑从库负载差异                                │
  │                                                           │
  │  权重（Weight）：                                          │
  │    slave0权重70，slave1权重30 → 7:3的请求比例             │
  │    适用：从库配置不同                                      │
  │                                                           │
  │  最小连接数（Least Connections）：                        │
  │    选当前连接数最少的从库                                  │
  │    适用：长连接场景                                        │
  │                                                           │
  │  最低延迟（Least Latency）：                              │
  │    选主从延迟最小的从库                                    │
  │    适用：对实时性要求高的场景                              │
  └──────────────────────────────────────────────────────────┘

问题4：从库故障转移

  ┌──────────────────────────────────────────────────────────┐
  │  某个从库挂了怎么办？                                      │
  │                                                           │
  │  方案1：客户端健康检查                                     │
  │    每30秒ping一次，挂了从负载均衡列表剔除                 │
  │    优点：简单                                              │
  │    缺点：故障感知延迟                                      │
  │                                                           │
  │  方案2：连接失败重试                                       │
  │    SELECT失败 → 自动切换到其他从库重试                    │
  │    优点：实时                                              │
  │    缺点：首次失败仍然返回错误                              │
  │                                                           │
  │  方案3：所有从库都挂了                                     │
  │    自动降级走主库（注意主库压力！）                        │
  │    告警通知DBA                                            │
  └──────────────────────────────────────────────────────────┘
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 用ShardingSphere做读写分离 |
| 高级开发 | 注解+AOP切换数据源，事务必须走主库 |
| 架构师 | 四种实现方式（手动/注解/SQL自动/Proxy）权衡，写后读必须主库（注解+事务+写后N秒+GTID四种方案），多从库负载均衡四策略（轮询/权重/最小连接/最低延迟），故障转移要考虑全挂场景降级主库 |

**补足方向**：被问"读写分离架构怎么设计"要给完整方案——"应用层用注解@DataSource强制走主库，框架层ShardingSphere自动SELECT走从库，负载均衡用加权轮询（从库配置不同），主从延迟监控用心跳表，全挂时自动降级主库并告警"。

## 题目四：半同步复制与MGR — 一致性的进阶方案

### 异步/半同步/全同步对比

```
MySQL复制的三种模式：

  ┌──────────────────────────────────────────────────────────┐
  │  异步复制（Async）— 默认                                  │
  │                                                           │
  │  流程：                                                   │
  │    1. 主库写binlog → 返回客户端成功                       │
  │    2. 异步推送给从库（不等确认）                          │
  │                                                           │
  │  优点：性能最好（主库不等从库）                            │
  │  缺点：主库挂了 → 已提交的事务可能丢失（从库还没收到）   │
  │  适用：非金融场景                                          │
  ├──────────────────────────────────────────────────────────┤
  │  半同步复制（Semi-sync）                                  │
  │                                                           │
  │  流程：                                                   │
  │    1. 主库写binlog                                        │
  │    2. 推送给从库                                          │
  │    3. 等至少1个从库ACK → 返回客户端成功                  │
  │    4. 其他从库异步继续                                    │
  │                                                           │
  │  优点：                                                   │
  │    至少有1个从库有完整数据                                │
  │    主库挂了切换到该从库 → 不丢数据                        │
  │                                                           │
  │  缺点：                                                   │
  │    主库写入延迟增加（等从库ACK）                          │
  │    从库都挂了 → 主库写入退化为异步（rpl_semi_sync_master_timeout）│
  │                                                           │
  │  适用：金融场景必选                                        │
  ├──────────────────────────────────────────────────────────┤
  │  全同步复制（Sync）                                       │
  │                                                           │
  │  流程：                                                   │
  │    1. 主库写binlog                                        │
  │    2. 推送给所有从库                                      │
  │    3. 等所有从库ACK → 返回客户端成功                     │
  │                                                           │
  │  优点：所有从库强一致                                      │
  │  缺点：                                                   │
  │    性能最差（任何一个从库慢就阻塞主库）                    │
  │    可用性差（任何从库挂了就不能写）                        │
  │                                                           │
  │  适用：几乎不用，太重了                                    │
  └──────────────────────────────────────────────────────────┘

  架构师选型：
    金融/订单 → 半同步（不丢数据 + 性能可接受）
    日志/统计 → 异步（性能优先）
    全同步 → 几乎不用，用MGR或Galera替代
```

### MGR（MySQL Group Replication）

```
MGR是MySQL官方的多主集群方案

  ┌──────────────────────────────────────────────────────────┐
  │  架构：                                                   │
  │                                                           │
  │  ┌────────┐  ┌────────┐  ┌────────┐                      │
  │  │ Node1  │←→│ Node2  │←→│ Node3  │                      │
  │  │ Master │  │ Master │  │ Master │                      │
  │  └────────┘  └────────┘  └────────┘                      │
  │                                                           │
  │  特点：                                                   │
  │    1. 多主架构：每个节点都可读可写                        │
  │    2. 基于Paxos协议：保证强一致                            │
  │    3. 自动故障转移：节点挂了自动剔除                      │
  │    4. 自动数据补齐：新节点加入自动同步                    │
  │                                                           │
  │  写入流程：                                                │
  │    1. 客户端写到任意节点                                  │
  │    2. 节点广播事务到其他节点                              │
  │    3. 多数节点（N/2+1）确认 → 提交                       │
  │    4. 类似Raft的多数派写入                                │
  └──────────────────────────────────────────────────────────┘

  MGR的两种模式：

  ┌──────────────────────────────────────────────────────────┐
  │  单主模式（推荐）— Single-Primary                        │
  │                                                           │
  │  只有一个节点可写，其他节点只读                            │
  │  优点：                                                   │
  │    无冲突（所有写都到同一个节点）                          │
  │    Primary挂了自动选举新Primary                            │
  │  适用：替代传统主从+MHA                                   │
  ├──────────────────────────────────────────────────────────┤
  │  多主模式 — Multi-Primary                                │
  │                                                           │
  │  所有节点都可写                                            │
  │  优点：写入能力线性扩展                                    │
  │  缺点：                                                   │
  │    并发写同一行 → 冲突检测 → 部分事务回滚                │
  │    需要业务层避免冲突                                      │
  │  适用：分片明确的场景（不同分片写不同节点）                │
  └──────────────────────────────────────────────────────────┘

  MGR vs 半同步：

  ┌──────────────────┬───────────────────┬───────────────────┐
  │                  │  半同步           │  MGR              │
  ├──────────────────┼───────────────────┼───────────────────┤
  │  一致性          │  弱（主切换可能丢│  强（多数派一致） │
  │                  │  少量binlog）    │                   │
  │  自动故障转移    │  需MHA/ORC等工具 │  内置             │
  │  脑裂处理        │  需要外部检测     │  内置（多数派）   │
  │  部署复杂度      │  低               │  高               │
  │  性能            │  好               │  中（多数派写）    │
  │  成熟度          │  高               │  8.0+较成熟       │
  └──────────────────┴───────────────────┴───────────────────┘
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | MySQL复制是异步的 |
| 高级开发 | 半同步复制能保证至少一个从库有数据，金融场景用 |
| 架构师 | 三种复制模式权衡：异步性能好但可能丢数据/半同步金融必选/全同步几乎不用。半同步退化机制：从库都挂了变异步（rpl_semi_sync_master_timeout）。MGR是新一代方案，单主模式替代MHA+半同步，多主模式需业务避冲突。选型：日志异步/订单半同步/核心金融MGR单主 |

**补足方向**：被问"如何保证主从一致性"要分层——异步复制（默认）→ 半同步复制（金融场景，至少1从同步）→ MGR（强一致+自动故障转移）。说明每种方案的成本和收益。

## 题目五：强制读主与一致性保证 — 写后读问题的完整解决方案

### 写后读问题的本质

```
读写分离架构下，写主读从导致的一致性问题：

  ┌──────────────────────────────────────────────────────────┐
  │  时间线：                                                 │
  │  T0: 用户A 写入 → 主库执行UPDATE (balance=900)            │
  │  T1: 主库返回成功                                          │
  │  T2: 用户A 立即查询 → 路由到从库                          │
  │  T3: 从库返回旧数据 (balance=1000) ← 不一致！             │
  │  T4: 主从同步完成（100ms后）                              │
  │  T5: 从库数据变为新值 (balance=900)                       │
  │                                                           │
  │  问题影响：                                                │
  │    用户体验：修改后立即查询看不到变更                      │
  │    业务正确性：基于旧值的判断可能错误                      │
  │    严重场景：用户重复提交                                  │
  └──────────────────────────────────────────────────────────┘
```

### 四种强制读主方案

```
方案1：注解强制（最常用）

  ┌──────────────────────────────────────────────────────────┐
  │  @DataSource("master")                                   │
  │  @Override                                                │
  │  public Account queryByUserId(Long userId) {             │
  │      return accountMapper.selectByUserId(userId);        │
  │  }                                                        │
  │                                                           │
  │  适用：明确知道这个查询需要最新数据                        │
  │  缺点：开发要判断哪些查询走主库 → 容易遗漏                │
  └──────────────────────────────────────────────────────────┘

方案2：事务内强制（框架默认）

  ┌──────────────────────────────────────────────────────────┐
  │  @Transactional                                          │
  │  public void transfer(Long from, Long to, BigDecimal amt) {│
  │      // 事务内所有操作自动走主库                          │
  │      Account fromAcc = query(from);    // 走主库         │
  │      account.setBalance(...);                            │
  │      update(fromAcc);                  // 走主库         │
  │      // ...                                              │
  │  }                                                        │
  │                                                           │
  │  原理：开启事务时绑定一个连接，整个事务在同一连接执行     │
  │  优点：框架自动处理，不用开发判断                          │
  │  适用：所有事务场景                                        │
  └──────────────────────────────────────────────────────────┘

方案3：写后N秒强制（智能路由）

  ┌──────────────────────────────────────────────────────────┐
  │  原理：用户写操作后，N秒内的查询全部走主库                │
  │                                                           │
  │  实现：                                                   │
  │  1. 用户写入 → 在Redis记录 user:123:write_ts = NOW()     │
  │  2. 用户查询 → 检查 user:123:write_ts                    │
  │       如果在3秒内 → 走主库                                │
  │       如果超过3秒 → 走从库                                │
  │                                                           │
  │  N值选择：                                                │
  │    < 平均主从延迟 → 仍可能读旧数据                        │
  │    > 平均主从延迟 → 安全                                  │
  │    推荐：3-5秒（覆盖95%场景）                             │
  │                                                           │
  │  优点：用户体验好，自动处理                                │
  │  缺点：                                                   │
  │    部分流量打主库 → 主库压力增加                          │
  │    需要维护Redis状态                                       │
  └──────────────────────────────────────────────────────────┘

方案4：GTID跟踪（最严格）

  ┌──────────────────────────────────────────────────────────┐
  │  原理：基于全局事务ID GTID 精确跟踪同步进度                │
  │                                                           │
  │  流程：                                                   │
  │  1. 用户写入主库 → 主库返回GTID（如 server1:1-1000）     │
  │  2. 客户端记录这个GTID                                    │
  │  3. 用户查询 → 客户端检查从库的GTID                       │
  │       从库GTID ≥ 客户端GTID → 走从库（已同步）           │
  │       从库GTID < 客户端GTID → 走主库或等待               │
  │                                                           │
  │  优点：精确，无浪费                                        │
  │  缺点：                                                   │
  │    实现复杂，需要客户端配合                                │
  │    每次查询都要比较GTID → 增加复杂度                      │
  │                                                           │
  │  适用：对一致性要求极高的金融场景                          │
  └──────────────────────────────────────────────────────────┘
```

### 四种方案的选型决策

```
  ┌──────────────────┬──────────┬──────────┬──────────┐
  │  场景            │ 注解     │ 事务     │ N秒强制  │
  ├──────────────────┼──────────┼──────────┼──────────┤
  │  事务内查询      │  -       │  ✓       │  -       │
  │  写后立即读      │  ✓       │  -       │  ✓       │
  │  特定关键查询    │  ✓       │  -       │  -       │
  │  普通读多写少    │  -       │  -       │  ✓       │
  │  金融极致一致    │  -       │  -       │  GTID    │
  └──────────────────┴──────────┴──────────┴──────────┘

  推荐组合方案（生产实践）：
    基础：事务内自动主库（框架默认）
    补充：注解@DataSource("master")标记关键查询
    可选：写后3秒强制主库（用户中心、订单中心）
    极致：GTID跟踪（资金、支付）

  这和Day2纵深防御一脉相承：
    每种方案覆盖不同场景，多层叠加
```

### 与架构师水平的差距

| 层次 | 回答 |
|------|------|
| 开发 | 直接走主库就行了 |
| 高级开发 | 注解切换数据源，事务内自动走主库 |
| 架构师 | 四种方案分层（事务内自动+注解显式+写后N秒+GTID跟踪），按一致性要求逐层叠加。N秒方案的N要大于平均主从延迟，3-5秒覆盖95%场景。GTID方案最严格但实现复杂，金融场景才用。核心原则：不是越严格越好，是按场景选刚好够用的 |

**补足方向**：被问"读写分离怎么解决一致性"，要给分层方案——"基础事务内自动主库，关键查询用注解强制，用户中心用写后3秒强制（Redis记录写时间戳），资金场景才上GTID。不是一刀切，按数据敏感度分层"。
