## 差距一：锁的加锁规则不够精确 — 知道概念但说不清具体范围
> Day4发现

### 现状
知道锁加在索引上、三种锁的定义、和索引类型有关。但具体到"主键BETWEEN加什么锁""普通索引等值命中为什么不退化"说不准确。把锁的"粒度"（Record/Gap/Next-Key）和"模式"（X/S）混在一起。

### 架构师为什么需要
面试官问"SELECT FOR UPDATE WHERE id BETWEEN 10 AND 20加了什么锁"，答"记录锁+间隙锁"太模糊。要能逐行说出扫描过程和退化结果。线上遇到锁冲突排查也需要精确判断锁范围。说"普通索引不退化因为可能有重复值"比"普通索引会加gap lock"强十倍。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | InnoDB有行锁和表锁 |
| 高级开发 | 知道Record/Gap/Next-Key Lock，锁加在索引上 |
| 架构师 | 锁的行为由索引类型×查询类型×隔离级别三维度决定。唯一索引等值命中退化成Record Lock，普通索引不退化——根因是普通索引可能有重复值必须锁间隙防同值插入。无索引=锁全表。能逐行说出扫描过程和退化结果 |

### 怎么补
1. **记住退化规则速查表**：5种场景（唯一命中/唯一未命中/普通命中/普通未命中/无索引）
2. **记住退化根因**：唯一索引保证唯一→不需要锁间隙；普通索引可能重复→必须锁间隙
3. **面试时先说本质**："锁加在索引上，行为由索引类型决定"，再给速查表

## 差距二：隔离级别选型缺系统化框架 — 只想到一个维度
> Day4发现

### 现状
知道RR有间隙锁容易死锁，但说不出阿里改RC的完整理由。五个维度只能想到一个。不知道半一致性读。RC下必须留RR的场景完全空白。

### 架构师为什么需要
"为什么改RC"是面试高频题。只答"减少死锁"勉强及格，答不出五个维度（锁范围/死锁/主从/可见性直觉/半一致性读）露馅。更关键的是"什么场景必须RR"——答不出说明没有系统思考过选型。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | RR是默认的，RC有不可重复读 |
| 高级开发 | RR有间隙锁容易死锁，可以考虑RC |
| 架构师 | RC选型五个理由（锁范围小/死锁少/row格式消除复制风险/可见性符合直觉/半一致性读），RR必留两个场景（对账/唯一性校验+插入，但后者用唯一索引更可靠）。半一致性读是RC杀手级优势：UPDATE遇锁行→读最新版本→不匹配直接跳过不加锁。选型决策树：OLTP用RC+唯一索引兜底，报表对账用RR+WITH CONSISTENT SNAPSHOT |

### 怎么补
1. **记住五个维度**：锁范围/死锁/主从/可见性直觉/半一致性读
2. **半一致性读是加分项**：面试时主动提"RC有个杀手级特性叫半一致性读"
3. **RC下必须RR的场景**：对账一致性 + 唯一性校验（但推荐唯一索引替代）

## 差距三：锁的"粒度"和"模式"维度混淆
> Day4发现

### 现状
回答时说"RC当前读加排它锁"，把锁的粒度（Record/Gap/Next-Key）和模式（X/S）混在一起。这两个维度是正交的。

### 架构师为什么需要
面试官追问"SELECT LOCK IN SHARE MODE加什么锁"，如果只想到"共享锁"没想"锁哪些行/间隙"，说明理解不完整。架构师要能同时回答"什么模式"和"什么粒度"。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | FOR UPDATE加排它锁 |
| 高级开发 | 加X锁，LOCK IN SHARE MODE加S锁 |
| 架构师 | 锁是两个正交维度：模式（X排它/S共享）× 粒度（Record/Gap/Next-Key）。FOR UPDATE = X模式，LOCK IN SHARE MODE = S模式。RR下粒度是Next-Key Lock，RC下粒度只有Record Lock。两个维度独立变化 |

### 怎么补
1. **记住"模式×粒度=完整的锁描述"**
2. **面试时两个维度都答**："加的是X模式的Next-Key Lock"而不是"加排它锁"
3. **写一张表**：模式（X/S）× 粒度（Record/Gap/Next-Key）× 隔离级别（RR/RC）= 12种组合

## 差距四：死锁排查缺SOP，长事务治理缺纵深
> Day4发现

### 现状
知道AB-BA死锁和固定顺序解法，但完整排查SOP说不出。长事务只想到"占连接"，不知道对五个层面的影响。治理方案停留在"加超时"。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 死锁重启就好，长事务加超时 |
| 高级开发 | SHOW ENGINE INNODB STATUS看死锁，按顺序加锁解决 |
| 架构师 | 死锁排查五步SOP（看状态→查锁等待→查长事务→匹配四种模式→选解法）。长事务影响五层（undo/redo/binlog/锁/主从），磁盘暴涨根因是undo purge被阻塞。根治五层（代码规范→ORM超时→连接池防泄漏→事务边界最小化→应用拆分）。监控四指标分级告警 |

### 怎么补
1. **死锁四种模式**：AB-BA固定顺序、Gap锁用RC、同行不同索引优化索引、批量更新排序
2. **长事务磁盘暴涨根因**：长事务挡住undo log purge，不是自己修改多
3. **事务边界设计**："事务内只做必须原子的事"——先查后改，RPC/MQ/通知全部事务外

## 差距五：数字直觉仍不足 — 延续第5周差距七
> Day4发现，延续第5周差距七

### 现状
答锁和事务相关题目时依然无量级数字——行锁TPS上限多少？History list length多少告警？事务超时多少秒？售罄后DB压力降多少？

| 缺失数字 | 应该说出的 |
|---------|-----------|
| 行锁串行化TPS上限 | ≈500（10万QPS秒杀场景远超） |
| 乐观锁冲突率>80% | 实际TPS反而比悲观锁低 |

### 怎么补
1. **每次答锁相关题带数字**：TPS上限、超时秒数、告警阈值
2. **回答时先给数字再解释**："行锁串行化TPS上限约500，10万QPS秒杀远超所以用Redis预扣"
3. **建立数字直觉**：行锁TPS≈500、Redis Lua TPS≈10万+、事务超时30s、History list>1000关注

## 差距六：高可用架构选型只知概念缺决策框架 — 知道MHA/MGR但不清楚什么时候选哪个
> Day5发现

### 现状
知道MySQL有主从复制、MHA、MGR这些方案，但选型时没有系统框架。只想到"数据重要用MGR"，说不出RPO/MTA/写性能三个维度的权衡。MHA已停止维护这个关键信息缺失。

### 架构师为什么需要
"高可用怎么做的"是面试必问题。只答"MGR"没有说服力，面试官会追问"为什么不用MHA""MGR的写性能开销能接受吗""什么场景必须用MGR"。没有三维决策框架，这些追问全答不上来。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | MySQL有主从复制可以做高可用 |
| 高级开发 | 知道MHA和MGR，数据重要用MGR |
| 架构师 | 高可用选型是三维决策：RPO（数据丢失容忍度）+ MTA（恢复时间）+ 写性能容忍度。金融RPO=0选MGR，电商核心RPO≈0选半同步+MHA，非核心异步+MHA。MHA已停止维护，新项目选MGR。MGR代价是写降15-30%，高写场景需评估 |

### 怎么补
1. **记住三维决策框架**：RPO + MTA + 写性能
2. **记住MHA停止维护**：新项目不再推荐MHA

## 差距七：慢SQL治理缺闭环思维 — 只会优化不会预防
> Day5发现，延续第5周差距（可观测性）

### 现状
遇到慢SQL能EXPLAIN分析、加索引优化，但缺"发现→分析→优化→预防"四阶段闭环。SQL审核平台概念模糊，不知道怎么从源头预防慢SQL上线。

### 架构师为什么需要
架构师不是只救火，更是防火。只优化不预防，慢SQL永远治不完。SQL审核平台是架构师推动的基础设施——自动EXPLAIN检查、影响评估、执行管控，从流程上阻断慢SQL上线。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 慢SQL加索引就好了 |
| 高级开发 | EXPLAIN分析，索引优化，参数调优 |
| 架构师 | 慢SQL治理是四阶段闭环：发现（slow_query_log+APM监控）→ 分析（EXPLAIN五步SOP）→ 优化（索引/SQL改写/架构）→ 预防（SQL审核平台自动检查+限流降级兜底）。治理比优化重要，预防1条>优化10条 |

### 怎么补
1. **记住四阶段闭环**：发现→分析→优化→预防
2. **SQL审核平台是架构师推动的基础设施**：上线前自动EXPLAIN+影响评估+执行管控
3. **面试时主动提预防**："除了优化，我还会推动SQL审核平台建设……"

## 差距八：大表在线变更只知道概念缺SOP — 知道pt-osc/gh-ost但没实操步骤
> Day5发现

### 现状
知道大表不能直接ALTER，听说过pt-osc和gh-ost，但说不清两者的核心区别（触发器vs binlog），更没有完整的变更SOP。变更时间预估、throttle调参、数据一致性验证全是空白。

### 架构师为什么需要
线上大表变更搞不好就是P0故障。面试官问"线上100GB的表怎么加字段"，只答"用gh-ost"太浅，追问"变更期间从库延迟怎么办""怎么保证数据一致性""变更要多久"答不上来。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 大表ALTER会锁表，用pt-osc |
| 高级开发 | pt-osc用触发器同步，gh-ost用binlog同步，高并发选gh-ost |
| 架构师 | 核心区别是增量同步方式：pt-osc触发器（有开销但不依赖binlog格式），gh-ost解析binlog（无触发器开销但需ROW格式）。TPS>5000必须gh-ost。变更SOP五步：评估影响→选工具+时间窗口→影子表验证→执行（加throttle限速）→验证收尾。100GB表约50分钟，throttle按Threads_running和从库延迟动态调速 |

### 怎么补
1. **核心区别一句话**：pt-osc触发器同步 vs gh-ost binlog同步
2. **记住五步SOP**：评估→选工具→验证→执行→收尾
3. **记住数字**：10GB≈5分钟，100GB≈50分钟，1TB≈8小时+

## 差距九：容量规划缺量化思维 — 只说"分库分表"不算"需要几个库几张表"
> Day5发现，延续第5周差距（容量规划）

### 现状
知道数据量大了要分库分表，但算不出"多少库多少表"。单机基线数字不牢（QPS极限/单表行数上限），安全系数意识缺失，扩容预案没有。

### 架构师为什么需要

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 数据多了就分库分表 |
| 高级开发 | 单表超2000万要拆，按QPS算需要几个库 |
| 架构师 | 容量规划四步法：基线→预估→计算→扩容。单机基线：读QPS≈8000、写QPS≈3000、单表上限≈2000万。算出所需库表数后×1.5安全系数。扩容预案按短期/中期/长期分层，冷热分离降低在线数据量 |

### 怎么补
1. **记住单机基线**：读≈8000-10000、写≈3000-5000、单表≈2000万、文件≈50GB
2. **记住四步法**：基线→预估→计算→扩容
3. **面试时先算再答**："按日增500万，1年后约19亿行，需要4库×32表，加50%余量"

## 差距十：故障排查缺优先级意识 — 想调优不想止血
> Day5发现

### 现状
遇到数据库故障第一反应是"查原因调优"，而不是"先止血"。故障排查四步法（现象→根因→止血→根治）中，止血步骤容易跳过。没有"止血>根治>优化"的优先级意识。

### 架构师为什么需要
线上故障分秒必争，P0故障时先止血再排查。先Kill慢SQL+限流降级把服务恢复，再慢慢查根因。在故障现场调优是最危险的行为——可能加重问题。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 慢SQL加索引，连接打满重启 |
| 高级开发 | SHOW PROCESSLIST查慢SQL，Kill掉 |
| 架构师 | 故障排查四步法：现象确认(What)→根因定位(Why)→止血优先(Stop)→根治防复(Fix)。关键原则：止血>根治>优化。止血三件套：Kill阻塞源+限流降级+切流量。不要在故障现场调优 |

### 怎么补
1. **记住优先级**：止血 > 根治 > 优化
2. **止血三件套**：Kill阻塞源 + 限流降级 + 切流量
3. **面试时先说止血再分析根因**："先Kill慢SQL恢复服务，再EXPLAIN查根因"

## 差距十一：锁的物理存在形式说不清 — 知道"加在索引上"但讲不出底层数据结构
> Day7深挖发现，延续Day4差距一

### 现状
能说"锁加在索引上"，但说不清锁的物理存储结构（lock_sys->rec_hash哈希表）、锁的 key 是什么（space_id+page_no+heap_no）、无索引为什么锁全表的完整推演链。看到 `SHOW ENGINE INNODB STATUS` 输出的 `space id X page no Y n bits Z` 看不懂每个字段含义。

### 架构师为什么需要
面试官追问"锁到底加在哪""无索引为什么锁全表"，只答"加在索引上"是高级开发水平。架构师要能从内存数据结构层面解释：锁是 `lock_t` 结构存储在全局哈希表，无索引时 InnoDB 必须先加锁再判断 WHERE，Server 层 5.7+ 才优化提前释放。线上看 INNODB STATUS 日志要能逐字段读出锁的物理位置。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | InnoDB 锁加在行上 |
| 高级开发 | 锁加在索引上，没索引会锁全表 |
| 架构师 | 锁物理上存在 `lock_sys->rec_hash` 全局哈希表，key=(space_id, page_no, heap_no)，每把锁是 `lock_t` 结构含事务ID+模式+粒度+位图。无索引时聚簇索引全扫描+逐行加 Next-Key Lock=锁全表+所有间隙；5.7+ Server 层判断 WHERE 不匹配后 `unlock_row()` 释放，RC 真释放 RR 不能释放 |

### 怎么补
1. **记住物理存储**：lock_sys 哈希表 + lock_t 结构 + (space_id, page_no, heap_no) key
2. **记住无索引锁全表完整链**：聚簇全扫描→逐行加锁→Server 层过滤→RC 释放 RR 保留
3. **能逐字段读 INNODB STATUS**：space id/page no/n bits/index/lock_mode/locks rec but not gap

## 差距十二：RR 防幻读边界讲不清 — 以为 RR 能完全防幻读
> Day7深挖发现，延续Day4差距二

### 现状
教科书背的是"RR 通过 Next-Key Lock 防止幻读"，但讲不清快照读 vs 当前读的两套机制（MVCC vs Next-Key Lock），更构造不出"RR 下仍能幻读"的反例（SELECT+UPDATE 混用场景）。

### 架构师为什么需要
"RR 真的能防幻读吗"是甄别架构师的题目。能答出"快照读靠 MVCC、当前读靠 Next-Key Lock，两者混用仍会幻读"才算到位。这个边界也决定了"为什么阿里规约改 RC + 业务层唯一性兜底"的因果链 — 间隙锁消失后必须靠唯一索引兜底。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | RR 是默认隔离级别，能防幻读 |
| 高级开发 | RR 通过 Next-Key Lock 防幻读 |
| 架构师 | RR 防幻读分两套机制：快照读靠 MVCC ReadView（看不见新行），当前读靠 Next-Key Lock（插不进新行）。但 SELECT+UPDATE 混用时 UPDATE 是当前读会读到最新版本，把别人新插入的行变成自己的修改，下次 SELECT 就能看见 — 这是 RR 防幻读的语义边界 |

### 怎么补
1. **记住两套机制**：快照读 MVCC + 当前读 Next-Key Lock
2. **会构造反例**：SELECT→其他事务 INSERT→UPDATE→SELECT 能看到新行
3. **理解因果链**：RR→RC 消除间隙锁→必须唯一索引兜底（不是 SELECT 后 INSERT）

## 差距十三：意向锁的存在价值讲不清 — 知道有 IS/IX 但不知道为什么需要
> Day7深挖发现

### 现状
知道 InnoDB 有意向锁 IS/IX，但讲不清"没有意向锁会怎样"（O(n) 全表扫描 vs O(1) 标志判断）、IX-IX 为什么兼容（这是高并发的关键）、完整的 4×4 锁兼容矩阵默写不出来。

### 架构师为什么需要
意向锁是面试高频题，更是理解"InnoDB 怎么支持高并发"的核心。IX-IX 兼容才让多个事务能同时持有不同行的 X 锁，否则数据库就退化成串行。架构师要能解释"表级锁判断从 O(n) 降到 O(1)"的优化本质，以及 Online DDL 为什么依赖意向锁兼容性。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | InnoDB 有意向锁 |
| 高级开发 | 加行锁前先加表级意向锁 IX/IS |
| 架构师 | 意向锁是表级锁，本质是"表中有行被锁"的标志位，把"判断表上有无行锁"从 O(n) 全表扫描降到 O(1) 标志判断。IX-IX 兼容是高并发关键 — 行级冲突交给行锁判断，表级只判断意图。完整兼容矩阵：X 与所有锁互斥，IX-IX/IX-IS/IS-IS/S-IS/S-S 兼容。Online DDL 依赖此 — MDL 读锁兼容 IX，DML 才能在 DDL 期间执行 |

### 怎么补
1. **记住意向锁本质**：表级标志位，O(n)→O(1) 优化
2. **能默写 4×4 兼容矩阵**：X/IX/S/IS 两两兼容性
3. **理解 IX-IX 兼容的意义**：行级冲突交给行锁判断，表级只表达意图
4. **关联 Online DDL**：MDL 读锁兼容 IX，所以 DDL 期间 DML 可执行

## 差距十四：死锁检测开销缺量化认知 — 不知道 O(n²) 是性能杀手
> Day7深挖发现

### 现状
知道 MySQL 有死锁检测但不知道算法复杂度（DFS+O(n²)）、不知道高并发热点行为什么死锁检测反而拖垮 TPS、不知道 `innodb_deadlock_detect=OFF` + `innodb_lock_wait_timeout=1` 的兜底组合、设计不出"DB+应用+缓存"三层防死锁完整方案。

### 架构师为什么需要
秒杀场景的核心痛点之一就是死锁检测开销。面试官问"秒杀系统数据库怎么设计"，能答出"关掉死锁检测+缩短超时+应用层重试+Redis 预扣分流"才是架构师水平。理解 O(n²) 推导才能解释为什么 1000 并发时检测开销爆炸。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | MySQL 会自动检测死锁 |
| 高级开发 | 死锁检测会回滚一个事务，高并发要小心 |
| 架构师 | 死锁检测是 DFS 等待图回路检测，单次 O(V+E)，热点行场景 E≈V² 单次约 O(n²)，1000 并发总开销 O(n³) ≈ 10 亿次操作 + 持 lock_sys 全局锁。高并发场景策略：关检测+ timeout=1s + 应用重试。完整防御三层：DB（关检测+单行操作+autocommit）+ 应用（请求合并+限流+异步化）+ 缓存（Redis 预扣+兜底+对账） |

### 怎么补
1. **记住算法复杂度**：DFS 单次 O(n²)，n=并发等待事务数
2. **记住关检测组合**：`innodb_deadlock_detect=OFF` + `innodb_lock_wait_timeout=1` + 应用重试
3. **能设计三层方案**：DB 层（关检测+极简表+单行操作）+ 应用层（合并+限流+异步+重试）+ 缓存层（预扣+兜底+对账）

## 差距十五：半一致性读理解不深 — 知道是 RC 优势但说不清触发条件
> Day7深挖发现，延续Day4差距二

### 现状
听说过"半一致性读是 RC 杀手级特性"，但讲不清三个触发条件（RC + UPDATE + ROW binlog）、为什么只对 UPDATE 生效（不是 SELECT/DELETE）、跳过的事务对业务的影响（影响行数=0 需应用层判断）、在自己业务里哪些场景能用哪些不能用。

### 架构师为什么需要
半一致性读是 RR→RC 决策的关键论据之一。能说出"RC 改造后 UPDATE 遇锁会读最新版本判断 WHERE 不匹配就跳过不加锁"，才算真正理解 RC 在高并发下的优势。架构师要能给业务团队画出"状态机推进 ✅、余额计息 ❌"的适用边界。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | RC 锁少一些 |
| 高级开发 | RC 没有间隙锁，半一致性读跳过冲突行 |
| 架构师 | 半一致性读三个触发条件：RC + UPDATE + ROW binlog；只对 UPDATE 是因为 SELECT FOR UPDATE 必须锁、DELETE 不能漏删；机制是 UPDATE 遇锁不等待→读最新已提交版本→WHERE 不匹配则跳过不加锁。适用边界：状态机推进/账户冻结/会话分配能用（语义上接受被改就跳过），余额计息/转账/对账不能用（依赖事务开始时快照） |

### 怎么补
1. **记住三触发条件**：RC + UPDATE + ROW binlog
2. **理解只对 UPDATE 的原因**：SELECT 必须锁，DELETE 不能漏
3. **建立业务边界**：状态机/分配类✅，依赖快照类❌
4. **应用层兜底**：UPDATE 返回影响行数=0 时应用层判断业务路径

## 差距十六：架构师"系统化原则"缺失 — 修死锁靠改 SQL 不靠设计原则
> Day7深挖发现

### 现状
死锁了改 SQL，长事务了拆事务，热点了加缓存 — 都是问题导向。没建立"架构设计阶段就规避"的系统化原则集合。比如转账场景，能修死锁但说不出"6 条设计原则（固定加锁顺序+缩短事务+分片键对齐+热点打散+异步化+监控驱动）"。

### 架构师为什么需要
架构师 vs 高级开发的本质差异：架构师在设计阶段把问题挡住，高级开发在事故现场修问题。面试官问"怎么设计才能从根上避免转账死锁"，能答出 6 条系统化原则才是架构师水平。每个技术决策点都应总结出 3-6 条可复用的设计原则。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 死锁了改 SQL |
| 高级开发 | 死锁排查 SOP + 加索引消除间隙锁 |
| 架构师 | 转账防死锁 6 原则：①固定加锁顺序（按账户 ID 升序）②缩短事务边界（禁 RPC/MQ/IO）③分片键对齐（同用户同分片）④热点行打散（大商户分子账户）⑤异步化非核心路径（通知/对账走 MQ）⑥监控驱动设计（死锁率>0.1%必复盘）。每个技术决策点都总结成可复用原则 |

### 怎么补
1. **建立"设计阶段原则集合"思维**：每个技术点总结 3-6 条原则
2. **6 原则示例**：固定顺序+缩短事务+分片对齐+热点打散+异步化+监控
3. **面试时主动说"设计阶段就规避"**：不要等问"线上死锁了怎么办"才答
