## 差距一：锁的加锁规则不够精确 — 知道概念但说不清具体范围
> Day4发现

### 现状
知道锁加在索引上、三种锁的定义、和索引类型有关。但具体到"主键BETWEEN加什么锁""普通索引等值命中为什么不退化"说不准确。把锁的"粒度"（Record/Gap/Next-Key）和"模式"（X/S）混在一起。

### 架构师为什么需要
面试官问"SELECT FOR UPDATE WHERE id BETWEEN 10 AND 20加了什么锁"，答"记录锁+间隙锁"太模糊。要能逐行说出扫描过程和退化结果。线上遇到锁冲突排查也需要精确判断锁范围。说"普通索引不退化因为可能有重复值"比"普通索引会加gap lock"强十倍。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | InnoDB有行锁和表锁 |
| 高级开发 | 知道Record/Gap/Next-Key Lock，锁加在索引上 |
| 架构师 | 锁的行为由索引类型×查询类型×隔离级别三维度决定。唯一索引等值命中退化成Record Lock，普通索引不退化——根因是普通索引可能有重复值必须锁间隙防同值插入。无索引=锁全表。能逐行说出扫描过程和退化结果 |

### 怎么补
1. **记住退化规则速查表**：5种场景（唯一命中/唯一未命中/普通命中/普通未命中/无索引）
2. **记住退化根因**：唯一索引保证唯一→不需要锁间隙；普通索引可能重复→必须锁间隙
3. **面试时先说本质**："锁加在索引上，行为由索引类型决定"，再给速查表

## 差距二：隔离级别选型缺系统化框架 — 只想到一个维度
> Day4发现

### 现状
知道RR有间隙锁容易死锁，但说不出阿里改RC的完整理由。五个维度只能想到一个。不知道半一致性读。RC下必须留RR的场景完全空白。

### 架构师为什么需要
"为什么改RC"是面试高频题。只答"减少死锁"勉强及格，答不出五个维度（锁范围/死锁/主从/可见性直觉/半一致性读）露馅。更关键的是"什么场景必须RR"——答不出说明没有系统思考过选型。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | RR是默认的，RC有不可重复读 |
| 高级开发 | RR有间隙锁容易死锁，可以考虑RC |
| 架构师 | RC选型五个理由（锁范围小/死锁少/row格式消除复制风险/可见性符合直觉/半一致性读），RR必留两个场景（对账/唯一性校验+插入，但后者用唯一索引更可靠）。半一致性读是RC杀手级优势：UPDATE遇锁行→读最新版本→不匹配直接跳过不加锁。选型决策树：OLTP用RC+唯一索引兜底，报表对账用RR+WITH CONSISTENT SNAPSHOT |

### 怎么补
1. **记住五个维度**：锁范围/死锁/主从/可见性直觉/半一致性读
2. **半一致性读是加分项**：面试时主动提"RC有个杀手级特性叫半一致性读"
3. **RC下必须RR的场景**：对账一致性 + 唯一性校验（但推荐唯一索引替代）

## 差距三：锁的"粒度"和"模式"维度混淆
> Day4发现

### 现状
回答时说"RC当前读加排它锁"，把锁的粒度（Record/Gap/Next-Key）和模式（X/S）混在一起。这两个维度是正交的。

### 架构师为什么需要
面试官追问"SELECT LOCK IN SHARE MODE加什么锁"，如果只想到"共享锁"没想"锁哪些行/间隙"，说明理解不完整。架构师要能同时回答"什么模式"和"什么粒度"。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | FOR UPDATE加排它锁 |
| 高级开发 | 加X锁，LOCK IN SHARE MODE加S锁 |
| 架构师 | 锁是两个正交维度：模式（X排它/S共享）× 粒度（Record/Gap/Next-Key）。FOR UPDATE = X模式，LOCK IN SHARE MODE = S模式。RR下粒度是Next-Key Lock，RC下粒度只有Record Lock。两个维度独立变化 |

### 怎么补
1. **记住"模式×粒度=完整的锁描述"**
2. **面试时两个维度都答**："加的是X模式的Next-Key Lock"而不是"加排它锁"
3. **写一张表**：模式（X/S）× 粒度（Record/Gap/Next-Key）× 隔离级别（RR/RC）= 12种组合

## 差距四：死锁排查缺SOP，长事务治理缺纵深
> Day4发现

### 现状
知道AB-BA死锁和固定顺序解法，但完整排查SOP说不出。长事务只想到"占连接"，不知道对五个层面的影响。治理方案停留在"加超时"。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 死锁重启就好，长事务加超时 |
| 高级开发 | SHOW ENGINE INNODB STATUS看死锁，按顺序加锁解决 |
| 架构师 | 死锁排查五步SOP（看状态→查锁等待→查长事务→匹配四种模式→选解法）。长事务影响五层（undo/redo/binlog/锁/主从），磁盘暴涨根因是undo purge被阻塞。根治五层（代码规范→ORM超时→连接池防泄漏→事务边界最小化→应用拆分）。监控四指标分级告警 |

### 怎么补
1. **死锁四种模式**：AB-BA固定顺序、Gap锁用RC、同行不同索引优化索引、批量更新排序
2. **长事务磁盘暴涨根因**：长事务挡住undo log purge，不是自己修改多
3. **事务边界设计**："事务内只做必须原子的事"——先查后改，RPC/MQ/通知全部事务外

## 差距五：数字直觉仍不足 — 延续第5周差距七
> Day4发现，延续第5周差距七

### 现状
答锁和事务相关题目时依然无量级数字——行锁TPS上限多少？History list length多少告警？事务超时多少秒？售罄后DB压力降多少？

| 缺失数字 | 应该说出的 |
|---------|-----------|
| 行锁串行化TPS上限 | ≈500（10万QPS秒杀场景远超） |
| 乐观锁冲突率>80% | 实际TPS反而比悲观锁低 |
| 事务超时 | 30秒（@Transactional(timeout=30)） |
| History list length告警 | >1000关注/>5000介入/>10000紧急 |
| 长事务持续时间告警 | >30s P3/>60s P2/>300s P1 |
| undo log膨胀 | 1小时长事务+100万UPDATE → GB级 |

### 怎么补
1. **每次答锁相关题带数字**：TPS上限、超时秒数、告警阈值
2. **回答时先给数字再解释**："行锁串行化TPS上限约500，10万QPS秒杀远超所以用Redis预扣"
3. **建立数字直觉**：行锁TPS≈500、Redis Lua TPS≈10万+、事务超时30s、History list>1000关注

## 本周自查清单（Day1-Day4）

每答完一道题，过一遍：

- [ ] **锁的精确范围**：能不能逐行说出扫描过程和退化结果？
- [ ] **选型框架**：隔离级别/扣减方案的选型有几个维度？每个维度为什么？

- [ ] **数字**：有没有量级数字？TPS上限/超时秒数/告警阈值
- [ ] **根因**：能不能说清"为什么"而不只是"是什么"？
- [ ] **因果链**：索引设计→锁行为→并发上限→分库分表，能不能串起来？

### 一句话总结

**Day4的核心差距是"知道概念但不够精确"：锁的分类知道但加锁规则说不准，RC选型知道但理由不系统，死锁知道排查但没SOP，长事务知道占连接但不知道五层影响。架构师要从"大概知道"升级到"能逐行说清"——锁的范围要逐行扫描过程，选型要系统化框架，排查要步骤化SOP，治理要纵深防御。**
