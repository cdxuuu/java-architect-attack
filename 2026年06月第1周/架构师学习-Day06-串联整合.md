### 场景一：蛋壳钱包资金路由 — 数据库强一致性下的全链路保障

**背景**：蛋壳钱包的资金路由系统，每日处理百万级交易，涉及账务流水记录、余额变更、对账清算。

**考题方向**：
1. 资金路由的分库分片键怎么选？天/用户/流水号三种路由策略的取舍
2. 余额变更用哪种隔离级别？RR还是RC？常见死锁场景怎么排查
3. 日切对账时需要全表一致性快照，隔离级别怎么处理？
4. 账户流水表每天增长百万行，慢SQL治理怎么落地？深分页问题怎么解？
5. 半夜对账跑批大事务导致主从延迟飙升，怎么止血和根治？

**涉及Day**：Day1(分片键) + Day2(主从延迟) + Day3(索引调优) + Day4(事务锁)+ Day5(故障排查)

### 场景二：医疗在线问诊 — 高并发挂号系统的数据库架构演进

**背景**：医疗在线问诊的挂号系统，专家号开抢时瞬间并发飙高（类似秒杀），涉及号源库存、患者信息、问诊记录等多类数据。

**考题方向**：
1. 单库扛不住挂号并发，先做读写分离还是直接分库分表？（区域/医生/患者三个分片键候选怎么选？基因法怎么落地？）
2. 号源扣减是数据库行锁还是Redis预扣？两者各自的风险是什么？怎么保证不超卖？
3. 问诊记录表很快过千万，索引怎么设计？按医生查询和按患者查看两种查询模式怎么兼顾？
4. 在线问诊高峰期连接数打满，排查SOP怎么执行？连接池参数怎么配？
5. 一年后的数据怎么冷热分离？归档后历史问诊查询怎么支持？

**涉及Day**：Day1(分库分表演进) + Day2(读写分离) + Day3(索引设计) + Day4(锁机制) + Day5(连接打满/冷热分离)

### 场景三：IM在线客服消息存储 — 海量消息的分库分表+冷热分离设计

**背景**：IM在线客服，每天产生数千万条聊天消息，需要支持用户按会话查询历史消息，也需要客服回溯查看。

**考题方向**：
1. 消息表用 conversation_id 分片，但用户查历史消息时按 user_id + time_range 查，分片键和查询条件不匹配——怎么处理？
2. 消息量太大需要冷热分离，热（7天）放MySQL，冷（30天+）放哪儿？MySQL中的热数据和归档库怎么同步？
3. 深分页翻查历史消息，传统 LIMIT OFFSET 越来越慢，怎么优化？
4. 高并发写入消息，索引怎么设计避免写瓶颈？覆盖索引怎么用？
5. 消息存储大表在线变更（加字段/加索引）走 pt-osc 还是 gh-ost？IO瓶颈怎么控制？

**涉及Day**：Day1(跨分片查询) + Day3(索引/深分页) + Day4(写并发锁) + Day5(大表变更/冷热分离)

---

### 场景四：保险商城账单系统 — 月结批处理+实时查询的混合负载架构

**背景**：平安健康险保险商城，每月账单结算批处理（批量更新大量保单状态）与日常用户实时查询保单信息共存，两种负载互相干扰。

**考题方向**：
1. 批处理大事务 UPDATE 上万行，锁范围和普通事务冲突怎么避免？能不能拆分成小批次？
2. 月结批处理导致主从延迟飙到数百秒，实时查询打到从库读到旧数据——怎么处理？
3. 保单表查询模式多样（按保单号/按用户/按时间/按状态），联合索引怎么设计？最左前缀怎么兼顾？
4. 随着用户量增长，保单表逼近2000万行上限，分库分表的决策判断怎么做？要不要做？
5. 高可用选型：保险业务RPO可以不为零吗？MGR还是半同步+MHA？写性能降15-30%能接受吗？

**涉及Day**：Day1(分库分表判断) + Day2(主从延迟) + Day3(索引设计) + Day4(大事务/锁冲突) + Day5(高可用选型/慢SQL治理)
