# Day 5：数据库性能优化实战综合 — 架构师梳理

## 一、MySQL高可用架构选型 — 决策框架

### 核心模型：RPO + MTA + 写性能 三维决策

```
高可用选型 = f(RPO, MTA, 写性能容忍度)

  维度1：RPO（Recovery Point Objective，数据丢失容忍度）
    ├─ RPO=0（一帧不能丢）→ MGR / InnoDB Cluster
    ├─ RPO≈0（极少丢）    → 半同步 + MHA
    └─ RPO>0（可接受丢失）→ 异步复制 + MHA/VIP

  维度2：MTA（Mean Time to Availability，恢复时间容忍度）
    ├─ MTA < 10秒  → InnoDB Cluster（Router自动路由）
    ├─ MTA < 30秒  → MGR（自动选主）
    ├─ MTA < 2分钟 → MHA（Manager切换）
    └─ MTA > 30分钟 → 手动VIP切换

  维度3：写性能容忍度
    ├─ 可接受降15-30% → MGR（多数派确认开销）
    ├─ 可接受降5-10%  → 半同步（等1从确认）
    └─ 不能降          → 异步复制

  一句话决策：
    "金融场景RPO=0选MGR，电商核心RPO≈0选半同步+MHA，
     非核心业务异步复制+MHA，新项目不再用MHA（停止维护）"
```

### MHA vs MGR 核心区别一句话

```
MHA是管理工具（不管复制，只管failover），MGR是复制引擎（替代原生复制，自带选举）
MHA已停止维护 → 新项目选MGR
MGR核心代价：写降15-30%，换来不丢数据+秒级切换
```

## 二、慢SQL治理体系 — 四阶段闭环

### 核心模型：发现→分析→优化→预防

```
慢SQL治理不是"加了索引就好了"，而是四阶段闭环：

  发现（线上监控）→ 分析（EXPLAIN五步SOP）→ 优化（四种手段）→ 预防（SQL审核平台）

  架构师视角的关键认知：
    1. 治理比优化更重要——预防1条慢SQL > 优化10条慢SQL
    2. SQL审核平台是最后一道防线——上线前自动EXPLAIN检查
    3. 慢SQL的根因往往不是"缺索引"——可能是锁竞争/深分页/架构设计问题
    4. 治理要有数字——慢SQL阈值0.1s，扫描行数>1万告警，>10万自动限流
```

### EXPLAIN五步SOP速查

```
1. type：ALL/index → 危险
2. possible_keys vs key：有可用但没用 → FORCE INDEX验证
3. rows × filtered：估算实际扫描量
4. Extra：filesort/temporary → 需优化
5. 优化后验证：EXPLAIN ANALYZE对比
```

### 常见慢SQL模式速查（8种）

```
1. SELECT *         → 覆盖索引/只查需要的列
2. WHERE+ORDER BY   → 联合索引(等值条件,排序字段)
3. 范围+排序        → (排序字段,范围字段) 或 ES
4. 深分页LIMIT      → 游标分页 WHERE id>last_id
5. IN(子查询)       → 改JOIN/EXISTS
6. COUNT(*)全表     → Redis计数/汇总表
7. GROUP BY无索引   → 加联合索引/先限缩
8. 函数/隐式转换    → 改写法/保持类型一致
```

## 三、大表在线变更 — 工具选型与SOP

### 核心模型：触发器 vs binlog 两种增量同步方式

```
pt-osc = 新表 + 触发器同步增量（简单但有性能开销）
gh-ost = 新表 + binlog同步增量（无触发器开销但需ROW格式）

  选型一句话：
    TPS<1000 → pt-osc（简单够用）
    TPS>5000 → gh-ost（触发器开销太大）
    中间地带 → gh-ost（长期方案更稳）

  gh-ost杀手级特性：throttle动态调速
    --max-load=Threads_running=25（超25自动暂停）
    --critical-load=Threads_running=50（超50立刻停）
    --max-lag-millis=1500（从库延迟>1.5s暂停）
```

### 大表DDL五步SOP

```
评估影响 → 选工具+时间窗口 → 影子表验证 → 执行（加throttle）→ 验证收尾

  数字直觉：
    10GB ≈ 5分钟 | 100GB ≈ 50分钟 | 1TB ≈ 8小时+
    拷贝速度 ≈ 2GB/分钟（取决于磁盘IO）
```

## 四、容量规划与架构演进

### 容量规划四步法

```
基线（单机极限）→ 预估（N月后数据量）→ 计算（所需库表数）→ 扩容（预留50%余量）

  单机基线速查：
    读QPS ≈ 8000-10000 | 写QPS ≈ 3000-5000
    单表行数上限 ≈ 2000万 | 单表文件上限 ≈ 50GB
    连接数上限 ≈ 3000-5000 | 磁盘上限 ≈ 2-4TB（SSD）
```

### 架构演进三阶段

```
阶段1：单库优化（QPS<5000）→ 索引+SQL+缓存+参数
阶段2：读写分离（读QPS 5000-30000）→ 一主多从+Redis
阶段3：分库分表（写QPS>5000/数据>2000万）→ 垂直+水平拆分

  四大反模式：
    ❌ QPS才2000就分库分表
    ❌ 读写分离解决写瓶颈
    ❌ 缓存替代分库分表
    ❌ 上来就微服务拆DB
```

### 冷热分离三层模型

```
热（3个月）→ MySQL+Redis | 温（3-12月）→ 归档库 | 冷（1年+）→ HBase/OSS
归档方法：pt-archiver / 分区表DROP PARTITION / 按月分表整表迁移
```

## 五、数据库故障排查 — 三大经典故障SOP

### 连接数打满

```
SHOW PROCESSLIST → 分类（Sleep/Query/Locked）
Sleep多 → 连接池配置 | Query多 → Kill慢SQL | Locked → Kill阻塞源
根治：连接池maxActive=50, wait_timeout=600, 监控>80%告警
```

### 主从延迟飙升

```
SHOW SLAVE STATUS → 确认延迟秒数
IO慢 → 网络/主库 | SQL慢 → 大事务/DDL/从库负载
根治：多线程复制(slave_parallel_workers=4-8) + 大事务拆分 + DDL走gh-ost
延迟阈值：< 1s正常 | 1-30s关注 | >30s告警 | >300s紧急
```

### 磁盘暴涨

```
定位膨胀源：数据文件/undo/binlog/redo/临时文件
undo暴涨 → 找长事务Kill → purge自动回收
binlog暴涨 → 检查expire_logs_days → PURGE清理
数据碎片 → ALTER TABLE ENGINE=InnoDB（重建表）
数字：undo<1GB正常 | >5GB告警 | >50GB紧急
```

### 通用排查四步法

```
现象确认(What) → 根因定位(Why) → 止血优先(Stop) → 根治防复(Fix)
关键原则：止血 > 根治 > 优化（不要在故障现场调优）
```
