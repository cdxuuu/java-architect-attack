# Day 4：MySQL事务与锁机制 — 架构师角度梳理

## 核心定位

Day4主题一句话概括：**锁的本质是索引上的锁，加锁行为由索引类型×查询类型×隔离级别三维度决定——架构师要能从"为什么加这把锁"反推到"怎么避免这把锁"**。

## 事务隔离级别：RR vs RC 的成本权衡

```
核心认知：隔离级别不是"哪个好"，是"你愿意付出什么代价"

  ┌─────────────────────────────────────────────────────────┐
  │  RR 的代价                         │  RC 的代价          │
  ├─────────────────────────────────────┼─────────────────────┤
  │  ① 间隙锁→死锁概率高               │  ① 幻读需业务兜底   │
  │  ② 锁范围大→并发度低               │  ② 不可重复读       │
  │  ③ 事务内看不到最新数据→业务踩坑    │  ③ 对账需RR连接     │
  └─────────────────────────────────────┴─────────────────────┘

  选型本质：
    RR = 用锁换一致性（强一致但并发受限）
    RC = 用一致性换并发（并发高但需要业务补偿）

  为什么互联网公司选RC：
    1. row格式binlog消除了RC的复制风险
    2. 半一致性读是RC杀手级优势（UPDATE遇锁行→读最新版本→不匹配跳过）
    3. 唯一索引可以替代间隙锁的唯一性校验功能
    4. 绝大多数OLTP场景不需要跨事务一致性读

  RR必须留的场景只有两类：
    1. 对账/报表 → 用 WITH CONSISTENT SNAPSHOT
    2. 唯一性校验+插入 → 用唯一索引更可靠
    → 结论：RC + 唯一索引 > RR 间隙锁
```

## MVCC：ReadView 是 RR 和 RC 的分水岭

```
  ┌──────────────────────────────────────────────────────┐
  │            ReadView 四字段                            │
  │  m_ids (活跃事务列表)                                 │
  │  min_trx_id (低水位)                                 │
  │  max_trx_id (高水位)                                 │
  │  creator_trx_id (创建者)                             │
  │                                                      │
  │  可见性判断：                                         │
  │    < 低水位 → 可见（已提交）                          │
  │    ≥ 高水位 → 不可见（之后开始）                      │
  │    中间区 → 在m_ids中不可见，不在则可见               │
  └──────────────────────────────────────────────────────┘

  RR vs RC 的唯一区别：ReadView 生成次数
    RR：一次（首次快照读）→ m_ids固化 → 可见性不变
    RC：每次快照读 → m_ids更新 → 新提交变可见

  面试陷阱：
    "RR的ReadView在BEGIN时生成" → ❌
    "RR的ReadView在第一次快照读时生成" → ✅
    想锁定 → START TRANSACTION WITH CONSISTENT SNAPSHOT
```

## 锁的分类：退化规则速查表

```
  ┌─────────────────────────────────────────────────────────────┐
  │  RR 隔离级别，等值查询 WHERE col = value                     │
  │                                                             │
  │  1. 唯一索引命中  → Record Lock（退化）                      │
  │  2. 唯一索引未命中 → Gap Lock（退化）                        │
  │  3. 普通索引命中   → Next-Key Lock + 后续Gap Lock + 回表锁   │
  │  4. 普通索引未命中 → Gap Lock（退化）                        │
  │  5. 无索引         → 全表Next-Key Lock = 锁全表              │
  └─────────────────────────────────────────────────────────────┘

  退化根因：唯一索引保证只有一行 → 不需要锁间隙防同值插入
  普通索引不退化：可能有重复值 → 必须锁间隙

  架构师实践：
    ① UPDATE/DELETE 必须走索引（否则锁全表）
    ② 唯一索引比普通索引锁范围小（能唯一就唯一）
    ③ RC 消除间隙锁（代价是幻读要业务兜底）
    ④ 范围查询注意尾部Gap Lock（多扫一行就多锁一个间隙）
```

## 死锁：四种模式与对应解法

```
  ┌─────────────────────┬──────────────────────┬───────────────┐
  │  死锁模式            │  解法                 │  适用场景      │
  ├─────────────────────┼──────────────────────┼───────────────┤
  │  AB-BA锁交叉        │  固定加锁顺序          │  转账/批量更新 │
  │  (本题场景)          │  Arrays.sort(ids)     │               │
  ├─────────────────────┼──────────────────────┼───────────────┤
  │  Gap Lock交叉       │  降RC 或 唯一索引      │  范围查询+插入 │
  │  + Insert Intention │  消除间隙锁            │               │
  ├─────────────────────┼──────────────────────┼───────────────┤
  │  同行不同索引锁      │  索引优化/减少索引数   │  多索引更新    │
  │  两个事务更新同一行   │                      │               │
  │  走不同索引          │                      │               │
  ├─────────────────────┼──────────────────────┼───────────────┤
  │  批量更新无序        │  排序后按顺序更新      │  批处理任务    │
  │                     │  ORDER BY id          │               │
  └─────────────────────┴──────────────────────┴───────────────┘

  排查SOP：
    SHOW ENGINE INNODB STATUS → 锁等待环路
    → information_schema/data_locks → 锁详情
    → INNODB_TRX → 长事务定位
    → 匹配四种模式 → 选解法
```

## 长事务：五层影响 + 五层根治

```
  ━━━ 五层影响 ━━━
    undo log  → purge阻塞→磁盘暴涨（根因：挡住其他事务版本清理）
    redo log  → checkpoint推进慢→文件撑满
    binlog    → 不完整写入→从库延迟
    锁        → 持有时长=事务时长→大面积超时
    主从复制  → 大事务回放慢→读写分离读到过期数据

  ━━━ 五层根治 ━━━

    L1 代码规范    → 事务内禁RPC/文件IO/耗时计算
    L2 ORM配置     → @Transactional(timeout=30) + statement-timeout=10
    L3 连接池      → leak-detection-threshold=60s 防连接泄漏
    L4 事务边界    → 最小化：事务内只做必须原子的操作
    L5 应用拆分    → 本地事务表+MQ代替分布式事务

    核心原则：事务内只做必须原子的事
    反模式：大事务包一切（查询+计算+RPC+MQ+通知）
    正确：事务只包"扣库存+创建订单"两步，其余全部事务外

  ━━━ 监控四指标 ━━━
    事务持续时间 > 30s/60s/300s 三级告警
    锁等待数量 > 10/50
    History list length > 1000/5000/10000（undo log堆积）
    行锁等待时间 > 1s/5s
```

## 高并发事务设计：Redis管快，DB管准

```
  ┌─────────────────────────────────────────────────────────┐
  │  为什么DB扣库存扛不住秒杀？                              │
  │                                                         │
  │  10万QPS → InnoDB行锁串行化 → 单行TPS上限≈500          │
  │  → 99995个请求等锁超时 → DB被打挂                       │
  │                                                         │
  │  解法：Redis预扣减 + MQ异步落库                         │
  │    Redis Lua: GET+DECR原子操作 → 10万QPS轻松扛         │
  │    售罄标记: stock=0时设标记 → 后续请求直接返回          │
  │    MQ异步: 秒杀服务不依赖DB和订单服务                    │
  └─────────────────────────────────────────────────────────┘

  三种扣减方案选型：
    悲观锁(FOR UPDATE) → 库存少+并发低（拍卖）
    乐观锁(version)    → 冲突率<5%（编辑保存）
    原子扣减(stock-1)  → 高并发（秒杀/抢购）

  Redis + DB对账架构：
    正常链路：Redis预扣→MQ→DB扣减（幂等+防超卖）
    对账链路：5分钟一次，Redis库存=DB库存=总库存-已售数
    不一致处理：以DB为准修正Redis
    三种不一致：Redis多扣/DB多扣/丢单

  面试一句话：
    "一致性在DB保证，性能在Redis保证，对账是桥梁"
```
