# Day 7：架构深挖 — InnoDB 锁机制底层原理深度剖析

## 深挖背景

本周 Day4 事务与锁机制中暴露三个能力差距：
- **差距一**：锁的加锁规则不够精确（说不清具体范围）
- **差距二**：隔离级别选型缺系统化框架
- **差距三**：锁的"粒度"和"模式"维度混淆

InnoDB 锁机制是 MySQL 区分"高级开发"和"架构师"的分水岭。今日深挖底层原理，把"知道概念"升级为"能逐行推演加锁过程"。

## 第 1 题：锁的物理存在形式 — 锁到底加在哪里？

### 1.1 行锁的物理存在形式

**结论**：InnoDB 的行锁**不加在数据行上**，而是加在**索引项（Index Record）的内存结构**上。

**底层数据结构**：
- 锁信息存储在 InnoDB 的**锁系统（lock system）内存结构**中，由全局哈希表 `lock_sys->rec_hash` 管理
- 每把锁的 key 是 `(space_id, page_no, heap_no)` — 即"表空间 ID + 页号 + 页内行号"
- 一把锁对应一个 `lock_t` 结构，记录了**事务 ID、锁模式（X/S/IX/IS）、锁粒度（Record/Gap/Next-Key）、被锁的索引项位图**

**为什么不加在数据行上**？
- 数据行在磁盘/Buffer Pool 中是被频繁读写的，加锁信息会污染数据页
- 锁是**逻辑概念**，行被删除后锁仍可能存在（事务未提交）
- 多个事务可能对同一行加不同模式的锁（如 IS 和 IX 兼容），需要独立的锁链

### 1.2 无索引表 UPDATE 为什么锁全表

**错误说法**："因为全表扫描，所以锁全表。"

**底层真相**：
1. `UPDATE t SET name='x' WHERE age=18`，无索引时优化器选择全表扫描
2. InnoDB 必须**对扫描到的每一行加 X 锁**（这是 2PL 两阶段锁协议的要求）
3. 由于没有索引可走，扫描路径是**聚簇索引的主键全扫描**
4. 每扫描一行就在该行的**主键索引项**上加 X 锁 + Next-Key Lock
5. 加完整张表的 Next-Key Lock = 锁全表 + 所有间隙

**关键点**：
- Server 层把所有行交给 InnoDB 时，InnoDB 必须先加锁再判断 WHERE 条件
- 即使 WHERE 不匹配（如 age≠18），锁也已经加上了
- **MySQL 5.7+ 优化**：Server 层判断 WHERE 不匹配时，会调用 `unlock_row()` 提前释放不匹配行的锁 — 但这是"加了再释放"，加锁瞬间仍是全表锁，并发场景下仍会阻塞

### 1.3 无索引列加 FOR UPDATE 的完整推演

`SELECT * FROM t WHERE name='x' FOR UPDATE`（name 无索引，只有主键 id）：

| 层次 | 行为 |
|------|------|
| **Server 层** | 调用 InnoDB 全表扫描接口，逐行获取数据，回到 Server 层做 WHERE 过滤 |
| **InnoDB Engine 层** | 扫描聚簇索引每一行，**先加 Next-Key Lock（X 模式）**，再返回给 Server 层 |
| **Server 层过滤** | 判断 name='x' 是否匹配，不匹配则调用 `handler::unlock_row()` 释放锁 |
| **5.7 之前** | unlock_row 实际不释放（避免幻读），所以全表都被锁 |
| **5.7+ RC 模式** | 真正释放不匹配行的锁，只保留匹配行的 X 锁 |
| **5.7+ RR 模式** | 不能释放（破坏可重复读语义），全表仍被锁 |

**架构师视角**：这就是为什么"无索引列的 UPDATE/DELETE/SELECT FOR UPDATE 是线上事故的高发场景" — RR 下直接锁全表，RC 下也有锁后释放的瞬间阻塞。

### 1.4 延伸：联合索引 (a,b,c) 上 WHERE a=1 AND c=3 FOR UPDATE

**加锁推演**：
1. 联合索引最左前缀原则：只能用 `a=1` 走索引，`c=3` 是索引下推或 Server 层过滤
2. 索引扫描范围：所有 `a=1` 的索引项
3. InnoDB 加锁：对所有 `a=1` 的索引项加 Next-Key Lock（X 模式）
4. 间隙锁范围：`(前一个a=0的最大索引项, 后一个a=2的最小索引项]`
5. **索引下推（ICP）优化**：MySQL 5.6+ 在引擎层就过滤 `c=3`，但**加锁仍然是对所有 a=1 的行**（ICP 只影响回表，不影响加锁）

**陷阱**：很多人以为 `c=3` 能减少锁范围，实际上 InnoDB 锁的是**索引扫描经过的所有项**，与 ICP 过滤无关。

## 第 2 题：Next-Key Lock 的"幻读"防御 — RR 真的能完全防止幻读吗？

### 2.1 快照读 vs 当前读防幻读机制

| 读类型 | 触发 SQL | 防幻读机制 |
|--------|----------|-----------|
| **快照读** | 普通 SELECT | **MVCC 版本链**：事务首次 SELECT 创建 ReadView，后续读都基于该 ReadView，看不到事务开始后新增的行 |
| **当前读** | SELECT FOR UPDATE / LOCK IN SHARE MODE / UPDATE / DELETE | **Next-Key Lock**：锁住扫描范围的所有索引项 + 间隙，防止其他事务插入 |

**核心区别**：
- 快照读用"看不见"防幻读（视觉层面）
- 当前读用"插不进"防幻读（物理层面）

### 2.2 RR 下仍能观察到幻读的反例

**反例 SQL 序列**：

```sql
-- 事务 A（RR 隔离级别）
START TRANSACTION;
SELECT * FROM t WHERE id > 100;  -- 快照读，假设返回 0 行，建立 ReadView

-- 事务 B
START TRANSACTION;
INSERT INTO t(id, name) VALUES (101, 'new');
COMMIT;

-- 事务 A 继续
SELECT * FROM t WHERE id > 100;  -- 仍返回 0 行（MVCC 看不见）
UPDATE t SET name='updated' WHERE id > 100;  -- 居然影响 1 行！
SELECT * FROM t WHERE id > 100;  -- 返回 1 行！幻读出现
```

**为什么会幻读**？
1. 第一次 SELECT 是快照读，基于事务 A 的 ReadView
2. 事务 B 插入并提交后，新行的 `trx_id` 在事务 A 的 ReadView 中是"未来事务"，看不见
3. 事务 A 的 UPDATE 是**当前读**，会读到最新版本，把事务 B 插入的行 UPDATE 了
4. UPDATE 后，事务 B 插入的行的 `trx_id` 变成事务 A 自己，事务 A **能看见自己的修改**
5. 第三次 SELECT 虽然还是快照读，但因为是"自己修改过的行"，所以能看见

### 2.3 RR 防幻读的语义边界

**核心边界**：RR 防幻读 = "快照读不会看到新行" + "当前读用间隙锁阻止新行插入"，但**两种读混用时，幻读仍可能出现**。

**向团队解释**：
> "RR 防的是同一类型的读之间的幻读。如果你在同一个事务里混用 SELECT 和 UPDATE/DELETE，UPDATE 会把别人新插入的行变成你自己的修改，下次 SELECT 就能看见。要彻底避免幻读，要么全程当前读（SELECT FOR UPDATE），要么业务上明确不允许并发插入相同范围。"

### 2.4 延伸：阿里规约 RR→RC + 业务层唯一性兜底的因果链

**因果关系**：
1. RR 改 RC 后，间隙锁消失 → 死锁减少、锁范围缩小、性能提升
2. 但间隙锁消失 → 无法阻止"先 SELECT 检查不存在，再 INSERT"的并发场景
3. 例：注册时先 SELECT 用户名是否存在，再 INSERT —— RR 下间隙锁能保护，RC 下两个并发请求都能 INSERT 成功，导致重复用户名
4. **业务层兜底方案**：
   - 加**唯一索引**（最可靠，靠 InnoDB 的唯一性约束）
   - INSERT ON DUPLICATE KEY UPDATE / INSERT IGNORE
   - 业务层先 INSERT 再判断异常（不要先 SELECT 后 INSERT）

**架构师结论**：RC + 唯一索引兜底 = 大多数 OLTP 场景的最优组合。

## 第 3 题：半一致性读（Semi-Consistent Read）— RC 的杀手锏

### 3.1 触发条件

**触发条件**（三个全部满足）：
1. **隔离级别 = RC**（或 `innodb_locks_unsafe_for_binlog=ON`，已废弃）
2. **语句类型 = UPDATE**（DELETE 不触发，SELECT 不触发）
3. **binlog 格式 = ROW 或 MIXED**（STATEMENT 不行，会导致主从不一致）

**为什么 SELECT 不触发**？
- SELECT FOR UPDATE 的语义就是"必须锁住读到的行"，跳过会破坏用户预期
- 普通 SELECT 是快照读，不加锁，没有触发场景

**为什么 DELETE 不触发**？
- DELETE 的语义是"删除匹配的行"，如果跳过，会漏删
- UPDATE 的语义是"修改匹配的行"，跳过只是不修改，业务上可以接受（最后会重新读）

### 3.2 RR vs RC 高并发资金扣减行为对比

**场景**：10 个并发事务执行 `UPDATE account SET balance=balance-10 WHERE user_id=100 AND status='active'`

**RR 下行为**：
```
T1: 持有 user_id=100 行的 X 锁
T2-T10: 全部阻塞，等待 T1 释放锁
T1 提交后，T2 获取锁，再次阻塞 T3-T10
...
吞吐量 = 1 / (单事务耗时)，串行执行
```

**RC + 半一致性读行为**：
```
T1: 持有 user_id=100 行的 X 锁，准备 UPDATE
T2: 也想 UPDATE 同一行，触发半一致性读：
    1. 读取 user_id=100 行的最新已提交版本（不等锁）
    2. 检查 status='active' 是否仍匹配
    3. 如果匹配 → 等锁
    4. 如果不匹配（如 T1 同时把 status 改成 'frozen'）→ 跳过这一行，不加锁
T3-T10: 同理
```

**关键差异**：
- RR 下：10 个事务全部阻塞，每个都要等
- RC 下：如果 WHERE 条件**已经被前一个事务改变**，后续事务直接跳过，不阻塞
- 跳过的事务的**最终结果**：UPDATE 影响行数 = 0，需要应用层根据返回值做业务判断

**这个特性的真正价值**：
- 减少锁等待，减少死锁
- 让 UPDATE 的"乐观行为"在数据库层就生效，不用应用层重试

### 3.3 半一致性读会破坏一致性吗？

**结论**：不会破坏一致性，但会改变"可见性语义"。

**MVCC 版本可见性论证**：
1. 半一致性读读取的是"最新已提交版本"，这个版本满足 ACID 的 D（持久性）
2. 半一致性读**不是基于事务 ReadView**，而是基于"全局最新已提交"，所以是单点决策，不会破坏事务一致性
3. UPDATE 跳过的行只是"放弃修改"，不会读到任何"中间状态"
4. 跳过决策是"读后丢弃"，对其他事务不可见

**但有一个副作用**：UPDATE 的"匹配语义"从"事务开始时匹配"变成了"执行那一刻匹配"，所以**不能用于"必须基于事务开始时状态做修改"的场景**。

### 3.4 延伸：业务场景适用边界

**适用场景**（半一致性读有价值）：
- ✅ **蛋壳钱包账户冻结**：`UPDATE account SET status='frozen' WHERE user_id=X AND status='active'` — 状态已变就跳过，业务正确
- ✅ **保险商城订单状态机推进**：`UPDATE order SET status='paid' WHERE order_id=X AND status='created'` — 状态已变说明被并发处理过
- ✅ **IM 客服会话分配**：`UPDATE session SET agent_id=Y WHERE session_id=X AND agent_id IS NULL` — 已分配就跳过

**不适用场景**（必须 RR 或显式锁）：
- ❌ **余额计算依赖事务开始时的快照**：如"按事务开始时余额计息" — 半一致性读会读到最新余额
- ❌ **多表关联的强一致更新**：如转账中"扣款 + 加款"必须基于事务开始时的快照
- ❌ **对账业务**：必须看到事务开始时的一致性快照

**边界判断口诀**：**"语义上能接受'被人改了就跳过'用 RC，必须'基于开始时快照'用 RR"**。

## 第 4 题：死锁检测的代价 — 高并发下为什么要关掉 innodb_deadlock_detect？

### 4.1 死锁检测算法本质

**算法本质**：**有向图回路检测（DFS 深度优先搜索）**

**数据结构**：
- InnoDB 维护一个**等待图（wait-for graph）**
- 节点 = 事务，边 = "事务 A 等待事务 B 持有的锁"
- 死锁 = 等待图中存在回路

**触发时机**：
- 每次事务**加锁失败需要等待**时，立即触发一次 DFS
- DFS 从当前事务出发，沿"等待边"递归查找
- 找到回路 → 选择一个事务作为"受害者"回滚（通常是 undo 量最小的）

**时间复杂度**：
- 单次检测：O(V+E)，V = 等待事务数，E = 等待关系数
- 高并发热点行：每个等待事务都可能与其他所有等待事务有关系 → E ≈ V²
- 单次检测复杂度 ≈ **O(n²)**

### 4.2 高并发热点行为什么是性能杀手

**推导**：
- 假设秒杀场景，1000 个事务同时抢同一行库存
- 第 1 个事务持有锁，第 2~1000 个事务进入等待队列
- 每个新事务进入等待时，触发一次 O(n²) 检测
- 1000 个事务的总检测开销 = O(n³) = **10 亿次操作**
- 死锁检测持有 `lock_sys->mutex` 全局互斥锁 → CPU 全部消耗在锁检测上，TPS 暴跌

**真实案例**：阿里曾在某次大促前压测发现，开启死锁检测时 TPS 只有 3 万，关闭后飙升到 30 万。

### 4.3 关掉死锁检测后的兜底策略

**核心参数组合**：
```sql
SET GLOBAL innodb_deadlock_detect = OFF;
SET GLOBAL innodb_lock_wait_timeout = 1;  -- 锁等待超时改为 1 秒（默认 50）
```

**策略本质**："**用超时代替检测**" — 不检测回路，靠超时自动打破死锁。

**代价**：
1. **死锁恢复变慢**：原来检测到立即回滚一个事务，现在所有死锁事务都要等 timeout
2. **正常锁等待也会超时**：业务上锁竞争激烈的事务会失败，需要应用层重试
3. **必须配合应用层重试**：捕获 `Lock wait timeout exceeded` 异常，重试事务

### 4.4 延伸：秒杀系统三层防死锁完整方案

**数据库层**：
1. 关闭死锁检测，超时设为 1 秒
2. 库存表设计极简：`(item_id PK, stock INT)`，只有主键索引
3. 扣减用 `UPDATE stock SET stock=stock-1 WHERE item_id=X AND stock>0`（避免事务包裹的"读-改-写"）
4. 单行操作 + 隐式事务（autocommit=1）

**应用层**：
1. **请求合并**：相同商品的扣减请求在应用层聚合，1 秒内合并提交
2. **限流降级**：每秒只放固定数量请求到数据库
3. **异步化**：扣减成功后，订单生成走异步队列，不阻塞主流程
4. **重试策略**：捕获锁超时，指数退避重试 + 最多 3 次

**缓存层**：
1. **Redis 预扣**：库存预扣放 Redis，原子操作 `DECR`，决定哪些请求能进入数据库
2. **Redis 兜底**：库存为 0 时直接返回"已售罄"，请求不进数据库
3. **异步对账**：Redis 和数据库定期对账，发现差异告警
4. **持久化保证**：Redis 用 AOF + 主从，避免缓存丢失导致超卖

**架构师思维**：数据库层负责**最后一道防线**，缓存层负责**削峰**，应用层负责**编排** — 数据库永远不应该是高并发的入口。

## 第 5 题：意向锁的存在意义 — 为什么需要 IS/IX？

### 5.1 意向锁的本质

**结论**：意向锁是**表级锁**，但表达的是"**表中有行被锁的意图**"。

**加锁位置**：
- 物理上加在**表的元数据**上（`table->locks` 链表）
- 一个事务对表加 IX/IS 锁，意味着"我将要在这张表的某些行上加 X/S 锁"

**加锁时机**：
- 任何行级 X 锁前，自动先加表级 IX 锁
- 任何行级 S 锁前，自动先加表级 IS 锁
- 这两步在同一原子操作中完成，业务无感知

### 5.2 没有意向锁会怎样？

**没有意向锁的场景**：
- 要执行 `LOCK TABLES t WRITE`（表级 X 锁）
- 必须先判断"表中是否有任何行被锁"
- 不知道哪些行被锁 → 必须**扫描所有行**检查锁状态 → O(n) 全表扫描

**这不可接受的原因**：
- 大表（千万行）的 DDL 要等几分钟检查锁
- 检查期间还要阻止新事务加锁，影响并发

**意向锁的优化**：
- 表级锁直接看表的 IX/IS 标志，**O(1) 判断**
- 加 X 锁前发现表上有 IX → 立即知道"有行被锁"，无需扫描
- 把"判断表有无行锁"从 O(n) 降到 O(1)

### 5.3 完整的锁兼容矩阵

|        | X | IX | S | IS |
|--------|---|----|----|----|
| **X**  | ✗ | ✗  | ✗ | ✗  |
| **IX** | ✗ | **✓** | ✗ | ✓  |
| **S**  | ✗ | ✗  | ✓ | ✓  |
| **IS** | ✗ | ✓  | ✓ | ✓  |

**关键点解析**：

**为什么 IX 和 IX 兼容**？
- IX 只表达"我要在某些行上加 X 锁"的意图
- 两个事务都想加行级 X 锁，**只要不是同一行**就不冲突
- 真正的冲突检测交给**行级锁**去做
- 表级兼容 + 行级冲突 = 高并发的关键

**为什么 X 和 IX 不兼容**？
- X 是表级排他锁，要求"整张表没有任何锁"
- IX 意味着"有行被加锁"，违反 X 的要求
- 所以 X 必须等所有 IX 释放（即所有行级锁释放）

**为什么 IS 和 IX 兼容**？
- IS：要在某些行加 S 锁
- IX：要在某些行加 X 锁
- 不同行 → 不冲突；同行 → 行级锁层面去判断

### 5.4 延伸：Online DDL 与意向锁

**传统 ALTER TABLE 流程**：
1. 申请表级 X 锁（**MDL 写锁**）
2. 等所有 IX 释放（等所有事务结束）
3. 拷贝表、重建索引
4. 释放锁

**问题**：步骤 1 会阻塞所有 DML，大表 DDL 时业务停摆。

**MySQL 5.6+ Online DDL**：
- 申请 **MDL 升级锁** → 短暂升级到 MDL 写锁拿元数据 → 降级到 MDL 读锁
- 持有 MDL 读锁期间允许 DML（IX 锁兼容 MDL 读锁）
- DML 产生的变更写入 **online log**，DDL 完成后回放
- 最后再次申请 MDL 写锁应用 online log

**pt-osc / gh-ost 的优势**：
- 完全在用户态完成，**不依赖 MySQL 内置 Online DDL**
- pt-osc：用触发器同步数据 → 老表锁定时间极短
- gh-ost：解析 binlog 同步数据 → 完全无锁（不用触发器），对主库压力最小
- **两者都避免了长时间持有 MDL 写锁，意向锁兼容性让 DML 持续可执行**

**架构师选型**：
- 小表（< 1G）：直接 Online DDL
- 中大表（1G-100G）：pt-osc
- 超大表（> 100G）+ 高并发：gh-ost（无触发器，对主库 0 压力）

## 第 6 题：综合实战 — 死锁排查 SOP 的底层推演

### 6.1 完整死锁排查 SOP

**四步 SOP**：

**Step 1：告警识别（监控层）**
- 监控指标：`Innodb_row_lock_waits`、`Innodb_deadlocks`、`Innodb_row_lock_time_avg`
- 应用层：`Deadlock found when trying to get lock; try restarting transaction` 异常计数
- 告警阈值：1 分钟内死锁 > 10 次 / 平均锁等待 > 100ms

**Step 2：现场抓取（数据库层）**
```sql
-- 抓取最近一次死锁详情
SHOW ENGINE INNODB STATUS\G

-- 抓取当前持锁和等待
SELECT * FROM performance_schema.data_locks;
SELECT * FROM performance_schema.data_lock_waits;

-- 抓取长事务
SELECT * FROM information_schema.innodb_trx
WHERE TIME_TO_SEC(TIMEDIFF(NOW(), trx_started)) > 5;
```

**Step 3：根因定位（分析层）**
- 解析 `LATEST DETECTED DEADLOCK` 段
- 还原两个事务的 SQL 执行顺序
- 找到循环等待的两把锁是什么对象

**Step 4：修复与根治（架构层）**
- 短期：调整 SQL 加锁顺序、加索引消除间隙锁
- 中期：拆分大事务、改隔离级别为 RC
- 长期：业务层改造（如热点行打散、改异步队列）

### 6.2 SHOW ENGINE INNODB STATUS 关键字段解读

**示例输出**：
```
*** (1) TRANSACTION:
TRANSACTION 4231, ACTIVE 5 sec starting index read
mysql tables in use 1, locked 1
LOCK WAIT 3 lock struct(s), heap size 1136, 2 row lock(s)
*** (1) WAITING FOR THIS LOCK TO BE GRANTED:
RECORD LOCKS space id 42 page no 5 n bits 72 index PRIMARY of table `db`.`account`
trx id 4231 lock_mode X locks rec but not gap waiting
```

**字段含义**：

| 字段 | 含义 |
|------|------|
| `space id 42` | 表空间 ID = 42（每张独立表空间一个 ID） |
| `page no 5` | 数据页号 = 5（页大小默认 16KB） |
| `n bits 72` | 锁位图大小 = 72 bit（表示该页能容纳的最大行数） |
| `index PRIMARY` | 锁加在主键索引上 |
| `lock_mode X` | 锁模式 = 排他锁 |
| `locks rec but not gap` | 锁粒度 = 仅记录锁（不锁间隙）— 通常是唯一索引等值命中或 RC 模式 |
| `waiting` | 当前在等待此锁 |

**其他常见锁粒度描述**：
- `locks gap before rec` = 仅间隙锁
- `locks rec but not gap` = 记录锁（Record Lock）
- `locks rec and gap` = Next-Key Lock（默认）
- `lock_mode S` = 共享锁
- `lock_mode IX` = 意向排他锁

### 6.3 真实死锁日志反推

**死锁场景**：
- 事务 A：持有 row1 的 X 锁，等待 row2 的 X 锁
- 事务 B：持有 row2 的 X 锁，等待 row1 的 X 锁

**SQL 形态反推**：

**场景 1：更新顺序不同**
```sql
-- 事务 A
UPDATE account SET balance=balance-100 WHERE id=1;  -- 锁 row1
UPDATE account SET balance=balance+100 WHERE id=2;  -- 等 row2

-- 事务 B（转账反向）
UPDATE account SET balance=balance-100 WHERE id=2;  -- 锁 row2
UPDATE account SET balance=balance+100 WHERE id=1;  -- 等 row1
```

**场景 2：间隙锁交错**
```sql
-- 事务 A
SELECT * FROM t WHERE id BETWEEN 1 AND 10 FOR UPDATE;  -- 锁 [1,10] 间隙
INSERT INTO t VALUES (15, ...);  -- 等 [11,20] 间隙

-- 事务 B
SELECT * FROM t WHERE id BETWEEN 11 AND 20 FOR UPDATE;  -- 锁 [11,20] 间隙
INSERT INTO t VALUES (5, ...);  -- 等 [1,10] 间隙
```

**循环等待形成原因**：
- 两个事务都已持有部分资源
- 各自需要的下一个资源被对方持有
- 形成 A→B→A 的等待回路

**修复方案（3 种）**：

| 方案 | 实现 | 适用场景 |
|------|------|---------|
| **统一加锁顺序** | 应用层强制按主键升序加锁（如转账时 min(id1,id2) 先锁） | 转账、多账户操作 |
| **降低锁粒度** | 用 RC 隔离级别消除间隙锁；加索引让锁范围更精确 | 范围查询为主的业务 |
| **改异步队列** | 把竞争同一行的操作放队列串行化（如同一商品扣减进同一队列） | 秒杀、热点账户 |
| **乐观锁替代** | 用版本号 CAS 替代 SELECT FOR UPDATE | 冲突率低的场景 |

### 6.4 延伸：架构设计阶段的防死锁原则

**蛋壳钱包转账场景的设计原则**：

**原则 1：固定加锁顺序**
- 任何涉及多账户的事务，**强制按账户 ID 升序加锁**
- 应用层封装在转账服务中，开发人员无感知
- 代码示例：`Arrays.sort(accountIds); for(id: accountIds) lock(id);`

**原则 2：缩短事务边界**
- 事务内只做必要的 DB 操作，**禁止 RPC、消息发送、文件 IO**
- 大事务拆小：批量转账拆成逐笔
- 单事务持锁时间 < 100ms

**原则 3：选对分片键**
- 资金路由表分片键 = `user_id`，保证同一用户的所有操作落到**同一分片**
- 避免跨分片事务（XA / TCC 死锁概率指数级上升）
- 同分片内的死锁可以靠 InnoDB 检测，跨分片死锁基本无法检测

**原则 4：热点行打散**
- 平台账户（大商户）容易成为热点
- 用**子账户分桶**：1 个商户账户拆 100 个子账户，按时间或 hash 路由
- 子账户加锁不冲突，定期合并到主账户

**原则 5：异步化非核心路径**
- 转账成功后的通知、对账、日志等走 MQ 异步
- 核心事务只做"扣款 + 加款 + 流水"
- 减少事务持有锁的时间

**原则 6：监控驱动设计**
- 死锁率 > 0.1% 必须复盘
- 每次新功能上线前压测死锁场景
- 建立"锁等待时长 P99"基线，回归测试
