# Day 3：分布式锁

## 题目

你的订单系统有个场景：用户抢购秒杀商品，库存只剩 1 件，但 10 个请求同时到达。

不加锁的后果很明显：10 个请求都读到库存=1，都扣减成功，超卖了。

请回答：
1. 用 Redis 怎么实现分布式锁？`SET key value NX EX` 够用吗？什么场景下会出问题？
2. Redisson 的看门狗机制解决了什么问题？它的续期逻辑是怎样的？什么情况下看门狗会失效？
3. Redis 集群下（主从切换），锁会丢吗？Redlock 是什么？为什么 Martin Kleppmann 说 Redlock 不靠谱，Antirez 怎么反驳的？
4. 你的秒杀场景会选 Redis 锁还是 ZooKeeper 锁？如果 QPS 是 10 万呢？

## 答案

### 1. Redis 分布式锁的基本实现与问题

**基础实现：**
```
加锁：SET lock_key unique_value NX EX 30
  NX — key 不存在才设置（互斥）
  EX 30 — 30 秒过期（防死锁）
  unique_value — 标识持有者，解锁时验证（防误删）

解锁（Lua 脚本保证原子性）：
  if redis.call("get", KEYS[1]) == ARGV[1] then
      return redis.call("del", KEYS[1])
  else
      return 0
  end
```

**为什么解锁要用 Lua？** 因为 GET + DEL 不是原子操作：

```
线程A GET → 值匹配，准备 DEL
    ↓ 此时锁过期
线程B 加锁成功
    ↓
线程A DEL → 把线程B的锁删了
    ↓
线程C 也加锁成功 → 两个线程同时持有锁
```

**SET NX EX 够用吗？不够，三个经典问题：**

| 问题 | 场景 | 后果 |
|------|------|------|
| 锁过期释放 | 业务执行超过 30s，锁自动过期 | 其他线程拿到锁，并发执行 |
| 误删别人的锁 | 上面 Lua 脚本的场景 | 同上 |
| 不可重入 | 同一线程嵌套调用 | 自己锁自己，死锁 |

误删用 Lua 解决了，锁过期和不可重入需要更高级的方案。

### 2. Redisson 看门狗机制

**锁过期问题的本质：** EX 时间设短了业务没执行完，设长了宕机恢复慢。你无法预知业务执行时间。

**看门狗的逻辑：**

```
1. 加锁时不设过期时间（或设 leaseTime = -1）
2. Redisson 默认设置 lockWatchdogTimeout = 30s
3. 加锁成功后，启动一个后台线程（看门狗）
4. 每隔 10s（30s / 3）检查：锁还被当前线程持有吗？
   ├─ 是 → 续期到 30s（重置过期时间）
   └─ 否 → 停止续期
5. 业务执行完，主动释放锁，看门狗停止
```

```
时间线：
0s    加锁成功，过期时间=30s，看门狗启动
10s   看门狗检查 → 锁还在 → 续期到 30s
20s   看门狗检查 → 锁还在 → 续期到 30s
25s   业务执行完，主动释放锁，看门狗停止
```

**看门狗失效的场景：**

| 场景 | 后果 |
|------|------|
| 持有锁的进程崩溃 | 看门狗线程也死了，无人续期，30s 后锁自动释放 ✓（这是正确行为） |
| 持有锁的进程 GC 暂停（STW） | 看门狗暂停续期，锁过期，其他线程拿到锁 → 并发执行 ✗ |
| 持有锁的进程网络分区 | 看门狗无法续期，锁过期，其他节点拿到锁 → 并发执行 ✗ |
| 手动设置了 leaseTime | 看门狗不启动（Redisson 认为你自己管理过期时间），锁到期直接释放 |

> 核心认知：看门狗解决的是"业务还没完锁就过期"的问题，解决不了"进程活着但和 Redis 断连"的问题。后者的本质是——Redis 锁不是强一致锁。

### 3. 主从切换丢锁与 Redlock 之争

**主从切换丢锁：**

```
1. 客户端A 在 Master 加锁成功
2. Master 还没把锁数据同步到 Slave
3. Master 宕机，Slave 提升为新 Master
4. 客户端B 在新 Master 加锁成功
5. 客户端A 和客户端B 同时持有锁 → 并发执行
```

**Redlock 算法（Antirez 提出）：**

用 5 个独立的 Redis 实例（不是主从集群），加锁时向所有实例发送 SET NX：

```
1. 记录开始时间 T1
2. 依次向 5 个 Redis 实例加锁
3. 计算成功加锁的实例数 N，和总耗时 T = T2 - T1
4. 如果 N >= 3（多数派）且 T < 锁过期时间 → 加锁成功
5. 加锁失败 → 向所有实例发送解锁请求
```

**Martin Kleppmann 的批评（2016）：**

1. **GC/网络延迟导致锁失效** — 进程 GC 暂停后，以为还持有锁，但锁已过期并被别人获取
2. **时钟跳跃问题** — Redlock 依赖各节点时钟，如果某节点时钟跳变，过期时间不准
3. **fencing token 更靠谱** — 用单调递增的 token（如 ZK 的 zxid），资源服务只接受更大的 token

```
Martin 的 fencing token 方案：
  客户端A 拿到锁，fencing token = 1
  客户端A GC 暂停，锁过期
  客户端B 拿到锁，fencing token = 2
  客户端A 恢复，写存储 → 存储服务检查 token=1 < 2，拒绝
```

**Antirez 的反驳：**

1. GC 暂停的问题所有分布式锁都有，ZK 也面临
2. 时钟跳跃可以控制在毫秒级，实际影响很小
3. fencing token 需要资源服务配合，不是所有场景都能改

**结论：**

| 方案 | 一致性 | 性能 | 适用场景 |
|------|-------|------|---------|
| Redis 单节点锁 | 弱 | 最高 | 允许极小概率并发 |
| Redlock | 较强 | 中等（5 次网络往返） | 对安全性有一定要求 |
| ZK 锁 | 强（CP） | 低 | 绝对不能并发 |
| fencing token | 最强 | 取决于锁实现 | 资源服务可控时 |

> 面试建议：不要站队，说清两边的观点，然后根据场景选。Martin 的批评核心是"不要把分布式锁当绝对互斥来用"，这个观点是对的。

### 4. 秒杀场景选 Redis 锁还是 ZK 锁？

**先问一个更根本的问题：秒杀场景该用分布式锁吗？**

秒杀的本质是"库存扣减"，不是"互斥访问"。分布式锁是串行化手段，10 万 QPS 全去抢一把锁，吞吐量直接崩。

**正确的秒杀方案分层：**

```
层1：前端限流 — 按钮置灰、验证码、滑动拼图
  ↓ 拦截 90% 请求

层2：网关限流 — 令牌桶/漏桶，单机 QPS 上限
  ↓ 再拦 90%

层3：Redis 原子扣减 — Lua 脚本：判断库存 + 扣减一步完成
  ↓ 不需要分布式锁，Redis 单线程天然串行

层4：MQ 异步下单 — 扣减成功后发 MQ，订单服务异步创建
  ↓ 削峰填谷

层5：数据库兜底 — 库存表乐观锁（UPDATE ... WHERE stock > 0）
```

**Redis Lua 扣减（层3，替代分布式锁）：**
```lua
local stock = redis.call("get", KEYS[1])
if not stock or tonumber(stock) <= 0 then
    return -1
end
redis.call("decr", KEYS[1])
return tonumber(stock) - 1
```

这个方案不需要分布式锁，因为 Redis 单线程执行 Lua 脚本，天然保证原子性。

**回到问题：如果非要用分布式锁呢？**

| 场景 | 选择 | 原因 |
|------|------|------|
| 秒杀扣库存 | 都不用 | Lua 原子扣减比锁快 100 倍 |
| 抢红包 | Redis 锁 | 允许极小概率不一致，性能优先 |
| 选主/分布式锁 | ZK 锁 | 绝对不能两个 Leader |
| 防重复提交 | Redis 锁 | 短时持有，过期自动释放 |

**QPS 10 万的判断：**

QPS 10 万意味着每秒 10 万次加锁/解锁操作。ZK 的吞吐量大概 1-2 万 QPS，扛不住。Redis 单节点 10 万 QPS 勉强能扛，但锁本身就成了瓶颈。

所以 QPS 10 万的秒杀，**不要用分布式锁**，用 Redis Lua 原子操作 + MQ 异步 + 数据库乐观锁的分层方案。
