## 一、熔断器在高可用链路中的定位

### 1.1 核心一句话
> **熔断 = 下游故障时的"自动保险丝"，用快速失败换取线程池不被打满、避免级联雪崩**

- 限流挡的是"入口流量超额"
- 熔断挡的是"下游慢/故障"引发的次生灾害
- 降级是熔断触发后的"善后动作"

### 1.2 为什么需要熔断——雪崩的物理过程

```
下游接口偶发慢（99线 50ms → 2s）
  → 上游调用方线程被占用
    → 上游线程池打满
      → 上游接口拒绝新请求 / 超时堆积
        → 上游的上游也开始堆积
          → 整条调用链雪崩
```

关键点：**慢调用比错误调用更危险**——错误是瞬时返回，慢调用会长期占用线程。

## 二、熔断器状态机

### 2.1 三态模型

```
       ┌──────────────────────────────────────┐
       │                                      │
       ▼                                      │
  ┌─────────┐  指标超阈值  ┌─────────┐  探测成功  │
  │ CLOSED  │ ──────────► │  OPEN   │ ────────► │
  │ (正常放行) │             │ (快速失败) │           │
  └─────────┘             └────┬────┘           │
       ▲                       │ 等待恢复窗口     │
       │                       ▼                │
       │ 探测全部成功      ┌────────────┐        │
       └────────────────── │ HALF_OPEN  │ ───────┘
                           │ (半开探测)  │
                           └─────┬──────┘
                                 │ 探测失败
                                 ▼
                              回到 OPEN
```

### 2.2 各状态的语义

| 状态 | 行为 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| CLOSED | 正常放行请求，统计指标 | 初始状态 / HALF_OPEN 探测全部成功 | 指标超阈值 → OPEN |
| OPEN | 快速失败，不调用下游，直接走 fallback | 统计窗口内慢调用比例 / 异常比例超阈值 | 等待 `recovery timeout` 后 → HALF_OPEN |
| HALF_OPEN | 放行有限探测请求（如 1~10 个） | OPEN 等待恢复时间到 | 探测成功 → CLOSED；失败 → OPEN |

### 2.3 关键参数

- `failureRateThreshold`：异常比例阈值（如 50%）
- `slowCallRateThreshold`：慢调用比例阈值（如 50%，慢定义由 `slowCallDurationThreshold` 决定）
- `minimumNumberOfCalls`：最少样本数，避免样本不足误判
- `waitDurationInOpenState`：OPEN 状态持续时间（熔断冷却期）
- `permittedNumberOfCallsInHalfOpenState`：HALF_OPEN 探测请求数
- `slidingWindowSize` / `slidingWindowType`：滑窗大小与类型（count-based / time-based）

## 三、三大主流实现对比

| 维度 | Hystrix（停止维护） | Resilience4j | Sentinel |
| --- | --- | --- | --- |
| 编程模型 | 命令模式（继承 `HystrixCommand`） | 函数式（装饰器 `CircuitBreaker.decorateSupplier`） | 注解 + API（`@SentinelResource` / `SphU.entry`） |
| 滑窗实现 | 桶式（BucketedCounterStream，基于 RxJava） | 滑窗（Count-based 或 Time-based，可配置） | LeapArray 滑动窗口（时间桶+样本数组） |
| 主指标 | 异常比例 + 慢调用（HystrixCommandProperties） | 慢调用比例 + 异常比例，可独立配置 | 慢调用比例 / 异常比例，可独立配置 |
| HALF_OPEN 探测 | 单次探测（半开期放行 1 个请求） | 可配置 N 个探测请求 | 探测期放行 N 个请求 |
| 限流能力 | 无（需配合其他组件） | 有 RateLimiter 模块 | 原生支持（流控+熔断一体） |
| 系统自适应 | 无 | 无 | 支持（Load 自适应） |
| 控制台 | Hystrix Dashboard（简陋） | 无原生 | Sentinel Dashboard（可视化规则推送） |
| 维护状态 | 2018 年起停止维护 | 活跃 | 活跃（阿里） |

### 3.1 Hystrix 为什么被淘汰

1. **命令模式重**：每个隔离点要写一个 `HystrixCommand` 子类，侵入性强
2. **基于 RxJava 1.x**：响应式模型过重，依赖陈旧
3. **不支持函数式组合**：无法像 Resilience4j 那样用装饰器链组合熔断、限流、重试、重试
4. **可观测性弱**：Dashboard 只能看断路器状态，规则推送能力差
5. **半开探测过于保守**：只放行 1 个请求，恢复慢

### 3.2 Resilience4j 的函数式改进

```java
Supplier<String> supplier = Decoration.ofSupplier(() -> callRemote())
    .decorate(CircuitBreaker.decorateSupplier(cb))
    .decorate(RateLimiter.decorateSupplier(rl))
    .decorate(Retry.decorateSupplier(retry, 3))
    .decorate(Bulkhead.decorateSupplier(bp))
    .get();
```

- 装饰器链：熔断 → 限流 → 重试 → 舱壁，顺序可调
- 无侵入：业务方法仍是纯函数
- 与 Spring Boot / Reactor 友好

### 3.3 Sentinel 的滑窗数据结构——LeapArray

- `LeapArray` 是一个**时间桶环形数组**，每个桶（`WindowWrap`）保存样本（`MetricBucket`）
- 限流、熔断共用同一套 LeapArray 结构，但读不同维度（通过量 / 异常数 / RT）
- 这就是为什么 Sentinel 能做"慢调用比例熔断"——它本身就在统计 RT
- 而 Guava RateLimiter 只维护令牌发放，**不统计 RT**，所以做不到熔断

## 四、降级策略分层设计

### 4.1 五层降级模型

| 层级 | 名称 | 触发条件 | 动作 | 秒杀场景例子 |
| --- | --- | --- | --- | --- |
| L1 | 兜底返回 | 单点接口故障 | 返回缓存/默认值 | 商品详情页降级为静态缓存 |
| L2 | 弱依赖熔断 | 非核心依赖慢 | 熔断+返回兜底 | 推荐系统挂了，秒杀页不展示推荐位 |
| L3 | 强依赖异步化 | 核心链路压力 | 同步转异步 | 下单成功消息改异步发送 |
| L4 | 读写降级 | DB 压力大 | 读走缓存 / 写暂存MQ | 库存查询走 Redis，下单写不直接落 DB |
| L5 | 流量入口裁剪 | 全链路过载 | 入口限流/排队 | 秒杀按钮置灰、答题验证码削峰 |

### 4.2 强依赖能不能降级

**原则：强依赖不降级，只熔断+重试+集群容灾**

- 扣减库存是强依赖：不能返回"假装扣成功"
- 方案：
  1. **本地缓存 + 异步扣减**：Redis 预扣 + MQ 异步落 DB
  2. **多副本**：Redis 主从 + Cluster，DB 主从
  3. **熔断保护**：Redis 集群熔断后走本地缓存兜底（牺牲一致性保可用）
  4. **限流前置**：在调用库存服务前限流，避免把库存服务打挂

### 4.3 降级触发方式

| 方式 | 触发主体 | 适用场景 | 风险 |
| --- | --- | --- | --- |
| **自动熔断降级** | 框架（Sentinel/CB） | 下游故障实时响应 | 误判、抖动恢复慢 |
| **手动开关降级** | 运维/SRE | 灰度、紧急止血 | 依赖人，响应慢 |
| **预案预降级** | 大促前提前执行 | 已知流量高峰 | 牺牲常态能力 |

**为什么大促前做预降级？**
- 大促瞬间流量是常态 10x+，等自动熔断触发时已晚
- 提前关闭非核心（推荐、评论、积分写入），把资源留给核心链路
- 提前预热缓存、提前扩容、提前降级，是"三预"标准动作

### 4.4 兜底数据设计

| 选项 | 适用场景 | 例子 |
| --- | --- | --- |
| 静态缓存 | 数据可旧、必须返回 | 商品详情页 → CDN 静态页 |
| 默认值 | 数据可缺失、不可空 | 推荐位 → 默认热门商品 |
| 空 | 前端可容忍空态 | 评论列表 → 空数组 |
| 稍后重试 | 强一致、不可旧 | 支付结果 → "查询中，请稍后" |

**选型原则**：业务能否容忍"旧" / "缺" / "无"。

### 4.5 真实降级案例模板（参考蛋壳钱包 / 平安健康险）

> 触发条件：支付通道接口 P99 > 2s 持续 1 分钟
> 降级动作：自动熔断该支付通道，路由到备用通道；前端展示"该通道暂不可用"
> 业务感知：用户切换支付方式，不影响下单
> 回滚流程：通道恢复后 HALF_OPEN 探测连续 5 次成功 → CLOSED → 重新路由

## 五、架构师视角的常见踩坑

### 5.1 熔断配置的常见错误

1. **`minimumNumberOfCalls` 设太小**：流量低时被噪声误触发
2. **`waitDurationInOpenState` 太短**：没等下游恢复就 HALF_OPEN，雪上加霜
3. **HALF_OPEN 探测数过多**：探测变成"二次过载"
4. **慢调用阈值设得过低**：正常波动被当成故障

### 5.2 降级的常见错误

1. **过度自动化**：所有降级都走自动，缺少人工开关兜底
2. **降级链路无回滚**：开了降级忘了关，常态能力一直降级
3. **兜底数据脏**：降级返回的默认值污染业务数据（如默认库存 0 导致无法下单）
4. **熔断 + 重试冲突**：熔断 OPEN 时重试还在打下游，放大流量

### 5.3 多层防护的协同

```
请求 → WAF/CDN（L5入口裁剪）
       → 网关限流（Sentinel 集群流控）
         → 应用熔断（Resilience4j / Sentinel）
           → 弱依赖降级（L2）
             → 强依赖异步化（L3）
               → DB 读写降级（L4）
                 → 兜底返回（L1）
```

每一层都该是"最后一道"，**不能依赖单层防护**。

## 六、能力差距与补足方向（Day02 视角）

### 6.1 与架构师水平的差距

1. **状态机设计细节**：HALF_OPEN 探测策略（固定 N 个 vs 滑动放行）的取舍
2. **指标体系**：慢调用比例 vs 异常比例，谁是主指标？业务不同答案不同
3. **多层级协同**：限流、熔断、降级、重试之间的"协同 + 防冲突"设计
4. **可观测性**：熔断器状态变化如何上报、告警如何去噪
5. **预案体系**：降级预案的演练频率、回滚 SOP

### 6.2 补足方向

- 阅读 Resilience4j 与 Sentinel 源码，对比滑窗实现差异
- 整理一份"降级预案 SOP"：每个降级开关的触发条件、影响范围、回滚步骤
- 在秒杀系统中实践"预降级 + 自动熔断"组合方案
- 学习混沌工程（Chaos Engineering）方法，做主动故障注入演练

## 七、面试速记口诀

- **熔断三态**：CLOSED 放行、OPEN 快速失败、HALF_OPEN 探测
- **熔断指标**：慢调用比例 + 异常比例 + 最小样本数
- **三大框架**：Hystrix 老去、4j 函数式、Sentinel 一体化
- **降级五层**：L1 兜底 / L2 弱依赖 / L3 异步化 / L4 读写 / L5 入口
- **强依赖不降级**：只熔断 + 多副本 + 限流前置
- **大促三预**：预热 + 预扩容 + 预降级
