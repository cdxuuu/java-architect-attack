## 作答区

### 题目一作答

#### A. 容量规划

##### 1. 网关层容量

```
业务目标：50000 QPS
Spring Cloud Gateway 单节点实测：10000-20000 QPS（取决于过滤器复杂度）
冗余系数：1.5（应对单节点故障、GC 抖动）
所需节点数 = 50000 / 15000 × 1.5 = 5

实际部署：2 台 8C16G 节点（按用户给定的部署规模）
单节点 QPS = 50000 / 2 = 25000，超出单节点能力
→ 必须扩容到至少 4 台 8C16G 网关节点
→ 或者：网关前置 Nginx 限流，把入口 QPS 限制到 20000，超出直接 429
```

**结论**：用户给定的 2 台网关节点不够，要么扩容到 4 台，要么前置 Nginx 限流挡量。

##### 2. 应用层容量

```
业务目标：50000 QPS（网关过滤后约 30000 QPS 到应用层）
单应用节点 4C8G：基于 Little's Law
  - 单节点 QPS = 30000 / 8 = 3750
  - 正常 RT = 100ms
  - 并发 = 3750 × 0.1 = 375
  - Tomcat maxThreads=200, acceptCount=100, 总容量 300
  - 375 > 300 → 单节点过载
  - 解决：扩容到 12 台应用节点，单节点 QPS = 2500，并发 = 250 ≤ 300 ✓

冗余设计：1.5 倍 → 12 × 1.5 = 18 台
用户给定 8 台 → 容量不足，需扩容或降级处理
```

**结论**：用户给定的 8 台应用节点不够。建议方案：
- 方案1：扩容到 16 台（大促期间弹性扩容）
- 方案2：网关层更激进修数（限流到 20000 QPS），8 台应用每台 2500 QPS 可承载
- 方案3：业务降级（关闭推荐、评论等弱依赖，降低单请求 RT 到 60ms）

##### 3. 库存服务容量

```
1000 件库存，50000 QPS 抢购，理论上 1000 / 50000 = 20ms 即售罄
但实际：
  - 防超卖：每次扣减前先检查库存（GET → DECR 原子操作）
  - Redis 单分片 QPS 上限：100000+（纯内存操作）
  - Redis 集群 3 主 3 从：300000+ QPS 承载
  - 库存扣减 QPS = 50000（每个抢购请求都扣减）
  - 50000 < 300000 → Redis 集群容量充足

但实际瓶颈在"库存预扣"逻辑：
  - 50000 抢购，1000 件库存
  - 第 1001 个请求开始，库存为 0，返回"已售罄"
  - 库存检查 QPS = 50000，但实际扣减 QPS = 1000
  - 设计：库存预扣 + 售罄标志位
    - 抢购前先 GET 售罄标志（O(1)，本地缓存）
    - 售罄后直接拒绝，不打 Redis
```

##### 4. 数据库容量

```
订单写入：1000 条（实际成交）
抢单请求：50000 次
库存预扣 → 异步落库模式：
  - Redis 扣减成功 → 发送 MQ 消息
  - 消费者异步消费 → 写入 MySQL
  - MySQL 写入 QPS = 1000 / 10min = 1.67 QPS（轻松承载）
  - 无需考虑 MySQL 容量瓶颈

如果不做异步落库：
  - 50000 抢购请求同步写 MySQL
  - MySQL 单机写入 QPS 上限约 5000-10000（取决于硬件）
  - 必然打爆 → 行锁等待 → 雪崩
```

##### 5. 冗余设计

```
按"任何一层都可能挂"假设，每层 1.5 倍冗余：
- 网关层：4 台（2 × 2 互备）
- 应用层：16 台（8 × 2 互备，单机房 8 台）
- Redis：3 主 3 从 + 1 主 1 从哨兵（异地容灾）
- MySQL：1 主 2 从 + 1 主 1 从异地
- MQ：3 节点集群 + 异地备份

容量公式：实际容量 = 单节点容量 × 节点数 / 1.5
即"设计容量是峰值的 1.5 倍"
```

#### B. 四层防线全链路配置

| 层级 | 组件 | 维度 | 阈值 | 兜底动作 | 数值依据 |
| --- | --- | --- | --- | --- | --- |
| 网关层 | Nginx + SCG | IP | 单 IP 10 QPS | 429 | 防刷，正常人 < 10 QPS |
| 网关层 | Nginx + SCG | 接口 | 单接口 50000 QPS | 429 | 业务峰值 |
| 应用接口层 | Sentinel FlowRule | 秒杀接口 | 单机 3000 QPS | fallback：返回"活动太火爆" | 16 台 × 3000 = 48000 ≈ 50000 |
| 应用接口层 | Sentinel FlowRule | 单 SKU | 单 SKU 5000 QPS | fallback：返回简化版 | 防热点打爆 |
| 集群限流层 | Sentinel Token Server | 全局 | 50000 QPS | fallback | 与业务峰值对齐 |
| 自适应限流层 | Sentinel SystemRule | 5 维 | Load>8/CPU>80%/RT>200ms/并发>300/QPS>3000 | 429 + 5s 重试 | Day04 推导 |
| 舱壁隔离层 | 线程池 | 风控 | core=10, max=20, queue=10, AbortPolicy | 走人工复核 | 第三方不可控，必须隔离 |
| 舱壁隔离层 | 信号量 | 库存 Redis | permits=500 | 快速失败 | RT 低，高 QPS |
| 舱壁隔离层 | 线程池 | 优惠券 | core=20, max=50, queue=100, AbortPolicy | 返回默认优惠券 | RT 中等 |
| 熔断降级层 | DegradeRule | 风控 | 慢调用 RT>1s 占比>50% 持续 10s | 走人工复核 | Day02 推导 |
| 熔断降级层 | DegradeRule | 优惠券 | 异常率 > 50% 持续 10s | 返回默认券 | Day02 推导 |

##### 三层阈值协同

```
网关层：50000 QPS（业务峰值，挡超量流量）
   ↓
集群限流：50000 QPS（全局总量，与网关对齐）
   ↓
应用接口限流：单机 3000 QPS × 16 台 = 48000 QPS（略低于集群限流）
   ↓
自适应限流：单机 QPS 3000 + 并发 300 + RT 200ms（系统保护）

协同原则：
- 网关 > 集群限流 > 应用接口限流 > 自适应限流
- 上层挡量，下层保命
- 自适应限流是"兜底"——前三层都过了但系统还是不行，最后由自适应限流兜底
```

##### 接口策略差异

| 接口 | 限流策略 | 原因 |
| --- | --- | --- |
| 秒杀接口（抢购） | 严格限流，单机 3000 QPS | 强一致，超量会打爆库存 |
| 商品详情页查询 | 宽松限流，单机 5000 QPS + 缓存兜底 | 弱一致，缓存兜底即可 |
| 库存扣减接口 | 集群限流 + 信号量隔离 | 强一致，必须串行化 |
| 下单接口 | 异步化 + MQ 削峰 | 最终一致，用 MQ 解耦 |

##### 演练方案表

| 故障类型 | 注入方式 | 预期系统反应 | 验证指标 | 失败标准 |
| --- | --- | --- | --- | --- |

| 下游超时 | Chaos Mesh 模拟服务 down | 客户端超时 + 熔断 OPEN | 5xx < 1%、熔断 10s 内 OPEN | 5xx > 5% 或熔断未触发 |

#### D. 应急预案 SOP

##### 1. 限流阈值过严

```
发现方式：拒绝率 > 30% 告警（P1），但 CPU/RT 正常
处置：
  1. 确认是否误判（看业务 QPS 是否真超容量）
  2. 通过 Nacos 推送新规则，调高接口限流阈值 20%
  3. 灰度发布 1 台 → 全量
  4. 观察 5 分钟，确认无雪崩
回滚：Nacos 一键回滚到上一版本
通信：通知业务方"已扩容限流配额，预计 X 分钟恢复"
```

##### 2. 限流阈值过松

```
发现方式：自适应限流持续触发告警（P0）
处置：
  1. 立即通过 Nacos 推送更严格的接口限流（降低 30%）
  2. 启动弹性扩容（增加 4 台应用节点）
  3. 降级非核心功能（推荐、评论）
  4. 网关层加严 IP 限流
回滚：扩容完成后逐步恢复原阈值
通信：P0 通知，5 分钟一次状态更新
```

##### 3. 下游熔断

```
发现方式：熔断 OPEN 告警（P1）
处置：
  1. 确认下游是否真异常（联系下游 owner）
  2. 确认 fallback 是否生效（看 5xx 是否在可控范围）
  3. 如下游持续异常，主动延长熔断冷却期
  4. 如下游恢复，HALF_OPEN 探测后自动 CLOSED
回滚：自动恢复
通信：通知下游 owner，记录工单
```

##### 4. MySQL 主库宕机

```
发现方式：MySQL 主库不可用告警（P0）
处置：
  1. 自动切换到从库（MHA / Orchestrator）
  2. 切换期间（10-30s）所有写请求拒绝
  3. 应用层降级：写请求返回"系统繁忙"，读请求走从库
  4. 确认主库恢复后回切
回滚：主库恢复后回切，需双写一段时间保证一致性
通信：P0 通知，DBA 值班
```

##### 5. Redis 集群宕机

```
发现方式：Redis 不可用告警（P0）
处置：
  1. 库存扣减切换到本地缓存兜底（保守值，可能少量超卖）
  2. 熔断 Redis 调用，所有依赖 Redis 的接口走 fallback
  3. 紧急扩容 Redis 集群（备用集群）
  4. 恢复后核对库存差异，人工对账
回滚：Redis 恢复后核对数据一致性
通信：P0 通知，可能涉及资金风险
```

##### 6. 网关节点宕机

```
发现方式：网关 5xx 告警（P1）
处置：
  1. Nginx 健康检查自动摘除故障节点
  2. 流量自动转移到其他网关节点
  3. 排查故障原因（OOM / 进程崩溃 / 网络）
  4. 重启或扩容网关节点
回滚：自动恢复
通信：P1 通知
```

##### 7. 整体雪崩

```
发现方式：全链路 5xx > 50%（P0）
处置：
  1. 立即"熔断全站"——网关层拒绝所有入口流量（fail-close）
  2. 启动应急预案：核心链路降级（仅保留支付回调）
  3. 逐层恢复：先 Redis → 再 MySQL → 再应用 → 再网关放流量
  4. 每 5 分钟状态同步到应急指挥群
回滚：逐层恢复，每层验证通过再放开下一层
通信：P0 通知，应急指挥群 1 分钟一次状态更新
```

#### E. 真实故障复盘：平安健康险商城 - JVM Full GC 引发的雪崩

**故障现象**：2024 年某次健康险产品秒杀活动，预估 QPS 2000，实际峰值 QPS 2500。投保接口配置固定阈值限流 3000 QPS，未配自适应限流。活动开始后 5 分钟，全链路雪崩，持续 8 分钟，影响订单 3000+。

**根因分析（5why）**：

```
Why1：为什么雪崩？→ Tomcat 线程池打满，下游订单服务连接池耗尽
Why2：为什么 Tomcat 线程池打满？→ JVM Full GC 暂停 1.2s，请求堆积，GC 后突刺流量涌入
Why3：为什么 Full GC 频繁？→ 老年代占用率快速上升，CMS 不适合大堆
Why4：为什么老年代快速填满？→ 投保请求创建大量临时对象（保单 VO、风控上下文）
Why5：为什么限流未触发？→ 固定阈值限流看 QPS，QPS 始终未超 3000，但并发已爆
```

**当时是否有四层防线**：

| 层级 | 是否有 | 是否生效 | 原因 |
| --- | --- | --- | --- |
| 网关层 | 有（Nginx IP 限流） | 部分生效 | 挡了刷子，但正常流量未挡 |
| 应用接口限流 | 有（Sentinel FlowRule 3000 QPS） | 未生效 | QPS 始终未超阈值 |
| 集群限流 | 无 | N/A | 未部署 Token Server |
| 自适应限流 | 无 | N/A | 未配置 SystemRule |
| 舱壁隔离 | 部分（风控有隔离） | 部分生效 | 风控隔离了，但订单服务未隔离 |
| 熔断降级 | 部分 | 未生效 | 熔断规则未配慢调用比例 |

**改造方案（基于 Day01~Day04 知识重新设计）**：

1. **新增自适应限流**（Day04）：SystemRule 配置 highestThreadCount=300, avgRt=200ms
2. **新增集群限流**（Day03）：Token Server 全局 20000 QPS，避免单机 QPS 不均
3. **补全熔断降级**（Day02）：风控/订单服务都配慢调用熔断（RT > 1s 占比 > 50% 持续 10s）
4. **舱壁隔离全覆盖**（Day03）：所有下游依赖都配独立线程池，统一 AbortPolicy
5. **JVM 调优**：CMS → G1，堆 4G → 6G，MaxGCPauseMillis=200ms
6. **acceptCount 调小**（Day04）：100 → 50，避免 GC 后突刺
7. **网关健康检查**（Day04）：GC 期间主动摘除节点
8. **故障演练**（Day05）：每月一次 Chaos Mesh 故障注入，验证四层防线

**改造后效果**：

```text
压测场景：模拟风控 RT 退化 + JVM Full GC

改造前：
  - 投保接口 P99 RT：30s+
  - 应用 Tomcat 线程池：100% 占用
  - 整体可用性：5xx 80%
  - 影响范围：全交易链路不可用

改造后：
  - 自适应限流 3s 内触发
  - 熔断 10s 内 OPEN
  - 投保接口 P99 RT：< 500ms（直接走 fallback）
  - 应用 Tomcat 线程池：< 30% 占用
  - 整体可用性：99.9%（仅风控降级订单走人工复核）
  - 影响范围：仅风控相关订单，核心投保链路无感知
```

**架构师视角反思**：
- **固定阈值限流是"虚假的安全感"**——QPS 没超阈值不代表系统健康，真正的负载信号是并发和 RT
- **四层防线缺一不可**——只配接口限流配不上自适应限流，等于"前门上锁后门敞开"
- **JVM 调优与限流是"双胞胎"**——GC 抖动会引发雪崩，限流挡不住 GC 期间的请求堆积，必须配合 acceptCount 调小 + 网关健康检查
- **故障演练是唯一验证手段**——配了不练等于没配，演练才能发现"防线协同失效"的盲区
- **核心教训**：高可用不是"配组件"，是"全链路协同设计 + 持续演练验证"

### 题目二作答

#### A. 限流降级熔断的本质

| 机制 | 本质（一句话） | 保护谁 | 牺牲什么 |
| --- | --- | --- | --- |
| 限流 | 主动控制进入系统的流量速率，超量拒绝 | 保护系统不被流量压垮 | 牺牲部分请求（拒绝） |
| 降级 | 弱依赖失败时返回兜底数据，保证主链路可用 | 保护主链路用户体验 | 牺牲完整功能（返回降级数据） |
| 熔断 | 下游异常时快速失败，避免拖垮主链路 | 保护自己不被下游拖垮 | 牺牲短期可用性（熔断期不可用） |
| 隔离 | 把下游依赖限制在独立资源池，故障不扩散 | 保护主链路和其他依赖 | 牺牲资源利用率（独立池有开销） |

**协同关系**：限流是"入口挡量"，隔离是"内部切传播路径"，熔断是"快速失败止损"，降级是"用户体验兜底"。四者形成"挡 → 切 → 断 → 兜"的完整防线。

#### B. 容量规划与限流配置的关系

**先有鸡还是先有蛋**：相互迭代，非单向因果
- 容量规划定下"系统能扛多少"，限流配置定下"放多少进来"
- 压测出容量 → 限流阈值 = 容量 × 安全系数
- 限流阈值反向要求扩容——业务量增长超阈值 → 扩容 → 重新压测 → 调高阈值

**三者演进逻辑**：

```
固定阈值限流（Day01）
  痛点：压测失真、容量动态退化
  ↓
自适应限流（Day04）
  痛点：参数调优难、阈值需要持续优化
  ↓
动态规则推送（Day04）
  痛点：规则变更需灰度、回滚
  ↓
全链路自动化（未来）
  目标：压测 → 自动配置规则 → 故障演练 → 自动调优
```

#### C. 故障演练的地位

**为什么"配了不演练"等于没配**：
- 限流规则可能有配置错误（阈值写错、resource 名写错）
- 多个规则之间可能有冲突（接口限流与自适应限流相互干扰）
- fallback 方法可能有 bug（编译通过但运行时 NPE）
- 监控告警可能采集不到关键指标
- 只有故障演练才能暴露这些"配置正确但运行错误"的问题

**演练频率与覆盖度**：
- 频率：核心系统每月一次，非核心每季度一次
- 覆盖度：至少覆盖 8 类典型故障（慢调用、超时、异常、宕机、GC、网络、Redis、MySQL）
- 强度：从单故障到组合故障（如"下游慢 + GC + Redis 抖动"三重故障）

**Chaos Mesh 与传统压测差异**：
- 压测：验证"正常流量下系统能扛多少"
- Chaos Mesh：验证"故障流量下系统是否存活"
- 互补：压测定容量，Chaos Mesh 定韧性

**故障演练文化的核心**（Google SRE 理念）：
- "故障是常态，不是异常"——系统一定会挂，问题是什么时候挂、挂了之后怎么办
- "主动制造故障"优于"被动等待故障"——主动故障可控，被动故障不可控
- "从故障中学习"——每次故障都是改进机会，不追责
- "演练文化 > 工具"——工具可以买，文化必须建

#### D. 高可用架构的元原则

1. **分层防御原则**：单一防线一定会失效，必须多层防线协同（网关 → 接口 → 集群 → 自适应 → 舱壁 → 熔断 → 降级）
2. **Fail-Fast 原则**：快速失败优于慢速等待——超时即返回，避免线程被拖垮
3. **优雅降级原则**：任何依赖都可能挂，挂了之后必须有兜底数据，不能让用户看到 5xx
4. **冗余设计原则**：任何单点都是风险，必须 N+1 冗余（多机房、多节点、多副本）
5. **反馈闭环原则**：限流配置不是"配好就完事"，必须监控 → 告警 → 调优 → 验证闭环
6. **演练验证原则**：配了不练等于没配，故障演练是唯一验证手段
7. **业务语义优先原则**：技术方案服务于业务语义（秒杀 fail-close、支付 fail-open），不是反过来

#### E. 与架构师水平的差距
