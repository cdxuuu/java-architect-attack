## 背景

Day01 讲了单机限流算法（令牌桶 / 漏桶 / 滑动窗口），本质都是**固定阈值**——QPS ≤ 1000 就放行，超过就拒绝。Day02 讲了熔断与降级——**异常反馈触发**，慢调用比例超阈值就熔断。Day03 讲了集群限流与舱壁隔离——**全局视角**与**资源隔离**。

但生产环境你会遇到三个新问题：

1. **压测容量 ≠ 真实容量**——压测时 QPS=3000 系统不挂，大促当天 QPS=2500 反而挂了。原因：压测环境下游 RT=50ms，生产环境下游 RT=300ms（共享数据库被打），同样的 QPS 在生产是 6 倍并发，固定阈值挡不住。
2. **系统容量随负载动态变化**——JVM GC 抖动、JIT 反优化、热点接口慢调用扩散、依赖方限流升级，都会让"昨天还能扛 2000 QPS"今天只能扛 1200 QPS。固定阈值对这种"动态退化"完全无感。
3. **固定阈值反应滞后**——固定阈值限流要等"流量超阈值"才动作，但系统可能"流量没超阈值、并发线程数已经爆了"（慢调用堆积）。等你拒绝时，Tomcat 线程池已经打满，下游连接池已经耗尽，雪崩已经开始。

自适应限流的本质：**不再用固定阈值，而是基于系统实时反馈信号（CPU、Load、RT、并发线程数）动态调整限流强度**——系统负载高就多拒绝、负载低就多放行，目标是"在保护系统不挂的前提下最大化吞吐"。

本日聚焦：
- 自适应限流的反馈信号体系（系统层 / 应用层 / 依赖层）
- 三大主流算法对比：Sentinel SystemRule / Netflix concurrency-limits / BBR-style 梯度限流
- 自适应限流与传统限流、熔断、隔离的四件套协同
- 监控告警闭环：从指标采集 → 告警规则 → 动态规则推送 → 故障自愈

## 题目一（原理题）：自适应限流的核心思想——从"固定阈值"到"系统反馈"

### 1.1 固定阈值限流的三重局限

### 作答区

#### 局限一：压测容量失真（Little's Law）

**核心公式**：并发数 = QPS × RT

压测环境与生产环境的差异，本质是 RT 不同导致并发数不同：

```
压测环境：QPS=3000, RT=50ms  → 并发 = 3000 × 0.05 = 150
生产环境：QPS=3000, RT=300ms → 并发 = 3000 × 0.3  = 900
                                            ↑ 6 倍
```

并发差 6 倍，但 Tomcat 容量是固定的（maxThreads=200 + acceptCount=100 = 300）：

```
压测环境：并发 150 ≤ 200 → 全部直接处理，RT 稳定，看似能扛 3000 QPS
生产环境：并发 900 > 300  → 600 个请求排队或被拒
                          → 排队导致 RT 进一步上升（队列等待时间）
                          → 恶性循环，1500 QPS 就崩
```

**根因**：压测环境下游 RT 假（mock 或独立部署），生产环境下游 RT 真（共享 DB、共享 Redis、被其他业务挤压）。压测出来的"3000 QPS 容量"本质是"压测环境并发 150 的容量"，迁移到生产毫无意义。

#### 局限二：容量动态退化无感

**场景1：JVM Full GC 暂停 800ms**
- GC 期间所有应用线程暂停，请求堆积在 acceptQueue
- GC 期间 QPS 信号没变化（请求还在队列里没被处理）
- GC 结束瞬间，堆积请求 + 正常请求同时涌入，瞬时并发是平时的 3 倍
- 固定阈值限流看 QPS，没超阈值 → 不触发
- 但实际并发已爆 → 触发二次 Full GC → 循环抖动

**场景2：下游 RT 退化（50ms → 500ms）**
- QPS 没变，但每个请求占用线程时间 10 倍
- 并发 = QPS × RT → 并发 10 倍
- 固定 QPS 限流无感，直到线程池打满才被动拒绝
- 此时雪崩已发生 30 秒

**场景3：热点 Key 缓存穿透**
- 单 SKU 缓存失效，所有请求打到 DB
- 该 SKU 查询 RT 从 10ms 飙到 1000ms
- 接口总 QPS 没超阈值（其他 SKU 还正常）
- 但 DB 已经被打爆——固定阈值限流盲区

#### 局限三：反应滞后（QPS 没超阈值但系统已崩）

```
接口配置：QPS ≤ 2500 限流
实际场景：
  下游风控服务 RT 从 50ms → 800ms
  QPS=2000（没超 2500 阈值）→ 限流器放行
  但并发 = 2000 × 0.8 = 1600
  Tomcat maxThreads=200 → 线程池瞬间打满
  后续请求全部 reject，但限流器还在"放行"
  雪崩已发生 30 秒，限流器仍未触发
```

**根本盲区**：固定阈值限流是**流量视角**（看 QPS），不是**负载视角**（看并发/CPU/RT）。系统挂不挂取决于负载，不是流量。

### 1.2 自适应限流的反馈信号体系

#### 三层信号体系对照表

| 层级 | 反馈信号 | 采集方式 | 典型阈值 | 失效场景 |
| --- | --- | --- | --- | --- |
| 系统层 | CPU 使用率、Load1、磁盘 IO、网络 IO | /proc/stat、JMX、node_exporter | CPU > 80%、Load1 > 核数 × 2 | 采样粒度太粗（1s），瞬时尖峰被平滑 |
| 应用层 | 入口 QPS、并发线程数、RT P99/P999、JVM 堆 | Sentinel slot、Micrometer | RT P99 > 500ms、并发 > 200 | 慢调用未抛异常，统计不到 |
| 依赖层 | 下游 RT、连接池占用率、拒绝次数 | APM、连接池 metrics | 连接池占用 > 80%、下游 RT > 1s | 异步调用未采集、跨线程上下文丢失 |

#### 为什么 Sentinel SystemRule 同时采集 5 个信号

**单信号盲区举例**：

| 场景 | CPU | Load | RT | 并发 | QPS | 单信号判定 |
| --- | --- | --- | --- | --- | --- | --- |
| 计算密集型（加密/压缩） | 95% | 1.0 | 50ms | 50 | 1000 | Load 正常，但 CPU 爆 |
| IO 等待型（下游慢） | 20% | 8.0 | 800ms | 200 | 250 | CPU 空闲，但 Load/RT 爆 |
| 突发流量（缓存击穿） | 60% | 2.0 | 100ms | 500 | 5000 | RT 正常，但并发/QPS 爆 |
| GC 抖动后突刺 | 90% | 4.0 | 1.2s | 400 | 2500 | QPS 正常，但 RT/并发爆 |

任一单信号都会漏报。5 信号交叉验证，任一超阈值即触发，覆盖所有典型故障形态。

#### CPU 用 1 分钟滑动平均 vs 瞬时值

**瞬时 CPU 飙高的常见场景**（不应限流）：
- JIT 编译：1-2s，编译完就降
- GC 标记：百毫秒级
- Linux 中断处理：毫秒级

**滑动平均的副作用**：反应慢，可能晚 10-30s 才触发。

**折中方案**：CPU 滑动平均（过滤瞬时尖峰）+ 并发线程数瞬时值（快速反应）双重判定。

#### RT P99 vs P999 vs 平均 RT

| 指标 | 优缺点 | 适用场景 |
| --- | --- | --- |
| 平均 RT | 被长尾拉高（1 个 10s 请求把 1000 个 50ms 平均成 60ms），不灵敏 | 不推荐 |
| P99 | 反映"绝大多数用户体验"，灵敏但不抖动 | 自适应限流首选 |
| P999 | 太敏感，正常 GC 抖动也触发 | 资金类强一致场景 |
| P50 | 完全反映不出长尾问题 | 不推荐 |

**结论**：自适应限流用 P99 较多，配合并发数双重判定。Sentinel SystemRule 出于实现简单用了 avgRt，更精细场景用 concurrency-limits 的 P99 信号。

### 1.3 三大主流自适应限流算法对比

#### A. Sentinel SystemRule（规则触发式，多维度阈值）

**5 个阈值是否独立**：不完全独立，常见"部分超阈值部分没超"的场景：

- **Load 没超但 CPU 超了**：计算密集型业务（加密/压缩/序列化），CPU 满载但 Load 不高（Load 主要反映等待 CPU 的进程数，纯计算不阻塞）
- **CPU 没超但 RT 超了**：IO 等待型业务（下游慢调用），CPU 空闲但线程池阻塞
- **RT 没超但并发超了**：突发流量瞬间堆积，RT 还没统计出来并发已经爆

**Sentinel SystemRule 默认行为**：
- 任一指标超阈值即触发，**限流所有入口接口**（entryType=IN）
- 粗暴但有效——系统级保护就是"先保命再保业务"
- 配置白名单可豁免核心接口（如支付回调标记为 OUT 类型不进系统流量统计）

**5 个阈值如何定**：基线法 + 容量法组合
- **基线法**：观察生产环境 1 周正常水位，取 P99 × 1.5 作为阈值
- **容量法**：基于硬件规格 + Little's Law 推导（详见题目二）
- 不能纯靠压测，压测环境 RT 失真

#### B. Netflix concurrency-limits（基于 Little's Law 动态调整并发上限）

**为什么限并发不限 QPS**：

| 限流维度 | 优势 | 劣势 |
| --- | --- | --- |
| 限 QPS | 直观，业务好理解 | 与 RT 耦合，RT 变化时阈值需手动调 |
| 限并发 | 直接对应"系统能同时处理多少请求"，与 RT 解耦 | 不直观，业务方难理解 |

**本质差异**：QPS 是"结果"（系统挂没挂看并发），并发是"原因"。限 QPS 需要预测 RT，而 RT 是动态的。限并发直接对应当前系统容量，RT 变化时并发上限自动跟随调整。

**AIMD 算法详解**：

```
加性增长（Additive Increase）：
  每次成功后并发上限 +1（或 +N）
  - N 太大：探测期间过载
  - N 太小：容量恢复慢
  - Netflix 默认 N=1，保守探测

乘性回退（Multiplicative Decrease）：
  超阈值或拒绝后并发上限 × 0.5
  - 不降到 1：避免"全停"和长恢复时间
  - 0.5 是经验值：保留一半容量，快速探测恢复
  - 类比 TCP 拥塞控制的"快重传快恢复"
```

**与 TCP 拥塞控制的关系**：concurrency-limits 直接借鉴 TCP Vegas / BBR，把"网络带宽"换成"系统并发容量"，把"RTT"换成"应用 RT"，把"丢包"换成"请求拒绝"。

**Gradient2Limit 避免两个极端**：

- **冷启动限流过严**：初始并发上限设低（如 10），逐步加性增长。如果初始就设高，冷启动期可能直接过载
- **长期运行限流过松**：基于"实际 RT / 最小 RT"梯度计算——RT 退化时立即降低并发上限，不等拒绝才降。最小 RT 持续更新，避免"历史最大值"长期生效
- **梯度公式**：`limit = current_limit × (minRT / currentRT)`——RT 翻倍时并发上限减半

#### C. BBR-style 梯度限流（探测最大吞吐量）

**BBR vs AIMD 本质差异**：

| 维度 | AIMD（concurrency-limits） | BBR |
| --- | --- | --- |
| 触发方式 | 被动回退——等系统过载（拒绝/超时）才回退 | 主动探测——持续测量找最优工作点 |
| 工作点 | 振荡在"过载-回退"之间 | 稳定在 BDP = maxQPS × minRT |
| 信号 | 拒绝/超时（事后） | QPS + RT（事前） |
| 类比 | "撞墙了退一步" | "持续测速维持最优" |

**BBR 探测窗口设计**：
- 周期性放行"探测流量"——每 10s 增加 10% 并发，测量这期间的 QPS 和 RT
- 探测期间可能短暂过载，控制代价：
  - 探测幅度小（10% 而非 100%）
  - 探测时间短（1s 而非 1min）
  - 探测失败立即回退（RT 退化 > 20% 立即降回）

**BBR 在应用限流普及度低的原因**：
- TCP 场景 RT 稳定（毫秒级），探测代价低；应用场景 RT 抖动大（GC、慢调用、热点），探测信号噪声大
- BBR 需要持续测量"最小 RT"，但应用最小 RT 受 GC 影响不稳定
- 算法复杂、调参难、故障场景不直观
- Sentinel 2.0+ 引入了 BBR 思想的 SystemSlot，但默认还是规则触发式，BBR 是可选实验特性

### 1.4 自适应限流与传统限流、熔断、隔离的协同

#### 四层限流协同图

```
请求进入
   ↓
┌──────────────────────────────────────────┐
│ 第一层：网关层固定阈值限流                │
│  - Nginx limit_req / Spring Cloud Gateway│
│  - 维度：IP、接口、User ID                │
│  - 作用：挡刷子、挡超量流量              │
│  - 触发条件：QPS > 阈值                   │
│  - 动作：429 Too Many Requests            │
└──────────────────────────────────────────┘
   ↓ 通过
┌──────────────────────────────────────────┐
│ 第二层：应用层接口限流                    │
│  - Sentinel FlowRule                     │
│  - 维度：接口、热点参数                   │
│  - 作用：保护配额、保护下游               │
│  - 触发条件：接口 QPS > 阈值              │
│  - 动作：fallback 兜底                    │
└──────────────────────────────────────────┘
   ↓ 通过
┌──────────────────────────────────────────┐
│ 第三层：应用层自适应限流                  │
│  - Sentinel SystemRule / concurrency-limits│
│  - 维度：系统全局（CPU、Load、RT、并发）  │
│  - 作用：保护系统不挂                     │
│  - 触发条件：任一系统指标超阈值           │
│  - 动作：拒绝入口请求（除白名单）         │
└──────────────────────────────────────────┘
   ↓ 通过
┌──────────────────────────────────────────┐
│ 第四层：舱壁隔离 + 熔断降级               │
│  - 线程池/信号量隔离 + CircuitBreaker     │
│  - 维度：单个下游依赖                     │
│  - 作用：保护下游、隔离故障               │
│  - 触发条件：线程池满 / 熔断器 OPEN       │
│  - 动作：fallback 兜底                    │
└──────────────────────────────────────────┘
   ↓ 通过
正常处理
```

#### 触发顺序与白名单

- Sentinel 默认执行顺序：FlowSlot（接口限流）→ SystemSlot（系统限流）→ 依赖隔离与熔断
- 自适应限流触发时，拒绝"所有非白名单入口接口"
- 白名单配置：核心接口（支付回调、对账回调）标记为 `entryType=OUT` 不计入系统流量
- 秒杀接口不进白名单——它是系统压力的主要来源，必须受系统限流约束

#### 自适应限流 vs 熔断

| 维度 | 自适应限流 | 熔断 |
| --- | --- | --- |
| 触发对象 | 自己不行（系统资源紧张） | 下游不行（依赖服务故障） |
| 触发信号 | CPU/Load/RT/并发 | 异常率/慢调用比例 |
| 动作 | 拒绝入口请求 | 快速失败 + fallback |
| 保护谁 | 保护自己 | 保护下游、保护自己不被下游拖垮 |

两者可同时触发：自己 CPU 高 + 下游慢 → 自适应限流 + 熔断同时动作。

#### 自适应限流触发后的兜底

- **HTTP 状态码**：429 Too Many Requests（不是 5xx，5xx 表示服务端错误，429 表示客户端请求过多）
- **业务码**：`{code: 429, msg: "系统繁忙，请稍后重试", retryAfter: 5}`
- **前端处理**：弹窗"系统繁忙" + 5s 后自动重试（指数退避，避免重试风暴）
- **关键接口（支付）**：返回"排队中"页面 + 异步轮询结果，不直接拒绝

## 题目二（实战题）：秒杀系统的自适应保护与监控告警体系

### 作答区

#### A. Sentinel SystemRule 实战配置

秒杀系统 4C8G 单机配置：

| 阈值 | 数值 | 依据 |
| --- | --- | --- |
| `highestSystemLoad` | 8 | 4 核 × 2 = 8，Linux Load1 超过核数 × 2 即过载 |
| `highestCpuUsage` | 80% | 4 核总 100%，80% 即 3.2 核满载，留 0.8 核兜底 |
| `highestThreadCount` | 300 | Tomcat maxThreads=200 + 接收线程 + GC 线程 + 业务线程池 ≈ 300 |
| `avgRt` | 200ms | 正常 RT 100ms × 2 = 200ms，超过即异常 |
| `qps` | 3000 | 单机峰值 2500 × 1.2 安全系数 = 3000 |

**推导过程（Little's Law）**：
```
单机峰值 QPS = 2500，正常 RT = 100ms
正常并发 = 2500 × 0.1 = 250
Tomcat maxThreads=200, acceptCount=100, 总容量 300
→ highestThreadCount=300（接近容量上限触发）
→ avgRt=200ms（正常 100ms，翻倍即异常）
→ qps=3000（峰值 2500 × 1.2）
→ CPU 80%（4C 满载 100%，留 20% 兜底）
→ Load1=8（核数 × 2，Linux 经验阈值）
```

**秒杀接口是否进 SystemRule 范围**：
- 进！秒杀接口是系统压力主要来源，必须受系统限流约束
- 配置白名单兜底——核心回调接口（支付回调、对账回调）标记为 OUT 不进系统流量
- 配置方式：`@SentinelResource(entryType = EntryType.OUT)` 标记回调接口

**触发后返回响应**：

```json
HTTP 429
{
  "code": 429,
  "msg": "系统繁忙，请稍后重试",
  "data": null,
  "retryAfter": 5
}
```

前端处理：
- 秒杀接口：弹窗"活动太火爆，请稍后再试" + 5s 倒计时
- 商品查询：返回缓存兜底数据 + "部分数据加载中"
- 下单接口：跳转"排队中"页面，5s 后异步查询结果

#### B. 四层防线协同：三种故障场景的演进

##### 场景1：风控服务 RT 退化到 800ms

```
请求进入（QPS=2000，没超 2500 阈值）
   ↓
第一层网关：QPS 没超 → 放行
   ↓
第二层接口限流：QPS 没超 2500 → 放行
   ↓
第三层自适应限流：
  - 并发线程数 = 2000 × 0.8 = 1600 > 300 → 触发
  - avgRt = 600ms > 200ms → 触发
  - 拒绝入口请求 70%
   ↓
第四层舱壁 + 熔断：
  - 风控线程池（core=10, max=20）3s 内打满
  - 风控慢调用 RT > 1s 占比 > 50% → 熔断 OPEN
  - 后续风控调用直接快速失败 → 走人工复核 fallback
   ↓
效果：
  - 自适应限流挡住入口 70% 流量
  - 熔断让风控调用快速失败
  - 系统存活，仅风控降级订单走人工
```

**哪一层先触发**：自适应限流（并发数信号最灵敏，秒级触发）；熔断需要积累统计窗口（10s），稍慢。

##### 场景2：JVM Full GC 1.2s 期间

```
GC 期间（1.2s）：
  - 应用线程暂停，请求堆积在 acceptQueue
  - 网关、接口限流、自适应限流都"无感"（QPS 信号没变化）
  - 但实际并发已超过 highestThreadCount
   ↓
GC 结束瞬间：
  - 突刺流量涌入（堆积的 + 正常进入的）
  - 第一层网关：QPS 飙升 → 触发 IP/接口限流
  - 第三层自适应：并发线程数 > 300 → 触发
  - 第四层：下游调用堆积，触发熔断
   ↓
能否挡住"GC 后突刺"：
  - 自适应限流能挡住一部分（并发线程数信号灵敏）
  - 但挡不住"GC 期间堆积的请求"——这些已经在 acceptQueue 里
  - 解决方案：
    1. JVM 调优（G1 替换 CMS、降低 GC 时间到 200ms 内）
    2. acceptCount 调小（默认 100 → 50），避免堆积请求在 GC 后涌入
    3. 网关层配置健康检查，GC 期间主动摘除节点（如 Spring Cloud Gateway 配合 Eureka 健康检查）
    4. 终极方案：自适应限流 + 容量冗余（双活机房分摊流量）
```

##### 场景3：库存 Redis 主从切换 3s

```
切换期间（3s）：
  - 所有 Redis 调用同步等待（或超时）
  - 单请求 RT 从 20ms → 3s（等待主从切换）
  - 并发 = QPS × 3s = 7500（正常 250）→ 瞬间爆表
   ↓
舱壁隔离（信号量 permits=500）：
  - 500 个 permit 被占满，后续请求被拒
  - 但主线程被阻塞（信号量不创建线程）→ Tomcat 线程被拖
   ↓
熔断：
  - 慢调用比例 100% > 50% → 熔断 OPEN
  - 后续 Redis 调用快速失败
   ↓
自适应限流：
  - 并发线程数 > 300 → 触发
  - 但熔断已经先一步 OPEN，自适应限流是"补刀"
   ↓
最佳实践：
  - 库存 Redis 调用配置超时 200ms（不等主从切换）
  - 超时即失败，走"本地缓存兜底"或"快速失败"
  - 熔断 + 自适应限流双保险
  - Redis 集群配置 Sentinel 模式 + 集群模式，主从切换 < 1s
```

#### C. 监控指标体系设计

| 维度 | 关键指标 | 采集来源 | 告警阈值 |
| --- | --- | --- | --- |
| 流量 | 通过 QPS、拒绝 QPS、拒绝率 | Sentinel metrics | 拒绝率 > 5% 持续 1min（P1）/ > 20%（P0） |
| 时延 | RT P50/P99/P999 | Sentinel + APM | P99 > 500ms 持续 1min（P1）/ > 2s（P0） |
| 并发 | 入口并发线程数、各依赖并发 | Tomcat + 连接池 | 入口并发 > 200 持续 30s（P1）/ > 280（P0） |
| 系统 | CPU、Load1、JVM 堆、GC 耗时 | Prometheus + JMX | CPU > 80% 持续 1min / Load > 8 / GC > 1s |
| 依赖 | 下游 RT、下游拒绝数、连接池占用率 | APM + 连接池监控 | 连接池占用 > 80%（P1）/ > 95%（P0） |

##### 拒绝率告警阈值如何定

- 不能纯看百分比——大促时正常拒绝率也可能 5%（流量超容量设计）
- 不能纯看绝对值——平时 50 拒绝是大问题，大促时 5000 拒绝是正常
- 折中方案：拒绝率 + 绝对值双条件，按时段区分（大促期间阈值放宽）
- 告警规则：`(拒绝率 > 5% AND 拒绝QPS > 100) AND NOT(时段为大促窗口)`

##### RT P99 突增告警的滞后性

- Sentinel 默认窗口 1s 滚动 2 个子窗口，P99 实际是"过去 1s 的 P99"，相对实时
- 但有些监控用 1min/5min 窗口，P99 滞后 1-5min

**降低告警延迟方案**：
- 短窗口告警：1s/10s 窗口的 P99 突增 → 立即告警（预警）
- 同比环比：当前 P99 vs 5min 前 P99，环比 > 50% 即告警
- 多窗口组合：1s 窗口触发预警，1min 窗口触发正式告警
- 实时流式统计：用 HdrHistogram 流式计算 P99，延迟 < 1s

##### 告警风暴问题

SystemRule 触发时，CPU、Load、RT、并发四个指标同时告警，会产生告警风暴。

**收敛方案**：
- 同一根因的多个告警合并——SystemRule 触发时四个告警收敛为一条"系统过载"
- 收敛规则：相同应用 + 相同时段（1min）+ 关联指标 → 合并
- Alertmanager 的 `inhibit_rules`：CPU 告警抑制 Load 告警，避免重复通知
- 告警平台层面：基于拓扑的根因分析（如 A 调 B，B 宕机则 A 的告警被抑制）

#### D. 告警规则与动态规则推送

##### 告警分级

| 级别 | 标准 | 典型场景 | 响应时间 |
| --- | --- | --- | --- |
| P0 | 核心接口不可用 / 资金风险 / 数据丢失 | 秒杀接口 5xx > 50%、支付失败率 > 10%、数据库宕机 | 立即介入（5min 内） |
| P1 | 部分功能降级 / 性能严重退化 | 自适应限流持续触发 5min、某个下游熔断、P99 > 2s | 5-30 分钟内 |
| P2 | 容量预警 / 慢依赖苗头 | 拒绝率 5%、连接池占用 70%、GC 频率上升 | 日常巡检（4h 内） |

##### 告警处置流程

```
SystemRule 触发
   ↓
告警平台收到（含应用名、指标、时间、关联链路）
   ↓
自动分诊：
  - 是否大促期间？→ 是 → 通知扩容
  - 是否单机？→ 是 → 自动摘除节点（如 Nacos 反注册 + 网关摘流）
  - 是否全集群？→ 是 → 立即 P0 通知
   ↓
人工介入：
  - 限流阈值是否需调整？→ Nacos 推送新规则
  - 是否需要降级非核心功能？→ Sentinel DegradeRule
  - 是否需要扩容？→ 弹性扩容脚本
   ↓
事后复盘：
  - 5why 分析根因
  - 更新 SOP 与告警规则
  - 故障注入演练验证
```

##### 动态规则推送（Nacos）

**规则存储格式**（dataId: `sentinel-rules-seckill.json`）：

```json
[
  {
    "resource": "seckillOrder",
    "limitApp": "default",
    "grade": 1,
    "count": 2500,
    "strategy": 0,
    "controlBehavior": 0,
    "clusterMode": false
  }
]
```

**推送链路**：

```
Nacos 配置变更
   ↓
Sentinel DataSourceListener 监听变更
   ↓
加载新规则 → FlowRuleManager.loadRules()
   ↓
新规则立即生效（下一个请求按新规则判定，毫秒级）
```

**灰度发布**：
- Nacos Beta 发布——先推 1 台 IP 验证 5min
- 监控该 IP 的 RT、拒绝率、错误率
- 无异常 → 全量推送
- 异常 → 一键回滚到上一版本

**回滚机制**：
- Nacos 保留历史版本（默认 30 个）
- 一键回滚：选历史版本 → 推送 → 秒级生效
- 紧急回滚脚本：`./rollback-sentinel.sh seckill-app v1.2.3`
- **双保险**：Sentinel 客户端本地缓存上一版本规则，Nacos 不可用时降级用本地

#### E. 真实案例：平安健康险商城 - JVM Full GC 引发的雪崩

**场景**：
2024 年某次健康险产品秒杀活动，预估 QPS 2000，实际峰值 QPS 2500（超预估 25%）。投保接口配置了固定阈值限流（3000 QPS），未配置自适应限流。

**故障链路**：

```
QPS 2500（未超 3000 阈值，限流未触发）
   ↓
JVM 老年代占用率快速上升（每秒 +2%）
   ↓
触发 Full GC（CMS），暂停 1.2s
   ↓
期间 2500 × 1.2 = 3000 个请求堆积在 acceptQueue
   ↓
GC 结束，突刺流量涌入
   ↓
老年代瞬间被新对象填满（1500ms 内）
   ↓
再次 Full GC，循环抖动
   ↓
5 分钟内 12 次 Full GC，平均 RT 从 100ms 飙到 3s
   ↓
Tomcat 线程池打满，下游订单服务连接池耗尽
   ↓
全链路雪崩，持续 8 分钟，影响订单 3000+
```

**当时是否有自适应限流**：
- 无。只配置了固定阈值限流（QPS ≤ 3000）
- QPS 始终未超阈值，限流未触发
- 但并发线程数从 250 飙到 1500，系统已经崩了

**如果当时有自适应限流**：

```
SystemRule 配置：highestThreadCount=300, avgRt=200ms
   ↓
第一次 Full GC 后突刺流量涌入：
  - 并发线程数 > 300 → 自适应限流触发
  - avgRt > 200ms → 自适应限流触发
  - 拒绝入口请求 70%
   ↓
老年代不会瞬间被填满
   ↓
不会触发第二次 Full GC
   ↓
系统抖动 30s 内恢复
   ↓
影响范围：仅 30% 用户被限流，无雪崩，无订单影响
```

**架构师视角反思**：
- 固定阈值限流是"流量视角"，自适应限流是"系统视角"，两者必须配合
- JVM 调优是根本——CMS 不适合大堆（> 4G），应换 G1 或 ZGC
- acceptCount 调小（默认 100 → 50），避免堆积请求在 GC 后涌入
- 网关层配置健康检查——GC 期间主动摘除节点（如 Spring Cloud Gateway 配合 Nacos 健康检查）
- **核心教训**：限流配置不是"配好就完事"，必须基于真实故障场景持续验证。固定阈值限流让"QPS 没超"成为虚假的安全感，真正的负载信号是并发和 RT。
