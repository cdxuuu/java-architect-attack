## 背景

昨天的限流是"系统过载前的主动闸门"，今天进入高可用三件套的后两环——**熔断**与**降级**。

你的秒杀系统在大促时，下游支付接口偶发超时，开始只是几十毫秒的抖动，但很快演变为整个支付服务线程池被打满、级联雪崩。事后复盘发现：限流挡住了入口流量，但没能挡住"下游慢调用"引发的次生灾害。这正是熔断降级要解决的问题。

本日聚焦：

## 题目一（原理题）：熔断器状态机与三大实现对比

请回答以下问题：

1. 画出熔断器的状态机（CLOSED → OPEN → HALF_OPEN → CLOSED），并解释每个状态的进入条件、退出条件、状态转移的触发指标。
2. Hystrix、Resilience4j、Sentinel 三者在熔断实现上的核心差异是什么？请从以下维度对比：
   - 熔断指标统计窗口（滑动窗口 / 时间桶 / 桶+滑动）
   - 慢调用比例 vs 异常比例，谁是主指标？
   - HALF_OPEN 的恢复探测机制（请求数 / 时间 / 探针）
   - 是否支持熔断后的自定义 fallback 与 fallback 链
3. 为什么 Hystrix 已停止维护？Resilience4j 在函数式编程模型上做了哪些改进？
4. Sentinel 的"熔断"和"限流"在内部实现上是同一套滑窗数据结构吗？为什么 Sentinel 能做到"慢调用比例熔断"而 Guava RateLimiter 做不到？

### 作答区

#### 1. 熔断器状态机

```
        ┌──────────────────────────────────────────┐
        │                                          │
        ▼                                          │
  ┌──────────┐  指标超阈值   ┌────────┐ 探测成功    │
  │ CLOSED   │ ───────────► │  OPEN  │ ──────────► │
  │ 正常放行  │              │ 快速失败 │             │
  └──────────┘              └───┬────┘             │
        ▲                       │ 等待 recovery    │
        │ 探测全成功              ▼                 │
        │                  ┌────────────┐          │
        └──────────────────│ HALF_OPEN  │──────────┘
                           │ 半开探测    │
                           └─────┬──────┘
                                 │ 探测失败
                                 ▼
                              回到 OPEN（重新计时）
```

| 状态 | 进入条件 | 退出条件 | 行为 |
| --- | --- | --- | --- |
| CLOSED | 初始 / HALF_OPEN 探测全部成功 | 滑窗内慢调用比例 OR 异常比例超阈值（且样本数 ≥ minimumNumberOfCalls） | 正常放行，统计指标 |
| OPEN | CLOSED 状态指标越线 | 等待 `waitDurationInOpenState` 到期 | 快速失败，直接走 fallback，不发起下游调用 |
| HALF_OPEN | OPEN 冷却期结束 | 探测 N 次成功 → CLOSED；任一失败 → OPEN | 放行有限个探测请求，重新统计指标 |

**触发指标（三类）**：
- **慢调用比例**：`RT > slowCallDurationThreshold` 的请求占比 ≥ `slowCallRateThreshold`
- **异常比例**：业务异常 / 框架异常占比 ≥ `failureRateThreshold`
- **最小样本数**：滑窗内请求数 ≥ `minimumNumberOfCalls`，避免低流量误判

**关键点**：OPEN 状态不调用下游，是"快速失败"；HALF_OPEN 是"试探性放行"，避免直接全量恢复造成二次雪崩。

#### 2. 三大实现对比

| 维度 | Hystrix | Resilience4j | Sentinel |
| --- | --- | --- | --- |
| 编程模型 | 命令模式（继承 HystrixCommand） | 函数式装饰器链 | 注解 + API（SphU.entry / @SentinelResource） |
| 滑窗实现 | 桶式（BucketedCounterStream，基于 RxJava） | Count-based 或 Time-based 滑窗 | LeapArray（时间桶环形数组 + 样本数组） |
| 主指标 | 异常比例为主，慢调用为辅 | 慢调用比例 + 异常比例可独立配置 | 慢调用比例 + 异常比例可独立配置 |
| HALF_OPEN 探测 | 单次探测（只放行 1 个请求） | 可配置 N 个探测请求 | 探测期放行 N 个请求 |
| 自定义 fallback | 支持（fallbackMethod） | 支持（Decorator 链下游 supplier） | 支持（blockHandler / fallback） |
| fallback 链 | 弱（单一 fallback） | 强（装饰器链可组合多层兜底） | 中（可配置 primary + fallback） |
| 限流能力 | 无 | 有（独立 RateLimiter 模块） | 原生（限流+熔断+热点+系统自适应一体） |
| 控制台 | Hystrix Dashboard（简陋） | 无原生控制台 | Sentinel Dashboard（可视化规则推送） |

#### 3. Hystrix 为什么被淘汰 / Resilience4j 的改进

**Hystrix 被淘汰的原因**：
1. **命令模式侵入性强**：每个隔离点要写一个 HystrixCommand 子类，代码膨胀
2. **基于 RxJava 1.x**：依赖陈旧，响应式模型过重
3. **半开探测过于保守**：HALF_OPEN 只放行 1 个请求，恢复慢
4. **可观测性弱**：Dashboard 只看断路器状态，规则推送能力差
5. **2018 年起停止维护**：社区已迁移到 Resilience4j

**Resilience4j 的函数式改进**：
```java
Supplier<String> call = Decoration.ofSupplier(() -> callRemote())
    .decorate(CircuitBreaker.decorateSupplier(cb))
    .decorate(RateLimiter.decorateSupplier(rl))
    .decorate(Retry.decorateSupplier(retry, 3))
    .decorate(Bulkhead.decorateSupplier(bp))
    .get();
```
- **装饰器链**：熔断 → 限流 → 重试 → 舱壁，顺序可调，可组合
- **无侵入**：业务方法仍是纯函数，不用继承任何类
- **模块解耦**：CircuitBreaker / RateLimiter / Retry / Bulkhead / TimeLimiter 各自独立
- **与 Reactor 友好**：原生支持 Mono/Flux 装饰

#### 4. Sentinel 的熔断与限流是否共用滑窗

**是的，共用同一套 LeapArray 数据结构**，但读不同维度。

LeapArray 是一个**时间桶环形数组**：
- 每个 `WindowWrap<MetricBucket>` 桶保存样本：成功数、异常数、RT、通过量
- 限流读"通过量 / QPS"维度
- 熔断读"慢调用比例 / 异常比例"维度
- **数据写入时一次写入，多种统计维度共享**——这是 Sentinel 性能高的关键

**为什么 Sentinel 能做慢调用熔断而 Guava RateLimiter 做不到？**
- Sentinel 的 LeapArray 在每次请求结束后写入 RT，**天然统计 RT 分布**
- Guava RateLimiter 内部只维护"令牌上次填充时间 + 已积累令牌数"，**根本没有 RT 统计结构**
- 想用 Guava 做慢调用熔断，必须自己外挂一个 RT 统计模块，本质就是重写 Sentinel

#### 与架构师水平的差距及补足方向

**差距**：
1. 状态机参数调优经验不足——`minimumNumberOfCalls`、`waitDurationInOpenState`、HALF_OPEN 探测数这三者的协同，没系统踩过坑
2. 对 Resilience4j 装饰器链的顺序敏感性（重试在熔断外还是内）理解不深
3. Sentinel 的 LeapArray 源码没读透，"高并发下桶切换的并发安全"还是黑盒
4. fallback 链设计经验少，业务里只用了单层 fallback

**补足方向**：
- 阅读 Resilience4j `CircuitBreakerStateMachine` 与 Sentinel `LeapArray` 源码
- 在秒杀系统中实践"熔断 → 限流 → 重试"装饰器链，对比不同顺序下的恢复曲线
- 整理一份熔断参数调优 SOP（基于 QPS 量级给推荐配置）

## 题目二（实战题）：秒杀系统的降级策略分层设计

在双11秒杀场景下，请设计一套**多层级降级方案**，回答：

1. 降级按"业务影响"应分几层？请给出你的分层模型（如：L1 兜底返回 / L2 弱依赖熔断 / L3 强依赖异步化 / L4 读写降级 / L5 流量入口裁剪），每层举一个秒杀场景的具体例子。
2. 强依赖（如扣减库存）能不能降级？如果不能降级，怎么保高可用？请给出你的方案。
3. 降级的触发方式有几种？**手动开关降级** vs **自动熔断降级** vs **预案降级（如大促前预降级）**，分别在什么场景使用？为什么大促前要做"预降级"？
4. 降级后的兜底数据怎么设计？例如商品详情页降级后返回什么？返回静态缓存 vs 返回空 vs 返回"稍后重试"，怎么选？
5. 在你负责的蛋壳钱包 / 平安健康险商城 / IM 客服系统中，举一个真实降级案例：触发条件、降级动作、业务感知、回滚流程。

### 作答区

#### 1. 五层降级模型

| 层级 | 名称 | 触发条件 | 动作 | 秒杀场景例子 |
| --- | --- | --- | --- | --- |
| L1 | 兜底返回 | 单点接口故障 | 返回缓存/默认值 | 商品详情页降级为 CDN 静态页，价格展示最近一次缓存值 |
| L2 | 弱依赖熔断 | 非核心依赖慢/异常 | 熔断+返回兜底 | 推荐系统挂了，秒杀页不展示推荐位，评论数显示"--" |
| L3 | 强依赖异步化 | 核心链路压力大 | 同步转异步 | 下单成功后短信/积分改 MQ 异步发送，发货通知异步化 |
| L4 | 读写降级 | DB 压力大 | 读走缓存/写暂存MQ | 库存查询走 Redis，下单请求先写 MQ 后异步入 DB |
| L5 | 流量入口裁剪 | 全链路过载 | 入口限流/排队/置灰 | 秒杀按钮加答题验证码削峰、活动入口排队、非白名单用户置灰 |

**执行顺序**：从 L5 开始往里推——先挡流量（L5），再降读写（L4），再异步化（L3），再熔断弱依赖（L2），最后兜底（L1）。**先降影响大的，再降影响小的**。

#### 2. 强依赖能不能降级

**原则：强依赖不降级，只熔断 + 重试 + 多副本 + 限流前置**

扣减库存、扣款、订单创建这些是强依赖，不能"假装成功"——否则超卖或资损。

**保高可用方案**：
1. **本地预扣 + 异步落库**：Redis 预扣库存（Lua 原子），MQ 异步落 DB，DB 短暂不可用时 MQ 堆积兜底
2. **多副本容灾**：Redis 主从 + Cluster 分片，DB 主从 + 分库分表
3. **熔断保护**：Redis 集群熔断后切到本地缓存兜底（牺牲一致性保可用，配合对账兜底）
4. **限流前置**：调用库存服务前先做 Sentinel 限流，把超额流量挡在库存服务之外
5. **降级到"拒绝服务"而非"假装成功"**：强依赖不可用时返回"请稍后重试"，而不是返回假成功

**核心心法**：强依赖的"降级"本质是**降低一致性级别**（强一致 → 最终一致），不是"不调用"。

#### 3. 降级触发方式：手动 / 自动 / 预案

| 方式 | 触发主体 | 响应速度 | 适用场景 | 风险 |
| --- | --- | --- | --- | --- |
| **自动熔断降级** | 框架（Sentinel/Resilience4j） | 毫秒级 | 下游故障实时响应、慢调用雪崩预防 | 误判、抖动恢复慢、过度自动化失控 |
| **手动开关降级** | 运维/SRE | 分钟级 | 灰度发布、紧急止血、复杂业务判断 | 依赖人响应、夜间延迟高 |
| **预案预降级** | 大促前提前执行 | 提前数小时 | 已知流量高峰（双11、618、抢购） | 牺牲常态能力，需事后回滚 |

**自动 vs 手动的边界**：
- **能自动化的**：指标明确、动作可逆、影响面小的（弱依赖熔断、单接口兜底）
- **必须人工的**：影响业务核心、需要业务判断的（关闭下单、降级支付通道、关闭活动入口）
- **架构原则**：自动保底 + 人工兜底，永远要有"一键开关"

**为什么大促前要做预降级？**
1. **大促瞬间流量是常态 10x+**，等自动熔断触发时雪崩已经发生
2. **提前释放资源**：关闭推荐、评论、积分写入等，把 CPU/DB/线程资源留给核心链路
3. **提前预热缓存**：避免缓存击穿
4. **提前扩容 + 预降级 + 预热**是"三预"标准动作，是大促保障的 SOP

#### 4. 兜底数据设计

| 选项 | 适用场景 | 秒杀例子 | 风险 |
| --- | --- | --- | --- |
| **静态缓存** | 数据可旧、必须返回内容 | 商品详情页 → CDN 静态页（价格可能稍旧） | 价格不一致风险，需标注"仅供参考" |
| **默认值** | 数据可缺失、不可空 | 推荐位 → 默认热门商品列表 | 默认值污染业务数据 |
| **空** | 前端可容忍空态 | 评论列表 → 空数组 + "暂无评论" | 用户体验差 |
| **稍后重试** | 强一致、不可旧 | 支付结果 → "查询中，请稍后" | 用户焦虑，需配套轮询机制 |

**选型决策树**：
```
业务能否容忍"旧"？
  ├─ 能 → 静态缓存（首选，用户体验最好）
  └─ 不能 → 业务能否容忍"缺"？
            ├─ 能 → 默认值 / 空
            └─ 不能 → "稍后重试"（强一致场景）
```

**秒杀商品详情页的典型兜底**：CDN 静态页（含商品基础信息） + Redis 缓存（库存/价格，TTL 1s） → 全部失效时返回"活动太火爆，请刷新重试"。

#### 5. 真实降级案例（蛋壳钱包 / 平安健康险商城）

**案例：平安健康险商城 - 健康告知接口降级**

- **背景**：投保流程中"健康告知"调用第三方风控接口，正常 RT 200ms，大促期间第三方接口抖动到 5s+
- **触发条件**：Sentinel 监控到该接口慢调用比例 > 60%（持续 30s，滑窗 10s）
- **降级动作**：
  1. 自动熔断该风控接口，进入 OPEN 状态
  2. fallback 走"轻量健康告知"——返回简化版告知问卷（不含第三方风控强校验）
  3. 投保流程不中断，标记该单为"待人工复核"
- **业务感知**：用户无感知，下单成功；次日运营在后台看到"待复核"订单，人工补做风控
- **回滚流程**：
  1. 第三方接口恢复后，Sentinel HALF_OPEN 探测连续 5 次成功 → CLOSED
  2. 自动熔断解除，但 fallback 期间生成的"待复核"订单仍需人工处理
  3. 复盘后调整阈值：慢调用比例 60% → 70%，避免过度敏感

**踩坑经验**：
- 一开始 fallback 直接返回"健康告知通过"，导致部分高风险订单流入——后改为"轻量告知 + 人工复核"
- 自动熔断阈值设太低（50%），第三方接口正常抖动就触发，导致大量订单走人工复核，运营压力暴增
- 后续引入"分级降级"：50% 触发告警不熔断，70% 才真正熔断降级

#### 与架构师水平的差距及补足方向

**差距**：
1. **降级预案体系不完整**——业务里只有零散的 fallback，没有形成"L1~L5 分层 + 触发条件 + 回滚 SOP"的体系
2. **自动 vs 手动的边界模糊**——要么过度自动化（误判频发），要么过度人工（响应慢）
3. **兜底数据设计粗糙**——多数场景直接返回空或抛错，没考虑"静态缓存 + 默认值"的精细化设计
4. **预降级演练少**——大促前没系统做"三预"，往往是出事后才补
5. **强依赖的"一致性降级"经验不足**——TCC 在蛋壳钱包用过，但和熔断/降级的协同设计没想透

**补足方向**：
- 整理一份"降级预案 SOP"：每个降级开关的触发条件、影响范围、回滚步骤、责任人
- 引入混沌工程（Chaos Engineering）做主动故障注入演练
- 在秒杀系统中实践"预降级 + 自动熔断 + 人工开关"三层联动
- 学习 Spring Cloud Alibaba + Sentinel 的规则推送机制，搭建可视化降级控制台
