> 日期：2026年06月24日（周三）

## 一、核心问题：为什么单机治理不够？

Day01 解决"单机怎么挡流量"，Day02 解决"故障怎么不扩散"，Day03 解决两个新问题：

1. **全局资源保护**：单机限流只看本机，挡不住"8 台节点 × 200 QPS = 1600 QPS"打穿库存服务
2. **依赖故障隔离**：熔断只解决"快速失败"，没解决"调用方主线程被占用"——舱壁隔离来补位

**架构师视角**：限流、熔断、降级、隔离是高可用四件套，缺一不可。前三件解决"流量与故障的传导"，隔离解决"故障的物理隔离"。

## 二、集群限流：三条路径的本质差异

### 1. 三条路径的本质

| 路径 | 本质 | 类比 |
| --- | --- | --- |
| Redis + Lua | 集中存储 + 集中计算 | 中央银行统一发钞 |
| Token Server | 集中签发 + 分布缓存 | 中央发令牌、地方批量领取 |
| 网关层 | 入口集中过滤 | 高速公路收费站 |

**架构师选型心法**：
- **想精确** → Redis + Lua（每次实时判断）
- **想高性能** → Token Server（批量拉取降低 RPC）
- **想简单** → 网关层（复用现有基建，运维成本最低）

### 2. 三层防线设计

```
[客户端] → [网关限流] → [应用集群限流] → [应用本地限流] → [资源]
            L1           L2                L3
         粗粒度        中粒度            细粒度
         挡异常流量    精确全局控制       兜底容灾
```

**每层职责**：
- **L1 网关层**：粗粒度，挡 80% 异常流量（如黑名单、IP 限流、地理限流）
- **L2 集群限流**：精确控制全局资源（如秒杀全局 1000 QPS）
- **L3 本地限流**：兜底，集群限流故障时启动，保证基本可用

**核心原则**：**精度递减、可用性递增**——越靠近资源越精确，越靠近入口越鲁棒。

### 3. CAP 取舍的工程答案

| 选项 | 一致性 | 可用性 | 适用场景 |
| --- | --- | --- | --- |
| Redis 同步限流 | 强 | 弱 | 强一致核心（库存扣减） |
| Token Server 批量 | 最终 | 中 | 通用业务限流 |
| 本地兜底 | 弱 | 强 | 容灾兜底 |

**架构师心法**：秒杀不追求"严格不多发一张令牌"，追求"任何故障下系统不崩"。**精度换可用性**是值得的，前提是事后对账。

## 三、舱壁隔离：选型决策的本质

### 1. 线程池 vs 信号量的本质差异

| 维度 | 线程池隔离 | 信号量隔离 |
| --- | --- | --- |
| 调用方式 | 异步（独立线程） | 同步（主线程） |
| 主线程阻塞 | 不阻塞 | 阻塞 |
| 慢调用影响 | 仅拖垮自己线程池 | 拖垮主线程池 |
| 性能开销 | 线程切换 1~10μs | 内存计数器，纳秒级 |
| 适用场景 | RT 不稳定、有慢调用 | RT 稳定且低 |

**架构师心法**：**信号量是"计数器"，线程池是"物理隔离"**。慢调用场景下，信号量的"计数器"会变成"阻塞主线程的锁"，必须用线程池。

### 2. 选型决策树

```
下游 RT 稳定吗？
├─ 不稳定（P99 > 1s）→ 线程池隔离（必须）
└─ 稳定
    └─ RT 高吗（> 100ms）？
        ├─ 高 → 线程池隔离（避免主线程阻塞）
        └─ 低 → 信号量隔离（性能优先）
```

**扩展维度**：
- **依赖强度**：强依赖倾向线程池（保护更彻底），弱依赖可信号量
- **QPS 量级**：极高 QPS（> 10000）倾向信号量（避免线程切换开销）
- **第三方依赖**：必须线程池（不可控因素多）

### 3. 线程池参数设计的 Little's Law

**Little's Law**：`并发数 = QPS × RT`

**示例**：推荐服务 QPS=3000，RT=200ms
- 理论并发：3000 × 0.2 = 600
- 慢调用场景：3000 × 2 = 6000（无法全容）
- **设计**：core=20 + queue=100 ≈ 120 并发，配合熔断+限流兜底剩余 5880

**架构师心法**：线程池不是"装下所有请求"，而是"装下正常态 + 触发熔断兜底异常态"。

## 四、热点参数限流：动态识别的工程价值

### 1. 为什么需要热点参数限流

- **普通限流**：接口维度，单 SKU 热点会"饿死"其他 SKU
- **集群总限流**：全局阈值，挡不住单 SKU 局部热点
- **热点参数限流**：参数维度，TopN 热点单独限流

### 2. LFU vs LRU 在秒杀场景的差异

| 维度 | LRU | LFU |
| --- | --- | --- |
| 驱逐依据 | 时间（最近最少使用） | 频率（最不经常使用） |
| 秒杀表现 | 长尾 SKU 干扰真热点 | 真热点稳定识别 |
| 突发响应 | 慢 | 快 |

**架构师心法**：秒杀热点是"频率驱动的"，不是"时间驱动的"——LFU 比 LRU 更匹配业务特征。Sentinel 用 W-TinyLFU 兼顾内存与响应。

### 3. 三级限流协同

```
单 SKU QPS > 1000 → 热点参数限流（精细到 SKU）
   ↓
全局 QPS > 50000 → 集群总限流（保护入口）
   ↓
Redis 故障 → 本地兜底限流（保可用）
```

## 五、舱壁 + 熔断 + 降级：三件套联动

### 1. 触发顺序

```
请求进入
   ↓
线程池未满 → 正常调用
   ↓ 线程池满
进队列等待 → 排队超时 → AbortPolicy 拒绝
   ↓ 慢调用比例超阈值
触发熔断 → OPEN 快速失败
   ↓ 熔断冷却期结束
HALF_OPEN 探测 → 成功 → CLOSED / 失败 → 重新 OPEN
   ↓ 全程
fallback 兜底数据返回
```

### 2. 拒绝策略选型

| 策略 | 秒杀场景适用性 |
| --- | --- |
| AbortPolicy | ✅ 推荐，触发 fallback |
| CallerRunsPolicy | ❌ 致命，主线程被拖垮 |
| DiscardPolicy | ❌ 静默丢弃，体验差 |
| DiscardOldestPolicy | ❌ 旧请求丢失，数据不一致 |

**架构师心法**：CallerRunsPolicy 看似"不丢请求"，实际是"用主线程陪葬"——秒杀场景必须禁用。

### 3. 兜底数据设计原则

| 数据类型 | 兜底策略 |
| --- | --- |
| 强一致（库存、价格） | "稍后重试"或保守兜底，不可假成功 |
| 弱依赖（推荐、评论） | 默认值或空态 |
| 历史数据（订单历史） | 缓存最近一次结果，标注"仅供参考" |

## 七、能力差距与补足方向（架构师视角）

### 差距一：四件套联动经验不足

**现状**：限流、熔断、降级、隔离分别用过，但没形成"超时 → 隔离 → 熔断 → 降级"的完整链路设计。

**架构师应具备**：
- 任何一个外部依赖调用，都能立即画出"四件套如何联动"的时序图
- 知道每个组件的触发顺序、阈值协同、回滚机制

**补足**：在秒杀系统为每个下游依赖画"四件套联动图"，做故障注入压测验证。

### 差距二：故障容灾的"精度 vs 可用性"权衡不清晰

**现状**：Redis 故障时是 fail-open 还是 fail-close，没统一规范。

**架构师应具备**：
- 每个限流组件故障时，能立即回答"业务放行还是拒绝"，并说明理由
- 知道"精度换可用性"的边界——什么场景能换，什么场景不能

**补足**：整理《限流容灾 SOP》，覆盖 Redis、Token Server、网关三类组件的故障场景。

### 差距三：参数化设计能力不足

**现状**：限流阈值、线程池参数靠拍脑袋，没基于 Little's Law 量化。

**架构师应具备**：
- 任何限流阈值都能给出"QPS × RT × 安全系数"的推导
- 线程池参数能基于并发模型计算，而非默认值

**补足**：建立"参数计算模板"，每个核心接口的限流/隔离参数都有量化依据。

### 差距四：热点治理意识薄弱

**现状**：秒杀只用了接口级限流，没做 SKU 维度热点治理。

**架构师应具备**：
- 识别业务中的"热点维度"（SKU、用户、IP、商户），主动设计热点参数限流
- 知道 LFU 在热点探测中的优势，并能选型

**补足**：在秒杀系统引入 Sentinel 热点参数限流，覆盖 SKU、用户 ID 两个维度。

### 差距五：演练缺失

**现状**：治理方案上线后没做故障注入演练，下次故障才发现问题。

**架构师应具备**：
- 治理方案必须有"故障注入验证报告"
- 定期做 Chaos Engineering 演练

**补足**：引入 Chaos Mesh，每月做一次"下游慢调用 / 下游超时 / 依赖故障"演练。
