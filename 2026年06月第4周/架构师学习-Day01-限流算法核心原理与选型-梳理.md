## 一、限流在高可用架构中的定位

### 1.1 核心一句话
> **限流 = 系统过载时的"主动防御闸门"，用可控的拒绝换取系统存活**
>
> 与熔断、降级合称高可用三件套，但限流是**第一道闸门**——它的目标是"不让系统进入过载状态"。

### 1.2 限流 / 熔断 / 降级的边界
| 机制 | 触发时机 | 作用 | 类比 |
| --- | --- | --- | --- |
| 限流 | 流量超过阈值 | 主动拒绝多余请求 | 水库闸门 |
| 熔断 | 下游故障/慢调用比例高 | 快速失败，不调用下游 | 电路保险丝 |
| 降级 | 系统压力过大 / 非核心功能 | 牺牲非核心，保核心 | 弃车保帅 |

```latex
请求进入
  → 限流（拒绝超额）       ← 第一道闸门
    → 熔断（快速失败）     ← 第二道闸门，保护下游
      → 降级（返回兜底）   ← 第三道闸门，保核心
        → 业务处理
```

### 1.3 限流的多层位置
```latex
 CDN / WAF        ← 黑名单、地理限流
   ↓
 Nginx / 网关     ← IP/接口级限流（10w+ QPS 第一道）
   ↓
 应用层 (Sentinel) ← 接口/方法级限流、热点参数
   ↓
 资源层 (Redis/DB) ← 连接池、线程池、慢SQL保护
```

## 二、四种限流算法：架构师必须能画出来

### 2.1 固定窗口计数器（Fixed Window）
```latex
|<----- 窗口1 (0-1s) -----|<----- 窗口2 (1-2s) -----|
       允许 100 次                允许 100 次

请求计数：cnt += 1
判断：cnt > 100 ? 拒绝 : 放行
窗口切换：cnt = 0
```

**致命缺陷：临界突发流量（也叫边界突刺）**
```latex
0.9s  来 100 个请求 → 全部放行
1.0s  窗口切换，cnt 归零
1.1s  来 100 个请求 → 全部放行

结果：0.9s~1.1s 这 200ms 内放行 200 个请求，是阈值的 2 倍
```

### 2.2 滑动窗口计数器（Sliding Window）
```latex
把固定窗口切成更小的"格子"，窗口滑动时累加格子计数

|格1|格2|格3|格4|格5|格6|      ← 每格 200ms
        ↑
     当前窗口（最近 1s = 5 格）

判断：sum(当前窗口内所有格子) > 阈值 ? 拒绝 : 放行
```

**关键改进**：窗口越细，临界突刺越小；格子无限多 → 退化为滑动日志。

**Sentinel 默认实现**：LeapArray 结构，每个 Sample 一个格子，时间轮转。

### 2.3 漏桶算法（Leaky Bucket）
```latex
       请求进入（任意速率）
          ↓
       ┌──────┐
       │ 漏桶  │ ← 容量 N（缓冲队列）
       │      │
       └──┬───┘
          ↓ 漏出（固定速率 R）
       拒绝? 桶满 → 直接丢弃/排队
```

**核心特征**：
- **入口任意速率，出口恒定速率**
- 强制平滑流量，**不允许突发**
- 适合保护下游弱系统（如数据库、第三方接口）

### 2.4 令牌桶算法（Token Bucket）
```latex
   令牌生成（固定速率 R）
          ↓
       ┌──────┐
       │ 令牌桶│ ← 容量 B（最多存 B 个令牌）
       │      │   桶满则丢弃多余令牌
       └──┬───┘
          ↓ 取令牌
       请求来了：拿到令牌 → 放行；没有 → 拒绝/等待
```

**核心特征**：
- **允许突发**：桶里攒的令牌可一次性消耗（最多 B 个突发）
- 长期平均速率 = R，短期可超 R
- 适合网关、API 入口（容忍用户偶发突发）

## 三、四种算法对比矩阵

| 维度 | 固定窗口 | 滑动窗口 | 漏桶 | 令牌桶 |
| --- | --- | --- | --- | --- |
| 实现复杂度 | ⭐ 最简单 | 中等 | 中等 | 中等 |
| 临界突发 | ❌ 严重 | ✅ 减弱 | ✅ 无 | ✅ 无 |
| 突发流量 | ✅ 允许 | ✅ 允许 | ❌ 不允许 | ✅ 允许（受桶容量限制）|
| 流量整形 | ❌ 无 | ❌ 弱 | ✅ 强（强制匀速）| ⚠️ 中等 |
| 内存开销 | O(1) | O(格子数) | O(队列) | O(1) |
| 分布式友好 | ✅ 易 | ✅ 易 | ⚠️ 队列要集中 | ✅ 易 |
| 典型实现 | 计数器 | Sentinel | Nginx leak bucket | Guava RateLimiter |

## 四、Sentinel vs Guava RateLimiter 的设计动因

### 4.1 Sentinel 为什么用滑动窗口
```latex
Sentinel 定位：微服务流量治理（限流 + 熔断 + 系统自适应保护）

需求：
1. 实时统计 QPS、慢调用比例、异常比例 → 熔断依据
2. 多维度限流：QPS、并发线程数、热点参数
3. 允许突发（业务流量本就不均匀）

滑动窗口的优势：
- 统计与限流共用一份数据结构（LeapArray）
- 窗口可精细到 500ms / 1s，临界突刺可控
- 不强制匀速，符合微服务真实流量模式
```

### 4.2 Guava RateLimiter 为什么用令牌桶
```latex
Guava 定位：单机通用速率限制器（如限制打印日志频率、调用第三方API）

需求：
1. 单机、单进程、低开销
2. 平滑输出请求，避免瞬时洪峰冲击下游
3. 支持预热（warmupPeriod）和突发（burst）

令牌桶的优势：
- 支持突发：攒了令牌可以一次性用
- 平滑限速：长期速率受控
- 实现简单：单机只需 AtomicLong + 上次请求时间

Guava 的两个经典实现：
- SmoothBursty    : 默认，允许突发
- SmoothWarmingUp : 带预热，刚启动时速率逐步提升
```

### 4.3 选型决策树
```latex
要保护下游弱系统，强制匀速 → 漏桶
要单机限速，允许突发         → 令牌桶（Guava）
要微服务多维度治理 + 熔断    → 滑动窗口（Sentinel）
要简单粗粒度，能容忍临界突刺 → 固定窗口（不推荐生产）
```

## 五、Redis + Lua 分布式限流：必须能讲清楚

### 5.1 为什么必须用 Lua
```latex
朴素 Redis 实现：
  1. GET key
  2. 判断 count < limit ?
  3. INCR key
  4. EXPIRE key 1

并发问题：
  线程A: GET → 99       线程B: GET → 99
  线程A: INCR → 100     线程B: INCR → 101 ❌ 超限

Lua 的作用：
  把"判断 + 计数 + 过期"打包成原子操作（Redis 单线程执行 Lua）
  一次网络往返，且不会被其他命令插入
```

### 5.2 滑动窗口版 Lua 脚本（核心思路）
```lua
-- KEYS[1] = 限流 key
-- ARGV[1] = 窗口大小(ms)
-- ARGV[2] = 阈值
-- ARGV[3] = 当前时间戳(ms)

local key = KEYS[1]
local window = tonumber(ARGV[1])
local limit = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local clear_before = now - window

-- 1. 移除过期记录
redis.call('ZREMRANGEBYSCORE', key, '-inf', clear_before)

-- 2. 当前窗口内请求数
local cnt = redis.call('ZCARD', key)

if cnt < limit then
    -- 3. 记录本次请求（member 唯一，score = now）
    redis.call('ZADD', key, now, now .. '-' .. math.random())
    redis.call('PEXPIRE', key, window)
    return 1  -- 放行
else
    return 0  -- 拒绝
end
```

**为什么用 ZSET**：member 唯一保证多次请求不合并，score 是时间戳便于范围删除。

### 5.3 令牌桶版 Lua 脚本（核心思路）
```lua
-- ARGV: capacity, rate, now, requested

local last_time = tonumber(redis.call('HGET', key, 'last_time') or now)
local tokens = tonumber(redis.call('HGET', key, 'tokens') or capacity)

-- 1. 按时间差补充令牌
local delta = math.max(0, now - last_time)
tokens = math.min(capacity, tokens + delta * rate / 1000)

-- 2. 判断 + 扣减
if tokens < requested then
    return 0
end
tokens = tokens - requested
redis.call('HMSET', key, 'tokens', tokens, 'last_time', now)
redis.call('PEXPIRE', key, ttl)
return 1
```

## 六、Redis 集群模式下的限流坑

### 6.1 hash tag 保证 key 同分片
```latex
问题：限流 key = user:1001, user:1002 ...
     如果落到不同分片，每个分片独立计数 → 阈值变成 N倍

解法：用 hash tag 强制路由
     key = {rate}:user:1001
     key = {rate}:user:1002
     → 都落到同一分片（按 {} 内文本计算 slot）

代价：所有限流 key 挤在一个分片，可能成为热点
取舍：单 key 限流规模小时用 hash tag；
     限流规模大时改用"分片计数 + 汇总"方案
```

### 6.2 分片计数方案（10w+ QPS 必备）
```latex
方案：把单 key 计数拆成 N 个子 key，分散到不同分片

  rate:user:1001:0  ──→ 分片 A
  rate:user:1001:1  ──→ 分片 B
  rate:user:1001:2  ──→ 分片 C
  ...

请求来了：
  idx = random(0, N-1)
  对 rate:user:1001:idx 做 INCR

放行判断：
  sum(所有子 key) < 阈值 ? 放行 : 拒绝
  （或用近似判断：单子 key 阈值 = 总阈值 / N）

优点：分摊热点，单分片压力 1/N
缺点：判断不严格（汇总有延迟），实现复杂
```

## 七、fail-open vs fail-close：架构师的核心取舍

### 7.1 两种策略
```latex
fail-open (放行)：
  Redis 挂了 → 限流失效 → 请求全放行
  代价：可能压垮下游
  适合：限流为辅、保业务可用性优先的场景

fail-close (拒绝)：
  Redis 挂了 → 限流失效 → 请求全拒绝
  代价：业务不可用
  适合：限流为命脉、保系统存活优先的场景
```

### 7.2 决策矩阵
| 业务类型 | Redis 挂时策略 | 理由 |
| --- | --- | --- |
| 秒杀下单 | fail-close | 不拒绝就 DB 雪崩 |
| 支付回调 | fail-open | 钱不能少，宁可慢 |
| 用户登录 | fail-open | 用户体验优先 |
| 第三方 API 调用 | fail-close | 保护下游不被罚款 |
| 普通查询 | fail-open | 业务可用性优先 |

### 7.3 工程上的混合策略
```latex
1. 双层限流：
   - 本地令牌桶（兜底，fail-close 阈值 = 平时QPS）
   - Redis 分布式限流（精确，fail-open 阈值 = 峰值QPS）
   - Redis 挂时退化为本地限流，保住基本盘

2. 限流降级开关：
   - 配置中心开关，Redis 挂时切换到本地限流或全放行
   - 配合监控告警，运维能快速决策

3. Redis 高可用兜底：
   - 主从 + 哨兵，避免单点
   - 限流 key 多副本，主挂从接管（短暂不准但能用）
```

## 八、限流方案选型矩阵

| 方案 | 适用规模 | 优点 | 缺点 | 典型场景 |
| --- | --- | --- | --- | --- |
| 本地计数器 | 单机 <1k QPS | 实现极简 | 不能集群 | 内部管理后台 |
| Guava RateLimiter | 单机 <1w QPS | 平滑、低开销 | 单机限流 | 应用内调用第三方 |
| Sentinel 单机 | 单机 <5w QPS | 多维度、可熔断 | 单机统计 | 微服务应用层 |
| Sentinel 集群流控 | 集群 <5w QPS | 集群精确 | Token Server 单点 | 微服务集群精确限流 |
| Redis + Lua | 集群 10w+ QPS | 精确、通用 | 依赖 Redis | 网关层、热点参数 |
| Nginx + Lua | 网关 10w+ QPS | 网关层拦截 | 与业务耦合 | 入口网关限流 |
| 网关自带限流 | 网关 10w+ QPS | 与路由一体 | 灵活性不足 | Spring Cloud Gateway |

## 九、架构师面试高频考点速记

```latex
Q1: 限流和熔断的区别？
   限流：流量超过阈值时主动拒绝，预防过载
   熔断：下游故障时快速失败，防止级联失败
   限流是入口防御，熔断是下游保护

Q2: 滑动窗口和固定窗口区别？
   固定窗口在切换瞬间会有 2 倍阈值的突刺
   滑动窗口通过细粒度格子减弱突刺
   格子越多越精确，但内存开销越大

Q3: 漏桶和令牌桶区别？
   漏桶：出口匀速，不允许突发，强制整形
   令牌桶：入口取令牌，允许突发（最多桶容量）
   选谁：保护弱系统选漏桶，容忍突发选令牌桶

Q4: 为什么 Redis 限流必须用 Lua？
   保证"判断 + 计数 + 过期"原子执行
   不用 Lua 会并发超限（GET 和 INCR 之间被插入）

Q5: Redis 集群下限流 key 怎么保证准确？
   方案1: hash tag 强制同分片（小规模）
   方案2: 分片计数 + 汇总（大规模，10w+ QPS）

Q6: fail-open 还是 fail-close？
   看业务：保命选 close，保业务选 open
   生产实践：双层限流（本地兜底 + 分布式精确）

Q7: Sentinel 为什么默认滑动窗口？
   1. 限流和熔断共用统计（一份数据多用途）
   2. 微服务流量本就不均匀，要允许突发
   3. 多维度限流（QPS、并发数、热点参数）

Q8: 热点参数限流怎么做？
   Sentinel HotParamLimitChecker：每个参数值独立计数
   蛋壳钱包场景：单 userId 限流防恶意刷单
   实现要点：LRU 缓存热点参数，冷数据淘汰
```

## 十、本日核心心智模型

```latex
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  限流的一切设计都围绕一个核心问题：                         │
│                                                          │
│    "如何在不杀死系统的前提下，尽可能多地放行请求？"         │
│                                                          │
│  四种算法 = 四种回答方式：                                 │
│   - 固定窗口：粗放，临界会爆（最差）                       │
│   - 滑动窗口：精确，允许突发（最常用）                     │
│   - 漏桶：强制匀速，保护弱系统                             │
│   - 令牌桶：允许突发，平滑长期速率                         │
│                                                          │
│  分布式限流的核心矛盾：                                   │
│    精确 vs 性能                                          │
│    - 全局精确 → 集中计数 → 单点瓶颈                       │
│    - 分散高性能 → 分片计数 → 不严格                       │
│    - 架构师 = 在两者间找到平衡                            │
│                                                          │
│  Fail 策略的核心矛盾：                                   │
│    可用性 vs 安全性                                      │
│    - fail-open：保可用，可能雪崩                          │
│    - fail-close：保安全，可能误杀                         │
│    - 架构师 = 双层兜底 + 配置可切换                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```
