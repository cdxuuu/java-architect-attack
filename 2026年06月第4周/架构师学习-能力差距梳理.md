## 一、概念定义化、缺故障链路推演（延续）

> Day2发现，延续第2周差距一

### 当前表现

回答"X 是什么"类问题时，倾向给出定义，例如"限流是流量超过阈值就拒绝"。概念本身正确，但缺链路。

### 与架构师水平的差距

架构师面试需要从"现象定义"推进到"链路推演"。例如限流缺失时：

```text
流量洪峰
  ↓
应用线程池打满
  ↓
Tomcat 拒绝连接 / 请求堆积
  ↓
DB 连接池打满
  ↓
慢查询堆积、行锁等待
  ↓
应用线程超时
  ↓
上游重试放大流量
  ↓
雪崩
```

### 补足方向

本周回答限流熔断类问题时统一使用三段式：
1. 现象定义：发生了什么
2. 链路推演：流量怎么一步步放大到系统雪崩
3. 治理组合：入口、网关、应用、资源多层防线

## 二、治理方案单点化、缺组合防线（延续）

> Day3发现，延续第2周差距二

### 当前表现

限流题容易只回答"用 Sentinel"或"用 Redis+Lua"，缺多层防线设计。

### 与架构师水平的差距

限流的组合防线应该是：

```text
第一层：CDN/WAF（黑名单、地理限流）
第二层：网关层（Nginx+Lua、Spring Cloud Gateway，IP/接口级）
第三层：应用层（Sentinel，方法/热点参数级）
第四层：资源层（连接池、线程池、慢SQL保护）
第五层：降级兜底（返回默认值、缓存旧值、快速失败）
```

### 补足方向

回答限流方案时按"多层防线 + fail 策略 + 兜底"组织，而不是只给一个 Redis+Lua。

## 三、主链路清晰、异常闭环与对账弱（延续）

> Day6发现，延续第2周差距六

### 当前表现

限流方案能讲主链路（Redis+Lua 计数 + 拒绝），但 Redis 故障、Lua 超时、集群分片不一致等异常分支表达不足。

### 与架构师水平的差距

异常闭环应包括：

```text
1. Redis 挂了 → fail-open 还是 fail-close，依据业务语义
2. Lua 执行超时 → 兜底策略（本地限流 / 放行）
3. 集群分片不均 → hash tag 还是分片计数
4. 限流误杀 → 监控告警 + 动态调整阈值
5. 限流统计偏差 → 对账（限流日志 vs 实际通过）
```

### 补足方向

每个限流方案都要补一段"异常分支 + 兜底 + 监控对账"。

## 四、技术方案缺业务语义对齐（延续）

> Day5发现，延续第2周差距五

### 当前表现

可能只说"秒杀用 fail-close、支付用 fail-open"，但缺业务语义层面的论证。

### 与架构师水平的差距

fail 策略本质是业务语义取舍：

```text
秒杀下单：保系统存活 > 保订单成交 → fail-close
支付回调：保资金不丢 > 保系统存活 → fail-open
用户登录：保用户体验 > 保系统存活 → fail-open
第三方API：保下游不被罚款 > 保本系统可用 → fail-close
```

### 补足方向

每个限流策略选择都落到业务语义：保什么、牺牲什么、阈值依据。
