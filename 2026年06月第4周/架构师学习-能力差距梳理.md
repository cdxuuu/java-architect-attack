## 一、概念定义化、缺故障链路推演（延续）

> Day2发现，延续第2周差距一

### 当前表现

回答"X 是什么"类问题时，倾向给出定义，例如"限流是流量超过阈值就拒绝"。概念本身正确，但缺链路。

### 与架构师水平的差距

架构师面试需要从"现象定义"推进到"链路推演"。例如限流缺失时：

```text
流量洪峰
  ↓
应用线程池打满
  ↓
Tomcat 拒绝连接 / 请求堆积
  ↓
DB 连接池打满
  ↓
慢查询堆积、行锁等待
  ↓
应用线程超时
  ↓
上游重试放大流量
  ↓
雪崩
```

### 补足方向

本周回答限流熔断类问题时统一使用三段式：
1. 现象定义：发生了什么
2. 链路推演：流量怎么一步步放大到系统雪崩
3. 治理组合：入口、网关、应用、资源多层防线

## 二、治理方案单点化、缺组合防线（延续）

> Day3发现，延续第2周差距二

### 当前表现

限流题容易只回答"用 Sentinel"或"用 Redis+Lua"，缺多层防线设计。

### 与架构师水平的差距

限流的组合防线应该是：

```text
第一层：CDN/WAF（黑名单、地理限流）
第二层：网关层（Nginx+Lua、Spring Cloud Gateway，IP/接口级）
第三层：应用层（Sentinel，方法/热点参数级）
第四层：资源层（连接池、线程池、慢SQL保护）
第五层：降级兜底（返回默认值、缓存旧值、快速失败）
```

### 补足方向

回答限流方案时按"多层防线 + fail 策略 + 兜底"组织，而不是只给一个 Redis+Lua。

## 三、主链路清晰、异常闭环与对账弱（延续）

> Day6发现，延续第2周差距六

### 当前表现

限流方案能讲主链路（Redis+Lua 计数 + 拒绝），但 Redis 故障、Lua 超时、集群分片不一致等异常分支表达不足。

### 与架构师水平的差距

异常闭环应包括：

```text
1. Redis 挂了 → fail-open 还是 fail-close，依据业务语义
2. Lua 执行超时 → 兜底策略（本地限流 / 放行）
3. 集群分片不均 → hash tag 还是分片计数
4. 限流误杀 → 监控告警 + 动态调整阈值
5. 限流统计偏差 → 对账（限流日志 vs 实际通过）
```

### 补足方向

每个限流方案都要补一段"异常分支 + 兜底 + 监控对账"。

## 六、四件套联动经验不足（Day3发现，本周新差距）

> Day3发现

### 当前表现

限流、熔断、降级、隔离这四件套在业务里都零散用过（Sentinel、Redisson、Hystrix 老项目），但没有形成"超时 → 隔离 → 熔断 → 降级"的完整链路设计。具体表现：
- 改造第三方依赖时只配了线程池隔离，漏配超时和熔断，故障时主线程虽不被拖垮但调用方仍同步等待 8s
- 拒绝策略默认用 `CallerRunsPolicy`（怕丢请求），故障时反而把 Tomcat 主线程拖垮
- 熔断和隔离分开配，没形成"线程池满 → 排队 → 熔断 → fallback"的有序链路

### 与架构师水平的差距

架构师对任何一个外部依赖调用，应能立即画出四件套联动时序图：

```text
请求进入
  ↓
线程池未满 → 正常调用（统计 RT 与异常率）
  ↓ 线程池满
进队列等待 → 超时拒绝（AbortPolicy，触发 fallback）
  ↓ 慢调用比例超阈值
触发熔断（OPEN）→ 快速失败
  ↓ 冷却期结束
HALF_OPEN 探测 → CLOSED / 重新 OPEN
  ↓ 全程
fallback 兜底数据返回
```

四件套缺一不可：
- **缺超时**：隔离了但主线程仍同步等待，等于没隔离
- **缺隔离**：熔断只能"快速失败"，主线程仍被占用
- **缺熔断**：故障持续时线程池反复打满，恢复慢
- **缺降级**：拒绝后用户看到 5xx，体验崩溃

### 补足方向

1. 在秒杀系统为每个下游依赖画"四件套联动图"，明确触发顺序与阈值协同
2. 引入 Chaos Mesh 主动注入"下游慢调用 / 超时 / 异常"故障，验证联动有效性
3. 整理《舱壁隔离选型 SOP》：按 RT、QPS、依赖强度三维度给推荐配置
4. 全局禁用 `CallerRunsPolicy`，统一 `AbortPolicy` + fallback 兜底

## 七、热点参数限流意识缺失（Day3发现，本周新差距）

> Day3发现

### 当前表现

秒杀系统只用了接口级限流（如查询 SKU 接口总 QPS ≤ 10000），没做 SKU 维度热点治理。曾出现"爆款 SKU 单独被刷爆缓存，其他 SKU 流量正常但全站受影响"的故障。

### 与架构师水平的差距

架构师应能识别业务中的"热点维度"并主动治理：

| 业务场景 | 热点维度 | 治理方式 |
| --- | --- | --- |
| 秒杀 | SKU ID | 热点参数限流 + 单 SKU 独立阈值 |
| 用户防刷 | User ID | 用户级限流 + 黑名单 |
| 商户限流 | Merchant ID | 商户级配额 |
| IP 防刷 | IP | IP 限流 + 地理位置 |

对 LFU vs LRU 在热点探测中的差异要有量化理解：
- LRU 在"多 SKU 轮流被刷"时会误判热点（长尾偶发访问挤掉真热点）
- LFU 基于频率，真热点稳定识别，秒杀场景必选
- Sentinel 用 W-TinyLFU，兼顾内存效率与响应速度

### 补足方向

1. 秒杀系统引入 Sentinel `ParamFlowRule`，覆盖 SKU、User ID 两个热点维度
2. 学习 W-TinyLFU 算法（Caffeine 实现），理解频率衰减机制
3. 整理《热点参数限流配置 SOP》：TopN 探测、单参数阈值、全局兜底三层配置

## 八、参数化设计能力不足（Day3发现，本周新差距）

> Day3发现

### 当前表现

限流阈值、线程池参数多靠"拍脑袋 + 经验值"，没基于 Little's Law 量化。例如推荐服务 QPS=3000、RT=200ms，理论并发 600，但线程池 core=20 + queue=100 ≈ 120，差距没说清楚。

### 与架构师水平的差距

架构师对任何限流/隔离参数应能给出量化推导：

```text
Little's Law：并发数 = QPS × RT

推荐服务案例：
  正常态：3000 × 0.2s = 600 并发
  慢调用：3000 × 2s = 6000 并发（无法全容）
  设计：core=20 + queue=100 ≈ 120 并发
  策略：正常态装下 1/5，剩余靠熔断+限流兜底
  安全系数：1.5（应对突发）
```

参数计算模板应包含：
- **限流阈值**：基于下游承载能力 × 安全系数
- **线程池大小**：基于 QPS × RT × 安全系数
- **队列长度**：基于"突发流量持续时间 × QPS"
- **超时时间**：基于下游 P99 RT × 2

### 补足方向

1. 建立《参数计算模板》，每个核心接口的限流/隔离参数都有量化依据
2. 学习 Netflix concurrency-limits 的自适应限流思想（基于 Little's Law 动态调整）
3. 在压测中验证参数合理性，形成"QPS-RT-并发数"对照表

## 九、自适应限流经验缺失（Day4发现，本周新差距）

> Day4发现

### 当前表现

业务里只用过固定阈值限流（Sentinel FlowRule + Redis+Lua），没系统化设计 SystemRule。平安健康险商城 JVM Full GC 雪崩案例中，QPS 始终未超 3000 阈值，固定阈值限流全程未触发，但并发已飙到 1500。

### 与架构师水平的差距

架构师对自适应限流应有完整方法论：

```text
1. 反馈信号体系：系统层 / 应用层 / 依赖层三层信号交叉验证
2. 算法选型：
   - Sentinel SystemRule（规则触发，5 维阈值）
   - Netflix concurrency-limits（AIMD 动态调整，限并发不限 QPS）
   - BBR-style（主动探测 BDP = maxQPS × minRT）
3. 与传统限流、熔断、隔离的四件套协同
4. 监控告警闭环：指标采集 → 告警分级 → 动态规则推送 → 故障自愈
```

核心心法：**固定阈值限流是"流量视角"，自适应限流是"系统视角"，两者必须配合**。

### 补足方向

1. 在秒杀系统落地 Sentinel SystemRule + concurrency-limits 双层自适应限流
2. 5 个阈值全部基于 Little's Law 量化推导（Load=8/CPU=80%/RT=200ms/并发=300/QPS=3000）
3. 精读 Netflix concurrency-limits 源码（Gradient2Limit、VegasLimit、AIMDLimit）
4. 引入 Nacos 动态规则推送 + 灰度发布 + 一键回滚

## 十、反馈信号体系与告警分级不清晰（Day4发现，本周新差距）

> Day4发现

### 当前表现

监控告警靠经验拍阈值，告警风暴未做收敛。SystemRule 触发时 CPU/Load/RT/并发四个告警同时炸，故障时被淹没。

### 与架构师水平的差距

架构师对告警体系应有分级与收敛设计：

```text
告警分级：
  P0：核心接口不可用 / 资金风险 / 数据丢失 → 立即介入
  P1：部分功能降级 / 性能严重退化 → 5-30 分钟内
  P2：容量预警 / 慢依赖苗头 → 日常巡检

告警收敛：
  - 同一根因的多个告警合并（CPU 告警抑制 Load 告警）
  - 相同应用 + 相同时段 + 关联指标 → 合并为一条
  - Alertmanager inhibit_rules 配置
```

### 补足方向

1. 整理《反馈信号采集 SOP》：每层信号采集方式、典型阈值、失效场景
2. 告警分级标准化：P0/P1/P2 三级，每级有响应时间与处置流程
3. 告警收敛：Alertmanager inhibit_rules + 拓扑根因分析

## 十一、全链路协同设计与容量规划不足（Day5发现，本周新差距）

> Day5发现

### 当前表现

业务里只关注单点（限流或熔断），没形成"挡 → 切 → 断 → 兜"的完整防线。容量规划靠拍脑袋，8 台 vs 16 台的扩容决策凭经验，没基于 Little's Law 量化推导。

### 与架构师水平的差距

架构师应有全链路视角：

```text
容量规划（Little's Law）：
  并发 = QPS × RT
  节点数 = 业务峰值 QPS / 单节点 QPS × 冗余系数（1.5）

四层防线全链路配置表：
  网关层 → 应用接口层 → 集群限流层 → 自适应限流层 → 舱壁隔离层 → 熔断降级层
  每层有维度、阈值、兜底动作、数值依据

协同原则：
  网关 > 集群限流 > 应用接口限流 > 自适应限流
  上层挡量，下层保命
```

### 补足方向

1. 在秒杀系统落地《四层防线全链路配置表》，每个核心接口一张表
2. 建立《容量规划 SOP》：业务目标 → Little's Law → 节点数 → 冗余系数
3. 每周 review 配置表，对比实际负载调整阈值

## 十二、故障演练与应急预案缺失（Day5发现，本周新差距）

> Day5发现

### 当前表现

业务里没做过 Chaos Mesh 故障注入，限流熔断配置未经验证。故障发生凭直觉处置，无 SOP，无回滚，无通信机制。

### 与架构师水平的差距

架构师应把故障演练作为高可用验证的唯一手段：

```text
故障演练覆盖度（至少 8 类）：
  1. 下游慢调用（RT 退化 10 倍）
  2. 下游超时
  3. 下游异常（5xx 50%）
  4. Redis 主从切换
  5. MySQL 慢 SQL
  6. JVM Full GC
  7. 网络抖动（丢包 30%）
  8. 节点宕机

应急预案（至少 7 场景）：
  限流过严 / 限流过松 / 下游熔断 / DB 宕机 / Redis 宕机 / 网关宕机 / 整体雪崩
  每个场景有：发现方式 + 处置步骤 + 回滚机制 + 通信机制
```

核心心法：**配了不练等于没配，故障演练是唯一验证手段**。

### 补足方向

1. 引入 Chaos Mesh 月度故障演练，覆盖 8 类典型故障
2. 编写《应急预案手册》——每个故障场景的 SOP + 回滚 + 通信，季度更新
3. 每次真实故障做 5why 复盘，追到"原则层面"而非"技术细节"
4. 建立"故障是常态"文化，主动制造故障优于被动等待
