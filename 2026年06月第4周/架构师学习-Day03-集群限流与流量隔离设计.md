## 背景

Day01 讲了单机限流算法（令牌桶 / 漏桶 / 滑动窗口），Day02 讲了熔断与降级。但你很快会遇到两个新问题：

1. **单机限流挡不住"全局总量"**——秒杀库存 1000 件，部署了 8 台应用节点，每台限流 200 QPS，全局就是 1600 QPS，照样把库存服务打穿。你需要的是**集群级精确限流**。

- 集群限流的三条实现路径（Redis+Lua / Token Server / 网关层）与选型权衡
- 舱壁隔离（线程池隔离 vs 信号量隔离）与热点参数限流的实战设计

## 题目一（原理题）：集群限流的三条实现路径与选型

### 1. 单机 vs 集群限流的边界

**判断准则：限流维度是否需要"全局视图"**

| 维度 | 单机限流 | 集群限流 |
| --- | --- | --- |
| 限流对象 | 本机资源（CPU、单机 QPS） | 全局资源（全局库存、全局限额、第三方配额） |
| 容错性 | 节点宕机不影响其他节点 | 限流组件故障影响全局 |
| 性能 | 内存操作，纳秒级 | 网络调用，毫秒级 |
| 适用场景 | 网关单机保护、应用线程池保护 | 秒杀全局总量、API 调用第三方配额、用户级防刷 |

**两个具体例子**：

1. **"全局库存 1000 件"必须用集群限流**
   - 8 台节点各 200 QPS，全局 1600 QPS，远超库存承受能力
   - 集群限流把全局阈值设为 1000 QPS（或 1000 TPS），各节点共享配额
   - 否则超卖、库存服务打穿、DB 行锁等待

2. **"用户级防刷"通常单机限流 + 集群限流组合**
   - 单机：防止单机被刷爆（同 IP/用户 ID 在单机 QPS ≤ 10）
   - 集群：防止用户跨节点刷（全局同用户 QPS ≤ 100）
   - **两层防线**：单机挡住大部分，集群兜底全局

**核心心法**：单机限流是"保护自己"，集群限流是"保护下游与全局资源"。

### 2. 三条实现路径对比

**架构对比图**：

路径C: 网关层限流
┌─────┐ ┌─────┐ ┌─────┐
│C-1  │ │C-2  │ │C-3  │  ← 客户端

| 维度 | Redis+Lua | Sentinel Token Server | 网关层限流 |
| --- | --- | --- | --- |
| 限流精度 | 滑动窗口 / 令牌桶（Lua 实现） | 滑动窗口 + 令牌桶 | 固定窗口 / 滑动窗口 / 令牌桶（看具体网关） |
| 性能瓶颈 | Redis 单节点 QPS（10w 左右），集群分片可扩展 | Token Server 内存 + CPU，单点瓶颈 30w+ QPS | 网关 CPU，水平扩展 |
| 容灾策略 | Redis 挂 → 限流失效，需本地兜底 | Token Server 挂 → Client 退化本地限流 | 网关挂 → 全站不可用（必须高可用部署） |
| 一致性窗口 | 毫秒级（实时） | 秒级（批量拉取有延迟） | 毫秒级（实时） |
| 运维成本 | 中（依赖 Redis 高可用） | 高（Token Server 集群部署 + 控制台） | 低（网关天然有，复用现有基建） |
| 适用场景 | 通用业务限流、秒杀全局总量 | Spring Cloud Alibaba 体系内 | 入口级流量治理、API 总量控制 |

**核心代码（Redis Lua 滑动窗口限流）**：

```lua
-- KEYS[1] = 限流 key
-- ARGV[1] = 当前时间戳（毫秒）
-- ARGV[2] = 窗口大小（毫秒）
-- ARGV[3] = 最大请求数
-- ARGV[4] = 当前请求的唯一 ID

local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local threshold = tonumber(ARGV[3])
local uid = ARGV[4]

-- 1. 移除窗口外的旧请求
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

-- 2. 统计当前窗口内请求数
local count = redis.call('ZCARD', key)

-- 3. 判断是否超阈值
if count >= threshold then
    return 0  -- 拒绝
end

-- 4. 记录本次请求
redis.call('ZADD', key, now, uid)

-- 5. 设置 key 过期时间，避免冷 key 永久占用内存
redis.call('PEXPIRE', key, window)

return 1  -- 放行
```

**原子性保证**：
- Lua 脚本在 Redis 单实例内是**原子执行**的，不会被其他命令插入
- 步骤 1~5 一气呵成，不存在"多个 Client 同时 ZCARD 都通过"的并发漏洞
- Redis Cluster 下，限流 key 必须落在同一个 slot（用 `{}` hashtag 保证，如 `rate_limit:{sku:123}`）

### 3. 高可用陷阱

**Redis 主从切换的几秒钟内**：
- 默认行为是**限流组件不可用 → 业务放行**（fail-open），避免限流故障导致全站不可用
- 但放行可能造成下游打穿，需配置 `fail-close`（拒绝）或本地兜底限流
- **秒杀场景建议**：Redis 故障时切到"本地粗粒度限流"（单机阈值 = 集群阈值 / 节点数），宁可错杀不可放过

**Sentinel Token Server 单点故障**：
- Client 内置**本地 fallback**——Token Server 不可达时退化为本地限流（单机阈值 = 集群阈值 / 节点数）
- 这是 Sentinel 设计的精华点：**牺牲全局精度，保住基本可用**
- 生产环境 Token Server 必须集群部署（3 节点以上），避免单点

**时钟漂移对固定窗口限流的影响**：
- 固定窗口依赖"窗口起始时间"，节点间时钟不一致会导致**两个节点对"当前窗口"理解不同**
- 例：节点 A 时钟 10:00:01，节点 B 时钟 09:59:59，两个节点在不同窗口内各自计数 100，实际全局已 200
- **滑动窗口缓解**：滑动窗口只看"当前时刻往前 N 秒"，与时钟绝对值关系小，对时钟漂移更鲁棒

**网络抖动导致 Lua 脚本超时**：
- **业务侧应降级而非重试**——重试会放大流量，造成 Redis 雪崩
- 推荐策略：Lua 超时 → 触发本地兜底限流 → 短时间放行，等 Redis 恢复
- 设置合理的客户端超时（如 50ms），超时后快速失败

### 4. 全局精确限流的 CAP 取舍

**为什么"全局精确限流"很难**：

CAP 定理在分区容忍（P）的前提下，一致性（C）和可用性（A）必须二选一：

- **强一致（C）**：所有节点实时共享同一份计数器，任何一次限流判断都要同步等待 → 性能极差，且分区时不可用
- **高可用（A）**：节点本地判断，最终汇总 → 性能好，但有"短暂超限"窗口

**秒杀场景的选择**：

```
方案1: 强一致（Redis 同步限流）
  - 每次请求打 Redis，Lua 原子计算
  - 优点：精确
  - 缺点：Redis 是单点瓶颈，QPS 上限 10w

方案2: 最终一致（Token Server 批量拉取）
  - Client 批量预取令牌（如每次拉 100 个），用完再拉
  - 优点：性能高，Token Server 压力小
  - 缺点：可能超发 100*N 个令牌（N=节点数）

方案3: 牺牲精度保可用（本地兜底）
  - 集群限流故障时退化为本地限流
  - 优点：永远可用
  - 缺点：精度下降
```

**我的选择（秒杀场景）**：

**三层组合限流**：
1. **网关层粗粒度限流**：挡住 80% 异常流量，QPS 上限 100w（保护整个入口）
2. **Redis Lua 集群限流**：精确控制全局库存相关流量，QPS 1000~10w（强一致核心）
3. **本地兜底限流**：Redis 故障时启动，单机阈值 = 集群阈值 / 节点数（保可用）

**核心心法**：秒杀不追求"严格不多发一张令牌"，而是追求"在 Redis 挂掉时整个系统不崩"。**精度换可用性**是值得的，前提是有"事后对账"机制（多发的库存事后补扣）。

## 题目二（实战题）：秒杀系统的热点参数限流与舱壁隔离设计

### 1. 热点参数限流方案

**问题剖析**：
- 普通 QPS 限流：用接口维度限流，秒杀接口总 QPS ≤ 10000，但单 SKU 可能独占 9000，其他 SKU 被饿死
- 集群总 QPS 限流：全局阈值挡不住"单 SKU 热点打爆下游缓存"
- **热点参数限流**：以参数值（如 SKU ID）为维度，对 TopN 热点参数单独限流

**Sentinel HotParamSlotChain 实现**：

```
请求进入
   ↓
ParamSlot.entry(skuId)
   ↓
┌──────────────────────────────┐
│ 热点探测（CounterLeaseArray） │
│  - 用 LFU 统计每个 skuId QPS  │
│  - 实时计算 TopN 热点          │
└──────────────────────────────┘
   ↓
判断 skuId 是否在 TopN 热点
   ├─ 是 → 走单参数限流规则（如该 SKU QPS ≤ 1000）
   └─ 否 → 走全局兜底限流规则（如总 QPS ≤ 50000）
   ↓
通过 → 放行 / 超限 → 拒绝
```

**为什么 Sentinel 能"动态识别热点"而不预知**：
1. **滑动窗口 + 参数级统计**：每个参数值在滑窗内单独维护计数器
2. **TopN 实时计算**：每次请求后更新参数排名，TopN 热点动态变化
3. **冷启动优化**：非热点的参数走"共享桶"统计（不浪费内存），进入 TopN 后升级为独立桶

**为什么用 LFU 而不是 LRU**：

| 维度 | LRU（最近最少使用） | LFU（最不经常使用） |
| --- | --- | --- |
| 驱逐策略 | 长时间未访问 | 访问频率最低 |
| 秒杀场景表现 | 长尾 SKU 偶发访问会"挤掉"真热点 | 真热点稳定在 TopN，不被偶发流量干扰 |
| 突发热点响应 | 慢（要先"挤掉"老热点） | 快（频率迅速攀升） |
| 内存效率 | 中 | 高（频率衰减算法可压缩冷数据） |

**秒杀场景下 LFU 的优势**：
- 秒杀 SKU 在活动开始前 5 分钟已经高频访问，LFU 能稳定识别
- LRU 在"多个 SKU 轮流被刷"时会误判热点，LFU 不会
- Sentinel 用的是 **W-TinyLFU**（Caffeine 实现），兼顾内存效率和热点响应速度

**核心代码**：

```java
// 业务方法
@SentinelResource(value = "querySku", blockHandler = "querySkuBlock")
public SkuVO querySku(Long skuId) {
    return skuService.getById(skuId);
}

// 兜底方法
public SkuVO querySkuBlock(Long skuId, BlockException ex) {
    if (ex instanceof ParamFlowException) {
        // 单 SKU 热点限流 → 返回简化版 SKU（仅基础信息）
        return skuCacheService.getSimpleSku(skuId);
    }
    // 全局限流 → 返回"活动太火爆"
    return SkuVO.busy();
}

// 规则配置：单 SKU 限流 1000 QPS，全局 50000 QPS
@PostConstruct
public void initHotParamRule() {
    ParamFlowRule rule = new ParamFlowRule("querySku")
        .setParamIdx(0)              // 第 0 个参数（skuId）
        .setGrade(RuleConstant.FLOW_GRADE_QPS)
        .setCount(1000)              // 单 SKU 默认阈值 1000
        .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_RATE_LIMITER)  // 排队等待
        .setMaxQueueingTimeMs(500);  // 排队超时 500ms

    // 全局兜底：特定热点 SKU 单独配规则
    ParamFlowItem item = new ParamFlowItem();
    item.setObject(String.valueOf(123456L));  // iPhone 17 Pro SKU
    item.setClassType(String.class.getName());
    item.setCount(500);  // 该 SKU 限流 500 QPS（更严格）
    rule.setParamFlowItemList(List.of(item));

    // 全局 QPS 兜底（用普通 FlowRule）
    FlowRule globalRule = new FlowRule("querySku")
        .setCount(50000)
        .setGrade(RuleConstant.FLOW_GRADE_QPS);

    ParamFlowRuleManager.loadRules(List.of(rule));
    FlowRuleManager.loadRules(List.of(globalRule));
}
```

### 2. 舱壁隔离选型

| 下游服务 | 隔离方式 | 线程池/信号量参数 | 选型理由 |
| --- | --- | --- | --- |
| 评论服务 | 信号量隔离 | permits=200 | RT 低（50ms）、QPS 高（5000）、本身轻量，线程池开销不划算 |
| 推荐服务 | 线程池隔离 | core=20, max=50, queue=100, AbortPolicy | RT 中等（200ms）但有慢调用（2s），线程池隔离避免拖垮主链路 |
| 库存服务 | 信号量隔离 | permits=500 | RT 极低（20ms）、强依赖、QPS 高（8000），信号量性能开销小 |
| 第三方风控 | 线程池隔离 | core=10, max=20, queue=10, AbortPolicy | RT 不稳定（P99 5s），必须隔离防止拖垮，且第三方故障不可控 |

**线程池核心参数设计**：

```java
// 推荐服务隔离线程池参数
ThreadPoolExecutor recommendPool = new ThreadPoolExecutor(
    /* corePoolSize    = */ 20,   // 按 Little's Law: QPS * RT = 3000 * 0.2s = 600，但考虑慢调用翻倍 → 队列+线程数应覆盖 600+，core=20 + queue=100 可承载约 600 并发
    /* maxPoolSize     = */ 50,   // 慢调用爆发时扩容，但不超过下游承受能力
    /* keepAliveTime   = */ 60L,  // 秒，空闲线程回收，避免长期占用
    /* unit            = */ TimeUnit.SECONDS,
    /* workQueue       = */ new LinkedBlockingQueue<>(100),  // 有界队列，防 OOM
    /* threadFactory   = */ new NamedThreadFactory("recommend-pool"),
    /* rejectedHandler = */ new ThreadPoolExecutor.AbortPolicy()  // 直接拒绝，触发 fallback
);
```

**参数设计思路**：
- **corePoolSize**：基于 Little's Law（并发数 = QPS × RT），3000 × 0.2 = 600，但考虑慢调用 2s 时并发会暴涨到 3000 × 2 = 6000，线程池不可能容下，所以配合限流+熔断兜底
- **队列长度**：100，缓冲短时抖动，太长会导致 RT 暴涨（请求堆积在队列里）
- **拒绝策略 AbortPolicy**：直接抛 RejectedExecutionException，触发 fallback，**保护主链路**

**为什么信号量不能应对慢调用**：

```
信号量隔离的工作机制：
  请求进入 → tryAcquire(permit) → 调用下游 → release(permit)
                                  ↑
                          如果下游慢，permit 一直被占用
                          信号量池被快速耗尽
                          后续请求全部拒绝
```

- 信号量是**计数器**，不创建线程，**调用是在主线程同步执行的**
- 慢调用会"占住" permit 不释放，导致 permit 池快速耗尽
- 但主线程被阻塞 → **Tomcat 工作线程被拖垮**（这就是信号量隔离的致命弱点）

**信号量适用场景**：
- 下游 RT 极低且稳定（< 50ms）
- 高 QPS 但不需要异步化
- 不想引入线程切换开销（如纯内存操作、本地缓存查询）

**线程池隔离的优势**：
- 调用是在**独立线程**执行的，主线程不阻塞
- 慢调用只拖垮自己的线程池，不影响主链路
- 代价：线程切换开销（约 1~10μs），不适合极高 QPS

**选型决策树**：

```
下游 RT 稳定吗？
├─ 不稳定（P99 > 1s）→ 线程池隔离
└─ 稳定
    └─ RT 高吗（> 100ms）？
        ├─ 高 → 线程池隔离（避免主线程阻塞）
        └─ 低 → 信号量隔离（性能优先）
```

### 3. 舱壁 + 熔断 + 降级协同

**触发顺序设计**：

```
请求进入
   ↓
调用推荐服务（线程池隔离）
   ↓
┌──────────────────────────────┐
│ Stage 1: 线程池未满           │
│  正常调用，统计 RT 与异常率   │
└──────────────────────────────┘
   ↓ 线程池满
┌──────────────────────────────┐
│ Stage 2: 线程池满 → 排队      │
│  进队列等待，超过 100ms 拒绝  │
│  此时不触发熔断（短时抖动）   │
└──────────────────────────────┘
   ↓ 排队持续超时
┌──────────────────────────────┐
│ Stage 3: 慢调用比例 > 60%    │
│  触发熔断（OPEN 状态）        │
│  快速失败，走 fallback        │
└──────────────────────────────┘
   ↓ 熔断冷却期结束
┌──────────────────────────────┐
│ Stage 4: HALF_OPEN 探测       │
│  放行 N 个请求，全成功 → CLOSED│
│  任一失败 → 重新 OPEN         │
└──────────────────────────────┘
```

**关键设计原则**：
- **先排队后熔断**：短时抖动用排队缓冲，持续异常才熔断
- **熔断优先于拒绝**：熔断是"快速失败 + fallback"，比"线程池拒绝 + 异常"对业务更友好
- **HALF_OPEN 探测必须受控**：探测请求数严格控制（如 5 个），避免探测期二次雪崩

**拒绝策略为什么选 AbortPolicy**：

| 策略 | 行为 | 秒杀场景风险 |
| --- | --- | --- |
| `AbortPolicy` | 抛 RejectedExecutionException | 触发 fallback，可控 |
| `CallerRunsPolicy` | 在调用者线程执行 | **致命**——主线程被阻塞，Tomcat 线程池被拖垮，整个应用不可用 |
| `DiscardPolicy` | 静默丢弃 | 用户无感知，体验差 |
| `DiscardOldestPolicy` | 丢弃队列最老请求 | 旧请求被丢弃，可能造成数据不一致 |

**CallerRunsPolicy 在秒杀场景的危险**：
- 推荐服务慢 → 线程池满 → CallerRuns 让 Tomcat 主线程同步调推荐服务
- 推荐服务 RT 2s → Tomcat 主线程阻塞 2s
- Tomcat 工作线程数有限（默认 200），很快全部被阻塞
- **整个应用不可用，包括下单、支付等核心链路**——这就是"舱壁失效导致全盘崩溃"

**正确的协同**：
- 线程池满 → AbortPolicy → 抛异常 → @SentinelResource fallback → 返回兜底数据
- 同时熔断器统计异常率，达到阈值熔断 → 后续请求不再进入线程池，直接快速失败

**隔离后的兜底数据设计**：

| 业务场景 | 兜底数据 |
| --- | --- |
| 商品详情页 - 推荐位 | 返回默认热门商品列表（运营预配置） |
| 商品详情页 - 评论数 | 显示 "--" 或 "查看更多" |
| 商品详情页 - 价格 | 返回最近一次缓存价格（标注"仅供参考"） |
| 商品详情页 - 库存 | 返回"有货"（保守兜底，避免漏单） |
| 下单 - 推荐凑单 | 跳过推荐，直接结算 |

**核心原则**：
- **强一致数据不能假成功**（库存、价格）→ 返回"稍后重试"或保守兜底
- **弱依赖数据可降级**（推荐、评论）→ 返回默认值或空态
- **永远标注"降级状态"** → 前端展示"部分数据加载中"，避免用户误判

### 4. 真实案例：平安健康险商城 - 第三方风控接口拖垮投保流程

**故障链路推演**：

```
2024 年某次活动，投保流程中调用第三方风控接口
   ↓
第三方风控服务因内部原因 RT 从 200ms 飙升到 8s
   ↓
投保接口调用风控接口（同步调用，无隔离）
   ↓
Tomcat 工作线程（200 个）被风控调用占满
   ↓
新进来的投保请求拿不到线程，排队等待
   ↓
Tomcat acceptQueue 满拒绝连接
   ↓
上游网关 5xx，触发网关限流
   ↓
下游订单服务、支付服务也因调用投保服务超时
   ↓
整个交易链路雪崩，持续 12 分钟
   ↓
事后复盘：1 个第三方接口慢，导致全线不可用
```

**改造方案**：

1. **线程池隔离**：为风控接口单独配置线程池（core=10, max=20, queue=10, AbortPolicy）
2. **熔断降级**：Sentinel 配置慢调用熔断（RT > 1s 占比 > 50% 持续 10s → OPEN）
3. **fallback 设计**：风控失败 → 走"轻量健康告知 + 人工复核"流程（参考 Day02 案例）
4. **超时控制**：风控接口客户端超时 800ms，超时即失败，不等待
5. **舱壁 + 熔断联动**：线程池拒绝 + 熔断快速失败，双重保护

**改造后效果验证**：

```text
压测场景：模拟风控接口 RT 飙升到 8s，持续 5 分钟

改造前：
  - 投保接口 P99 RT：30s+
  - 应用 Tomcat 线程池：100% 占用
  - 整体可用性：5xx 错误率 80%
  - 影响范围：全交易链路不可用

改造后：
  - 风控线程池：3s 内打满 → 触发熔断（5s 进入 OPEN）
  - 投保接口 P99 RT：< 500ms（直接走 fallback）
  - 应用 Tomcat 线程池：< 30% 占用
  - 整体可用性：99.9%（仅风控降级订单走人工复核）
  - 影响范围：仅风控相关订单，核心投保链路无感知
```

**踩坑经验**：
- 第一次改造时漏配**超时时间**，线程池隔离了但调用方仍同步等待 8s，没用
- 第二次加了超时，但**未配熔断**，超时后所有请求都报错，没快速失败机制
- 第三次完整方案：超时 + 线程池 + 熔断 + fallback 四件套，才彻底解决
- **教训**：舱壁隔离只是"切传播路径"，必须配合超时、熔断、降级才能形成完整防线

### 与架构师水平的差距及补足方向

**差距**：
1. **舱壁隔离经验单薄**——业务里只有 Hystrix/Sentinel 默认配置，没系统化设计"四个下游分别用什么隔离"
2. **热点参数限流未用过**——秒杀场景只用了普通限流，没针对单 SKU 做热点治理，曾出现"爆款 SKU 打爆缓存"
3. **拒绝策略选型意识不足**——默认用 CallerRunsPolicy（怕丢请求），反而在故障时把主链路拖垮
4. **熔断 + 隔离 + 降级联动不熟**——三件套是分开配的，没形成"超时 → 隔离 → 熔断 → 降级"的完整链路
5. **压测验证不足**——改造后没做故障注入压测，直到下一次故障才发现问题

**补足方向**：
- 在秒杀系统落地"四个下游各配独立线程池 + 熔断规则 + fallback"的完整方案
