# Day 6：MQ主题串联整合 — 从单点到系统
## 场景一：蛋壳钱包的转账链路 — 金额敏感场景的MQ设计
### 场景描述
```plain
用户A → 转账100元 → 用户B

涉及环节：
  1. 扣A的余额
  2. 加B的余额
  3. 记录转账流水
  4. 发送通知

要求：
  - 金额不能错（绝对不能少扣、多扣、重复扣）
  - 通知可以晚一点，但不能丢
  - 系统挂了也不能丢数据
```

### 架构师方案：TCC + MQ最终一致性
```plain
┌─────────────────────────────────────────────────────────────┐
│                    TCC（确保金额准确）                        │
├─────────────────────────────────────────────────────────────┤
│  Try：    冻结A的100元，记录转账流水（状态=处理中）          │
│  Confirm：扣除A的冻结金额，增加B的余额，状态=成功            │
│  Cancel：  解冻A的100元，状态=失败                          │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                  MQ（确保通知可靠）                          │
├─────────────────────────────────────────────────────────────┤
│  TCC成功后 → 发送"转账成功"消息到MQ                          │
│  消费端 → 发送短信/推送通知 → 幂等处理                        │
│  消费失败 → 重试16次 → 死信队列 → 人工补发                   │
└─────────────────────────────────────────────────────────────┘
```

### 本周知识点串联
| 知识点 | 在这个场景中的应用 |
| --- | --- |
| **消息不丢（Day1）** | MQ同步发送+同步刷盘+同步复制，确保通知消息不丢 |
| **消息幂等（Day1）** | 通知消费幂等，重复消息不会重复发送通知 |
| **MQ选型（Day2）** | 选RocketMQ：因为有内置重试+死信，可靠性高 |
| **事务消息（Day2）** | TCC成功后才发送MQ，或者用事务消息确保原子性 |
| **重试/死信（Day4）** | 通知消费失败重试16次，进死信人工处理 |


### 架构师思考：为什么不用MQ做金额流转？
```plain
MQ适合做通知、流水记录，但不适合做金额流转：

原因：
  1. MQ有延迟，用户转账后需要立即看到结果
  2. MQ消息可能重复，金额流转的幂等很难做对
  3. 金额是核心数据，应该用DB事务/TCC来保证

架构师决策：
  核心数据（金额）→ TCC/DB事务，强一致
  非核心数据（通知）→ MQ，最终一致
  → 职责分离，各自保证自己的可靠性
```

## 场景二：保险商城的投保链路 — 长流程异步化
### 场景描述
```plain
用户投保 → 流程：
  1. 验证投保信息
  2. 调用保费计算接口
  3. 调用核保接口
  4. 创建保单
  5. 发送保单邮件
  6. 发送短信通知

问题：
  流程太长，同步处理需要3-5秒，用户体验差
  某个环节（比如发邮件）慢了，整个流程都慢
```

### 架构师方案：流程拆分 + MQ异步解耦
```plain
同步部分（快速返回）：
  1. 验证投保信息
  2. 调用保费计算接口
  3. 创建投保记录（状态=处理中）
  4. 发送"投保中"消息到MQ
  5. 返回用户"投保处理中，请等待邮件通知"

异步部分（后台慢慢处理）：
  ┌───────────────────────────────────────────────────┐
  │ Topic：INSURANCE_APPLY                            │
  │ ↓                                                 │
  │ 消费端1：调用核保接口 → 成功继续，失败重试        │
  │ ↓                                                 │
  │ Topic：INSURANCE_UNDERWRITE_SUCCESS               │
  │ ↓                                                 │
  │ 消费端2：创建保单 → 发送"保单创建成功"消息        │
  │ ↓                                                 │
  │ Topic：INSURANCE_POLICY_CREATED                   │
  │ ↓                                                 │
  │ 消费端3：发送保单邮件                              │
  │ ↓                                                 │
  │ 消费端4：发送短信通知                              │
  └───────────────────────────────────────────────────┘
```

### 本周知识点串联
| 知识点 | 在这个场景中的应用 |
| --- | --- |
| **异步解耦（Day1）** | 长流程拆分成多个环节，用MQ串联，用户体验好 |
| **消息顺序（Day1）** | 同一投保单的消息发到同一Queue，确保顺序 |
| **延迟消息（Day3）** | 核保接口超时 → 延迟消息稍后重试 |
| **消费线程模型（Day4）** | 不同环节可以用不同的消费线程数 |
| **积压处理（Day4）** | 投保高峰期 → 监控积压 → 扩容消费端 |


### 架构师思考：流程拆分的粒度怎么定？
```plain
拆得太细：
  - Topic太多，复杂度高
  - 维护成本高

拆得太粗：
  - 还是同步，没有解耦
  - 一个环节慢了还是影响其他

架构师原则：
  1. 按业务领域拆分（核保、保单、通知是不同领域）
  2. 按是否需要同步返回拆分（需要快速返回的放前面）
  3. 按失败影响拆分（一个环节失败不应该影响其他）
  4. 按调用第三方拆分（调用外部接口的环节独立出来）
```

## 场景三：IM客服的消息推送 — 海量消息场景
### 场景描述
```plain
医生给患者发消息 → 需要推送给患者的App/小程序

特点：
  - 消息量巨大（百万级/天）
  - 推送可以延迟，但不能丢
  - 顺序不重要（但同一对话的消息最好按顺序）
  - 有些用户在线，有些用户离线

问题：
  直接调用推送接口 → 高峰期推送接口扛不住
  推送失败 → 用户收不到消息
```

### 架构师方案：MQ缓冲 + 批量推送 + 离线消息
```plain
┌─────────────────────────────────────────────────────────────┐
│                    医生发消息                                 │
│                      ↓                                       │
│              写入DB（主存储）                                │
│                      ↓                                       │
│              发送到MQ（推送缓冲）                            │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                  消费端批量推送                               │
├─────────────────────────────────────────────────────────────┤
│  1. 批量拉取消息（比如一次拉100条）                          │
│  2. 判断用户是否在线                                         │
│  3. 在线 → 批量调用推送接口                                  │
│  4. 离线 → 标记为离线消息，稍后推送                          │
│  5. 推送成功 → ACK                                           │
│  6. 推送失败 → 重试 → 死信队列                               │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                  离线消息补推                                 │
├─────────────────────────────────────────────────────────────┤
│  定时任务扫描离线消息表 → 用户上线时批量推送                  │
│  或用延迟消息：用户上线触发推送                               │
└─────────────────────────────────────────────────────────────┘
```

### 本周知识点串联
| 知识点 | 在这个场景中的应用 |
| --- | --- |
| **消息积压（Day1）** | MQ做缓冲，高峰期百万消息不怕积压 |
| **批量消费（Day4）** | 消费端批量拉取，批量推送，提高吞吐量 |
| **消费线程模型（Day4）** | 消费线程数调大，提高推送速度 |
| **集群消费（Day3）** | 多个消费端同时消费，提高吞吐量 |
| **消息回溯（Day4）** | 推送系统bug → 回溯重新推送 |
| **监控告警（Day5）** | 监控推送失败率、积压量，及时发现问题 |


### 架构师思考：离线消息怎么处理？
```plain
方案1：延迟消息
  用户离线 → 发延迟消息，30分钟后检查用户是否上线
  → 优点：实时性好
  → 缺点：消息量大时延迟消息很多

方案2：定时任务
  用户离线 → 标记为离线
  定时任务每5分钟扫描离线消息 → 推送
  → 优点：简单，可控
  → 缺点：实时性稍差

方案3：用户上线触发
  用户上线 → 查询是否有离线消息 → 推送
  → 优点：精确，不浪费
  → 缺点：需要端上配合

架构师决策：
  方案2 + 方案3结合
  定时任务兜底 + 上线触发实时推送
```

## 场景四：国美e卡资金路由 — 顺序消费场景
### 场景描述
```plain
用户使用e卡消费 → 资金路由系统：
  1. 扣减e卡余额
  2. 记录资金流水
  3. 通知账务系统

要求：
  - 同一e卡的消费必须按顺序处理（不能先扣100元，再扣50元，结果50元先处理了）
  - 流水不能丢
  - 余额不能错
```

### 架构师方案：MessageQueueSelector + 顺序消费
```plain
发送端：
  用MessageQueueSelector，同一e卡ID的消息发到同一Queue
  e卡A的消息 → Queue 0
  e卡B的消息 → Queue 1
  e卡C的消息 → Queue 2
  ...

消费端：
  用顺序消费（MessageListenerOrderly）
  同一Queue的消息，单线程按顺序消费
  消费失败 → 挂起当前Queue，不断重试（直到成功或跳过）

兜底：
  死信队列 → 人工介入
  对账脚本 → 定时核对e卡余额和流水
```

### 本周知识点串联
| 知识点 | 在这个场景中的应用 |
| --- | --- |
| **消息顺序（Day1）** | 同一e卡的消息发到同一Queue + 顺序消费 |
| **顺序消费陷阱（Day3）** | 消费失败会挂起当前Queue → 需要监控 |
| **幂等设计（Day1）** | 顺序消费可能重复 → 幂等保证 |
| **消息不丢（Day1）** | 同步刷盘+同步复制，确保消息不丢 |
| **重试/死信（Day4）** | 消费失败重试 → 死信队列人工处理 |


### 架构师思考：顺序消费的代价是什么？
```plain
顺序消费的代价：
  1. 同一Queue单线程消费 → 吞吐量低
  2. 消费失败挂起当前Queue → 可能积压
  3. Rebalance更复杂 → 可能导致重复消费

什么时候用顺序消费？
  - 业务必须严格顺序（比如资金流水）
  - 消息量不是特别大
  - 接受吞吐量降低

什么时候不用顺序消费？
  - 消息量巨大 → 用并发消费，业务层保证顺序
  - 顺序要求不高 → 用并发消费，性能优先

架构师权衡：
  顺序 vs 性能 → 看业务优先级
  资金场景 → 顺序优先
  通知场景 → 性能优先
```

## 架构师总结：MQ的"五维决策模型"
### 维度一：可靠性 vs 性能
```plain
高可靠性方案：
  同步发送 + 同步刷盘 + 同步复制
  TPS：4-6万
  适用：金额、订单等核心数据

高性能方案：
  异步发送 + 异步刷盘 + 异步复制
  TPS：8-12万
  适用：日志、通知等非核心数据

架构师决策：
  按业务重要性选择
  可以不同Topic用不同配置
```

### 维度二：同步 vs 异步
```plain
同步处理：
  优点：实时性好，用户立即知道结果
  缺点：用户等待时间长，一个环节慢全慢
  适用：需要立即知道结果的场景（比如登录、查询）

异步处理：
  优点：用户体验好，解耦，削峰
  缺点：实时性稍差，复杂度高
  适用：长流程、高并发场景（比如下单、投保）

架构师决策：
  核心查询同步，非核心流程异步
  同步快速返回，异步后台处理
```

### 维度三：顺序 vs 并发
```plain
顺序消费：
  优点：严格顺序
  缺点：吞吐量低，失败挂起
  适用：资金流水、状态机等

并发消费：
  优点：吞吐量高
  缺点：顺序不保证
  适用：通知、日志等

架构师决策：
  必须顺序的 → 顺序消费
  可以乱序的 → 并发消费
  业务层保证顺序的 → 并发消费
```

### 维度四：重试策略
```plain
快速重试（间隔短，次数少）：
  适用：临时故障（网络抖动、下游限流）
  比如：前3次间隔10s/30s/1min

慢速重试（间隔长，次数多）：
  适用：需要时间恢复的故障（DB恢复、服务重启）
  比如：后面13次间隔递增到2h

不重试直接死信：
  适用：数据错误（参数不对、订单不存在）
  比如：金额场景，不能无限重试

架构师决策：
  区分故障类型 → 不同类型用不同策略
```

### 维度五：兜底方案
```plain
L1：重试
  自动重试，大多数问题在这一层解决

L2：死信队列
  重试失败 → 死信队列 → 人工介入

L3：对账脚本
  定时核对数据 → 发现不一致 → 修复

L4：人工介入
  兜底方案，最后一道防线

架构师原则：
  任何环节都可能失败
  必须有多层兜底
```

## 面试时怎么串起来说？
### 面试官问："说说你在项目中怎么用MQ的？"
#### 普通回答
```plain
"我在项目中用RocketMQ发送消息，消费端处理消息。"
→ 太简单，不像架构师
```

#### 架构师回答
```plain
"我在项目中负责MQ的架构设计和运维，我从几个维度来说：

1. 选型：我们选了RocketMQ，因为它有内置重试+死信，可靠性高，适合我们的金融场景。

2. 可靠性保证：核心消息用同步发送+同步刷盘+同步复制，非核心消息用异步发送+异步刷盘。

3. 幂等设计：我们用状态机+分布式锁+唯一键三层幂等保证，重复消息不会影响业务。

4. 重试策略：我们区分临时故障和数据错误，临时故障快速重试，数据错误直接死信。

5. 兜底方案：我们有死信队列+对账脚本+人工介入三层兜底，确保不丢数据。

6. 监控运维：我们监控TPS、积压量、消费延迟、失败率，有完整的告警阈值和应急SOP。

举个具体例子：我们的秒杀系统，用MQ做削峰填谷，网关限流+Redis预扣拦截99%的请求，通过的请求写入MQ，消费端5000 TPS慢慢处理，幂等用状态机+分布式锁+唯一键，兜底用16次重试+死信队列+对账脚本，监控告警和应急SOP都很完善。"

→ 这样说，既有方法论，又有具体例子，架构师范儿就出来了！
```
