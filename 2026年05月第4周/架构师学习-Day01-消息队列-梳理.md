# Day 1：消息队列 — 架构师角度梳理

## 核心定位

消息队列是分布式系统的**异步通信基础设施**，解决的核心问题是：**如何让不同速度的组件协作而不互相拖垮**。

---

## 架构师视角：为什么不是RPC？

| 维度 | RPC（同步） | MQ（异步） |
|------|-----------|-----------|
| **调用方式** | 实时等待响应 | 发后即忘，消费者按节奏处理 |
| **耦合度** | 强耦合（上下游必须同时在线） | 松耦合（上游不关心下游状态） |
| **流量控制** | 下游扛不住就超时 | 天然削峰，下游按能力消费 |
| **一致性** | 强一致（调用成功即确认） | 最终一致（消息投递 → 异步处理） |
| **适用场景** | 查询、实时业务 | 削峰、解耦、异步通知 |

**架构师决策**：能同步就同步，需要异步才用MQ。MQ引入的复杂度（消息丢失、重复、顺序、积压）是真实成本。

---

## 三大场景的架构本质

### 1. 削峰填谷 = 流量整形

```
本质：用时间换空间——把瞬时流量平摊到更长时间窗口

秒杀场景：
  10:00:00 → 10万QPS瞬间涌入
  MQ缓冲 → 订单系统以1000 QPS稳定消费
  用户侧 → 几秒延迟换取系统不崩溃

架构决策：
  - 队列深度 = 峰值流量 × 可接受延迟
  - 消费者数量 = 下游系统承受能力
  - 积压告警阈值 = 队列深度 × 70%
```

### 2. 系统解耦 = 依赖反转

```
本质：上游不直接依赖下游，通过消息中间解耦

没有MQ：
  秒杀服务 → 订单服务 → 库存服务 → 积分服务 → 通知服务
  任何下游挂了，整个链路失败

有MQ：
  秒杀服务 → MQ → [订单、库存、积分、通知各自消费]
  下游挂了不影响秒杀，恢复后继续消费

架构决策：
  - 新增下游服务无需改动上游代码
  - 但要考虑：消息Schema演进、消费者上下线管理
```

### 3. 最终一致性 = 放弃强一致，换取可用性和性能

```
本质：与上周学的分布式事务形成对照

TCC（强一致）：  预留 → 确认/取消，保证一致性但性能差
MQ（最终一致）：  本地事务 + 发消息 → 异步处理，性能好但有窗口期不一致

你的钱包系统选择TCC是对的（金额不能错）
秒杀库存可以用MQ最终一致（库存超卖可通过对账修正）
```

## 全链路可靠性架构

```
                     可靠性保障层级
┌─────────────────────────────────────────────┐
│  L1: 生产端确认   同步发送 + 重试 + 幂等key   │
├─────────────────────────────────────────────┤
│  L2: Broker持久化  同步刷盘 + 同步双写        │
├─────────────────────────────────────────────┤
│  L3: 消费端保障   手动ACK + 幂等消费         │
├─────────────────────────────────────────────┤
│  L4: 补偿机制     定时对账 + 死信队列 + 告警   │
└─────────────────────────────────────────────┘

层级越往上越基础，越往下越兜底。
架构师要确保每一层都有保障，而不是只依赖某一层。
```

---

## 顺序性的架构权衡

```
全局有序：1个Partition → 1个消费者 → 吞吐量极低 → 基本不用
分区有序：N个Partition → N个消费者 → 吞吐量高 → 工程主流

架构决策关键：
  - 同一业务实体（订单ID）的消息路由到同一分区
  - 路由key的选择决定顺序粒度
  - 消费失败时必须阻塞当前分区，不能跳过
```

---

## 幂等性：不是可选项，是必选项

```
架构师认知：消息重复是常态，不是异常

幂等方案选择决策树：
  业务有状态机？ ──是──→ 用状态机保证幂等（零成本）
       │
       否
       │
  并发量高？ ──是──→ Redis SETNX + 异步对账
       │
       否
       │
  需要审计追踪？ ──是──→ 数据库唯一索引
       │
       否
       │
  → 乐观锁/版本号
```

## 消息积压：从运维角度设计

```
预防层面：
  - 监控：队列深度、消费延迟、消费TPS
  - 告警：积压量超阈值、消费延迟超阈值
  - 容量：按峰值2-3倍规划消费者数量

应急层面：
  - 扩容（横向加消费者）
  - 降级（跳过非关键消息）
  - 转发（新Topic + 更多消费者）

根因层面：
  - 消费逻辑优化（批量、异步）
  - 下游依赖治理（超时、重试）
  - 容量规划重新评估
```
