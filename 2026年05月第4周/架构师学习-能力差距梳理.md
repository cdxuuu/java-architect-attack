# 架构师面试能力差距梳理

## 当前定位：Java高级开发 → 目标：架构师

### 在MQ主题上的表现基准：85/100
- 知识储备够，实战经验是核心优势
- 差距不在"会不会"，在"深不深"和"全不全"

---

## 差距一：数字直觉不足

### 现状
知道同步刷盘比异步慢，但说不出具体慢多少；知道重试有次数，但说不出RocketMQ默认16次。

### 架构师为什么需要数字
数字体现的不是记忆力，是**用过多深**。面试官听到"同步刷盘吞吐量大概降30-50%"和"同步刷盘会更慢"，前者是架构师，后者是开发。

### 需要积累的数字清单

| 类别 | 关键数字 | 来源 |
|------|---------|------|
| MQ性能 | ASYNC_FLUSH ~8-12万TPS，SYNC_FLUSH ~4-6万TPS | 压测/官方文档 |
| MQ延迟 | 异步刷盘亚毫秒，同步刷盘毫秒级 | 压测 |
| MQ配置 | RocketMQ默认重试16次后进死信队列 | 官方文档 |
| MQ配置 | 投票超时150-300ms（Raft），选举多数票 | 源码/文档 |
| 磁盘IO | HDD fsync 5-10ms，SSD 0.5-2ms，NVMe 0.1-0.5ms | 系统知识 |
| Redis | SETNX单节点10万+QPS，集群可达数十万 | 压测 |
| MySQL | 单机写QPS约3000-5000（SSD），唯一索引更低 | 压测 |
| 网络 | 同机房RTT 0.1-0.5ms，跨机房1-5ms，跨域10-100ms | 基础知识 |

### 怎么补
1. **每次练完一个主题，把涉及的关键数字整理到笔记里**，不用背，写一遍就有印象
2. **自己压测一遍**：本地搭一套RocketMQ，用官方benchmark跑一遍ASYNC/SYNC的TPS差距，数字就变成你的了
3. **建立数字量级感**：不需要精确到个位，但要能说出"万级""十万级""毫秒级"这种量级

## 差距二：分层防御思维不足

### 现状
回答偏"选一个方案"，比如幂等选Redis还是数据库，会做单点选择，但不会组合使用。

### 架构师为什么需要分层
生产环境没有银弹，任何单层方案都有失效可能。架构师的思维是**每一层都可能有漏洞，所以要叠多层，层层兜底**。

### 对比

| 场景 | 高级开发答法 | 架构师答法 |
|------|------------|-----------|
| 幂等 | "用Redis SETNX" | "第一层Redis挡99%，第二层状态机兜底，第三层对账脚本巡检" |
| 消息不丢 | "同步发送+同步刷盘" | "L1生产端确认，L2 Broker持久化，L3消费端手动ACK，L4死信队列+对账补偿" |
| 顺序消费失败 | "重试+死信队列" | "重试N次→失败落库→放行后续消息→Job按ID维度补偿" |

### 怎么练
1. **每答完一道题，追问自己：如果这一层失效了怎么办？** 然后补一层
2. **面试时主动加一句**："当然这是第一道防线，如果这一层出了问题，我们还有XX兜底"，面试官会眼前一亮
3. **画架构图时养成习惯**：不是画一条线，而是画多条线（主路径+降级路径+补偿路径）

---

## 差距三：反向论证能力不足

### 现状
擅长说"用什么"，不擅长说"不用什么"和"为什么不用"。

### 架构师为什么需要反向论证
架构师的核心工作是**做选择**，做选择不只是选A，更是**排除B和C**。面试官判断你是不是架构师，听的不是你选了什么，而是你排除什么。

### 典型场景

| 题目 | 正向回答 | 反向论证（加分项） |
|------|---------|-----------------|
| 为什么用MQ | 削峰解耦 | 什么时候**不用**MQ？调用是同步查询场景、消息量极小、实时性要求亚秒级 |
| 幂等选Redis | 性能高 | 为什么**不全靠**Redis？缓存丢失时幂等失效，必须有业务层兜底 |
| 同步刷盘 | 不丢消息 | 什么时候**不用**同步刷盘？日志场景丢几条无所谓，SYNC_FLUSH白白损失50%吞吐量 |
| TCC vs MQ最终一致 | 各有适用场景 | 你的钱包系统选TCC对，因为金额不能错；但如果用MQ，哪个环节可能出问题？ |

### 怎么练
1. **每答完一道题，加一个自问：为什么不选另一个方案？** 强迫自己练反面论证
2. **建立"反模式"清单**：每个最佳实践对应一个"什么时候不该用"的场景
3. **面试话术模板**："我选X是因为Y，但不选Z是因为Z在XX场景下有YY问题"

## 差距四：全局串联能力不足

### 现状
单题答得好，但5道题之间是独立的，没有形成体系。

### 架构师为什么需要全局视角
面试不只是5道独立题，面试官在评估你能不能**站在系统层面做决策**。如果5道题的答案之间互相矛盾（比如Q2说要同步刷盘保可靠，Q5又说吞吐不够要扩容），面试官会认为你在背答案。

### MQ五道题的体系关系

```
消息不丢（Q2）←→ 消息重复（Q4）
    │                  │
    │  同步发送保证不丢，但可能重发 → 必须幂等
    │                  │
    ↓                  ↓
消息顺序（Q3）←→ 消息积压（Q5）
    │                  │
    │  顺序消费要单线程 → 消费速度慢 → 容易积压
    │                  │
    │  积压时要扩容 → Queue:Consumer=1:1 → 顺序限制了扩容
    │                  │
    ↓                  ↓
    这四个问题互相牵制，架构师要做的是在它们之间找平衡点，
    而不是对每个问题独立求最优解
```

### 怎么练
1. **周六串联日重点练这个**：不是简单列关联，而是画出主题之间的矛盾和权衡
2. **每学完一个新主题，回头画一张与已学主题的关系图**
3. **面试时如果5道题都答完了，主动说一句**："其实这5个问题不是独立的，比如消息不丢和消息重复是同一件事的两面"，直接拉高面试官的评估档次

---

## 每日练习改进计划

### 答题时的自查清单

每答完一道题，过一遍这个清单：

- [ ] **数字**：有没有可以量化的地方？TPS、延迟、重试次数、超时时间
- [ ] **分层**：如果当前方案失效了，下一层兜底是什么？
- [ ] **反向**：为什么不选另一个方案？什么场景不该用这个方案？
- [ ] **串联**：这道题和之前的主题有什么关联或矛盾？

### 一句话总结

**你的知识储备和实战经验已经是架构师水平，缺的是表达架构师的思维方式：数字感、分层防御、反向论证、全局串联。这不是多学几个主题能补的，而是每答一道题都要刻意练习的思维习惯。**

## 差距五：延迟消息原理不深入

### 现状
知道RocketMQ有18个延迟级别，但说不清内部SCHEDULE_TOPIC转换机制，不知道5.x时间轮改进。

### 架构师为什么需要理解原理
延迟消息不只是"设个级别"——理解SCHEDULE_TOPIC的流转才能排查"延迟消息没按时投递"的线上问题，理解时间轮才能评估5.x升级的必要性。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | setDelayTimeLevel(14)就行 |
| 高级开发 | 内部有个SCHEDULE_TOPIC，定时线程转投 |
| 架构师 | 延迟消息是时间维度的路由。旧版18个时间桶，5.x时间轮O(1)入出。选方案时按量级、精度、可靠性三维度决策 |

### 怎么补
1. 本地搭建RocketMQ，发送延迟消息后观察SCHEDULE_TOPIC_XXXX的Queue变化
2. 画一遍完整流转图：生产→SCHEDULE_TOPIC→定时线程→目标Topic→消费
3. 对比Redis ZSET方案，理解"为什么要用原生功能而不是自研"

## 差距六：Rebalance影响低估

### 现状
知道Rebalance是"重新分配Queue"，但没意识到会暂停消费、导致重复消费、可能引发风暴。

### 架构师为什么需要重视
线上Consumer宕机是常态，Rebalance是高频事件。不理解的后果：高峰期扩容触发Rebalance → 全组暂停 → 积压暴增 → 雪崩。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | Consumer数量变了就Rebalance |
| 高级开发 | Rebalance会短暂暂停消费 |
| 架构师 | Rebalance三大问题（暂停/重复/风暴），三层防御（预防/缓解/补偿），扩缩容要错峰 |

### 怎么补
1. 本地模拟Consumer宕机，观察Rebalance期间消费暂停的时间窗口
2. 练习回答"秒杀高峰期Consumer挂了怎么办"——Rebalance + 幂等 + 转接方案
3. 监控Rebalance频率作为核心指标，>1次/分钟需要排查

## 差距七：重试与死信的完整链路不熟

### 现状
知道"消费失败会重试"，但说不出RocketMQ默认16次、间隔递增规则、死信Topic命名、死信处理方式。

### 架构师为什么需要
线上消费失败是家常便饭，架构师要能快速判断"这条消息该不该重试、重试几次、什么时候进死信、进了死信怎么处理"。答不出16次重试间隔，面试官会认为你没深度使用过。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 失败了就重试 |
| 高级开发 | 有重试机制，失败多了进死信 |
| 架构师 | 16次递增重试（10s→2h），不同失败原因用不同策略（临时故障重试、数据错误直进死信），死信队列要有监控+自动告警+可操作后台 |

### 怎么补
1. 背16次重试间隔表，重点记第1次10s、第16次2h、总耗时~4h47min
2. 写一个订阅%DLQ%ConsumerGroup的Consumer，熟悉死信消费代码
3. 练习回答"金额场景重试策略和通知场景有什么区别"

## 差距八：积压应急没有结构化方法

### 现状
知道积压可以扩容，但想到什么做什么，没有SOP，也没考虑扩容的副作用（Rebalance）。

### 架构师为什么需要
线上积压是P0事件，架构师要在5分钟内做出正确决策。没有SOP = 慌乱中可能做错（比如Queue不够时盲目扩Consumer、积压时扩容触发Rebalance反而更慢）。

### 对比

| 层次 | 回答 |
|------|------|
| 开发 | 加机器 |
| 高级开发 | 扩Consumer，但Queue不够扩不了 |
| 架构师 | 三板斧（确认原因→止血→加速），Queue不够时临时Topic转发，消息分级跳过非核心，关键：扩容触发Rebalance要提前不要临时扩 |

### 怎么补
1. 记住应急SOP：0-1分钟确认积压量 → 1-3分钟定位原因 → 3-5分钟止血 → 5-15分钟加速消费
2. 练习画临时Topic转发方案的架构图
3. 在简历项目里主动提"设计过积压应急预案和消息分级策略"

## 差距九：回溯和消费线程模型知识空白

### 现状
不知道消息回溯怎么操作、回溯有什么坑；不清楚consumeThreadMin/Max和ProcessQueue的关系。

### 架构师为什么需要
回溯是"代码bug导致批量错误"的唯一修复手段，不会回溯=数据错了没法修。线程模型不熟=无法调优消费性能。

### 怎么补
1. 练习在RocketMQ Dashboard上做一次消息回溯操作
2. 背关键默认值：consumeThreadMin=20, consumeThreadMax=64, pullThresholdForQueue=1000
3. 画消费端线程模型图：Pull线程→ProcessQueue→消费线程池→业务处理
