# Day 2：MQ选型与高可用 — 架构师角度梳理

## 核心定位

MQ选型不是"哪个更好"的技术PK，而是**"我的业务最不能容忍什么"的反向排除**。

---

## 选型的架构本质

```
选MQ = 回答三个问题：

1. 我的业务最不能容忍什么？
   - 丢消息？→ RocketMQ（同步刷盘+同步双写）
   - 吞吐不够？→ Kafka（10万+TPS）
   - 功能缺失？→ RocketMQ（事务/延迟/死信）
   - 运维复杂？→ RabbitMQ（管理界面友好）

2. 我的团队能驾驭什么？
   - Java团队 → RocketMQ（源码可读，二次开发）
   - 大数据团队 → Kafka（生态整合）
   - 运维薄弱 → RabbitMQ（开箱即用）

3. 我的业务会往哪演进？
   - 数据量会爆发 → Kafka（水平扩展能力强）
   - 业务链路会复杂 → RocketMQ（事务/延迟/回溯）
   - 系统集成为主 → RabbitMQ（路由灵活）
```

---

## 高可用的架构本质

```
MQ高可用 = 解决两个问题：

1. Broker挂了怎么办？ → 副本机制（主从/ISR）
2. 怎么快速恢复？      → 自动故障转移

所有MQ的高可用方案都是这两个问题的不同解法：
  Kafka  → ISR + Controller自动选主
  RocketMQ → 主从复制 + Dledger/Controller自动切换
  RabbitMQ → 镜像队列 + 手动/自动故障转移
```

## 存储模型决定性能上限

```
Kafka为什么快？
  顺序写 + 零拷贝 + 批量发送 + 页缓存 + 分区并行
  本质：把磁盘当顺序日志用，把OS当缓存管理器用

RocketMQ为什么功能全？
  CommitLog（顺序写） + ConsumeQueue（逻辑读）
  一份物理存储，多份逻辑索引 → 天然支持回溯、延迟、事务

RabbitMQ为什么延迟低？
  Erlang的进程模型 + 内存消息优先 + 极轻量路由
  但吞吐量上不去 → 单队列无分区，并行度受限
```

---

## 事务消息：Kafka vs RocketMQ — 场景完全不同

```
Kafka事务 = 流处理场景的Exactly Once
  解决的问题：消费→处理→生产 这个链路的原子性
  典型场景：Flink消费Kafka → 处理 → 写回Kafka，保证不重不丢
  不是给你的业务事务用的！

RocketMQ事务 = 业务场景的本地事务+消息原子性
  解决的问题：本地事务和消息发送要么都成功，要么都失败
  典型场景：扣余额 + 发消息通知，保证一致性
  你的蛋壳钱包TCC → 可以用这个替代

架构师认知：
  不要用Kafka事务消息解决业务事务问题
  不要用RocketMQ事务消息解决流处理Exactly Once
  两个"事务"说的是不同的事
```

## 元数据管理：强一致 vs 最终一致

```
Kafka（ZK/KRaft）：
  元数据需要强一致 → 分区Leader选举不能出错
  代价：依赖外部ZK或内置Raft，复杂度高

RocketMQ（NameServer）：
  元数据允许短暂不一致 → 路由信息差几秒可接受
  代价：Producer可能发到已下线的Broker → 重试即可

哪个更好？
  没有更好，只有更适合：
  - Kafka的分区Leader是数据一致性的关键 → 必须强一致
  - RocketMQ的路由只是指引 → 最终一致够用

和你的Nacos经验呼应：
  Nacos的Distro协议也是AP → 服务注册允许短暂不一致
  Nacos的Raft协议用于配置管理 → 配置需要强一致
  同一个系统内部，不同场景用不同一致性级别
```
