# Day 1：消息队列 — 削峰、解耦、最终一致性的核心基础设施（练习整理）

## 题目一：为什么需要消息队列？三大场景说清楚

### 四大核心场景

| 场景 | 没有MQ时的问题 | MQ如何解决 | 秒杀场景映射 |
|------|---------------|-----------|-------------|
| **异步** | 同步调用等待响应，链路长延迟高 | 发后即忘，消费者按节奏处理 | 秒杀下单后异步通知积分、短信 |
| **解耦** | 上游直接依赖下游，下游挂了上游也挂 | 上游只管发消息，不关心谁消费 | 秒杀成功后通知库存、物流、积分，任一服务挂了不影响下单 |
| **削峰填谷** | 瞬时高并发直接打垮下游 | 请求先入队，消费者按能力消费 | 秒杀下单请求先入MQ，订单系统按DB承受力消费 |
| **最终一致性** | 分布式事务太重，性能差 | 本地事务 + 消息 + 补偿 | TCC钱包系统，也可以用MQ做最终一致性方案 |

> 注意：异步和解耦是两个独立概念——异步强调"不等结果"，解耦强调"不关心谁消费"，面试时要说清楚区别。

### 削峰的代价

- **延迟增加**：消息从入队到被消费有延迟，秒杀场景用户可能几秒后才看到订单
- **消息积压风险**：消费速度跟不上生产速度，积压导致延迟越来越大
- **复杂度上升**：需要考虑消息丢失、重复消费、顺序消费等问题

### MQ vs RPC 的架构本质

| 维度 | RPC（同步） | MQ（异步） |
|------|-----------|-----------|
| 调用方式 | 实时等待响应 | 发后即忘 |
| 耦合度 | 强耦合 | 松耦合 |
| 流量控制 | 下游扛不住就超时 | 天然削峰 |
| 一致性 | 强一致 | 最终一致 |

> 架构师决策：能同步就同步，需要异步才用MQ。MQ引入的复杂度（丢失、重复、顺序、积压）是真实成本。

## 题目二：如何保证消息不丢失？三环节全链路分析

这是消息队列面试中的**第一高频题**。

```
生产者 ──①──→ Broker ──②──→ 消费者
         发送阶段      存储阶段      消费阶段
```

### ① 生产端保障

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| 发送失败 | 网络抖动、Broker不可用 | **同步发送 + 重试**（RocketMQ默认2次重试） |
| 异步发送不确认 | send异步调用，不关心结果 | 改为**同步发送**或使用**SendCallback确认** |
| Broker收到但响应丢失 | 网络问题导致生产者以为发送失败 | **幂等生产者**：消息带唯一key，Broker去重 |

**重试失败后的兜底（生产级方案）**：
1. 重试超过配置次数 → 消息落入**本地事务表**并告警
2. 人工介入或通过**定时Job**补偿发送
3. 不能只说"重试"就结束，重试失败后的处理才是区分度的关键

```java
DefaultMQProducer producer = new DefaultMQProducer("producer_group");
producer.start();

Message msg = new Message("TopicTest", "TagA", "orderId", "秒杀订单内容".getBytes());
SendResult result = producer.send(msg, 3000); // 同步发送，超时3秒

if (result.getSendStatus() != SendStatus.SEND_OK) {
    // 重试失败 → 落本地事务表 + 告警，后续Job补偿
    localMsgTableMapper.insert(msg);
    alertService.warn("消息发送失败，已落库: " + msg.getKeys());
}
```

### ② Broker端保障

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| 消息还在内存，Broker宕机 | 异步刷盘，消息未持久化 | **同步刷盘**（SYNC_FLUSH） |
| 主Broker宕机，从Broker未同步 | 异步复制，从节点数据落后 | **同步双写**（SYNC_MASTER） |

#### 同步刷盘 vs 异步刷盘的性能差距

| 配置 | 单Broker吞吐量（参考值） | 延迟 |
|------|------------------------|------|
| ASYNC_FLUSH | 8~12万 TPS | 亚毫秒级 |
| SYNC_FLUSH | 4~6万 TPS | 毫秒级 |

大概 **30%~50% 的吞吐量下降**，具体取决于磁盘类型：

- **机械硬盘（HDD）**：差距更大，fsync耗时约5-10ms，同步刷盘可能只剩2-3万TPS
- **SSD**：差距缩小，fsync耗时约0.5-2ms，同步刷盘还能保持5-6万TPS
- **NVMe SSD**：差距最小，fsync耗时约0.1-0.5ms

#### 主从复制的性能差距

| 配置 | 吞吐量影响 |
|------|-----------|
| ASYNC_MASTER | 无额外开销 |
| SYNC_MASTER | 吞吐量再降20-30%（写入延迟 = 主写入 + 主从网络RTT + 从写入） |

#### 四种组合的吞吐量阶梯

```
最高性能，最低可靠：
  ASYNC_FLUSH + ASYNC_MASTER → ~12万TPS

平衡方案：
  ASYNC_FLUSH + SYNC_MASTER  → ~9万TPS（主从同步防单点，但不等刷盘）

最高可靠，最低性能：
  SYNC_FLUSH + SYNC_MASTER   → ~4万TPS（全链路同步）
```

#### 架构师选型决策

| 业务场景 | 推荐配置 | 理由 |
|---------|---------|------|
| 日志/埋点 | ASYNC + ASYNC | 丢几条可接受，吞吐优先 |
| 普通业务消息 | ASYNC + SYNC | 主从同步防单点，刷盘靠OS保障 |
| 金融/支付 | SYNC + SYNC | 一条都不能丢，牺牲性能换可靠 |
| 秒杀 | ASYNC + SYNC | 削峰场景吞吐优先，主从同步防单点，极端丢消息可通过对账补偿 |

> 不是所有消息都值得最高可靠性。架构师要按业务容忍度分级，不同Topic用不同配置。

### ③ 消费端保障

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| 先提交offset再处理 | 处理失败但offset已提交，消息丢失 | **关闭自动提交，处理完再手动ACK** |
| 处理过程中异常 | 消息被确认但业务未完成 | **消费幂等** + 手动确认 |

**完整消费端方案**：
1. 关闭自动提交offset
2. 业务执行完后手动提交 CONSUME_SUCCESS / RECONSUME_LATER
3. 消费失败进入重试（RocketMQ默认重试16次）
4. 超过重试次数 → 进入**死信队列** `%DLQ%ConsumerGroup`
5. 死信队列消费 + 告警机制，人工介入或Job补偿

```java
consumer.registerMessageListener(new MessageListenerConcurrently() {
    @Override
    public ConsumeConcurrentlyStatus consumeMessage(
            List<MessageExt> msgs, ConsumeConcurrentlyContext context) {
        try {
            for (MessageExt msg : msgs) {
                // 1. 幂等检查
                if (orderService.isProcessed(msg.getKeys())) {
                    continue;
                }
                // 2. 业务处理
                orderService.processOrder(msg.getBody());
            }
            // 3. 业务成功后才手动确认
            return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
        } catch (Exception e) {
            // 4. 处理失败，稍后重试
            return ConsumeConcurrentlyStatus.RECONSUME_LATER;
        }
    }
});
```

### 全链路可靠性保障层级

```
                     可靠性保障层级
┌─────────────────────────────────────────────┐
│  L1: 生产端确认   同步发送 + 重试 + 幂等key   │
├─────────────────────────────────────────────┤
│  L2: Broker持久化  同步刷盘 + 同步双写        │
├─────────────────────────────────────────────┤
│  L3: 消费端保障   手动ACK + 幂等消费         │
├─────────────────────────────────────────────┤
│  L4: 补偿机制     定时对账 + 死信队列 + 告警   │
└─────────────────────────────────────────────┘

层级越往上越基础，越往下越兜底。
架构师要确保每一层都有保障，而不是只依赖某一层。
```

> Kafka差异提示：Kafka没有死信队列概念，靠消费者自己管理offset和重试逻辑。

## 题目三：如何保证消息顺序？全局有序 vs 分区有序

### 两种顺序保证

| 级别 | 实现方式 | 性能影响 | 适用场景 |
|------|---------|---------|---------|
| **全局有序** | 只有一个Queue，单消费者 | 吞吐量极低 | 极少使用 |
| **分区有序** | 同一key的消息发到同一Queue | 吞吐量高 | 绝大多数场景 |

### 实现三环节

```
生产端：sharding key路由 → 同一业务ID的消息发到同一Queue
Broker端：无需额外处理，同一Queue内消息天然有序
消费端：单线程串行消费同一Queue，保证消费顺序
```

```java
// 生产端：MessageQueueSelector 路由
producer.send(msg, new MessageQueueSelector() {
    @Override
    public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
        Long orderId = (Long) arg;
        return mqs.get((int) (orderId % mqs.size()));
    }
}, orderId);

// 消费端：MessageListenerOrderly 单线程消费
consumer.registerMessageListener(new MessageListenerOrderly() {
    @Override
    public ConsumeOrderlyStatus consumeMessage(
            List<MessageExt> msgs, ConsumeOrderlyContext context) {
        for (MessageExt msg : msgs) {
            processOrder(msg);
        }
        return ConsumeOrderlyStatus.SUCCESS;
    }
});
```

### 消费失败时的保序方案（重点）

**常规方案**：消费失败 → 阻塞当前Queue → 重试 → 死信队列。问题：一条消息卡住整个分区。

**更好的方案 — 失败落库 + Job补偿**：

```
消息流入 → 检查该订单ID是否在失败表中
  ├── 不在失败表 → 正常消费
  │      └── 消费失败 → 重试N次 → 落库到失败表 + 告警
  └── 在失败表 → 当前消息也落库，不走业务消费

Job补偿 → 按订单ID维度，顺序重放该ID下所有消息
  ├── 全部成功 → 清除失败表记录
  └── 仍有失败 → 保持落库，继续重试
```

优势：
1. 分区不会因一条消息卡住，其他订单的消息正常流转
2. Job补偿时只处理失败的订单ID，按ID维度顺序重放
3. 积压风险更低，可用性更好

代价：每条消息消费前需要查失败表，可用Redis缓存失败订单ID集合降低DB压力。

> RocketMQ的Queue:Consumer = 1:1硬限制：一个Queue只能被一个Consumer消费，扩容Consumer时要注意这个约束。

## 题目四：如何处理消息重复？幂等性设计

### 为什么会重复？

| 原因 | 说明 |
|------|------|
| 生产者重试 | 网络超时，消息已到达但响应丢失，生产者重发 |
| 消费者重试 | 消费超时，Broker重新投递 |
| Broker主从切换 | 主从同步后主宕机，新主可能重复投递 |

**核心认知：MQ语义是 At Least Once，消息重复不可避免，只能通过消费端幂等来解决！**

### 四种幂等方案对比

| 方案 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| **数据库唯一索引** | 消息key作为唯一索引 | 实现简单，可审计 | 高并发下唯一索引成为热点瓶颈 |
| **Redis SETNX** | 消息key存Redis | 性能高 | 缓存丢失时幂等失效 |
| **业务状态机** | 状态只能单向流转（如乐观锁） | 天然幂等，零额外成本 | 仅限有状态的业务 |
| **乐观锁/版本号** | 更新时带版本号 | 无额外存储 | 需要业务表加版本字段 |

### 各方案细节

**数据库唯一索引瓶颈优化**：高并发下唯一索引的插入会成为热点，可以用分表或先查Redis挡一层缓解。

**Redis SETNX的兜底问题**：Redis宕机时SETNX失效，重复消息可能进入业务。所以SETNX更适合做第一道防线，业务层还要有兜底。

**业务状态机 — 秒杀订单天然适用**：待支付→已支付→已发货，状态只能单向流转，重复消费同一个"支付成功"消息，第二次执行时状态已经不是待支付了，直接跳过。

### 实战：组合使用（推荐）

```
第一层：Redis SETNX 快速过滤（挡住99%重复）
第二层：业务状态机 兜底保障（Redis失效时兜底）
第三层：对账脚本 定时巡检（兜底的兜底）
```

### 幂等方案选择决策树

```
业务有状态机？ ──是──→ 用状态机保证幂等（零成本）
     │
     否
     │
并发量高？ ──是──→ Redis SETNX + 异步对账
     │
     否
     │
需要审计追踪？ ──是──→ 数据库唯一索引
     │
     否
     │
→ 乐观锁/版本号
```

## 题目五：消息积压了怎么办？线上最常遇到的故障

### 第一步：判断消息重要性

- 无关紧要的消息（日志、统计）→ 可以直接丢弃
- 业务消息 → 进入下面的处理流程

### 第二步：应急处理

**扩容（5分钟内）**：
1. 增加消费者实例数
2. 增加每个消费者的消费线程数
3. 注意：RocketMQ的Queue:Consumer = 1:1，如果已经1:1则扩容无效

**转接方案（Queue:Consumer已1:1时）**：
```
原Topic（4个Queue，4个Consumer，积压100万）
       ↓ 转发到新Topic
新Topic（16个Queue，16个Consumer，消费能力4倍）
       ↓ 消费完成后切回
原Topic恢复正常
```

**异步落库 + Job消化**：
- 消费端先异步落库，不直接走业务逻辑
- 本质是把MQ的积压转移到DB的积压，MQ积压触发告警和阈值风险，DB积压则可控得多
- 通过Job慢慢消化DB中的消息

**降级**：跳过非关键消息，降级非核心业务，释放消费能力

### 第三步：根因分析

| 根因 | 处理方式 |
|------|---------|
| 消费端慢（Bug/慢SQL/大对象序列化） | 优化代码和技术方案 |
| 下游依赖慢（第三方接口超时） | 超时控制、熔断降级 |
| 突增流量（秒杀活动等非常态） | 容量规划调整（扩Queue、扩Consumer），性价比高 |

### 预防层面

- **监控**：队列深度、消费延迟、消费TPS
- **告警**：积压量超阈值、消费延迟超阈值
- **容量**：按峰值2-3倍规划消费者数量
- **过期处理**：积压消息超过过期时间被清理，生产端做好过期检测和重发，消费端做时间戳校验
