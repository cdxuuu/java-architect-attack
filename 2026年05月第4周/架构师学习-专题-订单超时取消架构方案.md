# 专题：订单超时取消 — 为什么大厂不用MQ延迟消息，用什么替代

## 一、问题的本质

### 业务场景

| 你的业务 | 场景 | 超时时间 |
|---------|------|---------|
| 保险商城 | 投保后未支付自动取消 | 15~30分钟 |
| 秒杀系统 | 下单后未支付释放库存 | 15分钟 |
| 蛋壳钱包 | 转账超时关单 | 5~10分钟 |
| 普通电商 | 订单超时取消 | 30分钟 |

看似简单：创建订单时发一条延迟消息，到期消费时检查状态，未支付则取消。

**但这个方案在大规模场景下有致命问题。**

---

## 二、MQ延迟消息方案的问题

### 问题1：无效投递风暴

```
场景：电商大促，一天1000万订单

假设：
  90%的订单在超时前支付（正常情况）
  10%的订单真正超时需要取消

MQ延迟消息方案：
  1000万条延迟消息全部投递到MQ
  → 900万条到期后消费，查DB发现已支付，直接跳过（白跑）
  → 只有100万条真正需要取消

问题：90%的消息是无效的，但MQ照存、照投、照消费
  - Broker存储压力：1000万条消息占磁盘
  - Consumer处理压力：1000万次DB查询（900万次白查）
  - 网络带宽：1000万条消息的传输
```

### 问题2：延迟消息无法撤回

```
用户10:00下单 → 发30分钟延迟消息 → 消息将在10:30投递
用户10:05支付 → 订单状态变为已支付

但10:30到了 → 延迟消息照样投递 → Consumer查DB发现已支付 → 跳过
消息投出去了，你拦不住

对比：DB定时扫描方案
  10:30定时任务扫描 → WHERE status = UNPAID → 只查出真正未支付的
  已支付的订单根本不在扫描结果里 → 零无效操作
```

### 问题3：延迟精度与MQ负载的矛盾

```
RocketMQ旧版18个延迟级别 → 最接近15分钟的是10分钟或20分钟 → 不精确
RocketMQ 5.x任意延迟 → 每条消息一个独立延迟 → 时间轮/定时器压力大

大促场景：
  1000万条延迟消息，每条都有不同的到期时间
  → SCHEDULE_TOPIC内部要维护1000万个定时任务
  → Broker内存和CPU压力巨大
```

### 问题4：消息积压加剧

```
延迟消息到期后集中投递 → 大促时10:30到期的可能几十万条同时投递
→ Consumer瞬间收到几十万条消息
→ 如果Consumer处理慢（查DB+调库存服务）→ 积压
→ 积压导致后续消息延迟 → 更多订单超时误判
→ 恶性循环
```

### 问题5：可靠性依赖MQ

```
MQ延迟消息丢了 → 订单永远不会被取消 → 库存永远被占
MQ Broker故障 → 延迟消息无法投递 → 大量订单堆积
消费端故障 → 延迟消息积压 → 恢复后集中消费 → 瞬时压力

架构原则：核心业务不能依赖单一中间件的可靠性
```

## 三、时间轮思想 — 问题的正确抽象

### 从"轮询全量"到"精确命中小范围"

```
朴素方案：每分钟扫描全表
  SELECT * FROM order WHERE status = UNPAID
  → 1000万订单，每次扫全表 → DB直接被打死

改进1：加时间条件
  SELECT * FROM order 
  WHERE status = UNPAID AND create_time < NOW() - INTERVAL 30 MINUTE
  → 还是扫全量未支付订单，只是缩小了结果集

改进2：加expire_time字段 + 索引
  创建订单时：INSERT ... , expire_time = NOW() + INTERVAL 30 MINUTE
  支付完成时：UPDATE SET status = PAID  → 已支付订单不再被扫到
  定时扫描：SELECT * FROM order 
            WHERE status = UNPAID 
              AND expire_time BETWEEN '10:30:00' AND '10:30:59'
  → 只扫"当前分钟到期且未支付"的订单 → 极小范围

这就是时间轮的思想：
  全表轮询 = 遍历时间轮的所有槽位
  expire_time分桶 = 精确命中一个槽位
```

### Netty时间轮 vs 分布式时间轮

```
Netty HashedWheelTimer（单机版）：
  ┌──┬──┬──┬──┬──┬──┬──┬──┐
  │  │  │  │  │  │  │  │  │  ← 时间槽数组
  └──┴──┴──┴──┴──┴──┴──┴──┘
     ↑ 指针转动，到哪个槽执行哪个槽的任务

  任务加入：hash(延迟时间) % 槽数 → 放入对应槽
  指针转动：每tick前进一格，执行当前槽的所有任务

  限制：
    - 纯内存，重启丢数据
    - 单JVM，多实例重复触发
    - 百万级Timeout对象占内存

分布式时间轮（DB版）：
  ┌──────────────────────────────────────────┐
  │  order表（expire_time字段 + 索引）         │
  │  ┌──────┬────────┬─────────┬──────────┐ │
  │  │ 10:30│ 10:31  │ 10:32   │ 10:33    │ │  ← 按expire_time自然分桶
  │  │ 订单A│ 订单C  │ 订单E   │ 订单G    │ │
  │  │ 订单B│ 订单D  │ 订单F   │          │ │
  │  └──────┴────────┴─────────┴──────────┘ │
  └──────────────────────────────────────────┘
       ↑ 定时任务只查当前时间片的桶

  优势：
    - DB天然持久化，不丢数据
    - 索引即分桶，查询O(logN)级别的数据量
    - 分布式定时任务天然多实例
    - 已支付订单不在扫描范围，零无效操作
```

### 时间轮的层级扩展（多层时间轮）

```
Netty实现中：延迟时间超过一轮 → 需要多层时间轮
  第1层：1秒一格，60格（覆盖1分钟）
  第2层：1分钟一格，60格（覆盖1小时）
  第3层：1小时一格，24格（覆盖1天）

DB实现中：expire_time索引天然支持多层
  WHERE expire_time BETWEEN '10:30:00' AND '10:30:59'  ← 精确到分钟
  WHERE expire_time BETWEEN '10:00:00' AND '10:59:59'  ← 精确到小时
  WHERE expire_time BETWEEN '2026-05-20 00:00:00' AND '2026-05-20 23:59:59'  ← 精确到天

DB索引不需要你手动分层，B+Tree天然支持范围查询的高效定位
```

## 四、大厂实际方案详解

### 方案1：DB时间桶 + 分布式定时任务（最常见，推荐）

#### 表设计

```sql
CREATE TABLE `t_order` (
    `id` bigint NOT NULL AUTO_INCREMENT,
    `order_no` varchar(64) NOT NULL COMMENT '订单号',
    `user_id` bigint NOT NULL,
    `amount` decimal(12,2) NOT NULL,
    `status` tinyint NOT NULL DEFAULT 0 COMMENT '0-待支付 1-已支付 2-已取消 3-已超时关闭',
    `expire_time` datetime NOT NULL COMMENT '超时时间',
    `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_order_no` (`order_no`),
    KEY `idx_expire` (`status`, `expire_time`)  -- 核心：联合索引，定时扫描用
) ENGINE=InnoDB;
```

**索引设计要点**：

```
为什么是 (status, expire_time) 联合索引而不是 (expire_time) 单列？

查询语句：
  SELECT * FROM t_order 
  WHERE status = 0 AND expire_time <= NOW()
  LIMIT 1000

联合索引 (status, expire_time)：
  → 先定位 status=0，再在范围内按expire_time排序
  → 极其高效，扫描数据量 = 未支付+已到期的订单数

单列索引 (expire_time)：
  → 按expire_time定位范围后，还要回表过滤status
  → 如果10:30到期的订单有10万条，但其中9万已支付
  → 回表过滤9万条无效数据 → 浪费
```

#### 创建订单

```java
@Service
public class OrderService {
    
    public OrderCreateResult createOrder(CreateOrderRequest request) {
        Order order = new Order();
        order.setOrderNo(generateOrderNo());
        order.setStatus(OrderStatus.UNPAID);
        // 核心：设置超时时间
        order.setExpireTime(LocalDateTime.now().plusMinutes(30));
        
        orderMapper.insert(order);
        return OrderCreateResult.success(order.getOrderNo());
    }
}
```

#### 支付回调

```java
public void onPaymentSuccess(String orderNo) {
    // 支付成功 → 更新状态
    // 已支付的订单不会被定时任务扫到（status != 0）
    orderMapper.updateStatus(orderNo, OrderStatus.PAID);
}
```

#### 定时任务扫描

```java
@XxlJob("orderTimeoutCheckJob")
public void orderTimeoutCheck() {
    LocalDateTime now = LocalDateTime.now();
    
    // 每次只扫1000条，避免大事务
    List<Order> orders = orderMapper.selectExpired(now, 1000);
    
    while (!orders.isEmpty()) {
        for (Order order : orders) {
            try {
                cancelOrder(order);
            } catch (Exception e) {
                log.error("取消订单失败: {}", order.getOrderNo(), e);
                // 失败不阻塞，下次扫描重试
            }
        }
        orders = orderMapper.selectExpired(now, 1000);
    }
}

// Mapper
@Select("SELECT * FROM t_order " +
        "WHERE status = 0 AND expire_time <= #{now} " +
        "LIMIT #{limit}")
List<Order> selectExpired(@Param("now") LocalDateTime now, 
                          @Param("limit") int limit);
```

#### 分片扫描（分布式扩展）

```
单实例扫描：
  一个Job实例扫全表 → 大促时扫不过来

分片扫描（XXL-Job分片模式）：
  分片参数：shardTotal=4, shardIndex=0~3
  → 每个实例扫 order_id % 4 == shardIndex 的那部分
  → 4个实例并行扫描，4倍吞吐

  @XxlJob("orderTimeoutCheckJob")
  public void orderTimeoutCheck() {
      int shardTotal = XxlJobHelper.getShardTotal();
      int shardIndex = XxlJobHelper.getShardIndex();
      
      List<Order> orders = orderMapper.selectExpiredSharded(
          now, 1000, shardTotal, shardIndex);
      // ...
  }
  
  SQL：
  SELECT * FROM t_order 
  WHERE status = 0 AND expire_time <= #{now}
    AND id % #{shardTotal} = #{shardIndex}
  LIMIT #{limit}
```

#### 关键配置参数

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| 扫描间隔 | 30秒~1分钟 | 不需要秒级，订单超时是分钟级 |
| 每批数量 | 500~1000 | 避免大事务，分批处理 |
| 订单超时容差 | 扫描间隔 × 2 | 如1分钟扫描，实际超时可能是30~31分钟，业务可接受 |
| 分片数 | 按峰值订单量/每批处理速度计算 | 如峰值10万单/分钟，每批1000条处理1秒，需要2个分片 |

### 方案2：Redis ZSET做分布式时间轮（中等规模）

#### 数据结构

```
Redis ZSET：score = 到期时间戳，member = orderId

创建订单：
  ZADD order:expire <timestamp_30min_later> <orderId>

支付成功：
  ZREM order:expire <orderId>    ← 直接移除，不再检测

定时检测：
  ZRANGEBYSCORE order:expire 0 <current_timestamp> LIMIT 0 1000
  → 取出到期的订单ID
  → 批量查DB确认状态 → 未支付则取消
  ZREM order:expire <已处理的orderId>
```

#### 与MQ延迟消息的对比

```
              MQ延迟消息              Redis ZSET
创建时      发延迟消息到MQ             ZADD写入ZSET
支付时      无法撤回，消息照样投递      ZREM直接移除，零无效操作
到期检测    Broker投递 → Consumer消费  Job轮询ZSET → 查DB确认
持久化      Broker磁盘持久化           Redis RDB/AOF（可能丢少量）
规模限制    Broker内存/CPU             Redis内存（百万级OK，千万级要评估）
```

**Redis ZSET的核心优势：支付时可以移除，彻底消除无效操作。**

#### 实现

```java
@Service
public class OrderExpireService {
    
    private static final String EXPIRE_ZSET_KEY = "order:expire";
    
    // 创建订单时
    public void addExpireOrder(String orderId, int timeoutMinutes) {
        long expireTimestamp = System.currentTimeMillis() 
            + timeoutMinutes * 60 * 1000L;
        stringRedisTemplate.opsForZSet().add(
            EXPIRE_ZSET_KEY, orderId, expireTimestamp);
    }
    
    // 支付成功时
    public void removeExpireOrder(String orderId) {
        stringRedisTemplate.opsForZSet().remove(EXPIRE_ZSET_KEY, orderId);
    }
    
    // 定时检测
    @XxlJob("orderExpireCheckJob")
    public void checkExpiredOrders() {
        long now = System.currentTimeMillis();
        
        // 取出到期的订单（每次最多1000条）
        Set<String> expiredOrderIds = stringRedisTemplate.opsForZSet()
            .rangeByScore(EXPIRE_ZSET_KEY, 0, now, 0, 1000);
        
        if (expiredOrderIds == null || expiredOrderIds.isEmpty()) {
            return;
        }
        
        for (String orderId : expiredOrderIds) {
            try {
                Order order = orderMapper.selectByOrderNo(orderId);
                if (order != null && order.getStatus() == OrderStatus.UNPAID) {
                    cancelOrder(order);
                }
                // 不管取消还是已支付，都从ZSET移除
                stringRedisTemplate.opsForZSet()
                    .remove(EXPIRE_ZSET_KEY, orderId);
            } catch (Exception e) {
                log.error("处理过期订单失败: {}", orderId, e);
            }
        }
    }
}
```

#### Redis ZSET的注意事项

```
1. 内存评估
   每个orderId约32字节 + score 8字节 + 指针开销 ≈ 80字节
   100万订单 ≈ 80MB → 可接受
   1000万订单 ≈ 800MB → 需要评估Redis内存

2. Redis故障兜底
   Redis宕机 → ZSET数据可能丢失 → 订单永远不会被取消？
   解决：DB扫描做L2兜底（见下方分层防御）

3. 大key问题
   单个ZSET存千万级member → ZRANGEBYSCORE可能阻塞Redis
   解决：按时间分片
     order:expire:2026052010  ← 按小时分ZSET
     order:expire:2026052011
   定时任务只查当前小时的ZSET
```

## 五、分层防御：生产级方案

任何单层方案都可能失效，架构师要做的是层层兜底。

```
┌─────────────────────────────────────────────────┐
│  L1: Redis ZSET 快速到期检测（秒级/分钟级）        │
│  → 支付时ZREM移除，零无效操作                      │
│  → 到期时ZRANGEBYSCORE取出，查DB确认后取消          │
├─────────────────────────────────────────────────┤
│  L2: DB定时扫描 兜底（分钟级）                      │
│  → Redis故障/数据丢失时兜底                         │
│  → WHERE status=0 AND expire_time <= NOW()        │
│  → 扫描间隔可以长一些（5分钟），因为是兜底不是主力    │
├─────────────────────────────────────────────────┤
│  L3: MQ延迟消息 最后防线（分钟级）                   │
│  → 前两层都挂了时的最终保障                         │
│  → 每个订单发一条延迟消息，到期消费时再查DB           │
│  → 有无效投递问题，但作为最后一道防线可以接受         │
├─────────────────────────────────────────────────┤
│  L4: 对账脚本 巡检（小时级/天级）                    │
│  → 定时统计：已超时但仍为UNPAID的订单                 │
│  → 发现漏网之鱼 → 人工确认后批量取消                  │
│  → 这是兜底的兜底，确保数据最终一致                   │
└─────────────────────────────────────────────────┘

层级越往上越实时，越往下越兜底
L1处理99%的场景，L2处理Redis故障的场景，L3处理DB Job也挂的场景，L4确保不会遗漏
```

### 为什么还要保留MQ延迟消息（作为L3）？

```
极端场景：Redis宕机 + DB定时任务Job也挂了
  → 订单永远不会被取消
  → 库存永远被占
  → MQ延迟消息作为独立通道，不依赖Redis和DB Job

但注意：
  L3是"保底"不是"主力"
  所以即使有90%无效投递也无所谓——极端情况下才触发
  正常情况下L1和L2已经处理完了，L3的消息消费时查DB发现已取消，直接跳过
```

## 六、方案选型决策树

```
你的业务峰值订单量是多少？

├── 万级/天（普通业务）
│   └── DB时间桶 + XXL-Job（方案1）
│       简单可靠，不需要Redis
│       你的保险商城大概率就是这个量级
│
├── 十万级~百万级/天（中等规模）
│   └── Redis ZSET + DB扫描（方案1+2组合）
│       L1: Redis ZSET做快速检测
│       L2: DB扫描做兜底
│
├── 千万级/天（大促场景）
│   └── DB分片扫描 + Redis ZSET + MQ延迟消息（方案1+2+3组合）
│       L1: Redis ZSET
│       L2: DB分片扫描（XXL-Job分片模式）
│       L3: MQ延迟消息保底
│       L4: 对账脚本巡检
│
└── 亿级/天（头部大厂）
    └── 自研分布式时间轮服务 + 全链路分层防御
        投入产出比只在超大规模下才成立
```

## 七、面试怎么答

### 普通回答

"订单超时取消可以用RocketMQ的延迟消息，创建订单时发一条延迟消息，到期消费检查状态，未支付就取消。"

### 架构师回答

"订单超时取消的核心问题不是'能不能实现'，而是'规模大了之后怎么做才不浪费'。

MQ延迟消息的问题在于：大部分订单会在超时前支付，但延迟消息无法撤回，到期照样投递，导致90%以上的无效消费和DB查询。而且Broker要维护大量未到期的延迟消息，大促时存储和调度压力都很大。

大厂的主流方案是DB时间桶 + 分布式定时任务。核心思路来自时间轮：给订单加expire_time字段和联合索引(status, expire_time)，定时任务只扫描'当前时间片到期且未支付'的极小范围。已支付的订单不在扫描结果里，天然零无效操作。这本质上是把Netty时间轮的分桶思想搬到了分布式层面——DB索引就是时间轮的槽位，expire_time就是槽位的key。

生产环境我会做分层防御：L1用Redis ZSET做快速到期检测，支付时ZREM直接移除；L2用DB定时扫描兜底Redis故障；L3用MQ延迟消息做最后防线；L4用对账脚本巡检确保不遗漏。L1处理99%的流量，越往下越兜底。"
