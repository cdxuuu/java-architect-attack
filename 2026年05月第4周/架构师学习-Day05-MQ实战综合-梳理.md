# Day 5 架构师梳理：MQ实战综合 — 分布式事务 / IM场景 / 运维排查

## 一、分布式事务：不是"选哪个"，是"怎么组合"

### 架构师视角

五种分布式事务方案不是互斥的选择题，而是**工具箱里的不同工具**。真实系统往往是多种方案组合使用：

```
核心资产段（蛋壳钱包转账）→ TCC（强一致）
外部交互段（e卡资金路由→资方放款）→ 本地消息表（最终一致）
通知段（通知借款人）→ 最大努力通知（尽力）
长流程段（投保→核保→出单）→ Saga（编排）
```

**架构师的核心判断力不是"知道5种方案"，而是"在同一个系统里，不同段用不同方案，且段与段之间如何衔接"。**

### 关键洞察

1. **TCC vs 事务消息不是同层面的选择**：TCC解决的是"多参与者的一致性"，事务消息解决的是"本地事务和发消息的原子性"。两者可以组合——本地用TCC，跨服务用事务消息

2. **本地消息表是最朴素但最可靠的方案**：不绑定MQ、不依赖Broker回查、天然支持对账。大厂不是不用，而是**在核心链路用更重的方案，在通用链路用消息表**

3. **对账是一致性的最后防线**：不管用什么方案，最终一致性系统必须配对账。没有对账的最终一致性 = "可能一致"

4. **事务消息回查15次后丢弃**：这是面试高频考点，也是线上排坑必备知识。回查接口必须幂等，否则回查本身可能产生脏数据

### 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 全链路TCC | 第三方系统不支持Try/Confirm/Cancel | 核心段TCC + 外部段消息表 |
| 全链路事务消息 | 依赖RocketMQ，换MQ就得重构 | 本地消息表解耦MQ依赖 |
| 没有对账 | 最终一致性永远只是"可能"一致 | 日切对账兜底 |
| 回查接口不幂等 | 回查重复执行可能产生脏数据 | 查本地事务状态，按状态返回 |

## 二、MQ降级不是"MQ挂了就完了"

### 架构师视角

面试官问"MQ宕机怎么办"，不是要听"集群高可用"，而是要听**降级方案的设计**。架构师的思路是：

```
正常链路：A → MQ → B
降级链路：A → 同步调B（核心）/ 暂存DB（非核心）
恢复链路：暂存消息 → 按序重发 → 幂等消费
兜底链路：对账脚本 → 差异补偿
```

**四条链路齐全 = 架构师思维。只有第一条 = 开发思维。**

### 关键洞察

1. **降级开关是标配**：配置中心一键切换同步/异步模式，不是代码里写if-else

2. **同步降级的代价**：RT从毫秒级变成百毫秒级，要限流保护DB

3. **恢复后不能瞬间灌消息**：MQ恢复后，暂存消息要限速重发，否则造成二次积压

4. **对账不受降级影响**：无论走同步还是异步，日终对账照常运行

## 三、IM消息系统：三种场景三种策略

### 架构师视角

IM消息投递不是一种方案打天下，而是**在线/离线/多端三种场景分别设计**：

```
在线：推送（WebSocket）+ 送达回执
离线：分层存储（Redis→MySQL→HBase）+ 拉取
多端：连接表管理 + 全端推送 + 全局未读数
```

### 关键洞察

1. **可靠投递 = 至少一次投递 + 幂等消费**：不追求"恰好一次"（Exactly Once），那是分布式系统的理论问题。实际做法是At Least Once + 幂等

2. **微信不显示已读是产品+技术双重决策**：面试时能从两个维度分析是加分项。纯技术视角会说"写放大"，加上产品视角说"社交压力"才是架构师

3. **离线消息分层存储是成本-延迟权衡**：
   - 全Redis：成本不可接受
   - 全DB：延迟不可接受
   - 分层：热数据Redis保体验，冷数据DB/HBase保成本

4. **30天离线消息的架构决策**：不自动全量拉取，按需分页加载。这是"用户体验"和"系统成本"的权衡——30天前的消息大概率用户不需要

## 四、读扩散vs写扩散：没有银弹，只有权衡

### 架构师视角

这不是一个技术选择题，而是一个**读写比例+群规模+实时性**的三维决策：

```
              群规模小(<500)
              /          \
        读多写少        写多读少
         ↓                ↓
      写扩散            写扩散
    （预推好，读取快）  （群小，写放大可接受）

              群规模大(>1000)
              /          \
        读多写少        写多读少
         ↓                ↓
      写扩散            读扩散
    （微博粉丝模式）   （钉钉大群模式）

              群规模超大(>10000)
                    ↓
              推拉结合
    （在线推+离线拉）
```

### 关键洞察

1. **10000人群写扩散 = 1条消息×10000次写**：这不是"能不能做"的问题，而是值不值得做的问题

2. **大厂最终方案都是推拉结合**：纯写扩散扛不住大群，纯读扩散实时性差。推拉结合取两者优点

3. **异步扩散是写扩散的核心优化**：发送者不感知扩散延迟，MQ异步写收件人收件箱

4. **未读数一致性 = 写扩散的副产品问题**：写扩散每条消息要更新所有收件人未读数，10000人群=10000次Redis写。必须异步批量处理

5. **删除/撤回在写扩散下是噩梦**：10000份副本都要改。读扩散只改1份。这也是大群倾向读扩散的原因之一

## 五、运维排查：架构师=排障SOP的设计者

### 架构师视角

运维排查不是"会看监控面板"，而是**设计排障SOP和容量体系**：

```
开发：看到lag告警 → 去Dashboard看 → 凭经验猜原因
架构师：设计分阶段SOP → 自动化告警分级 → 容量评估公式 → 预案文档
```

### 关键洞察

1. **0-5分钟止血比定位根因更重要**：先止血再排查。非核心链路直接暂停，核心链路回滚最近部署

2. **lag飙升四段定位法**：生产端突增？消费端变慢？Rebalance风暴？Broker问题？每一段有对应的排查路径

3. **磁盘水位85%是"最后警告"**：到90%就难救了。磁盘问题往往不是磁盘本身的问题，而是消费积压或Topic治理的体现

4. **安全水位60%不是保守，是工程常识**：
   - 日常3倍突发流量是正常的
   - 一台Broker宕机，流量要分给其他节点
   - GC/IO抖动会造成水位瞬间冲高
   - 跑80% = 没有容错空间 = 一次抖动就雪崩

5. **Topic生命周期治理不是可选项**：大厂集群上千个Topic，没治理 = 废弃Topic持续消耗磁盘/带宽/CPU = 永远"磁盘不够用"

## 七、面试话术模板

### 分布式事务

> "我做过蛋壳钱包TCC，资金链路必须强一致。但我们不是全链路TCC——核心段（扣款）用TCC，外部交互段（调支付通道）用本地消息表+对账兜底。事务消息我们评估过，但当时MQ不是RocketMQ，而且核心链路对一致性要求太高，本地消息表+对账更可控。"

### MQ降级

> "MQ不可用时我们有三层降级：核心链路同步调用保业务不中断，非核心消息本地暂存，MQ恢复后限速重发。但最关键的是日终对账——不管走哪条链路，最终一致性要靠对账兜底。"

### IM场景

> "IM消息投递的核心是At Least Once + 幂等。在线推送、离线分层存储、多端独立ACK。微信不显示已读，既有技术原因（写放大），也有产品原因（社交压力）。我们做客服IM时，已读回执走IM通道，不需要MQ。"

### 运维排查

> "lag飙升我先止血再定位——0-5分钟看是否核心链路，非核心暂停；5-30分钟四段定位（生产/消费/Rebalance/Broker）。日常做容量评估，安全水位定60%，因为要留突发buffer和宕机冗余。"
