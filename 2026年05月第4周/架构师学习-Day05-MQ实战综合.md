# Day 5：MQ实战综合 — 分布式事务整合 / IM在线客服 / 运维排查

## 题目一：分布式事务方案选型 — TCC/本地消息表/事务消息/Saga/最大努力通知

### 场景映射

| 你的业务 | 分布式事务需求 |
|---------|-------------|
| 蛋壳钱包 | 转账必须强一致，金额不能错 |
| 国美e卡资金路由（小贷平台） | 借款人放款→资金路由匹配资方→资方放款→落账 |
| 保险商城 | 投保→核保→出单，最终一致可接受 |
| 秒杀系统 | 扣库存→创建订单→支付，中间态要可控 |

### 五种方案对比

| 维度 | TCC | 本地消息表 | 事务消息(RocketMQ) | Saga | 最大努力通知 |
|------|-----|-----------|-------------------|------|------------|
| 一致性 | 强一致（最终） | 最终一致 | 最终一致 | 最终一致 | 最终一致（尽力） |
| 性能 | 低（3次RPC） | 中（1次DB+1次MQ） | 高（1次half+1次MQ） | 中 | 高 |
| 侵入性 | 高（写Try/Confirm/Cancel） | 中（写消息表） | 中（写本地事务+回查） | 高（写正反操作） | 低 |
| 适用场景 | 资金/库存等核心链路 | 通用，对账兜底 | RocketMQ技术栈 | 长流程编排 | 通知类（支付结果） |
| 补偿方式 | Cancel回滚 | 定时任务重发 | 回查机制 | 反向补偿 | 多次重试→人工 |
| 最多参与者 | 建议≤3 | 无限制 | 无限制 | 无限制 | 1（通知方） |

### 选型决策树

```
是否涉及资金/库存等核心资产？
├── 是 → 是否能接受最终一致？
│   ├── 不能（必须强一致）→ TCC
│   └── 能 → 参与方≤3且能写Try/Confirm/Cancel？
│       ├── 是 → TCC（更可控）
│       └── 否 → 本地消息表 + 对账
└── 否 → 是否有长流程编排需求？
    ├── 是 → Saga
    └── 否 → 是否为通知类场景？
        ├── 是 → 最大努力通知
        └── 否 → 事务消息 or 本地消息表
            ├── 技术栈是RocketMQ → 事务消息
            └── 其他MQ/无MQ → 本地消息表
```

### 蛋壳钱包为什么选TCC而不是事务消息？

1. **业务本质决定**：转账是资金操作，A扣100和B加100必须是原子性的。事务消息只能保证"A扣100"和"发消息"的原子性，不能保证"B加100"一定成功
2. **中间态可控**：TCC的Try阶段冻结金额，如果Cancel可以原样解冻。事务消息做不到"冻结"——钱要么扣了要么没扣
3. **补偿确定性**：TCC的Cancel是业务级补偿（解冻金额），事务消息的回查只能"重试"，如果下游一直失败，只能进死信

### 国美e卡资金路由选型

场景：借款人申请放款 → 资金路由匹配资方（利率/额度/通过率）→ 资方放款 → 落账

**推荐：本地消息表 + 对账**

```
第一步（强一致）：创建放款订单 + 写本地消息表（同一DB事务）
  → 订单状态：INIT
  → 消息状态：待发送

第二步（最终一致）：资金路由 + 资方交互
  → 定时任务扫描待发送消息，发MQ
  → 消费者执行资金路由：匹配资方 → 调资方放款接口
  → 资方返回成功 → 更新订单状态：SUCCESS
  → 资方返回失败 → 更新订单状态：FAIL，触发换资方重试
  → 资方无响应 → 超时后查询资方状态，幂等处理

第三步（兜底）：对账
  → 每日对账：本地放款流水 vs 资方放款流水
  → 差异自动生成补偿单
```

为什么不用TCC？资方（银行/信托等）是外部系统，不可能实现Try/Confirm/Cancel接口。资金路由的核心难点不是"强一致"，而是**资方接口的不确定性**（超时/拒绝/额度不足）——需要的是重试+换资方+对账，不是回滚。

为什么不用事务消息？资金路由涉及多资方匹配和重试，业务逻辑复杂，本地消息表+定时任务比事务消息+回查更可控。

### 事务消息完整流程（RocketMQ）

```
1. 生产者发送half消息
   → Broker收到后写入RMQ_SYS_TRANS_HALF_TOPIC
   → 对生产者返回OK（此时消费者不可见）

2. 执行本地事务
   → 成功：Commit → Broker将消息移到目标Topic → 消费者可见
   → 失败：Rollback → Broker删除half消息
   → 未知：不做任何操作

3. 回查机制
   → Broker定时扫描RMQ_SYS_TRANS_HALF_TOPIC
   → 发现超时未Commit/Rollback的消息（默认15秒）
   → 向生产者发起回查（最多15次）
   → 回查接口返回Commit/Rollback
   → 15次回查仍无结果 → Rollback（消息丢弃）

4. 关键参数
   - 回查间隔：15秒起
   - 最大回查次数：15次（4.x）/ 无上限但可配置checkMax（5.x）
   - half消息存储：RMQ_SYS_TRANS_HALF_TOPIC
   - 回查进度：RMQ_SYS_TRANS_OP_HALF_TOPIC
```

**回查接口幂等设计**：

```java
public class OrderTransactionListener implements TransactionListener {
    
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        String orderId = msg.getKeys();
        // 1. 查本地事务表，根据状态决定
        OrderTransaction trans = orderTransMapper.selectByOrderId(orderId);
        
        if (trans == null) {
            // 事务记录不存在，可能本地事务还没执行完
            return LocalTransactionState.UNKNOW; // 继续回查
        }
        
        switch (trans.getStatus()) {
            case COMMITTED:
                return LocalTransactionState.COMMIT_MESSAGE;
            case ROLLED_BACK:
                return LocalTransactionState.ROLLBACK_MESSAGE;
            case PROCESSING:
                return LocalTransactionState.UNKNOW; // 还在处理，继续回查
            default:
                return LocalTransactionState.ROLLBACK_MESSAGE; // 未知状态，回滚保安全
        }
    }
}
```

### 本地消息表 vs 事务消息：核心差异

| 维度 | 本地消息表 | 事务消息 |
|------|-----------|---------|
| 原理 | 业务操作和写消息表在同一DB事务 | half消息+本地事务+回查 |
| 依赖 | 只依赖DB | 依赖RocketMQ |
| 性能 | 额外1次DB写 | 额外1次half消息 |
| 可靠性 | 依赖DB可靠性 | 依赖MQ可靠性 |
| 回查 | 不需要（定时扫描即可） | 需要 |
| MQ选型 | 任何MQ都可以 | 仅RocketMQ |

**大厂为什么还在用本地消息表**：

1. **不绑定MQ**：阿里内部很多系统用MetaQ（RocketMQ内部版），但更多公司MQ会迁移。本地消息表只依赖DB，MQ随时可以换
2. **简单可控**：就是一张表+一个定时任务，排查问题一目了然。事务消息的half→回查链路出了问题很难排查
3. **对账天然支持**：消息表本身就是最好的对账数据源
4. **MySQL 5.7+的XA问题**：虽然本地消息表看起来"土"，但事务消息的half消息依赖Broker的回查机制，如果Broker有问题，消息会卡在half状态

## 题目二：MQ最终一致性的边界 — 全链路问题点与降级方案

### 全链路问题点与兜底

#### 1. 生产端

| 问题 | 场景 | 兜底 |
|------|------|------|
| 消息发不出去 | 网络抖动/ Broker不可用 | 同步发送+重试3次+本地暂存 |
| 消息发了但没ACK | 网络超时，不知道发没发成功 | 幂等设计+查询接口确认 |
| 消息内容错误 | 代码bug序列化错误 | 消息回溯+补偿重发 |
| 本地事务提交但消息没发 | 事务和发消息不在同一操作 | 事务消息 or 本地消息表 |

#### 2. Broker端

| 问题 | 场景 | 兜底 |
|------|------|------|
| 消息未持久化就返回ACK | 异步刷盘宕机 | 核心链路用同步刷盘 |
| 主从同步延迟 | 主宕机，从还没同步完 | 同步复制（SYNC_MASTER） |
| 磁盘满 | 消息无法写入 | 磁盘水位监控+自动清理 |
| CommitLog损坏 | 硬件故障 | 多副本+跨机房部署 |

#### 3. 消费端

| 问题 | 场景 | 兜底 |
|------|------|------|
| 消费失败 | 下游服务不可用/数据异常 | 重试→死信队列→人工 |
| 重复消费 | Rebalance/网络重发 | 幂等（Redis+状态机） |
| 消费顺序错乱 | 多Queue多Consumer | 业务层排序/状态机校验 |
| 消费超时 | 业务处理慢 | 合理设超时+异步处理 |
| 消费积压 | 消费速度<生产速度 | 扩容/消息分级/临时Topic转发 |

#### 4. 业务层

| 问题 | 场景 | 兜底 |
|------|------|------|
| 重复落账 | 消息重复消费 | 幂等键+状态机 |
| 金额不一致 | 部分成功部分失败 | 对账脚本+人工补偿 |
| 状态不一致 | 上下游系统状态不同步 | 状态查询接口+补偿Job |
| 超时未确认 | 长时间无确认 | 超时自动关闭+补偿 |

### MQ宕机30分钟的降级方案

```
MQ不可用时的分层降级策略：

L1: 同步降级（0-5分钟）
  → 核心链路：同步调用替代异步消息
  → 例：付款成功 → 同步调账户系统加余额
  → 代价：RT增加，但业务不中断
  → 降级开关：配置中心一键切换同步/异步

L2: 本地暂存（5-15分钟）
  → 非核心消息写入本地DB/文件
  → 标记状态为"待发送"
  → MQ恢复后由定时任务扫描重发

L3: 熔断限流（15-30分钟）
  → 非核心业务直接熔断（如营销通知）
  → 核心业务限流，保护DB不被同步调用压垮
  → 触发条件：MQ不可用>10分钟

L4: MQ恢复后补偿
  → 定时任务扫描本地暂存消息，按时间顺序重发
  → 注意限速：不能瞬间灌入大量消息
  → 消费端要做好幂等

L5: 对账兜底（始终运行）
  → 无论如何降级，日终对账不能少
  → 订单流水 vs 账户流水 vs 支付流水
  → 差异自动生成补偿单，人工审核后执行
```

**关键：降级不是"MQ挂了就完了"，而是一套完整的降级-恢复-补偿体系。架构师面试的核心就是这套体系的完整性。**

### 对账系统设计

#### 日切对账 vs 实时对账

| 维度 | 日切对账 | 实时对账 |
|------|---------|---------|
| 频率 | 每日一次（T+1） | 秒级/分钟级 |
| 数据源 | 业务流水表全量比对 | 增量事件驱动 |
| 延迟 | 最差24小时 | 分钟级 |
| 成本 | 低（离线计算） | 高（实时计算） |
| 覆盖率 | 100% | 依赖事件完整性 |
| 适用 | 资金类（必须100%覆盖） | 通知类（允许少量遗漏） |

#### 对账系统架构

```
对账核心流程：

1. 数据采集
   → 业务系统导出T日流水（订单/支付/账户）
   → 格式：对账文件（CSV/JSON），按约定时间上传

2. 对账匹配
   → 按业务单号做双向匹配
   → 匹配结果：平账 / 单边账 / 差异账

3. 差异处理
   → 自动处理：规则引擎（如超时自动关闭）
   → 半自动：生成补偿单，人工审核后执行
   → 手动处理：疑难差异，人工逐笔排查

4. 对账报表
   → 每日对账报告：总笔数/匹配率/差异笔数
   → 差异明细：单号/差异类型/金额/处理状态
```

**你的蛋壳钱包/国美e卡资金路由一定是做了对账的。面试时主动提"我做过对账系统"——这是区分开发和架构师的关键标志。**

## 题目三：IM在线客服 — 消息可靠投递

### 可靠投递链路设计

#### B在线场景

```
A发送消息的完整链路：

A客户端 → A服务端 → MQ → IM服务端 → B服务端 → B客户端
   │          │        │        │          │         │
   │          │        │        │          │         └── 收到消息，发ACK
   │          │        │        │          └── 推送（WebSocket长连接）
   │          │        │        └── 消费MQ消息，查B的在线状态
   │          │        └── 持久化消息，返回ACK
   │          └── 写消息DB + 发MQ（同一事务/本地消息表）
   └── 等待服务端ACK，超时重发

关键：每一步都有ACK/超时重试，保证至少投递一次（At Least Once），
     幂等设计保证重复投递不会显示两条。
```

#### B离线场景

```
A发送消息 → 服务端写入离线消息表 → B上线后：

方案一：拉取（Pull）
  → B上线后，服务端推送"你有N条未读消息"的通知
  → B客户端按页拉取离线消息
  → 优点：B可控节奏，不压连接
  → 缺点：多次RT

方案二：推送（Push）
  → B上线建立连接后，服务端批量推送离线消息
  → 优点：实时性好
  → 缺点：消息量大时可能阻塞连接

大厂实践：推拉结合
  → B上线时先推一个"未读计数"
  → B进入聊天页后拉离线消息
  → 实时新消息走推送
```

#### B多端登录场景

```
B同时在手机和PC在线：

每条消息需要推送到B的所有端：

IM服务端 → 查B的连接表（哪些端在线）
         → 推送到每个端
         → 每个端独立ACK
         → 任意一个端ACK即视为"已送达"

未读数同步：
  → 服务端维护全局未读数（Redis）
  → 每个端拉取时取同一份未读数
  → 任一端标记已读 → 全局未读数-1 → 推送其他端更新
```

### 送达回执与已读回执

#### 送达回执

```
消息投递 → B端收到 → B端发"已送达"回执 → 服务端 → A端显示"已送达"

实现：
  → B端收到消息后发送Delivered ACK
  → 服务端更新消息状态为"delivered"
  → 推送A端更新状态
  → 需要MQ吗？可以不需要，直接走IM长连接通道即可
  → 如果A端不在线，存DB，A上线后拉取
```

#### 已读回执

```
B打开消息 → B端发"已读"回执 → 服务端 → A端显示"已读"

实现：
  → B端打开聊天页后发送Read ACK
  → 服务端更新消息状态为"read"
  → 推送A端更新
  → 同样走IM通道，不需要MQ
```

#### 为什么微信不显示已读？

1. **产品层面**：降低社交压力。显示已读会让发送方期待即时回复，增加焦虑
2. **技术层面**：
   - 微信日活10亿+，每条消息的已读回执是巨大的写放大（1条消息→N个已读回执）
   - 群消息已读数维护成本极高（100人群=100条已读状态/消息）
   - 已读状态是高频变化数据，存Redis成本高，存DB写压力大
3. **权衡**：微信选择牺牲"已读"功能换取系统简洁和用户体验

**面试加分点**：能从产品+技术两个维度分析，而不是只说"技术做不到"。

### 离线消息存储

| 方案 | 存储 | 适用场景 | 问题 |
|------|------|---------|------|
| MQ存储 | Kafka/RocketMQ | 实时流，短保留期 | 消息过期后丢失，不适合长期离线 |
| Redis | Sorted Set | 7天内离线消息 | 内存成本高，大量离线消息不划算 |
| MySQL | 离线消息表 | 任意时长 | 查询性能，需分表 |
| HBase | 行键=userId+msgId | 海量历史消息 | 延迟较高 |

**大厂实践（分层存储）**：

```
离线消息分层存储：

热数据（7天内）：Redis Sorted Set
  → score=timestamp，member=消息ID
  → 上线后按时间范围拉取
  → 过期时间7天

温数据（7-30天）：MySQL离线消息表
  → 按userId分表
  → 上线后先查Redis，查不到再查MySQL

冷数据（30天+）：HBase/对象存储
  → 压缩存储
  → 极少访问

用户离线30天后上线的处理：
  → 先拉Redis（可能已过期）
  → 再拉MySQL（7-30天的消息）
  → 再拉HBase（30天前的消息）
  → 按需分页加载，不一次性全量拉取
  → UI上显示"以上是历史消息"，不自动滚动到底部

架构权衡：
  → 全量存Redis：成本太高（每条消息×每对用户）
  → 全量存DB：延迟太高
  → 分层存储：成本和延迟的平衡点
  → 关键：不是所有离线消息都要"送达"，30天前的消息用户大概率不需要
```

## 题目四：IM超大群消息扩散 — 读扩散 vs 写扩散

### 读扩散 vs 写扩散

#### 核心概念

```
写扩散（Push模型）：
  发送者写1次 → 系统复制N份 → 每个收件人收件箱各1份
  读的时候：直接读自己收件箱，O(1)
  写的时候：O(N)，N=群人数

读扩散（Pull模型）：
  发送者写1次 → 只存发送者发件箱
  读的时候：遍历所有关注的人的发件箱，聚合排序
  写的时候：O(1)
  读的时候：O(M)，M=关注的人数
```

#### 对比

| 维度 | 写扩散 | 读扩散 |
|------|--------|--------|
| 写成本 | O(N)，群越大越慢 | O(1) |
| 读成本 | O(1)，读自己的收件箱 | O(M)，聚合多人发件箱 |
| 存储成本 | N份副本 | 1份原稿 |
| 实时性 | 高（预先推好） | 低（拉取有延迟） |
| 一致性 | 最终一致（异步复制） | 强一致（读原稿） |
| 删除/修改 | 需要改N份 | 改1份 |
| 适用 | 小群、读多写少 | 大群、写多读少 |

#### 大厂选择

| 产品 | 模式 | 原因 |
|------|------|------|
| 微信普通群（<500人） | 写扩散 | 群小，写放大可控，读取快 |
| 微信大群/公众号 | 读扩散 | 粉丝量太大，写扩散扛不住 |
| 微博 | 写扩散 | 读多写少，明星粉丝千万级需要预推 |
| 钉钉大群 | 推拉结合 | 大群读扩散降写压力，在线用户推送保实时性 |

### 10000人群的写扩散问题与优化

#### 写扩散的问题

```
1条消息 × 10000人 = 10000次写操作

问题：
  ① 写放大：1条消息产生10000条副本
  ② 延迟：最后一个人收到消息可能要几秒
  ③ 存储膨胀：10000份消息存储
  ④ 失败处理：第5000份写失败怎么办？部分人收不到？
  ⑤ 删除/撤回：要删除10000份副本
```

#### 大厂优化方案

**1. 异步扩散（核心优化）**

```
发送者 → 写自己的发件箱（同步，用户可感知）
       → 写扩散任务到MQ（异步）
       → 返回发送成功

MQ消费者：
  → 批量写收件人收件箱
  → 批量大小100-500
  → 写失败的记录补偿表，定时重试

好处：发送者不感知扩散延迟
代价：接收者可能有1-2秒延迟
```

**2. 分片扩散**

```
10000人群 → 按收件人ID分片
  → 10个分片，每片1000人
  → 10个Consumer并行扩散
  → 扩散速度提升10倍

分片策略：userId % shardCount
```

**3. 合并扩散**

```
高频发言场景：群内1分钟内发了50条消息

优化：不是每条消息都扩散10000次
  → 累积N条消息后批量扩散
  → 1次扩散包含50条消息ID
  → 收件人收件箱只写1次，包含50个消息指针

适用：消息量大但实时性要求不极致的场景
```

**4. 推拉结合（终极方案）**

```
在线用户：推（写扩散，但只推在线的人，可能只有1000人在线）
离线用户：拉（读扩散，上线后拉取未读消息）

效果：
  → 实际写扩散量 = 在线人数（远小于群总人数）
  → 离线用户上线后拉取，不占用写扩散资源
  → 微信大群就是这个方案
```

### 未读数一致性

#### 设计方案

```
未读数存储：Redis Hash
  Key: unread:{userId}
  Field: {conversationId}
  Value: 未读消息数

更新逻辑：
  → 群内有人发消息：所有群成员的未读数+1（写扩散）
  → 用户打开会话：未读数清零
  → 多端同步：清零操作通过IM通道推送所有端

问题1：写扩散更新10000人的未读数
  → 用Redis Pipeline批量HINCRBY
  → 一次Pipeline处理1000人
  → 10000人 = 10次Pipeline，耗时约50ms

问题2：消息丢了，未读数不一致
  → 定期对账：Redis未读数 vs DB实际未读消息数
  → 对账频率：5分钟一次
  → 不一致时以DB为准修复Redis

问题3：多端同时操作
  → 用户手机端和PC端同时打开同一会话
  → 每个端独立ACK，服务端以最先ACK的为准
  → 未读数变化推送到所有端

问题4：超大群未读数风暴
  → 1条消息 → 10000次HINCRBY → Redis写压力
  → 优化：异步批量更新（MQ消费）
  → 允许短暂不一致（1-2秒延迟可接受）
```

## 题目五：MQ运维监控 — 架构排查视角

### 消费者lag飙升排查SOP

#### 0-5分钟：止血+评估

```
1. 确认严重程度
   → lag当前值？增速？（50万→200万用了多久？）
   → 涉及哪个Topic/ConsumerGroup？
   → 是否核心链路？（资金/订单 vs 日志/通知）

2. 一眼定位常见原因
   → 消费端是否有新部署？（看部署记录）
   → 下游服务是否正常？（调用下游的健康检查接口）
   → DB是否慢查询？（看DB监控/慢查询日志）
   → 消费端日志是否有ERROR？（看消费失败日志）

3. 止血决策
   → 如果是非核心ConsumerGroup → 临时暂停消费，释放资源
   → 如果是核心链路且消费端异常 → 快速回滚最近部署
   → 如果是下游问题 → 联系下游团队，必要时降级
```

#### 5-30分钟：定位根因

```
4. 区分问题段
   → 生产端突增？看生产TPS是否异常上升
   → 消费端变慢？看消费RT是否上升
   → Rebalance风暴？看ConsumerGroup成员变化频率

5. 常见根因清单
   ┌──────────────────────────────────────────────────┐
   │ 消费端变慢                                        │
   │   → 下游服务RT升高（查下游监控）                    │
   │   → DB慢查询（查慢查询日志）                        │
   │   → GC频繁（查JVM监控）                            │
   │   → 线程池满（查线程池队列深度）                     │
   │   → 代码bug（如死循环、大对象序列化）                │
   │                                                    │
   │ 生产端突增                                          │
   │   → 上游流量高峰（查网关监控）                       │
   │   → 定时任务触发（查任务调度日志）                    │
   │   → 补偿任务重发大量历史消息                         │
   │                                                    │
   │ Rebalance风暴                                      │
   │   → Consumer频繁上下线（查Consumer变化日志）          │
   │   → 心跳超时配置不合理                              │
   │   → 容器环境Pod驱逐                                 │
   │                                                    │
   │ Broker问题                                         │
   │   → 磁盘IO高（查iostat）                           │
   │   → 网络问题（查Broker网络监控）                     │
   │   → PageCache竞争（同步刷盘+消费同时进行）           │
   └──────────────────────────────────────────────────┘

6. 确认后采取措施
   → 消费端变慢：优化下游/加消费线程/限流
   → 生产突增：上游限流/消息分级（非核心跳过）
   → Rebalance：固定Consumer数量/调大心跳超时
   → Broker：扩容/加SSD/调刷盘策略
```

#### 30分钟+：根治+预防

```
7. 根治
   → 增加容量水位预警（lag > 阈值自动告警）
   → 消息分级机制（核心/非核心不同ConsumerGroup）
   → 消费端熔断降级（下游不可用时快速跳过非核心）
   → 扩容预案（Consumer数量 × Queue数量的映射关系提前规划）

8. 复盘
   → 根因是什么？
   → 从告警到止血用了多久？
   → 哪个环节可以自动化？
   → 容量评估是否需要重新做？
```

### Broker磁盘水位85%告警

#### 可能的根因

```
1. 消费太慢 → 消息积压 → CommitLog持续增长
   → 回到上一题的排查SOP

2. 消息保留时间过长
   → 默认72小时，某些Topic设了168小时甚至更长
   → 检查Topic级别的fileReservedTime配置

3. 大消息 Topic
   → 某个Topic消息体特别大（如JSON报文>1MB）
   → 检查各Topic的消息大小分布

4. 过期消息未清理
   → RocketMQ的过期清理依赖CommitLog删除
   → 如果磁盘使用率>75%触发清理，但清理速度<写入速度
   → 检查deleteWhen和diskMaxUsedSpaceRatio配置

5. 没做Topic生命周期治理
   → 大量废弃Topic还在写入
   → 临时Topic未清理
   → 测试Topic混在生产集群
```

#### 短期/中期/长期处理

```
短期（1小时内）：
  → 确认是否消费积压导致 → 加速消费
  → 手动清理已确认废弃的Topic
  → 降低非核心Topic的消息保留时间
  → 如果磁盘>90%：紧急扩磁盘或迁移Broker

中期（1-7天）：
  → 接入Topic生命周期管理系统
  → 设置磁盘水位多级告警（75%/85%/90%）
  → 评估是否需要扩容（加Broker或加磁盘）

长期（持续）：
  → Topic治理：申请审批 → 创建 → 使用 → 销毁全生命周期管理
  → 为什么大厂必须做Topic生命周期治理？
    → 一个废弃Topic的持续写入 = 每天几十GB磁盘消耗
    → 100个废弃Topic = 每天几TB浪费
    → 大厂集群通常有上千个Topic，人工管理不可行
    → 治理不做好，磁盘永远不够用
  → 容量规划：按业务增长预测磁盘需求，提前扩容
```

### MQ集群容量评估方法论

#### 评估步骤

```
1. 单机基准测试
   → 使用官方benchmark工具
   → 测试场景：1Topic/16Queue/不同消息大小
   → 记录指标：TPS / RT / CPU / 磁盘IO / 网络IO
   → 压到极限后看瓶颈在哪（CPU？磁盘？网络？）

2. 瓶颈识别
   → CPU瓶颈：TPS上不去，CPU>80%
   → 磁盘瓶颈：iowait>30%，写RT飙升
   → 网络瓶颈：带宽跑满，网卡丢包
   → 内存瓶颈：PageCache不够，频繁换页

3. 容量水位线
   → 安全水位：单机最大TPS的60%
   → 告警水位：单机最大TPS的70%
   → 熔断水位：单机最大TPS的80%
   → 为什么不能跑80%+？
     → 留buffer应对突发流量（日常3倍高峰很常见）
     → 一台Broker宕机后，流量要分给其他Broker
     → GC/IO抖动会导致水位瞬间冲高
     → 80%运行 = 没有任何容错空间

4. 限流阈值
   → 生产端限流：安全水位 × 集群Broker数 × 0.8
   → 消费端限流：按ConsumerGroup分别设置
   → 限流策略：令牌桶/滑动窗口，超限返回REJECT

5. 容量评估公式
   → 单机安全TPS = 基准TPS × 60%
   → 集群安全TPS = 单机安全TPS × Broker数 × 0.8
     （0.8考虑一台Broker宕机的冗余）
   → 需要Broker数 = 业务峰值TPS / 单机安全TPS + 1（冗余）
```

#### 评估报告模板

```
集群容量评估报告

1. 基准数据
   - 单机最大TPS：____（消息大小1KB）
   - 瓶颈：____（CPU/磁盘IO/网络）
   - 单机最大吞吐：____MB/s

2. 当前负载
   - 集群Broker数：____
   - 当前峰值TPS：____
   - 当前磁盘使用率：____%
   - 当前CPU使用率：____%

3. 容量水位
   - 安全TPS：____（基准×60%×Broker数×0.8）
   - 当前水位：____%（当前峰值/安全TPS）
   - 剩余容量：____TPS

4. 扩容建议
   - 当前集群可支撑业务增长：____倍
   - 预计达到告警水位时间：____个月
   - 建议扩容时间：提前____个月
   - 建议扩容Broker数：____台
```
