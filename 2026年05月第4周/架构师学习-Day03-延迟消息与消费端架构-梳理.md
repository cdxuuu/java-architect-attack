# Day 3：延迟消息与消费端架构 — 架构师梳理

## 架构师视角：延迟消息的本质是"时间维度的路由"

### 从开发到架构师的思维跃迁

| 层次 | 思维方式 | 典型回答 |
|------|---------|---------|
| 开发 | "延迟消息就是消息晚一点投递" | 用RocketMQ的setDelayTimeLevel就行 |
| 高级开发 | "延迟消息需要定时调度机制" | RocketMQ用内部Topic + 定时线程，5.x有时间轮 |
| 架构师 | "延迟消息是时间维度的路由，和空间维度（Topic/Queue）路由是同一件事" | 延迟消息 = 按时间路由到不同的"虚拟Queue"，时间到了再路由到真实Topic |

### 延迟消息的架构本质

```
普通消息路由：按Key路由到空间维度（哪个Queue）
延迟消息路由：按时间路由到时间维度（什么时候投递）

两者统一视角：
  普通消息：路由条件 = 业务Key → 路由目标 = Queue
  延迟消息：路由条件 = 时间戳 → 路由目标 = 时间桶（Time Bucket）

RocketMQ旧版：18个时间桶 = 18个延迟级别
RocketMQ 5.x：时间轮 = 精细时间桶
Kafka：没有时间桶 → 不支持延迟消息
```

### 架构师选延迟消息方案的决策框架

```
第一问：现有MQ是否原生支持？
  是 → 用原生支持（零额外成本）
  否 → 进入第二问

第二问：延迟消息量级？
  百级/千级 → Redis ZSET + 定时轮询（够用）
  万级/十万级 → 时间轮 + DB持久化（Redis内存扛不住）
  百万级以上 → 自研延迟服务（需要集群化）

第三问：精度要求？
  分钟级 → Redis ZSET轮询5秒一次够了
  秒级 → 时间轮
  毫秒级 → 内存时间轮（但不能持久化，重启丢消息）

第四问：可靠性要求？
  可丢（日志延迟处理） → 内存队列即可
  不可丢（支付超时取消） → 必须持久化 + 补偿机制
```

### 反向论证：什么时候不用延迟消息

| 场景 | 不用延迟消息 | 用什么 |
|------|------------|--------|
| 延迟时间极短（<1秒） | 延迟消息调度开销比延迟本身大 | 直接定时任务或延迟线程 |
| 延迟时间极长（天/周级） | MQ存大量未到期消息浪费存储 | DB定时Job扫描 |
| 延迟触发逻辑简单 | 引入MQ过重 | 数据库事件 + 定时器 |
| 需要取消延迟 | 延迟消息投出后无法撤回 | DB状态标记 + 消费时检查 |

## 架构师视角：Rebalance 是分布式协调的缩影

### Rebalance 的本质是什么？

```
Rebalance = 消费者组内的资源重分配

本质是分布式协调问题：
  谁来协调？→ Broker/Coordinator
  怎么感知变化？→ 心跳
  怎么分配？→ 分配策略
  怎么保证一致性？→ 全组暂停 + 重新分配

和哪些分布式问题是同一类？
  - 分布式锁的租约续期（心跳 = 租约续期）
  - 服务注册发现（Consumer注册 = 服务注册）
  - Raft的Leader选举（Coordinator = Leader）
  - 一致性Hash的数据迁移（Queue重分配 = 数据迁移）
```

### Rebalance 问题的分层防御

```
L1 预防：减少Rebalance触发
  - 心跳超时调大（默认20s → 60s，减少GC/网络抖动误判）
  - Consumer端健康检查（处理慢时主动暂停消费而非超时）
  - 扩缩容错峰（秒杀前提前扩，不要在高峰期扩）

L2 缓解：减少Rebalance影响
  - Sticky分配策略（尽量保持原分配，减少迁移量）
  - Cooperative Rebalance（增量分配，不全组暂停）
  - 幂等消费（兜底重复消费问题）

L3 补偿：处理Rebalance后果
  - 重复消息 → 幂等兜底
  - 消息积压 → 扩容 + 转接（Day1方案）
  - 顺序消费中断 → 失败落库 + Job补偿（Day1方案）
```

### 秒杀系统的扩缩容架构

```
秒杀前1小时：
  1. 从4个Consumer扩到16个
  2. 确保Queue数量 ≥ 16（否则扩Consumer无效）
  3. 验证Rebalance完成，所有Consumer正常消费
  4. 预热消费者连接和本地缓存

秒杀进行中：
  1. 监控Rebalance频率（>1次/分钟 = 异常）
  2. 不做任何扩缩容操作（避免触发Rebalance）
  3. 积压时用转接方案（Day1），不扩Consumer

秒杀后30分钟：
  1. 消息消费完毕后缩回4个Consumer
  2. 触发Rebalance → 此时流量已低，影响小

架构师关键决策：
  Queue数量在Topic创建时就按峰值规划（如32个）
  Consumer数量按需扩缩，但永远不超过Queue数量
```

## 架构师视角：Push vs Pull 不是选择，是场景适配

### 统一理解框架

```
所有消息消费的本质：Consumer从Broker获取消息

区别只在"谁来发起请求"：
  Pull：Consumer主动 → Consumer控制节奏
  Push：Broker主动 → Broker控制节奏

为什么RocketMQ的"Push"是Long Polling？
  因为纯Push有不可解的问题：
    1. Broker要维护每个Consumer的推送状态 → 有状态 → 难扩展
    2. Consumer处理慢 → Broker推送积压 → 内存溢出
    3. Consumer宕机 → Broker推送失败 → 重试风暴

  Long Polling = Pull + 长连接挂起
  效果：Consumer主动（无状态）+ 接近实时（挂起等待）
  兼得了Push的实时性和Pull的简洁性
```

### 消费模式对架构设计的影响

```
集群消费 → 负载均衡架构
  天然横向扩展 → 加Consumer提升吞吐
  但有Queue数量上限 → Queue数 = 最大并行度
  适用：订单处理、秒杀下单、库存扣减

广播消费 → 扇出架构
  每个Consumer独立消费全量 → 扩容不提升吞吐
  天然解耦 → 上游不关心下游有几个实例
  适用：配置变更、缓存刷新、事件通知
```

### 消息过滤对Topic设计的指导

```
架构师设计Topic时的三个层次：

层次1：按业务域分Topic
  ORDER_TOPIC / PAY_TOPIC / LOGISTICS_TOPIC
  → 物理隔离，独立监控、独立扩缩容

层次2：同一Topic内用Tag分消息类型
  ORDER_TOPIC + Tag: CREATE/PAY/SHIP/CANCEL
  → 逻辑隔离，共享存储，Tag过滤高效

层次3：复杂条件用SQL92过滤
  ORDER_TOPIC + 属性: amount=100, region=BJ
  → 按属性条件过滤，灵活但性能低

设计原则：
  先按业务域分Topic（粗粒度隔离）
  同一业务域内用Tag分类型（细粒度路由）
  只有跨Tag的复杂条件才用SQL92
  不要把所有消息放一个Topic靠Tag/SQL过滤
```

### Topic设计的反向论证

```
过度拆Topic的代价：
  - Topic多了 → 管理复杂，运维成本高
  - Kafka中Topic多了 → Partition文件争抢页缓存 → 性能退化
  - 小流量Topic浪费资源

过度合并Topic的代价：
  - 不同业务消息混在一起 → 监控无法区分
  - 消息量差异大 → 小业务被大业务淹没
  - 无法独立扩缩容

架构师平衡点：
  按业务域拆（3-10个核心Topic）
  业务域内用Tag（每个Topic2-5个Tag）
  超过5个Tag考虑拆Topic
```

## 今日架构师思维训练总结

### 三道架构师级追问

**追问1：秒杀系统的订单超时取消，延迟消息和定时任务扫描哪个好？**

```
定时任务扫描方案：
  每分钟扫描 order 表 where status=UNPAID and create_time < now()-15min
  优势：实现简单，可查可审计
  劣势：订单量大时扫描慢，数据库压力大

延迟消息方案：
  下单时发延迟消息 → 15分钟后消费 → 检查订单状态 → 未支付则取消
  优势：不查DB，消息驱动
  劣势：消息丢了就不取消（需要补偿）

架构师答案：两者组合使用
  L1：延迟消息做实时取消（秒级精度）
  L2：定时任务做兜底巡检（分钟级兜底）
  这就是分层防御——延迟消息可能丢，定时任务兜底
```

**追问2：广播消费的Consumer宕机，消息丢了怎么办？**

```
集群消费：Consumer宕机 → Queue被其他Consumer接管 → 消息不丢
广播消费：Consumer宕机 → 没人接管 → 消息丢了

解决思路：
  1. 广播消费的offset存在Consumer本地 → 重启后从上次位置继续
  2. 如果是永久宕机（实例被删除）→ 消息真的丢了
  3. 所以广播消费只适合"丢了也无所谓"或"可以从其他地方恢复"的场景
     - 缓存刷新 → 丢了无所谓，下次刷新就好
     - 配置变更 → 丢了可以从配置中心重新拉取
  4. 如果广播消费的消息不能丢 → 不要用广播消费，改用多Group集群消费
```

**追问3：Consumer处理速度差异大，怎么保证负载均衡？**

```
Push模式（RabbitMQ真Push）：
  prefetchCount控制 → 慢的Consumer少推，快的多推 → 自然均衡

Pull模式（Kafka）：
  慢的Consumer poll慢 → 拉得少 → 快的poll快拉得多
  但！Kafka是按分区分配的 → 一个分区只给一个Consumer
  → 快的Consumer处理完了也只能等 → 负载不均衡
  → 需要增加分区数来提高并行度

RocketMQ "Push"模式：
  也是按Queue分配 → 同Kafka的问题
  → Queue:Consumer = 1:1 → 快的Consumer等慢的

架构师方案：
  1. 增加Queue数量 → 更细粒度分配
  2. 消费端异步处理 → 消费线程池 → 快慢不取决于业务处理速度
  3. 如果业务处理确实差异大 → 按消息类型分Topic → 独立Consumer组
```
