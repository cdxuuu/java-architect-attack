# Day 4：消息重试·死信队列·积压实战 — 架构师梳理

## 一、架构师视角：重试不是兜底，是设计

### 开发思维 vs 架构师思维

```
开发：消费失败了怎么办？→ 重试
架构师：消费失败了怎么办？→ 先问三个问题：
  1. 为什么失败？是临时故障还是永久错误？
  2. 重试有没有副作用？会不会重复处理？
  3. 重试到什么时候停止？停了之后怎么办？
```

### 重试策略的决策矩阵

| 失败原因 | 重试策略 | 理由 |
|---------|---------|------|
| 下游服务超时 | 递增重试 | 临时故障，恢复后自动成功 |
| 下游服务宕机 | 递增重试+限制次数 | 等恢复但不能无限等 |
| 数据格式错误 | 不重试，直接进死信 | 重试也不会成功 |
| DB死锁 | 快速重试 | 死锁释放后马上能成功 |
| 业务校验失败 | 不重试，记录日志 | 重试无意义，需人工排查 |
| 网络闪断 | 快速重试1-2次 | 网络瞬时恢复概率高 |

### 架构师要设计的：全链路重试策略

```
                    消费失败
                       │
              ┌────────┴────────┐
              │                 │
         可重试错误         不可重试错误
              │                 │
     ┌────────┴────────┐       │
     │                 │       ▼
  快速恢复型       慢恢复型    直接进死信
  (网络闪断/死锁)  (服务宕机)  (数据错误)
     │                 │       │
  快速重试1-3次    递增重试     告警
  (间隔1-5秒)     (10s→2h)   人工介入
     │                 │
     │            16次失败
     │                 │
     │                 ▼
     │            进死信队列
     │                 │
     │            告警+人工介入
     │                 │
     ▼                 ▼
  成功 → 继续      对账补偿 → 最终一致
```

## 二、死信队列：最后一道防线的工程化

### 死信队列不是终点，是处理的起点

```
错误理解：消息进死信 = 丢了
正确理解：消息进死信 = 自动重试结束，人工接管开始

死信队列工程化三件事：
  1. 监控：死信队列消息数 > 0 → 告警
  2. 可视化：管理后台能看到死信消息内容和失败原因
  3. 可操作：支持一键重发、修改后重发、标记已处理
```

### 死信处理的分层策略

```
L1：自动重发（80%场景）
  修复下游问题后 → 自动重发死信到原Topic → 正常消费

L2：修改后重发（15%场景）
  数据格式错误 → 人工修正消息内容 → 重发到原Topic

L3：手动补偿（4%场景）
  业务逻辑变化 → 手动在DB中补偿数据 → 标记死信已处理

L4：数据修复（1%场景）
  多条死信关联 → 需要批量修复DB → DBA介入
```

### 死信队列的运维指标

| 指标 | 阈值 | 告警级别 |
|------|------|---------|
| 死信队列消息数 | > 0 | P1 |
| 死信队列消息数 | > 10 | P0 |
| 死信消息滞留时间 | > 1小时 | P1 |
| 死信消息涉及金额 | > 0 | P0 |
| 死信消息占比 | > 0.01% | P2 |

## 三、积压：架构师要的是"不积压"而不是"积压了怎么办"

### 积压的本质：生产速度 > 消费速度

```
架构师要解决的不是"积压了怎么办"，而是"为什么消费速度上不去"：

消费速度上不去的原因：
  1. 消费逻辑慢（DB操作、远程调用、复杂计算）
  2. 消费并行度低（Queue数量少、顺序消费限制）
  3. 消费端资源不够（线程池小、CPU/内存不足）
  4. 下游限流（DB扛不住、第三方限速）
  5. 消费端异常（GC频繁、OOM、代码bug）
```

### 从设计层面防积压

```
设计阶段就考虑容量：
  1. 生产峰值是多少？→ 压测
  2. 消费能力是多少？→ 压测
  3. 消费能力 >= 峰值 × 1.5 → 安全
  4. 消费能力 < 峰值 → MQ就是缓冲，必须设计降级

弹性设计：
  Consumer部署在K8s → HPA自动扩缩容
  Queue数量预留余量（平时8个，需要时可以扩到32个）
  消费端支持动态线程池调整

降级设计：
  消息分级（P0-P3）→ 积压时跳过低优先级
  消费端熔断 → 下游不可用时暂停消费，不浪费重试
  旁路消费 → 积压时转发到临时Topic加速
```

### 积压应急的SOP（标准操作流程）

```
0-1分钟：发现 → 确认积压量和增长速率
1-3分钟：定位原因 → 消费端/生产端/Rebalance
3-5分钟：止血 → 重启/回滚/限流
5-15分钟：加速消费 → 扩容/转发/批量消费/跳过非核心
15-30分钟：监控恢复 → 确认积压量下降 → 调整策略
30分钟+：事后复盘 → 根因分析 → 优化设计防止再发生
```

## 四、消息回溯：安全的"后悔药"

### 回溯的使用场景分类

| 场景 | 回溯方式 | 注意事项 |
|------|---------|---------|
| 代码bug修复后 | 按时间戳回溯到bug上线前 | 必须幂等 |
| 数据统计修正 | 按时间范围回溯 | 限流消费避免冲击 |
| 消费端重启补数据 | 从最早开始消费 | 确认不会重复处理 |
| 精确排查问题 | 按Offset回溯 | 先确认Offset对应的消息 |

### 回溯的架构设计

```
回溯不是随便回的，需要"回溯基础设施"：

1. 消息保留策略
   - 保留时间足够长（建议7天以上）
   - 磁盘容量规划：每天消息量 × 保留天数 × 1.2（安全系数）

2. 回溯操作审批
   - 回溯影响范围大 → 需要审批
   - 金额相关Topic回溯 → P0审批
   - 通知类Topic回溯 → P2审批

3. 回溯执行方案
   - 评估回溯消息量
   - 限流消费速率
   - 准备回滚方案（回溯导致问题时怎么恢复）

4. 回溯后验证
   - 对比回溯前后的数据一致性
   - 检查是否有重复处理
   - 确认消费进度正常推进
```

## 五、消费端性能：架构师的系统化调优

### 消费端性能调优框架

```
从全局到局部，依次排查：

1. 架构层面
   - Queue数量是否够？→ 并行度上限
   - Consumer数量是否够？→ 资源利用
   - 消费模式是否合适？→ 并发 vs 顺序

2. 消费端配置
   - consumeThreadMin/Max → 线程池大小
   - pullBatchSize → 拉取批量
   - consumeMessageBatchMaxSize → 消费批量
   - pullThresholdForQueue → 本地缓存限制

3. 业务逻辑
   - DB操作是否批量？→ 逐条改批量
   - 远程调用是否可以异步？→ 同步改异步
   - 热点数据是否缓存？→ 减少DB查询

4. 下游依赖
   - DB是否扛得住？→ 读写分离/分库
   - 第三方是否限速？→ 限流/熔断
   - 网络是否延迟高？→ 同机房部署

5. JVM层面
   - GC是否频繁？→ 调整堆大小/GC算法
   - 是否OOM？→ 检查ProcessQueue大小配置
   - 线程是否阻塞？→ 线程dump分析
```

### 消费端配置决策表

| 场景 | consumeThreadMin | consumeThreadMax | pullBatchSize | consumeMessageBatchMaxSize |
|------|-----------------|-----------------|---------------|--------------------------|
| 轻量业务（通知推送） | 20 | 64 | 32 | 1 |
| 重量业务（涉及DB事务） | 10 | 20 | 16 | 1 |
| 批量写入场景 | 20 | 64 | 32 | 32 |
| 顺序消费场景 | 20 | 20 | 32 | 1 |
| 积压应急 | 40 | 128 | 64 | 1 |

## 六、今日主题的全局串联

### Day1-Day4的完整链路

```
生产端                    Broker                   消费端
  │                        │                        │
  │── 发送消息 ──────────→│                        │
  │   (Day1: 同步发送     │                        │
  │    保证不丢)          │                        │
  │                        │── 投递消息 ──────────→│
  │                        │   (Day3: Push/Pull)    │
  │                        │                        │── 消费处理
  │                        │                        │   (Day4: 线程池/流控)
  │                        │                        │
  │                        │←── 消费结果 ───────────│
  │                        │   成功: ACK            │
  │                        │   失败: RECONSUME_LATER│
  │                        │                        │
  │                        │── 重试投递 ──────────→│
  │                        │   (Day4: 16次递增重试) │
  │                        │                        │
  │                        │  16次失败              │
  │                        │── 进死信队列            │
  │                        │   (Day4: DLQ)          │
  │                        │                        │
  │                        │←── 回溯请求            │
  │                        │── 从历史offset重新投递  │
  │                        │   (Day4: 消息回溯)     │

问题链：
  消息不丢(Day1) → 可能重复 → 必须幂等(Day1)
  消费失败 → 重试(Day4) → 重复消费 → 幂等兜底(Day1)
  重试耗尽 → 死信(Day4) → 人工介入 → 对账补偿
  消费太慢 → 积压(Day1+Day4) → 扩容 → Rebalance(Day3) → 可能更慢
  代码bug → 回溯(Day4) → 重复消费 → 幂等兜底(Day1)

核心闭环：幂等是所有消费端问题的兜底
```

## 七、面试答题模板

### 题目：消息消费失败怎么处理？

```
第一层（基础）：消费失败返回RECONSUME_LATER，RocketMQ自动重试16次

第二层（数字）：重试间隔递增，第1次10秒，第16次2小时，总计约4小时47分钟

第三层（分层）：
  L1 自动重试（临时故障自愈）
  L2 死信队列（重试耗尽，人工接管）
  L3 对账补偿（定时Job巡检，兜底一致性）
  L4 人工修复（极端情况）

第四层（反向论证）：
  不是所有失败都应该重试——数据格式错误重试也不会成功，应该直接进死信
  金额场景不能等4小时，重试次数要调少

第五层（串联）：
  重试必然重复消费 → 幂等是前提（Day1）
  顺序消费失败会阻塞整个Queue → 容易积压（Day1+Day4）
  广播消费没有重试机制 → 需要自己实现（Day3）
```

### 题目：线上消息积压了怎么办？

```
第一层（应急）：三板斧——确认原因、止血、加速消费

第二层（方案）：扩容Consumer / 临时Topic转发 / 批量消费 / 跳过非核心消息

第三层（数字）：Queue数量是并行度上限；单Consumer简单业务1-2万TPS，涉及DB约1000-3000TPS

第四层（架构）：积压是设计问题不是运维问题——容量规划要留1.5倍余量，监控要提前告警

第五层（串联）：扩容触发Rebalance（Day3）→ 短暂暂停 → 可能加重积压 → 要提前扩不要临时扩
```
